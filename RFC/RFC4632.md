## 1. 서론 (Introduction)

### 1.1 문서 목적
- 32비트 IPv4 주소 공간의 할당 전략 정의
- 주소 공간 보존과 글로벌 라우팅 상태 증가 제한이 목표
- RFC 1519 폐기 및 12년 이상의 배포 결과 반영

---

## 2. 역사 및 문제 설명 (History and Problem Description)

### 2.1 배경
- 인터넷: 1970년대 연구 프로젝트로 시작
- 32비트 주소 공간을 세 가지 클래스로 구성

### 2.2 클래스 기반 주소 체계

| 클래스 | 네트워크 수 | 호스트 수/네트워크 | 용도 |
|--------|------------|-------------------|------|
| Class A | 128개 | 16,777,214개 | 대규모 조직 |
| Class B | 16,384개 | 65,534개 | 중규모 조직 |
| Class C | 2,097,152개 | 254개 | 소규모 조직 |

### 2.3 1992년 ROAD 그룹이 식별한 세 가지 주요 문제

#### 문제 1: Class B 주소 공간 고갈
- 중규모 조직에 적합한 크기의 클래스 부재
- Class C (254개 호스트)는 너무 작음
- Class B (65,534개 호스트)는 너무 큼
- Class B 할당 요청 폭증

#### 문제 2: 라우팅 테이블 폭발적 증가
- 소프트웨어/하드웨어 처리 능력 초과
- 운영 관리 능력 한계
- 라우터 메모리 및 CPU 부담

#### 문제 3: 32비트 IPv4 주소 공간 장기 고갈
- 약 43억 개의 주소 한계
- 근본적인 주소 공간 부족 문제
- 1993-1995년 사이 심각해질 것으로 예상

### 2.4 대응 문서
- RFC 1338: Supernetting 개념 도입
- RFC 1519: CIDR 명세 최초 발행

## 3. 해결책으로서의 클래스리스 주소 지정 (Classless Addressing as a Solution)

### 3.1 기본 개념
- 클래스 기반 체계 폐기
- "클래스리스" 계층적 주소 블록(프리픽스) 도입
- 프리픽스 할당이 인터넷 토폴로지를 따름
- 라우팅 정보 집계 가능

### 3.2 CIDR 표기법 (Slash Notation)

#### 형식
```
a.b.c.d/X
```
- `a.b.c.d`: 네트워크 주소
- `/X`: 프리픽스 길이 (네트워크 비트 수)

#### 예시
| CIDR 표기 | 설명 | 서브넷 마스크 |
|-----------|------|--------------|
| `172.16.0.0/16` | 레거시 Class B | 255.255.0.0 |
| `192.168.99.0/24` | 레거시 Class C | 255.255.255.0 |
| `10.0.0.0/8` | 레거시 Class A | 255.0.0.0 |

### 3.3 프리픽스 크기 테이블

| 프리픽스 | 주소 수 | 설명 |
|---------|---------|------|
| /8 | 16,777,216 | 레거시 Class A |
| /16 | 65,536 | 레거시 Class B |
| /24 | 256 | 레거시 Class C |
| /25 | 128 | Class C의 1/2 |
| /26 | 64 | Class C의 1/4 |
| /27 | 32 | Class C의 1/8 |
| /28 | 16 | Class C의 1/16 |
| /29 | 8 | 소규모 네트워크 |
| /30 | 4 | 지점 간 링크 (2개 사용 가능) |
| /31 | 2 | 지점 간 링크 (RFC 3021) |
| /32 | 1 | 호스트 라우트 |
| /0 | 4,294,967,296 | 기본 라우트 (전체) |

### 3.4 가변 길이 서브넷 마스크 (VLSM)
- 네트워크를 다양한 크기의 서브넷으로 분할 가능
- 주소 공간의 효율적 활용
- 필요한 만큼만 주소 할당

## 4. 주소 할당 및 라우트 집계 (Address Assignment and Routing Aggregation)

### 4.1 라우트 집계 (Route Aggregation / Supernetting)

#### 개념
```
192.168.0.0/24 \
192.168.1.0/24  |---> 192.168.0.0/22 (집계)
192.168.2.0/24  |
192.168.3.0/24 /
```

#### 중요성
- **"집계는 라우팅 상태 감소의 유일하게 일반적으로 이해되는 방법"**
- CIDR 성공의 핵심 요소
- 토폴로지 기반 주소 할당 필수

### 4.2 집계 시나리오

#### 단순한 경우: 단일 ISP 연결
- 조직이 ISP의 주소 공간 사용
- 명시적 라우트 광고 불필요
- 완전한 집계 가능

#### 복잡한 경우 1: 멀티호밍 (Multihoming)
- 둘 이상의 ISP에 연결
- 모든 ISP가 해당 네트워크를 명시적으로 광고해야 함
- 집계 이점 제한적
- 글로벌 라우팅 테이블 항목 증가

#### 복잡한 경우 2: ISP 변경
- 새 ISP가 구체적 프리픽스 광고 필요
- 기존 ISP의 집계된 라우트에서 "구멍" 발생
- 재번호화(Renumbering) 권장되나 어려움

### 4.3 계층적 주소 할당 구조

```
IANA
  └── /8 블록 할당
        └── Regional Internet Registries (RIRs)
              ├── ARIN (북미)
              ├── RIPE NCC (유럽/중동/중앙아시아)
              ├── APNIC (아시아태평양)
              ├── LACNIC (라틴아메리카/카리브해)
              └── AfriNIC (아프리카)
                    └── ISP/LIR에 블록 할당
                          └── 고객에게 추가 세분화
```

### 4.4 주소 할당 원칙
- 초기: 중앙 NIC이 Class A/B/C 임의 할당
- CIDR 이후: ISP에게 대량 블록 할당 → 고객에게 세분화
- 토폴로지에 따른 주소 할당으로 집계 효율 극대화

## 5. 라우팅 구현 고려사항 (Routing Implementation Considerations)

### 5.1 라우팅 프로토콜 지원

#### 클래스리스 지원 프로토콜 (권장)
| 유형 | 프로토콜 | 특징 |
|------|---------|------|
| Interior | OSPF | 링크 상태, VLSM 지원 |
| Interior | IS-IS | 링크 상태, VLSM 지원 |
| Interior | RIPv2 | 거리 벡터, 서브넷 마스크 포함 |
| Interior | EIGRP | 하이브리드, Cisco 독점 |
| Exterior | BGP4 | 경로 벡터, CIDR 완전 지원 |

#### 클래스리스 미지원 프로토콜 (폐기됨)
| 프로토콜 | 문제점 |
|---------|--------|
| RIPv1 | 서브넷 마스크 전달 불가 |
| HELLO | 클래스 기반 |
| IGRP | 클래스 기반 |
| EGP | 클래스 기반 |

### 5.2 라우트 광고 규칙

#### 규칙 1: 가장 긴 매치 (Longest Match) 기반 포워딩
- 여러 라우트가 일치하면 가장 구체적인(프리픽스가 긴) 라우트 선택
- 예: `192.168.16.0/20`과 `192.168.17.0/24` 중 후자 우선

#### 규칙 2: Null 라우팅 요구
- **"집계 라우트를 생성하는 라우터는 그 라우트와 일치하지만 더 구체적 라우트와 일치하지 않는 패킷을 폐기해야 함"**
- Null/Discard 인터페이스로 설정
- 라우팅 루프 방지

### 5.3 프리픽스 필터 형식 변경

#### 레거시 형식 (문제)
```
accept 172.16.0.0
```
- 마스크가 암묵적 (클래스 기반 추론)

#### CIDR 형식 (권장)
```
accept 172.16.0.0/16
```
- 마스크가 명시적
- 모호함 제거

### 5.4 집계 구성 원칙

#### AS (자율 시스템) 책임
- 할당받은 프리픽스의 집계 책임
- 가능한 한 집계하여 광고

#### 주의사항
- 동적 집계 시 라우트 플래핑 방지 필요
- **"프록시 집계"** (다른 AS 동의 없이 라우트 집계) 주의
  - 의도하지 않은 트래픽 경로 변경 가능
  - 트래픽 엔지니어링 문제 발생 가능

### 5.5 내부 라우팅에서 BGP 사용
- CIDR 이전: 외부 라우팅 → 내부 라우팅 전파 (EGP → OSPF/RIP)
- CIDR 이후: **iBGP (Internal BGP)** 사용
  - 다른 ISP로부터 학습한 라우트 전파
  - 풀 메시 또는 라우트 리플렉터 사용

## 6. 새 주소 할당 및 라우팅 예시

### 6.1 주소 위임 시나리오

#### ISP "PA"에 할당된 블록
- 블록: `10.24.0.0/13` (524,288 주소)

#### 고객 할당

| 고객 | 할당 블록 | 주소 수 | 비고 |
|------|----------|---------|------|
| C1 | `10.24.0.0/21` | 2,048 | 단일 연결 |
| C2 | `10.24.16.0/20` | 4,096 | 단일 연결 |
| C3 | `10.24.8.0/22` | 1,024 | 단일 연결 |
| C4 | `10.24.12.0/22` | 1,024 | 멀티홈 (PA+PB) |
| C5 | `10.24.32.0/23` | 512 | 멀티홈 (PA+PB) |
| C6 | `10.24.34.0/23` | 512 | 단일 연결 |

### 6.2 멀티호밍 및 ISP 변경 예시

#### C7: PB에서 PA로 이동한 사이트
- 기존 주소: PB 주소 공간 (`10.32.0.0/20`)
- 새 ISP: PA
- PA가 `10.32.0.0/20`을 명시적으로 광고해야 함

### 6.3 라우팅 광고

#### PA → 백본 (BB) 광고
```
10.24.12.0/22  (C4 - 멀티홈, 주 경로)
10.32.0.0/20   (C7 - 이전 고객, 주 경로)
10.24.0.0/13   (PA의 나머지 집계)
```

#### PB → 백본 (BB) 광고
```
10.24.12.0/22  (C4 - 멀티홈, 백업 경로)
10.24.32.0/23  (C5 - 멀티홈, 주 경로)
10.32.0.0/13   (PB의 나머지 집계)
```

## 7. DNS 고려사항 (Domain Name Service Considerations)

### 7.1 IN-ADDR.ARPA 문제
- 역방향 DNS 위임이 옥텟 경계에서만 가능
  - `/8`, `/16`, `/24` 경계에서만 자연스러운 위임
- 비트마스크 기반 CIDR 할당 시 유지 작업 증가
  - `/25`, `/26` 등은 옥텟 경계를 넘음

### 7.2 해결책
- **RFC 2317**: Classless IN-ADDR.ARPA 위임
  - CNAME 레코드를 사용한 위임 기법
  - 비옥텟 경계 주소 블록 지원

## 8. 장기 솔루션으로의 전환 (Transition to a Long-Term Solution)

### 8.1 CIDR의 위치
- 원래 3-5년 단기 솔루션으로 설계
- IPv4 문제의 ** 임시 완화책**
- 장기 솔루션(IPv6) 개발 동안 인터넷 기능 유지

### 8.2 한계
- IPv4의 근본적 주소 고갈 문제는 해결 불가
- 라우팅 테이블 증가 완전 억제 불가
- 멀티호밍 및 트래픽 엔지니어링 증가로 인한 비집계 라우트 증가

## 9. CIDR의 글로벌 라우팅 상태 영향 분석 (1988-2005)

### 9.1 시기별 분석

| 기간 | 특징 | 원인 |
|------|------|------|
| 1988-1994 | 지수 증가 | 레거시 클래스 할당, 초기 슈퍼넷 |
| 1994 | 급격한 감소 | BGP4 배포 |
| 1994-1999 | 선형 증가 | CIDR 정착, 적절한 집계 |
| 1999-2001 | 지수 증가 | 인터넷 버블, 신규 ISP 대거 등장 |
| 2001-2003 | 안정화 | 닷컴 붕괴, "CIDR 경찰" 활동 강화 |
| 2004-2005 | 새로운 지수 증가 | 경제 회복, 의도적 비집계 증가 |

### 9.2 2005년 3월 기준 통계
- 집계 엔트리: 약 **75,000개**
- 더 구체적(비집계) 엔트리: 약 **85,000개**
- 총 라우팅 테이블 크기: 약 **160,000개**

### 9.3 비집계 라우트 증가 원인
- 멀티호밍 확대
- 트래픽 엔지니어링 목적
- 부하 분산
- 장애 복구 (failover)

## 10. 결론 및 권장사항 (Conclusions and Recommendations)

### 10.1 CIDR 배포 성과
- ** 단기 위기 해결**
  - Class B 고갈 문제 완화
  - 라우팅 테이블 폭증 억제
- ** 성공 요인**
  - BGP4 지원과의 결합
  - 장비 제조업체와 ISP 커뮤니티의 협력
  - ISP 간 신뢰 관계 구축

### 10.2 현재 IPv4 라우팅 시스템 구성
- 집계 주소 광고 기반 토폴로지 정보
- 더 구체적 광고:
  - 트래픽 엔지니어링
  - 멀티호밍
  - 로컬 구성/정책

### 10.3 권장사항

#### 지수 성장 재개 완화를 위한 조치
1. **ISP 교육 강화**
   - 효율적 집계 전략
   - 적절한 주소 할당 정책
2. ** 장비 구성 개선**
   - 적절한 필터링
   - Null 라우팅 설정
3. ** 멀티홈 기술 개발**
   - 글로벌 라우팅 상태 불필요한 방식
   - 로컬/지역 수준 해결
4. ** 트래픽 부하 분산 개선**
   - 비집계 라우트 의존도 감소

### 10.4 핵심 메시지
> **"집계는 글로벌 인터넷에서 라우팅 확장성을 위한 유일한 도구입니다."**

## 12. 보안 고려사항 (Security Considerations)

### 12.1 트래픽 하이재킹

#### 공격 방식
- 더 구체적 프리픽스를 광고하여 트래픽 가로채기
- 예: `192.168.16.0/20` 대신 `192.168.17.0/24` 광고
- 가장 긴 매치 규칙으로 인해 트래픽 유인

#### 대응책
- ISP 간 신뢰 관계 구축
- 강력한 라우트 정책 필터
- RIR 지침 기반 프리픽스 길이 상한선 적용
  - 예: `/24`보다 작은 프리픽스는 거부

### 12.2 서비스 거부 공격 (DoS)

#### 공격 방식
- 수많은 라우트를 광고하여 라우팅/포워딩 테이블 오버플로우 유발
- **"AS 7007" 사건**: 실제 발생한 유명한 예시

#### 대응책
- 이웃으로부터 허용할 최대 프리픽스 수 구성
- 예: `maximum-prefix` 설정

```
router bgp 65001
  neighbor 10.0.0.1 maximum-prefix 1000
```

## 부록: CIDR 계산 예시

### A.1 프리픽스 길이와 호스트 수 계산

```
호스트 수 = 2^(32 - 프리픽스 길이) - 2
```

| 프리픽스 | 계산 | 사용 가능 호스트 |
|---------|------|-----------------|
| /24 | 2^8 - 2 | 254 |
| /25 | 2^7 - 2 | 126 |
| /26 | 2^6 - 2 | 62 |
| /27 | 2^5 - 2 | 30 |
| /28 | 2^4 - 2 | 14 |
| /29 | 2^3 - 2 | 6 |
| /30 | 2^2 - 2 | 2 |

※ 네트워크 주소와 브로드캐스트 주소 제외

### A.2 라우트 집계 예시

#### 4개의 /24 네트워크를 /22로 집계

```
192.168.0.0/24   (192.168.0.0 - 192.168.0.255)   \
192.168.1.0/24   (192.168.1.0 - 192.168.1.255)    |
192.168.2.0/24   (192.168.2.0 - 192.168.2.255)    |---> 192.168.0.0/22
192.168.3.0/24   (192.168.3.0 - 192.168.3.255)   /      (192.168.0.0 - 192.168.3.255)
```

#### 집계 가능 조건
- 연속된 주소 블록
- 2^n 개의 동일 크기 블록
- 집계 후 프리픽스가 정확히 시작 주소와 일치
