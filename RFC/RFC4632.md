# RFC 4632: Classless Inter-Domain Routing (CIDR)

## 개요

RFC 4632는 IPv4 주소를 할당하면서 주소 공간을 보존하고 글로벌 라우팅 테이블의 성장을 제한하는 전략을 문서화한 문서입니다.

## 핵심 문제점과 해결책

### 기존 문제점

원래의 IPv4 주소 체계는 세 가지 고정된 클래스(A, B, C)를 사용했는데, 이로 인해 두 가지 심각한 문제가 발생했습니다:

1. **Class B 주소의 급격한 고갈**: Class B 주소가 빠르게 소진되고 있었음
2. **라우팅 테이블의 기하급수적 증가**: 전 세계 라우팅 테이블이 급격히 커지고 있었음

### CIDR의 해결책

CIDR은 이러한 경직된 클래스 시스템을 **유연하고 계층적인 프리픽스 기반 주소 체계**로 대체했습니다.

## 핵심 개념

### 프리픽스 표기법 (Prefix Notation)

주소는 `a.b.c.d/X` 형식으로 표기됩니다. 여기서 X는 네트워크 부분을 정의하는 비트 수를 나타냅니다.

**예시:**
- `172.16.0.0/16`: 기존 Class B 네트워크에 해당, 65,536개의 주소 보유
- `192.168.1.0/24`: 256개의 주소 보유
- `10.0.0.0/8`: 16,777,216개의 주소 보유

### 유연한 크기 조정 (Flexible Sizing)

기존의 세 가지 고정 클래스와 달리, CIDR은 **2의 거듭제곱 크기의 모든 주소 블록**을 허용합니다:

| 프리픽스 | 주소 개수 | 설명 |
|---------|----------|------|
| /8 | 16,777,216 | 대규모 조직용 |
| /16 | 65,536 | 중규모 조직용 |
| /24 | 256 | 소규모 네트워크용 |
| /32 | 1 | 단일 호스트 |

## 계층적 주소 분배

주소 공간은 여러 레지스트리 레벨을 통해 분배됩니다:

```
IANA (Internet Assigned Numbers Authority)
    ↓ /8 블록 할당
Regional Internet Registries (RIRs)
    ↓ 더 작은 블록 할당
Internet Service Providers (ISPs)
    ↓ 추가 블록 위임
최종 고객 (End Customers)
```

### 분배 구조

1. **IANA**: /8 블록을 지역 인터넷 레지스트리(RIR)에 할당
2. **지역 인터넷 레지스트리 (RIR)**: ISP에 더 작은 블록을 할당
   - ARIN (북미)
   - RIPE NCC (유럽, 중동, 중앙아시아)
   - APNIC (아시아 태평양)
   - LACNIC (라틴 아메리카, 카리브해)
   - AFRINIC (아프리카)
3. **ISP**: 고객에게 추가적인 블록 위임

이 분산 모델은 이전의 중앙 집중식 권한 체계를 대체했습니다.

## VLSM (Variable Length Subnet Mask)

### VLSM 개념

VLSM(가변 길이 서브넷 마스크)은 CIDR의 핵심 구성 요소로, **하나의 네트워크 내에서 서로 다른 크기의 서브넷 마스크를 사용**할 수 있게 합니다.

### VLSM의 필요성

기존의 고정 길이 서브넷 마스크(FLSM) 방식에서는 모든 서브넷이 동일한 크기여야 했습니다. 이로 인해 주소 낭비가 발생했습니다.

**예시 - 고정 길이 서브넷의 문제:**
```
회사 네트워크: 192.168.1.0/24 (256개 주소)
- 부서 A: 100명 필요 → /25 (128개) 할당 → 28개 낭비
- 부서 B: 50명 필요 → /25 (128개) 할당 → 78개 낭비
총 낭비: 106개 주소
```

### VLSM 적용 예시

VLSM을 사용하면 실제 필요에 맞게 서브넷 크기를 조정할 수 있습니다:

```
회사 네트워크: 192.168.1.0/24 (256개 주소)

세부 할당:
├── 192.168.1.0/25   (128개) - 본사 (100명)
├── 192.168.1.128/26 (64개)  - 지사 A (50명)
├── 192.168.1.192/27 (32개)  - 지사 B (25명)
├── 192.168.1.224/28 (16개)  - 서버실 (10대)
└── 192.168.1.240/30 (4개)   - 점대점 링크 (2대)
    나머지: 192.168.1.244/30 ~ 192.168.1.252/30 예비
```

### VLSM 설계 원칙

1. **큰 서브넷부터 할당**: 가장 많은 호스트가 필요한 네트워크부터 시작
2. **2의 거듭제곱 단위로 계획**: 필요 호스트 수 + 2 (네트워크/브로드캐스트 주소)
3. **연속된 주소 블록 사용**: 향후 확장과 집계를 고려
4. **여유 공간 확보**: 미래 성장을 위한 예비 공간 배치

### VLSM 계산 방법

| 필요 호스트 수 | 권장 프리픽스 | 실제 호스트 수 |
|---------------|--------------|---------------|
| 1-2 | /30 | 2 |
| 3-6 | /29 | 6 |
| 7-14 | /28 | 14 |
| 15-30 | /27 | 30 |
| 31-62 | /26 | 62 |
| 63-126 | /25 | 126 |
| 127-254 | /24 | 254 |

## 라우팅 규칙

### 1. 최장 프리픽스 매칭 (Longest-Prefix Matching)

포워딩은 **최장 매칭 라우팅**을 사용합니다. 더 구체적인 프리픽스가 우선순위를 가집니다.

**예시:**
- 목적지: `192.168.1.100`
- 라우팅 테이블:
  - `192.168.0.0/16` → Router A
  - `192.168.1.0/24` → Router B
- 결과: `/24`가 더 구체적이므로 **Router B**로 전송

### 최장 프리픽스 매칭 알고리즘

라우터가 패킷을 포워딩할 때 사용하는 알고리즘입니다:

**알고리즘 동작 순서:**

```
1. 목적지 IP 주소 추출
2. 라우팅 테이블의 모든 엔트리와 비교
3. 각 엔트리에 대해:
   - 프리픽스 마스크를 목적지 주소에 적용
   - 결과가 해당 네트워크 주소와 일치하는지 확인
4. 매칭되는 모든 엔트리 중 가장 긴 프리픽스 선택
5. 해당 엔트리의 next-hop으로 패킷 전달
```

**상세 예시:**

```
목적지 IP: 10.1.2.100

라우팅 테이블:
┌─────────────────┬─────────────┬──────────┐
│ 네트워크        │ 프리픽스    │ Next-Hop │
├─────────────────┼─────────────┼──────────┤
│ 0.0.0.0         │ /0          │ R1       │ ← 기본 경로
│ 10.0.0.0        │ /8          │ R2       │ ← 매칭
│ 10.1.0.0        │ /16         │ R3       │ ← 매칭
│ 10.1.2.0        │ /24         │ R4       │ ← 매칭 (최장!)
│ 10.2.0.0        │ /16         │ R5       │ ← 비매칭
└─────────────────┴─────────────┴──────────┘

결과: /24가 가장 긴 매칭이므로 R4로 전송
```

**구현 자료구조:**

| 자료구조 | 검색 시간 복잡도 | 특징 |
|---------|-----------------|------|
| 선형 검색 | O(n) | 단순하지만 느림 |
| 트라이(Trie) | O(W) | W = 주소 비트 수 (32) |
| Patricia Trie | O(W) | 메모리 효율적 |
| 다중 비트 트라이 | O(W/k) | k = 스트라이드 크기 |
| TCAM | O(1) | 하드웨어 기반, 고속 |

### 2. 집계 라우트 (Aggregate Routes)

집계 라우트는 해당 집계에는 매칭되지만 특정 서브넷에는 매칭되지 않는 트래픽을 **폐기(discard)**해야 합니다.

### 경로 집계(Aggregation) 조건

경로 집계(Route Aggregation)는 여러 개의 세부 경로를 하나의 요약 경로로 통합하는 기술입니다. 효과적인 집계를 위해서는 다음 조건들이 충족되어야 합니다:

**필수 조건:**

1. **연속된 주소 공간**: 집계할 네트워크들이 연속적인 주소 범위에 있어야 함
2. **공통 상위 비트**: 집계 대상 네트워크들이 동일한 상위 비트를 공유해야 함
3. **2의 거듭제곱 정렬**: 집계 블록의 시작 주소가 해당 블록 크기의 배수여야 함
4. **동일한 Next-Hop**: 모든 세부 경로가 같은 출구 방향을 가져야 함

**집계 가능 여부 판단:**

```
집계 가능한 예시:
192.168.0.0/24 + 192.168.1.0/24 → 192.168.0.0/23 ✓
(상위 23비트가 동일: 11000000.10101000.0000000x)

집계 불가능한 예시:
192.168.1.0/24 + 192.168.2.0/24 → 집계 불가 ✗
(192.168.1.0은 /23 블록의 중간에서 시작)
```

**집계 규칙 정리:**

| 조건 | 설명 | 예시 |
|-----|------|------|
| 주소 연속성 | 빈 공간 없이 연속되어야 함 | 10.0.0.0/24, 10.0.1.0/24 |
| 경계 정렬 | 블록 크기의 배수에서 시작 | /23은 짝수 네트워크에서 시작 |
| 동일 정책 | 같은 라우팅 정책 적용 | 모두 같은 AS로 전송 |
| 완전 포함 | 집계 범위 내 모든 네트워크 포함 | 부분 집계 시 블랙홀 위험 |

**집계 시 주의사항:**

1. **블랙홀 방지**: 집계 범위 내에 실제로 사용하지 않는 주소가 있으면, 해당 주소로의 트래픽은 폐기됨
2. **Null 라우트 설정**: 집계 경로 광고 시, 사용하지 않는 범위에 대해 null 라우트 설정 권장
3. **정책 일관성**: 집계된 모든 네트워크에 동일한 라우팅 정책이 적용되어야 함

## 영향과 결과

### 긍정적 효과

- **IPv4 고갈 지연**: CIDR 배포로 IPv4 주소 고갈 속도가 늦춰짐
- **라우팅 성장 안정화**: 1990년대를 통해 라우팅 테이블 성장이 안정화됨
- **효율적인 주소 사용**: 필요한 만큼만 주소를 할당할 수 있게 됨

### 한계점

2005년 이후로, 다음과 같은 이유로 기하급수적 성장이 재개되었습니다:
- **멀티호밍 (Multi-homing)**: 여러 ISP에 연결하기 위한 추가 경로 광고
- **트래픽 엔지니어링**: 더 구체적인 경로 광고가 전 세계적으로 필요해짐

## 실제 활용 예시

### 주소 할당 예시

회사에 1,000개의 호스트가 필요한 경우:

```
기존 클래스 방식:
- Class C (/24) = 256개 주소 → 부족
- Class B (/16) = 65,536개 주소 → 과다

CIDR 방식:
- /22 = 1,024개 주소 → 적절!
```

### 슈퍼넷팅 예시

여러 개의 /24 네트워크를 하나로 집약:

```
192.168.0.0/24
192.168.1.0/24
192.168.2.0/24
192.168.3.0/24
    ↓ 집약
192.168.0.0/22 (하나의 라우팅 엔트리)
```

## 요약

CIDR은 인터넷의 지속적인 성장을 가능하게 한 핵심 기술입니다. 클래스 기반 주소 체계의 비효율성을 해결하고, 유연한 주소 할당과 효율적인 라우팅을 가능하게 했습니다. 오늘날 모든 네트워크 엔지니어가 반드시 이해해야 하는 기본 개념입니다.
