# RFC 1918 - Address Allocation for Private Internets

> 1996년 2월 발행, BCP 5 (Best Current Practice)
> 사설 인터넷을 위한 주소 할당 표준

## 1. 개요

RFC 1918은 인터넷에 직접 연결되지 않는 사설 네트워크에서 사용할 수 있는 IP 주소 대역을 정의합니다. 이 주소들은 전역적으로 라우팅되지 않으며, 조직 내부 네트워크에서 자유롭게 사용할 수 있습니다.

## 2. 배경 및 동기

### 주요 동기

> "지구적으로 고유한 주소 공간 고갈 우려"와 라우팅 오버헤드 증가

1. **IPv4 주소 고갈**: 32비트 주소 공간의 한계
2. **라우팅 테이블 크기 증가**: 글로벌 라우팅 효율성 저하
3. **불필요한 전역 주소 사용**: 인터넷 접근이 불필요한 호스트에도 전역 주소 할당

### 문제점

- 모든 호스트에 전역 고유 주소를 할당하기 어려움
- 인터넷 성장에 따른 주소 공간 압박
- 라우팅 인프라에 대한 부담 증가

## 3. 사설 IP 주소 대역

IANA(Internet Assigned Numbers Authority)가 사설용으로 예약한 세 가지 주소 블록:

### 사설 주소 범위

| 블록 | 주소 범위 | CIDR 표기 | 클래스 | 주소 개수 |
|------|-----------|-----------|--------|-----------|
| 24비트 블록 | 10.0.0.0 ~ 10.255.255.255 | 10.0.0.0/8 | Class A | 16,777,216개 |
| 20비트 블록 | 172.16.0.0 ~ 172.31.255.255 | 172.16.0.0/12 | Class B | 1,048,576개 |
| 16비트 블록 | 192.168.0.0 ~ 192.168.255.255 | 192.168.0.0/16 | Class C | 65,536개 |

### 각 블록의 특성

**10.0.0.0/8 (Class A 대역)**
```
주소 범위: 10.0.0.0 - 10.255.255.255
네트워크 비트: 8
호스트 비트: 24
사용 가능 호스트: 16,777,214개 (브로드캐스트/네트워크 주소 제외)
용도: 대규모 조직, 데이터센터
```

**172.16.0.0/12 (Class B 대역)**
```
주소 범위: 172.16.0.0 - 172.31.255.255
네트워크 비트: 12
호스트 비트: 20
사용 가능 호스트: 1,048,574개
용도: 중규모 조직
```

**192.168.0.0/16 (Class C 대역)**
```
주소 범위: 192.168.0.0 - 192.168.255.255
네트워크 비트: 16
호스트 비트: 16
사용 가능 호스트: 65,534개
용도: 소규모 조직, 가정용 네트워크
```

## 4. 호스트 분류

RFC 1918은 호스트를 인터넷 연결 요구사항에 따라 세 가지 카테고리로 분류합니다.

### Category 1: 외부 접근 불필요

- 인터넷 계층에서 외부 호스트 접근이 필요하지 않은 호스트
- 완전히 격리된 내부 네트워크
- **사설 주소 사용 가능**

**예시:**
- 내부 데이터베이스 서버
- 프린터, IoT 장치
- 개발/테스트 환경

### Category 2: 제한된 외부 서비스 필요

- 외부 서비스에 접근해야 하지만 직접적인 IP 연결은 불필요
- 응용 계층 게이트웨이를 통해 외부 접근
- **사설 주소 + 게이트웨이(NAT/프록시) 사용**

**예시:**
- 웹 브라우징이 필요한 사용자 PC
- 이메일 클라이언트
- 소프트웨어 업데이트가 필요한 시스템

### Category 3: 네트워크 계층 외부 접근 필요

- 직접적인 IP 계층 접근이 필요한 호스트
- **전역 주소 필요**

**예시:**
- 공개 웹 서버
- 메일 서버
- VPN 게이트웨이

### 분류 다이어그램

```
┌─────────────────────────────────────────────────────────────┐
│                        인터넷                                │
└───────────────────────────┬─────────────────────────────────┘
                            │
                    ┌───────┴───────┐
                    │   라우터/방화벽  │
                    │  (NAT Gateway) │
                    └───────┬───────┘
                            │
        ┌───────────────────┼───────────────────┐
        │                   │                   │
┌───────┴───────┐   ┌───────┴───────┐   ┌───────┴───────┐
│  Category 3   │   │  Category 2   │   │  Category 1   │
│  (전역 주소)   │   │  (사설+NAT)    │   │  (사설 전용)   │
│               │   │               │   │               │
│  웹서버       │   │  사용자 PC    │   │  DB 서버      │
│  메일서버     │   │  프록시 접근   │   │  프린터       │
└───────────────┘   └───────────────┘   └───────────────┘
```

## 5. 장점

### 유연성 확보

> "기업들은 더 큰 사설 주소 공간으로부터 유연성을 얻는다"

1. **자유로운 주소 할당**: IANA 승인 없이 내부 주소 사용 가능
2. **충분한 주소 공간**: 대규모 조직도 충분한 주소 확보
3. **독립적인 네트워크 설계**: 외부 제약 없이 내부 구조 설계

### 전역 주소 공간 보존

- 인터넷 라우팅 테이블 크기 감소
- IPv4 주소 고갈 지연
- 효율적인 주소 사용

### 보안 강화

- 외부에서 직접 접근 불가
- NAT를 통한 추가 보안 계층
- 내부 네트워크 구조 은폐

## 6. 단점 및 주의사항

### 주소 재할당(Renumbering) 필요

사설 주소를 사용하다가 인터넷 연결이 필요해지면:
- 전역 주소로 변경 필요
- 네트워크 재설계 비용 발생
- 서비스 중단 가능성

### 주소 충돌 위험

```
기업 A (10.0.0.0/8 사용) + 기업 B (10.0.0.0/8 사용)
                    ↓
              합병 시 충돌 발생!
```

- 여러 조직 합병 시 동일 사설 대역 사용으로 충돌
- VPN 연결 시 주소 중복 문제
- 해결을 위한 주소 재할당 필요

### 주소 재사용 시 충돌 문제

사설 주소는 전 세계적으로 동일한 대역을 재사용하기 때문에, 네트워크 간 연결 시 심각한 충돌 문제가 발생할 수 있습니다.

**충돌 시나리오:**

```
┌─────────────────────────────────────────────────────────────────┐
│                     주소 충돌 시나리오                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  시나리오 1: M&A (인수합병)                                       │
│  ┌─────────────┐              ┌─────────────┐                  │
│  │   회사 A     │   합병 후    │   회사 B     │                  │
│  │ 10.1.0.0/16 │ ←─────────→ │ 10.1.0.0/16 │ ← 동일 대역!      │
│  └─────────────┘              └─────────────┘                  │
│                                                                 │
│  시나리오 2: VPN 연결                                            │
│  ┌─────────────┐   Site-to-Site  ┌─────────────┐              │
│  │  본사        │ ←─────VPN────→ │  지사        │              │
│  │ 192.168.1.0 │                 │ 192.168.1.0 │ ← 라우팅 불가! │
│  └─────────────┘                 └─────────────┘              │
│                                                                 │
│  시나리오 3: 클라우드 하이브리드                                  │
│  ┌─────────────┐   Direct Connect ┌─────────────┐             │
│  │  온프레미스  │ ←──────────────→ │  AWS VPC    │             │
│  │ 172.16.0.0  │                  │ 172.16.0.0  │ ← CIDR 충돌! │
│  └─────────────┘                  └─────────────┘             │
└─────────────────────────────────────────────────────────────────┘
```

**충돌로 인한 문제:**

1. **라우팅 불가**: 동일 대역 간 직접 통신 불가능
2. **서비스 장애**: 애플리케이션 연결 실패
3. **복잡한 해결 과정**: 대규모 주소 재할당(Renumbering) 필요
4. **비용 증가**: 네트워크 재설계 및 다운타임 비용

**해결 방안:**

| 방법 | 설명 | 장단점 |
|------|------|--------|
| **이중 NAT** | 양측 네트워크에서 NAT 적용 | 복잡하지만 즉시 적용 가능 |
| **주소 재할당** | 한쪽 네트워크 주소 변경 | 근본적 해결, 비용/시간 소요 |
| **NAT64/DNS64** | IPv6 전환 시 활용 | 장기적 해결책 |
| **오버레이 네트워크** | VXLAN, SD-WAN 활용 | 기존 주소 유지 가능 |

**이중 NAT (Double NAT) 구성 예시:**

```
회사 A (10.1.0.0/16)          회사 B (10.1.0.0/16)
        ↓                            ↓
    NAT 변환                      NAT 변환
  10.1.x.x → 172.20.x.x        10.1.x.x → 172.21.x.x
        ↓                            ↓
        └────────── VPN ─────────────┘
              (172.20.0.0 ↔ 172.21.0.0)
```

**충돌 예방 전략:**

1. **고유 서브넷 선택**: 일반적이지 않은 주소 대역 선택
   - 피해야 할 대역: 10.0.0.0/24, 192.168.0.0/24, 192.168.1.0/24
   - 권장 대역 예시: 10.47.0.0/16, 172.23.0.0/16

2. **IPAM 도입**: IP Address Management 도구로 중앙 관리

3. **문서화**: 사용 중인 주소 대역 명확히 기록

4. **미래 계획**: M&A, 파트너십 가능성 고려한 주소 설계

### NAT 관련 제약

- 일부 프로토콜 호환성 문제 (FTP, SIP 등)
- End-to-End 연결성 손실
- 복잡한 트러블슈팅

## 7. 운영 고려사항

### 네트워크 설계 원칙

1. **사설 네트워크 설계 우선**: 공개 주소 사용 전에 사설 주소 사용 검토
2. **미래 성장 고려**: 충분한 주소 공간 확보
3. **체계적인 서브넷 계획**: 관리 용이성 확보

### 패킷 필터링

**경계 라우터에서 필터링 필수:**

```
# 인바운드 필터 (외부 → 내부)
deny ip 10.0.0.0/8 any
deny ip 172.16.0.0/12 any
deny ip 192.168.0.0/16 any

# 아웃바운드 필터 (내부 → 외부)
deny ip any 10.0.0.0/8
deny ip any 172.16.0.0/12
deny ip any 192.168.0.0/16
```

- 사설 주소의 외부 유출 방지
- 스푸핑 공격 차단
- 라우팅 오류 방지

### DNS 분리 운영

**Split DNS 구성:**

```
┌─────────────────────────────────────────────────┐
│                 외부 DNS 서버                    │
│  example.com → 203.0.113.10 (공인 IP)          │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│                 내부 DNS 서버                    │
│  server.internal → 10.0.1.100 (사설 IP)        │
│  example.com → 10.0.1.50 (내부 서버)           │
└─────────────────────────────────────────────────┘
```

- 공개 DNS: 외부 사용자용 공인 주소만 공개
- 사설 DNS: 내부 사용자용 사설 주소 제공
- 사설 주소 정보 외부 노출 방지

### 캡슐화 사용

공개 네트워크를 통해 사설 네트워크 연결 시:
- VPN 터널링 사용
- IPsec, GRE, VXLAN 등 활용
- 사설 주소 트래픽을 공인 주소로 캡슐화

### ISP/백본 책임 및 필터링 요구사항

> "사설 주소 공간을 사용하는 기업의 인터넷 연결을 제공하는 ISP는 사설 네트워크의 라우팅 정보가 외부로 전파되지 않도록 해야 한다"

**ISP의 필터링 책임:**

1. **경계 라우터 필터링**: ISP는 고객 경계에서 사설 주소 트래픽 필터링 필수
2. **BGP 공지 차단**: 사설 주소 대역의 라우팅 정보가 인터넷에 광고되지 않도록 차단
3. **역방향 경로 필터링(RPF)**: 소스 주소 검증을 통한 스푸핑 방지

**백본 네트워크 필터링:**

```
# BGP 프리픽스 필터 예시
ip prefix-list PRIVATE-BLOCK deny 10.0.0.0/8 le 32
ip prefix-list PRIVATE-BLOCK deny 172.16.0.0/12 le 32
ip prefix-list PRIVATE-BLOCK deny 192.168.0.0/16 le 32
ip prefix-list PRIVATE-BLOCK permit 0.0.0.0/0 le 32
```

**중요 포인트:**
- 개별 조직뿐만 아니라 ISP도 필터링 책임이 있음
- 사설 주소가 인터넷에 유출되면 라우팅 문제 및 보안 위협 발생
- 상위 백본에서도 다중 계층 필터링 필요

### 사설 주소 라우팅 정책

사설 주소의 올바른 라우팅 정책 수립이 필수적입니다.

**라우팅 정책 원칙:**

1. **내부 라우팅만 허용**: 사설 주소는 조직 내부에서만 라우팅
2. **기본 경로 사용**: 외부 통신은 NAT 게이트웨이로 향하는 기본 경로 활용
3. **라우팅 프로토콜 분리**: 내부(OSPF, EIGRP)와 외부(BGP) 프로토콜 분리 운영

**정책 라우팅 예시:**

```
┌─────────────────────────────────────────────────────────┐
│                    라우팅 정책 구조                       │
├─────────────────────────────────────────────────────────┤
│  내부 라우팅 도메인                                       │
│  ├── 10.0.0.0/8 → 내부 OSPF/EIGRP로 전파               │
│  ├── 172.16.0.0/12 → 내부 라우팅 테이블에만 존재         │
│  └── 192.168.0.0/16 → 사설 영역 내 라우팅               │
│                                                         │
│  외부 라우팅 도메인                                       │
│  ├── 0.0.0.0/0 → NAT 게이트웨이로 기본 경로              │
│  └── 사설 주소 프리픽스 재배포 금지                       │
└─────────────────────────────────────────────────────────┘
```

**재배포 정책:**
- 사설 주소 프리픽스는 외부 라우팅 프로토콜(BGP)로 재배포 금지
- ACL 또는 프리픽스 리스트로 재배포 필터링 필수
- Route-map을 활용한 세밀한 제어 권장

### 역방향 DNS (Reverse DNS) 고려사항

사설 IP 주소에 대한 역방향 DNS 조회는 특별한 주의가 필요합니다.

**문제점:**

```
역방향 DNS 쿼리 예시:
100.1.168.192.in-addr.arpa → 인터넷 DNS 서버로 유출 가능!
```

- 사설 주소의 역방향 DNS 쿼리가 외부로 유출되면 정보 노출 위험
- 루트 DNS 서버에 불필요한 트래픽 발생
- 잘못된 응답으로 인한 애플리케이션 오류

**해결 방안:**

1. **내부 DNS 서버에서 역방향 존 운영**
```
# named.conf 설정 예시
zone "168.192.in-addr.arpa" {
    type master;
    file "db.192.168";
};

zone "16.172.in-addr.arpa" {
    type master;
    file "db.172.16";
};

zone "10.in-addr.arpa" {
    type master;
    file "db.10";
};
```

2. **빈 존(Empty Zone) 설정**: 외부 쿼리 차단
3. **DNS 정책 서버**: 사설 주소 쿼리를 내부로 리다이렉트

**RFC 권장사항 (RFC 6303 참조):**
- AS112 프로젝트: 사설 주소 역방향 DNS 쿼리를 로컬에서 처리
- 내부 DNS 리졸버에서 사설 존 우선 참조 설정

## 8. NAT (Network Address Translation)

RFC 1918 사설 주소와 함께 가장 많이 사용되는 기술입니다.

### NAT는 보안 솔루션이 아님

> **경고**: NAT는 보안을 위해 설계된 것이 아니며, 방화벽이나 보안 솔루션을 대체할 수 없습니다.

**흔한 오해:**
- "NAT가 내부 네트워크를 보호해준다" - **잘못된 인식**
- "사설 IP를 사용하면 해킹당하지 않는다" - **잘못된 인식**

**NAT의 보안적 한계:**

1. **주소 은폐 != 보안**
   - NAT는 주소 변환 기술일 뿐, 보안 기능이 아님
   - 내부 구조가 숨겨지지만 이것이 보안을 보장하지 않음

2. **아웃바운드 연결 취약점**
   ```
   악성코드 → 외부 C&C 서버 연결 → NAT 통과 가능!
   NAT는 내부에서 시작된 연결을 차단하지 않음
   ```

3. **상태 기반 공격 가능**
   - NAT 테이블 오버플로우 공격
   - 세션 하이재킹
   - 포트 예측 공격

4. **프로토콜 취약점 그대로 노출**
   - 애플리케이션 레벨 취약점은 NAT로 보호되지 않음
   - 피싱, 악성코드 다운로드 등 사용자 기반 공격에 무방비

**올바른 보안 구성:**

```
┌─────────────────────────────────────────────────────────┐
│                    보안 계층 구조                         │
├─────────────────────────────────────────────────────────┤
│  1. 경계 방화벽 (Stateful Inspection)                   │
│  2. IDS/IPS (침입 탐지/방지 시스템)                      │
│  3. 웹 애플리케이션 방화벽 (WAF)                         │
│  4. NAT (주소 변환 - 보안 목적 아님)                     │
│  5. 내부 세그멘테이션 및 마이크로세그멘테이션             │
│  6. 엔드포인트 보안                                     │
└─────────────────────────────────────────────────────────┘
```

**권장사항:**
- NAT와 함께 반드시 상태 기반 방화벽 사용
- 심층 방어(Defense in Depth) 전략 적용
- 정기적인 보안 감사 및 침투 테스트 수행

### NAT 유형

| 유형 | 설명 |
|------|------|
| **Static NAT** | 1:1 고정 매핑 |
| **Dynamic NAT** | 동적 주소 풀에서 할당 |
| **PAT/NAPT** | 포트 기반 다대일 변환 (가장 일반적) |

### PAT (Port Address Translation) 동작

```
내부 호스트               NAT Gateway              외부 서버
192.168.1.10:5000  →  203.0.113.1:40000  →  8.8.8.8:53
192.168.1.11:5001  →  203.0.113.1:40001  →  8.8.8.8:53
192.168.1.12:5002  →  203.0.113.1:40002  →  1.1.1.1:80
```

## 9. 현대적 활용

### 일반적인 사용 사례

1. **가정용 네트워크**: 192.168.0.0/24 또는 192.168.1.0/24
2. **기업 네트워크**: 10.0.0.0/8 또는 172.16.0.0/12
3. **클라우드 VPC**: 모든 사설 대역 활용 가능
4. **컨테이너 네트워크**: Docker 기본값 172.17.0.0/16

### 클라우드 환경

```
AWS VPC: 10.0.0.0/16, 172.16.0.0/16, 192.168.0.0/16 지원
Azure VNet: RFC 1918 주소 공간 사용
GCP VPC: 사설 주소 범위 내 서브넷 생성
```

### 권장 주소 선택

| 규모 | 권장 대역 | 이유 |
|------|-----------|------|
| 가정/소규모 | 192.168.0.0/24 | 간단하고 익숙함 |
| 중규모 기업 | 172.16.0.0/16 | VPN 충돌 회피 |
| 대규모 기업 | 10.0.0.0/8 | 충분한 확장성 |

## 10. 요약

RFC 1918은 사설 네트워크를 위한 IP 주소 할당 표준을 정의합니다:

- **세 가지 사설 대역**: 10/8, 172.16/12, 192.168/16
- **목적**: IPv4 주소 보존, 라우팅 효율성, 네트워크 보안
- **제약사항**: 인터넷 직접 라우팅 불가, NAT 필요
- **현재 상태**: NAT와 함께 인터넷의 핵심 인프라로 자리잡음

사설 IP 주소는 인터넷 주소 고갈 문제를 완화하고, 조직 내부 네트워크의 유연한 설계를 가능하게 하는 핵심 요소입니다.
