## 2. INTRODUCTION

### 2.1 배경과 필요성

#### 기존 HOSTS.TXT 시스템의 문제점

| 문제 | 설명 |
|------|------|
| ** 확장성 문제** | 중앙집중식 HOSTS.TXT 파일 방식의 한계 |
| ** 대역폭 소비** | 새 버전 배포 시 네트워크 대역폭 소비량이 ** 호스트 수의 제곱에 비례** |
| ** 관리 자율성 부재** | 로컬 조직이 자체 인프라를 관리하면서도 글로벌 이름 서비스 업데이트 불가 |
| **NIC 병목** | 모든 변경사항에 NIC(Network Information Center) 개입 필요 |
| ** 애플리케이션 요구사항** | 단순 호스트-주소 변환 이상의 범용 이름 서비스 필요 |

### 2.2 역사 (History)

| **RFC 1034, 1035** | 개선된 명세로 재작성 |

### 2.3 DNS 설계 목표

#### 5가지 기본 원칙

| 원칙 | 설명 |
|------|------|
| ** 일관된 네임스페이스** | 이름은 네트워크 토폴로지, 주소, 라우팅 정보와 독립적인 ** 중립 식별자** |
| ** 분산 유지관리** | 대규모 데이터베이스는 로컬 캐싱과 함께 ** 분산 파티셔닝** 필요, 중앙집중식 일관성 병목 회피 |
| ** 소스 권한** | 데이터 원작자가 ** 획득 비용, 업데이트 속도, 캐시 정확도** 간 트레이드오프 제어 |
| ** 범용 유틸리티** | 호스트 주소, 메일박스 데이터, 미래 정보 유형 등 다양한 애플리케이션 지원, ** 타입 태그** 로 선택적 쿼리 |
| ** 프로토콜 독립성** | ** 클래스 태그** 를 통해 다양한 프로토콜 패밀리의 병렬 네임스페이스 및 데이터 형식 지원 |

#### 추가 목표

| 목표 | 설명 |
|------|------|
| ** 통신 독립성** | UDP 데이터그램 또는 TCP 연결 모두에서 독립적 동작 |
| ** 확장성** | 개인용 컴퓨터부터 대형 호스트까지 적절히 참여 가능 |

### 2.4 사용에 대한 가정 (Assumptions About Usage)

| 가정 | 설명 |
|------|------|
| ** 데이터 성장** | 초기에는 호스트 수에 비례, 이후 사용자 수에 비례 (메일박스, 메타데이터 증가) |
| ** 변경 패턴** | 대부분의 데이터는 드물게 변경 (주소, 메일박스 바인딩), 일부는 빠르게 변경 가능 (초/분 단위) |
| ** 관리 경계** | 책임 분배는 조직 구조와 일치, 각 조직은 ** 중복 네임 서버** 운영 |
| ** 신뢰 인프라** | 클라이언트는 외부 참조 수락 전 ** 선호 네임 서버** 식별 필요 |
| ** 가용성 우선** | 시스템은 즉각 업데이트보다 ** 접근성 우선**, 캐싱을 통해 점진적 전파 |
| ** 쿼리 해석 모드** | ** 재귀(recursive)** 와 ** 반복(iterative)** 모드 모두 지원, 데이터그램 운영에는 반복 방식이 더 적합 |
| ** 마스터 파일 아키텍처** | 데이터는 로컬 관리자가 유지하는 분산 마스터 파일에서 시작, 표준 형식으로 호스트 간 교환 가능 |

### 2.5 이 문서의 범위
- DNS의 ** 개념과 기능** 에 초점
- 구현 세부사항은 RFC 1035에서 다룸

## 3. DOMAIN NAME SPACE AND RESOURCE RECORDS

### 3.1 Name Space Specifications (이름 공간 명세)

#### 계층적 구조

```
                               . (root)
                               |
         ___________________________________________________
         |          |          |          |          |
        MIL        EDU        COM       ARPA       ORG
         |          |                     |
    _____|_____   __|__________      IN-ADDR
    |    |    |   |    |    |  |
   BRL NOSC DARPA UCI MIT YALE ISI
                   |
              _____|_____
              |    |    |
             LCS   XX ACHILLES
```

- 도메인 네임스페이스는 ** 트리 구조**
- 각 노드는 하나의 ** 도메인** 을 나타냄
- 도메인 이름 = 노드에서 루트까지의 경로에 있는 ** 라벨들의 목록**

#### 라벨 규칙

| 속성 | 규칙 |
|------|------|
| 길이 | **0~63 옥텟** |
| 고유성 | 형제 노드는 반드시 다른 라벨 사용 (비형제 노드는 동일 라벨 허용) |
| 루트 | 단일 **null 라벨** (길이 0) 예약 |
| 대소문자 | ** 보존** 되나 비교 시 ** 구분 없음** (case-insensitive) |
| 문자 인코딩 | ASCII 가정, 최상위 비트 0 |

#### 이름 표현

| 표현 방식 | 설명 | 예시 |
|-----------|------|------|
| 절대 이름 (FQDN) | 끝에 점(.) 포함, 루트까지 완전한 경로 | `poneria.ISI.EDU.` |
| 상대 이름 | 점 없음, 로컬 컨텍스트로 완성 | `poneria` (ISI.EDU 컨텍스트에서) |
| 내부 표현 | 길이-옥텟 접두사 + 0바이트 종료 | - |
| 텍스트 표현 | 라벨을 점(.)으로 구분 | - |

#### 길이 제한

| 항목 | 최대 길이 |
|------|-----------|
| 라벨 | **63 옥텟** |
| FQDN 전체 | **255 옥텟** (라벨 + 길이 바이트 포함) |

#### 도메인 관계

- 도메인 X가 서브도메인 Y를 ** 포함**: Y의 이름이 X의 이름으로 끝남
- 예: `A.B.C.D`는 `B.C.D`, `C.D`, `D`, 루트(`''`)의 서브도메인

### 3.2 Administrative Guidelines on Use (관리 지침)

#### 관리 권한 분배

- DNS 기술 명세는 트리 구조와 라벨 선택 규칙에 대해 ** 중립적**
- 다양한 애플리케이션 요구사항 허용

#### 네이밍 의미론

| 도메인 영역 | 조직 체계 |
|-------------|-----------|
| 일반 도메인 | 조직적/지역적 계층 |
| IN-ADDR.ARPA | 네트워크/호스트 구조 (역방향 조회용) |
| NetBIOS 도메인 | 플랫 구조 |

#### 성장 대비
- 하위 도메인이 결국 다중 Zone이 되면 ** 최상위 레벨에서 분기**
- 이름 변경 없이 분해 가능

#### 소프트웨어 호환성
- 특수 문자나 선행 숫자가 있는 라벨은 ** 레거시 소프트웨어** 에서 문제 발생 가능
- ARPANET 규칙을 기대하는 소프트웨어와 호환성 문제

#### 관련 RFC

| RFC | 내용 |
|-----|------|
| RFC-920 | 최상위 레벨 정책 |
| RFC-1032 | 도메인 설정 가이드라인 |
| RFC-1031 | MILNET 마이그레이션 |

### 3.3 Technical Guidelines on Use (기술 지침)

#### 새로운 객체 유형 지원 시 필요 사항

1. ** 이름-도메인 매핑 규칙**: 객체 정보를 도메인 이름 쿼리로 매핑하는 방법
2. ** 리소스 레코드 타입과 형식**: RR 타입과 RDATA 구조

- 범위: 단순(호스트)부터 복잡(메일박스, 다중 매핑 레벨)까지

#### 호스트 매핑

- 기존 호스트명 구문 규칙 준수
- **A 타입 RR** 로 주소 저장
- **IN-ADDR.ARPA** 도메인으로 신뢰할 수 있는 역방향 매핑 제공

#### 메일박스 매핑

- 이메일 주소 형식: `<local-part>@<mail-domain>`
- 도메인 이름 변환:
  - local-part → ** 단일 라벨**
  - mail-domain → 표준 도메인 형식
  - 결합하여 통합 도메인 이름 생성
- 예: `HOSTMASTER@SRI-NIC.ARPA` → `HOSTMASTER.SRI-NIC.ARPA`

#### 설계 결정
- 이전 사용법과의 상향 호환성
- 다른 객체 정의 간 상호작용
- 새 기능 추가 욕구 간의 ** 다양한 절충** 포함

### 3.4 Example Name Space (예시 도메인 트리)

```
                                    . (root)
                                    |
         _________________________________________________________
         |                          |                            |
        MIL                        EDU                         ARPA
         |                          |                            |
    _____|_____              _______|_______              _______|_______
    |    |    |              |      |      |              |      |      |
   BRL  NOSC DARPA         UCI    MIT   YALE          SRI-NIC IN-ADDR ACC
                            |
                       _____|_____
                       |    |    |
                      LCS   XX ACHILLES
```

- ** 리프 노드** (UCI, MIT, YALE 등): 그 자체로 도메인
- ** 비리프 노드** (EDU, ISI): 도메인이면서 서브도메인 포함

### 3.5 Preferred Name Syntax (선호 이름 문법)

#### BNF 문법 (RFC-822, HOSTS.TXT 호환)

```
<domain>      ::= <subdomain> | " "
<subdomain>   ::= <label> | <subdomain> "." <label>
<label>       ::= <letter> [ [ <ldh-str> ] <let-dig> ]
<ldh-str>     ::= <let-dig-hyp> | <let-dig-hyp> <ldh-str>
<let-dig-hyp> ::= <let-dig> | "-"
<let-dig>     ::= <letter> | <digit>
<letter>      ::= A-Z, a-z (52자)
<digit>       ::= 0-9
```

#### 라벨 문자 제한 규칙 (ARPANET 규칙)

| 규칙 | 설명 |
|------|------|
| 시작 문자 | 반드시 ** 문자** 로 시작 |
| 끝 문자 | ** 문자** 또는 ** 숫자** 로 끝남 |
| 내부 문자 | 문자, 숫자, 하이픈(-) 허용 |
| 최대 길이 | **63자** |

#### 유효한 호스트명 예시

- `A.ISI.EDU`
- `XX.LCS.MIT.EDU`
- `SRI-NIC.ARPA`

#### 대소문자 처리
- 유연하게 처리
- 양쪽 케이스 허용에도 불구하고 ** 의미 부여 없음**

### 3.6 Resource Records (리소스 레코드)

#### 핵심 개념

- 각 노드는 관련 정보를 담은 ** 리소스 레코드(RR) 집합** 보유
- RR 순서는 ** 의미 없음**

#### RR 구성 요소

| 구성 요소 | 비트 | 설명 |
|-----------|------|------|
| **Owner** | - | RR 위치를 식별하는 도메인 이름 |
| **Type** | 16 | 리소스 유형 지정 코드 |
| **Class** | 16 | 프로토콜 패밀리/인스턴스 |
| **TTL** | 32 | 캐시 수명 제어 (초 단위) |
| **RDATA** | - | 타입과 클래스에 따른 리소스 데이터 |

#### 주요 RR 타입

| Type | 설명 | RDATA |
|------|------|-------|
| **A** | 호스트 주소 | IN: 32비트 IP, CH: 도메인 이름 + 16비트 Chaos 주소 |
| **NS** | 권위 있는 네임 서버 | 호스트명 |
| **CNAME** | 별칭(alias) | RDATA에 정규 이름 지정 |
| **SOA** | Zone 권한 정의 | 여러 필드 (아래 상세) |
| **MB** | 메일박스 도메인 이름 | - |
| **MG** | 메일 그룹 멤버 | - |
| **MR** | 메일 리네임 도메인 이름 | - |
| **NULL** | Null RR | - |
| **WKS** | 잘 알려진 서비스 설명 | - |
| **PTR** | 도메인 이름 포인터 (역방향 매핑) | 도메인 이름 |
| **HINFO** | 호스트 정보 | CPU, OS 식별 |
| **MINFO** | 메일박스/메일 리스트 정보 | - |
| **MX** | 메일 교환기 | 16비트 우선순위 + 호스트명 |
| **TXT** | 텍스트 문자열 | - |

#### 주요 클래스

| Class | 설명 |
|-------|------|
| **IN** | 인터넷 시스템 |
| **CH** | Chaos 시스템 |

#### TTL 의미론

| 용도 | 설명 |
|------|------|
| ** 리졸버 캐싱용** | TTL은 캐시 수명의 ** 상한** |
| ** 권위 있는 Zone 데이터** | Zone 새로고침 정책에 의해 관리, TTL 적용 안 됨 |
| ** 권장 값** | 일반적인 호스트의 경우 ** 일(days) 단위** |
| ** 변경 예상 시** | TTL 감소로 불일치 기간 최소화 |
| **TTL 0** | 캐싱 ** 완전 금지** |

#### RR 텍스트 표현 형식

```
<owner> [<TTL>] [<class>] <type> <RDATA>
```

| 규칙 | 설명 |
|------|------|
| 첫 줄 | owner 이름으로 시작 |
| 빈 줄 접두사 | 이전 RR의 owner 가정 |
| 괄호 | 여러 줄 RR 허용 |
| 기본 클래스 | IN |
| 순서 | TTL (선택), class (선택, 기본 IN), type, RDATA |

#### 예시

```
ISI.EDU.        MX      10 VENERA.ISI.EDU.
                MX      10 VAXA.ISI.EDU.
VENERA.ISI.EDU. A       128.9.0.32
                A       10.1.0.52
VAXA.ISI.EDU.   A       10.2.0.27
                A       128.9.0.33
```

- 3개 도메인 이름에 6개 RR
- MX RDATA: 16비트 우선순위 값 + 호스트명

#### 다중 클래스 예시

```
XX.LCS.MIT.EDU. IN      A       10.0.0.44
                CH      A       MIT.EDU. 2420
```

### 3.6.1 Aliases and Canonical Names (별칭과 정규 이름)

#### 문제

- 리소스가 여러 동등한 이름을 가질 때 ** 주 식별자 지정** 필요

#### CNAME 해결책

- Canonical Name RR이 별칭 소유권 식별 및 해당 정규 이름 지정
- ** 제약**: CNAME RR이 노드에 있으면 ** 다른 데이터 없어야 함**
  - 정규 이름과 별칭의 데이터가 다를 수 없음 보장

#### 처리 규칙

| 상황 | 처리 |
|------|------|
| CNAME 발견 (QTYPE ≠ CNAME) | 지정된 정규 이름에서 쿼리 재시작, CNAME RR 응답에 포함 |
| QTYPE = CNAME | CNAME 반환, 재시작 없음 |

#### 예시

```
USC-ISIC.ARPA   IN      CNAME   C.ISI.EDU
C.ISI.EDU       IN      A       10.0.0.52
```

- A 타입 쿼리 for `USC-ISIC.ARPA` → CNAME과 A RR ** 모두 반환**
- CNAME 타입 또는 와일드카드 쿼리 → **CNAME만 반환**

#### 모범 사례

| 지침 | 설명 |
|------|------|
| 참조 대상 | 다른 이름을 참조하는 RR은 별칭이 아닌 ** 정규 이름** 을 가리켜야 함 (간접 참조 회피) |
| PTR RR | 역방향 매핑 PTR RR은 ** 정규 이름** 을 대상으로 함 |
| 체인 추적 | CNAME 체인 추적해야 함 |
| 루프 처리 | 루프는 ** 오류로 신호** |
| 견고성 원칙 | 소프트웨어는 CNAME 체인이나 루프에서 실패하면 안 됨 |

#### 역방향 매핑 예시

```
52.0.0.10.IN-ADDR.ARPA  IN      PTR     C.ISI.EDU
```

- 별칭 USC-ISIC.ARPA가 아닌 정규 이름을 가리킴

### 3.7 Queries (쿼리)

#### 정의

- 네임 서버에 응답, 답변, 참조, 오류 조건을 요청하는 ** 메시지**

#### 전송

- **UDP 데이터그램** 또는 **TCP 연결** (인터넷 환경)

#### 처리 모델

1. 사용자 프로그램 → 리졸버에 요청
2. 리졸버가 쿼리 생성 및 응답/참조/오류 처리

#### 메시지 형식

| 섹션 | 내용 |
|------|------|
| ** 헤더** | 고정 헤더 (4비트 opcode 필드로 쿼리 유형 구분) |
| **Question** | 쿼리 이름(QNAME), 쿼리 타입(QTYPE), 쿼리 클래스(QCLASS) |
| **Answer** | 쿼리에 직접 응답하는 RR |
| **Authority** | 권위 있는 서버를 설명하는 RR, 권위 있는 응답용 SOA 포함 가능 |
| **Additional** | 다른 섹션을 지원하는 유용한 RR |

#### Opcode 값 (16개 가능)

| Opcode | 상태 |
|--------|------|
| Standard query | ** 공식 프로토콜** |
| Inverse query | 선택적 |
| Status query | 선택적 |
| Completion | ** 폐지됨** |
| 나머지 값 | 미할당 |

### 3.7.1 Standard Queries (표준 쿼리)

#### 구성 요소

| 요소 | 설명 |
|------|------|
| **QNAME** | 도메인 이름 |
| **QTYPE** | 요청 정보 유형 |
| **QCLASS** | 요청 클래스 |

- DNS 쿼리의 ** 대다수** 를 차지하여 특별히 지정하지 않으면 "쿼리"는 표준 쿼리를 의미

#### QTYPE 필드 값

| 값 | 설명 |
|-----|------|
| `<any type>` | 특정 타입 매칭 (A, PTR 등) |
| `AXFR` | Zone 전송 쿼리 |
| `MAILB` | 메일박스 관련 RR (MB, MG) |
| `*` | ** 모든 RR 타입** |

#### QCLASS 필드 값

| 값 | 설명 |
|-----|------|
| `<any class>` | 특정 클래스 매칭 (IN, CH) |
| `*` | ** 모든 클래스** |

#### 응답 전략

- 일치하는 RR 반환
- 유용한 지원 레코드 추가 (권위 있는 서버로 향하는 NS RR, 해당 서버 바인딩 A RR)

#### 예시: MX 쿼리

** 쿼리**: `Mockapetris@ISI.EDU`에 대한 메일 교환기 요청

```
QNAME = ISI.EDU
QTYPE = MX
QCLASS = IN
```

**Answer 섹션**:
```
ISI.EDU.        MX      10 VENERA.ISI.EDU.
                MX      10 VAXA.ISI.EDU.
```

**Additional 섹션** (후속 조회 예상):
```
VAXA.ISI.EDU.   A       10.2.0.27
                A       128.9.0.33
VENERA.ISI.EDU. A       10.1.0.52
                A       128.9.0.32
```

#### 와일드카드 쿼리

- `QCLASS=*` 쿼리는 특별한 권한 해석
- 특정 네임 서버가 모든 클래스를 알 수 없으므로 ** 응답이 절대 권위 있을 수 없음**

### 3.7.2 Inverse Queries (역방향 쿼리) - 선택적

#### 목적

- 리소스를 도메인 이름으로 ** 역매핑**
- 예: SOA RR → 도메인 이름

#### 구현 상태

- ** 선택적 지원**
- 모든 네임 서버는 메시지를 이해하고 **"not-implemented" 응답 반환 필수**

#### 제한 사항

| 제한 | 설명 |
|------|------|
| 완전성/고유성 보장 불가 | 시스템이 도메인 이름으로 조직됨 |
| TTL 값 보장 불가 | 적절한 TTL 값 보장 불가 |
| 다중 값 집합 멤버십 표시 불가 | 여러 주소를 가진 호스트의 RR 멤버십 |
| 캐싱 금지 | 역방향 쿼리의 RR은 ** 절대 캐싱하면 안 됨** |
| 주소→호스트명 매핑 불가 | **IN-ADDR.ARPA 사용해야 함** |

#### 용도

- 주로 ** 디버깅** 과 ** 데이터베이스 유지보수** 활동에 유용

#### 상세 명세
- RFC-1035에서 다룸

### 3.8 Status Queries (상태 쿼리) - 실험적

- 미래 정의를 위해 섹션 ** 예약**

### 3.9 Completion Queries (완성 쿼리) - 폐지됨

- RFC 882-883의 선택적 완성 서비스 ** 삭제됨**
- 미래에 재설계된 서비스 가능 또는 opcode 재사용 가능

## 4. NAME SERVERS

### 4.1 Introduction (소개)

#### 네임 서버의 역할

- Zone으로 분할된 ** 분산 도메인 데이터베이스 저장소**
- ** 핵심 임무**: Zone의 데이터를 사용하여 쿼리에 응답

#### 설계 장점

| 장점 | 설명 |
|------|------|
| 단순한 쿼리 응답 | ** 로컬 데이터만** 사용 |
| 응답 유형 | 질문에 직접 응답 ** 또는** 더 가까운 네임 서버로 참조 |
| 다단계 없음 | 쿼리가 답변 또는 참조 생성, ** 다단계 요청 없음** |

#### 중복 요구사항

> "관리적 명령에 의해, 모든 Zone은 ** 최소 2개의 서버** 에서 사용 가능해야 하며, 많은 Zone은 그 이상의 중복성을 가짐"

#### 지식 범위

| 항목 | 설명 |
|------|------|
| 권위 있는 정보 | 하나 이상의 Zone 지원, 도메인 트리의 작은 섹션만 커버 |
| 비권위적 데이터 | 다른 트리 섹션에 대한 데이터 캐싱 가능 |
| 응답 소스 표시 | 권위 있는지 비권위적인지 ** 명시적으로 표시** |

### 4.2 How the Database is Divided Into Zones (Zone 분할 방법)

#### 이중 파티셔닝 전략

##### 1. 클래스 파티션

| 특성 | 설명 |
|------|------|
| 각 클래스 | ** 별도로 조직, 위임, 유지** 되는 데이터베이스 |
| 클래스별 병렬 네임스페이스 트리 | 다른 첨부 데이터 |
| 새 클래스 생성 이유 | 기존 타입의 새 데이터 형식 필요, 기존 이름 공간의 별도 관리 버전 필요 |

##### 2. 네임스페이스 "컷(Cuts)"

- 인접 노드 사이의 컷이 ** 별도 Zone** 생성
- 모든 컷 후, 연결된 이름 공간 각 그룹이 별도 Zone

#### Zone 특성

| 특성 | 설명 |
|------|------|
| 권위 있는 노드 | 각 Zone은 ** 최소 1개** 의 권위 있는 노드/이름 보유 |
| 연결 유지 | Zone 내 모든 노드는 ** 연결됨** |
| 최상위 노드 | Zone의 최상위 노드는 다른 Zone 노드보다 ** 루트에 가까움** |
| Zone 이름 | 일반적으로 Zone을 식별 |
| 극단적 경우 | 각 이름이 별도 Zone, 또는 단일 Zone이 모든 노드 포함 |
| 실질적 파티션 | ** 조직이 통제권을 가지는 지점** 에서 발생 |

#### 조직적 유연성
- 하위 구조가 있는 조직은 관리 편의를 위해 ** 내부 파티션** 생성 가능
- 중첩 위임으로 관리

### 4.2.1 Technical Considerations (기술적 고려사항)

#### Zone 데이터 구성 요소

| 구성 요소 | 설명 |
|-----------|------|
| ** 권위 있는 데이터** | Zone 최상위 노드부터 리프 노드 또는 하단 컷 위 노드까지의 ** 모든 RR** |
| ** 최상위 노드 설명** | Zone 관리에 특히 중요한 RR - 모든 서버 나열 **NS RR**, Zone 파라미터 설명 ** 단일 SOA RR** |
| ** 서브존 위임** | 서브존 주위의 하단 컷 설명 NS RR (** 권위 있는 데이터 아님**, 서브존 최상위 노드 RR과 정확히 일치) |
| **Glue 데이터** | 서브존 접근 가능하게 하는 서버 ** 주소 RR** (서버 이름이 서브존 내에 있을 때 특히 필요) |

#### Zone 전송

- RR 전송을 통해 네임 서버 간 Zone 전송 가능
- ** 일련의 메시지** 또는 **FTP 마스터 파일**

#### NS RR 위치

| 위치 | 유형 |
|------|------|
| Zone 최상위 노드 | ** 권위 있음** |
| 하단 컷 | ** 비권위적** |
| 중간 | ** 절대 없음** |

#### Glue 데이터 목적

- ** 부트스트랩 문제 해결**: NS RR이 자식 Zone 내 서버를 참조할 때
- Glue 주소 없이는 서버가 서버 주소를 알기 위해 해석 필요
- Glue RR은 ** 권위 있는 데이터의 일부가 아님**

### 4.2.2 Administrative Considerations (관리적 고려사항)

#### 위임 설정 프로세스

1. 적절한 ** 부모 Zone** 식별
2. 부모 소유자의 ** 위임 동의** 획득
3. 새 서브존 이름 선택 (RFC-1032 최상위 및 로컬 가이드라인)
4. ** 중복 네임 서버 지원** 시연
5. 부모 Zone에 ** 위임 NS RR과 Glue RR** 추가
6. 양쪽의 Zone 컷 NS와 Glue 레코드 간 ** 일관성 유지**

#### 서버 위치 유연성

- 네임 서버가 Zone의 도메인 내에 ** 있을 필요 없음**
- Zone의 서버가 ** 널리 분산** 되면 인터넷 전체에서 더 접근 가능
- 예: UK 도메인 네임 서버가 US에 위치하여 대서양 횡단 대역폭 보존

### 4.3 Name Server Internals (네임 서버 내부)

### 4.3.1 Queries and Responses (쿼리와 응답)

#### 주요 기능

- 표준 메시지 형식(RFC-1035)을 통해 ** 표준 쿼리 응답**

#### 쿼리 구성 요소

| 요소 | 설명 |
|------|------|
| QTYPE | 원하는 정보 유형 |
| QCLASS | 클래스 |
| QNAME | 관심 이름 |

#### 응답 모드

##### 비재귀 모드 (Non-Recursive)

| 특성 | 설명 |
|------|------|
| 데이터 사용 | ** 로컬 정보만** 사용 |
| 반환 | 오류, 답변, 또는 더 가까운 서버로 참조 |
| 구현 | ** 모든 네임 서버에 필수** |
| 적합한 경우 | 요청자가 참조를 추적할 수 있고 캐싱 지원 원할 때 |

##### 재귀 모드 (Recursive)

| 특성 | 설명 |
|------|------|
| 역할 | 서버가 ** 리졸버 역할** 수행 |
| 반환 | 오류 또는 답변만 (** 참조 없음**) |
| 구현 | ** 선택적** |
| 제한 가능 | 승인된 클라이언트로 제한 가능 |

#### 모드 협상 (2비트 신호 메커니즘)

| 비트 | 설명 |
|------|------|
| **RA (Recursion Available)** | 서버가 응답에서 설정/해제, 요청과 관계없이 재귀 서비스 제공 ** 의사 표시** |
| **RD (Recursion Desired)** | 쿼리에서 설정, 클라이언트의 재귀 서비스 ** 선호 표시** |

- 클라이언트는 이전에 RA 신호를 보낸 서버나 명시적 합의가 있는 서버에서만 재귀 서비스에 의존해야 함

#### 재귀 응답 옵션

- 쿼리 답변 (앞에 CNAME RR 가능, 별칭 표시)
- 이름 오류 (비존재 표시, CNAME RR로 원래 쿼리 별칭 상태 표시 가능)
- 임시 오류 표시

#### 비재귀 응답 옵션

- 권위 있는 이름 오류
- 임시 오류
- 답변 조합 (권위/캐시 표시)
- 더 가까운 조상 서버로 참조
- 유용한 지원 RR

### 4.3.2 Algorithm (알고리즘)

네임 서버 알고리즘은 Zone별 트리 구조와 ** 별도 캐시** 로 조직된 RR 가정

#### 단계 1 - 재귀 설정

- 의지에 따라 **RA 설정**
- RD 요청되고 가능하면 → ** 단계 5**
- 그렇지 않으면 → ** 단계 2**

#### 단계 2 - Zone 위치

- QNAME의 가장 가까운 조상에 대해 사용 가능한 Zone 검색
- 발견 → ** 단계 3**
- 미발견 → ** 단계 4**

#### 단계 3 - 매칭 프로세스 (라벨별 매칭)

##### a. 완전 매칭

| 조건 | 처리 |
|------|------|
| CNAME 존재 & QTYPE ≠ CNAME | CNAME을 answer에 복사, QNAME을 정규 이름으로 변경, ** 단계 1로** |
| 그 외 | 일치하는 RR을 answer에 복사, ** 단계 6으로** |

##### b. Zone 외부 매칭 (참조)

- 서브존 NS RR을 authority 섹션에 복사
- 가용 주소를 additional 섹션에 추가 (필요시 Glue RR)
- ** 단계 4로**

##### c. 매칭 없음, 와일드카드 확인 ("*" 라벨)

| 조건 | 처리 |
|------|------|
| "*" 없음 | 원래 vs CNAME 추적 이름 확인; 원래=권위 있는 이름 오류 종료; CNAME 추적=그냥 종료 |
| "*" 존재 | QTYPE에 대해 RR 매칭; 일치하면 owner=QNAME (not "*")으로 answer에 복사, ** 단계 6으로** |

#### 단계 4 - 캐시 검색

- QNAME에 대해 캐시 검색
- 일치하는 RR을 answer에 복사
- 권위 있는 데이터의 위임 없으면 authority 섹션에 최선의 캐시 위임 찾기
- ** 단계 6으로**

#### 단계 5 - 재귀 해석

- ** 로컬 리졸버 알고리즘** 으로 쿼리 응답
- 결과(중간 CNAME 포함)를 answer 섹션에 저장

#### 단계 6 - Additional 섹션 채우기

- ** 로컬 데이터만** 사용
- additional 섹션에 유용한 RR 추가 시도
- ** 종료**

### 4.3.3 Wildcards (와일드카드)

#### 정의

- `*` 라벨로 시작하는 owner 이름을 가진 RR
- ** 합성 지침** 으로 기능

#### 동작

> "적절한 조건이 충족되면, 네임 서버는 쿼리 이름과 같은 owner 이름과 와일드카드 RR에서 가져온 내용으로 RR을 생성"

#### 주요 용도

- 인터넷에서 외부 시스템으로 ** 메일 전달**
- Zone 이름에 명시적 데이터 없는 경우 ** 기본 속성** 제공

#### 형식

- Owner 이름: `*.<anydomain>`
- anydomain은 다른 `*` 라벨 ** 없음**
- Zone 권위 있는 데이터에 ** 있어야 함**

#### 매칭 범위

- 와일드카드는 anydomain의 ** 자손에 적용** (anydomain 자체 제외)
- `*` 라벨은 ** 최소 하나의 전체 라벨** 매칭 (때로는 더 많음)

#### 비적용 조건

| 조건 | 설명 |
|------|------|
| 다른 Zone의 쿼리 | ** 위임이 와일드카드 취소** |
| 쿼리 이름 또는 중간 이름 존재 | 예: `*.X`는 `B.X`에 명시적 데이터 있으면 `B.X`나 `A.B.X`에 적용 안 됨 |

#### 쿼리 이름의 "*"

- 쿼리 이름의 `*`는 특별한 효과 없음
- 와일드카드 테스트 가능
- 결과는 ** 캐싱하면 안 됨**

#### 콘텐츠 보존

- RR 합성 시 와일드카드 RR 내용 ** 변경 없음**

#### 상세 예시 (메일 게이트웨이 회사)

회사: X.COM, 게이트웨이: A.X.COM (IP: 1.2.3.4)

**COM zone 항목**:
```
X.COM           MX      10      A.X.COM
*.X.COM         MX      10      A.X.COM
A.X.COM         A       1.2.3.4
A.X.COM         MX      10      A.X.COM
*.A.X.COM       MX      10      A.X.COM
```

** 효과**:

| 상황 | 결과 |
|------|------|
| X.COM으로 끝나는 모든 MX 쿼리 | A.X.COM 참조 반환 |
| 두 와일드카드 RR 필요 이유 | `*.X.COM`은 A.X.COM 서브트리 내에서 명시적 데이터에 의해 억제됨 |
| X.COM과 A.X.COM의 명시적 MX 데이터 | 필요 |
| XX.COM 쿼리 | 와일드카드 매칭 ** 없음** |

### 4.3.4 Negative Response Caching (부정 응답 캐싱) - 선택적

#### 개념

- 네임 서버가 **TTL과 함께 부정 결과** 배포
- 리졸버가 권위 있는 데이터 조회 없이 TTL 기간 동안 ** 비존재 가정** 하며 캐싱

#### 구현 방법

- 권위 있을 때 네임 서버가 응답 **additional 섹션에 SOA RR 추가**
- SOA 소스: answer 섹션 데이터 Zone, 또는 이름 오류 적용 Zone
- **SOA MINIMUM 필드** 가 부정 결과 캐시 수명 제어

#### 다중 Owner 응답

- 여러 owner 이름 포함 answer 섹션은 **QNAME 일치 데이터에만 SOA** 필요 (섹션 내 유일한 권위 있는 데이터)

#### 비권위적 응답

- 네임 서버와 리졸버는 비권위적 응답에 **SOA 추가하거나 미명시 결과 추론 금지**
- 캐시된 정보는 RR과 Zone 이름 매칭에 보통 충분하지 않음
- 직접 SOA 쿼리로 SOA RR이 캐시될 수 있음
- 네임 서버가 모든 권위 있는 응답에 SOA 출력 필수 아님

#### 선택적 상태

| 항목 | 상태 |
|------|------|
| 네임 서버 | 모든 권위 있는 응답에 SOA 추가 ** 필수 아님** |
| 리졸버 | 부정 결과 캐싱 ** 필수 아님** |
| 권장 | ** 둘 다 권장** |
| 필수 | 모든 리졸버와 재귀 서버: 존재 시 SOA RR ** 무시 필수** |

#### 실험적 향상 (계획만, 권장 안 함)

- 캐시된 데이터의 Zone SOA 획득 및 SERIAL 변경 없으면
- 더 작으면 TTL을 Zone MINIMUM으로 리셋

### 4.3.5 Zone Maintenance and Transfers (Zone 유지보수와 전송)

#### 자동 새로고침 모델

##### 마스터/프라이머리 모델

| 요소 | 설명 |
|------|------|
| 마스터 네임 서버 | 하나의 Zone 네임 서버가 ** 마스터로 지정** |
| 변경 조정 | 프라이머리에서 조정 (일반적으로 ** 마스터 파일 편집**) |
| 관리자 신호 | 새 Zone 로딩 신호 |
| 세컨더리 서버 | 주기적으로 ** 변경 확인** |

#### 변경 감지 방법

- 세컨더리가 **SOA SERIAL 필드** 모니터링
- Zone에 변경 있을 때마다 **SERIAL 필드 항상 진행**
- 시리얼 번호 진행과 비교는 ** 시퀀스 공간 산술** 사용
- Zone 업데이트 속도에 ** 이론적 한계** 존재

#### 폴링 제어 파라미터 (SOA RR)

| 파라미터 | 설명 |
|----------|------|
| **REFRESH** | 세컨더리가 프라이머리에 새 SERIAL 확인 전 ** 최소 초** |
| **RETRY** | 프라이머리 연락 실패 시 확인 간 ** 초** |
| **EXPIRE** | 성공적 시리얼 확인 없이 Zone을 ** 폐기 처리 전 최대 초** |

#### 프로세스 흐름

1. 세컨더리가 새 Zone 로드 후 **REFRESH 초** 대기
2. 프라이머리 **SOA RR 시리얼** 쿼리
3. 시리얼 변경 없음 → **REFRESH 대기 재시작**
4. EXPIRE 간격 동안 확인 완료 불가 → Zone을 ** 폐기로 처리**
5. 시리얼 변경됨 → **AXFR Zone 전송 요청 시작**

#### Zone 전송 실행

- AXFR은 오류(거부) 또는 응답 시퀀스 생성 가능
- ** 첫 번째와 마지막 메시지**: 최상위 권위 노드 데이터 포함
- ** 중간 메시지**: 나머지 모든 RR (권위 및 비권위)
- 메시지로 세컨더리 **Zone 재구성** 가능
- ** 정확성 필수로 TCP 또는 다른 신뢰할 수 있는 프로토콜 사용 필요**

#### 선택적 확장

- 세컨더리 서버가 다른 세컨더리에 ** 선택적으로 쿼리** 가능
- 프라이머리 다운타임 또는 더 나은 네트워크 접근 시 ** 가용성 향상**

## 5. RESOLVERS

### 5.1 Introduction (소개)

#### 정의

- 사용자 프로그램을 네임 서버에 ** 인터페이스하는 프로그램**

#### 동작 모델

| 단계 | 설명 |
|------|------|
| 입력 | 사용자 프로그램(메일, TELNET, FTP)에서 ** 서브루틴/시스템 호출** 수신 |
| 출력 | ** 로컬 호스트 호환 형식** 으로 정보 반환 |

#### 네트워크 특성

- 요청 프로그램과 같은 머신의 리졸버가 ** 원격 네임 서버** 조회 가능
- 완료 시간 ** 매우 가변적** (밀리초~초)

#### 핵심 성능 목표

> "리졸버의 매우 중요한 목표는 이전 결과의 캐시에서 대부분의 요청에 응답하여 ** 네트워크 지연과 네임 서버 부하를 제거** 하는 것"

#### 캐시 최적화

- 여러 프로세스/사용자/머신 간 ** 공유 캐시** 가 비공유 캐시보다 효율적

### 5.2 Client-Resolver Interface (클라이언트-리졸버 인터페이스)

### 5.2.1 Typical Functions (일반적 기능)

#### 기능 1 - 호스트 이름 → 주소 변환

| 항목 | 설명 |
|------|------|
| 동작 | 이전 HOSTS.TXT 기능 모방 |
| 입력 | 문자열 |
| 출력 | 하나 이상의 **32비트 IP 주소** |
| DNS 변환 | **A 타입 RR** 요청 |
| DNS 장점 | 여러 주소 반환 권장; 순서 비보장, 리졸버 ** 정렬/선택** 허용 |

#### 기능 2 - 호스트 주소 → 호스트 이름 변환

| 항목 | 설명 |
|------|------|
| 동작 | ** 역방향 매핑** |
| 입력 | 32비트 IP 주소 |
| 출력 | 문자열 |
| 처리 | IP 주소 옥텟 ** 역순** + `IN-ADDR.ARPA` 접미사 |
| 예시 | IP `1.2.3.4` → `4.3.2.1.IN-ADDR.ARPA` **PTR 쿼리** |

#### 기능 3 - 일반 조회 기능

| 항목 | 설명 |
|------|------|
| 동작 | ** 임의 DNS 정보** 검색 (HOSTS.TXT에 대응 없음) |
| 입력 | QNAME, QTYPE, QCLASS |
| 출력 | 일치하는 RR |
| 특징 | DNS 형식으로 ** 모든 RR 콘텐츠(TTL 등)** 반환 |

#### 결과 가능성

| 결과 | 설명 |
|------|------|
| ** 성공적 매칭** | 요청 데이터 제공하는 하나 이상의 RR (적절한 형식으로 반환) |
| ** 이름 오류 (NE)** | 참조된 이름 ** 존재 안 함** (사용자 오타, 유효하지 않은 도메인) |
| ** 데이터 없음** | 참조된 이름 존재하나 요청 ** 데이터 타입 없음** (메일박스 이름에 주소 쿼리) |

#### 오류 구분 중요성

- 호스트 주소/이름 변환 기능: "이름 오류"와 "데이터 없음" 단일 오류로 결합 ** 가능**
- 일반 기능: ** 반드시 구분 필요**
- 이유: 애플리케이션이 같은 이름에 대해 다른 타입 정보를 ** 순차 요청** 할 수 있음

### 5.2.2 Aliases (별칭)

#### 문제

- 해석 중 쿼리 이름이 ** 별칭 식별** 가능
- 예: 호스트→주소 변환 중 CNAME RR 발견

#### 신호 요구사항

> "가능하면, 별칭 조건을 리졸버에서 클라이언트로 다시 ** 신호해야 함**"

#### 처리 로직

| 상황 | 처리 |
|------|------|
| CNAME 발견 | 리졸버가 새 이름에서 ** 쿼리 재시작** |
| CNAME 타입 쿼리 수행 시 | 별칭 추적 안 함 (CNAME 자체 반환) |

#### 오류 조건

| 조건 | 처리 |
|------|------|
| 여러 별칭 레벨 | 비효율로 인해 ** 피해야 함**; 오류 아님 |
| 별칭 루프 | 잡아서 ** 오류 신호** |
| 존재하지 않는 이름으로의 별칭 | 잡아서 ** 오류 신호** |

### 5.2.3 Temporary Failures (임시 실패)

#### 실패 소스

- 네트워크 격리 (링크/게이트웨이 실패)
- 도메인의 ** 모든 서버 불가용**

#### 중요한 처리

> "이런 종류의 조건을 애플리케이션에 ** 이름 또는 데이터 없음 오류로 신호하면 안 됨**"

- 잘못된 신호는 ** 해로운 메일 시스템 동작** 가능

#### 권장 솔루션

> "리졸버 함수의 가능한 결과 중 하나로 ** 항상 임시 실패** 를 가져야 함"

- 기존 HOSTS.TXT 함수 에뮬레이션이 더 어려워질 수 있음
- 무기한 차단은 ** 부적절**, 특히 다른 작업이 있는 서버 프로세스에서

### 5.3 Resolver Internals (리졸버 내부)

- 구현은 알고리즘과 오류 처리 복잡성에서 ** 크게 다름**
- 이 섹션은 권장 전략 개요; 세부사항은 RFC-1035

### 5.3.1 Stub Resolvers (스텁 리졸버)

#### 개념

- ** 재귀 쿼리 지원 네임 서버** 로 해석 기능 오프로드
- 리소스 제한 플랫폼(PC) 또는 조직 ** 중앙 캐싱** 에 단순한 방법 제공

#### 최소 요구사항

- 재귀 요청 수행하는 ** 네임 서버 주소 목록**
- 일반적으로 플랫폼 제한으로 ** 구성 파일** 에서

#### 검증 필요성

- 사용자는 나열된 서버가 재귀 서비스 수행하는지 ** 확인 필요**
- 서버가 모든 클라이언트 ** 거부 가능**
- 로컬 시스템 관리자와 상담하여 서비스 제공 의사 있는 네임 서버 찾기

#### 단점

| 단점 | 설명 |
|------|------|
| 임의 지속시간 | 재귀 요청 지속시간 ** 최적화 어려움** (재전송 간격) |
| 네임 서버 과부하 | 열정적 스텁이 재전송을 ** 새 요청으로 해석** |
| TCP 대안 | 실제 리졸버와 ** 유사한 호스트 능력 부담** |

### 5.3.2 Resources (리소스)

- 리졸버가 공유 로컬 네임 서버 Zone 접근으로 ** 더 빠른 접근** 가능
- ** 제약**: 캐시된 정보가 Zone 데이터를 ** 절대 재정의하면 안 됨**
- "로컬 정보" 용어: 캐시와 공유 Zone 포함, ** 권위 있는 데이터 선호**

#### 데이터 구조

| 구조 | 설명 |
|------|------|
| **SNAME** | 검색 중인 도메인 이름 |
| **STYPE** | 쿼리 타입 (QTYPE) |
| **SCLASS** | 쿼리 클래스 (QCLASS) |
| **SLIST** | 현재 네임 서버 집합과 Zone 정보 |
| **SBELT** | "안전 벨트" 구성 구조 (정보 없는 상황용) |
| **CACHE** | 이전 응답 결과 저장 |

#### SLIST 상세

| 항목 | 설명 |
|------|------|
| Zone 이름 동등성 | 루트에서 ** 라벨 매칭 수** ("가까움" 표시) |
| 알려진 네임 서버 | - |
| 알려진 주소 | - |
| 이력 데이터 | ** 최선의 서버** 제안 |

#### SBELT 상세

| 항목 | 설명 |
|------|------|
| 형식 | SLIST와 ** 동일** |
| 초기화 | ** 구성 파일** 에서 |
| 용도 | 정보 없는 상황용 나열된 서버 |
| 매칭 카운트 | **-1** (0개 라벨 알려짐 표시) |

#### CACHE 상세

- 일반적으로 RR 저장 시 간격을 ** 절대 시간으로 변환**
- ** 주기적 스윕** 또는 발견 기반 폐기로 만료된 RR 제거

### 5.3.3 Algorithm (알고리즘)

#### 단계 1 - 로컬 검색

- 답변을 위해 ** 캐시 검색**
- 발견되면 클라이언트에 반환
- 일부 구현: 캐시 우회하고 권위 있는 서버 조회 ** 옵션** (기본값 비권장)

#### 단계 2 - 서버 선택

- SNAME에서 위로 **NS RR 위치** 찾기 (SNAME, 부모, 조부모, 루트로)

** 예시 경로** (`Mockapetris.ISI.EDU`):
1. Mockapetris.ISI.EDU
2. ISI.EDU
3. EDU
4. . (root)

- NS RR 이름을 **SLIST에 복사**
- 로컬 데이터에서 ** 주소 채우기**
- 모든 주소 없을 수 있음
- 권장: 알 수 없는 주소에 ** 병렬 리졸버 프로세스** 시작하면서 가용 주소로 계속

#### 설계자 우선순위 (순서대로)

1. ** 무한 루프/연쇄 반응 방지 작업 제한** (잘못된 구성에서도 중요)
2. 가능하면 ** 답변 획득**
3. ** 불필요한 전송 방지**
4. ** 속도 최적화**

#### SBELT 폴백

- NS RR 검색 실패 시 ** 안전 벨트에서 SLIST 초기화**
- 일반적 구성: **2개 루트 서버 + 2개 로컬 도메인 서버** (중복 + 인터넷 접근 + 로컬 격리 복원력)

#### SLIST 최적화

- ** 주소 지역성** 선호
- ** 과거 응답 통계**
- ** 배팅 평균** 으로 정렬

#### 단계 3 - 쿼리 전송

- 전송 간 ** 타임아웃** 으로 모든 서버의 모든 주소 순환

> "실제로 다중홈 호스트의 ** 모든 주소를 사용** 하는 것이 중요하고, 너무 공격적인 재전송 정책은 여러 리졸버가 사용할 때 실제로 응답을 ** 늦춤**"

- SLIST에 ** 타임아웃 제어 및 전송 이력** 포함

#### 단계 4 - 응답 분석

- ** 매우 편집증적 파싱 필요**
- **ID 필드** 로 응답이 보낸 쿼리와 일치하는지 확인 필요

##### 이상적 답변

- 권위 있는 서버가 ** 데이터 또는 이름 오류** 제공
- 전달하고 ** 캐시** (TTL > 0)

##### 위임 처리

- ** 라벨 매칭 카운트** 비교로 SLIST 서버 대비 위임 가까움 검증
- 가짜 위임 ** 무시**
- 유효한 위임: NS와 주소 RR ** 캐시**, SLIST ** 업데이트**, 검색 ** 재시작**

##### CNAME 처리

- 응답에 답변 데이터 포함하거나 CNAME 자체가 답변이 아닌 한 CNAME에서 검색 ** 재시작**

#### 세부사항
- RFC-1035에서 구현 힌트와 세부사항 제공

## 6. SCENARIO (시나리오)

### 예시 Zone 할당

예시 도메인 공간 (root, MIL, EDU, MIT.EDU, ISI.EDU)에 대한 샘플 Zone 할당:

```
네임 서버별 Zone:
- C.ISI.EDU: root, EDU (연속)
- A.ISI.EDU: root, MIL, ISI.EDU (연속+비연속)
- SRI-NIC.ARPA: root, MIL, EDU
```

- 서버는 ** 연속 또는 분리된 Zone** 유지 가능
- "A.ISI.EDU는 root와 MIL 도메인에서 연속 Zone을 가지지만, ISI.EDU에서 ** 비연속 Zone** 도 가짐"

### 6.1 C.ISI.EDU Name Server

C.ISI.EDU는 root, MIL, EDU 도메인 (IN 클래스) 서비스

#### Root Zone 데이터 예시

```
.       IN      SOA     SRI-NIC.ARPA. HOSTMASTER.SRI-NIC.ARPA. (
                        870611          ;serial
                        1800            ;refresh every 30 min
                        300             ;retry every 5 min
                        604800          ;expire after a week
                        86400)          ;minimum of a day
                NS      A.ISI.EDU.
                NS      C.ISI.EDU.
                NS      SRI-NIC.ARPA.

MIL.    86400   NS      SRI-NIC.ARPA.
        86400   NS      A.ISI.EDU.

EDU.    86400   NS      SRI-NIC.ARPA.
        86400   NS      C.ISI.EDU.

SRI-NIC.ARPA.   A       26.0.0.73
                A       10.0.0.51
                MX      0 SRI-NIC.ARPA.
                HINFO   DEC-2060 TOPS20

ACC.ARPA.       A       26.6.0.65
                HINFO   PDP-11/70 UNIX
                MX      10 ACC.ARPA.

USC-ISIC.ARPA.  CNAME   C.ISI.EDU.

73.0.0.26.IN-ADDR.ARPA.  PTR    SRI-NIC.ARPA.
65.0.6.26.IN-ADDR.ARPA.  PTR    ACC.ARPA.
51.0.0.10.IN-ADDR.ARPA.  PTR    SRI-NIC.ARPA.
52.0.0.10.IN-ADDR.ARPA.  PTR    C.ISI.EDU.
103.0.3.26.IN-ADDR.ARPA. PTR    A.ISI.EDU.

A.ISI.EDU. 86400 A      26.3.0.103
C.ISI.EDU. 86400 A      10.0.0.52
```

#### 형식 특성

| 특성 | 설명 |
|------|------|
| 다중 줄 SOA RR | ** 괄호** 사용 |
| 클래스 지정 | 첫 Zone RR에서 지정 (** 모두 상속**) |
| TTL 강제 | SOA MINIMUM (86400초/일)으로 ** 권위 RR의 최소 TTL** |
| 위임 NS RR | MIL/EDU 경계 표시, ** 명시적 TTL** (비권위적) |
| Glue RR | 서버 주소용 |

#### Root 노드

- **SOA** + **3개 NS RR** (Zone 관리 및 서버 설명)

#### Zone 관리 데이터 (SOA)

| 필드 | 값 | 설명 |
|------|-----|------|
| **MNAME** | SRI-NIC.ARPA | 마스터 |
| **RNAME** | HOSTMASTER@SRI-NIC.ARPA | 책임자 |
| **MINIMUM** | 86400초 | Zone RR의 ** 최소 TTL** |

#### 위임 표시

- MIL, EDU 경계의 NS RR이 ** 비루트 Zone 서버** 표시

#### EDU Zone 데이터 예시 (origin EDU)

```
EDU.  IN SOA SRI-NIC.ARPA. HOSTMASTER.SRI-NIC.ARPA. (
                        870729 ;serial
                        1800 ;refresh 30 min
                        300 ;retry 5 min
                        604800 ;expire week
                        86400 ;minimum day
                        )
                NS SRI-NIC.ARPA.
                NS C.ISI.EDU.

UCI 172800 NS ICS.UCI
            172800 NS ROME.UCI

                172800 A 128.9.0.33

                172800 A 128.9.0.32

                    172800 NS UMN-REI-UC.ARPA.
```

|------|------|

### 6.2 Example Standard Queries (표준 쿼리 예시)

- 비재귀 쿼리 (RD 미설정)는 요청자 아닌 ** 쿼리된 서버에 따른** 답변 반환

#### 6.2.1 QNAME=SRI-NIC.ARPA, QTYPE=A

- 서버가 권위 있는 데이터 보유 시 직접 응답

## 7. GLOSSARY (용어집)

| 용어 | 정의 |
|------|------|
| **Authoritative** | 권위 있는 - Zone 데이터에서 직접 제공되는 데이터 |
| **Cache** | 캐시 - 빠른 검색을 위해 저장된 이전 쿼리 결과 |
| **CNAME** | Canonical Name - 별칭의 정규 이름 |
| **Delegation** | 위임 - 서브존에 대한 권한 이전 |
| **Domain** | 도메인 - DNS 트리의 노드 |
| **FQDN** | Fully Qualified Domain Name - 완전한 도메인 이름 |
| **Glue** | 글루 - 서브존 서버 주소를 제공하는 비권위적 RR |
| **Label** | 라벨 - 도메인 이름의 단일 구성 요소 |
| **Master** | 마스터 - Zone의 원본 데이터를 가진 서버 |
| **Name Server** | 네임 서버 - DNS 쿼리에 응답하는 서버 |
| **Primary** | 프라이머리 - 마스터 파일에서 Zone 데이터를 로드하는 서버 |
| **QCLASS** | Query Class - 쿼리의 클래스 |
| **QNAME** | Query Name - 쿼리의 도메인 이름 |
| **QTYPE** | Query Type - 쿼리의 타입 |
| **RDATA** | Resource Data - RR의 실제 데이터 |
| **Recursive** | 재귀적 - 서버가 완전한 답변을 제공하는 쿼리 모드 |
| **Resolver** | 리졸버 - 클라이언트와 네임 서버 사이의 인터페이스 |
| **RR** | Resource Record - 리소스 레코드 |
| **Secondary** | 세컨더리 - 다른 서버에서 Zone 데이터를 전송받는 서버 |
| **SOA** | Start of Authority - Zone의 시작과 파라미터를 정의하는 RR |
| **TTL** | Time To Live - RR의 캐시 수명 |
| **Zone** | 존 - 단일 관리 단위로 관리되는 도메인 트리의 연속된 부분 |
| **Zone Transfer** | Zone 전송 - 서버 간 Zone 데이터 복제 |

## 요약

RFC 1034는 DNS의 아키텍처 원칙을 수립합니다:

| 원칙 | 설명 |
|------|------|
| ** 분산 계층적 네이밍** | 트리 구조의 도메인 네임스페이스 |
| ** 위임된 권한** | Zone 기반 관리로 분산 운영 |
| ** 캐싱** | TTL 기반 캐시로 성능과 가용성 향상 |
| ** 재귀/반복 해석** | 클라이언트와 서버의 다양한 사용 사례 지원 |
| **Zone 기반 관리** | 조직 구조에 맞는 관리 경계 |

핵심 메커니즘(리소스 레코드, 네임 서버, 리졸버)은 프로토콜 안정성을 유지하면서 ** 확장성** 을 지원합니다. 명세는 즉각적 업데이트보다 ** 견고성, 중복성, 점진적 일관성** 을 강조하며, 이는 DNS의 장기적 인터넷 확장성을 가능하게 하는 기본 원칙입니다.
