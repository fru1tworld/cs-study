# RFC 2119: Key words for use in RFCs to Indicate Requirement Levels

> RFC 문서에서 요구사항 수준을 나타내기 위해 사용되는 핵심 단어들

## 문서 정보

| 항목 | 내용 |
|------|------|
| **RFC 번호** | 2119 |
| **분류** | BCP 14 (Best Current Practice) |
| **작성자** | S. Bradner (Harvard University) |
| **발행일** | 1997년 3월 |
| **상태** | 현행 표준 |

## 개요

많은 표준 문서에서 명세의 요구사항을 나타내기 위해 여러 단어들이 사용됩니다. 이러한 단어들은 종종 대문자로 표기됩니다. RFC 2119는 IETF 문서에서 이 단어들이 어떻게 해석되어야 하는지를 정의합니다.

이 문서는 **모든 RFC의 범례** 가 되는 메타 문서로, RFC 문서를 읽기 전에 반드시 이해해야 합니다.

---

## 핵심 키워드 정의

### 1. MUST / REQUIRED / SHALL (필수)

```
강제성: ████████████ 100% (절대적 요구사항)
```

**정의**: 해당 정의가 명세의 **절대적 요구사항** 임을 의미합니다.

- 반드시 구현해야 함
- 예외가 허용되지 않음
- 미준수 시 명세를 위반하는 것으로 간주

**예시**:
- "모든 구현체는 이 프로토콜을 지원해야 한다(MUST support)"
- "클라이언트는 유효한 인증서를 제시해야 한다(SHALL present)"

---

### 2. MUST NOT / SHALL NOT (금지)

```
강제성: ████████████ 100% (절대적 금지)
```

**정의**: 해당 정의가 명세의 **절대적 금지사항** 임을 의미합니다.

- 절대 해서는 안 됨
- 어떤 상황에서도 금지
- 이를 어기면 명세를 위반하는 것

**예시**:
- "서버는 이 필드를 수정해서는 안 된다(MUST NOT modify)"
- "구현체는 이 오래된 알고리즘을 사용해서는 안 된다(SHALL NOT use)"

---

### 3. SHOULD / RECOMMENDED (권장)

```
강제성: ████████░░░░ 70% (강한 권장)
```

**정의**: 특정 상황에서 특정 항목을 무시할 **타당한 이유가 존재할 수 있음** 을 의미합니다. 그러나 다른 방향을 선택하기 전에 **전체적인 영향을 이해하고 신중하게 검토** 해야 합니다.

- 일반적으로 따라야 함
- 예외적인 상황에서만 무시 가능
- 무시하기 전에 영향을 완전히 이해해야 함

**예시**:
- "구현체는 이 최적화를 지원하는 것이 좋다(SHOULD support)"
- "새로운 알고리즘을 사용하는 것이 권장된다(RECOMMENDED)"

---

### 4. SHOULD NOT / NOT RECOMMENDED (비권장)

```
강제성: ████████░░░░ 70% (강한 비권장)
```

**정의**: 특정 행동이 허용되거나 심지어 유용할 수 있는 **타당한 이유가 특정 상황에서 존재할 수 있음** 을 의미합니다. 그러나 이 라벨이 붙은 동작을 구현하기 전에 **전체적인 영향을 이해하고 신중하게 검토** 해야 합니다.

- 일반적으로 하지 않아야 함
- 특별한 사유가 있는 경우에만 예외 가능
- 예외를 적용하기 전에 신중한 검토 필요

**예시**:
- "이 방식은 사용하지 않는 것이 좋다(SHOULD NOT use)"
- "이 옵션은 권장되지 않는다(NOT RECOMMENDED)"

---

### 5. MAY / OPTIONAL (선택)

```
강제성: ░░░░░░░░░░░░ 0% (완전한 선택사항)
```

**정의**: 해당 항목이 **완전히 선택적** 임을 의미합니다. 한 벤더는 특정 시장에서 요구하거나 제품 향상을 위해 해당 항목을 포함할 수 있고, 다른 벤더는 같은 항목을 생략할 수 있습니다.

**중요한 상호운용성 요구사항**:

MAY/OPTIONAL 키워드는 단순히 "해도 되고 안 해도 된다"는 의미가 아닙니다. **상호운용성 보장** 이라는 중요한 책임이 따릅니다:

| 구현체 유형 | 상호운용성 의무 |
|------------|----------------|
| 옵션을 **포함하지 않은** 구현체 | 해당 옵션을 포함한 다른 구현체와 **상호운용할 준비가 되어 있어야 한다(MUST)** |
| 옵션을 **포함한** 구현체 | 해당 옵션을 포함하지 않은 다른 구현체와 **상호운용할 준비가 되어 있어야 한다(MUST)** |

**상호운용성 조항의 실제 적용**:

```
예: HTTP 서버가 "Keep-Alive" 연결을 지원하는 것이 MAY(선택)인 경우

✅ Keep-Alive를 지원하는 서버:
   - Keep-Alive를 지원하지 않는 클라이언트의 요청도 정상 처리해야 함

✅ Keep-Alive를 지원하지 않는 서버:
   - Keep-Alive를 요청하는 클라이언트에게 적절한 응답을 제공해야 함
   - 연결을 정상적으로 종료하고 클라이언트가 재연결할 수 있게 해야 함

❌ 잘못된 구현:
   - Keep-Alive만 지원하고 일반 연결 요청을 거부하는 것
   - Keep-Alive 요청에 대해 오류를 반환하는 것
```

**예시**:
- "구현체는 이 확장을 지원할 수 있다(MAY support)"
- "이 필드는 선택적이다(OPTIONAL)"
- "서버는 캐시 힌트를 제공할 수 있다(MAY provide)"

## SHOULD NOT vs MUST NOT 차이 명확화

이 두 키워드는 모두 "하지 말라"는 의미를 담고 있지만, **강제성과 예외 허용 여부** 에서 중요한 차이가 있습니다.

### 비교표

| 구분 | MUST NOT | SHOULD NOT |
|------|----------|------------|
| **강제성** | 100% (절대적) | ~70% (강한 권장) |
| **예외 허용** | 불가 | 가능 (타당한 이유 필요) |
| **위반 시** | 명세 비준수 | 권장사항 미준수 |
| **결정권** | 명세가 결정 | 구현자가 결정 가능 |

### 실제 차이 예시

```
📌 MUST NOT 예시 (TLS 1.3):
   "구현체는 TLS 1.2의 압축 기능을 사용해서는 안 된다(MUST NOT)"

   → 이유: CRIME 공격 등 보안 취약점이 있으므로 어떤 상황에서도 사용 불가
   → 결과: 이를 사용하면 TLS 1.3 비준수 구현체로 간주됨

📌 SHOULD NOT 예시 (HTTP/1.1):
   "클라이언트는 동일 서버에 2개 이상의 동시 연결을 열지 않는 것이 좋다(SHOULD NOT)"

   → 이유: 서버 리소스 보호를 위한 권장사항
   → 예외: 고대역폭이 필요한 특수한 경우 더 많은 연결 가능
   → 결과: 예외적으로 더 많은 연결을 열어도 명세 준수로 인정됨
```

### 선택 기준

| 상황 | 권장 키워드 |
|------|------------|
| 보안 취약점 또는 심각한 문제 유발 | **MUST NOT** |
| 상호운용성 파괴 가능성 | **MUST NOT** |
| 프로토콜 손상 가능성 | **MUST NOT** |
| 성능 저하 우려 | SHOULD NOT |
| 비효율적인 구현 | SHOULD NOT |
| 향후 호환성 우려 | SHOULD NOT |
| 특수 상황에서 필요할 수 있음 | SHOULD NOT |

## 요약 비교표

| 키워드 | 한국어 | 강제성 | 설명 |
|--------|--------|--------|------|
| **MUST** / REQUIRED / SHALL | 필수 | 100% | 절대적 요구사항 |
| **MUST NOT** / SHALL NOT | 금지 | 100% | 절대적 금지사항 |
| **SHOULD** / RECOMMENDED | 권장 | ~70% | 강하게 권장하나 예외 가능 |
| **SHOULD NOT** / NOT RECOMMENDED | 비권장 | ~70% | 강하게 비권장하나 예외 가능 |
| **MAY** / OPTIONAL | 선택 | 0% | 완전한 선택사항 |

## 사용 지침

### 문서에 포함할 표준 문구

이 가이드라인을 따르는 저자는 문서 시작 부분에 다음 문구를 포함해야 합니다:

> 이 문서에서 사용된 "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", "OPTIONAL"이라는 키워드는 RFC 2119에 기술된 대로 해석되어야 합니다.

### 키워드 사용 시 주의사항

이 키워드들은 **신중하고 절제 있게** 사용해야 합니다:

1. **상호운용성에 실제로 필요한 경우** 에만 사용
2. **해로운 동작을 제한** 하기 위해 필요한 경우에만 사용
3. 상호운용성에 필요하지 않은데 **특정 구현 방법을 강제하기 위해 사용해서는 안 됨**

```
❌ 잘못된 사용: 특정 구현 방법 강제
   "구현체는 연결 리스트를 사용해야 한다(MUST use)"

✅ 올바른 사용: 상호운용성을 위한 요구사항
   "구현체는 UTF-8 인코딩을 지원해야 한다(MUST support)"
```

### 추가 주의사항

#### 1. 과도한 MUST 사용 자제

MUST를 남용하면 다음과 같은 문제가 발생합니다:

| 문제점 | 설명 |
|--------|------|
| 구현 부담 증가 | 모든 요구사항을 만족해야 하므로 개발 비용 증가 |
| 혁신 저해 | 새로운 방식의 구현을 원천적으로 차단 |
| 준수 불가능 | 현실적으로 모든 MUST를 만족하기 어려워짐 |

```
❌ 과도한 MUST:
   "메시지는 64바이트 이하여야 한다(MUST)"
   "응답 시간은 100ms 이내여야 한다(MUST)"

✅ 적절한 표현:
   "메시지는 64바이트 이하인 것이 좋다(SHOULD)"
   "응답 시간은 100ms 이내를 권장한다(RECOMMENDED)"
```

#### 2. 모호한 표현 피하기

키워드와 함께 사용되는 문장은 **명확하고 검증 가능** 해야 합니다:

```
❌ 모호한 표현:
   "구현체는 빠르게 응답해야 한다(MUST respond quickly)"
   "서버는 적절한 보안을 제공해야 한다(MUST provide adequate security)"

✅ 명확한 표현:
   "구현체는 30초 이내에 응답해야 한다(MUST respond within 30 seconds)"
   "서버는 TLS 1.2 이상을 지원해야 한다(MUST support TLS 1.2 or higher)"
```

#### 3. 일관된 키워드 사용

동일한 문서 내에서 **동일한 의미** 에는 **동일한 키워드** 를 사용해야 합니다:

```
❌ 비일관적 사용:
   "클라이언트는 인증서를 제시해야 한다(MUST)"
   "클라이언트는 인증서를 제시하는 것이 필수이다(REQUIRED)"
   "클라이언트는 인증서를 제시한다(SHALL)"  ← 같은 문서에서 혼용

✅ 일관적 사용:
   같은 수준의 요구사항에는 하나의 키워드만 선택하여 사용
```

#### 4. 부정형 키워드의 명확한 범위 지정

MUST NOT이나 SHOULD NOT 사용 시 **금지 범위를 명확히** 해야 합니다:

```
❌ 범위 불명확:
   "이 필드를 사용해서는 안 된다(MUST NOT use)"

✅ 범위 명확:
   "이 필드를 새로운 구현에서 사용해서는 안 된다(MUST NOT use in new implementations)"
   "이 필드를 인증 목적으로 사용해서는 안 된다(MUST NOT use for authentication)"
```

## 보안 고려사항

이 용어들은 **보안과 관련된 동작** 을 명시하는 데 자주 사용됩니다.

**주의**: MUST나 SHOULD를 구현하지 않거나, MUST NOT이나 SHOULD NOT으로 명시된 것을 수행했을 때의 **보안에 대한 영향은 매우 미묘** 할 수 있습니다.

문서 저자는 다음을 해야 합니다:
- 권장사항이나 요구사항을 따르지 않았을 때의 **보안 영향을 상세히 설명**
- 대부분의 구현자들이 명세를 만든 경험과 논의의 혜택을 받지 못했음을 고려

## 실제 활용 예시

### HTTP/2 명세에서의 사용 예

```
클라이언트는 HTTP/2 연결을 열기 전에 서버가 HTTP/2를 지원하는지
확인해야 한다(MUST).

클라이언트는 알 수 없는 프레임 타입을 무시해야 한다(MUST).

서버는 GOAWAY 프레임을 보내기 전에 진행 중인 스트림 처리를
완료하는 것이 좋다(SHOULD).

구현체는 흐름 제어 윈도우 크기를 조정할 수 있다(MAY).
```

### TLS 1.3 명세에서의 사용 예

```
구현체는 TLS 1.2 이하 버전의 압축 메서드를 지원해서는 안 된다
(MUST NOT).

서버는 인증서 요청 시 신뢰할 수 있는 CA 목록을 제공하는 것이
좋다(SHOULD).

클라이언트는 세션 재개를 위해 PSK를 사용할 수 있다(MAY).
```

## RFC 8174: 대소문자 규칙

### 배경

RFC 2119가 발행된 이후, 키워드가 **대문자일 때만 특별한 의미** 를 갖는지에 대한 혼란이 있었습니다. RFC 8174 (2017년 발행)는 이러한 모호성을 해소하기 위해 작성되었습니다.

### 핵심 규칙

RFC 8174는 다음을 명확히 합니다:

> **키워드가 모두 대문자(ALL UPPERCASE)로 작성된 경우에만** RFC 2119의 특별한 의미를 갖습니다.

| 표기 | 의미 |
|------|------|
| `MUST` | RFC 2119 키워드 (절대적 요구사항) |
| `Must` | 일반 영어 단어 |
| `must` | 일반 영어 단어 |

### 예시

```
✅ RFC 2119 키워드로 해석:
   "Implementations MUST support TLS 1.2"
   → 구현체는 반드시 TLS 1.2를 지원해야 함 (강제)

✅ 일반 영어로 해석:
   "The implementation must be well-documented"
   → 구현체가 잘 문서화되어야 함 (일반적 서술)

✅ 일반 영어로 해석:
   "This should be obvious to most readers"
   → 대부분의 독자에게 명확해야 함 (일반적 서술)
```

### 업데이트된 표준 문구

RFC 8174에 따라 문서 시작 부분에 포함해야 하는 권장 문구:

> 이 문서에서 사용된 "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", "OPTIONAL"이라는 키워드는 **모두 대문자로 나타날 때만** BCP 14 [RFC 2119] [RFC 8174]에 기술된 대로 해석되어야 합니다.

### 실제 적용

| 상황 | 처리 방법 |
|------|----------|
| 키워드 의미로 사용하고 싶은 경우 | **반드시 모두 대문자** 로 표기 |
| 일반 영어로 사용하고 싶은 경우 | 소문자 또는 문장 첫 글자만 대문자로 표기 |
| 기존 문서 해석 시 | 문맥에 따라 판단 (RFC 8174 이전 문서는 모호할 수 있음) |

```
📌 문서 작성 팁:
   기술 문서를 작성할 때 "must", "should", "may" 등을
   일반 영어로 사용하려면 소문자로 작성하여 혼란을 방지하세요.

   예: "The reader should note that..." (일반 서술)
   예: "Implementations SHOULD cache responses..." (RFC 2119 키워드)
```

## 키워드별 구체적 사용 예시

각 키워드의 실제 프로토콜 문서에서의 사용 예시를 정리합니다.

### MUST / REQUIRED / SHALL

```
📋 DNS (RFC 1035):
   "UDP 메시지의 최대 크기는 512 옥텟이어야 한다(MUST)"

📋 HTTP/1.1 (RFC 7230):
   "HTTP 메시지는 CRLF로 줄을 구분해야 한다(MUST)"

📋 TLS 1.3 (RFC 8446):
   "서버는 지원하는 버전 목록에서 가장 높은 버전을 선택해야 한다(MUST)"

📋 JSON (RFC 8259):
   "JSON 텍스트는 UTF-8, UTF-16, 또는 UTF-32로 인코딩되어야 한다(SHALL)"
```

### MUST NOT / SHALL NOT

```
📋 HTTP/2 (RFC 7540):
   "클라이언트는 연결 프리페이스를 보내기 전에 다른 프레임을 보내서는 안 된다(MUST NOT)"

📋 WebSocket (RFC 6455):
   "클라이언트는 마스킹되지 않은 프레임을 보내서는 안 된다(MUST NOT)"

📋 OAuth 2.0 (RFC 6749):
   "클라이언트는 인가 코드를 두 번 이상 사용해서는 안 된다(MUST NOT)"
```

### SHOULD / RECOMMENDED

```
📋 HTTP/1.1 (RFC 7231):
   "서버는 5xx 응답에 에러 설명을 포함하는 것이 좋다(SHOULD)"

📋 SMTP (RFC 5321):
   "메일 서버는 최소 5분 동안 연결 재시도를 기다리는 것이 좋다(SHOULD)"

📋 TLS 1.3 (RFC 8446):
   "구현체는 forward secrecy를 제공하는 키 교환을 선호하는 것이 좋다(RECOMMENDED)"
```

### SHOULD NOT / NOT RECOMMENDED

```
📋 HTTP (RFC 7231):
   "클라이언트는 민감한 정보를 URI에 포함하지 않는 것이 좋다(SHOULD NOT)"

📋 REST API 설계:
   "DELETE 요청에 body를 포함하지 않는 것이 권장되지 않는다(NOT RECOMMENDED)"

📋 Cookie (RFC 6265):
   "Set-Cookie 헤더에서 Secure 속성 없이 민감한 쿠키를 설정하지 않는 것이 좋다(SHOULD NOT)"
```

### MAY / OPTIONAL

```
📋 HTTP/1.1 (RFC 7230):
   "클라이언트는 요청에 Accept 헤더를 포함할 수 있다(MAY)"

📋 DNS (RFC 1035):
   "리졸버는 응답을 캐시할 수 있다(MAY)"

📋 TLS 1.3 (RFC 8446):
   "클라이언트는 early data를 전송할 수 있다(MAY)"

📋 HTTP/2 (RFC 7540):
   "PRIORITY 프레임 지원은 선택적이다(OPTIONAL)"
```
