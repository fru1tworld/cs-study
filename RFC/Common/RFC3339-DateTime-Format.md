# RFC 3339: Date and Time on the Internet: Timestamps

> 인터넷에서 사용되는 날짜와 시간 형식: 타임스탬프

## 문서 정보

| 항목 | 내용 |
|------|------|
| **RFC 번호** | 3339 |
| **분류** | Proposed Standard |
| **작성자** | G. Klyne (Clearswift Corporation), C. Newman (Sun Microsystems) |
| **발행일** | 2002년 7월 |
| **상태** | 현행 표준 |

## 개요

RFC 3339는 인터넷 프로토콜에서 사용하기 위한 **ISO 8601 기반의 날짜/시간 형식 프로파일**을 정의합니다. 이 표준은 시스템 간 일관된 날짜/시간 표현을 보장하여 상호운용성 문제를 해결합니다.

이 문서의 핵심 목표는 **그레고리력**을 사용하여 날짜와 시간을 표현하는 **명확하고 정렬 가능한 형식**을 제공하는 것입니다.

---

## 핵심 정의

### 1. UTC (협정 세계시)

```
정의: 국제 도량형국(BIPM)에서 유지하는 시간 표준
```

**UTC**는 전 세계 시간 측정의 기준이 되는 시간 척도입니다. 원자 시계를 기반으로 하며, 지구 자전과의 오차를 보정하기 위해 **윤초(leap second)**가 추가됩니다.

---

### 2. 타임스탬프 (Timestamp)

**정의**: 특정 시점을 **모호하지 않게** 표현한 것

타임스탬프는 다음 요소를 포함합니다:
- 날짜 (년, 월, 일)
- 시간 (시, 분, 초)
- 시간대 오프셋

---

### 3. 시간대 (Timezone)

| 표기법 | 의미 | 예시 |
|--------|------|------|
| **Z** | UTC 오프셋 00:00 (Zulu time) | `2024-01-15T09:30:00Z` |
| **+HH:MM** | UTC보다 앞서는 시간대 | `+09:00` (한국 표준시) |
| **-HH:MM** | UTC보다 뒤처지는 시간대 | `-05:00` (미국 동부 표준시) |

---

### 4. 윤년 (Leap Year)

윤년 판별 규칙:

```
조건 1: 연도가 4로 나누어 떨어지면 윤년
조건 2: 단, 100으로 나누어 떨어지면 평년
조건 3: 단, 400으로 나누어 떨어지면 윤년
```

**예시**:

| 연도 | 윤년 여부 | 이유 |
|------|----------|------|
| 2024 | 윤년 | 4로 나누어 떨어짐 |
| 1900 | 평년 | 100으로 나누어 떨어지지만 400으로는 안 됨 |
| 2000 | 윤년 | 400으로 나누어 떨어짐 |

## 인터넷 날짜/시간 형식

### 기본 구조

RFC 3339의 날짜/시간 형식은 다음과 같이 구성됩니다:

```
date-time = full-date "T" full-time

full-date = YYYY-MM-DD
full-time = HH:MM:SS[.fraction]time-offset
```

### 구성 요소

| 요소 | 설명 | 자릿수 | 범위 |
|------|------|--------|------|
| **YYYY** | 연도 | 4자리 | 0000-9999 |
| **MM** | 월 | 2자리 | 01-12 |
| **DD** | 일 | 2자리 | 01-28/29/30/31 |
| **HH** | 시 | 2자리 | 00-23 |
| **MM** | 분 | 2자리 | 00-59 |
| **SS** | 초 | 2자리 | 00-59 (윤초 시 60) |
| **fraction** | 소수점 이하 초 | 선택적 | 임의 정밀도 |
| **time-offset** | 시간대 오프셋 | 필수 | Z 또는 ±HH:MM |

### ABNF 문법

```abnf
date-fullyear   = 4DIGIT
date-month      = 2DIGIT  ; 01-12
date-mday       = 2DIGIT  ; 01-28, 01-29, 01-30, 01-31 (월에 따라)
time-hour       = 2DIGIT  ; 00-23
time-minute     = 2DIGIT  ; 00-59
time-second     = 2DIGIT  ; 00-58, 00-59, 00-60 (윤초 허용)
time-secfrac    = "." 1*DIGIT
time-numoffset  = ("+" / "-") time-hour ":" time-minute
time-offset     = "Z" / time-numoffset

partial-time    = time-hour ":" time-minute ":" time-second [time-secfrac]
full-date       = date-fullyear "-" date-month "-" date-mday
full-time       = partial-time time-offset

date-time       = full-date "T" full-time
```

## 형식 예시

### 기본 예시

```
1985-04-12T23:20:50.52Z
```

**해석**:
- 날짜: 1985년 4월 12일
- 시간: 23시 20분 50.52초
- 시간대: UTC (Z)

---

### 시간대 오프셋 예시

```
1996-12-19T16:39:57-08:00
```

**해석**:
- 날짜: 1996년 12월 19일
- 시간: 16시 39분 57초
- 시간대: UTC-8 (미국 태평양 표준시)
- **UTC 기준**: 1996-12-20T00:39:57Z

---

### 윤초 예시

```
1990-12-31T23:59:60Z
```

**해석**:
- 날짜: 1990년 12월 31일
- 시간: 23시 59분 **60초** (윤초)
- 이 윤초는 1990년 말에 실제로 삽입됨

---

### 한국 표준시 (KST) 예시

```
2024-01-15T09:30:00+09:00
```

**해석**:
- 날짜: 2024년 1월 15일
- 시간: 오전 9시 30분
- 시간대: UTC+9 (한국 표준시)
- **UTC 기준**: 2024-01-15T00:30:00Z

## 핵심 요구사항

### 필수 사항 (MUST)

| 요구사항 | 설명 |
|----------|------|
| **4자리 연도** | 연도는 반드시 4자리여야 함 (2자리 연도 금지) |
| **시간대 필수** | 모든 타임스탬프에 시간대 정보 포함 필수 |
| **구분자 사용** | 날짜는 `-`, 시간은 `:`, 날짜와 시간 사이는 `T` 사용 |

### 금지 사항 (MUST NOT)

| 금지사항 | 설명 |
|----------|------|
| **2자리 연도** | Y2K 문제 방지를 위해 금지됨 |
| **24시 표기** | 시간 값은 00-23만 허용 (24:00 금지) |

### 허용 사항 (MAY)

| 허용사항 | 설명 |
|----------|------|
| **소문자 t, z** | 구분자 `T`와 UTC 표기 `Z`를 소문자로 사용 가능 |
| **소수점 이하 초** | 임의의 정밀도로 소수점 이하 초 표현 가능 |

## 윤초 (Leap Second)

### 개념

지구의 자전 속도는 일정하지 않습니다. UTC는 원자 시계 기반이므로, 지구 자전과의 오차를 보정하기 위해 **윤초**가 추가됩니다.

### 삽입 시점

윤초는 다음 시점에만 삽입될 수 있습니다:

```
6월 30일 23:59:60 UTC
또는
12월 31일 23:59:60 UTC
```

### 윤초 표현

```
1998-12-31T23:59:60Z   ← 윤초
1999-01-01T00:00:00Z   ← 다음 초
```

### 주의사항

- 대부분의 시스템은 윤초를 제대로 처리하지 못함
- NTP 등의 프로토콜로 동기화되는 시스템은 윤초 처리 구현이 필요함
- 윤초가 포함된 타임스탬프를 파싱할 때 59초로 처리하거나 별도 처리 필요

## 설계 원칙

RFC 3339 형식은 다음 원칙을 따릅니다:

### 1. 정렬 가능성 (Sortability)

```
✅ 문자열 정렬 = 시간순 정렬

예:
2024-01-01T00:00:00Z
2024-01-01T12:00:00Z
2024-01-02T00:00:00Z
```

같은 시간대의 타임스탬프는 문자열 정렬만으로 시간순 정렬이 가능합니다.

### 2. 가독성 (Human Readability)

```
RFC 3339: 2024-01-15T09:30:00+09:00
Unix 시간: 1705278600

→ RFC 3339 형식이 훨씬 가독성이 높음
```

### 3. 단순성 (Simplicity)

- ISO 8601의 복잡한 옵션들을 제거
- 인터넷 프로토콜에 필요한 최소한의 형식만 정의

### 4. 명확성 (Unambiguity)

```
❌ 모호한 형식: 01/02/2024 (미국식? 유럽식?)
✅ 명확한 형식: 2024-01-02 (항상 YYYY-MM-DD)
```

## ISO 8601과의 차이점

RFC 3339는 ISO 8601의 **프로파일**입니다. 주요 차이점:

| 항목 | ISO 8601 | RFC 3339 |
|------|----------|----------|
| **연도 범위** | 확장 가능 | 4자리 고정 |
| **기본 형식** | 구분자 생략 가능 (`YYYYMMDD`) | 구분자 필수 (`YYYY-MM-DD`) |
| **시간대** | 선택적 | 필수 |
| **주차 형식** | 지원 (`2024-W03`) | 미지원 |
| **서수일 형식** | 지원 (`2024-015`) | 미지원 |
| **시간 구분자** | `T`만 허용 | `T` 또는 `t` (또는 공백) |

## 구현 고려사항

### 파싱 (Parsing)

```python
# Python 예시
from datetime import datetime

# RFC 3339 파싱
dt = datetime.fromisoformat("2024-01-15T09:30:00+09:00")
```

### 포맷팅 (Formatting)

```python
# Python 예시
from datetime import datetime, timezone

dt = datetime.now(timezone.utc)
rfc3339 = dt.isoformat()  # "2024-01-15T00:30:00+00:00"
```

### 비교 (Comparison)

시간대가 다른 타임스탬프 비교 시 **반드시 UTC로 변환** 후 비교:

```
2024-01-15T09:30:00+09:00  (한국)
2024-01-15T00:30:00Z       (UTC)

→ 두 타임스탬프는 동일한 시점을 나타냄
```

## 일반적인 실수

### 1. 시간대 누락

```
❌ 2024-01-15T09:30:00        ← 시간대 없음
✅ 2024-01-15T09:30:00+09:00  ← 시간대 포함
✅ 2024-01-15T00:30:00Z       ← UTC 명시
```

### 2. 잘못된 구분자

```
❌ 2024/01/15 09:30:00  ← 슬래시, 공백
❌ 20240115T093000Z      ← 구분자 없음
✅ 2024-01-15T09:30:00Z  ← 올바른 형식
```

### 3. 잘못된 시간 범위

```
❌ 2024-01-15T24:00:00Z  ← 24시 금지
✅ 2024-01-16T00:00:00Z  ← 다음 날 00시로 표현
```

### 4. 2자리 연도 사용

```
❌ 24-01-15T09:30:00Z    ← 2자리 연도
✅ 2024-01-15T09:30:00Z  ← 4자리 연도
```

## 실제 활용 예시

### HTTP 헤더

```http
Date: Mon, 15 Jan 2024 00:30:00 GMT
Last-Modified: Sun, 14 Jan 2024 12:00:00 GMT
```

> 참고: HTTP 헤더는 RFC 3339가 아닌 RFC 7231의 IMF-fixdate 형식 사용

### JSON API

```json
{
  "created_at": "2024-01-15T09:30:00+09:00",
  "updated_at": "2024-01-15T10:45:30.123+09:00",
  "expires_at": "2024-01-22T09:30:00Z"
}
```

### 로그 파일

```
[2024-01-15T09:30:00.123+09:00] INFO: Server started
[2024-01-15T09:30:01.456+09:00] DEBUG: Connection established
[2024-01-15T09:30:02.789+09:00] ERROR: Connection timeout
```

### 데이터베이스

```sql
-- PostgreSQL
SELECT * FROM events
WHERE created_at >= '2024-01-15T00:00:00Z'
AND created_at < '2024-01-16T00:00:00Z';
```

## 보안 고려사항

### 1. 시간대 조작

악의적인 사용자가 시간대를 조작하여 타임스탬프를 위조할 수 있습니다:

```
❌ 클라이언트가 제공한 타임스탬프를 무조건 신뢰
✅ 서버에서 타임스탬프 생성 또는 검증
```

### 2. 윤초 처리

윤초가 포함된 타임스탬프(`23:59:60`)를 파싱하지 못하는 시스템은 서비스 거부(DoS) 공격에 취약할 수 있습니다.

### 3. 정밀도 공격

매우 긴 소수점 이하 초 값을 포함한 타임스탬프로 버퍼 오버플로우 공격 가능:

```
❌ 2024-01-15T09:30:00.123456789012345678901234567890Z
✅ 소수점 이하 자릿수 제한 구현
```
