## 1. Abstract (개요)

- **Origin 개념 정의**
  - 사용자 에이전트가 권한(authority)과 특권(privilege)의 범위로 사용하는 개념
  - 서로 다른 origin의 콘텐츠를 격리하여 악의적 간섭 방지

- **문서 범위**
  - Origin 원칙 설명
  - URI에서 origin을 결정하는 방법
  - Origin을 문자열로 직렬화하는 방법
  - 요청과 연관된 origin을 나타내는 HTTP "Origin" 헤더 필드 정의

---

## 2. Introduction (소개)

### 문제 배경
- 사용자 에이전트는 다양한 출처의 콘텐츠와 상호작용
  - 일부 콘텐츠는 악의적일 수 있음
- 악의적 간섭을 방지하기 위해 콘텐츠를 "origin"으로 구분

### 동일 출처 정책 (Same-Origin Policy)
- **동일 origin 콘텐츠**: 자유롭게 상호작용 가능
- **다른 origin 콘텐츠**: 제한적 상호작용

### 문서의 목적
- 동일 출처 정책의 원칙 설명
- Origin 비교 및 직렬화의 실용적 세부사항 제공
- 애플리케이션별 구현 세부사항은 다른 명세서(HTML, WebSocket 등)에서 다룸

## 4. Principles of the Same-Origin Policy (동일 출처 정책 원칙)

### 4.1 Trust (신뢰)

#### 신뢰 관계 정의
- 동일 출처 정책은 **URI로 신뢰를 명시**
- 문서가 URI를 통해 신뢰하는 스크립트와 폼 목적지를 지정
- 리소스를 import하거나 데이터를 전송하면 해당 리소스에 origin의 전체 권한 부여

#### 신뢰의 의미
- **라이브러리 import**: 해당 라이브러리의 무결성(integrity) 신뢰
- **데이터 전송**: 해당 목적지의 기밀성(confidentiality) 신뢰

#### 4.1.1 함정 (Pitfalls)
- 설계자는 중요한 신뢰 구분이 URI에 가시적으로 유지되도록 해야 함
- **HTTPS vs HTTP 예시**
  - "https" 사용 시 TLS로 보호된 검색 지정 가능
  - 능동적 네트워크 공격자로부터 보호
- URI scheme을 통해 보안 속성이 모호해지면 동일 출처 모델 약화

### 4.2 Origin (출처)

#### Origin의 필요성
- 모든 URI를 별도 보호 도메인으로 취급하면 개발자 부담 증가
- 웹 애플리케이션은 일반적으로 **여러 조정된 리소스로 구성**

#### Origin의 정의
- 사용자 에이전트는 URI를 "origin"으로 그룹화
- **Origin = 보호 도메인 (Protection Domain)**
- 두 URI가 동일 origin에 속하려면 다음이 **모두 동일**해야 함:

| 구성 요소 | 설명 | 예시 |
|----------|------|------|
| scheme | 프로토콜 | http, https |
| host | 호스트명 | example.com |
| port | 포트 번호 | 80, 443 |

#### Scheme을 포함하는 이유
- scheme 미포함 시 `http://example.com`과 `https://example.com`이 동일 origin
- 네트워크 공격자가 HTTP 콘텐츠를 손상시켜 HTTPS 보호 무력화 가능
- **보안을 위해 scheme 필수**

#### 완전 정규화 호스트명 사용 이유
- DNS는 계층적으로 위임하지만 신뢰 관계는 배포마다 다름
- 예: 교육기관
  - `example.edu/~student/` - 학생 콘텐츠
  - `grades.example.edu/` - 성적 시스템
  - 이들은 동일 origin이면 안 됨
- Origin 세분화는 보안 모델 발전의 역사적 산물

#### 4.2.1 Examples (예시)

**동일 Origin**

| URL | 설명 |
|-----|------|
| `http://example.com/` | 기준 URL |
| `http://example.com:80/` | 기본 포트 명시 (동일) |
| `http://example.com/path/file` | 경로 다름 (origin 동일) |

**다른 Origin**

| URL A | URL B | 이유 |
|-------|-------|------|
| `http://example.com/` | `http://example.com:8080/` | 포트 다름 |
| `http://example.com/` | `http://www.example.com/` | 호스트 다름 |
| `http://example.com/` | `https://example.com:80/` | scheme 다름 |
| `http://example.com/` | `https://example.com/` | scheme 다름 |
| `http://example.com/` | `http://example.org/` | 호스트 다름 |

### 4.3 Authority (권한)

#### 리소스별 권한 차이
- origin 내 모든 리소스가 동일한 권한을 갖지 않음
- **이미지**: 수동적, 권한 없음
- **HTML 문서**: 전체 origin 권한 보유

#### 권한 결정 방식
- 사용자 에이전트는 **미디어 타입**을 검사하여 권한 결정
- `image/png`: 최소 권한
- `text/html`: 전체 origin 접근 권한

#### 신뢰할 수 없는 콘텐츠 호스팅
- 웹 애플리케이션은 미디어 타입 제한으로 권한 제한 가능
- 그러나 HTML 문서에 신뢰할 수 없는 콘텐츠 포함 시 **XSS 취약점** 위험
  - Origin 권한 유출 가능

#### 4.3.1 함정 (Pitfalls)
- 새로운 웹 플랫폼 기능은 미디어 타입과 무관하게 권한 부여하면 안 됨
- 많은 애플리케이션이 제한된 타입으로 신뢰할 수 없는 콘텐츠 제공
- 그런 콘텐츠에 새 권한 부여 시 취약점 도입 위험

**권장 사항**
- 이미 전체 origin 권한을 가진 타입에 권한 부여
- 또는 해당 권한을 위해 새로 설계된 타입 사용

**콘텐츠 스니핑 (Content Sniffing) 위험**
- 제공된 것과 다른 미디어 타입으로 콘텐츠 처리
- 저권한 타입(이미지)에 고권한 타입(HTML) 권한 부여 위험
- 신중한 구현 필요

### 4.4 Policy (정책)

#### 일반 원칙
- 사용자 에이전트는 **다른 origin을 격리**
- **제어된 교차 origin 통신** 허용
- 구현 세부사항은 컨텍스트에 따라 크게 다름

#### 4.4.1 객체 접근 (Object Access)
- 대부분의 사용자 에이전트 객체(API)는 **동일 origin에만** 제공
- 한 URI의 콘텐츠가 다른 URI의 객체에 접근하려면:
  - 두 URI가 **동일 origin**이어야 함 (scheme, host, port 모두 동일)

**예외 사항**
- HTML의 `Location` 인터페이스: 네비게이션을 위한 제한적 교차 origin 접근 허용
- `postMessage`: 의도적으로 교차 origin 통신 가능

**주의**: 교차 origin 객체 노출은 위험하며 세심한 주의 필요

#### 4.4.2 네트워크 접근 (Network Access)

**읽기 제한**
- 다른 origin에서 읽기는 일반적으로 **금지**
- 예외: CORS(Cross-Origin Resource Sharing)로 옵트인 시 허용

**교차 origin 리소스 사용 허용**
- 스크립트 실행
- 이미지 렌더링
- 스타일시트 적용
- 외부 콘텐츠 표시 (프레임)

**쓰기 제한**
- 다른 origin으로 정보 전송은 **허용**
- 그러나 임의 형식 전송은 위험
- 사용자 에이전트는 문서를 특정 프로토콜로 제한
  - HTTP 요청 (커스텀 헤더 없이)
- 프로토콜 확장(WebSocket 등)은 신중한 보안 평가 필요

#### 4.4.3 함정 (Pitfalls)

**교차 origin 상호작용의 위험**
- 교차 origin 이미지 로딩: 높이, 너비 정보 유출
- 교차 origin 네트워크 요청: CSRF(Cross-Site Request Forgery) 취약점

**균형**
- 사용자 에이전트는 위험과 사용성 이점 사이 균형
- 교차 origin 요청 완전 차단 시 하이퍼링크 따라가기 불가능
  - 웹의 핵심 기능 상실

**새로운 기능 설계 권장사항**
- 동일 origin 내 일부 리소스에만 권한 부여/거부하면 안 됨
- 사용자 에이전트는 origin 내 리소스를 격리하지 않음
- **전체 origin 단위**로 권한 부여/거부해야 함

### 4.5 Conclusion (결론)

- 동일 출처 정책은 **URI로 신뢰 관계 지정**
- URI를 **origin(보호 도메인)**으로 그룹화
- Origin 내 일부 리소스(활성 콘텐츠)는 전체 권한 보유
- 다른 리소스(수동 콘텐츠)는 권한 없음
- 권한 보유 콘텐츠는 동일 origin 객체/리소스 접근 가능
- 제한적 교차 origin 접근은 취약점 방지를 위해 신중히 설계

## 5. Origin of a URI (URI의 Origin)

### Origin 결정 알고리즘

```
1. 비계층적 URI 처리
   - URI가 계층적 명명 권한이 없거나 절대 URI가 아닌 경우
   - 새로운 전역 고유 식별자 생성 후 반환
   - 참고: 동일 URI에 대한 여러 실행이 다른 값 생성 가능
   - 사용자 에이전트는 일반적으로 origin을 한 번 계산 후 재사용

2. Scheme 처리
   - uri-scheme = URI의 scheme 컴포넌트
   - 소문자로 변환

3. 미지원 프로토콜 처리
   - 구현이 uri-scheme을 지원하지 않으면
   - 새로운 전역 고유 식별자 생성 후 반환

4. File Scheme 처리
   - uri-scheme이 "file"인 경우
   - 구현 정의 값 반환 가능 (MAY)

   참고: File scheme 콘텐츠의 권한 처리
   - 역사적으로 file scheme 콘텐츠는 막대한 권한 보유
   - 모든 로컬 파일에 그런 권한 부여 시 권한 상승 공격 가능
   - 일부 에이전트: 디렉토리 기반 권한 부여
   - 일부 에이전트: 파일 URI마다 전역 고유 식별자 사용 (가장 안전)

5. Host 처리
   - uri-host = URI의 host 컴포넌트
   - i;ascii-casemap 대조를 사용하여 소문자로 변환

   참고: IDNA 처리 가정
   - 사용자 에이전트가 URI 구성 시 IDNA 처리/검증 수행 가정
   - uri-host는 A-label 변환을 통해 LDH 레이블만 포함
   - Origin 보안 정책은 IDNA 알고리즘에 크게 의존

6. Port 처리
   - 포트 컴포넌트가 없는 경우:
     - uri-port = uri-scheme의 기본 포트
   - 그렇지 않으면:
     - uri-port = 포트 컴포넌트

7. 반환
   - (uri-scheme, uri-host, uri-port) 트리플 반환
```

### Origin 결정 흐름도

```
URI 입력
    │
    ▼
┌─────────────────────────────────┐
│ 계층적 명명 권한 있고            │
│ 절대 URI인가?                    │
└─────────────────────────────────┘
    │ No              │ Yes
    ▼                 ▼
전역 고유        Scheme 소문자 변환
식별자 반환           │
                      ▼
              ┌─────────────────┐
              │ Scheme 지원?     │
              └─────────────────┘
                │ No        │ Yes
                ▼           ▼
           전역 고유    ┌─────────────┐
           식별자 반환  │ "file" scheme?│
                       └─────────────┘
                        │ Yes      │ No
                        ▼          ▼
                   구현 정의    Host, Port 추출
                   값 반환     (scheme, host, port) 반환
```

## 6. Comparing Origins (Origin 비교)

### 비교 규칙

#### Scheme/Host/Port 트리플 비교
- 두 origin이 **동일**하려면:
  - scheme이 동일
  - host가 동일
  - port가 동일

#### 전역 고유 식별자 비교
- 전역 고유 식별자는 scheme/host/port 트리플과 **일치할 수 없음**

### 동일 Origin 판단
- 두 URI가 **same-origin**이려면 해당 origin이 일치해야 함

### 특수 케이스

**Data URI의 origin**
- Data URI는 서버 기반 명명 권한이 없음
- 전역 고유 식별자 origin을 가짐
- **결과**: Data URI는 자기 자신과도 same-origin이 아님

| URI 유형 | Origin 유형 | 자기 자신과 same-origin? |
|----------|------------|-------------------------|
| HTTP/HTTPS URI | (scheme, host, port) 트리플 | Yes |
| Data URI | 전역 고유 식별자 | **No** |

### Origin 비교 예시

| URL A | URL B | 동일 Origin? | 이유 |
|-------|-------|-------------|------|
| `http://example.com/a` | `http://example.com/b` | Yes | scheme, host, port 모두 동일 |
| `http://example.com` | `http://example.com:80` | Yes | 80은 HTTP 기본 포트 |
| `http://example.com` | `https://example.com` | No | scheme 다름 (http vs https) |
| `http://example.com` | `http://example.com:8080` | No | port 다름 (80 vs 8080) |
| `http://example.com` | `http://www.example.com` | No | host 다름 |
| `data:text/html,...` | `data:text/html,...` | No | 각각 고유 식별자 |

## 7. Serializing Origins (Origin 직렬화)

### 7.1 Unicode 직렬화 (Unicode Serialization)

#### 알고리즘

```
1. Origin이 트리플이 아닌 경우
   → "null" 반환하고 중단

2. result = origin 트리플의 scheme 부분

3. result에 "://" 추가

4. host의 각 컴포넌트를 result에 추가 ("."로 구분):
   - 컴포넌트가 A-label인 경우:
     → 해당 U-label 사용
   - 그렇지 않으면:
     → 컴포넌트 그대로 사용

5. 포트가 scheme의 기본 포트와 다른 경우:
   → ":" 추가
   → 10진수로 포트 추가

6. result 반환
```

#### Unicode 직렬화 예시

| Origin | Unicode 직렬화 |
|--------|---------------|
| (http, example.com, 80) | `http://example.com` |
| (https, example.com, 443) | `https://example.com` |
| (http, example.com, 8080) | `http://example.com:8080` |
| (http, xn--nxasmq5b.com, 80) | `http://βόλος.com` |
| 전역 고유 식별자 | `null` |

### 7.2 ASCII 직렬화 (ASCII Serialization)

#### 알고리즘

```
1. Origin이 트리플이 아닌 경우
   → "null" 반환하고 중단

2. result = origin 트리플의 scheme 부분

3. result에 "://" 추가

4. host 부분을 result에 그대로 추가

5. 포트가 scheme의 기본 포트와 다른 경우:
   → ":" 추가
   → 10진수로 포트 추가

6. result 반환
```

#### ASCII 직렬화 예시

| Origin | ASCII 직렬화 |
|--------|-------------|
| (http, example.com, 80) | `http://example.com` |
| (https, example.com, 443) | `https://example.com` |
| (http, example.com, 8080) | `http://example.com:8080` |
| (http, xn--nxasmq5b.com, 80) | `http://xn--nxasmq5b.com` |
| 전역 고유 식별자 | `null` |

### Unicode vs ASCII 직렬화 비교

| 특징 | Unicode 직렬화 | ASCII 직렬화 |
|------|---------------|-------------|
| 국제화 도메인 | U-label (읽기 쉬움) | A-label (Punycode) |
| 사용 목적 | 사용자 표시용 | 프로토콜 전송용 |
| HTTP Origin 헤더 | 사용 안 함 | **사용** |

## 8. The HTTP Origin Header Field (HTTP Origin 헤더 필드)

### 8.1 Syntax (구문)

#### ABNF 정의

```abnf
origin              = "Origin:" OWS origin-list-or-null OWS
origin-list-or-null = %x6E %x75 %x6C %x6C / origin-list
                    ; "null" 또는 origin-list
origin-list         = serialized-origin *( SP serialized-origin )
serialized-origin   = scheme "://" host [ ":" port ]
```

- `scheme`, `host`, `port`는 RFC 3986 참조
- `OWS`: 선택적 공백 (Optional White Space)
- `SP`: 단일 공백

### 8.2 Semantics (의미론)

#### Origin 헤더의 목적
- HTTP 요청에 포함 시: 요청을 **"유발한" origin**을 나타냄
- 트리거 API에 의해 정의됨

#### 예시
- 스크립트 실행 컨텍스트가 HTTP 요청 유발
- 사용자 에이전트는 Origin으로 서버에 보안 컨텍스트 알림 가능 (MAY)

#### 다중 Origin
- 여러 origin이 요청에 기여한 경우
- 사용자 에이전트는 모든 origin을 나열 가능 (MAY)
- **예시**: 한 origin이 요청을 발행하고 다른 origin이 리다이렉트한 경우
  - 두 origin 모두 관련되었음을 서버에 알림 가능

### 8.3 User Agent Requirements (사용자 에이전트 요구사항)

#### 기본 요구사항

| 요구사항 | 수준 | 설명 |
|----------|------|------|
| HTTP 요청에 Origin 포함 | MAY | 어떤 HTTP 요청에든 포함 가능 |
| 단일 Origin 헤더 | MUST NOT | 하나의 요청에 둘 이상의 Origin 헤더 포함 금지 |
| 프라이버시 민감 컨텍스트 | MUST | "null" 전송 |

#### 프라이버시 민감 컨텍스트
- 이 문서는 프라이버시 민감 컨텍스트를 정의하지 않음
- 애플리케이션이 지정 가능

#### Origin 생성 규칙

1. **각 serialized-origin**
   - origin의 **ascii-serialization**이어야 함 (MUST)

2. **중복 방지**
   - 연속된 두 serialized-origin이 동일할 수 없음
   - 동일한 serialized-origin이 발생할 경우
   - 사용자 에이전트는 두 번째를 생성하면 안 됨 (MUST NOT)

### Origin 헤더 예시

```http
# 단일 Origin
Origin: http://example.com

# 비기본 포트
Origin: http://example.com:8080

# HTTPS
Origin: https://secure.example.com

# 다중 Origin (리다이렉트 체인)
Origin: http://example.com http://redirect.example.org

# 프라이버시 민감 컨텍스트
Origin: null
```

## 9. Security Considerations (보안 고려사항)

### 개요
- 동일 출처 정책은 **사용자 에이전트 보안의 기본**
- 특히 브라우저에서 중요
- 역사적으로 일부 에이전트가 taint tracking과 유출 방지 시도
  - 구현이 어려웠음
  - 최근 일부 접근법 부활에 관심

### 평가의 어려움
- Origin이 보안에서 핵심적 역할을 하기 때문에 평가 어려움
- Origin은 **불완전한 격리 단위**
- 여러 체계적 약점 존재

### 9.1 DNS 의존성 (Reliance on DNS)

#### 문제점
- 동일 출처 정책은 **DNS(Domain Name System)에 의존**
- 많은 URI scheme(http)이 DNS 기반 명명 권한 사용
- DNS 완전 또는 부분 손상 시 동일 출처 정책 보안 속성 실패

#### DNS 저항성 있는 Scheme

| Scheme | DNS 저항성 | 메커니즘 |
|--------|-----------|----------|
| http | 낮음 | DNS 의존 |
| https | 중간 | 인증서로 콘텐츠 소스 검증 |
| chrome-extension | 높음 | 공개키 기반 명명 권한, DNS 손상에 완전 안전 |

#### Origin 개념의 역할
- 다른 URI scheme의 격리는 **DNS 손상 영향 억제**에 필수적

### 9.2 분기된 격리 단위 (Divergent Units of Isolation)

#### 문제점
- 많은 기술(쿠키, RFC 6265)이 현대적 origin 개념보다 앞섬
- **다른 격리 단위** 사용으로 취약점 발생

#### "레지스트리 제어" 도메인 제안

일부에서 완전 정규화 도메인명 대신 "레지스트리 제어" 도메인 사용 제안
- 예: `www.example.com` 대신 `example.com`

#### 비권장 이유

| 이유 | 설명 |
|------|------|
| 정의적 어려움 | "레지스트리 제어" 도메인은 DNS 고유 속성이 아닌 인간의 DNS 관행의 함수 |
| 유지보수 어려움 | 공개 접미사 목록이 존재하지만 유지 어렵고 구현마다 다름 |
| 복잡한 사례 | 일본 지자체 등 DNS 계층 깊은 곳에서 공개 레지스트리 운영 |
| Scheme 비호환성 | 비DNS 명명 권한에 실패 |
| 공개키 Scheme | "레지스트리 제어 공개키"는 의미 없음 |
| 역순 Scheme | nntp(alt.usenet.kooks)처럼 역순 점 표기 위임 사용 |

**결론**: "레지스트리 제어" 도메인은 URI-scheme 및 구현 특정적
- Scheme과 구현 간 차이로 취약점 위험

### 9.3 주변 권한 (Ambient Authority)

#### 문제점
- 동일 출처 정책은 **콘텐츠 URI에 기반**하여 권한 부여
- 지정 가능한 객체가 아님
- 권한-지정 분리는 **주변 권한(ambient authority)**의 예
- 취약점 발생

#### XSS 예시
- HTML 문서의 **크로스 사이트 스크립팅(XSS)** 취약점
- 공격자가 스크립트 콘텐츠 주입 시
  - 스크립트가 문서의 origin 권한으로 실행
  - 민감한 정보 접근 가능

#### 권한 설계 개선 방향
- 스크립트 권한이 지정 가능한 객체로 제한되었다면
- 주입 공격이 이점을 제공하지 않았을 것

### 9.4 IDNA 의존성 및 마이그레이션 (IDNA Dependency and Migration)

#### 문제점
- 동일 출처 보안은 **IDNA 알고리즘 세부사항에 크게 의존**
- 사용자 에이전트마다 국제화 도메인명 매핑이 다를 수 있음

#### IDNA 버전 차이

| 버전 | RFC | 예시 문자 U+00DF 처리 |
|------|-----|----------------------|
| IDNA2003 | RFC 3490 | 다른 처리 |
| IDNA2008 | RFC 5890 | 다른 처리 |

#### 마이그레이션 위험
- IDNA 알고리즘 간 마이그레이션은 **보안 경계 재설정**
- 가능한 결과:
  - 새로운 경계 생성
  - 기존 경계 제거
- **위험**: 상호 불신하는 엔티티 간 경계 변경
- 불신 엔티티가 동일 origin으로 결합되면 **공격 가능**
