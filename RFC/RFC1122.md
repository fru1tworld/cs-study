## 1. INTRODUCTION (소개)

### 1.1 Internet Architecture (인터넷 아키텍처)

#### 1.1.1 Internet Hosts (인터넷 호스트)
- ** 호스트 정의**: 통신 서비스의 최종 소비자
  - 사용자를 대신하여 애플리케이션 실행
  - 네트워크 통신 프로토콜 사용
- ** 호스트 범위**: 마이크로프로세서부터 슈퍼컴퓨터까지
- ** 기능 범위**: 단일 목적 시스템부터 원격 로그인, 파일 전송, 이메일 지원 전체 서비스 플랫폼까지
- ** 멀티홈드 호스트(Multihomed Host)**: 네트워크에 대한 다중 인터페이스를 가진 호스트

#### 1.1.2 Architectural Assumptions (아키텍처 가정)

인터넷 호스트 설계의 4가지 기본 가정:

| 가정 | 설명 |
|------|------|
| **(a) 네트워크들의 네트워크** | 각 호스트는 특정 네트워크에 직접 연결됨. 동일 네트워크의 두 호스트와 원거리 네트워크 간 통신에 동일한 프로토콜 사용 |
| **(b) 상태 비저장 게이트웨이** | 게이트웨이는 상태를 유지하지 않고 데이터그램을 독립적으로 전달. 장애 시에도 중복 경로 활용 가능 |
| **(c) 게이트웨이의 라우팅 복잡성** | 라우팅 복잡성은 게이트웨이에 위치. 호스트 소프트웨어를 라우팅 아키텍처 진화로부터 보호 |
| **(d) 광범위한 네트워크 허용** | 인터넷은 다양한 대역폭, 지연, 패킷 손실, 재정렬, 패킷 크기를 수용하면서 개별 구성요소 장애에 대한 견고성 유지 |

#### 1.1.3 Internet Protocol Suite (인터넷 프로토콜 스위트)

계층화된 프로토콜 구조:

```
┌─────────────────────────────────────────────┐
│  Application Layer (애플리케이션 계층)        │
│  - 사용자 프로토콜: Telnet, FTP, SMTP        │
│  - 지원 프로토콜: SNMP, BOOTP, DNS          │
├─────────────────────────────────────────────┤
│  Transport Layer (전송 계층)                 │
│  - TCP: 신뢰성 있는 연결 지향 서비스         │
│  - UDP: 비연결형 데이터그램 서비스           │
├─────────────────────────────────────────────┤
│  Internet Layer (인터넷 계층)                │
│  - IP: 비연결형 데이터그램 서비스            │
│  - ICMP: 오류/제어                          │
│  - IGMP: 멀티캐스트 그룹 관리               │
├─────────────────────────────────────────────┤
│  Link Layer (링크 계층)                      │
│  - 네트워크별 미디어 액세스 프로토콜         │
└─────────────────────────────────────────────┘
```

#### 1.1.4 Embedded Gateway Code (내장 게이트웨이 코드)
- 내장 게이트웨이 기능이 있는 호스트:
  - 게이트웨이 기능: Gateway Requirements RFC 준수
  - 호스트 기능: 본 문서 준수
- ** 장점**: 편의성, 멀티홈드 시나리오에서 최적 라우팅 결정
- ** 단점**: 라우팅 프로토콜 진화 추적 복잡성
- ** 중요 요구사항**:
  - 관리자가 게이트웨이 활성화에 대한 의식적 제어 유지
  - 구성 스위치를 통한 제어
  - ** 기본값: 비-게이트웨이 모드**

### 1.2 General Considerations (일반 고려사항)

#### 1.2.1 Continuing Internet Evolution (지속적인 인터넷 진화)
- 인터넷의 엄청난 성장으로 관리 및 확장 과제 발생
- 벤더 구현은 지속적인 프로토콜 진화 수용 필요
- 구현 후 유지보수 포기 시 고객 불만족이 기술 커뮤니티에 빠르게 전파

#### 1.2.2 Robustness Principle (강건성 원칙)

> **"받는 것에는 관대하게, 보내는 것에는 보수적으로"**
>
> *"Be liberal in what you accept, and conservative in what you send."*

- 모든 가능한 오류 처리 필수
- ** 가정**: "네트워크는 악의적인 개체들로 가득 차 있다"
- 프로토콜 열거를 불완전한 것으로 취급
  - 예: 4개의 오류 코드가 정의되어 있어도 5번째를 만나면 실패하면 안 됨
- ** 결과**: 오작동 호스트 감시 및 방해 용량 제한

#### 1.2.3 Error Logging (오류 로깅)
- 인터넷 복잡성으로 신중한 진단 로깅 필요
- ** 구현 요구사항**:
  - 잘못된 프로토콜 이벤트를 최대 진단 세부 정보와 함께 기록
  - 오류 발생 시 패킷 헤더 포함
  - 순환 로그 또는 선택적 활성화를 통한 오버플로 방지
  - 관리 프로토콜을 통해 접근 가능한 이상 현상 카운트
  - 선택적 로깅 지원 (예: "호스트 X의 모든 것을 로깅")
- ** 관리 철학**:
  - 일부: 무해한 이벤트 무시 선호
  - 일부: 적극적인 이상 감지 및 제거 요구

#### 1.2.4 Configuration (구성)
- 완전한 자체 구성은 달성되지 않음
- ** 구성 요구사항 이유**:
  - 최적값에 대한 현재 불확실성으로 향후 업데이트 필요
  - 값이 외부 요인에 의존 (호스트 크기, 통신 부하, 네트워크 속도)
  - 구식 또는 잘못된 프로토콜 구현이 지속됨
    - 때때로 올바른 시스템의 "잘못된 구성"이 공존을 위해 필요
- ** 권장사항**:
  - 재컴파일 없이 재정의 가능한 기본값 설정
  - 구성 파일 재정의 기능 제공
  - 기본값은 공식 프로토콜 구현 (결함 시스템 수용이 아님)

### 1.3 Reading this Document (문서 읽기)

#### 1.3.1 Organization (구성)
- 프로토콜 계층별로 구성: 링크 계층, 인터넷 계층, 전송 계층
- 엄격한 계층화는 프로토콜과 구현 방식을 불완전하게 모델링
- 프로토콜은 계층 간에 복잡하게 상호작용

** 주요 섹션 구조**:
1. Introduction (소개)
2. Protocol Walk-Through (프로토콜 상세 검토 - 수정/명확화 포함)
3. Specific Issues (설계 및 구현 주제)
4. Interfaces (다음 계층에 대한 서비스 인터페이스)
5. Summary (요구사항 색인)

#### 1.3.2 Requirements (요구사항)

대문자 키워드의 요구사항 의미:

| 키워드 | 의미 |
|--------|------|
| **MUST / REQUIRED** | 절대적인 명세 요구사항 |
| **SHOULD / RECOMMENDED** | 무시할 타당한 이유가 있을 수 있지만, 전체 영향을 신중히 평가 필요 |
| **MAY / OPTIONAL** | 진정한 선택사항. 벤더가 시장 또는 향상 고려에 따라 선택 |

** 준수(Compliance)**:
- MUST 요구사항 미충족 구현: ** 비준수(non-compliant)**
- 모든 MUST + SHOULD 충족: ** 무조건 준수(unconditionally compliant)**
- MUST 충족, 일부 SHOULD 미충족: ** 조건부 준수(conditionally compliant)**

#### 1.3.3 Terminology (용어)

| 용어 | 정의 |
|------|------|
| **Segment (세그먼트)** | TCP 프로토콜 전송 단위 (TCP 헤더 + 애플리케이션 데이터) |
| **Message (메시지)** | 전송 계층 전송 단위 |
| **IP Datagram (IP 데이터그램)** | 인터넷 계층 전송 단위 (IP 헤더 + 전송 데이터) |
| **Packet (패킷)** | IP 계층/링크 계층 인터페이스 단위 (완전한 데이터그램 또는 단편) |
| **Frame (프레임)** | 링크 계층 전송 단위 (링크 헤더 + 패킷) |
| **Connected Network (연결된 네트워크)** | 호스트가 인터페이스하는 네트워크 (로컬/서브네트워크) |
| **Multihomed (멀티홈드)** | 다중 IP 주소를 가진 호스트 |
| **Physical network interface** | 고유 링크 계층 주소를 가진 물리적 네트워크 연결 |
| **Logical interface (논리적 인터페이스)** | 고유 IP 주소로 구분되는 논리적 경로 |
| **Specific-destination address** | 효과적인 데이터그램 목적지 (브로드캐스트/멀티캐스트의 경우 로컬 IP 주소) |
| **Path (경로)** | 데이터그램이 통과하는 단방향 게이트웨이 시퀀스 |
| **MTU** | 최대 전송 단위 크기 |

## 2. LINK LAYER (링크 계층)

### 2.1 Introduction (소개)
- 모든 인터넷 시스템(호스트와 게이트웨이)은 동일한 링크 계층 요구사항 공유
- 게이트웨이 요구사항 문서에 명시
- 본 섹션에서 추가 자료 보충

### 2.2 Protocol Walk-Through (프로토콜 상세 검토)
- 상세 내용 없음 (게이트웨이 요구사항 참조)

### 2.3 Specific Issues (특정 이슈)

#### 2.3.1 Trailer Protocol Negotiation (트레일러 프로토콜 협상)

| 요구사항 | 레벨 |
|----------|------|
| 트레일러 프로토콜 사용 | MAY |
| 양 시스템이 구현 확인 후에만 사용 | MUST |
| 동적 협상 미수행 시 기본적으로 비활성화 | MUST |

** 근거**:
- 트레일러는 패킷 데이터 재배열로 때때로 복사 오버헤드 감소를 통해 처리량 향상
- 부적절한 사용 시 혼란스러운 증상 발생
- 특정 크기의 패킷만 트레일러 캡슐화를 받음
- 한 시스템만 트레일러 사용 시 선택적 패킷 손실 발생

** 구현 접근법** (Ethernet):
- ARP 교환이 정상적으로 완료
- 트레일러 지원 호스트가 추가 "트레일러 ARP 응답" 패킷 전송
- 요청 또는 응답 호스트 모두 트레일러 수신 요청 가능

#### 2.3.2 Address Resolution Protocol -- ARP

##### 2.3.2.1 ARP Cache Validation (ARP 캐시 검증)

| 요구사항 | 레벨 |
|----------|------|
| 오래된 캐시 항목 플러시 메커니즘 제공 (타임아웃 또는 기타) | MUST |
| ARP 플러딩 방지 (권장: 목적지당 초당 최대 1 요청) | MUST |
| 구성 가능한 타임아웃 | SHOULD |

** 근거**:
- 프록시 ARP 보급으로 잘못된 캐시 항목 가능성 크게 증가
- 어떤 메커니즘이든 이제 필요
- 프록시 ARP 없이도 타임아웃이 잘못 캐시된 데이터 무효화

** 메커니즘**:

| 메커니즘 | 설명 |
|----------|------|
| ** 타임아웃** | 주기적으로 항목 무효화 (사용 중이어도). 브로드캐스트 관찰로 새로 고침 시 타임아웃 재시작. 순서: 분 단위. 프록시 ARP에 유용 |
| ** 유니캐스트 폴링** | 점대점 ARP 요청으로 능동적 폴링. N회 연속 실패 후 삭제. 일반적 N=2, 분 단위 타임아웃 |
| ** 링크 계층 조언** | 링크 계층 전달 문제 시 항목 플러시 |
| ** 상위 계층 조언** | 전송 계층 호출이 전달 문제 표시 시 해당 항목 무효화 |

** 트레이드오프**: 대규모 Ethernet에서 분 단위 타임아웃은 눈에 띄는 트래픽 오버헤드 생성. 구성으로 타임아웃 연장 가능

##### 2.3.2.2 ARP Packet Queue (ARP 패킷 큐)

| 요구사항 | 레벨 |
|----------|------|
| 미해결 IP 주소로 향하는 최소 1개 (최신) 패킷 저장 | SHOULD |
| 해결 시 전송 | SHOULD |

** 근거**:
- 실패 시 모든 교환에서 첫 번째 패킷 손실 발생
- 상위 계층이 일반적으로 재전송으로 대처하지만 성능 저하:
  - TCP 열기 요청 손실 시 초기 RTT 추정치 팽창
  - UDP 애플리케이션 (DNS)은 더 심각하게 영향 받음

#### 2.3.3 Ethernet and IEEE 802 Encapsulation (Ethernet 및 IEEE 802 캡슐화)

요구사항은 RFC-894 (Ethernet) 및 RFC-1042 (IEEE 802) 명세 준수:

| 기능 | MUST | SHOULD | MAY | MUST NOT |
|------|------|--------|-----|----------|
| RFC-894 캡슐화 송신/수신 | ✓ | | | |
| RFC-1042 캡슐화 수신 (RFC-894와 혼합) | | ✓ | | |
| RFC-1042 캡슐화 송신 | | | ✓ | |
| 양 캡슐화 구현 시 구성 가능한 선택 스위치 (기본값: RFC-894) | ✓ | | | |
| K1=6 인코딩 사용 802 패킷 송신 | | | | ✓ |
| 주소 변환을 위한 ARP 사용 | ✓ | | | |

** 표준 준수**:
- K1=6 인코딩은 IEEE가 IP용으로 예약 - 사용 금지
- RFC-1042는 SNAP 확장과 함께 K1=170 사용

**MTU 값**:
- Ethernet: 1500
- IEEE 802.3: 1492

** 논의**:
- IEEE 802.3 명세는 10Mbps Ethernet 케이블 운용 허용 (프레임이 물리적으로 혼합)
- 수신자는 Length 필드로 구분 (802.3에서 ≤1500, Ether-Type에서 >1500)
- 브로드캐스트 프레이밍 호환성 문제:
  - 한 프레이밍으로 전송된 브로드캐스트는 다른 프레이밍만 수신하는 호스트에서 누락
- 조항들은 894-capable과 1042-capable 시스템 간 직접 상호운용 지원
- 894-only와 1042-only 시스템은 IP 게이트웨이 없이 직접 상호운용 불가

### 2.4 Link/Internet Layer Interface (링크/인터넷 계층 인터페이스)

| 인터페이스 | 요구사항 | 레벨 |
|------------|----------|------|
| ** 수신** | 들어오는 패킷의 링크 계층 브로드캐스트 주소 상태를 나타내는 패킷 플래그 | MUST |
| ** 송신** | 5비트 TOS 필드 포함 | MUST |
| ** 오류 보고** | ARP 캐시 항목 부재만으로 Destination Unreachable 보고 금지 | MUST NOT |

** 근거** (수신):

- 브로드캐스트 스톰 방지와 관련 (인터넷 계층 섹션 참조)

## 3. INTERNET LAYER PROTOCOLS (인터넷 계층 프로토콜)

### 3.1 Introduction (소개)

** 강건성 원칙의 중요성**: 여기서 "특히 중요" - 하나의 오작동 호스트가 많은 다른 호스트에 서비스 거부 가능

** 표준**:
- **RFC-791**: IP 프로토콜 및 아키텍처 소개
- **RFC-792**: ICMP (라우팅, 진단, 오류 기능)
- **RFC-950**: 필수 서브넷 주소 지정 확장
- **RFC-1112**: IGMP (멀티캐스트 그룹 관리 확장)

** 핵심 요구사항**:
- 호스트 IP 계층은 **IP와 ICMP 모두 구현 MUST**

** 기본 기능**:
1. 나가는 데이터그램에 대한 "다음 홉" 게이트웨이/호스트 선택
2. 들어오는 데이터그램 재조립
3. (선택적) 의도적 단편화 구현
4. 진단 및 오류 기능 제공

** 처리 흐름**:

| 방향 | 단계 |
|------|------|
| ** 들어오는 데이터그램** | 1. 올바른 형식 검증<br>2. 로컬 목적지 검증<br>3. 옵션 처리<br>4. 필요 시 재조립<br>5. 적절한 전송 모듈로 전달 |
| ** 나가는 데이터그램** | 1. 전송 계층이 설정하지 않은 필드 설정<br>2. 올바른 첫 번째 홉 선택 (라우팅)<br>3. 필요/구현 시 단편화<br>4. 링크 계층 드라이버로 전달 |

** 멀티호밍 고려사항**:
- 멀티홈드 호스트는 다중 IP 주소를 가짐 → 상당한 복잡성 도입
- ** 두 가지 구분되는 문제**:
  1. ** 로컬 멀티호밍**: 호스트 자체가 멀티홈드
  2. ** 원격 멀티호밍**: 로컬 호스트가 원격 멀티홈드 호스트와 통신
- 원격 멀티호밍: ** 애플리케이션 계층에서 처리 MUST**
- 호스트: 로컬 멀티호밍 지원 **MAY** (섹션 3.3.4 참조)

** 게이트웨이 기능**:

| 요구사항 | 레벨 |
|----------|------|
| 다른 호스트의 데이터그램 전달 시 게이트웨이 요구사항 충족 | MUST |
| 내장 게이트웨이 코드의 비활성화 구성 스위치 보유 | MUST |
| 기본값: 비-게이트웨이 모드 | MUST |
| 다중 인터페이스 감지 시 게이트웨이 모드 자동 활성화 금지 | MUST NOT |

** 묵시적 폐기 (Silent Discard)**:
- ** 정의**: 추가 처리나 ICMP 오류 메시지 전송 없이 데이터그램 제거
- ** 그러나**: 오류 로깅 및 이벤트 통계 기록 **SHOULD**
- ** 목적**: 브로드캐스트 스톰 방지

### 3.2 Protocol Walk-Through (프로토콜 상세 검토)

#### 3.2.1 Internet Protocol -- IP

##### 3.2.1.1 Version Number (버전 번호)

| 요구사항 | 레벨 |
|----------|------|
| 버전 번호 ≠ 4인 데이터그램 묵시적 폐기 | MUST |

##### 3.2.1.2 Checksum (체크섬)

| 요구사항 | 레벨 |
|----------|------|
| 수신된 모든 데이터그램에서 IP 헤더 체크섬 검증 | MUST |
| 검증 실패 체크섬 묵시적 폐기 | MUST |

##### 3.2.1.3 Addressing (주소 지정)

**5개의 IP 주소 클래스**:
- **Class A, B, C**: 일반 주소
- **Class D**: IP 멀티캐스팅 지원 (28비트 논리적 그룹 주소)
  - 영구적 또는 일시적
  - 영구 할당: Internet Assigned Number Authority
  - 일시적 주소: 동적 할당
  - IGMP가 그룹 멤버십 동적 결정
- **Class E**: 실험용

** 특수 주소 케이스** (표기법: {Network-number, Host-number} 또는 {Network-number, Subnet-number, Host-number}; "-1"은 모든 1 비트 필드):

| 주소 | 설명 | 송신 가능 | 소스 주소 |
|------|------|-----------|-----------|
| **(a) {0,0}** | 이 네트워크의 이 호스트 | 초기화 시에만 (IP 주소 학습 중) | 초기화 시에만 |
| **(b) {0, Host-number}** | 이 네트워크의 지정된 호스트 | 초기화 시에만 | 초기화 시에만 |
| **(c) {-1,-1}** | 제한된 브로드캐스트 - 연결된 물리적 네트워크의 모든 호스트가 수신 | 외부 전달 MUST NOT | MUST NOT |
| **(d) {Network-number, -1}** | 지정된 네트워크로의 직접 브로드캐스트 | - | MUST NOT |
| **(e) {Network-number, Subnet-number, -1}** | 지정된 서브넷으로의 직접 브로드캐스트 | - | MUST NOT |
| **(f) {Network-number, -1, -1}** | 모든 서브넷으로의 직접 브로드캐스트 | - | MUST NOT |
| **(g) {127, any}** | 내부 호스트 루프백 | 호스트 외부 노출 MUST NOT | - |

**Network-number 값**:
- 관리적으로 할당되어 전 세계적 고유성 보장
- IP 주소는 Host-number, Network-number, Subnet-number 필드에 0 또는 -1 금지 (특수 케이스 제외)
- 최소 2비트 필드 길이 암시

** 서브넷 지원**:

| 요구사항 | 레벨 |
|----------|------|
| 서브넷 확장 지원 (RFC-950) | MUST |
| 각 로컬 IP 주소와 주소 마스크 {-1,-1,0} 연결 | MUST |

** 소스 주소 요구사항**:

| 요구사항 | 레벨 |
|----------|------|
| 데이터그램 송신 시 IP 소스 주소는 호스트 자신의 IP 주소여야 함 (브로드캐스트/멀티캐스트 아님) | MUST |

** 들어오는 데이터그램 필터링**:

| 요구사항 | 레벨 |
|----------|------|
| 호스트로 향하지 않는 들어오는 데이터그램 묵시적 폐기 | MUST |

** 데이터그램이 목적지로 향하는 경우**:
1. 목적지 주소가 호스트 IP 주소와 같음; 또는
2. 목적지 주소가 연결된 네트워크에 유효한 IP 브로드캐스트; 또는
3. 목적지 주소가 호스트가 들어오는 물리적 인터페이스에서 멤버십을 유지하는 멀티캐스트 그룹

**Specific-destination address (특정 목적지 주소)**:
- 브로드캐스트/멀티캐스트 데이터그램의 경우: 동등한 로컬 IP 주소
- 헤더에 브로드캐스트/멀티캐스트 주소가 있으면: specific-destination = 수신 물리적 인터페이스의 IP 주소

** 소스 주소 검증**:

| 요구사항 | 레벨 |
|----------|------|
| 본 섹션 규칙에 따라 잘못된 소스 주소를 가진 들어오는 데이터그램 묵시적 폐기 | MUST |
| 검증은 IP 또는 전송 계층에서 발생 | - |

** 논의**:
- 잘못 주소 지정된 데이터그램: 유니캐스트 데이터그램의 링크 계층 브로드캐스트 또는 혼란/잘못 구성된 게이트웨이/호스트에서 발생
- 아키텍처 목표: 특징 없는 32비트 IP 주소 허용 (주소 형식 지식을 요구하는 알고리즘 회피)
  - 향후 형식 변경 시 호스트 소프트웨어 변경 불필요
- 그러나 브로드캐스트/멀티캐스트 검증이 이 목표 위반
- 모든 서브넷 직접 브로드캐스트 주소 (f)에 의존하는 애플리케이션: 사용 불가능할 수 있음
  - 광범위한 벤더 게이트웨이 구현이 제한적
  - 구현되어도 특정 네트워크 관리가 비활성화할 수 있음

##### 3.2.1.4 Fragmentation and Reassembly (단편화 및 재조립)
- 인터넷 모델: 모든 호스트가 재조립 지원 요구
- 단편화/재조립 요구사항: 섹션 3.3.2 및 3.3.3 참조

##### 3.2.1.5 Identification (식별)

| 요구사항 | 레벨 |
|----------|------|
| 동일한 이전 데이터그램 복사본 송신 시 동일한 Identification 필드 값 유지 | MAY |

** 논의**:
- 일부 전문가: 동일 복사본은 원래 Identification 값 유지해야 함
- 제안된 장점:
  1. 원본 및 복사본 단편으로부터 재조립 갭 채우기
  2. 혼잡한 게이트웨이가 Identification 필드와 Fragment Offset으로 중복 데이터그램 폐기 가능
- 관찰된 인터넷 데이터그램 손실 패턴: 재전송된 단편 재조립 갭 채우기 확률 불리
- 다른 메커니즘 (재전송 시 TCP 재패키징): 동일 데이터그램 재전송 방지
- UDP 같은 비연결 전송 프로토콜: Identification 값 유지를 위한 애플리케이션 협력 필요
- ** 결론**: 동일 Identification 재전송은 도움이 되지 않는 것으로 보임

##### 3.2.1.6 Type-of-Service (서비스 유형)

**TOS 바이트 구조**:
- **Precedence 필드**: 상위 3비트 - 국방부 애플리케이션 대상, 본 문서 범위 외
- **TOS 필드**: 하위 5비트 - 문서에서 "TOS" 언급 시 이 부분만 지칭

| 요구사항 | 레벨 |
|----------|------|
| IP 계층이 전송 계층에 모든 송신 데이터그램의 TOS 필드 설정 기능 제공 (기본값: 모든 0 비트) | MUST |
| 수신된 TOS 값을 전송 계층으로 전달 | SHOULD |
| 링크 계층 TOS 매핑 (RFC-795) 구현 | SHOULD NOT |

** 논의**:
- 역사적으로 활용도 낮았으나, TOS 필드는 향후 역할 증가 예상
- 게이트웨이 라우팅 및 큐잉 알고리즘 제어
- TOS가 링크 계층 서비스 선택자로 매핑될 수 있음 (직렬 회선을 TCP 트래픽 클래스 간 효과적으로 공유)
- RFC-795 매핑 (1981 인터넷에 포함된 네트워크용): 현재 구식

##### 3.2.1.7 Time-to-Live (생존 시간)

| 요구사항 | 레벨 |
|----------|------|
| TTL 값 0인 데이터그램 송신 금지 | MUST NOT |
| TTL < 2로 수신된 데이터그램을 단지 그 이유만으로 폐기 금지 | MUST NOT |
| IP 계층이 전송 계층에 모든 송신 데이터그램의 TTL 필드 설정 기능 제공 | MUST |
| 고정 TTL 값은 구성 가능해야 함 | MUST |

** 논의**:
- **TTL의 이중 기능**:
  1. TCP 세그먼트 수명 제한
  2. 인터넷 라우팅 루프 종료
- TTL은 초 단위 시간을 나타내지만 홉 카운트 속성 보임 (게이트웨이가 TTL을 최소 1씩 감소)
- ** 의도**: TTL 만료가 게이트웨이에서 데이터그램을 폐기하지만 목적지 호스트에서는 아님
- 데이터그램 전달 호스트: 게이트웨이 TTL 규칙 준수 필요
- 상위 계층 프로토콜: "확장 범위" 리소스 검색 구현을 위해 TTL 설정 가능
  - 진단 도구 및 IP 멀티캐스팅을 통한 가장 가까운 서버 위치 찾기에 유용
- 고정 TTL 값: 인터넷 "직경" (가능한 가장 긴 경로) 초과 필요
- ** 합리적 값**: 직경의 약 2배 (지속적 성장 수용)

##### 3.2.1.8 Options (옵션)

| 요구사항 | 레벨 |
|----------|------|
| 전송 계층이 전송된 데이터그램에 IP 옵션 지정 수단 보유 (섹션 3.4 참조) | MUST |
| 수신된 모든 IP 옵션 (NOP/END-OF-LIST 제외)을 전송 계층 또는 ICMP 처리로 전달 | MUST |
| IP 및 전송 계층이 이해하는 옵션 해석, 나머지는 묵시적 무시 | MUST |

** 논의**:
- 수신된 모든 옵션 전달: 의도적으로 "엄격한 계층화" 위반
- 향후 전송 관련 IP 옵션 도입 용이
- 각 계층이 관련 옵션 추출, 나머지 무시
- 문서는 옵션 길이 명세 (NOP/END-OF-LIST 제외)를 지정하지만 다중 옵션 처리 순서는 미지정
- 다중 옵션 수신 시 소스 라우트 옵션과 결합된 모호성 의미 인정 필요

** 구현**:
- IP 계층은 가능한 범위 외의 옵션 길이 오류로 충돌하면 안 됨
- 잘못된 길이가 일부 구현에서 무한 루프 유발

** 특정 옵션 요구사항**:

| 옵션 | 설명 |
|------|------|
| **(a) Security Option** | 일부 환경에서 모든 데이터그램에 Security 옵션 요구 (본 문서 범위 및 IP 명세 외). RFC-791 및 RFC-1038 보안 옵션은 구식. DoD 애플리케이션은 가이드 문서 참조 필요 |
| **(b) Stream Identifier Option** | 상세 내용은 별도 문서 참조 |
| **(c) Source Route Option** | 섹션 3.3.5 참조 |
| **(d) Record Route Option** | 선택적 구현 |
| **(e) Timestamp Option** | 선택적 구현 |

#### 3.2.2 Internet Control Message Protocol -- ICMP

**ICMP 개요**:
- 라우팅, 진단, 오류 기능 제공
- 아키텍처적으로 IP 위에 계층화됨 (TCP/UDP처럼 IP를 사용하여 종단 간 데이터 전달)
- 그러나 ICMP 처리는 IP 계층 기능의 필수 부분

##### 3.2.2.1 Destination Unreachable (목적지 도달 불가)

| 요구사항 | 레벨 |
|----------|------|
| 전송 프로토콜이 지원되지 않을 때 코드 2 (Protocol Unreachable) 생성 | MUST |
| UDP가 데이터그램을 역다중화할 수 없을 때 코드 3 (Port Unreachable) 생성 | MUST |
| 수신된 도달 불가 메시지를 전송 계층에 보고 | MUST |

** 코드별 의미**:

| 코드 | 의미 | 비고 |
|------|------|------|
| 0 | Net Unreachable | 도달 불가능의 힌트일 뿐, 증거 아님 |
| 1 | Host Unreachable | 도달 불가능의 힌트일 뿐, 증거 아님 |
| 2 | Protocol Unreachable | - |
| 3 | Port Unreachable | - |
| 4 | Fragmentation Needed and DF Set | - |
| 5 | Source Route Failed | 도달 불가능의 힌트일 뿐, 증거 아님 |

** 중요**: 코드 0, 1, 5 메시지는 "힌트일 뿐, 증거가 아님" - 죽은 게이트웨이의 증거로 사용 불가

##### 3.2.2.2 Redirect (재지향)

| 요구사항 | 레벨 |
|----------|------|
| 호스트는 ICMP Redirect 메시지 송신 금지 (게이트웨이만 전송) | SHOULD NOT |
| 리디렉트 수신 시 라우팅 정보 업데이트 | MUST |
| 연결된 네트워크의 게이트웨이로부터의 리디렉트만 수락 | MUST |
| 리디렉트 처리를 구성 가능한 옵션으로 구현 | SHOULD |

##### 3.2.2.3 Source Quench (소스 억제)

| 요구사항 | 레벨 |
|----------|------|
| Source Quench 메시지 수신 시 소스로의 전송 속도 감소 | 구현 옵션 |

** 참고**: 이 메커니즘은 현대 인터넷 환경에서 제한된 효과

##### 3.2.2.4 Time Exceeded (시간 초과)

** 두 가지 코드 유형**:
- ** 코드 0**: TTL 초과
- ** 코드 1**: 단편 재조립 타임아웃

| 요구사항 | 레벨 |
|----------|------|
| 이 메시지를 수락하고 적절한 전송 계층으로 전달 | MUST |

##### 3.2.2.5 Parameter Problem (파라미터 문제)

| 요구사항 | 레벨 |
|----------|------|
| 잘못된 헤더 필드 또는 인식되지 않는 중요 옵션이 있는 데이터그램 수신 시 Parameter Problem 메시지 생성 | SHOULD |

** 코드**:
- ** 코드 0**: 포인터 오류
- ** 코드 1**: 필수 옵션 누락

##### 3.2.2.6 Echo Request/Reply (에코 요청/응답)

| 요구사항 | 레벨 |
|----------|------|
| 데이터 옥텟을 에코백하는 ICMP Echo 서버 기능 구현 | MUST |
| 에코 요청에 동일 데이터로 응답, 해당 시 수신된 타임스탬프 값 사용 | MUST |

##### 3.2.2.7 Information Request/Reply (정보 요청/응답)

| 요구사항 | 레벨 |
|----------|------|
| 정보 요청 처리 | OPTIONAL |

** 참고**: 이 기능은 현대 검색 프로토콜 배포로 구식이 됨

##### 3.2.2.8 Timestamp and Timestamp Reply (타임스탬프 및 타임스탬프 응답)

| 요구사항 | 레벨 |
|----------|------|
| 타임스탬프 값은 자정 UTC 이후 밀리초 표시 | MUST |
| 메시지 발신 시 또는 중간 홉이 미리 지정되지 않은 경우 타임스탬프 기록 | MUST |
| 목적지 호스트는 상위 계층으로 전달 전 현재 타임스탬프 추가 | SHOULD |

##### 3.2.2.9 Address Mask Request/Reply (주소 마스크 요청/응답)

| 요구사항 | 레벨 |
|----------|------|
| 서브넷 구성 검색을 위한 주소 마스크 요청 구현 | MAY |
| 응답은 요청을 수신한 인터페이스와 연결된 서브넷 마스크 제공 | SHOULD |

#### 3.2.3 Internet Group Management Protocol -- IGMP

| 요구사항 | 레벨 |
|----------|------|
| IP 멀티캐스팅을 위한 동적 호스트 그룹 설정 지원 | MAY |

### 3.3 Specific Issues (특정 이슈)

#### 3.3.1 Routing Outbound Datagrams (나가는 데이터그램 라우팅)

##### 로컬/원격 결정
- 호스트는 서브넷 마스크와 목적지 주소를 비교하여 로컬 또는 원격 여부 결정
- ** 로컬 목적지**: 직접 링크 계층 전달 사용
- ** 원격 목적지**: 게이트웨이 전달 필요

##### 게이트웨이 선택
- 다중 경로는 알고리즘 선택 필요
- ** 권장**: 구성 가능한 메트릭 및 라우팅 테이블 지원 구현
- 하드코딩된 단일 게이트웨이 가정 회피

##### 라우트 캐시
- 목적지 주소를 게이트웨이로 매핑하는 캐시 유지 구현 권장
- 전달 실패 또는 관리 신호에 따라 오래된 항목 무효화 메커니즘 필요

##### Dead Gateway Detection (죽은 게이트웨이 감지)

| 요구사항 | 레벨 |
|----------|------|
| 고정 게이트웨이 사용 호스트는 게이트웨이 사망 감지 및 백업 게이트웨이 전환 능력 보유 | SHOULD |

** 감지 메커니즘**:
- ICMP 도달 불가 메시지
- ARP 해결 실패
- 타임아웃 기반 접근

##### New Gateway Selection (새 게이트웨이 선택)
- 게이트웨이 실패 감지 후, 구성된 게이트웨이 목록에서 대안 선택
- 또는 ICMP 리디렉트 메시지를 통해 새 경로 발견

##### Initialization (초기화)
- 시작 시 초기 게이트웨이 정보 획득을 위해 BOOTP 또는 수동 구성 지원 권장

#### 3.3.2 Reassembly (재조립)

| 요구사항 | 레벨 |
|----------|------|
| IP 데이터그램 재조립 지원 | MUST |

** 명세 요구사항**:
- 소스 주소, 식별자, 프로토콜로 인덱싱된 단편 버퍼 유지
- 최대 재조립 시간 제한 적용 (기본값: 15초)
- 겹치는 단편 올바르게 처리
- 타임아웃 시 불완전한 재조립 묵시적 폐기

#### 3.3.3 Fragmentation (단편화)

| 요구사항 | 레벨 |
|----------|------|
| MTU보다 큰 데이터그램 송신 시 의도적 단편화 구현 | SHOULD |

- 명세는 호스트 시작 단편화를 허용 (필수 아님)
- 구현 시:
  - 구성된 MTU 값 존중
  - 식별자 및 단편 오프셋 필드 올바르게 설정

#### 3.3.4 Local Multihoming (로컬 멀티호밍)

##### 소개
- 멀티홈드 호스트: 다른 인터페이스 또는 동일 인터페이스에 다중 IP 주소 보유
- 소스 주소 선택 및 라우팅 결정에 복잡성 도입

##### 요구사항

| 요구사항 | 레벨 |
|----------|------|
| 인터페이스에 다중 주소 할당 지원 | MUST |
| 각 주소에 연관된 서브넷 마스크 필요 | MUST |
| 라우팅 결정은 모든 로컬 주소 고려 | MUST |

##### Source Address Selection (소스 주소 선택)
- 나가는 데이터그램: 선택된 게이트웨이의 연결된 네트워크와 일치하는 소스 주소 사용 필수
- 응답 데이터그램: 가능한 경우 목적지의 들어오는 인터페이스 주소와 일치하는 소스 사용 권장

#### 3.3.5 Source Route Forwarding (소스 라우트 전달)

** 중간 소스 라우트 홉 역할 호스트 요구사항**:
1. 포인터 유효성 검증
2. 현재 주소를 다음 홉 주소로 교체
3. 포인터를 4만큼 감소
4. 다음 홉으로 전달

** 최종 목적지 호스트 요구사항**:
- 응답 데이터그램을 위해 기록된 라우트 역방향 처리
- 소스 호스트가 기록된 라우트에 나타나도 반환 경로 올바르게 재구성

#### 3.3.6 Broadcasts (브로드캐스트)

| 브로드캐스트 유형 | 범위 |
|------------------|------|
| ** 제한된 브로드캐스트 {-1,-1}** | 연결된 네트워크만 도달 |
| ** 직접 브로드캐스트** | 특정 네트워크 또는 서브넷 대상 |

| 요구사항 | 레벨 |
|----------|------|
| 인터페이스에 대한 유효한 브로드캐스트 주소 인식 | MUST |
| 브로드캐스트 주소 지정된 데이터그램 수락 | MUST |

** 강조**: 브로드캐스트에 대한 ICMP 오류 응답 제한으로 "브로드캐스트 스톰" 방지

#### 3.3.7 IP Multicasting (IP 멀티캐스팅)

| 요구사항 | 레벨 |
|----------|------|
| IP 멀티캐스팅 지원 | MAY |

** 구현 시 요구사항**:
- 그룹 멤버십 관리를 위해 IGMP 사용
- 가입된 그룹으로 향하는 멀티캐스트 데이터그램 처리
- 멀티캐스트 전달을 위한 TTL 범위 지정 존중

#### 3.3.8 Error Reporting (오류 보고)

| 요구사항 | 레벨 |
|----------|------|
| 전달 불가능한 데이터그램에 대해 ICMP 오류 메시지 생성 | MUST |

** 엄격한 제한 - 다음에 대해 오류 메시지 금지**:
- 오류 메시지에 대한 응답
- 브로드캐스트
- 멀티캐스트
- 링크 계층 브로드캐스트
- 비초기 단편
- 잘못된 소스 주소를 가진 데이터그램

## 4. TRANSPORT PROTOCOLS (전송 프로토콜)

### 4.1 User Datagram Protocol -- UDP

#### 4.1.1 Introduction (소개)
- UDP: 신뢰성 보장 또는 흐름 제어 메커니즘 없는 비연결형 데이터그램 전송 서비스 제공

#### 4.1.2 Protocol Walk-Through (프로토콜 상세 검토)
- RFC-768 참조

#### 4.1.3 Specific Issues (특정 이슈)

##### Ports (포트)
- 포트 필드: 송신 및 수신 애플리케이션 프로세스 식별
- **Well-known 포트 (0-1023)**: 표준 서비스용으로 예약

##### IP Options (IP 옵션)

| 요구사항 | 레벨 |
|----------|------|
| 데이터그램에 수신된 IP 옵션을 애플리케이션 계층으로 전달 지원 | MUST |
| 애플리케이션이 전송된 데이터그램에 IP 옵션 지정 허용 | MUST |

##### ICMP Messages (ICMP 메시지)

| 요구사항 | 레벨 |
|----------|------|
| "Destination Unreachable" 또는 "Time Exceeded" 알림 등 전달 문제를 나타내는 ICMP 메시지 적절히 처리 | SHOULD |
| 이러한 조건을 영향받는 애플리케이션에 잠재적으로 보고 | SHOULD |

##### UDP Checksums (UDP 체크섬)

| 요구사항 | 레벨 |
|----------|------|
| 체크섬이 0인 UDP 데이터그램 수신 가능 | MUST |

#### 4.1.4 UDP/Application Layer Interface (UDP/애플리케이션 계층 인터페이스)

### 4.2 Transmission Control Protocol -- TCP

#### 4.2.1 Introduction (소개)
- TCP: 흐름 제어 및 혼잡 관리와 함께 신뢰성 있는 연결 지향 전송 서비스 제공
- RFC-793: 핵심 프로토콜 명세 정의

#### 4.2.2 Protocol Walk-Through (프로토콜 상세 검토)

##### Well-Known Ports (잘 알려진 포트)
- TCP는 표준화된 서비스용으로 포트 0-1023 예약 (Telnet, SMTP, FTP 등)
- 애플리케이션: 일반적으로 사용자 정의 사용을 위해 이 예약된 포트 회피 권장

##### Use of Push (Push 사용)

| 요구사항 | 레벨 |
|----------|------|
| 세그먼트의 PUSH 플래그는 데이터가 즉시 애플리케이션에 전달되어야 함을 나타냄 | MUST |
| 적시의 애플리케이션 데이터 전달을 위해 PUSH 플래그 동작 존중 | SHOULD |

##### Window Size (윈도우 크기)
- 윈도우 필드: 수신 버퍼 가용성 광고

| 요구사항 | 레벨 |
|----------|------|
| 윈도우 크기가 0인 TCP 세그먼트 수락 | MUST |

- 윈도우 크기 0: 흐름 제어 제약 신호

##### Urgent Pointer (긴급 포인터)
- 이 필드: 우선순위 처리가 필요한 데이터 식별
- URG 플래그 설정 시: 긴급 포인터가 세그먼트 내 긴급 데이터의 끝을 나타냄

##### TCP Options (TCP 옵션)
- 옵션: Maximum Segment Size (MSS), Window Scale, Selective Acknowledgment (SACK) 등 기능에 확장성 제공

| 요구사항 | 레벨 |
|----------|------|
| 인식되지 않는 옵션 묵시적 무시 | MUST |

##### Maximum Segment Size Option (최대 세그먼트 크기 옵션)

| 요구사항 | 레벨 |
|----------|------|
| 모든 SYN 세그먼트에 MSS 옵션 송신 | SHOULD |

- 일반적으로 경로 MTU에서 IP 및 TCP 헤더 크기를 뺀 값으로 설정 (최소 536 바이트)

##### TCP Checksum (TCP 체크섬)

| 요구사항 | 레벨 |
|----------|------|
| 수신된 모든 세그먼트에서 TCP 체크섬 검증 | MUST |
| 잘못된 체크섬을 가진 세그먼트 폐기 | MUST |

##### TCP Connection State Diagram (TCP 연결 상태 다이어그램)

** 상태 전이**:

```
CLOSED ─────────────────────────────────────────────────────────────┐
   │                                                                │
   │ (passive open)                              (active open)      │
   ▼                                                   │            │

   │                                                                │
   │ (recv SYN)                                                     │
   ▼                                                                │

   │                                                                │
   │ (recv ACK of SYN)                                             │
   ▼                                                                │
ESTABLISHED ◄──────────────────────────────────────────────────────┤
   │                                                                │
   │ (send FIN)                          (recv FIN)                │
   ▼                                         │                      │

   │                                    CLOSE-WAIT                  │
   │ (recv ACK of FIN)                      │                      │

   │                                    LAST-ACK                    │
   │ (recv FIN)                             │                      │
   ▼                                        │ (recv ACK of FIN)    │

   │                                    CLOSED                      │
   │ (recv ACK of FIN)                                             │
   ▼                                                                │
TIME-WAIT ──────────────────────────────────────────────────────────┘
   │
   │ (timeout=2MSL)
   ▼
CLOSED
```

##### Initial Sequence Number Selection (초기 순서 번호 선택)

| 요구사항 | 레벨 |
|----------|------|
| 초기 순서 번호는 무작위 또는 의사 무작위 메커니즘을 사용하여 생성 | MUST |

##### Simultaneous Open Attempts (동시 열기 시도)
- 양 끝점이 동시에 연결 시작 시: 둘 다 SYN-RECEIVED 상태 진입 가능

| 요구사항 | 레벨 |
|----------|------|
| 이 드물지만 유효한 시나리오를 적절히 처리 | MUST |

##### Recovery from Old Duplicate SYN (오래된 중복 SYN 복구)

| 요구사항 | 레벨 |
|----------|------|
| 이전 연결 시도에서 온 중복 SYN 세그먼트 감지 및 폐기 | MUST |

- ** 목적**: 상태 머신 손상 방지

##### RST Segment (RST 세그먼트)
- 리셋 세그먼트: 연결을 갑자기 종료

| 요구사항 | 레벨 |
|----------|------|
| RST 세그먼트에 데이터 필드 포함 금지 | MUST NOT |

- RST 생성: 연결 거부 또는 프로토콜 오류

##### Closing a Connection (연결 종료)
- 표준 종료 시퀀스: FIN-ACK 교환 포함

| 요구사항 | 레벨 |
|----------|------|
| 응답에서 데이터가 예상되지 않으면 FIN 송신 후 종료 가능 | MAY |
| TIME-WAIT 상태 대기 권장 | RECOMMENDED |

##### Data Communication (데이터 통신)
- 데이터 전달: 시퀀싱 및 흐름 제어 윈도우 존중
- 송신자: 광고된 윈도우 크기 초과 금지

##### Retransmission Timeout (재전송 타임아웃)
- 혼잡 인식 구현: RTT 측정과 RTO 계산을 결합한 Jacobson 알고리즘 사용
- 타임아웃 값 적응적 조정

##### Managing the Window (윈도우 관리)

| 요구사항 | 레벨 |
|----------|------|
| 수신자는 오른쪽 윈도우 가장자리를 축소하면 안 됨 | MUST NOT |

- 버퍼 공간이 사용 가능해지면 윈도우 업데이트 전송 권장

##### Probing Zero Windows (제로 윈도우 프로빙)
- 송신자가 제로 윈도우 광고를 만나면:

| 요구사항 | 레벨 |
|----------|------|
| 추가 버퍼 공간이 사용 가능해질 때 감지하기 위해 주기적으로 수신자 프로빙 | MUST |

##### Passive OPEN Calls (수동 OPEN 호출)
- 애플리케이션: 특정 포트에서 리스닝 요청
- 구현: 일반적으로 애플리케이션의 수락을 위해 보류 중인 연결을 백로그에 큐잉

##### Time to Live (생존 시간)

| 요구사항 | 레벨 |
|----------|------|
| TCP는 전송 계층으로부터의 호출을 통해 전송된 세그먼트의 TTL 값 설정 지원 | MUST |

- 애플리케이션이 데이터그램 범위 제한 허용

##### Event Processing (이벤트 처리)

| 요구사항 | 레벨 |
|----------|------|
| 들어오는 세그먼트, 타임아웃, 애플리케이션 요청, 오류 알림을 조정된 방식으로 처리 | MUST |

##### Acknowledging Queued Segments (큐된 세그먼트 확인)
- 지연 ACK: 데이터 세그먼트에 확인을 피기백하여 효율성 향상

| 요구사항 | 레벨 |
|----------|------|
| 과도한 재전송 방지를 위해 ACK를 신속히 전송 | SHOULD |

#### 4.2.3 Specific Issues (특정 이슈)

##### 4.2.3.1 Retransmission Timeout Calculation (재전송 타임아웃 계산)

| 요구사항 | 레벨 |
|----------|------|
| RTO 계산은 지수 백오프와 함께 Jacobson 알고리즘 사용 | SHOULD |

- 초기 RTO 값: 일반적으로 1-3초에서 시작
- 각 재전송마다 최대값까지 두 배로 증가

##### 4.2.3.2 When to Send an ACK Segment (ACK 세그먼트 송신 시점)

| 요구사항 | 레벨 |
|----------|------|
| 전체 크기 세그먼트 두 개마다 ACK 전송 | SHOULD |
| 순서가 맞지 않는 세그먼트 수신 시 ACK 전송 | MUST |

- 지연 ACK: 응답성 유지하면서 효율성 향상

##### 4.2.3.3 When to Send a Window Update (윈도우 업데이트 송신 시점)

| 요구사항 | 레벨 |
|----------|------|
| 광고된 윈도우가 증가하면 윈도우 업데이트 전송 | MUST |

- 충분한 버퍼 공간이 사용 가능해지면 업데이트 전송 권장

##### 4.2.3.4 When to Send Data (데이터 송신 시점)
- 구현: 단편화 감소를 위해 Nagle 알고리즘 존중 권장
- 전체 MSS 크기 세그먼트 또는 push 플래그 설정 시 즉시 전송 허용

##### 4.2.3.5 TCP Connection Failures (TCP 연결 실패)

| 요구사항 | 레벨 |
|----------|------|
| 재전송 시도의 지수 백오프를 통해 연결 실패 감지 | SHOULD |

- 적절한 재전송 횟수 후 연결 중단

##### 4.2.3.6 TCP Keep-Alives (TCP Keep-Alives)

| 요구사항 | 레벨 |
|----------|------|
| 실패한 연결 감지를 위해 Keep-alive 세그먼트 전송 | MAY |

- ** 이유**: 오버헤드 추가 및 실패한 경로를 통과할 수 있음

##### 4.2.3.7 TCP Multihoming (TCP 멀티호밍)
- 멀티홈드 호스트의 애플리케이션: 바인딩할 소스 주소 지정 가능해야 함

| 요구사항 | 레벨 |

##### 4.2.3.12 Efficiency (효율성)
- 구현: 적시 통신 유지하면서 불필요한 패킷 전송 회피
