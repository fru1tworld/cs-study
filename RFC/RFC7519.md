## 1. Introduction (소개)

### 1.1 JWT의 정의
- 두 당사자 간에 전송될 클레임(claims)을 나타내는 컴팩트하고 URL-safe한 수단
- JWS(JSON Web Signature) 또는 JWE(JSON Web Encryption) 구조의 페이로드로 사용
- 디지털 서명, MAC(Message Authentication Code), 암호화 지원

### 1.2 JWT의 목적
- 클레임 기반 토큰 표현
- 컴팩트한 형식으로 HTTP 헤더, URI 쿼리 파라미터에 적합
- JSON 기반으로 다양한 프로그래밍 언어에서 쉽게 처리 가능

### 1.3 발음
- JWT는 영어 단어 "jot"과 동일하게 발음

### 1.4 사용 사례
- OpenID Connect 인증
- OAuth 2.0 액세스 토큰
- API 인증 및 권한 부여
- 정보 교환

### 1.5 Notational Conventions (표기법)

#### RFC 2119 키워드

| 키워드 | 의미 |
|--------|------|
| MUST | 절대적 요구사항 |
| MUST NOT | 절대적 금지사항 |
| REQUIRED | MUST와 동의어 |
| SHALL | MUST와 동의어 |
| SHALL NOT | MUST NOT과 동의어 |
| SHOULD | 권장사항 (특별한 이유가 없으면 따라야 함) |
| SHOULD NOT | 비권장사항 |
| RECOMMENDED | SHOULD와 동의어 |
| NOT RECOMMENDED | SHOULD NOT과 동의어 |
| MAY | 선택사항 |
| OPTIONAL | MAY와 동의어 |

## 2. Terminology (용어 정의)

### 2.1 핵심 용어

| 용어 | 정의 |
|------|------|
| JSON Web Token (JWT) | 클레임을 문자열로 표현하는 형식. JWS 또는 JWE로 인코딩되어 디지털 서명, MAC, 암호화 지원 |
| JWT Claims Set | JWT가 전달하는 클레임을 포함한 JSON 객체 |
| Claim | 주체(subject)에 대해 주장되는 정보. 이름/값 쌍으로 구성 |
| Claim Name | 클레임의 이름 부분. 항상 문자열 |
| Claim Value | 클레임의 값 부분. 모든 JSON 값 가능 |
| Nested JWT | 중첩된 서명 또는 암호화가 적용된 JWT. JWT를 JWS 또는 JWE의 페이로드로 사용 |
| Unsecured JWT | 무결성 보호나 암호화가 없는 JWT. JWS의 "alg" 값이 "none" |

### 2.2 StringOrURI
- JSON 문자열 값
- 콜론(":")을 포함하면 반드시 URI여야 함
- 대소문자 구분
- 변환/정규화 없음

### 2.3 NumericDate
- 숫자 JSON 값
- 1970-01-01T00:00:00Z UTC(Unix Epoch)부터의 초 단위
- 윤초(leap second) 무시
- 정수가 아닌 값도 표현 가능

## 3. JSON Web Token (JWT) Overview (개요)

### 3.1 JWT 구조
```
Header.Payload.Signature
```

- **세 부분**이 마침표('.')로 구분
- 각 부분은 **base64url 인코딩**

### 3.2 JWT 유형

| 유형 | 설명 |
|------|------|
| JWS Compact Serialization | 서명된 JWT (디지털 서명 또는 MAC) |
| JWE Compact Serialization | 암호화된 JWT |
| Unsecured JWT | 서명/암호화 없는 JWT (alg: none) |
| Nested JWT | 서명 후 암호화 또는 암호화 후 서명 |

### 3.3 Example JWT (예시)

#### JOSE Header
```json
{"typ":"JWT",
 "alg":"HS256"}
```
Base64url 인코딩:
```
eyJ0eXAiOiJKV1QiLA0KICJhbGciOiJIUzI1NiJ9
```

#### JWT Claims Set
```json
{"iss":"joe",
 "exp":1300819380,
 "http://example.com/is_root":true}
```
Base64url 인코딩:
```
eyJpc3MiOiJqb2UiLA0KICJleHAiOjEzMDA4MTkzODAsDQogImh0dHA6Ly9leGFtcGxlLmNvbS9pc19yb290Ijp0cnVlfQ
```

#### 완전한 JWT (HMAC SHA-256 서명)
```
eyJ0eXAiOiJKV1QiLA0KICJhbGciOiJIUzI1NiJ9
.
eyJpc3MiOiJqb2UiLA0KICJleHAiOjEzMDA4MTkzODAsDQogImh0dHA6Ly9leGFtcGxlLmNvbS9pc19yb290Ijp0cnVlfQ
.
dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk
```

## 4. JWT Claims (클레임)

### 4.1 Registered Claim Names (등록된 클레임)

| 클레임 | 전체 이름 | 설명 | 타입 | 필수 여부 |
|--------|----------|------|------|-----------|
| iss | Issuer | JWT를 발급한 주체 식별 | StringOrURI | 선택 |
| sub | Subject | JWT의 주체 식별. 발급자 컨텍스트에서 로컬 고유 또는 전역 고유 | StringOrURI | 선택 |
| aud | Audience | JWT의 의도된 수신자 식별. 배열 또는 단일 값 가능 | StringOrURI (배열 가능) | 선택 |
| exp | Expiration Time | JWT 만료 시간. 이 시간 이후 JWT 거부 | NumericDate | 선택 |
| nbf | Not Before | JWT 활성화 시간. 이 시간 이전 JWT 거부 | NumericDate | 선택 |
| iat | Issued At | JWT 발급 시간. JWT의 나이(age) 결정에 사용 | NumericDate | 선택 |
| jti | JWT ID | JWT의 고유 식별자. 재생 공격(replay attack) 방지 | String (대소문자 구분) | 선택 |

#### 4.1.1 iss (Issuer)
- JWT를 발급한 주체(principal) 식별
- 애플리케이션 특정 처리
- 대소문자 구분 문자열
- 선택 사항 (OPTIONAL)

#### 4.1.2 sub (Subject)
- JWT의 주체 식별
- 발급자 컨텍스트에서 로컬로 고유하거나 전역적으로 고유해야 함
- 애플리케이션 특정 처리
- 선택 사항 (OPTIONAL)

#### 4.1.3 aud (Audience)
- JWT가 의도된 수신자 식별
- 수신자는 자신을 "aud" 값으로 식별해야 함
- 자신이 의도된 수신자가 아니면 JWT 거부해야 함 (MUST)
- 단일 문자열 또는 문자열 배열
- 선택 사항 (OPTIONAL)

#### 4.1.4 exp (Expiration Time)
- JWT가 처리되어서는 안 되는 만료 시간
- 현재 시간이 "exp" 이상이면 JWT 거부해야 함 (MUST)
- 구현 시 클록 스큐(clock skew) 대비 몇 분 정도 여유 제공 가능 (MAY)
- 선택 사항 (OPTIONAL)

#### 4.1.5 nbf (Not Before)
- JWT가 처리되어서는 안 되는 시간 이전
- 현재 시간이 "nbf" 미만이면 JWT 거부해야 함 (MUST)
- 클록 스큐 대비 여유 제공 가능 (MAY)
- 선택 사항 (OPTIONAL)

#### 4.1.6 iat (Issued At)
- JWT가 발급된 시간
- JWT의 나이 결정에 사용 가능
- 선택 사항 (OPTIONAL)

#### 4.1.7 jti (JWT ID)
- JWT를 위한 고유 식별자
- 재생 공격(replay attack) 방지에 사용
- 발급자별 충돌 방지 필수 (MUST)
- 대소문자 구분 문자열
- 선택 사항 (OPTIONAL)

### 4.2 Public Claim Names (공개 클레임)

- IANA "JSON Web Token Claims" 레지스트리에 등록
- 또는 충돌 방지(Collision-Resistant) 이름 사용
  - 예: URI 형식 (`http://example.com/is_root`)
- 이름 정의자의 네임스페이스 제어 필수
- 공개 클레임 정의자는 다른 정의자의 네임스페이스와 충돌 방지 책임

### 4.3 Private Claim Names (비공개 클레임)

- JWT 생산자(producer)와 소비자(consumer) 간의 협약
- 등록되거나 공개된 이름이 아님
- 충돌 위험 있음 - 주의 필요
- 사용 환경 내에서만 의미 있음

## 5. JOSE Header

### 5.1 개요
- JWT의 첫 번째 부분
- JSON 객체로 구성
- JWT에 적용된 암호화 작업 설명
- JWS 또는 JWE의 헤더 파라미터 포함

### 5.2 "typ" (Type) Header Parameter

- JWT 애플리케이션의 미디어 타입 선언
- 권장 값: "JWT" (대문자 권장)
- JWT 구현에서는 무시됨
- JWT가 아닌 객체가 존재할 수 있는 컨텍스트에서 유용
- 선택 사항 (OPTIONAL)
- 대소문자 무관 (case-insensitive)

### 5.3 "cty" (Content Type) Header Parameter

- 구조적 정보 전달
- 중첩 서명/암호화가 없으면 사용 권장하지 않음 (NOT RECOMMENDED)
- 중첩이 있으면 반드시 "JWT" 값 사용 (MUST)
- 선택 사항 (OPTIONAL)
- "application/" 접두사 생략 권장

### 5.4 Replicating Claims as Header Parameters (헤더에 클레임 복제)

- 암호화된 JWT에서 일부 클레임의 암호화되지 않은 표현 제공 가능
- 복제 가능한 클레임:
  - "iss" (Issuer)
  - "sub" (Subject)
  - "aud" (Audience)
- 목적: 복호화 전 처리 결정 지원
- 수신 애플리케이션은 헤더와 클레임 값이 동일함을 검증해야 함 (SHOULD)
- 암호화되지 않은 전송이 안전한 클레임만 복제 권장 (SHOULD)

## 6. Unsecured JWTs (비보안 JWT)

### 6.1 정의
- 무결성 보호나 암호화가 없는 JWT
- JWE 또는 JWS 외부의 다른 수단으로 보안이 제공되는 경우 사용

### 6.2 특징
- "alg" 헤더 파라미터 값: "none"
- JWS 서명: 빈 문자열
- 세 부분 구조 (마지막 부분이 비어있음)

### 6.3 Unsecured JWT 예시

#### JOSE Header
```json
{"alg":"none"}
```
Base64url 인코딩:
```
eyJhbGciOiJub25lIn0
```

#### JWT Claims Set
```json
{"iss":"joe",
 "exp":1300819380,
 "http://example.com/is_root":true}
```
Base64url 인코딩:
```
eyJpc3MiOiJqb2UiLA0KICJleHAiOjEzMDA4MTkzODAsDQogImh0dHA6Ly9leGFtcGxlLmNvbS9pc19yb290Ijp0cnVlfQ
```

#### 완전한 Unsecured JWT
```
eyJhbGciOiJub25lIn0
.
eyJpc3MiOiJqb2UiLA0KICJleHAiOjEzMDA4MTkzODAsDQogImh0dHA6Ly9leGFtcGxlLmNvbS9pc19yb290Ijp0cnVlfQ
.
```
- 마지막 마침표 뒤에 빈 문자열 (서명 없음)

## 7. Creating and Validating JWTs (JWT 생성 및 검증)

### 7.1 Creating a JWT (JWT 생성 절차)

1. **JWT Claims Set 작성**
   - 원하는 클레임을 포함하는 JSON 객체 생성
   - 공백/줄바꿈 허용
   - 정규화 불필요

2. **메시지 생성**
   - JWT Claims Set의 UTF-8 표현을 옥텟(octets) 시퀀스로 변환

3. **JOSE Header 생성**
   - 원하는 헤더 파라미터 세트를 포함하는 JSON 객체 생성

4. **JWS 또는 JWE 생성**
   - **JWS의 경우**: 메시지를 JWS 페이로드로 사용
   - **JWE의 경우**: 메시지를 JWE 평문(plaintext)으로 사용

5. **중첩 작업** (해당하는 경우)
   - 이전 단계의 결과를 새 메시지로 설정
   - "cty" (Content Type) 값을 "JWT"로 설정
   - 단계 3으로 돌아가 반복

6. **결과 JWT**
   - JWS 또는 JWE가 최종 결과

### 7.2 Validating a JWT (JWT 검증 절차)

1. **마침표('.') 포함 확인**
   - 적어도 하나의 마침표 문자 포함 확인

2. **JOSE Header 추출**
   - 첫 번째 마침표 이전의 부분

3. **JOSE Header Base64url 디코딩**
   - 표준 Base64url 디코딩 규칙 적용

4. **UTF-8이 유효한 JSON인지 확인**
   - UTF-8 인코딩된 표현이 완전히 유효한 JSON 객체여야 함

5. **JOSE Header 파라미터 지원 확인**
   - 결과 JOSE Header의 파라미터가 이해되고 지원되는지 확인
   - "crit" (Critical) 파라미터가 있으면 해당 파라미터도 확인

6. **JWS 또는 JWE 판정**
   - JOSE Header가 JWS인지 JWE인지 판별
   - "enc" (encryption algorithm) 파라미터가 있으면 JWE
   - "enc" 파라미터가 없으면 JWS

7. **JWS/JWE 검증**
   - **JWS의 경우**: RFC 7515 (JWS) Section 5.2의 검증 절차 수행
   - **JWE의 경우**: RFC 7516 (JWE) Section 5.2의 복호화 절차 수행

8. **중첩 JWT 확인**
   - "cty" 값이 "JWT"면 중첩 JWT
   - 중첩 JWT인 경우 단계 1로 돌아가 반복

9. **메시지 Base64url 디코딩**
   - JWT Claims Set이 포함된 메시지 디코딩

10. **UTF-8이 유효한 JSON 객체인지 확인**
    - UTF-8 인코딩된 표현이 완전히 유효한 JSON 객체여야 함

**중요**: 모든 단계에서 실패 시 JWT를 거부해야 함 (MUST)

### 7.3 String Comparison Rules (문자열 비교 규칙)

- RFC 7159 Section 8.3의 JSON 규칙 사용
- **동등성/부등식 비교만 수행** (대소 비교 없음)
- **대소문자 구분** (case-sensitive)
- **변환/정규화 없음**
- 이스케이프 시퀀스가 있는 유니코드 문자는 비교 전 이스케이프 해제
- "typ"과 "cty" 값은 다른 규칙 적용 가능 (대소문자 무관 비교)

## 8. Implementation Requirements (구현 요구사항)

### 8.1 서명/MAC 알고리즘

| 알고리즘 | "alg" 값 | 요구 수준 |
|----------|----------|-----------|
| HMAC SHA-256 | HS256 | 필수 구현 (MUST) |
| None | none | 필수 구현 (MUST) |
| RSASSA-PKCS1-v1_5 SHA-256 | RS256 | 권장 구현 (RECOMMENDED) |
| ECDSA P-256 SHA-256 | ES256 | 권장 구현 (RECOMMENDED) |

### 8.2 암호화 알고리즘 (암호화 지원 시)

#### 키 관리 알고리즘 (필수)

| 알고리즘 | "alg" 값 |
|----------|----------|
| RSAES-PKCS1-v1_5 | RSA1_5 |
| AES-128 Key Wrap | A128KW |
| AES-256 Key Wrap | A256KW |

#### 콘텐츠 암호화 알고리즘 (필수)

| 알고리즘 | "enc" 값 |
|----------|----------|
| AES-128-CBC + HMAC SHA-256 | A128CBC-HS256 |
| AES-256-CBC + HMAC SHA-512 | A256CBC-HS512 |

#### 권장 알고리즘

| 키 관리 알고리즘 | "alg" 값 |
|------------------|----------|
| ECDH-ES + A128KW | ECDH-ES+A128KW |
| ECDH-ES + A256KW | ECDH-ES+A256KW |

| 콘텐츠 암호화 알고리즘 | "enc" 값 |
|------------------------|----------|
| AES-128-GCM | A128GCM |
| AES-256-GCM | A256GCM |

### 8.3 Nested JWT 지원
- 선택 사항 (OPTIONAL)
- 서명 후 암호화 또는 암호화 후 서명 지원

## 9. URI for Declaring that Content is a JWT (JWT 콘텐츠 선언 URI)

### URN
```
urn:ietf:params:oauth:token-type:jwt
```

### 용도
- 콘텐츠가 JWT임을 선언하는 애플리케이션에서 사용
- 미디어 타입(application/jwt) 대신 URI 기반 선언 필요 시 사용

## 10. Security Considerations (보안 고려사항)

### 10.1 Trust Decisions (신뢰 결정)

- JWT의 내용은 **암호화적으로 보안되고 신뢰 결정에 필요한 컨텍스트에 바인딩되지 않는 한** 신뢰할 수 없음
- 검증해야 할 사항:
  - JWT 서명/암호화에 사용된 키가 발급자(issuer)의 통제 하에 있는지
  - 발급자가 신뢰할 수 있는지
  - "aud" (Audience) 클레임이 현재 애플리케이션에 유효한지
  - "exp" (Expiration Time), "nbf" (Not Before) 클레임의 유효성

### 10.2 Signing and Encryption Order (서명과 암호화 순서)

#### 중첩 JWT에서 둘 다 필요한 경우

| 순서 | 설명 | 이점 |
|------|------|------|
| 서명 후 암호화 (권장) | 메시지 서명 → 암호화 | 서명 제거 공격 방지, 서명자 신원 프라이버시 보호 |
| 암호화 후 서명 | 메시지 암호화 → 서명 | 특정 법적 요구사항 충족 가능 |

#### 고려 사항
- 일반적으로 **서명 후 암호화** 권장
- 암호화된 텍스트에 서명하는 것의 법적 문제 고려 필요
- 애플리케이션 요구사항에 따라 순서 결정

### 10.3 Certificate Thumbprint Confirmation (인증서 지문 확인)

- X.509 인증서를 사용하는 경우 인증서 지문(thumbprint) 확인 가능
- 클레임과 헤더 파라미터 간의 일관성 검증

## 11. Privacy Considerations (프라이버시 고려사항)

### 11.1 개인정보 보호 방법

JWT가 프라이버시에 민감한 정보를 포함하는 경우:

1. **암호화된 JWT 사용**
   - 수신자 인증과 함께 JWE 사용
   - 의도된 수신자만 내용 확인 가능

2. **암호화된 채널 사용**
   - TLS 등 암호화 프로토콜 사용
   - 엔드포인트 인증 지원

3. **정보 생략**
   - 프라이버시에 민감한 정보를 JWT에서 제외
   - 가장 간단한 방법

### 11.2 주의 사항
- 암호화되지 않은 JWT의 클레임은 쉽게 읽을 수 있음
- Base64url 인코딩은 암호화가 아님
- 민감한 정보는 반드시 암호화 필요

## Appendix A: JWT Examples (JWT 예제)

### A.1 Example Encrypted JWT (암호화된 JWT 예제)

#### 사용 알고리즘
- 키 암호화: RSAES-PKCS1-v1_5
- 콘텐츠 암호화: AES_128_CBC_HMAC_SHA_256

#### JOSE Header
```json
{"alg":"RSA1_5","enc":"A128CBC-HS256"}
```

#### 구조
JWE Compact Serialization에 따른 5부분 구조:
1. JOSE Header (Base64url 인코딩)
2. 암호화된 키 (Base64url 인코딩)
3. 초기화 벡터 (Base64url 인코딩)
4. 암호문 (Base64url 인코딩)
5. 인증 태그 (Base64url 인코딩)

### A.2 Example Nested JWT (중첩 JWT 예제)

#### 구조
- 내부 JWT: 서명된 JWT (JWS)
- 외부 JWT: 암호화된 JWT (JWE)

#### 외부 JOSE Header
```json
{"alg":"RSA1_5",
 "enc":"A128CBC-HS256",
 "cty":"JWT"}
```

#### 특징
- "cty":"JWT"로 중첩 구조 표시
- 내부 JWT가 평문으로 사용됨
- 완전한 평문 JWT 크기: 458 옥텟

## Appendix B: Relationship of JWTs to SAML Assertions (JWT와 SAML 관계)

### B.1 비교

| 특성 | JWT | SAML 2.0 |
|------|-----|----------|
| 형식 | JSON | XML |
| 서명 | JWS (JSON 기반) | XML-DSIG (정규화 필요) |
| 크기 | 컴팩트 | 상대적으로 큼 |
| 복잡성 | 간단 | 복잡 |
| 표현력 | 제한적 | 풍부 |
| 사용 환경 | HTTP 헤더, URI 쿼리 파라미터 | 다양 |

### B.2 JWT의 위치
- SAML의 완전한 대체가 아닌 **보완재**
- 컴팩트함이 중요한 시나리오에서 사용
- 단순성 우선 시 선택

### B.3 클레임 모델
- JWT의 "iss", "sub" 등이 SAML의 유사 개념과 대응
- 모든 클레임이 선택 사항이므로 다양한 사용 가능
