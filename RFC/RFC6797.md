## 1. Abstract

- HSTS는 웹사이트가 ** 보안 연결(HTTPS)로만 접근 가능** 함을 선언하는 메커니즘
- `Strict-Transport-Security` HTTP 응답 헤더 필드를 통해 정책 전달
- UA(User Agent) 설정을 통해서도 구성 가능
- 목적: 사용자가 지정된 사이트와 ** 암호화된 채널로만 통신** 하도록 보장

## 3. Overview

### 3.1 Use Cases (사용 사례)

- ** 주요 시나리오**:
  - 웹 브라우저 사용자: 다양한 웹사이트와 안전한 상호작용 원함
  - 웹사이트 배포자: 사용자를 위해 명시적으로 안전한 서비스 제공

### 3.2 HSTS Policy Effects (정책 효과)

HSTS 정책을 구현한 적합한 UA가 HSTS 호스트에 대해 달성하는 효과:

1. **URI 변환**:
   - 비보안 URI 참조(http)를 보안 URI 참조(https)로 자동 변환
   - 역참조(dereferencing) 전에 변환 수행

2. ** 보안 연결 강제**:
   - 보안 전송 오류/경고 발생 시 연결 즉시 종료
   - 사용자 우회 옵션 제공 안 함

### 3.3 Threat Model (위협 모델)

#### 3.3.1 해결되는 위협

| 위협 유형 | 설명 | HSTS 대응 |
|-----------|------|----------|
| **Passive Network Attacker** | 무선 네트워크 도청으로 세션 쿠키 탈취 | 암호화된 연결 강제로 세션 ID 가로채기 방지 |
| **Active Network Attacker** | DNS 스푸핑, 라우터 침해, AP 위장을 통한 MITM 공격 | 암호화 연결 강제 + 사용자 우회 옵션 제거 |
| **Web Site Development Bugs** | 혼합 콘텐츠 취약점 (스타일시트/미디어를 비보안으로 로딩) | 초기 비보안 연결 방지 |

#### 3.3.2 해결되지 않는 위협

| 위협 유형 | 설명 | 이유 |
|-----------|------|------|
| **Phishing** | 다른 도메인의 가짜 사이트에서 자격 증명 수집 | HSTS는 다른 도메인을 제어할 수 없음 |
| **Malware/Browser Vulnerabilities** | 시스템 손상 | HSTS는 시스템 신뢰성에 의존 |

### 3.4 Requirements (요구사항)

#### 3.4.1 전체 요구사항

- 네트워크 공격자로부터의 위험 최소화
- 개발자 버그로부터의 위험 최소화
- 사용자의 비보안 행동으로부터의 위험 최소화

#### 3.4.2 세부 핵심 요구사항

1. ** 정책 선언**: 웹사이트가 UA에 엄격한 보안 정책 선언 필수
2. ** 암호화 지시**: 웹사이트가 비보안 연결을 암호화 사용하도록 지시
3. ** 정책 유지**: UA가 지정된 기간 동안 영구적인 정책 데이터 보유, "최신" 정보 캐시
4. **URI 재작성**: UA가 정책 활성화된 사이트에 대해 모든 "http" URI를 "https"로 재작성
5. ** 서브도메인 적용**: 관리자가 서브도메인에 정책 적용 신호 가능, UA가 이를 강제
6. ** 상위 도메인 보호**: UA가 피어 또는 상위 레벨 도메인에 정책 적용 방지
7. ** 경고 우회 방지**: UA가 사용자의 보안 경고 우회 방지

#### 3.4.3 부가 요구사항

1. ** 혼합 보안 컨텍스트 금지**: 보안/비보안 혼합 로드 불허
2. ** 사용자 정의 정책**: 서버 신호와 관계없이 사용자가 엄격한 보안 사이트 선언 가능

## 6. HSTS Mechanism Overview (HSTS 메커니즘 개요)

### 6.1 HSTS Host Declaration (HSTS 호스트 선언)

- HTTP 호스트가 ** 보안 전송(TLS)을 통해** `Strict-Transport-Security` 헤더 발행
- 적합한 UA는 오류 없는 수신 시 해당 호스트를 **Known HSTS Host** 로 간주

### 6.2 HSTS Policy (HSTS 정책)

HSTS 정책이 지시하는 내용:

1. ** 보안 채널 전용 통신**: Known HSTS Host와 보안 채널로만 통신
2. ** 유지 기간 지정**: 정책 보유 기간 명시
3. ** 서브도메인 확장** (선택): `includeSubDomains` 디렉티브로 서브도메인에 정책 확장

### 6.3 HSTS Policy Storage and Maintenance (정책 저장 및 유지)

- UA가 발행 호스트 도메인 이름으로 ** 인덱싱** 하여 정책 유지
- 슈퍼도메인/서브도메인과 ** 별도로** 저장
- ** 오직 발행 호스트만** 자신의 정책 업데이트/삭제 가능
- `max-age=0`은 정책 삭제 신호

### 6.4 User Agent HSTS Policy Enforcement (UA HSTS 정책 집행)

연결 수립 시 UA의 동작:

1. Known HSTS Host 검사하여 적용 가능성 결정
2. 적용 조건:
   - `includeSubDomains`를 가진 슈퍼도메인이 적용되는 경우, 또는
   - 호스트 자체가 Known HSTS Host인 경우

## 7. Strict-Transport-Security Header Field Syntax

### 7.1 헤더 필드 의미

- UA에게 응답 메시지를 발행하는 호스트에 대해 **HSTS 정책을 강제** 하도록 지시

### 7.2 ABNF 문법

```abnf
Strict-Transport-Security = "Strict-Transport-Security" ":"
                             [ directive ]  *( ";" [ directive ] )

directive         = directive-name [ "=" directive-value ]
directive-name    = token
directive-value   = token | quoted-string
```

### 7.3 디렉티브 요구사항

| 규칙 | 설명 |
|------|------|
| 순서 무관 | 디렉티브 순서는 중요하지 않음 |
| 단일 출현 | 각 디렉티브는 최대 한 번만 출현 |
| 대소문자 무관 | 디렉티브 이름은 대소문자 구분 안 함 |
| 비적합 헤더 무시 | UA는 문법에 맞지 않는 헤더 무시 |
| 미인식 디렉티브 | 인식되지 않는 디렉티브는 무시하되 인식된 디렉티브는 처리 |

### 7.4 max-age Directive

- ** 필수** 디렉티브
- UA가 호스트를 Known HSTS Host로 간주하는 ** 초 단위 기간** 지정
- 값이 0이면 ** 정책 중단** 신호
- 문법: `max-age=delta-seconds` (delta-seconds는 숫자)

### 7.5 includeSubDomains Directive

- ** 선택적** 디렉티브
- 값 없음 (valueless)
- 존재 시: HSTS 정책이 ** 해당 호스트와 모든 서브도메인** 에 적용됨을 신호

### 7.6 Examples (예시)

```
# 1년 정책, 현재 도메인만
Strict-Transport-Security: max-age=31536000

# 6개월 정책, 서브도메인 포함
Strict-Transport-Security: max-age=15768000 ; includeSubDomains

# 정책 삭제
Strict-Transport-Security: max-age=0
```

## 8. Server Processing Model (서버 처리 모델)

### 8.1 HTTP-over-Secure-Transport Requests (보안 전송 요청)

- 보안 전송을 통해 응답 시, HSTS 호스트는 STS 헤더를 ** 포함해야 함 (SHOULD)**
- 포함 시, ** 정확히 하나의 헤더** 만 있어야 함 (MUST)
- Known HSTS Host 상태 수립 방법:
  - 보안 HTTP를 통해, 또는
  - 클라이언트 측 사전 로드 목록을 통해

### 8.2 HTTP Requests (Non-Secure Transport) (비보안 전송 요청)

- 비보안 HTTP 요청 수신 시:
  - ** 영구 리다이렉트(HTTP 301)** 로 응답해야 함 (SHOULD)
  - HTTPS URI로 (스킴만 변경)

- ** 중요**: HSTS 호스트는 비보안 전송을 통한 응답에 STS 헤더를 ** 포함하면 안 됨 (MUST NOT)**

## 9. User Agent Processing Model (UA 처리 모델)

### 전제 조건

- IDNA2008 (또는 IDNA2003) 구현
- IDNA 정규화된 도메인 이름 사용

### 9.1 Strict-Transport-Security Header Processing

#### 처리 조건

- 보안 전송을 통해 적합한 STS 헤더 수신
- 기본 오류 없음

#### 처리 동작

| 상황 | 동작 |
|------|------|
| 아직 기록되지 않음 | Known HSTS Host로 기록 |
| 이미 기록됨 | max-age 또는 includeSubDomains가 다르면 캐시 정보 업데이트 |

#### 처리 규칙

| 규칙 | 설명 |
|------|------|
| max-age | 헤더 수신 시점부터의 TTL(time-to-live) |
| max-age=0 | 캐시된 정책 제거 (includeSubDomains 포함) |
| 다중 STS 헤더 | 첫 번째만 처리 |
| 비보안 전송 헤더 | 무시 |
| 비적합 헤더 | 무시 |

#### 9.1.1 Noting an HSTS Host (HSTS 호스트 기록)

- UA는 IP 주소를 Known HSTS Host로 기록하면 ** 안 됨 (MUST NOT)**
- 기존 Known HSTS Host와 매칭되지 않는 도메인 이름의 경우:
  - 호스트 기록 ** 필수 (MUST)**
  - 캐시 내용: 도메인 이름, 만료 시간(max-age 기준), includeSubDomains 상태
- UA는 슈퍼도메인 엔트리의 만료 또는 디렉티브를 ** 수정하면 안 됨 (MUST NOT)**
- 만료된 엔트리는 발견 시 캐시에서 ** 제거 필수**

### 9.2 Known HSTS Host Domain Name Matching (도메인 이름 매칭)

- ** 매칭 방법**: 레이블별, 오른쪽에서 왼쪽으로, ASCII 대소문자 무관 비교

#### 매칭 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| **Superdomain Match** | Known HSTS Host 도메인이 주어진 도메인 이름의 전체 오른쪽 부분과 매칭 | "bar.foo.example.com"이 "qaz.bar.foo.example.com"과 매칭 |
| **Congruent Match** | Known HSTS Host 도메인이 주어진 도메인 이름과 정확히 매칭 | "foo.example.com"이 "foo.example.com"과 매칭 |

### 9.3 URI Loading and Port Mapping (URI 로딩 및 포트 매핑)

"http" URI 로딩 시 UA 동작:

```
1. URI에서 호스트 컴포넌트 추출
2. null이면 → 매칭 없음 반환
3. IP-literal 또는 IPv4address와 구문적으로 매칭되면 → 매칭 없음 반환
4. 추출된 도메인 이름을 Known HSTS Host와 매칭 (9.2 참조)
5. includeSubDomains를 가진 슈퍼도메인 또는 일치 매칭 발견 시:
   - 스킴을 "https"로 교체
   - 명시적 포트 80을 443으로 변환
   - 다른 명시적 포트는 보존
   - 포트 없으면 포트 추가 안 함
```

### 9.4 Secure Transport Errors (보안 전송 오류)

Known HSTS Host에 대한 기본 보안 전송 오류(경고 또는 치명적) 발생 시:

- UA는 연결을 ** 종료해야 함 (MUST)**
- 포함되는 오류:
  - 인증서 유효성 오류 (CRL, OCSP 검사)
  - TLS 서버 ID 검증 실패

### 9.5 HTTP-Equiv Meta Elements

```
UA는 <meta> 요소의 http-equiv="Strict-Transport-Security" 속성 설정을
무시해야 함 (MUST NOT heed)
```

### 9.6 Missing STS Headers (누락된 STS 헤더)

- 보안 채널을 통한 STS 헤더 누락에도 불구하고:
  - max-age 만료까지 호스트를 Known HSTS Host로 계속 취급
  - 사전 구성된 목록의 경우 max-age가 사실상 ** 무한** 일 수 있음

## 10. Constructing Effective Request URI (효과적인 요청 URI 구성)

### 10.1 기본 정의

#### HTTP Request-Line

```
Method SP Request-URI SP HTTP-Version CRLF
```

#### Request-URI 형태

| 형태 | 설명 |
|------|------|
| `*` | 서버 자체에 대한 요청 |
| `absoluteURI` | 전체 URI |
| `abs_path` | 절대 경로 |
| `authority` | 권한 형태 |

#### Host 헤더

```
"Host" ":" host [ ":" port ]
```

### 10.2 Effective Request URI 결정

| Request-URI | 조건 | Effective Request URI |
|-------------|------|----------------------|
| absoluteURI | - | Request-URI 자체 |
| abs_path 또는 `*` | Host 헤더 있음 | 스킴 + "://" + Host 헤더의 host:port + Request-URI |
| abs_path 또는 `*` | Host 헤더 없음 | undefined |
| authority | - | undefined |

- ** 스킴 결정**: 비보안 → "http", 보안 → "https"
- Effective URI 비교는 RFC 2616 Section 3.2.3에 따름
- 빈 경로 컴포넌트는 "/"와 구별됨

### 10.3 Examples (예시)

#### 예시 1

```http
GET /pub/WWW/TheProject.html HTTP/1.1
Host: www.example.org:8080
```

- 비보안 연결
- Effective Request URI: `http://www.example.org:8080/pub/WWW/TheProject.html`

#### 예시 2

```http
OPTIONS * HTTP/1.1
Host: www.example.org
```

- SSL/TLS 연결
- Effective Request URI: `https://www.example.org`

## 12. Server Implementation Advice (서버 구현 조언) - 비규범적

### 12.1 Non-Conformant User Agent Considerations (비적합 UA 고려사항)

- 헤더를 무시하는 비적합 UA는 Section 2.3.1의 공격에 ** 취약**
- 웹 애플리케이션은 HSTS 지원과 관계없이 개발 버그 방지에 ** 동일한 주의** 필요

### 12.2 HSTS Policy Expiration Time (정책 만료 시간)

#### 만료 시간 설정 방법

| 방법 | 설명 |
|------|------|
| **Constant future** | 동일한 max-age 반복 전송 (예: 90일 = 7776000초) |
| **Fixed point** | 원하는 만료까지 남은 시간 계산 |

#### 권장사항

- 도메인 인증서 유효성과 만료 정렬 고려
- 기본값: `max-age=0` (의도적 구성 필요)

### 12.3 HSTS with Self-Signed Certificates (자체 서명 인증서)

#### 연결 실패 조건

다음 ** 네 가지 조건이 모두** 존재하면 연결 실패:

1. 조직이 자체 인증서 생성
2. 루트 CA가 브라우저 저장소에 없음
3. 자체 서명 인증서로 HSTS 활성화
4. TLSA 사용 가능한 연결과 매칭되지 않는 인증서

#### 해결 방안

| 해결책 | 설명 |
|--------|------|
| 루트 CA 배포 | 브라우저/OS 인증서 저장소에 배포 |
| 최종 엔터티 인증서 배포 | 클라이언트에 직접 배포 |
| TLSA 프로토콜 배포 | DANE 사용 |
| 적절한 인증서 사용 | 인정된 CA의 인증서 사용 |

### 12.4 includeSubDomains Implications (서브도메인 포함 의미)

#### 12.4.1 대체 포트/서브도메인의 비보안 서비스

** 문제 예시**: 인증 기관이 서브도메인에서 HTTP CRL/OCSP 서비스 제공

** 해결 방안**:

| 해결책 | 설명 |
|--------|------|
| includeSubDomains 생략 | 서브도메인에 정책 미적용 |
| HTTP 서비스를 TLS/SSL로 제공 | 모든 서비스 암호화 |
| 다른 도메인에서 호스팅 | HTTP 서비스를 다른 도메인으로 이동 |
| 대안 활용 | OCSP 스테이플링 등 |

#### 12.4.2 HSTS 호스트 서브도메인의 웹 애플리케이션

** 문제**: apex에만 STS 헤더가 있고 서브도메인에 별도 애플리케이션이 있는 경우

- 일부 UA는 직접 apex 접촉 없이 정책 강제 안 함

** 해결책**: 모든 잘 알려진 진입점에서 STS 헤더 구성

## 13. User Agent Implementation Advice (UA 구현 조언) - 비규범적

### 13.1 No User Recourse (사용자 구제 없음)

- 경고/오류 시 보안 연결 실패 (Section 8.4)는 **"사용자 구제 없음"** 필요
- ** 우회 다이얼로그 없음**
- 서버 오류로 취급, 재시도 옵션만 제공
- **"클릭스루" 공격** 방지

### 13.2 User-Declared HSTS Policy (사용자 선언 정책)

- 사용자가 도메인을 HSTS 호스트로 ** 명시적 선언** 허용
- 상호작용 전 Known HSTS Host ** 시드(seeding)**
- ** 부트스트랩 MITM 취약점** 완화
- ** 어려움**: 임의 사이트가 URI를 "https"로만 구체화하지 않을 수 있음

### 13.3 HSTS Pre-Loaded List (사전 로드 목록)

- 벤더가 루트 CA 배포와 유사하게 **"공장 로드"** 목록으로 HSTS 정책 사전 구성
- ** 부트스트랩 MITM 취약점** 완화
- 사용자 선언 정책 보완

### 13.4 Disallow Mixed Security Context Loads (혼합 보안 컨텍스트 로드 금지)

- 기본 리소스가 보안으로 로드될 때 비보안 채널로 리소스 가져오기 ** 방지**
- 더 명확한 용어와 UA 간 행동 균일성 정의 표준화 필요

### 13.5 HSTS Policy Deletion (정책 삭제)

- 신중한 UI/보안 설계로 HSTS 호스트별 캐시 엔트리 삭제 ** 활성화**
- ** 우발적 삭제 방지**
- 스크립트 주입이 조용히 엔트리 제거하는 것 ** 방어**

## 15. Security Considerations (보안 고려사항)

### 15.1 Underlying Secure Transport (기본 보안 전송)

- HSTS는 Section 2의 위협 분석에 따라 **TLS 또는 SSL** 을 기본 보안 전송으로 전제
- 다른 프로토콜 사용 시 해당 프로토콜의 보안 모델과 HTTP 레이어링 평가 필요

### 15.2 Non-Conformant User Agent Implications (비적합 UA 영향)

#### Passive 공격 취약점
- Secure가 아닌 쿠키가 있는 비보안 참조로 스니핑 가능

#### Active 공격 취약점
- HSTS 강제 없이 사용자가 경고 "클릭스루"하여 MITM 악용 가능

### 15.3 HSTS Policy Establishment Over Error-Free Transport (오류 없는 전송을 통한 정책 수립)

#### 정책 수립 요구사항
- ** 오류 없는 보안 전송** 필요

#### 이유: MITM 프록시가 할 수 있는 것
- 무단 호스트 기록으로 ** 서비스 거부** 유발
- max-age 조작으로 **TTL 재설정**
- `max-age=0` 전달로 호스트 ** 목록 제거**

#### 함의
- MITM 프록시 뒤의 기업 사용자:
  - Unknown HSTS Host에 대해 레거시 오류 다이얼로그 만남
  - 기록하지 않음
- 오류 없는 연결 성공 후 Known 상태 되면:
  - 이후 프록시 연결 실패

### 15.4 includeSubDomains Necessity (서브도메인 포함 필요성)

#### 문제 시나리오

```
1. 공격자가 있을 법하지 않은 서브도메인 등록
2. 해당 서브도메인으로 HSTS 쿠키와 함께 UA 요청 유발
3. 공격자 인증서 제시
4. 사용자 "클릭스루" 시 쿠키 탈취
```

#### 해결책
- `includeSubDomains`가 암호화를 선제적으로 강제하여 ** 방지**

### 15.5 Denial of Service (서비스 거부)

#### HTTP 전용 애플리케이션 공격

- 공격자가 HTTP 전용 서비스에 HSTS 정책 설정
- 암호화된 연결을 통해 도달할 수 없게 됨

#### 공격 방법

| 방법 | 설명 |
|------|------|
| HTTP 응답 분할/헤더 주입 | 악성 헤더 삽입 |
| 리다이렉트 스푸핑 | 공격자 인증서로 리다이렉트 |
| 사용자 조작 | 사회 공학 |
| 스크립터블 UA 구성 | UA 설정 악용 |

#### includeSubDomains 오용

- 적절한 암호화가 없는 서브도메인이 ** 도달 불가** 가 됨

### 15.6 Bootstrap MITM Vulnerability (부트스트랩 MITM 취약점)

- 사용자가 처음 Unknown HSTS Host에 비보안 "http" URI로 접촉할 때 ** 취약점 존재**
- 초기 상호작용이 비보안 채널 사용, 공격에 취약
- Section 12의 기능/시설로 ** 완화**

### 15.7 Network Time Attacks (네트워크 시간 공격)

- NTP를 전복하는 능동 공격자가 HSTS 효과 ** 약화**
- 영향받는 클라이언트:
  - NTP를 신뢰하는 클라이언트
  - 실시간 시계가 없는 클라이언트
- 공격은 명세 범위를 ** 초과**
- 현대 OS는 기본적으로 NTP 사용 (RFC 4732 Section 2.10 참조)

### 15.8 Bogus Root CA plus DNS Cache Poisoning (가짜 루트 CA + DNS 캐시 포이즈닝)

#### 공격 시나리오

1. 피싱으로 가짜 CA 인증서 설치
2. DNS 캐시 포이즈닝으로 공격자 사이트로 리다이렉트
3. HSTS 정책으로 공격자 사이트 신뢰

#### 완화 방법

| 방법 | 설명 |
|------|------|
| DNS 보안 확장 | RFC 4033 |
| 이메일 피싱 차단 | 피싱 메일 필터링 |
| 인증서 주입 차단 | 악성 CA 설치 방지 |

### 15.9 Creative HSTS Policy Store Manipulation (정책 저장소 조작)

#### 문제

- HSTS 호스트가 도메인 이름 선택으로 서브도메인에 정보 인코딩
- UA 정책 저장소에 캐시됨
- 영리하게 구성된 웹 리소스를 통해 변형 쿼리로 ** 검색 가능**
- 이전 UA 방문 노출 가능 - **"웹 추적"** 기법
- 잠재적 ** 악용 가능**

### 15.10 Internationalized Domain Names (국제화 도메인 이름)

- IDNA 구현 복잡성과 IDNA2003/2008 간 차이가 ** 보안 고려사항** 제시
- UA는 도메인 이름을 적절히 ** 정규화 필수**
- 문자 목록 검증과 유니코드 정규화가 ** 호모그래프 공격** 방지

## 18. Appendix (부록)

### Appendix A: Design Decision Notes (설계 결정 노트)

| 결정 | 이유 |
|------|------|
| 보안 전송을 통해서만 정책 수립 | MITM 주입 방지 |
| max-age는 상대적 TTL | 절대 만료보다 수신 시점 기준이 더 실용적 |
| 도메인 이름의 일치 매칭 | 부모/형제 도메인으로 정책 누출 방지 |
| "사용자 구제 없음" 접근 | 클릭스루 취약점 제거 |
| 선제적 HTTPS 재작성 | 혼합 콘텐츠 취약점 방지 |
| 명시적 포트 처리 | 비표준 포트 보존, 표준 포트 80은 443으로 매핑 |

### Appendix B: Differences between HSTS Policy and Same-Origin Policy

#### 목적 차이

| 정책 | 목적 | 적용 범위 |
|------|------|----------|
| **Same-Origin Policy** (RFC 6454) | 스크립트 리소스 접근 제어, 샌드박싱 | 리소스별 |
| **HSTS Policy** | 전송 계층 보안 강제 | 호스트 전체 (모든 TCP 포트) |

#### 핵심 차이점

- HSTS는 교차 출처 리소스 접근을 ** 제어하지 않음**
- Same-Origin은 리소스별로 적용, HSTS는 호스트 전체에 적용
- 근본적으로 다른 위협 클래스를 해결하는 ** 다른 메커니즘**

| 2년 | 63072000 |

## HSTS Preload List

### 개념

브라우저 공급업체(Chrome, Firefox, Safari 등)는 HSTS 정책이 미리 로드된 도메인 목록을 브라우저에 포함합니다. 이를 통해 부트스트랩 MITM 취약점을 제거합니다.

### Preload 등록 요구사항

1. 유효한 인증서 사용
2. 포트 80에서 443으로 리다이렉트
3. 모든 서브도메인을 HTTPS로 서비스
4. 기본 도메인에서 HSTS 헤더 제공:
   - `max-age`가 최소 1년(31536000초) 이상
   - `includeSubDomains` 디렉티브 포함
   - `preload` 디렉티브 포함

### 예시

```http
Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
```
