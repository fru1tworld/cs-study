# RFC 3022 - Traditional IP Network Address Translator (NAT)

> **발행일**: 2001년 1월
> **폐기 문서**: RFC 1631
> **상태**: Informational

## 1. 개요

RFC 3022는 전통적인 NAT(Network Address Translator)의 동작 원리를 설명하는 문서입니다. NAT는 사설 IP 네트워크와 외부 네트워크 간의 통신을 가능하게 하는 핵심 기술로, 공유기 환경에서 필수적으로 사용됩니다.

### NAT가 필요한 이유

- **주소 고갈 문제**: IPv4 주소의 부족으로 인해 모든 장치에 공인 IP를 할당할 수 없음
- **보안/프라이버시**: 내부 네트워크 구조를 외부로부터 숨김
- **네트워크 토폴로지 변경**: ISP 변경, 조직 합병 등으로 인한 주소 재할당을 투명하게 처리

## 2. NAT의 정의와 동작 원리

NAT는 두 가지 상호보완적인 메커니즘을 통해 동작합니다:

### 기본 동작 흐름

```
[내부 호스트]     [NAT 라우터]      [외부 서버]
10.0.0.5:1234  →  198.76.29.7:5678  →  203.0.113.1:80
     (사설 IP)        (공인 IP)          (목적지)
```

1. **송신 시**: 사설 IP 주소가 공인 IP 주소로 변환됨
2. **수신 시**: 공인 IP 주소가 다시 사설 IP 주소로 역변환됨

## 3. NAT의 유형

### 3.1 Basic NAT (기본 NAT)

사설 IP 주소를 전역 고유 주소로 **1:1 매핑**하는 방식입니다.

**특징**:
- 각 내부 호스트에 대응하는 외부 주소 할당
- 정적 또는 동적 주소 바인딩 지원
- 단일 매핑 주소에서 여러 세션 동시 지원

**예시**:
```
내부 주소          외부 주소
10.0.0.1    ↔    198.76.29.1
10.0.0.2    ↔    198.76.29.2
10.0.0.3    ↔    198.76.29.3
```

### 3.2 NAPT (Network Address Port Translation)

IP 주소와 **포트 번호를 함께 변환**하여 여러 사설 호스트가 하나의 공인 IP를 공유할 수 있게 합니다. **PAT(Port Address Translation)** 또는 **IP Masquerade**라고도 합니다.

**특징**:
- **N:1 매핑**: 여러 내부 호스트가 단일 외부 IP 공유
- SOHO(Small Office/Home Office) 환경에 적합
- 포트 번호로 세션 구분

**예시**:
```
내부 주소:포트              외부 주소:포트
10.0.0.1:1234    ↔    198.76.29.7:5001
10.0.0.2:1234    ↔    198.76.29.7:5002
10.0.0.3:5678    ↔    198.76.29.7:5003
```

**튜플 변환**:
```
(사설 IP, 사설 포트) → (공인 IP, 할당된 포트)
```

## 4. 주소 변환 메커니즘

### 4.1 변환의 3단계

| 단계 | 설명 |
|------|------|
| **Address Binding (바인딩)** | 첫 아웃바운드 세션 시작 시 매핑 생성 |
| **Address Lookup & Translation (조회 및 변환)** | 후속 패킷에 대해 매핑 조회 후 변환 적용 |
| **Address Unbinding (해제)** | 마지막 세션 종료 시 매핑 해제 |

### 4.2 패킷 수정 상세

NAT는 여러 프로토콜 계층에서 수정을 수행합니다:

#### IP 헤더 변경
- **아웃바운드**: 소스 주소 변환
- **인바운드**: 목적지 주소 변환
- IP 체크섬 재계산 필요

#### TCP/UDP 변경
```
TCP/UDP 체크섬은 소스와 목적지 IP 주소를 포함하는
의사 헤더(pseudo header)도 커버하므로 반드시 재계산해야 함
```

#### ICMP 처리
- **Query 패킷**: 식별자(Identifier) 변환
- **Error 메시지**: 페이로드 내 임베디드 패킷 수정

### 4.3 체크섬 조정 알고리즘

전체 체크섬을 재계산하는 대신, **1의 보수 연산 특성**을 활용한 효율적인 방법:

```c
void checksum_adjust(unsigned char *chksum, unsigned char *optr,
                     int olen, unsigned char *nptr, int nlen) {
    long x, old, new_;
    x = chksum[0] * 256 + chksum[1];
    x = ~x & 0xFFFF;
    while (olen) {
        old = optr[0] * 256 + optr[1];
        optr += 2;
        x -= old & 0xFFFF;
        if (x <= 0) { x--; x &= 0xFFFF; }
        olen -= 2;
    }
    while (nlen) {
        new_ = nptr[0] * 256 + nptr[1];
        nptr += 2;
        x += new_ & 0xFFFF;
        if (x & 0x10000) { x++; x &= 0xFFFF; }
        nlen -= 2;
    }
    x = ~x & 0xFFFF;
    chksum[0] = x / 256;
    chksum[1] = x & 0xFF;
}
```

## 5. 세션/연결 관리

### 5.1 연결 상태 유지

NAT 구현체는 다음 정보를 추적하는 상태 테이블을 유지합니다:

| 정보 | 설명 |
|------|------|
| 주소/포트 튜플 매핑 | 내부↔외부 주소 대응 관계 |
| 세션 생명주기 정보 | 연결 시작/종료 상태 |
| 타임아웃 값 | 연결 만료 시간 |

### 5.2 FTP ALG (Application Layer Gateway)

FTP는 특별한 처리가 필요합니다:

```
1. FTP 제어 세션 페이로드 모니터링
2. PORT 명령 및 PASV 응답 감지
3. TCP 시퀀스/ACK 번호에 대한 델타 조정 유지
4. FTP 프로토콜 메시지 내 주소/포트 옥텟 변환
```

### 5.3 DNS 고려사항

NAT별 DNS 변환 대신 아키텍처 분리 권장:

- **내부 DNS 서버**: 사설 호스트 매핑 유지
- **외부 DNS 서버**: 공인 주소만 처리
- **내부 DNS 없는 사설 네트워크**: 외부로 요청 전달

## 6. 제한사항 및 고려사항

### 6.1 필수 제약사항

| 제약 | 설명 |
|------|------|
| **세션 라우팅 요구** | 세션의 모든 요청/응답은 동일한 NAT 라우터를 통해 라우팅되어야 함 |
| **주소 공간 분리** | 사설과 전역 라우팅 가능 주소 공간은 완전히 분리되어야 함 |
| **TCP/UDP 단편화** | NAPT는 단편화된 TCP/UDP 패킷 처리에 어려움 (첫 조각만 포트 정보 포함) |

### 6.2 보안 및 프라이버시 트레이드오프

**장점**:
- 단방향 프라이버시 제공 (사설 호스트 주소 숨김)

**단점**:
- 내부 호스트의 악의적 활동 시 포렌식 분석 복잡화
- NAT 라우터 주소가 실제 공격자를 가림

### 6.3 IP 옵션 처리

NAT 장치는 다음 IP 옵션을 지원하지 않거나 주소를 변환하지 않을 수 있음:
- Record Route
- Loose Source Routing

→ 사설 주소가 종단 간 노출될 가능성

### 6.4 ARP 및 LAN 인터페이스

NAT 매핑 주소가 직접 연결된 LAN 세그먼트에 있을 때:
- NAT 라우터는 자신의 MAC 주소로 ARP 요청에 응답해야 함 (Proxy ARP)

## 7. 사설 주소 공간

RFC 1918에서 지정한 사설 IP 주소 블록:

| 클래스 | 주소 범위 | CIDR |
|--------|-----------|------|
| A | 10.0.0.0 ~ 10.255.255.255 | 10.0.0.0/8 |
| B | 172.16.0.0 ~ 172.31.255.255 | 172.16.0.0/12 |
| C | 192.168.0.0 ~ 192.168.255.255 | 192.168.0.0/16 |

## 8. 아키텍처 권장사항

### 경계 라우터 배치

```
[사설 네트워크] ←→ [NAT (경계 라우터)] ←→ [인터넷]
     10.x.x.x              변환              공인 IP
```

- NAT는 스텁 도메인 경계에만 배치
- 사설 네트워크는 외부에 광고되지 않음
- 전역 라우팅 가능 주소만 백본 라우팅 인프라에 전파

### Basic NAT에서 NAPT로의 전환

Basic NAT 주소 풀 고갈 시 자동으로 NAPT로 전환하는 우아한 폴백 메커니즘 구현 가능
(단, 애플리케이션 동작 기대가 변경될 수 있음)

## 9. 참고 구현체

- **Linux**: IP masquerade
- **FreeBSD**: NAPT daemon

## 요약

NAT는 IPv4 주소 고갈 문제를 해결하고 네트워크 보안을 강화하는 핵심 기술입니다. Basic NAT는 1:1 주소 변환을, NAPT는 포트를 활용한 N:1 변환을 제공하며, 대부분의 가정용/소규모 사무실 환경에서 NAPT가 사용됩니다.
