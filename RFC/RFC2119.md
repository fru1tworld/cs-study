# RFC 2119: Key words for use in RFCs to Indicate Requirement Levels

> RFC 문서에서 요구사항 수준을 나타내기 위해 사용되는 핵심 단어들

## 문서 정보

| 항목 | 내용 |
|------|------|
| **RFC 번호** | 2119 |
| ** 분류** | BCP 14 (Best Current Practice) |
| ** 작성자** | S. Bradner (Harvard University) |
| ** 발행일** | 1997년 3월 |
| ** 상태** | 현행 표준 |

## 개요

많은 표준 문서에서 명세의 요구사항을 나타내기 위해 여러 단어들이 사용됩니다. 이러한 단어들은 종종 대문자로 표기됩니다. RFC 2119는 IETF 문서에서 이 단어들이 어떻게 해석되어야 하는지를 정의합니다.

이 문서는 ** 모든 RFC의 범례** 가 되는 메타 문서로, RFC 문서를 읽기 전에 반드시 이해해야 합니다.

---

## 핵심 키워드 정의

### 1. MUST / REQUIRED / SHALL (필수)

```
강제성: ████████████ 100% (절대적 요구사항)
```

** 정의**: 해당 정의가 명세의 ** 절대적 요구사항** 임을 의미합니다.

- 반드시 구현해야 함
- 예외가 허용되지 않음
- 미준수 시 명세를 위반하는 것으로 간주

** 예시**:
- "모든 구현체는 이 프로토콜을 지원해야 한다(MUST support)"
- "클라이언트는 유효한 인증서를 제시해야 한다(SHALL present)"

---

### 2. MUST NOT / SHALL NOT (금지)

```
강제성: ████████████ 100% (절대적 금지)
```

** 정의**: 해당 정의가 명세의 ** 절대적 금지사항** 임을 의미합니다.

- 절대 해서는 안 됨
- 어떤 상황에서도 금지
- 이를 어기면 명세를 위반하는 것

** 예시**:
- "서버는 이 필드를 수정해서는 안 된다(MUST NOT modify)"
- "구현체는 이 오래된 알고리즘을 사용해서는 안 된다(SHALL NOT use)"

---

### 3. SHOULD / RECOMMENDED (권장)

```
강제성: ████████░░░░ 70% (강한 권장)
```

** 정의**: 특정 상황에서 특정 항목을 무시할 ** 타당한 이유가 존재할 수 있음** 을 의미합니다. 그러나 다른 방향을 선택하기 전에 ** 전체적인 영향을 이해하고 신중하게 검토** 해야 합니다.

- 일반적으로 따라야 함
- 예외적인 상황에서만 무시 가능
- 무시하기 전에 영향을 완전히 이해해야 함

** 예시**:
- "구현체는 이 최적화를 지원하는 것이 좋다(SHOULD support)"
- "새로운 알고리즘을 사용하는 것이 권장된다(RECOMMENDED)"

---

### 4. SHOULD NOT / NOT RECOMMENDED (비권장)

```
강제성: ████████░░░░ 70% (강한 비권장)
```

** 정의**: 특정 행동이 허용되거나 심지어 유용할 수 있는 ** 타당한 이유가 특정 상황에서 존재할 수 있음** 을 의미합니다. 그러나 이 라벨이 붙은 동작을 구현하기 전에 ** 전체적인 영향을 이해하고 신중하게 검토** 해야 합니다.

- 일반적으로 하지 않아야 함
- 특별한 사유가 있는 경우에만 예외 가능
- 예외를 적용하기 전에 신중한 검토 필요

** 예시**:
- "이 방식은 사용하지 않는 것이 좋다(SHOULD NOT use)"
- "이 옵션은 권장되지 않는다(NOT RECOMMENDED)"

---

### 5. MAY / OPTIONAL (선택)

```
강제성: ░░░░░░░░░░░░ 0% (완전한 선택사항)
```

** 정의**: 해당 항목이 ** 완전히 선택적** 임을 의미합니다. 한 벤더는 특정 시장에서 요구하거나 제품 향상을 위해 해당 항목을 포함할 수 있고, 다른 벤더는 같은 항목을 생략할 수 있습니다.

** 중요한 상호운용성 요구사항**:
- 특정 옵션을 포함하지 ** 않은** 구현체는 해당 옵션을 포함한 다른 구현체와 ** 상호운용할 준비가 되어 있어야 한다(MUST)**
- 특정 옵션을 ** 포함한** 구현체는 해당 옵션을 포함하지 않은 다른 구현체와 ** 상호운용할 준비가 되어 있어야 한다(MUST)**

** 예시**:
- "구현체는 이 확장을 지원할 수 있다(MAY support)"
- "이 필드는 선택적이다(OPTIONAL)"

---

## 요약 비교표

| 키워드 | 한국어 | 강제성 | 설명 |
|--------|--------|--------|------|
| **MUST** / REQUIRED / SHALL | 필수 | 100% | 절대적 요구사항 |
| **MUST NOT** / SHALL NOT | 금지 | 100% | 절대적 금지사항 |
| **SHOULD** / RECOMMENDED | 권장 | ~70% | 강하게 권장하나 예외 가능 |
| **SHOULD NOT** / NOT RECOMMENDED | 비권장 | ~70% | 강하게 비권장하나 예외 가능 |
| **MAY** / OPTIONAL | 선택 | 0% | 완전한 선택사항 |

## 사용 지침

### 문서에 포함할 표준 문구

이 가이드라인을 따르는 저자는 문서 시작 부분에 다음 문구를 포함해야 합니다:

> 이 문서에서 사용된 "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", "OPTIONAL"이라는 키워드는 RFC 2119에 기술된 대로 해석되어야 합니다.

### 키워드 사용 시 주의사항

이 키워드들은 ** 신중하고 절제 있게** 사용해야 합니다:

1. ** 상호운용성에 실제로 필요한 경우** 에만 사용
2. ** 해로운 동작을 제한** 하기 위해 필요한 경우에만 사용
3. 상호운용성에 필요하지 않은데 ** 특정 구현 방법을 강제하기 위해 사용해서는 안 됨**

```
❌ 잘못된 사용: 특정 구현 방법 강제
   "구현체는 연결 리스트를 사용해야 한다(MUST use)"

✅ 올바른 사용: 상호운용성을 위한 요구사항
   "구현체는 UTF-8 인코딩을 지원해야 한다(MUST support)"
```

## 보안 고려사항

이 용어들은 ** 보안과 관련된 동작** 을 명시하는 데 자주 사용됩니다.

** 주의**: MUST나 SHOULD를 구현하지 않거나, MUST NOT이나 SHOULD NOT으로 명시된 것을 수행했을 때의 ** 보안에 대한 영향은 매우 미묘** 할 수 있습니다.

문서 저자는 다음을 해야 합니다:
- 권장사항이나 요구사항을 따르지 않았을 때의 ** 보안 영향을 상세히 설명**
- 대부분의 구현자들이 명세를 만든 경험과 논의의 혜택을 받지 못했음을 고려

## 실제 활용 예시

### HTTP/2 명세에서의 사용 예

```
클라이언트는 HTTP/2 연결을 열기 전에 서버가 HTTP/2를 지원하는지
확인해야 한다(MUST).

클라이언트는 알 수 없는 프레임 타입을 무시해야 한다(MUST).

서버는 GOAWAY 프레임을 보내기 전에 진행 중인 스트림 처리를
완료하는 것이 좋다(SHOULD).

구현체는 흐름 제어 윈도우 크기를 조정할 수 있다(MAY).
```

### TLS 1.3 명세에서의 사용 예

```
구현체는 TLS 1.2 이하 버전의 압축 메서드를 지원해서는 안 된다
(MUST NOT).

서버는 인증서 요청 시 신뢰할 수 있는 CA 목록을 제공하는 것이
좋다(SHOULD).

클라이언트는 세션 재개를 위해 PSK를 사용할 수 있다(MAY).
```
