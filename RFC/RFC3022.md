## 초록 (Abstract)

- NAT(Network Address Translator)의 두 가지 방법론 설명
  - **Basic NAT**: 주소 매핑만 수행
  - **NAPT**: 주소와 포트 매핑을 함께 수행
- 사설 주소를 가진 영역(realm)과 전역 등록 주소를 가진 외부 영역을 연결하는 메커니즘 제공

## 1. 소개 (Introduction)

### 1.1 NAT의 배경

- 내부 IP 주소가 외부에서 사용될 수 없는 시나리오 해결
  - 프라이버시 요구사항
  - 주소의 유효성 문제 (사설 주소는 인터넷에서 라우팅 불가)
- NAT를 통해 조직의 내부 토폴로지 변경을 외부 사용자에게 숨김
- 경계 라우터(Border Router)에서 주소 변환 집중 수행

### 1.2 주요 사용 사례

- ** 간헐적 외부 접근이 필요한 조직**
  - 가끔씩 외부 네트워크 접근이 필요한 환경
  - 전체 호스트에 공인 IP 할당이 비효율적인 경우
- **SOHO (Small Office/Home Office)**
  - 여러 노드가 하나의 IP 주소 공유
  - 제한된 공인 IP 주소 할당 환경 (ISP에서 단일 IP만 제공)
- ** 투명한 접근**
  - 호스트 소프트웨어 수정 없이 외부 네트워크 접근
  - 라우터 수정 없이 동작

### 1.3 핵심 요구사항

- ** 세션 일관성**: 세션에 속한 모든 요청과 응답이 ** 반드시 동일한 NAT 라우터** 를 통해 라우팅되어야 함
- ** 스텁 도메인(Stub Domain)**: NAT가 적용되는 영역의 유일한 출구점(sole exit point)

### 1.4 NAT의 역할

```
내부 네트워크 (사설 주소)     NAT 장치           외부 네트워크 (공인 주소)
      [Stub Domain]    <-->  [Border Router]  <-->  [Internet]
```

## 2. Traditional NAT 개요 (Overview of Traditional NAT)

### 2.1 Basic NAT

#### 동작 방식
- 사설 주소를 전역 유효 주소로 동적 매핑
- **1:1 주소 매핑**: 하나의 사설 주소가 하나의 공인 주소에 매핑

#### 특징
- 동일 매핑에서 여러 동시 세션 가능
- 정적 매핑 지원: 보장된 외부 접근을 위해 설정 가능
- 다중 출구점 사용 시 동일한 변환 테이블 유지 필요

#### 주소 풀 관리
- 전역 주소 풀(NAT Pool)에서 동적으로 주소 할당
- 주소 고갈 시 새로운 세션 생성 불가

#### Basic NAT 동작 예시

```
                       사설 네트워크              NAT               외부 네트워크
                      ┌─────────────┐         ┌─────┐         ┌──────────────┐
Stub A:               │ 10.33.96.5  │ ──────> │ NAT │ ──────> │ 198.76.29.7  │
                      └─────────────┘         │  A  │         └──────────────┘
                                              └─────┘
                                                 │
                                                 ▼
                      ┌─────────────┐         ┌─────┐         ┌──────────────┐
Stub B:               │ 10.81.13.22 │ <────── │ NAT │ <────── │ 198.76.28.4  │
                      └─────────────┘         │  B  │         └──────────────┘
                                              └─────┘

[Outbound] 출발지: 10.33.96.5 → 198.76.29.7 (변환됨)
           목적지: 198.76.28.4 (변환 없음)
[Inbound]  출발지: 198.76.28.4 (변환 없음)
           목적지: 198.76.29.7 → 10.33.96.5 (역변환됨)
```

### 2.2 NAPT (Network Address Port Translation)

#### 동작 방식
- (사설 IP, 로컬 포트) 튜플을 (등록된 IP, 할당 포트) 튜플로 매핑
- **N:1 주소 매핑**: 여러 내부 호스트가 단일 외부 주소 공유

#### 특징
- SOHO 환경에 필수적
- **PAT (Port Address Translation)** 또는 **IP Masquerade** 라고도 함
- 단일 공인 IP로 다수의 내부 호스트 지원

#### 프로토콜 지원
- **TCP/UDP 세션**: 아웃바운드 세션 우선 지원
- **ICMP 쿼리 패킷**: 식별자(Identifier) 필드 변환
- ** 인바운드 접근**: 정적 포트 매핑을 통해 가능

#### NAPT 동작 예시

```
내부 네트워크                    NAPT                         외부 네트워크
┌──────────────────┐         ┌───────────────────────┐
│ 10.0.0.10:3017   │ ──────> │ 138.76.28.4:1024      │ ──────> [Internet]
│ 10.0.0.10:3018   │ ──────> │ 138.76.28.4:1025      │ ──────>
│ 10.0.0.20:3017   │ ──────> │ 138.76.28.4:1026      │ ──────>
└──────────────────┘         └───────────────────────┘

변환 테이블:
| 내부 주소       | 내부 포트 | 외부 주소      | 외부 포트 | 프로토콜 |
|-----------------|-----------|----------------|-----------|----------|
| 10.0.0.10       | 3017      | 138.76.28.4    | 1024      | TCP      |
| 10.0.0.10       | 3018      | 138.76.28.4    | 1025      | TCP      |
| 10.0.0.20       | 3017      | 138.76.28.4    | 1026      | TCP      |
```

### 2.3 Basic NAT vs NAPT 비교

| 특성 | Basic NAT | NAPT |
|------|-----------|------|
| 매핑 방식 | 1:1 (IP 대 IP) | N:1 (IP:Port 대 IP:Port) |
| 공인 IP 요구량 | 내부 호스트 수만큼 필요 | 최소 1개로 다수 지원 |
| 포트 변환 | 없음 | 있음 |
| 사용 환경 | 중대형 기업 | SOHO, 가정 |
| 프로토콜 호환성 | 높음 | 일부 제한 |

## 3. NAT 용어 (NAT Terminology)

RFC 2663에서 정의된 NAT 관련 핵심 용어:

### 3.1 주소 용어

| 용어 | 영문 | 설명 |
|------|------|------|
| 내부 로컬 주소 | Inside Local Address | 내부 네트워크에서 사용하는 내부 호스트의 IP 주소 (사설 주소) |
| 내부 전역 주소 | Inside Global Address | 외부에서 보이는 내부 호스트의 IP 주소 (NAT 변환 후 공인 주소) |
| 외부 로컬 주소 | Outside Local Address | 내부에서 보이는 외부 호스트의 IP 주소 |
| 외부 전역 주소 | Outside Global Address | 외부 네트워크에서 사용하는 외부 호스트의 IP 주소 |

### 3.2 기타 용어

| 용어 | 영문 | 설명 |
|------|------|------|
| NAT 풀 | NAT Pool | NAT 변환에 사용 가능한 전역 주소 집합 |
| 주소 바인딩 | Address Binding | 사설 주소와 전역 주소 간의 연관 관계 |
| 세션 | Session | 특정 전송 계층 연결 또는 연결 없는 통신 흐름 |

### 3.3 주소 변환 예시

```
┌─────────────────────────────────────────────────────────────────────┐
│                        주소 변환 테이블                              │
├─────────────────┬─────────────────┬─────────────────┬───────────────┤
│ Inside Local    │ Inside Global   │ Outside Global  │ 상태          │
├─────────────────┼─────────────────┼─────────────────┼───────────────┤
│ 10.0.0.10:3017  │ 138.76.28.4:1024│ 64.233.160.0:80 │ ESTABLISHED   │
│ 10.0.0.10:3018  │ 138.76.28.4:1025│ 64.233.160.0:443│ ESTABLISHED   │
│ 10.0.0.20:4000  │ 138.76.28.4:1026│ 151.101.1.69:80 │ TIME_WAIT     │
└─────────────────┴─────────────────┴─────────────────┴───────────────┘
```

## 4. NAT 동작 (NAT Operation)

### 4.1 세션 변환 단계 (Translation Phases)

#### 4.1.1 주소 바인딩 (Address Binding)

- **Basic NAT**: 첫 번째 아웃바운드 세션 시작 시 사설 주소가 외부 주소에 바인딩
- **NAPT**: (로컬 IP, 로컬 TU 포트) 튜플이 (등록된 IP, 할당 TU 포트) 튜플에 바인딩
- 후속 세션은 기존 바인딩 재사용

```
[첫 번째 세션]
내부 호스트 10.0.0.10:3017 → NAT 바인딩 생성 → 138.76.28.4:1024

[후속 세션]
내부 호스트 10.0.0.10:3018 → 기존 IP 바인딩 재사용 → 138.76.28.4:1025
```

#### 4.1.2 주소 조회 및 변환 (Address Lookup and Translation)

- 세션 기반 소프트 스테이트(Soft State) 유지
- 패킷별로 바인딩 테이블 조회
- 헤더 필드 변환 수행

#### 4.1.3 주소 언바인딩 (Address Unbinding)

- 마지막 세션 종료 시 바인딩 해제
- 타임아웃 기반 자동 해제
  - **TCP**: FIN/RST 또는 유휴 타임아웃
  - **UDP**: 유휴 타임아웃 (일반적으로 수 분)
  - **ICMP**: 응답 수신 또는 타임아웃

### 4.2 주소 변환 (Address Translation)

#### 4.2.1 헤더 조작 (Header Manipulations)

**Basic NAT**
- IP 주소 필드 수정
- IP 헤더 체크섬 재계산

**TCP/UDP**
- 전송 계층 체크섬 업데이트
- UDP 체크섬이 0인 경우 예외 (체크섬 계산 건너뜀)

**NAPT 추가 작업**
- 포트 번호 변환
- ICMP 쿼리 식별자 변환

#### 4.2.2 체크섬 조정 알고리즘 (Checksum Adjustment Algorithm)

- 효율적인 체크섬 재계산을 위한 증분 업데이트 방식
- 변환 전후 주소의 산술적 차이를 계산하여 체크섬에 추가

```
원리: "변환 전 주소와 변환 후 주소 간의 산술적 차이를 계산하고
       이를 체크섬에 더하는 것으로 충분함"

조건: 짝수 오프셋과 길이에서만 적용 가능
```

**C 코드 예시 (RFC 원문)**
```c
void checksumadjust(unsigned char *chksum, unsigned char *optr,
                    int olen, unsigned char *nptr, int nlen) {
    long x, old, new;
    x = chksum[0] * 256 + chksum[1];
    x = ~x & 0xFFFF;
    while (olen) {
        old = optr[0] * 256 + optr[1];
        optr += 2;
        x -= old & 0xFFFF;
        if (x <= 0) { x--; x &= 0xFFFF; }
        olen -= 2;
    }
    while (nlen) {
        new = nptr[0] * 256 + nptr[1];
        nptr += 2;
        x += new & 0xFFFF;
        if (x & 0x10000) { x++; x &= 0xFFFF; }
        nlen -= 2;
    }
    x = ~x & 0xFFFF;
    chksum[0] = x / 256;
    chksum[1] = x & 0xFF;
}
```

### 4.3 ICMP 패킷 수정 (ICMP Packet Modifications)

#### 4.3.1 ICMP 쿼리 패킷
- Echo Request/Reply, Timestamp 등
- NAPT에서 식별자(Identifier) 필드 변환 필요

#### 4.3.2 ICMP 에러 패킷

ICMP 에러 메시지는 원본 패킷의 일부를 포함하므로 ** 다중 레이어 수정 필요**

```
ICMP 에러 패킷 구조:
┌─────────────────────────────────────────┐
│ 외부 IP 헤더 (수정 필요)                │
├─────────────────────────────────────────┤
│ ICMP 헤더 (체크섬 수정 필요)            │
├─────────────────────────────────────────┤
│ 내장된 IP 헤더 (주소/체크섬 수정 필요)  │
├─────────────────────────────────────────┤
│ 내장된 TCP/UDP 헤더 (포트 수정 - NAPT)  │
└─────────────────────────────────────────┘
```

** 수정 항목**
1. 외부 IP 헤더의 주소 및 체크섬
2. ICMP 헤더 체크섬
3. 내장된 IP 헤더의 주소 및 체크섬
4. NAPT의 경우: 내장된 TCP/UDP 포트 또는 ICMP 쿼리 ID

### 4.4 IP 단편화 고려사항 (IP Fragmentation Considerations)

#### 4.4.1 Basic NAT의 단편화 처리
- IP 주소만 변환하므로 단편화 처리 가능
- 모든 단편에 동일한 주소 변환 적용

#### 4.4.2 NAPT의 단편화 문제

** 심각한 제한사항**
- 첫 번째 단편만 전송 계층 헤더(포트 정보) 포함
- 후속 단편은 단편화 ID만 보유

** 충돌 시나리오**
```
호스트 A (10.0.0.10) → 단편 ID: 1234 → NAT → ?
호스트 B (10.0.0.20) → 단편 ID: 1234 → NAT → ?

문제: 동일한 단편 ID를 가진 두 호스트의 단편을
      NAT가 구분할 수 없음 → 두 세션 모두 손상
```

**DF 비트 처리**
- DF(Don't Fragment) 비트 설정 패킷은 단편화되지 않음
- Path MTU Discovery에 영향

### 4.5 세션 매핑 (Session Mapping)

#### 4.5.1 세션 상태 테이블

NAT 구현체가 유지하는 정보:

| 정보 | 설명 |
|------|------|
| 주소/포트 튜플 매핑 | 내부↔외부 주소 대응 관계 |
| 세션 생명주기 정보 | 연결 시작/종료 상태 |
| 타임아웃 값 | 연결 만료 시간 |
| 프로토콜 유형 | TCP, UDP, ICMP 등 |

#### 4.5.2 타임아웃 처리

| 프로토콜 | 타임아웃 조건 |
|----------|---------------|
| TCP | FIN/RST 수신 또는 유휴 타임아웃 (일반적으로 24시간) |
| UDP | 유휴 타임아웃 (일반적으로 수 분) |
| ICMP | 응답 수신 또는 짧은 타임아웃 |

## 5. NAT 제한사항 (NAT Limitations)

### 5.1 End-to-End 연결성 문제

- 진정한 end-to-end 연결 불가
- 외부에서 내부로의 세션 시작이 어려움
- P2P 애플리케이션에 문제 발생
- ** 해결책**: 정적 매핑, 포트 포워딩, NAT Traversal 기술

### 5.2 응용 계층 프로토콜 문제

#### IP 주소가 페이로드에 포함된 경우
- **FTP**: PORT/PASV 명령에 IP 주소 포함
- **SIP, H.323**: VoIP 시그널링에 IP 주소 포함
- ** 일부 게임 프로토콜**: 직접 연결 정보 교환

#### 해결책
- **ALG (Application Level Gateway)** 구현
- 프로토콜별 특수 처리

### 5.3 프라이버시와 보안 (Privacy and Security)

#### 5.3.1 프라이버시 향상
- 내부 호스트 주소 은닉
- 네트워크 토폴로지 보호
- 외부에서 내부 구조 파악 어려움

#### 5.3.2 보안 역설
- 익명성으로 인한 남용 추적 어려움
- 보안 위반 조사 복잡성 증가
- 로깅 및 감사 제한

```
프라이버시 vs 추적성:
┌─────────────────────────────────────────────────────┐
│ [장점] 외부에서 내부 구조 파악 불가                 │
│ [단점] 내부 호스트의 외부 남용 시 추적 어려움       │
└─────────────────────────────────────────────────────┘
```

### 5.4 FTP 예시 (FTP as an Example)

#### FTP ALG (Application Level Gateway) 필요성
- FTP는 제어 세션에서 데이터 연결 주소/포트를 전송
- 페이로드에 포함된 IP 주소 변환 필요

#### FTP ALG 동작

```
FTP 제어 세션 모니터링:
┌─────────────────────────────────────────────────────────────┐
│ 1. PORT 명령 감지                                          │
│    PORT 10,0,0,10,12,45 → PORT 138,76,28,4,4,0            │
│    (10.0.0.10:3117 → 138.76.28.4:1024)                    │
├─────────────────────────────────────────────────────────────┤
│ 2. PASV 응답 감지                                          │
│    227 Entering Passive Mode (10,0,0,10,12,45)            │
│    → 227 Entering Passive Mode (138,76,28,4,4,0)          │
└─────────────────────────────────────────────────────────────┘
```

#### TCP 시퀀스/ACK 번호 조정
- 페이로드 길이 변경으로 인한 시퀀스 번호 불일치 보정
- 특수 테이블로 델타 값 관리

```
시퀀스 번호 조정:
- 아웃바운드: 시퀀스 델타 증가
- 인바운드: ACK 번호 델타 감소

예시:
원본 PORT 명령: "PORT 10,0,0,10,12,45\r\n" (22바이트)
변환 PORT 명령: "PORT 138,76,28,4,4,0\r\n" (24바이트)
델타: +2 바이트 → 후속 패킷 시퀀스 번호 조정 필요
```

### 5.5 IPsec과의 호환성 문제

- NAT가 IP 헤더를 수정하므로 무결성 검증 실패
- **AH (Authentication Header)**: 완전 비호환 (IP 헤더 포함 인증)
- **ESP (Encapsulating Security Payload)**: 터널 모드 제한적 지원

```
IPsec 호환성 문제:
┌─────────────────────────────────────────────────────┐
│ AH 모드: IP 헤더 포함 인증 → NAT 변환 시 인증 실패 │
│ ESP 모드: 페이로드만 암호화 → 터널 모드에서 제한적  │
└─────────────────────────────────────────────────────┘

해결책: NAT-T (NAT Traversal) - UDP 캡슐화
```

### 5.6 ARP 응답 (ARP Responses on LAN Interfaces)

- 전역 주소 공간이 직접 연결된 LAN 서브넷과 겹칠 때 발생
- NAT 장치가 매핑된 전역 주소에 대해 **Proxy ARP** 응답 필수

```
Proxy ARP 동작:
┌─────────────────────────────────────────────────────┐
│ ARP 요청: "138.76.28.4의 MAC 주소는?"               │
│ NAT 응답: "내 MAC 주소로 보내라" (Proxy ARP)        │
└─────────────────────────────────────────────────────┘
```

### 5.7 단방향 세션 (Unidirectional Sessions)

- 기본적으로 사설 도메인에서 외부로 나가는 세션만 지원
- 외부에서 내부로의 세션 시작이 어려움
- ** 해결책**: 정적 매핑 또는 포트 포워딩

## 6. 기타 고려사항 (Miscellaneous Issues)

### 6.1 주소 공간 분할 (Address Space Partitioning)

- ** 핵심 원칙**: 주어진 주소는 사설 주소이거나 전역 주소 중 하나여야 함
- 중첩 없음: 라우팅 모호성 방지
- 명확한 경계 정의 필수

### 6.2 사설 주소 공간 (Private Address Space)

RFC 1918에서 할당한 세 개의 사설 주소 블록:

| 블록 | 주소 범위 | CIDR 표기 | 호스트 수 |
|------|-----------|-----------|-----------|
| Class A | 10.0.0.0 - 10.255.255.255 | 10.0.0.0/8 | 16,777,214 |
| Class B | 172.16.0.0 - 172.31.255.255 | 172.16.0.0/12 | 1,048,574 |
| Class C | 192.168.0.0 - 192.168.255.255 | 192.168.0.0/16 | 65,534 |

- IANA 협의 없이 사용 가능
- 인터넷에서 라우팅되지 않음

### 6.3 NAT를 통한 라우팅 (Routing Across NAT)

- NAT 라우터는 사설 네트워크를 외부에 광고하지 않아야 함
- 일반적으로 서비스 제공자 라우터로의 정적 라우트 사용
- 라우팅 프로토콜 제한:
  - BGP에서 사설 네트워크 광고 금지
  - 내부 라우팅 프로토콜과 외부 라우팅 프로토콜 분리

### 6.4 Basic NAT에서 NAPT로 전환 (Basic NAT to NAPT Switch-Over)

- 전역 주소 고갈 시 선택적 NAPT 전환 가능
- 접근 거부 방지 목적
- 단, 애플리케이션 호환성 저하 가능

```
주소 풀 상태 모니터링:
┌─────────────────────────────────────┐
│ 주소 풀: 5개 중 5개 사용 중         │
│ → NAPT 모드로 자동 전환             │
│ → 단일 주소로 다중 세션 지원        │
└─────────────────────────────────────┘
```

### 6.5 DNS 지원 (DNS Support)

- 내부 DNS 서버: 사설 영역 매핑 유지
- 외부 DNS 서버: 공인 주소 처리
- 일반적인 아웃바운드 세션에서는 DNS ALG 불필요

```
DNS 구성:
┌─────────────────┐     ┌─────────────────┐
│ 내부 DNS 서버   │     │ 외부 DNS 서버   │
│ (사설 주소 관리)│     │ (공인 주소 관리)│
└────────┬────────┘     └────────┬────────┘
         │                       │
         ▼                       ▼
    ┌─────────┐             ┌─────────┐
    │10.0.0.10│             │외부 호스트│
    └─────────┘             └─────────┘
```

### 6.6 IP 옵션 처리 (IP Option Handling)

- 라우터가 무시하거나 변환하지 않을 수 있음
- 영향 받는 옵션:
  - Record Route
  - Strict Source Route
  - Loose Source Route
- 사설 주소가 노출될 수 있으나 통과성에는 영향 없음

## 요약

RFC 3022는 Traditional NAT의 두 가지 주요 유형인 **Basic NAT** 와 **NAPT** 를 정의하고, 그 동작 방식, 구현 고려사항, 제한사항을 상세히 설명한다.

### 핵심 포인트

1. **Basic NAT**: 1:1 IP 주소 매핑, 공인 IP 풀 필요
2. **NAPT**: N:1 매핑으로 단일 공인 IP에 다수 호스트 지원
3. ** 세션 일관성**: 모든 세션 트래픽은 동일 NAT 라우터 통과 필수
4. **ALG 필요**: FTP 등 페이로드에 IP 포함하는 프로토콜
5. ** 제한사항**: End-to-end 연결성, IPsec 호환성, 단편화 문제

IPv4 주소 고갈 문제에 대한 실용적인 해결책으로 널리 채택되었으며, 특히 NAPT는 가정 및 소규모 사무실 환경에서 필수적인 기술이 되었다. 그러나 end-to-end 연결성 저하, 특정 프로토콜과의 비호환성, IPsec 문제 등의 제한사항이 존재하며, 이러한 문제들은 NAT-T, ALG 등의 보완 기술로 해결하고 있다.
