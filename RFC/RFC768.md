# RFC 768: User Datagram Protocol (UDP)

## 개요

RFC 768은 **사용자 데이터그램 프로토콜(UDP)**을 정의하는 문서입니다. 1980년 8월에 발표된 이 문서는 매우 짧으며, UDP가 얼마나 단순한 프로토콜인지를 직접 확인할 수 있습니다.

> "이 프로토콜은 애플리케이션 프로그램이 최소한의 프로토콜 메커니즘으로 다른 프로그램에 메시지를 보낼 수 있는 절차를 제공한다."

## 핵심 목적

UDP는 IP(Internet Protocol)를 기반으로 상호 연결된 컴퓨터 네트워크 간에 **데이터그램 통신**을 가능하게 합니다.

## 주요 특징

| 특징 | 설명 |
|-----|------|
| **트랜잭션 지향** | 연결 설정 없이 개별 메시지 전송 |
| **전달 보장 없음** | 패킷 전달이 보장되지 않음 |
| **중복 보호 없음** | 중복 패킷에 대한 보호 기능 없음 |
| **최소 오버헤드** | TCP에 비해 헤더가 매우 작음 (8바이트) |
| **비연결성** | 연결 수립 과정 없음 |

### UDP vs TCP 비교

| 항목 | UDP | TCP |
|-----|-----|-----|
| 연결 방식 | 비연결형 | 연결 지향형 |
| 신뢰성 | 없음 | 있음 |
| 순서 보장 | 없음 | 있음 |
| 헤더 크기 | 8바이트 | 20바이트 이상 |
| 속도 | 빠름 | 상대적으로 느림 |
| 흐름 제어 | 없음 | 있음 |
| 혼잡 제어 | 없음 | 있음 |

> **참고**: 신뢰성 있고 순서가 보장된 데이터 스트림이 필요한 애플리케이션은 UDP 대신 **TCP**를 사용해야 합니다.

## UDP 헤더 구조

UDP 데이터그램 헤더는 단 **8바이트**로 구성됩니다:

```
 0      7 8     15 16    23 24    31
+--------+--------+--------+--------+
|     Source      |   Destination   |
|      Port       |      Port       |
+--------+--------+--------+--------+
|                 |                 |
|     Length      |    Checksum     |
+--------+--------+--------+--------+
|                                   |
|              Data                 |
|                                   |
+-----------------------------------+
```

### 헤더 필드 설명

| 필드 | 크기 | 설명 |
|-----|------|------|
| **Source Port** | 16비트 | 송신 포트 번호 (선택적) |
| **Destination Port** | 16비트 | 수신 포트 번호 (특정 인터넷 주소 컨텍스트에서 의미 있음) |
| **Length** | 16비트 | UDP 헤더와 데이터의 총 길이 (옥텟 단위, 최소값 8) |
| **Checksum** | 16비트 | 오류 검출을 위한 16비트 1의 보수 계산 |

### Source Port (송신 포트)
- **선택적 필드**입니다
- 송신자의 포트를 나타냅니다
- 응답이 필요 없는 경우 0으로 설정할 수 있습니다

### Destination Port (수신 포트)
- 특정 인터넷 주소 컨텍스트 내에서 의미를 가집니다
- 수신 애플리케이션을 식별하는 데 사용됩니다

### Length (길이)
- 옥텟(바이트) 단위로 측정됩니다
- **최소값은 8** (헤더만 있는 경우)
- 최대값은 65,535 - IP 헤더 크기

### Checksum (체크섬)
- IP 헤더 의사 헤더(pseudo-header), UDP 헤더, 데이터 페이로드를 포함하여 계산
- 16비트 1의 보수(one's complement) 계산 사용

## 체크섬 계산

### 의사 헤더 (Pseudo-Header)

체크섬 계산에는 IP 헤더의 일부 정보를 포함한 **의사 헤더**가 사용됩니다:

```
 0      7 8     15 16    23 24    31
+--------+--------+--------+--------+
|          Source Address           |
+--------+--------+--------+--------+
|        Destination Address        |
+--------+--------+--------+--------+
|  zero  |protocol|   UDP Length    |
+--------+--------+--------+--------+
```

### 체크섬 규칙

- 계산된 체크섬이 **0**이면, **모든 비트가 1**로 전송됩니다
- **모든 비트가 0**인 값은 체크섬이 생성되지 않았음을 나타냅니다
- 이 규칙은 0과 "체크섬 없음"을 구분하기 위한 것입니다

## 프로토콜 식별

UDP는 인터넷 프로토콜 스위트 내에서 **프로토콜 번호 17** (8진수 21)로 식별됩니다.

```
IP 헤더의 Protocol 필드:
- TCP: 6
- UDP: 17
- ICMP: 1
```

## 실제 활용 사례

### 초기 활용
RFC 768에서 언급된 주요 활용처:
- **인터넷 네임 서버 (DNS)**: 도메인 이름 조회
- **TFTP (Trivial File Transfer Protocol)**: 간단한 파일 전송

### 현대적 활용

| 프로토콜/서비스 | 포트 | 용도 |
|---------------|-----|------|
| DNS | 53 | 도메인 이름 조회 |
| DHCP | 67, 68 | IP 주소 자동 할당 |
| TFTP | 69 | 간단한 파일 전송 |
| NTP | 123 | 시간 동기화 |
| SNMP | 161, 162 | 네트워크 관리 |
| RIP | 520 | 라우팅 정보 교환 |

### UDP를 선택하는 경우

1. **실시간 애플리케이션**
   - VoIP (음성 통화)
   - 비디오 스트리밍
   - 온라인 게임

2. **단순 요청-응답**
   - DNS 쿼리
   - NTP 시간 동기화

3. **브로드캐스트/멀티캐스트**
   - DHCP
   - 서비스 디스커버리

4. **낮은 지연 시간이 중요한 경우**
   - 재전송보다 새로운 데이터가 더 중요할 때

## UDP 통신 흐름

```
송신자                                    수신자
   │                                        │
   │    UDP 데이터그램 (손실 가능)          │
   │───────────────────────────────────────→│
   │                                        │
   │    (확인 응답 없음)                     │
   │                                        │
   │    UDP 데이터그램 (손실 가능)          │
   │───────────────────────────────────────→│
   │                                        │
   │    (확인 응답 없음)                     │
   │                                        │
```

- 연결 설정 과정 없음 (핸드셰이크 없음)
- 각 데이터그램은 독립적으로 처리
- 확인 응답(ACK) 없음
- 재전송 메커니즘 없음

## UDP의 장점

1. **낮은 오버헤드**: 8바이트의 작은 헤더
2. **빠른 전송**: 연결 설정 지연 없음
3. **단순성**: 구현이 간단함
4. **유연성**: 애플리케이션 레벨에서 필요한 기능 구현 가능

## UDP의 단점

1. **신뢰성 없음**: 패킷 손실 가능
2. **순서 보장 없음**: 도착 순서가 다를 수 있음
3. **흐름/혼잡 제어 없음**: 네트워크 상황에 적응하지 않음
4. **중복 가능**: 동일 패킷이 여러 번 도착할 수 있음

## UDP 기반 신뢰성 프로토콜

UDP의 속도 이점을 유지하면서 신뢰성을 추가한 프로토콜들:

- **QUIC**: Google에서 개발, HTTP/3의 기반
- **DTLS**: TLS의 데이터그램 버전
- **RTP**: 실시간 미디어 전송
- **RUDP**: 신뢰성 있는 UDP

## 요약

UDP는 인터넷 프로토콜 스위트에서 가장 단순한 전송 계층 프로토콜입니다:

- **최소한의 프로토콜 메커니즘**으로 메시지 전송
- **연결 설정 없이** 즉시 데이터 전송 가능
- **낮은 지연 시간**이 신뢰성보다 중요한 경우 적합
- **8바이트**의 최소 헤더로 오버헤드 최소화

UDP의 단순함은 오히려 강점이 되어, 실시간 통신, DNS 쿼리 등 다양한 분야에서 광범위하게 사용되고 있습니다. TCP가 제공하는 신뢰성이 필요 없거나, 애플리케이션 레벨에서 자체적인 신뢰성 메커니즘을 구현하는 경우 UDP가 더 나은 선택이 될 수 있습니다.
