## 1. Introduction (서론)

### 1.1 기업(Enterprise)의 정의
- TCP/IP를 사용하여 네트워크를 ** 자율적으로 운영하는 주체**
- 자체적인 주소 지정 전략을 결정하는 권한 보유

### 1.2 문서의 목적
- 사설 인터넷에서 사용할 수 있는 IP 주소 대역 정의
- 기업 내 모든 호스트 간 ** 완전한 네트워크 계층 연결성** 제공
- 내부 네트워크 설계에 유연성 부여

### 1.3 주소 변경(Renumbering) 비용
- 사설 주소와 공인 주소 간 전환 시 발생하는 비용:
  - 호스트 IP 주소 변경
  - 해당 DNS 항목 수정
  - 해당 호스트를 IP 주소로 참조하는 다른 호스트의 설정 파일 변경

## 2. Motivation (동기)

### 2.1 배경: 인터넷 주소 고갈 문제
- 인터넷 성장으로 인한 ** 두 가지 주요 우려**:
  - ** 주소 공간 고갈**: 전역 고유 IP 주소의 부족
  - ** 라우팅 오버헤드 증가**: ISP의 처리 능력을 초과하는 라우팅 정보 확장

### 2.2 핵심 관찰
- 모든 호스트가 글로벌 인터넷 연결을 필요로 하지 않음
- 기업 내 대부분의 호스트는 제한된 외부 연결만 필요

### 2.3 호스트 카테고리 (3가지 분류)

| 카테고리 | 연결 요구사항 | 주소 요구사항 |
|----------|---------------|---------------|
| ** 카테고리 1** | 외부 액세스 불필요 | 기업 내 고유하면 됨, 기업 간 중복 가능 |
| ** 카테고리 2** | 제한된 외부 서비스 필요 (이메일, FTP, 원격 로그인 등), 중개 게이트웨이 통해 처리 | 기업 내 고유하면 됨, 기업 간 중복 가능 |
| ** 카테고리 3** | 네트워크 계층 수준의 외부 연결 필요 | ** 전역적으로 고유한 IP 주소 필수** |

### 2.4 용어 정의
- ** 사설(Private)**: 카테고리 1, 2에 해당하는 호스트
- ** 공인(Public)**: 카테고리 3에 해당하는 호스트

### 2.5 실제 사용 사례

#### 카테고리 1 예시 (외부 연결 불필요)
- 공항 출발 안내 디스플레이
  - 내부 TCP/IP 연결만 필요
  - 외부 인터넷 연결 불필요

#### 카테고리 2 예시 (제한된 외부 연결)
- 소매점/은행의 금전등록기 및 워크스테이션
  - 최소한의 외부 요구사항
  - 중개 게이트웨이를 통한 서비스 이용

#### 기타 사례
- 보안 격리를 위해 애플리케이션 계층 게이트웨이를 사용하는 기업 네트워크
- 외부 접근성이 필요 없는 내부 라우터 인터페이스

## 3. Private Address Space (사설 주소 공간)

IANA(Internet Assigned Numbers Authority)가 사설용으로 예약한 세 가지 주소 블록:

### 3.1 IANA 예약 주소 블록

| 블록 | 주소 범위 | CIDR 표기 | 클래스 | 주소 개수 |
|------|-----------|-----------|--------|-----------|
| 24비트 블록 | 10.0.0.0 ~ 10.255.255.255 | 10.0.0.0/8 | Class A | 16,777,216개 |
| 20비트 블록 | 172.16.0.0 ~ 172.31.255.255 | 172.16.0.0/12 | Class B | 1,048,576개 |
| 16비트 블록 | 192.168.0.0 ~ 192.168.255.255 | 192.168.0.0/16 | Class C | 65,536개 |

### 각 블록의 상세 정보

**10.0.0.0/8 (Class A 대역)**
```
주소 범위: 10.0.0.0 - 10.255.255.255
네트워크 비트: 8
호스트 비트: 24
사용 가능 호스트: 16,777,214개 (브로드캐스트/네트워크 주소 제외)
용도: 대규모 조직, 데이터센터
```

**172.16.0.0/12 (Class B 대역)**
```
주소 범위: 172.16.0.0 - 172.31.255.255
네트워크 비트: 12
호스트 비트: 20
사용 가능 호스트: 1,048,574개
용도: 중규모 조직
```

**192.168.0.0/16 (Class C 대역)**
```
주소 범위: 192.168.0.0 - 192.168.255.255
네트워크 비트: 16
호스트 비트: 16
사용 가능 호스트: 65,534개
용도: 소규모 조직, 가정용 네트워크
```

### 3.2 사설 주소의 특성

#### 사용 권한
- **IANA나 인터넷 레지스트리와의 조율 없이** 자유롭게 사용 가능
- 조직 내부에서만 고유성 보장

#### 고유성 범위
- 기업 내부 또는 협력하는 사설 네트워크 내에서만 고유
- ** 전역적 고유성 없음**

#### 외부 연결 요구사항
- 외부 연결이 필요한 경우 인터넷 레지스트리로부터 ** 전역 고유 주소 별도 필요**

### 3.3 사설 호스트와 공인 호스트의 연결성

| 호스트 유형 | 내부 통신 | 외부 통신 |
|-------------|-----------|-----------|
| 사설 호스트 | 기업 내 모든 호스트(사설/공인)와 통신 가능 | ** 기업 외부 호스트와 IP 연결 불가** |
| 공인 호스트 | 기업 내 모든 호스트와 통신 가능 | 외부 호스트와 통신 가능 |

### 3.4 라우팅 요구사항
- ** 사설 네트워크에 대한 라우팅 정보는 기업 간 링크로 전파되어서는 안 됨**
- 라우터는 이러한 정보를 ** 필터링** 해야 함

### 3.5 DNS 고려사항
- 인터넷 서비스 제공자는 사설 주소 참조가 기업 외부로 ** 유출되는 것을 방지** 하는 조치 필요

## 4. Advantages and Disadvantages (장점과 단점)

### 4.1 장점

#### 4.1.1 글로벌 주소 공간 절약
- 전역 고유 주소 공간 보존
- IP 주소 공간의 수명 연장에 기여

#### 4.1.2 네트워크 설계 유연성
- 더 큰 가용 주소 풀을 통한 ** 설계 유연성** 제공
- ** 운영 및 관리상 편리한 주소 체계** 구현 가능

#### 4.1.3 성장 경로 용이
- ** 더 쉬운 성장 경로** 제공
- 내부 네트워크 확장 시 추가 주소 할당 절차 불필요

#### 4.1.4 주소 충돌 방지
- 이전에 연결되지 않았던 기업들이 나중에 인터넷 연결을 원할 때 주소 충돌 방지
- 외부 연결 전환 시 충돌을 피하는 **"안전한 선택(safe choice)"**

### 4.2 단점

#### 4.2.1 인터넷 액세스 유연성 감소
- 사설 주소에서 공인 주소로 전환 시 ** 재번호 부여(renumbering) 필요**
- 재번호 부여에 대한 약속(commitment) 필요

#### 4.2.2 재번호 부여 비용
- 비용 = ** 사설에서 공인으로 전환해야 하는 호스트 수** 에 비례
- 호스트 IP 주소 변경, DNS 항목 수정, 설정 파일 변경 필요

#### 4.2.3 사설 인터넷 병합 시 문제
- 여러 사설 인터넷을 병합할 때 ** 주소 충돌 발생 가능**
- 주소 충돌 시 재번호 부여 필요

#### 4.2.4 기업 합병 시 문제
- 조율되지 않은 사설 인터넷을 가진 기업들의 합병 시 ** 특별한 도전 과제** 발생

### 4.3 완화 전략
- **DHCP와 같은 벤더 도구** 를 조사하여 재번호 부여 프로세스 용이화 권장

## 5. Operational Considerations (운영 고려사항)

### 5.1 네트워크 설계 전략
- **1단계**: 사설 주소 공간을 사용하여 사설 네트워크 먼저 설계
- **2단계**: 필요한 위치에 공인 서브넷 계획

### 5.2 유연성 확보

#### 주소 변경 시 필요한 작업
- 관련 호스트만 재번호 부여
- 필요 시 물리적 연결 변경

#### 물리적 미디어 분리 권장
- 공인/사설 서브넷에 대해 ** 별도의 물리적 미디어** 사용
- 향후 변경 용이성 확보

### 5.3 서브넷팅 권장사항

| 상황 | 권장 블록 |
|------|-----------|
| 서브넷팅이 가능한 경우 | **24비트 블록 (Class A 네트워크, 10/8)** 선호 |
| 서브넷팅에 제약이 있는 경우 | 16비트 블록 (Class C, 192.168/16) 또는 20비트 블록 (Class B, 172.16/12) |

#### 호스트 그룹화 원칙
- ** 유사한 연결 요구사항** 을 가진 호스트를 동일 서브넷에 그룹화

### 5.4 혼합 주소 지정 주의사항
- 단일 미디어에 공인/사설 혼합 사용은 기술적으로 가능
- 그러나 ** 복잡성을 유발** 하므로 권장하지 않음

### 5.5 필터링 요구사항

#### 패킷 및 라우팅 필터
- 기업을 외부 네트워크에 연결하는 라우터에 적절한 패킷 및 라우팅 필터 설정
- ** 양쪽 끝 모두에 설정 필요**
- 유출 방지

```
# 인바운드 필터 (외부 → 내부)
deny ip 10.0.0.0/8 any
deny ip 172.16.0.0/12 any
deny ip 192.168.0.0/16 any

# 아웃바운드 필터 (내부 → 외부)
deny ip any 10.0.0.0/8
deny ip any 172.16.0.0/12
deny ip any 192.168.0.0/16
```

#### 인바운드 필터링
- 인바운드 라우팅 정보에서 사설 네트워크 필터링 필요

### 5.6 조직 간 연결

#### 사설 네트워크 간 통신 방법
- 두 협력 사이트가 통신 가능
- 공용 네트워크로의 ** 경계에서 캡슐화(encapsulation)** 사용
- 사설 주소를 비공개로 유지

### 5.7 주소 충돌 위험 완화
- 사설 주소 예약 풀에서 ** 무작위로** 선택하여 하위 블록 할당
- 다른 조직과의 주소 충돌 위험 최소화

### 5.8 DNS 아키텍처

#### 분할 DNS(Split DNS) 구성 권장
- ** 공인/사설 주소를 모두 포함하는 DNS 존에 대해 2개의 권한 서버 운영**

| DNS 서버 | 가시성 | 포함 내용 |
|----------|--------|-----------|
| 공개용 서버 | 외부에서 접근 가능 | 공인 주소만 포함 |
| 내부용 서버 | 내부에서만 접근 가능 | 전체 데이터 (사설 주소 포함) |

```
┌─────────────────────────────────────────────────┐
│                 외부 DNS 서버                    │
│  example.com → 203.0.113.10 (공인 IP)          │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│                 내부 DNS 서버                    │
│  server.internal → 10.0.1.100 (사설 IP)        │
│  example.com → 10.0.1.50 (내부 서버)           │
└─────────────────────────────────────────────────┘
```

## 7. Conclusion (결론)

### 7.1 대규모 기업의 이점
- 대규모 기업이 전역 고유 IP 주소 공간에서 ** 상대적으로 작은 블록만** 필요로 하게 됨

### 7.2 전체 인터넷의 이점
- ** 전역 고유 주소 공간 보존**
- IP 주소 공간의 수명을 ** 효과적으로 연장**

### 7.3 절충점
- 조직은 유연성을 얻는 대신 연결 요구사항이 변화함에 따라 ** 잠재적인 재번호 부여 의무** 부담
