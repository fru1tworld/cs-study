## 1. 소개 (Introduction)

### 1.1 개요
- OAuth 2.0은 ** 인가 계층(authorization layer)을 도입** 하여 클라이언트 역할과 리소스 소유자 역할을 분리
- 제3자 애플리케이션이 ** 사용자 자격 증명을 직접 다루지 않고** HTTP 서비스에 제한된 접근 권한 획득 가능
- 리소스 소유자를 대신하여 리소스 소유자와 HTTP 서비스 간의 승인 상호작용 조율
- 또는 제3자 애플리케이션이 자체적으로 접근 권한 획득

### 1.2 역할 (Roles)

OAuth 2.0은 네 가지 역할을 정의:

| 역할 | 설명 |
|------|------|
| **Resource Owner** | 보호된 리소스에 대한 접근 권한을 부여할 수 있는 엔티티. 일반적으로 최종 사용자(end-user) |
| **Resource Server** | 보호된 리소스를 호스팅하는 서버. 액세스 토큰을 사용하여 보호된 리소스 요청 수락 |
| **Client** | 리소스 소유자를 대신하여 인가를 받아 보호된 리소스 요청을 하는 애플리케이션 |
| **Authorization Server** | 리소스 소유자 인증 및 인가 획득 후 클라이언트에게 액세스 토큰 발급 |

- 인가 서버와 리소스 서버 간의 상호작용은 이 명세의 범위를 벗어남
- 인가 서버와 리소스 서버는 같은 서버일 수도 있고 분리될 수도 있음
- 단일 인가 서버가 여러 리소스 서버에 대한 액세스 토큰 발급 가능

### 1.3 프로토콜 흐름 (Protocol Flow)

```
     +--------+                               +---------------+
     |        |--(A)- Authorization Request ->|   Resource    |
     |        |                               |     Owner     |
     |        |<-(B)-- Authorization Grant ---|               |
     |        |                               +---------------+
     |        |
     |        |                               +---------------+
     |        |--(C)-- Authorization Grant -->| Authorization |
     | Client |                               |     Server    |
     |        |<-(D)----- Access Token -------|               |
     |        |                               +---------------+
     |        |
     |        |                               +---------------+
     |        |--(E)----- Access Token ------>|    Resource   |
     |        |                               |     Server    |
     |        |<-(F)--- Protected Resource ---|               |
     +--------+                               +---------------+
```

| 단계 | 설명 |
|------|------|
| (A) | 클라이언트가 리소스 소유자에게 인가 요청 |
| (B) | 클라이언트가 인가 승인(authorization grant) 수신 - 리소스 소유자의 인가를 나타내는 자격 증명 |
| (C) | 클라이언트가 인가 서버에 인증하고 인가 승인 제시하여 액세스 토큰 요청 |
| (D) | 인가 서버가 클라이언트 인증 및 인가 승인 검증 후 액세스 토큰 발급 |
| (E) | 클라이언트가 액세스 토큰을 제시하여 리소스 서버에 보호된 리소스 요청 |
| (F) | 리소스 서버가 액세스 토큰 검증 후 요청 처리 |

### 1.4 인가 승인 (Authorization Grant)

인가 승인은 리소스 소유자의 인가를 나타내는 자격 증명으로, 클라이언트가 액세스 토큰을 얻는 데 사용

#### 1.4.1 Authorization Code
- 인가 서버가 클라이언트와 리소스 소유자 사이의 ** 중개자** 역할
- 리소스 소유자의 자격 증명이 ** 클라이언트와 공유되지 않음**
- 클라이언트 인증 기능 제공
- 액세스 토큰이 리소스 소유자의 user-agent를 통해 전달되지 않아 ** 잠재적 노출 방지**

#### 1.4.2 Implicit
- ** 브라우저에서 JavaScript 등 스크립팅 언어로 구현된 클라이언트에 최적화** 된 단순화된 authorization code 흐름
- 인가 코드 대신 ** 액세스 토큰 직접 발급**
- 리프레시 토큰 ** 지원 안 함**
- 왕복 횟수 감소로 일부 클라이언트의 응답성 및 효율성 향상
- 보안 영향 고려 필요 (Section 10.3, 10.16)

#### 1.4.3 Resource Owner Password Credentials
- 리소스 소유자와 클라이언트 간에 ** 높은 수준의 신뢰** 가 있을 때만 적합
- 사용자 이름과 비밀번호를 직접 인가 승인으로 사용
- 단일 요청으로 액세스 토큰 교환
- 기존 직접 인증 방식을 OAuth로 마이그레이션 시 유용

#### 1.4.4 Client Credentials
- 인가 범위가 ** 클라이언트 통제 하의 보호된 리소스** 로 제한된 경우 사용
- 또는 인가 서버와 ** 사전에 배열된 인가** 에 대한 접근 요청 시 사용
- 클라이언트가 자체적으로(리소스 소유자 역할도 함) 또는 사전 배열된 인가 기반으로 동작

### 1.5 액세스 토큰 (Access Token)

- 클라이언트에 발급된 ** 인가를 나타내는 자격 증명**
- 특정 범위(scope)와 접근 기간을 나타냄
- 리소스 소유자가 부여하고 리소스 서버와 인가 서버가 시행
- 토큰 문자열은 일반적으로 ** 클라이언트에게 불투명(opaque)**
- 토큰은 인가 정보 검색용 식별자이거나, 인가 정보를 서명/검증 가능한 방식으로 자체 포함

### 1.6 리프레시 토큰 (Refresh Token)

- 액세스 토큰 ** 갱신** 을 위한 선택적 자격 증명
- 리프레시 토큰은 ** 인가 서버에서만 사용** - 리소스 서버로 전송되지 않음

```
  +--------+                                           +---------------+
  |        |--(A)------- Authorization Grant --------->|               |
  |        |                                           |               |
  |        |<-(B)----------- Access Token -------------|               |
  |        |               & Refresh Token             |               |
  |        |                                           |               |
  |        |                            +----------+   |               |
  |        |--(C)---- Access Token ---->|          |   |               |
  |        |                            |          |   |               |
  |        |<-(D)- Protected Resource --|          |   | Authorization |
  | Client |                            | Resource |   |     Server    |
  |        |--(E)---- Access Token ---->|  Server  |   |               |
  |        |                            |          |   |               |
  |        |<-(F)- Invalid Token Error -|          |   |               |
  |        |                            +----------+   |               |
  |        |                                           |               |
  |        |--(G)----------- Refresh Token ----------->|               |
  |        |                                           |               |
  |        |<-(H)----------- Access Token -------------|               |
  +--------+           & Optional Refresh Token        +---------------+
```

### 1.7 TLS 버전 (TLS Version)

- 이 명세 작성 시점에 TLS 버전 1.2가 가장 최신 버전
- TLS 1.0은 ** 가장 광범위한 상호운용성** 제공
- 구현은 보안 요구사항에 따라 추가 전송 계층 보안 메커니즘 지원 가능

### 1.8 HTTP 리다이렉션 (HTTP Redirections)

- 이 명세는 HTTP 리다이렉션을 ** 광범위하게 사용**
- 예제에서 HTTP 302 상태 코드 사용하지만, **user-agent를 통해 리다이렉션을 수행하는 다른 방법도 허용**

### 1.9 상호운용성 (Interoperability)

- OAuth 2.0은 ** 많은 선택적 구성요소를 가진 풍부하고 확장 가능한 프레임워크**
- 이 명세만으로는 ** 상호운용 불가능한 구현이 발생할 가능성** 높음
- 완전한 웹 스케일 상호운용성을 위해 ** 규범적 프로파일과 확장 정의** 필요

### 1.9 표기 규칙 (Notational Conventions)

- RFC 2119 키워드 사용:
  - **MUST / MUST NOT**: 절대적 요구사항/금지
  - **REQUIRED / SHALL**: MUST와 동의어
  - **SHOULD / SHOULD NOT**: 권장/비권장
  - **RECOMMENDED**: SHOULD와 동의어
  - **MAY / OPTIONAL**: 선택사항
- RFC 5234의 ABNF 표기법 사용
- RFC 3986의 URI-reference 규칙 적용
- 달리 명시되지 않으면 모든 프로토콜 파라미터 이름과 값은 ** 대소문자 구분**

## 2. 클라이언트 등록 (Client Registration)

프로토콜 시작 전 클라이언트는 인가 서버에 등록해야 함

등록 시 클라이언트가 제공해야 하는 정보:
- 클라이언트 유형 지정
- 리다이렉션 URI 제공
- 인가 서버가 요구하는 기타 정보 (이름, 웹사이트, 로고, 법적 약관 등)

### 2.1 클라이언트 유형 (Client Types)

| 유형 | 설명 |
|------|------|
| **Confidential** | 자격 증명의 기밀성을 유지할 수 있음 (예: 접근이 제한된 보안 서버에서 구현된 클라이언트) |
| **Public** | 자격 증명의 기밀성을 유지할 수 없음 (예: 리소스 소유자가 사용하는 장치에서 실행되는 네이티브 앱 또는 브라우저 기반 앱) |

#### 클라이언트 프로파일

| 프로파일 | 유형 | 설명 |
|----------|------|------|
| **Web Application** | Confidential | 웹 서버에서 실행. 클라이언트 자격 증명과 액세스 토큰이 서버 측에 저장되어 리소스 소유자에게 노출되지 않음 |
| **User-Agent-Based Application** | Public | 웹 서버에서 다운로드한 코드가 브라우저 내에서 실행. 프로토콜 데이터와 자격 증명이 리소스 소유자에게 쉽게 접근 가능 |
| **Native Application** | Public | 장치에 설치 및 실행. 정적 자격 증명은 추출 가능하다고 가정. 동적 발급 토큰(액세스/리프레시)은 수용 가능한 수준의 보호 가능 |

### 2.2 클라이언트 식별자 (Client Identifier)

- 인가 서버가 클라이언트에게 발급하는 ** 고유한 문자열**
- 클라이언트 식별자는 ** 비밀이 아님** - 리소스 소유자에게 노출됨
- 클라이언트 인증에 ** 단독으로 사용해서는 안 됨**

### 2.3 클라이언트 인증 (Client Authentication)

- Confidential 클라이언트는 인가 서버와 인증 방법 설정
- 인가 서버는 보안 요구사항을 충족하는 ** 모든 형태의 클라이언트 인증 수락 가능**
- 일반적으로 클라이언트 자격 증명 발급: 클라이언트 비밀번호 또는 공개/개인 키 쌍

#### 2.3.1 클라이언트 비밀번호 (Client Password)

- 클라이언트 비밀번호가 있는 클라이언트는 RFC 2617의 **HTTP Basic 인증** 사용 가능
- 클라이언트 식별자와 비밀번호를 `application/x-www-form-urlencoded` 알고리즘으로 인코딩
- 대안: 요청 본문에 `client_id`와 `client_secret` 파라미터 포함 (** 권장하지 않음**)

요구사항:
- 인가 서버는 비밀번호 인증 요청 시 **TLS 사용 필수**
- ** 무차별 대입 공격 방지** 대책 필수

#### 2.3.2 기타 인증 방법 (Other Authentication Methods)

- 인가 서버는 보안 요구사항에 맞는 ** 적절한 HTTP 인증 방식 지원 가능**
- 클라이언트 식별자와 인증 방식 간의 ** 매핑 정의** 필요

### 2.4 미등록 클라이언트 (Unregistered Clients)

- 이 명세는 미등록 클라이언트 사용을 배제하지 않음
- 그러나 사용은 ** 이 명세의 범위를 벗어나며 추가 보안 분석 필요**

## 3. 프로토콜 엔드포인트 (Protocol Endpoints)

인가 프로세스는 두 개의 서버 엔드포인트와 하나의 클라이언트 엔드포인트 사용:

| 엔드포인트 | 위치 | 용도 |
|------------|------|------|
| **Authorization Endpoint** | 서버 | user-agent 리다이렉션을 통해 리소스 소유자로부터 인가 획득 |
| **Token Endpoint** | 서버 | 클라이언트가 인가 승인을 액세스 토큰으로 교환 |
| **Redirection Endpoint** | 클라이언트 | 인가 서버가 리소스 소유자 user-agent를 통해 자격 증명 반환 |

### 3.1 인가 엔드포인트 (Authorization Endpoint)

- 리소스 소유자와 상호작용하여 인가 승인 획득
- 인가 서버는 먼저 ** 리소스 소유자의 신원 확인** 필요
- 엔드포인트는 `application/x-www-form-urlencoded` 쿼리 컴포넌트 포함 가능 - 추가 파라미터 추가 시 유지해야 함
- 프래그먼트 컴포넌트는 ** 포함하면 안 됨**
- 인가 엔드포인트 요청 시 **TLS 사용 필수**
- **HTTP GET 메서드 지원 필수**, POST 메서드는 선택적으로 지원 가능

#### 3.1.1 응답 유형 (Response Type)

`response_type` 파라미터로 원하는 권한 부여 유형 지정:

| 값 | 설명 |
|----|------|
| `code` | Authorization Code 요청 (Section 4.1.1) |
| `token` | Access Token / Implicit Grant 요청 (Section 4.2.1) |
| 등록된 확장 값 | Section 8.4에 따름 |

- 확장 응답 유형은 ** 공백으로 구분된 목록** 포함 가능 (순서 무관)
- `response_type` 파라미터가 없거나 인식할 수 없으면 ** 에러 응답 반환 필수**

#### 3.1.2 리다이렉션 엔드포인트 (Redirection Endpoint)

인가 서버가 user-agent를 클라이언트 엔드포인트로 리다이렉션

- 리다이렉션 엔드포인트 URI는 ** 절대 URI** 여야 함
- 쿼리 컴포넌트 포함 가능 (추가 파라미터 시 유지)
- 프래그먼트 컴포넌트는 ** 포함하면 안 됨**

##### 3.1.2.1 엔드포인트 요청 기밀성

- 응답 유형이 `code` 또는 `token`일 때 **TLS 사용 권장**
- 전송 계층 보안 부재는 ** 클라이언트와 보호된 리소스 보안에 심각한 영향**

##### 3.1.2.2 등록 요구사항

인가 서버는 다음 클라이언트에 대해 리다이렉션 엔드포인트 등록 ** 필수**:
- Public 클라이언트
- Implicit grant를 사용하는 Confidential 클라이언트

등록 권장사항:
- 완전한 리다이렉션 URI 등록 요구 또는 최소한 URI 스킴, 권한, 경로 등록
- 여러 리다이렉션 엔드포인트 허용 가능
- 등록 요구 부재는 ** 공격자가 인가 엔드포인트를 오픈 리다이렉터로 악용** 가능

##### 3.1.2.3 동적 구성 (Dynamic Configuration)

클라이언트는 다음 경우 리다이렉션 URI 파라미터 포함 필수:
- 여러 리다이렉션 URI 등록
- 부분 리다이렉션 URI만 등록
- 리다이렉션 URI 미등록

인가 서버는 수신 값을 ** 등록된 리다이렉션 URI 중 하나와 비교/매칭** 필수

##### 3.1.2.4 잘못된 엔드포인트 (Invalid Endpoint)

엔드포인트 검증 실패 시:
- 리소스 소유자에게 ** 에러 알림 권장**
- 잘못된 리다이렉션 URI로 ** 자동 리다이렉션 금지**

##### 3.1.2.5 엔드포인트 콘텐츠 (Endpoint Content)

- 리다이렉션 응답은 일반적으로 HTML 문서 반환
- 클라이언트는 리다이렉션 엔드포인트 응답에 ** 서드파티 스크립트 포함 금지** 권장
- URI에서 자격 증명 추출 후 ** 자격 증명 노출 없이 다른 엔드포인트로 다시 리다이렉션** 권장

### 3.2 토큰 엔드포인트 (Token Endpoint)

- 클라이언트가 인가 승인 또는 리프레시 토큰을 제시하여 액세스 토큰 획득
- Implicit grant를 제외한 ** 모든 grant 유형** 에 사용
- 토큰 엔드포인트 요청 시 **TLS 사용 필수**
- 액세스 토큰 요청 시 **HTTP POST 메서드 사용 필수**

#### 3.2.1 클라이언트 인증 (Client Authentication)

- Confidential 클라이언트 또는 자격 증명이 발급된 클라이언트는 토큰 엔드포인트 요청 시 ** 인가 서버에 인증 필수**

클라이언트 인증 목적:
- 리프레시 토큰과 인가 코드를 발급받은 클라이언트에 ** 바인딩** 강제
- 침해된 클라이언트의 ** 자격 증명 비활성화** 로 남용 방지
- ** 자격 증명 순환** 모범 사례 구현

- 인증되지 않은 Public 클라이언트는 `client_id` 파라미터로 자신 식별 가능

### 3.3 액세스 토큰 범위 (Access Token Scope)

- 인가 및 토큰 엔드포인트는 `scope` 요청 파라미터로 범위 지정 가능
- 인가 서버는 `scope` 응답 파라미터로 발급된 토큰 범위 알림

범위 값 형식:
- ** 공백으로 구분된, 대소문자 구분 문자열 목록**
- 문자열은 인가 서버가 정의
- 구문: `scope = scope-token *( SP scope-token )`

규칙:
- 인가 서버는 클라이언트 요청 범위를 ** 전체 또는 부분 무시 가능**
- 발급된 범위가 요청과 다르면 **`scope` 응답 파라미터 포함 필수**
- 범위 파라미터 생략 시 ** 사전 정의된 기본값 사용하거나 요청 실패**

## 4. 인가 획득 (Obtaining Authorization)

OAuth는 네 가지 권한 부여 유형과 추가 유형을 위한 확장 메커니즘 정의

### 4.1 Authorization Code Grant

**Confidential 클라이언트에 최적화** 된 리다이렉션 기반 흐름

장점:
- ** 클라이언트 인증** 기능
- 액세스 토큰이 리소스 소유자의 user-agent를 통과하지 않고 ** 클라이언트에 직접 전송**

```
     +----------+
     | Resource |
     |   Owner  |
     |          |
     +----------+
          ^
          |
         (B)
     +----|-----+          Client Identifier      +---------------+
     |         -+----(A)-- & Redirection URI ---->|               |
     |  User-   |                                 | Authorization |
     |  Agent  -+----(B)-- User authenticates --->|     Server    |
     |          |                                 |               |
     |         -+----(C)-- Authorization Code ---<|               |
     +-|----|---+                                 +---------------+
       |    |                                         ^      v
      (A)  (C)                                        |      |
       |    |                                         |      |
       ^    v                                         |      |
     +---------+                                      |      |
     |         |>---(D)-- Authorization Code ---------'      |
     |  Client |          & Redirection URI                  |
     |         |                                             |
     |         |<---(E)----- Access Token -------------------'
     +---------+       (w/ Optional Refresh Token)
```

| 단계 | 설명 |
|------|------|
| (A) | 클라이언트가 user-agent를 인가 엔드포인트로 리다이렉션. 클라이언트 식별자, 요청 범위, 로컬 상태, 리다이렉션 URI 포함 |
| (B) | 인가 서버가 user-agent를 통해 리소스 소유자 인증, 권한 부여/거부 결정 |
| (C) | 인가 서버가 user-agent를 클라이언트로 리다이렉션. 인가 코드와 state 파라미터 포함 |
| (D) | 클라이언트가 토큰 엔드포인트에 액세스 토큰 요청. 인가 코드 포함, 인가 서버에 인증 |
| (E) | 인가 서버가 클라이언트 인증, 인가 코드 검증, 리다이렉션 URI 일치 확인 후 액세스 토큰과 선택적 리프레시 토큰 발급 |

#### 4.1.1 인가 요청 (Authorization Request)

클라이언트가 `application/x-www-form-urlencoded` 형식으로 쿼리 컴포넌트에 파라미터 추가:

| 파라미터 | 필수 여부 | 설명 |
|----------|-----------|------|
| `response_type` | ** 필수** | 값은 `code`여야 함 |
| `client_id` | ** 필수** | 클라이언트 식별자 |
| `redirect_uri` | 선택 | 리다이렉션 엔드포인트 |
| `scope` | 선택 | 요청 범위 |
| `state` | ** 권장** | 불투명한 값. CSRF 방지용 (Section 10.12) |

예시:
```http
GET /authorize?response_type=code&client_id=s6BhdRkqt3&state=xyz
    &redirect_uri=https%3A%2F%2Fclient%2Eexample%2Ecom%2Fcb HTTP/1.1
Host: server.example.com
```

#### 4.1.2 인가 응답 (Authorization Response)

** 성공 응답:**

인가 서버가 리다이렉션 URI 쿼리 컴포넌트에 파라미터 추가:

| 파라미터 | 필수 여부 | 설명 |
|----------|-----------|------|
| `code` | ** 필수** | 인가 코드. 발급 후 ** 짧은 시간 내 만료** 필수 (최대 10분 권장). 1회만 사용 가능. 클라이언트 식별자와 리다이렉션 URI에 바인딩 |
| `state` | 조건부 필수 | 요청에 있었다면 ** 정확히 동일한 값** |

예시:
```http
HTTP/1.1 302 Found
Location: https://client.example.com/cb?code=SplxlOBeZQQYbYS6WxSbIA&state=xyz
```

** 에러 응답:**

리다이렉션 URI 관련 문제 또는 클라이언트 식별자 문제 시:
- 리소스 소유자에게 ** 에러 알림 권장**
- 잘못된 리다이렉션 URI로 ** 자동 리다이렉션 금지**

기타 에러 시 쿼리 컴포넌트에 파라미터 추가:

| 파라미터 | 필수 여부 | 에러 코드 |
|----------|-----------|-----------|
| `error` | ** 필수** | `invalid_request`, `unauthorized_client`, `access_denied`, `unsupported_response_type`, `invalid_scope`, `server_error`, `temporarily_unavailable` |
| `error_description` | 선택 | 사람이 읽을 수 있는 ASCII 텍스트 |
| `error_uri` | 선택 | 에러 정보가 있는 웹 페이지 URI |
| `state` | 조건부 필수 | 요청에 있었다면 정확히 동일한 값 |

예시:
```http
HTTP/1.1 302 Found
Location: https://client.example.com/cb?error=access_denied&state=xyz
```

#### 4.1.3 액세스 토큰 요청 (Access Token Request)

클라이언트가 `application/x-www-form-urlencoded` 형식으로 요청 본문에 파라미터 추가:

| 파라미터 | 필수 여부 | 설명 |
|----------|-----------|------|
| `grant_type` | ** 필수** | 값은 `authorization_code`여야 함 |
| `code` | ** 필수** | 인가 서버에서 받은 인가 코드 |
| `redirect_uri` | 조건부 필수 | 인가 요청에 포함되었다면 ** 동일한 값** 필수 |
| `client_id` | 조건부 필수 | Section 3.2.1에 따라 클라이언트가 인증하지 않는 경우 |

예시:
```http
POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&code=SplxlOBeZQQYbYS6WxSbIA
&redirect_uri=https%3A%2F%2Fclient%2Eexample%2Ecom%2Fcb
```

인가 서버 검증 필수 항목:
- Confidential 클라이언트 또는 자격 증명 발급 클라이언트에 대해 ** 클라이언트 인증 필수**
- 인증 포함 시 ** 클라이언트 인증**
- 인가 코드가 인증된 Confidential 클라이언트에 발급되었는지 또는 (Public인 경우) 요청의 `client_id`에 발급되었는지 확인
- ** 인가 코드 유효성** 검증
- 초기 요청에 `redirect_uri`가 있었다면 ** 동일한 값인지** 확인

#### 4.1.4 액세스 토큰 응답 (Access Token Response)

- 유효하고 인가되면 Section 5.1에 따라 액세스 토큰과 선택적 리프레시 토큰 발급
- 인증 실패 또는 유효하지 않으면 Section 5.2에 따라 에러 반환

예시:
```http
HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Cache-Control: no-store
Pragma: no-cache

{
  "access_token":"2YotnFZFEjr1zCsicMWpAA",
  "token_type":"example",
  "expires_in":3600,
  "refresh_token":"tGzv3JOkF0XG5Qx2TlKWIA",
  "example_parameter":"example_value"
}
```

### 4.2 Implicit Grant

** 브라우저에서 JavaScript로 구현된 Public 클라이언트에 최적화** 된 단순화된 흐름

특징:
- 리프레시 토큰 발급 ** 지원 안 함**
- 클라이언트가 인가 요청 결과로 ** 액세스 토큰 직접 수신**
- 인가 서버가 ** 클라이언트 인증 안 함**
- 액세스 토큰이 리소스 소유자 또는 다른 애플리케이션에 ** 노출될 수 있음**

```
     +----------+
     | Resource |
     |  Owner   |
     |          |
     +----------+
          ^
          |
         (B)
     +----|-----+          Client Identifier     +---------------+
     |         -+----(A)-- & Redirection URI --->|               |
     |  User-   |                                | Authorization |
     |  Agent  -|----(B)-- User authenticates -->|     Server    |
     |          |                                |               |
     |          |<---(C)--- Redirection URI ----<|               |
     |          |          with Access Token     +---------------+
     |          |            in Fragment
     |          |                                +---------------+
     |          |----(D)--- Redirection URI ---->|   Web-Hosted  |
     |          |          without Fragment      |     Client    |
     |          |                                |    Resource   |
     |     (F)  |<---(E)------- Script ---------<|               |
     |          |                                +---------------+
     +-|--------+
       |    |
      (A)  (G) Access Token
       |    |
       ^    v
     +---------+
     |         |
     |  Client |
     |         |
     +---------+
```

| 단계 | 설명 |
|------|------|
| (A) | 클라이언트가 user-agent를 인가 엔드포인트로 리다이렉션 |
| (B) | 인가 서버가 user-agent를 통해 리소스 소유자 인증 |
| (C) | 인가 서버가 user-agent를 클라이언트로 리다이렉션. **URI 프래그먼트에 액세스 토큰** 포함 |
| (D) | user-agent가 리다이렉션 따라 웹 호스팅 클라이언트 리소스로 이동 (** 프래그먼트 포함 안 됨**) |
| (E) | 웹 호스팅 클라이언트 리소스가 전체 리다이렉션 URI에 접근하는 ** 스크립트 포함 웹 페이지** 반환 |
| (F) | user-agent가 스크립트를 로컬에서 실행 |
| (G) | user-agent가 액세스 토큰을 클라이언트에 전달 |

#### 4.2.1 인가 요청 (Authorization Request)

| 파라미터 | 필수 여부 | 설명 |
|----------|-----------|------|
| `response_type` | ** 필수** | 값은 `token`이어야 함 |
| `client_id` | ** 필수** | 클라이언트 식별자 |
| `redirect_uri` | 선택 | 리다이렉션 엔드포인트 |
| `scope` | 선택 | 요청 범위 |
| `state` | ** 권장** | 불투명한 값. CSRF 방지용 |

예시:
```http
GET /authorize?response_type=token&client_id=s6BhdRkqt3&state=xyz
    &redirect_uri=https%3A%2F%2Fclient%2Eexample%2Ecom%2Fcb HTTP/1.1
Host: server.example.com
```

#### 4.2.2 액세스 토큰 응답 (Access Token Response)

** 성공 응답:**

리다이렉션 URI ** 프래그먼트 컴포넌트** 에 파라미터 추가:

| 파라미터 | 필수 여부 | 설명 |
|----------|-----------|------|
| `access_token` | ** 필수** | 인가 서버가 발급한 액세스 토큰 |
| `token_type` | ** 필수** | 토큰 유형 (대소문자 구분 안 함) |
| `expires_in` | ** 권장** | 액세스 토큰 수명 (초 단위) |
| `scope` | 조건부 | 요청과 동일하면 선택, 다르면 필수 |
| `state` | 조건부 필수 | 요청에 있었다면 정확히 동일한 값 |

** 주의:** 인가 서버는 ** 리프레시 토큰 발급 금지**

예시:
```http
HTTP/1.1 302 Found
Location: http://example.com/cb#access_token=2YotnFZFEjr1zCsicMWpAA
          &state=xyz&token_type=example&expires_in=3600
```

** 에러 응답:**

에러 시 ** 프래그먼트 컴포넌트** 에 파라미터 추가:

| 파라미터 | 필수 여부 | 에러 코드 |
|----------|-----------|-----------|
| `error` | ** 필수** | `invalid_request`, `unauthorized_client`, `access_denied`, `unsupported_response_type`, `invalid_scope`, `server_error`, `temporarily_unavailable` |
| `error_description` | 선택 | 사람이 읽을 수 있는 ASCII 텍스트 |
| `error_uri` | 선택 | 에러 정보가 있는 웹 페이지 URI |
| `state` | 조건부 필수 | 요청에 있었다면 정확히 동일한 값 |

### 4.3 Resource Owner Password Credentials Grant

리소스 소유자와 클라이언트 간에 ** 높은 수준의 신뢰** 가 있을 때만 적합

사용 시나리오:
- 클라이언트가 사용자 이름/비밀번호 수집 기능 보유
- 기존 직접 인증 방식을 OAuth로 마이그레이션

```
     +----------+
     | Resource |
     |  Owner   |
     |          |
     +----------+
          v
          |    Resource Owner
         (A) Password Credentials
          |
          v
     +---------+                                  +---------------+
     |         |>--(B)---- Resource Owner ------->|               |
     |         |         Password Credentials     | Authorization |
     | Client  |                                  |     Server    |
     |         |<--(C)---- Access Token ---------<|               |
     |         |    (w/ Optional Refresh Token)   |               |
     +---------+                                  +---------------+
```

| 단계 | 설명 |
|------|------|
| (A) | 리소스 소유자가 클라이언트에 사용자 이름과 비밀번호 제공 |
| (B) | 클라이언트가 토큰 엔드포인트에 자격 증명과 함께 액세스 토큰 요청. 인가 서버에 인증 |
| (C) | 인가 서버가 클라이언트 인증, 리소스 소유자 자격 증명 검증 후 액세스 토큰 발급 |

#### 4.3.1 인가 요청 및 응답 (Authorization Request and Response)

- 리소스 소유자 자격 증명 획득 방법은 이 명세의 ** 범위 밖**
- 클라이언트는 액세스 토큰 획득 후 ** 자격 증명 폐기 필수**

#### 4.3.2 액세스 토큰 요청 (Access Token Request)

| 파라미터 | 필수 여부 | 설명 |
|----------|-----------|------|
| `grant_type` | ** 필수** | 값은 `password`여야 함 |
| `username` | ** 필수** | 리소스 소유자 사용자 이름 |
| `password` | ** 필수** | 리소스 소유자 비밀번호 |
| `scope` | 선택 | 요청 범위 |

예시:
```http
POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=password&username=johndoe&password=A3ddj3w
```

인가 서버 필수 항목:
- Confidential 클라이언트에 대해 ** 클라이언트 인증 필수**
- 인증 포함 시 ** 클라이언트 인증**
- 기존 비밀번호 검증 알고리즘으로 ** 리소스 소유자 비밀번호 자격 증명 검증**

#### 4.3.3 액세스 토큰 응답 (Access Token Response)

- 유효하면 Section 5.1에 따라 액세스 토큰과 선택적 리프레시 토큰 발급
- 인증 실패 또는 유효하지 않으면 Section 5.2에 따라 에러 반환

### 4.4 Client Credentials Grant

** 클라이언트 통제 하의 보호된 리소스** 에 대한 인가 범위이거나 ** 사전에 배열된 인가** 요청 시 사용

클라이언트가 자체적으로 (리소스 소유자 역할도 함) 또는 사전 배열된 인가 기반으로 동작

```
     +---------+                                  +---------------+
     |         |                                  |               |
     |         |>--(A)- Client Authentication --->| Authorization |
     | Client  |                                  |     Server    |
     |         |<--(B)---- Access Token ---------<|               |
     |         |                                  |               |
     +---------+                                  +---------------+
```

| 단계 | 설명 |
|------|------|
| (A) | 클라이언트가 인가 서버에 인증하고 인가 승인(클라이언트 자격 증명) 제시 |
| (B) | 인가 서버가 클라이언트 인증 및 검증 후 액세스 토큰 발급 |

#### 4.4.1 인가 요청 및 응답 (Authorization Request and Response)

이 권한 부여 유형은 인가 요청/응답을 ** 사용하지 않음**. 클라이언트가 토큰을 직접 요청

#### 4.4.2 액세스 토큰 요청 (Access Token Request)

| 파라미터 | 필수 여부 | 설명 |
|----------|-----------|------|
| `grant_type` | ** 필수** | 값은 `client_credentials`여야 함 |
| `scope` | 선택 | 요청 범위 |

클라이언트는 Section 3.2.1에 따라 ** 인가 서버에 인증 필수**

예시:
```http
POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
```

인가 서버 필수 항목:
- ** 클라이언트 인증 필수**
- ** 클라이언트 인증**
- ** 클라이언트 자격 증명 검증**

#### 4.4.3 액세스 토큰 응답 (Access Token Response)

- 유효하면 Section 5.1에 따라 액세스 토큰 발급
- ** 리프레시 토큰 발급 금지**
- 유효하지 않으면 Section 5.2에 따라 에러 반환

### 4.5 확장 권한 부여 (Extension Grants)

- 인가 서버는 이 문서에 정의된 것 외에 ** 추가 권한 부여 유형 지원 가능**

---

## 5. 액세스 토큰 발급 (Issuing an Access Token)

### 5.1 성공 응답 (Successful Response)

인가 서버가 HTTP 200 응답 본문에 `application/json` 콘텐츠 타입으로 JSON 파라미터 반환:

| 파라미터 | 필수 여부 | 설명 |
|----------|-----------|------|
| `access_token` | ** 필수** | 인가 서버가 발급한 액세스 토큰 |
| `token_type` | ** 필수** | 토큰 유형 (대소문자 구분 안 함). 예: RFC 6750의 "Bearer" |
| `expires_in` | ** 권장** | 액세스 토큰 수명 (초 단위) |
| `refresh_token` | 선택 | 리프레시 토큰 (지원하는 권한 부여 유형에만 발급) |
| `scope` | 조건부 | 요청과 동일하면 선택, 다르면 필수 |

필수 헤더:
- `Cache-Control: no-store` ** 포함 필수**
- `Pragma: no-cache` (HTTP/1.0 호환)

추가 파라미터 포함 가능. 클라이언트는 ** 인식하지 못하는 응답 파라미터 무시 필수**

### 5.2 에러 응답 (Error Response)

인가 요청 또는 토큰 요청 실패 시 HTTP 400 상태와 `application/json` 콘텐츠 타입 반환:

| 파라미터 | 필수 여부 | 설명 |
|----------|-----------|------|
| `error` | ** 필수** | ASCII 에러 코드 |
| `error_description` | 선택 | 사람이 읽을 수 있는 ASCII 텍스트 |
| `error_uri` | 선택 | 에러 정보가 있는 웹 페이지 URI |

#### 에러 코드

| 에러 코드 | 설명 |
|-----------|------|
| `invalid_request` | 요청에 필수 파라미터 누락, 잘못된 파라미터 값, 중복 파라미터, 또는 기타 형식 오류 |
| `invalid_client` | 클라이언트 인증 실패 (알 수 없는 클라이언트, 인증 미포함, 지원하지 않는 인증 방법). HTTP 401 상태 코드 반환 가능 |
| `invalid_grant` | 제공된 인가 승인 또는 리프레시 토큰이 유효하지 않음, 만료됨, 취소됨, 리다이렉션 URI 불일치, 다른 클라이언트에 발급됨 |
| `unauthorized_client` | 인증된 클라이언트가 이 인가 승인 유형 사용 권한 없음 |
| `unsupported_grant_type` | 인가 서버가 지원하지 않는 인가 승인 유형 |
| `invalid_scope` | 요청된 범위가 유효하지 않음, 알 수 없음, 형식 오류, 또는 리소스 소유자가 부여한 범위 초과 |

필수 헤더:
- `Cache-Control: no-store` ** 포함 필수**

예시:
```http
HTTP/1.1 400 Bad Request
Content-Type: application/json;charset=UTF-8
Cache-Control: no-store
Pragma: no-cache

{
  "error":"invalid_request"
}
```

## 6. 액세스 토큰 갱신 (Refreshing an Access Token)

클라이언트가 리프레시 토큰을 사용하여 새 액세스 토큰 획득

| 파라미터 | 필수 여부 | 설명 |
|----------|-----------|------|
| `grant_type` | ** 필수** | 값은 `refresh_token`이어야 함 |
| `refresh_token` | ** 필수** | 클라이언트에 발급된 리프레시 토큰 |
| `scope` | 선택 | 요청 범위. 원래 부여된 범위를 ** 초과할 수 없음** |

Confidential 클라이언트는 Section 3.2.1에 따라 ** 인가 서버에 인증 필수**

예시:
```http
POST /token HTTP/1.1
Host: server.example.com
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token&refresh_token=tGzv3JOkF0XG5Qx2TlKWIA
```

인가 서버 필수 항목:
- Confidential 클라이언트에 대해 ** 클라이언트 인증 필수**
- 인증 포함 시 ** 클라이언트 인증**
- ** 리프레시 토큰 검증**
- 리프레시 토큰이 인증된 Confidential 클라이언트 또는 요청의 `client_id`에 발급되었는지 확인

인가 서버는 Section 5.1에 따라 액세스 토큰과 선택적으로 ** 새 리프레시 토큰** 반환

## 7. 보호된 리소스 접근 (Accessing Protected Resources)

### 7.1 액세스 토큰 유형 (Access Token Types)

- 토큰 형식/구조/활용 방법은 이 명세의 ** 범위 밖**
- RFC 6750 등 동반 명세에서 정의

### 7.2 에러 응답 (Error Response)

- 리소스 서버가 액세스 토큰 검증
- 유효하지 않거나 만료되면 **HTTP 401 Unauthorized** 반환
- `WWW-Authenticate` 응답 헤더 필드에 RFC 6750 또는 동반 명세에 따른 에러 정보 포함

## 8. 확장성 (Extensibility)

### 8.1 액세스 토큰 유형 정의 (Defining Access Token Types)

- 새 토큰 유형은 **IANA OAuth Access Token Types Registry** 에 등록하여 정의 가능
- 토큰 유형 명세는 속성과 보호 방법 정의

### 8.2 새 엔드포인트 파라미터 정의 (Defining New Endpoint Parameters)

- 엔드포인트는 새 파라미터로 확장 가능
- 새 파라미터는 Section 11.2 절차에 따라 **IANA OAuth Parameters Registry** 에 등록

### 8.3 새 인가 승인 유형 정의 (Defining New Authorization Grant Types)

- 새 권한 부여 유형 정의 가능
- 명세에 클라이언트와 인가 서버 간 상호작용 상세, 보안 고려사항 포함
- IANA에 권한 부여 유형 등록

### 8.4 새 인가 엔드포인트 응답 유형 정의 (Defining New Authorization Endpoint Response Types)

- 인가 엔드포인트에 대한 새 응답 유형 정의 가능
- 명세에 상호작용 단계와 파라미터 상세, 보안 고려사항 포함
- Section 11.3 절차에 따라 IANA에 응답 유형 등록

### 8.5 추가 에러 코드 정의 (Defining Additional Error Codes)

- 새 에러 코드 정의 가능
- 명세에 에러 조건과 의미 상세
- Section 11.4에 따라 **IANA OAuth Extensions Error Registry** 에 등록

** 표준 에러 코드:**
- Authorization Endpoint: `invalid_request`, `unauthorized_client`, `access_denied`, `unsupported_response_type`, `invalid_scope`, `server_error`, `temporarily_unavailable`
- Token Endpoint: `invalid_client`, `invalid_grant`, `unauthorized_client`, `unsupported_grant_type`

** 에러 코드 값 요구사항:**
- 문자 제한 준수 필수: `%x20-21 / %x23-5B / %x5D-7E`

## 9. 네이티브 애플리케이션 (Native Applications)

- 네이티브 애플리케이션은 장치에 설치 및 실행되는 **Public 클라이언트**
- 프로토콜 데이터와 자격 증명이 ** 리소스 소유자에게 접근 가능**
- 애플리케이션에 포함된 클라이언트 인증 자격 증명은 ** 추출 가능** 하다고 가정
- 그러나 동적으로 발급된 자격 증명(액세스/리프레시 토큰)은 ** 수용 가능한 수준의 보호** 가능
  - 최소한 애플리케이션이 상호작용하는 적대적 서버로부터 보호됨
- 네이티브 애플리케이션은 **authorization code 권한 부여 유형 사용 가능**
  - 리소스 소유자가 인가 과정 전반에 걸쳐 애플리케이션과 상호작용할 것으로 예상

## 10. 보안 고려사항 (Security Considerations)

### 10.1 클라이언트 인증 (Client Authentication)

- Confidential 클라이언트는 ** 강력한 인증** 활용 권장
- Public 클라이언트 인증은 ** 클라이언트 식별 목적으로 의존해서는 안 됨**
- 인가 서버는 인증 방법의 보안 속성을 신중히 고려

### 10.2 클라이언트 가장 (Client Impersonation)

- 미등록 클라이언트는 ** 가장 위험** 야기
- 공격자가 합법적 클라이언트로 등록하고 합법적 클라이언트 사이트의 오픈 리다이렉션 URI를 사용하여 사용자 리다이렉션 가능
- 인가 서버는 ** 가장 방지 보안 정책** 적용

### 10.3 액세스 토큰 (Access Tokens)

액세스 토큰 권장사항:
- ** 제한된 수명**
- ** 필요한 범위로 제한**
- ** 추출/수정으로부터 보호**
- 클라이언트는 토큰 값을 ** 추측하거나 예측하려 해서는 안 됨**

### 10.4 리프레시 토큰 (Refresh Tokens)

리프레시 토큰은 리소스 소유자 자격 증명 저장 없이 ** 장기 접근** 가능

권장사항:
- ** 업계 모범 사례** 로 보호
- 침해 시 ** 취소**
- **TLS로만 전송**
- ** 필요한 범위로 제한**

### 10.5 인가 코드 (Authorization Codes)

인가 코드 권장사항:
- 발급 후 ** 짧은 시간 내 만료**
- **1회만 사용**
- ** 정확한 리다이렉션 URI 매칭** 필요

### 10.6 인가 코드 리다이렉션 URI 조작 (Authorization Code Redirection URI Manipulation)

- 공격자가 리다이렉션 URI를 조작하여 인가 코드 캡처 가능
- 인가 코드는 ** 클라이언트 식별자와 리다이렉션 URI에 바인딩** 필수

### 10.7 리소스 소유자 비밀번호 자격 증명 (Resource Owner Password Credentials)

이 권한 부여 유형은 자격 증명을 노출하므로 ** 다른 흐름을 사용할 수 없거나 높은 신뢰 시나리오** 에서만 사용

클라이언트 권장사항:
- 자격 증명 ** 저장 안 함**
- ** 단일 요청 교환 모델** 사용
- **TLS로만 전송**

### 10.8 요청 기밀성 (Request Confidentiality)

인가 엔드포인트, 토큰 엔드포인트, 리다이렉션 엔드포인트에 대한 모든 요청은 ** 기밀성 보호를 위해 TLS 사용** 권장

### 10.9 엔드포인트 진위 확인 (Ensuring Endpoint Authenticity)

클라이언트는 ** 인증된 연결과 엔드포인트 검색 메커니즘** 을 통해 엔드포인트 진위 검증 권장

### 10.10 자격 증명 추측 공격 (Credentials-Guessing Attacks)

인가 서버 권장사항:
- ** 속도 제한** 구현
- ** 강력한 자격 증명** 요구
- ** 계정 잠금 메커니즘** 활용

### 10.11 피싱 공격 (Phishing Attacks)

- 사용자는 ** 신뢰할 수 있는 인가 서버에서만 인증**
- 클라이언트는 사용자를 ** 신뢰할 수 없는 사이트로 리다이렉션 금지**
- 인가 서버 권장사항:
  - ** 표준 인증 메커니즘** 사용
  - 의심스러운 활동에 대해 ** 사용자 경고**
  - ** 보안 표시기** 구현

### 10.12 교차 사이트 요청 위조 (Cross-Site Request Forgery, CSRF)

- 인가 엔드포인트는 **CSRF 보호** 적용
- **`state` 파라미터 사용 권장** 하여 CSRF 방지

### 10.13 클릭재킹 (Clickjacking)

- 리다이렉션 엔드포인트는 투명한 프레임을 통한 ** 클릭재킹 공격** 대상
- 서버는 **`X-Frame-Options` 응답 헤더를 사용하여 프레이밍 거부**

### 10.14 코드 주입 및 입력 검증 (Code Injection and Input Validation)

- 모든 엔드포인트에서 ** 입력 검증 중요**
- 파라미터의 특수 문자가 ** 코드 주입** 가능하게 할 수 있음

### 10.15 오픈 리다이렉터 (Open Redirectors)

- 미등록 리다이렉션 URI는 ** 오픈 리다이렉터 공격** 가능하게 함
- 리다이렉션 URI 등록 요구 부재는 공격자가 ** 인가 엔드포인트를 오픈 리다이렉터로 악용** 가능하게 함

### 10.16 Implicit 흐름에서 액세스 토큰 오용 (Misuse of Access Token to Impersonate Resource Owner in Implicit Flow)

- URI 프래그먼트의 액세스 토큰이 리소스 소유자 또는 다른 애플리케이션에 ** 노출될 수 있음**
- 토큰 권장사항:
  - ** 제한된 수명**
  - **1회 또는 제한 사용**
  - ** 신뢰할 수 없는 사이트로 전송 금지**

## 부록 B: application/x-www-form-urlencoded 미디어 유형

### 인코딩 규칙

`application/x-www-form-urlencoded` 형식은 파라미터를 키-값 쌍으로 인코딩

**Token 응답:**
- `application/json` 미디어 타입으로 HTTP 응답 엔티티 본문에 포함

**Authorization 요청:**
- `application/x-www-form-urlencoded` 형식으로 쿼리 구성요소에 추가

### 구현 가이드

1. ** 쿼리 구성요소 인코딩:**
   - URI에 추가되는 파라미터는 RFC 3986에 따라 퍼센트 인코딩
   - 공백 문자는 `%20`으로 인코딩

2. ** 엔티티 본문 인코딩:**
   - 폼 인코딩 요청 본문:
     - `&`로 구분된 키-값 쌍
     - 키와 값은 퍼센트 인코딩
     - `+`는 공백을 나타냄 (레거시) 또는 `%20` (권장)

3. ** 문자 집합:**
   - 요청 본문의 비ASCII 문자에 UTF-8 인코딩 필수

### 인코딩 요약

- 폼 인코딩 값은 **UTF-8 인코딩** 사용
- 예약되지 않은 문자는 인코딩하지 않음
- 기타 모든 옥텟은 RFC 3986 Section 2.1에 따라 ** 퍼센트 인코딩**
- 공백 문자는 ** 플러스 기호(+)** 또는 ** 퍼센트 인코딩(%20)**
- POST 요청의 경우 `Content-Type: application/x-www-form-urlencoded`로 요청 본문에 파라미터 전송

### 예시

```http
POST /token HTTP/1.1
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token&refresh_token=tGzv3JOkF0XG5Qx2TlKWIA
&client_id=s6BhdRkqt3&client_secret=7Fjfp0ZBr1KtDRbnfVdmIw
```

** 중요:**
- 본문 형식의 클라이언트 자격증명은 요청 본문으로만 전송 가능
- 요청 URI에 포함 금지 (MUST NOT)
