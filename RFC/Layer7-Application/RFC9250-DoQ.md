# RFC 9250 - DNS over Dedicated QUIC Connections (DoQ)

> **발행일**: 2022년 5월
> **저자**: Christian Huitema, Sara Dickinson, Allison Mankin
> **상태**: Standards Track

## 1. 개요

RFC 9250은 **QUIC 전송을 사용하여 DNS 메시지를 전송**하는 프로토콜인 DNS over QUIC (DoQ)를 정의합니다. QUIC이 제공하는 암호화와 함께 전통적인 DNS 전송 대비 개선된 지연 시간 및 효율적인 패킷 손실 복구를 제공합니다.

### DoQ가 제공하는 핵심 기능

| 기능 | 설명 |
|------|------|
| **암호화** | QUIC/TLS 1.3을 통한 DNS 트래픽 암호화 |
| **낮은 지연** | 0-RTT/1-RTT 핸드셰이크로 빠른 연결 |
| **병렬 처리** | 독립 스트림으로 Head-of-line blocking 제거 |
| **연결 마이그레이션** | 네트워크 변경 시 연결 유지 |

### DoQ vs DoH vs DoT 비교

```
┌─────────────┬─────────────┬─────────────┬─────────────┐
│             │    DoQ      │    DoH      │    DoT      │
├─────────────┼─────────────┼─────────────┼─────────────┤
│ 전송 계층    │ QUIC (UDP)  │ HTTP (TCP)  │ TCP         │
│ 암호화      │ TLS 1.3     │ TLS 1.3     │ TLS 1.2/1.3 │
│ 포트        │ 853/UDP     │ 443/TCP     │ 853/TCP     │
│ 0-RTT       │ 지원        │ HTTP/3만    │ 미지원      │
│ 멀티플렉싱  │ 네이티브    │ HTTP/2+     │ 미지원      │
└─────────────┴─────────────┴─────────────┴─────────────┘
```

## 2. 연결 설정

### 2.1 포트 및 ALPN

```
기본 포트: UDP 853

ALPN (Application-Layer Protocol Negotiation):
- 토큰: "doq"
- TLS 핸드셰이크 중 DoQ 지원 표시

⚠️ 포트 53 사용 금지:
- 표준 DNS와의 혼동 방지
- 명확한 프로토콜 구분
```

### 2.2 연결 흐름

```
클라이언트                                서버
    │                                      │
    │──── QUIC Initial ──────────────────>│
    │     (ClientHello + ALPN "doq")       │
    │                                      │
    │<─── QUIC Initial ───────────────────│
    │     (ServerHello)                    │
    │                                      │
    │──── QUIC Handshake ────────────────>│
    │     (Finished)                       │
    │                                      │
    │<─── QUIC Handshake ─────────────────│
    │     (Finished)                       │
    │                                      │
    │     ═══════════════════════════════  │
    │          1-RTT 연결 완료             │
    │     ═══════════════════════════════  │
    │                                      │
    │──── Stream 0: DNS Query ───────────>│
    │<─── Stream 0: DNS Response ─────────│
    │                                      │
    │──── Stream 4: DNS Query ───────────>│
    │<─── Stream 4: DNS Response ─────────│
```

이전 연결 정보로 즉시 데이터 전송:

클라이언트                                서버
    │                                      │
    │──── QUIC Initial ──────────────────>│
    │     + 0-RTT Data (DNS Query)         │
    │                                      │

    │                                      │

```

## 3. 스트림 매핑

### 3.1 스트림 구조

```
각 DNS 트랜잭션은 별도의 양방향 스트림 사용:

스트림 ID 규칙 (클라이언트 시작):
- 스트림 0: 첫 번째 쿼리
- 스트림 4: 두 번째 쿼리
- 스트림 8: 세 번째 쿼리
- ...
- 4씩 증가

QUIC 스트림 타입:
┌─────────────────────────────────────┐
│ Least Significant 2 Bits            │
├──────────┬──────────────────────────┤
│   0x0    │ 클라이언트 시작, 양방향   │ ← DoQ 사용
│   0x1    │ 서버 시작, 양방향         │
│   0x2    │ 클라이언트 시작, 단방향   │
│   0x3    │ 서버 시작, 단방향         │
└──────────┴──────────────────────────┘
```

### 3.2 메시지 형식

```
DoQ 메시지 구조:

┌─────────────────────────────────────┐
│      Length (2 bytes)               │
├─────────────────────────────────────┤
│                                     │
│      DNS Message                    │
│      (RFC 1035 형식)                │
│                                     │
└─────────────────────────────────────┘

특징:
- DNS-over-TCP와 동일한 2바이트 길이 접두사
- 최대 65,535 바이트
```

### 3.3 DNS 메시지 ID

```
⚠️ 중요 규칙:

"QUIC 연결을 통해 쿼리를 전송할 때,
DNS 메시지 ID는 반드시 0으로 설정되어야 합니다 (MUST)"

이유:
- QUIC 스트림이 요청/응답 상관관계 제공
- Message ID 불필요
- 캐싱 효율성 향상 (동일 쿼리 = 동일 바이트)

예시:
┌────────────────┬──────────────────┐
│ 전통적 DNS     │ DNS Message ID   │
│                │ = 랜덤 값        │
├────────────────┼──────────────────┤
│ DoQ            │ DNS Message ID   │
│                │ = 0              │
└────────────────┴──────────────────┘
```

### 3.4 스트림 병렬 처리

```
여러 쿼리 동시 처리:

┌─────────────────────────────────────────────────────┐
│ QUIC Connection                                     │
│                                                     │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐            │
│  │ Stream 0 │ │ Stream 4 │ │ Stream 8 │ ...        │
│  │ Query A  │ │ Query B  │ │ Query C  │            │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘            │
│       ▼            ▼            ▼                   │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐            │
│  │Response A│ │Response B│ │Response C│            │
│  └──────────┘ └──────────┘ └──────────┘            │
│                                                     │
│  → 각 스트림 독립적, 패킷 손실이 다른 쿼리에 영향 없음  │
└─────────────────────────────────────────────────────┘
```

## 4. 오류 처리

### 4.1 DoQ 오류 코드

| 코드 | 이름 | 값 | 설명 |
|------|------|-----|------|
| DOQ_NO_ERROR | 정상 종료 | 0x0 | 오류 없이 연결/스트림 종료 |
| DOQ_INTERNAL_ERROR | 내부 오류 | 0x1 | 구현 내부 오류 |
| DOQ_PROTOCOL_ERROR | 프로토콜 오류 | 0x2 | DoQ 프로토콜 위반 |
| DOQ_REQUEST_CANCELLED | 요청 취소 | 0x3 | 클라이언트가 요청 취소 |
| DOQ_EXCESSIVE_LOAD | 과부하 | 0x4 | 서버 과부하로 거부 |
| DOQ_UNSPECIFIED_ERROR | 미지정 오류 | 0x5 | 기타 오류 |

### 4.2 오류 사용 예시

```
스트림 수준 오류:

클라이언트가 쿼리 취소:
RESET_STREAM(DOQ_REQUEST_CANCELLED)

서버가 과부하:
RESET_STREAM(DOQ_EXCESSIVE_LOAD)

프로토콜 위반:
RESET_STREAM(DOQ_PROTOCOL_ERROR)

연결 수준 오류:

심각한 프로토콜 위반:
CONNECTION_CLOSE(DOQ_PROTOCOL_ERROR)

정상 종료:
CONNECTION_CLOSE(DOQ_NO_ERROR)
```

### 4.3 QUIC vs DoQ 오류

```
계층별 오류 구분:

QUIC 전송 오류:
- 연결 실패
- 흐름 제어 위반
- 타임아웃

DoQ 애플리케이션 오류:
- 잘못된 DNS 메시지 형식
- 프로토콜 위반
- 서버 과부하

DNS 수준 오류:
- SERVFAIL
- NXDOMAIN
- REFUSED
(이들은 정상적인 DNS 응답으로 처리)
```

## 5. 보안 및 프라이버시

### 5.1 TLS 1.3 기반 보안

```
DoQ 보안 기반:

QUIC → TLS 1.3 → 보안 제공

보안 속성:
┌─────────────────────────────────────┐
│ 기밀성 (Confidentiality)            │
│ - 모든 DNS 데이터 암호화            │
│ - 패킷 헤더 일부도 암호화           │
└─────────────────────────────────────┘
┌─────────────────────────────────────┐
│ 무결성 (Integrity)                  │
│ - AEAD 암호화로 변조 감지           │
│ - 재전송 공격 방지                  │
└─────────────────────────────────────┘
┌─────────────────────────────────────┐
│ 인증 (Authentication)               │
│ - 서버 인증서 검증                  │
│ - 클라이언트 인증 선택적            │
└─────────────────────────────────────┘
┌─────────────────────────────────────┐
│ 다운그레이드 방지                   │
│ - TLS 1.3 필수                      │
│ - 레거시 프로토콜 불허              │
└─────────────────────────────────────┘
```

### 5.2 인증 요구사항

```
RFC 7858 (DoT)와 유사한 인증:

서버 인증:
- X.509 인증서 필수
- 인증서 체인 검증
- 호스트명 검증

클라이언트 인증:
- 선택적
- mTLS (mutual TLS) 가능
```

### 5.3 0-RTT 보안 제한

```
⚠️ 0-RTT 주의사항:

"0-RTT 메커니즘은 '재생 가능한' 트랜잭션이 아닌
DNS 요청을 보내는 데 사용되어서는 안 됩니다 (MUST NOT)"

재생 가능한 트랜잭션:
✓ QUERY (표준 DNS 조회)
✓ NOTIFY

재생 불가능한 트랜잭션:
✗ UPDATE (DNS 동적 업데이트)
✗ 상태 변경 작업

이유:
- 0-RTT 데이터는 재생 공격에 취약
- 공격자가 0-RTT 패킷을 복사하여 재전송 가능
- 멱등(idempotent) 작업만 안전
```

### 5.4 세션 재개 프라이버시

```
세션 재개의 추적 위험:

문제:
- 세션 티켓이 클라이언트 식별자로 동작
- 서버가 세션 간 클라이언트 추적 가능

권장사항:

1. 일회용 티켓
   - 각 티켓을 한 번만 사용
   - 재사용 시 새 티켓 발급

2. 주소 변경 시 재개 거부
   - IP 주소 변경 감지
   - 새 연결 강제

3. 클라이언트 측 티켓 관리
   - 오래된 티켓 폐기
   - 티켓 재사용 제한
```

## 6. 연결 관리

### 6.1 유휴 타임아웃

```
QUIC idle_timeout 협상:

목적:
- 비활성 연결 정리
- 리소스 해제

권장:
- 클라이언트와 서버가 협상
- 너무 짧으면 재연결 오버헤드
- 너무 길면 리소스 낭비

예시:
max_idle_timeout = 30초
→ 30초간 활동 없으면 연결 종료
```

### 6.2 연결 재사용

```
연결 재사용 권장:

이점:
- 핸드셰이크 오버헤드 제거
- 0-RTT 가능
- 암호화 컨텍스트 유지

클라이언트 동작:
1. 기존 연결 확인
2. 유휴 시간 확인
3. 재사용 가능하면 재사용
4. 아니면 새 연결

예시:
┌─────────────────────────────────────┐
│ 쿼리 1 ───> [연결 생성] ───> 응답 1 │
│ 쿼리 2 ───> [연결 재사용] ──> 응답 2│
│ 쿼리 3 ───> [연결 재사용] ──> 응답 3│
│ ...                                 │
│ (idle timeout 후)                   │
│ 쿼리 N ───> [새 연결] ───────> 응답 N│
└─────────────────────────────────────┘
```

### 6.3 연결 마이그레이션

```
네트워크 변경 시 연결 유지:

시나리오:
WiFi → 셀룰러 전환
IP 주소 변경

QUIC 동작:
- Connection ID로 연결 식별
- IP 주소 변경해도 연결 유지
- 진행 중인 쿼리 계속 처리

⚠️ 프라이버시 고려:
- 연결 마이그레이션 = 추적 가능성
- 프라이버시 중요 시 새 연결 권장
```

## 7. 흐름 제어 및 리소스 보호

### 7.1 QUIC 흐름 제어

```
스트림 및 연결 수준 흐름 제어:

스트림 수준:
- 각 스트림별 데이터 제한
- 수신자가 한계 설정

연결 수준:
- 모든 스트림 합계 제한
- 전체 버퍼 관리

DoQ 영향:
- 대형 DNS 응답 분할 가능
- 흐름 제어 한계 존중 필수
```

### 7.2 증폭 공격 방지

```
주소 검증:

QUIC 3-way 핸드셰이크:
1. 클라이언트 → 서버: Initial
2. 서버 → 클라이언트: Initial (주소 검증)
3. 클라이언트 → 서버: Handshake

증폭 제한:
- 검증 전 응답 크기 제한
- 3배 증폭 제한 (RFC 9000)

DoQ 적용:
- 0-RTT 응답도 제한 적용
- 대형 응답은 핸드셰이크 후
```

### 7.3 리소스 관리

```
서버 리소스 보호:

1. 동시 스트림 제한
   initial_max_streams_bidi = N
   → 동시 쿼리 수 제한

2. 연결 수 제한
   - 클라이언트 IP당 연결 제한
   - 전역 연결 제한

3. 과부하 처리
   DOQ_EXCESSIVE_LOAD 오류로 거부
```

## 8. 배포 시나리오

### 8.1 지원 시나리오

```
DoQ 세 가지 배포 모델:

1. Stub → Recursive
   ┌────────┐         ┌───────────┐
   │ 클라이언트│ ─DoQ─> │재귀 해석기 │
   └────────┘         └───────────┘
   - 가장 일반적
   - 최종 사용자 프라이버시

2. Recursive → Authoritative
   ┌───────────┐         ┌──────────┐
   │재귀 해석기 │ ─DoQ─> │권한 서버  │
   └───────────┘         └──────────┘
   - 해석기-서버 간 보호
   - 서버 운영자 선택

3. Zone Transfer
   ┌──────────┐         ┌──────────┐
   │ 주 서버   │ ─DoQ─> │ 보조 서버 │
   └──────────┘         └──────────┘
   - 영역 전송 암호화
   - AXFR/IXFR 보호
```

### 8.2 DNS-over-TCP와 비교

```
DoQ가 DoT 대비 개선한 점:

1. 연결 오버헤드 감소
   DoT (TCP+TLS): 3-RTT 핸드셰이크
   DoQ (QUIC):    1-RTT 핸드셰이크

2. 0-RTT 지원
   DoT: 미지원
   DoQ: 재연결 시 0-RTT

3. Head-of-line blocking 제거
   DoT: 하나의 TCP 스트림, 손실 시 전체 지연
   DoQ: 독립 스트림, 손실이 다른 쿼리에 영향 없음

4. 연결 마이그레이션
   DoT: IP 변경 시 재연결 필요
   DoQ: 연결 유지 가능
```

## 9. 트래픽 분석 보호

### 9.1 패딩 요구사항

```
트래픽 분석 위협:

관찰 가능:
- 패킷 크기
- 타이밍
- 패턴

DoQ 패딩 권장:

1. 쿼리 패딩
   - 모든 쿼리를 유사한 크기로
   - EDNS Padding 옵션 사용

2. 응답 패딩
   - 서버가 응답 패딩
   - 블록 크기로 정규화

3. QUIC 프레임 패딩
   - PADDING 프레임 사용
   - 패킷 크기 균일화

예시:
┌────────────────────────────────────┐
│ 원본 쿼리: 45 바이트               │
│ 패딩 후:   128 바이트 (블록 크기)  │
└────────────────────────────────────┘
```

### 9.2 타이밍 분석 방어

```
타이밍 정보 노출:

위험:
- 쿼리 간격으로 사용자 행동 추론
- 응답 시간으로 서버 상태 추론

완화:
- 쿼리 배치 처리
- 응답 지연 추가 (트레이드오프)
- 지터(jitter) 도입
```

2. 병렬 쿼리
   - 여러 스트림 동시 사용
   - 응답 순서 무관하게 처리

3. 0-RTT 활용

   - 실패 시 1-RTT 폴백

4. 오류 처리
   - DOQ 오류 코드 적절히 처리
   - 재시도 전략 구현
```

## 12. 설계 철학

### 12.1 목표

```
DoQ 설계 목표:

1. DNS over TLS와 유사한 프라이버시
   - 동등한 암호화 수준
   - 동등한 인증 요구사항

2. 전통적 DNS와 유사한 지연 특성
   - UDP DNS 수준의 응답성
   - 빠른 쿼리 완료

3. 단순성
   - HTTP 계층 없음 (DoH와 차이)
   - 직접 QUIC 사용
```

### 12.2 명시적 비목표

```
DoQ가 시도하지 않는 것:

1. 중간박스 탐지 회피
   - 포트 853 사용으로 식별 가능
   - 의도적으로 회피하지 않음

2. 서버 시작 트랜잭션
   - 서버 푸시 미지원
   - 클라이언트 시작만

3. HTTP 호환성
   - HTTP 계층 없음
   - 웹 캐시 미활용
   (DoH와 차별점)
```

## 요약

RFC 9250 DNS over QUIC (DoQ)는 다음을 제공합니다:

- **QUIC 기반 암호화**: TLS 1.3으로 DNS 트래픽 보호
- **낮은 지연**: 1-RTT/0-RTT 핸드셰이크
- **병렬 처리**: 독립 스트림으로 Head-of-line blocking 제거
- **연결 마이그레이션**: 네트워크 변경 시 연결 유지
- **효율적 오류 처리**: 스트림/연결 수준 오류 분리

DoQ는 **DoT의 보안과 UDP DNS의 성능을 결합**한 현대적인 DNS 전송 프로토콜입니다.
