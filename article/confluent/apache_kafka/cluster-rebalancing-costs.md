>

## 서론

Apache Kafka는 분산 이벤트 스트리밍의 사실상 표준이 되었다. 조직들은 실시간 데이터 파이프라인과 스트리밍 애플리케이션을 구축하기 위해 Kafka에 의존하고 있다. 그러나 Kafka 클러스터를 운영하는 데 있어 종종 간과되는 비용 중 하나가 바로 **클러스터 리밸런싱**이다.

리밸런싱은 Kafka의 핵심 메커니즘으로, 컨슈머 그룹 내에서 파티션 할당을 재분배하는 프로세스이다. 컨슈머가 그룹에 참여하거나 떠날 때, 또는 토픽의 파티션 수가 변경될 때 리밸런싱이 트리거된다. 이 프로세스는 정확성과 공정한 부하 분산을 보장하지만, 상당한 운영 비용을 수반할 수 있다.

---

## 리밸런싱이란 무엇인가?

Kafka 컨슈머 그룹에서 각 컨슈머 인스턴스는 토픽의 하나 이상의 파티션에서 메시지를 읽는다. **리밸런싱**은 그룹 내 컨슈머들 간에 파티션 소유권을 재분배하는 프로토콜이다.

리밸런싱은 다음과 같은 상황에서 트리거된다:

- **컨슈머가 그룹에 참여할 때** - 새로운 컨슈머 인스턴스가 시작되는 경우
- **컨슈머가 그룹을 떠날 때** - 컨슈머가 종료되거나 크래시되는 경우
- **컨슈머가 비정상으로 간주될 때** - 하트비트 타임아웃이 만료되는 경우
- **토픽 메타데이터가 변경될 때** - 파티션이 추가되는 경우
- **구독이 변경될 때** - 컨슈머가 새로운 토픽을 구독하는 경우

리밸런싱 프로세스는 그룹 코디네이터(Group Coordinator)라 불리는 특정 Kafka 브로커에 의해 조정된다. 그룹 코디네이터는 컨슈머 그룹의 멤버십을 관리하고, 리밸런싱이 필요할 때 프로토콜을 시작한다.

## 전통적인 "Stop-the-World" 리밸런싱의 비용

Kafka의 초기 리밸런싱 프로토콜은 **Eager Rebalancing**(즉시 리밸런싱)이라고 불리며, 이는 본질적으로 "stop-the-world" 접근 방식이다. 이 프로토콜에서는 리밸런싱이 트리거될 때 다음과 같은 일이 발생한다:

1. **모든 컨슈머가 파티션 할당을 포기한다** - 그룹의 모든 컨슈머가 현재 소유한 모든 파티션을 반납한다.
2. **모든 처리가 중단된다** - 파티션 할당이 없으므로 어떤 컨슈머도 메시지를 읽을 수 없다.
3. **그룹이 다시 참여한다** - 모든 컨슈머가 JoinGroup 요청을 보내고, 리더가 새로운 할당을 계산한다.
4. **새로운 할당이 배포된다** - SyncGroup 응답을 통해 각 컨슈머에게 새로운 파티션이 할당된다.
5. **처리가 재개된다** - 컨슈머가 할당받은 파티션에서 다시 메시지를 소비하기 시작한다.

### 이 접근 방식의 문제점

Eager Rebalancing의 가장 큰 문제는 **리밸런싱 기간 동안 전체 컨슈머 그룹이 완전히 중단**된다는 것이다. 이로 인해 다음과 같은 비용이 발생한다:

- **처리 중단 시간:** 리밸런싱이 진행되는 동안 어떤 메시지도 소비되지 않는다. 대규모 컨슈머 그룹에서는 이 중단이 수 초에서 수 분까지 지속될 수 있다.

- **컨슈머 랙 증가:** 처리가 중단되는 동안 프로듀서는 계속 메시지를 보내므로, 컨슈머 랙이 급격히 증가한다. 리밸런싱이 완료된 후 이 랙을 따라잡는 데 추가 시간이 소요된다.

- **상태 재구축 비용:** Kafka Streams와 같은 상태 기반 애플리케이션에서는 리밸런싱 후 로컬 상태 저장소를 재구축해야 할 수 있다. 이는 changelog 토픽에서 대량의 데이터를 다시 읽어야 함을 의미하며, 복구 시간이 상당히 길어질 수 있다.

- **중복 처리 가능성:** `enable.auto.commit=true`가 설정된 경우, 리밸런싱 전에 처리되었지만 아직 커밋되지 않은 오프셋이 있을 수 있다. 리밸런싱 후 새로운 컨슈머가 해당 파티션을 인수하면 마지막으로 커밋된 오프셋부터 다시 처리하게 되어 중복이 발생할 수 있다.

- **연쇄 리밸런싱(Cascading Rebalances):** 리밸런싱이 오래 걸리면, 일부 컨슈머가 `session.timeout.ms` 내에 하트비트를 보내지 못해 그룹에서 제외될 수 있다. 이는 또 다른 리밸런싱을 트리거하여, 이른바 "리밸런싱 폭풍(rebalance storm)"이 발생할 수 있다.

## 리밸런싱의 실제 비용 측정

리밸런싱의 비용은 여러 요인에 따라 달라진다:

### 컨슈머 그룹의 크기

컨슈머 그룹이 클수록 리밸런싱에 더 오래 걸린다. 모든 컨슈머가 JoinGroup 요청을 보내고 SyncGroup 응답을 기다려야 하기 때문이다. `max.poll.interval.ms` 타임아웃 내에 리밸런싱이 완료되지 않으면 컨슈머가 제외되어 연쇄 리밸런싱이 발생할 수 있다.

### 파티션 수

파티션 수가 많을수록 할당 알고리즘의 계산 시간이 증가하고, 각 컨슈머가 처리해야 하는 메타데이터 양도 늘어난다.

### 상태 기반 처리

Kafka Streams를 사용하는 경우, 파티션 재할당은 로컬 상태 저장소의 마이그레이션을 의미한다. 상태 저장소의 크기가 클수록 복구 시간이 길어진다.

### 네트워크 지연

분산 환경에서 컨슈머들이 지리적으로 분산되어 있다면, JoinGroup과 SyncGroup 프로토콜의 왕복 시간이 증가하여 리밸런싱이 더 오래 걸린다.

## 비용을 줄이기 위한 개선: 협력적 리밸런싱

Kafka 2.4부터 도입된 **Cooperative Rebalancing**(협력적 리밸런싱, 일명 Incremental Cooperative Rebalancing)은 전통적인 Eager Rebalancing의 "stop-the-world" 문제를 해결하기 위해 설계되었다.

### 협력적 리밸런싱의 동작 방식

Eager 프로토콜과 달리, 협력적 리밸런싱에서는:

1. **컨슈머가 기존 할당을 유지한다** - 리밸런싱이 시작되더라도 컨슈머는 현재 소유한 파티션을 즉시 반납하지 않는다.
2. **점진적 재할당이 이루어진다** - 실제로 이동이 필요한 파티션만 해제되고 재할당된다.
3. **두 단계로 진행된다** - 첫 번째 리밸런싱에서 이동할 파티션을 식별하고, 두 번째 리밸런싱에서 해당 파티션을 새 소유자에게 할당한다.

이 접근 방식의 핵심 이점은 **이동이 필요하지 않은 파티션의 처리가 계속된다**는 것이다. 즉, 리밸런싱 중에도 대부분의 파티션에서 메시지 소비가 중단되지 않는다.

### 협력적 리밸런싱 활성화

협력적 리밸런싱을 사용하려면 컨슈머 설정에서 `partition.assignment.strategy`를 변경해야 한다:

```properties
# 컨슈머 설정
partition.assignment.strategy=org.apache.kafka.clients.consumer.CooperativeStickyAssignor
```

Kafka Streams의 경우:

```properties
# Kafka Streams 설정
upgrade.from=<이전 버전>
```

### Eager에서 Cooperative로의 롤링 업그레이드

기존 Eager 프로토콜에서 Cooperative 프로토콜로 전환할 때는 **롤링 업그레이드**를 수행해야 한다. 모든 컨슈머를 동시에 변경하면 호환성 문제가 발생할 수 있기 때문이다.

1. **1단계:** 모든 컨슈머의 `partition.assignment.strategy`에 두 가지 전략을 모두 포함하도록 설정한다:
   ```properties
   partition.assignment.strategy=org.apache.kafka.clients.consumer.CooperativeStickyAssignor,org.apache.kafka.clients.consumer.RangeAssignor
   ```

2. **2단계:** 모든 컨슈머가 업데이트된 후, Eager 전략을 제거한다:
   ```properties
   partition.assignment.strategy=org.apache.kafka.clients.consumer.CooperativeStickyAssignor
   ```

## 정적 그룹 멤버십(Static Group Membership)

Kafka 2.3에서 도입된 **정적 그룹 멤버십**은 리밸런싱의 빈도를 줄이는 또 다른 중요한 최적화이다.

### 문제: 불필요한 리밸런싱

기본적으로 Kafka 컨슈머는 그룹에 참여할 때마다 새로운 멤버 ID를 부여받는다. 이는 컨슈머가 일시적으로 재시작되는 경우(예: 롤링 배포)에도 "새로운" 컨슈머로 인식되어 리밸런싱이 트리거됨을 의미한다.

### 해결: group.instance.id

정적 그룹 멤버십은 각 컨슈머에 고유한 `group.instance.id`를 할당하여 이 문제를 해결한다:

```properties
# 컨슈머 설정
group.instance.id=consumer-host-1
```

정적 그룹 멤버십이 활성화되면:

- **컨슈머가 재시작될 때 리밸런싱이 즉시 트리거되지 않는다.** 그룹 코디네이터는 `session.timeout.ms` 동안 해당 멤버가 다시 돌아오기를 기다린다.
- **같은 `group.instance.id`로 다시 참여하는 컨슈머는 이전과 동일한 파티션 할당을 받는다.** 이는 파티션 이동을 최소화한다.
- **롤링 배포 시 리밸런싱이 완전히 방지될 수 있다.** 각 컨슈머가 `session.timeout.ms` 내에 재시작된다면 그룹 멤버십이 유지되므로 리밸런싱이 발생하지 않는다.

### 권장 설정

정적 그룹 멤버십을 사용할 때는 `session.timeout.ms`를 충분히 크게 설정해야 한다. 컨슈머 재시작에 걸리는 시간보다 커야 하므로, 일반적으로 더 높은 값(예: 5분에서 10분)이 권장된다:

```properties
session.timeout.ms=300000  # 5분
group.instance.id=consumer-host-1
heartbeat.interval.ms=10000
```

## 브로커 수준 리밸런싱: 파티션 재할당

컨슈머 그룹 리밸런싱 외에도, **브로커 수준의 파티션 재할당**은 또 다른 형태의 리밸런싱이다. 이는 다음과 같은 상황에서 발생한다:

- **새로운 브로커 추가:** 기존 파티션이 새 브로커로 이동해야 한다.
- **브로커 해제:** 브로커를 제거하기 전에 해당 브로커의 파티션을 다른 브로커로 이동해야 한다.
- **부하 불균형 해소:** 특정 브로커에 파티션이 집중된 경우 재분배가 필요하다.

### 파티션 재할당의 비용

브로커 간 파티션 재할당은 대량의 데이터 복제를 수반한다. 이는 다음과 같은 비용을 발생시킨다:

- **네트워크 대역폭 소비:** 파티션 데이터를 새 브로커로 복제하면 브로커 간 네트워크 트래픽이 급증한다.
- **디스크 I/O 증가:** 소스 브로커에서의 읽기와 대상 브로커에서의 쓰기가 동시에 발생한다.
- **프로덕션 트래픽에 대한 영향:** 재할당 트래픽이 클라이언트 요청과 네트워크 및 디스크 리소스를 경쟁한다.
- **복제 지연 시간 증가:** 기존 파티션의 팔로워 복제가 느려질 수 있다.

### 스로틀링으로 영향 제어

Kafka는 파티션 재할당의 속도를 제한하기 위한 **스로틀링** 메커니즘을 제공한다:

```bash
# kafka-reassign-partitions.sh를 사용한 스로틀링 예시
kafka-reassign-partitions.sh --bootstrap-server localhost:9092 \
  --reassignment-json-file reassignment.json \
  --execute \
  --throttle 50000000  # 50MB/s로 제한
```

스로틀 값은 바이트/초 단위이며, 너무 낮게 설정하면 재할당이 매우 오래 걸리고, 너무 높게 설정하면 프로덕션 트래픽에 영향을 미친다. 적절한 밸런스를 찾는 것이 중요하다.

## 셀프 밸런싱 클러스터(Self-Balancing Clusters)

Confluent Platform과 Confluent Cloud는 **셀프 밸런싱 클러스터** 기능을 제공한다. 이 기능은 수동 파티션 재할당의 운영 부담을 크게 줄여준다.

### 셀프 밸런싱의 동작 방식

셀프 밸런싱 클러스터는 다음과 같이 동작한다:

- **자동 감지:** 클러스터의 부하 불균형, 새 브로커 추가, 브로커 제거 등의 이벤트를 자동으로 감지한다.
- **지능적 계획:** 최소한의 데이터 이동으로 최적의 균형을 달성하는 재할당 계획을 자동으로 생성한다.
- **점진적 실행:** 프로덕션 워크로드에 미치는 영향을 최소화하면서 백그라운드에서 점진적으로 파티션을 이동한다.
- **지속적 모니터링:** 클러스터 상태를 지속적으로 모니터링하고 필요에 따라 조정한다.

### 셀프 밸런싱의 이점

- **운영 복잡성 감소:** 수동으로 재할당 계획을 작성하고 실행할 필요가 없다.
- **더 빠른 확장:** 브로커를 추가하거나 제거할 때 자동으로 부하가 재분배된다.
- **프로덕션 영향 최소화:** 지능적인 스로틀링과 스케줄링으로 클라이언트 성능에 미치는 영향을 최소화한다.

## 리밸런싱 비용을 최소화하기 위한 모범 사례

### 1. 협력적 리밸런싱 프로토콜 사용

가능하다면 항상 `CooperativeStickyAssignor`를 사용한다. 이는 리밸런싱 중 처리 중단을 최소화하는 가장 효과적인 방법이다.

### 2. 정적 그룹 멤버십 활성화

모든 컨슈머에 `group.instance.id`를 설정하여 불필요한 리밸런싱을 방지한다. 특히 롤링 배포가 빈번한 환경에서 큰 효과를 발휘한다.

### 3. 세션 타임아웃 적절히 설정

`session.timeout.ms`와 `max.poll.interval.ms`를 워크로드에 맞게 적절히 설정한다. 너무 짧으면 불필요한 리밸런싱이 발생하고, 너무 길면 실제 장애 감지가 늦어진다.

```properties
session.timeout.ms=45000
heartbeat.interval.ms=15000
max.poll.interval.ms=300000
```

### 4. 파티션 수 신중하게 결정

토픽의 파티션 수를 신중하게 결정한다. 파티션 수가 많을수록 리밸런싱에 더 많은 시간이 소요된다. 처리량 요구 사항과 리밸런싱 비용 사이의 균형을 고려해야 한다.

### 5. 모니터링 및 알림 설정

리밸런싱 관련 메트릭을 모니터링하고 적절한 알림을 설정한다:

- **컨슈머 랙:** 리밸런싱 후 컨슈머 랙이 증가하는지 모니터링한다.
- **리밸런싱 빈도:** 비정상적으로 빈번한 리밸런싱은 문제의 징후이다.
- **리밸런싱 시간:** 각 리밸런싱에 소요되는 시간을 추적한다.
- **JoinGroup/SyncGroup 지연:** 프로토콜 단계별 지연 시간을 모니터링한다.

### 6. 브로커 재할당 시 스로틀링 활용

브로커 수준의 파티션 재할당을 수행할 때는 항상 스로틀링을 설정하여 프로덕션 트래픽에 대한 영향을 제어한다.

## 결론

클러스터 리밸런싱은 Kafka 운영에서 피할 수 없는 부분이지만, 그 비용을 이해하고 관리하는 것이 중요하다. 전통적인 Eager Rebalancing의 "stop-the-world" 접근 방식은 상당한 처리 중단과 연쇄 장애를 유발할 수 있다.

다행히 Kafka 커뮤니티는 이러한 비용을 줄이기 위한 여러 개선 사항을 도입했다:

- **협력적 리밸런싱(Cooperative Rebalancing):** 점진적 파티션 재할당으로 처리 중단을 최소화한다.
- **정적 그룹 멤버십(Static Group Membership):** 불필요한 리밸런싱을 방지한다.
- **셀프 밸런싱 클러스터(Self-Balancing Clusters):** 브로커 수준의 파티션 재할당을 자동화한다.

이러한 기능들을 적절히 활용하고, 모니터링과 올바른 설정을 통해 리밸런싱의 숨겨진 비용을 최소화함으로써, Kafka 클러스터를 보다 안정적이고 효율적으로 운영할 수 있다. 리밸런싱이 불가피한 상황에서도 그 영향을 최소화하는 것이 핵심이며, 이를 위해 협력적 프로토콜 채택, 정적 멤버십 설정, 적절한 타임아웃 튜닝, 그리고 지속적인 모니터링이 필수적이다.
