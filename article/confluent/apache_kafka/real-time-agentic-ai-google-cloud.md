# Confluent: 에이전틱 시스템을 위한 실시간 백본

> **원문:** [Confluent: The Real-Time Backbone for Agentic Systems](https://www.confluent.io/blog/real-time-agentic-ai-google-cloud/)
>
> **저자:** Dustin Shammo (시니어 솔루션 엔지니어 - Google Cloud 파트너십, Confluent), Merlin Yamssi (생성형 AI 리드, Google)
>
> **게시일:** 2025년 10월 16일

---

에이전틱 시스템의 진화하는 환경에서, Confluent와 Google Cloud는 효율적이고 안정적이며 지능적인 데이터 흐름을 뒷받침하는 실시간 인프라를 제공하는 핵심 조력자로 함께 부상하고 있습니다. 이 강력한 시너지는 에이전트 간(A2A) 통신, 외부 리소스와의 상호작용, 그리고 복잡한 멀티 에이전트 환경의 전반적인 안정성과 관측 가능성에서의 주요 과제들을 해결합니다.

생성형 AI가 주목을 받고 있지만, 그 진정한 힘은 종종 숨겨진 과제인 데이터의 신선도에 의해 제한됩니다. 대규모 언어 모델(LLM)은 방대한 양의 정보로 훈련되지만, 이 데이터는 빠르게 구식이 될 수 있어 시스템이 실시간 맥락을 제공하지 못하게 됩니다. 클라우드 네이티브 Apache Kafka로 구동되는 Confluent의 데이터 스트리밍 플랫폼은 Google Cloud의 Vertex AI와 같은 AI 서비스에 실시간 데이터를 제공함으로써 이 격차를 해결합니다.

이 글에서는 Confluent와 Google Cloud가 어떻게 멀티 에이전트 시스템의 실시간 백본을 구동하고 있는지 안내하겠습니다.

## 격차 해소: 지능형 AI 에이전트를 위한 실시간 데이터

데이터 이동의 중추 신경계 역할을 함으로써, Confluent는 기업이 이러한 한계를 극복할 수 있도록 돕습니다. 다양한 소스에서 신선한 실시간 데이터를 Google Cloud의 Vertex AI와 같은 강력한 AI 서비스로 직접 스트리밍합니다. 하지만 이 데이터의 여정은 거기서 끝나지 않습니다. 정교한 멀티 에이전트 시스템에서 LLM은 하나의 구성 요소에 불과합니다. 여기서 두 가지 중요한 통신 프로토콜인 모델 컨텍스트 프로토콜(MCP)과 A2A가 등장합니다.

Confluent의 플랫폼은 4단계 프로세스를 통해 생성형 AI 애플리케이션을 위한 실시간의 신뢰할 수 있는 지식 기반을 제공합니다:

- **스트림(Stream):** Confluent는 모든 환경에서 데이터를 지속적으로 스트리밍하여, LLM, 벡터 데이터베이스 및 기타 애플리케이션에 가용한 최신 정보를 제공합니다.
- **연결(Connect):** 120개 이상의 사전 구축된 커넥터를 통해 Confluent는 다양한 소스의 데이터 통합 프로세스를 간소화하고, 데이터 사일로를 허물며 원활한 스트리밍 파이프라인을 생성합니다.
- **거버넌스(Govern):** Confluent의 스트림 거버넌스는 데이터 계보, 품질 및 추적성을 보장하여, 데이터의 출처와 사용에 대한 명확한 이해를 제공함으로써 AI 솔루션에 대한 신뢰와 투명성을 구축합니다.
- **처리(Process):** Apache Flink 프로토콜을 사용하여, Confluent는 데이터가 수집되는 즉시 처리하여 변환 및 AI 모델 추론이 최소 지연 시간으로 이루어질 수 있도록 합니다. 이는 데이터 처리와 AI 워크플로를 통합하여 모델이 실시간 이벤트에 반응할 수 있게 합니다.

![Confluent와 Google Cloud로 생성형 AI 데이터를 움직이게 하기](ConfluentAgenticBackbone-Image1.png)

이는 생성형 AI LLM의 기반을 열어, AI 에이전트가 다음 에이전트를 참여시키기 전에 작업을 완료하기 위한 가장 최신의 정보를 보유하도록 보장합니다.

그 이점은 명확합니다: 이러한 모델의 훈련 및 배포에서 향상된 정확성, 더 나은 성능, 그리고 비용 절감. 더 나아가, A2A 프로토콜은 이 동일한 실시간 데이터 백본을 활용하여 에이전트 간 대화를 가능하게 합니다. 에이전트가 MCP를 사용하여 고객 이슈를 식별하면, 표준화된 '이벤트'인 A2A 메시지를 Kafka 토픽에 게시할 수 있습니다. 주문 처리를 전문으로 하는 다른 에이전트가 해당 메시지를 구독하여 즉각적인 조치를 취할 수 있습니다.

Confluent를 활용함으로써 조직은 더 풍부한 고객 경험을 제공하고, 더 나은 추천으로 매출을 증대하며, 지원 작업을 더 효과적으로 자동화할 수 있습니다. 이 모든 것은 급변하는 세상에 보조를 맞추는 실시간 데이터에 의해 구동되며, A2A와 MCP의 강력한 조합에 의해 오케스트레이션됩니다.

## A2A와 MCP 정의

A2A와 MCP는 멀티 에이전트 환경에서 에이전트와 시스템이 어떻게 통신하는지를 정의하는 두 가지 별개이지만 상호 보완적인 프로토콜입니다.

**A2A(에이전트 간 프로토콜)**는 에이전트가 다른 에이전트와 대화하기 위한 통신 메커니즘을 구체적으로 정의합니다. 이는 협력 시스템 내의 서로 다른 에이전트 간 정보 교환, 작업 위임 및 워크플로 오케스트레이션을 가능하게 합니다.

**MCP(모델 컨텍스트 프로토콜)**는 에이전트가 외부 리소스 및 도구와 어떻게 통합하고 접근하는지를 정의합니다. 여기에는 데이터베이스와의 상호작용, API 호출, 레거시 애플리케이션 사용, 또는 스트리밍 소스로부터의 데이터 소비가 포함되어, 에이전트가 실시간 맥락을 수집하고 자체 내부 로직 외부에서 작업을 수행할 수 있도록 합니다.

![AI 에이전트는 A2A 프로토콜을 활용하여 서로 상호작용하고, MCP를 사용하여 외부 도구 및 리소스를 호출합니다](ConfluentAgenticBackbone-Image2.png)

Confluent와 같은 플랫폼이 없다면, A2A와 MCP는 밀결합된 점대점 연결에 의존해야 합니다. A2A의 경우, 에이전트는 상호작용하는 모든 다른 에이전트의 구체적인 위치를 알아야 하며, 이는 어떤 변경이든 광범위한 업데이트를 요구하고 메시지의 내구성이 부족한 취약한 시스템을 만듭니다. MCP의 경우, 에이전트는 각 외부 도구, 데이터베이스 또는 API에 대해 맞춤형 클라이언트를 구축해야 하며, 이는 프로토콜 불일치, 개발 복잡성 증가, 비효율적인 데이터 폴링으로 이어져 실시간 맥락과 확장성을 저해합니다.

이러한 직접 통신 모델은 관측 가능성, 메시지 내구성 및 프로토콜 표준화를 위한 중앙 집중식 시스템이 부족하여, 전체 멀티 에이전트 시스템을 관리, 확장 및 디버그하기 어렵게 만듭니다.

## MCP/A2A 프록시: AI 에이전트를 위한 통신 브리지

이 아키텍처의 핵심은 Confluent의 MCP/A2A 프록시입니다. 이 핵심적인 중개자는 AI 에이전트와 도구가 다양한 시스템에서 어떻게 통신하는지를 간소화합니다. MCP와 A2A에 의해 정의된 것과 같은 다양한 통신 프로토콜을 Confluent Cloud 내에서 관리되는 표준화되고 견고한 메시지로 지능적으로 변환합니다.

핵심 포인트:

1. **통합 레지스트리:** 프록시는 중앙 집중식 레지스트리를 제공하여, 에이전트가 사용 가능한 에이전트, 도구 및 외부 리소스를 쉽게 발견하고 연결할 수 있게 하며, 복잡한 점대점 구성을 제거합니다.

2. **프로토콜 변환:** 서로 다른 MCP와 A2A 상호작용을 Confluent Cloud 내의 일관된 메시지 스트림으로 원활하게 변환합니다. 이러한 추상화는 개발을 간소화하고 안정적인 데이터 교환을 보장합니다.

3. **서비스 노출:** 결정적으로, 프록시를 통해 기존 마이크로서비스, 레거시 애플리케이션, 심지어 실시간 Flink 작업까지 접근 가능한 MCP 도구 및 리소스 또는 A2A 에이전트로 쉽게 노출할 수 있습니다. 이는 기존 투자에서 막대한 가치를 발굴하여 AI 에이전트가 이를 활용할 수 있게 합니다.

Confluent는 에이전트와 외부 시스템 간의 분리된 통신을 촉진하고, 에이전틱 시스템의 원칙과 완벽하게 정렬된 이벤트 기반 아키텍처를 본질적으로 가능하게 합니다. A2A 상호작용과 MCP를 통한 에이전트-외부 리소스 작업 모두 Kafka 토픽을 활용하여 이벤트를 게시할 수 있습니다. 도구와 다른 에이전트는 이러한 토픽을 구독하여 통신이 밀결합되지 않도록 보장합니다.

이러한 비동기, 이벤트 기반 아키텍처는 에이전트 워크플로가 자체 속도로 운영될 수 있게 합니다. Confluent는 에이전트가 바쁘거나 처리 시간이 오래 걸리는 경우에도 메시지를 안정적으로 전달하여, 중요한 정보가 손실되지 않도록 보장합니다. 이는 개별 에이전트의 워크로드나 처리 능력이 다양할 수 있는 견고한 에이전틱 시스템에서 특히 중요합니다.

에이전트는 다른 에이전트의 메시지나 외부 이벤트에 동적으로 반응할 수 있어, 높은 응답성과 적응성을 갖춘 환경을 조성합니다. 이러한 이벤트 기반 접근 방식은 느슨한 결합과 확장성을 촉진하여, 정교한 멀티 에이전트 시스템의 설계, 배포 및 관리를 더 쉽게 만듭니다.

복잡한 에이전트는 종종 대량의 데이터를 처리합니다. Confluent는 실시간 시나리오에서도 에이전트 시스템이 요구하는 속도에 맞춰 이 데이터의 안정적인 전달과 처리를 보장합니다. 이 안정성은 사기 탐지, 실시간 고객 서비스, 제조 최적화 및 동적 지리 분석과 같이 즉각적인 통찰을 요구하는 애플리케이션에 가장 중요합니다. Confluent를 데이터 백본으로 사용하면, 에이전트는 항상 가장 최신의 정보에 접근하여 시의적절하고 정확한 결정을 내릴 수 있습니다.

견고한 에이전틱 시스템을 구축하는 데 있어 핵심적인 부분은 에이전트가 서로 간에, 그리고 외부 리소스와 효과적으로 통신할 수 있도록 하는 것입니다. MCP와 A2A는 이러한 상호작용의 청사진을 제공합니다. Confluent는 모든 데이터 흐름을 위한 중추 신경계를 제공함으로써, 이러한 프로토콜을 이론적 설계에서 확장 가능하고 탄력적이며 관측 가능한 실제 애플리케이션으로 전환하는 불가결한 실시간 백본 역할을 합니다.

## 사용 사례: 실시간 금융 사기 탐지 및 대응

실시간으로 사기를 잡는 것은 엄청난 도전입니다. 기존 시스템은 종종 어려움을 겪어 지연과 놓친 위협으로 이어집니다. 지능형 에이전트와 견고한 메시징을 사용하여 빠르고 정확하며 확장 가능한 시스템을 구축할 수 있다면 어떨까요?

실시간 탐지를 심층 조사 및 대응으로부터 분리하는 스트리밍 아키텍처를 살펴보겠습니다. 이 아키텍처는 Google의 Agent Development Kit(ADK), LangGraph 및 Confluent로 구동됩니다. 이 아키텍처는 Google Cloud의 강력한 AI 에이전트와 Confluent의 실시간 데이터 처리 도구를 활용하여 빠르고 정확하며 확장 가능한 시스템을 구축합니다.

### 에이전트 1: 트랜잭션 모니터

Google의 ADK로 구축된 이 고성능 에이전트는 첫 번째 방어선입니다. Confluent 클라이언트 애플리케이션(Python 또는 JavaScript)은 Kafka에서 대량의 실시간 트랜잭션 데이터를 소비하며, MCP/A2A 프록시가 안전하고 효율적인 데이터 교환을 처리합니다. Flink SQL 쿼리가 의심스러운 패턴을 빠르게 플래그합니다. 중요한 것은, 이 에이전트는 자체적으로 조치를 취하지 않고 단순히 식별하고 경고한다는 것입니다.

### 에이전트 2: 조사 에이전트

트랜잭션 모니터가 이상 징후를 플래그하면, A2A 작업 메시지(예: `{'request': 'Investigate suspicious transaction for user X'}`)를 `fraud-alerts` Kafka 토픽에 전송합니다. 감사 가능한 단계별 추론을 위해 LangGraph로 구축된 조사 에이전트가 이 메시지를 소비합니다. 그런 다음 도구를 사용하여 고객 관계 관리(CRM) 시스템에서 고객 이력에 접근하고, 알려진 사기 데이터베이스를 확인하여 사기를 확정합니다.

### 에이전트 3: 대응 에이전트

사기가 확정되면, 조사 에이전트는 또 다른 A2A 작업 메시지(예: `{'request': 'Block account for user X immediately'}`)를 `remediation-tasks` Kafka 토픽에 전송합니다. 대응 에이전트가 이 메시지를 소비하며, 그 유일한 임무는 핵심 뱅킹 시스템에서 사용자의 계정을 잠그는 것입니다.

### 워크플로 실행 흐름

1. **의심 탐지:** 트랜잭션 모니터(에이전트 1)는 Flink SQL 쿼리를 사용하여 실시간 트랜잭션 데이터를 지속적으로 스캔합니다. 의심스러운 패턴을 발견하면 직접 조치를 취하지 않습니다. 대신 A2A 작업 메시지(예: `{'task_id': '123', 'request': 'Investigate suspicious transaction for user X'}`)를 생성하여 `fraud-alerts` Kafka 토픽에 게시합니다. 이것이 탐지와 행동을 분리하는 중요한 첫 번째 단계입니다.

2. **조사 시작:** `fraud-alerts`를 구독하는 조사 에이전트(에이전트 2)가 이 A2A 메시지를 소비합니다. 감사 가능한 단계별 추론을 위해 LangGraph를 활용하여 심층 분석에 들어갑니다. CRM에서 고객 이력을 가져오고 알려진 사기 데이터베이스와 대조하여 해당 활동이 실제로 사기인지 확인합니다.

3. **조치 요청:** 사기가 확정되면, 조사 에이전트도 직접 조치를 취하지 않습니다. 새로운 A2A 작업 메시지(예: `{'task_id': '456', 'request': 'Block account for user X immediately'}`)를 `remediation-tasks` Kafka 토픽에 발행합니다.

4. **대응 실행:** 마지막으로, `remediation-tasks`를 수신하는 대응 에이전트(에이전트 3)가 이 메시지를 소비합니다. 핵심 뱅킹 시스템에서 사용자의 계정을 잠그는 단순하고 집중된 작업을 수행하여 프로세스를 완료합니다.

### 시스템의 장점

이 시스템의 아름다움은 Confluent를 통한 원활한 A2A 통신에 있습니다: 트랜잭션 모니터가 탐지하고 경고하며, 조사 에이전트가 소비하고 분석하고 확정하며, 대응 에이전트가 실행합니다.

이 분리된 에이전트 기반 아키텍처는 상당한 이점을 제공합니다:

- **확장성:** 각 에이전트가 독립적으로 확장됩니다
- **탄력성:** Kafka가 메시지 내구성을 보장합니다
- **감사 추적:** 모든 A2A 메시지가 명확한 기록입니다
- **유연성:** 에이전트를 독립적으로 업데이트할 수 있습니다
- **명확한 책임:** 개발과 유지보수를 간소화합니다

이 접근 방식은 실시간 사기에 대처하기 위한 강력하고 효율적이며 투명한 방법으로, 시스템을 더 견고하고 적응력 있게 만듭니다.
