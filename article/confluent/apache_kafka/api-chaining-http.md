# 점 연결하기: 다중 API 데이터 흐름을 Apache Kafka로 간소화하기

> **원문:** [Connecting the Dots: Simplifying Multi-API Data Flows into Apache Kafka](https://www.confluent.io/blog/api-chaining-http/)
> **저자:** Sparsh Gupta (시니어 소프트웨어 엔지니어) | **게시일:** 2025년 12월 1일 | **읽는 시간:** 약 8분

오늘날 데이터 중심의 서비스형 소프트웨어(SaaS) 환경에서, 완전한 고객 인사이트를 얻으려면 여러 API 엔드포인트에 분산된 데이터를 가져와 공유해야 하는 경우가 많습니다. 이것이 바로 많은 고객이 Confluent의 데이터 스트리밍 및 통합 기능을 사용하여 실시간 API 체이닝을 구현하고자 하는 이유입니다. API 체이닝은 API 간의 관계를 자동으로 추적할 수 있게 해주는 기술입니다.

예를 들어, 사용자 데이터는 하나의 API에서 가져오고, 활동 이력이나 상호작용 기록은 또 다른 API에서 조회할 수 있습니다. 이렇게 흩어져 있지만 관련된 데이터를 통합하는 것은 복잡할 수 있으며, 특히 아키텍처에 Apache Kafka를 추가하면 더욱 그렇지만, 실시간 분석, 개인화 또는 규정 준수를 위해 여전히 필수적입니다. 이러한 API들을 체이닝하면 사용자를 가져온 다음 거래 내역이나 방문 장소 등 관련 레코드를 즉시 조회할 수 있습니다.

Confluent의 HTTP Source V2 커넥터에 대한 최근 업데이트를 통해, 이 패턴을 커스텀 코드나 외부 오케스트레이터 없이 구성만으로 선언적으로 구현할 수 있게 되었습니다. 그 결과는? 최소한의 노력으로 풍부하고 연결된 데이터를 Apache Kafka로 실시간 수집할 수 있습니다.

이 블로그 게시물에서 다루는 내용:

- API 체이닝의 작동 방식과 유사한 패턴과의 비교
- Apache Kafka에서 API 체이닝을 구현해야 하는 이유와 시점
- 실시간 API 체이닝에서 부모-자식 데이터 관계가 작동하는 방식
- Confluent로 실시간 API 체이닝을 선언적으로 구현하는 방법

## API 체이닝이란 무엇이며, 어떻게 작동하는가?

API 체이닝은 하나의 API 호출 결과가 다른 API의 입력으로 사용되는 기술입니다. 일반적으로 첫 번째 응답의 데이터(예: ID)가 후속 요청의 경로 또는 쿼리 매개변수로 전달됩니다. 이를 통해 여러 엔드포인트에 걸쳐 관련 데이터를 순회하거나 조합하는 종속적인 호출 시퀀스가 생성됩니다. 엔드포인트 간의 관계를 구축하고 깊이 중첩되거나 관련된 데이터를 추출할 수 있습니다.

### API 체이닝을 사용해야 하는 시점(그리고 이유)

일반적인 경험 법칙으로, 주어진 데이터셋을 조회하는 데 필요한 원격 API 호출 수를 최소화하는 것을 목표로 해야 합니다. 예를 들어, 목록 API 엔드포인트는 때때로 각 항목에 대한 상세 정보를 포함하도록 매개변수화할 수 있어, 여러 번의 후속 요청을 피할 수 있습니다.

그러나 이러한 옵션을 사용할 수 없거나, 단일 엔드포인트에서 다루기 힘든 양의 데이터를 반환하는 경우, API 체이닝이 실용적인 대안이 될 수 있습니다. 이를 통해 여러 엔드포인트에 걸쳐 점진적으로 데이터를 가져오고 풍부하게 만들어, 실시간으로 완전한 데이터셋을 효과적으로 구성할 수 있습니다.

이 접근 방식은 유연성을 제공하고 통합 로직을 단순화할 수 있지만, 체이닝된 호출 간의 지연 시간 증가, 속도 제한 및 의존성 장애와 같은 잠재적 단점에 주의해야 합니다. API 체이닝에서 구성 가능한 신중한 배치 처리 및 페이지네이션 전략이 이러한 문제를 완화하고 예측 가능한 성능을 유지하는 데 도움이 될 수 있습니다.

### 부모-자식 데이터 관계에 API 체이닝 적용하기

API 체이닝을 보물 지도처럼 생각해 보세요. 각 API 응답이 다음 데이터 계층을 더 깊이 파고들 수 있는 값을 제공합니다. API 체이닝의 작동 방식을 이해하기 위해, 최신 SaaS 애플리케이션에서 흔히 볼 수 있는 패턴인 REST API를 통해 노출되는 부모-자식 데이터 관계를 살펴보고, 전통적인 방식과 Kafka를 사용한 구현 방식을 논의하겠습니다.

두 개의 API가 있다고 가정해 봅시다:

- api1 → `/users` - 시스템의 모든 사용자 목록을 반환
- api2 → `/users/${id}/places` - 특정 사용자가 방문한 장소를 반환

`GET /users`

부모 API 엔드포인트는 시스템에 등록된 모든 사용자를 조회합니다.

`GET /users/${id}/places`

그런 다음 자식 API 엔드포인트는 고유 ID로 식별되는 특정 사용자가 방문한 모든 장소를 반환합니다.

이것은 산업 전반에 걸쳐 일반적인 구조입니다: 기본 엔티티(예: 사용자, 고객, 환자)와 관련 하위 리소스(예: 방문한 장소, 주문한 내역, 예약된 약속). 이는 단순화되었지만 현실적인 패턴으로, RESTful 시스템에서 계층적 데이터가 모델링되는 방식의 본질을 포착합니다.

## API 요청을 Apache Kafka로 스트리밍하기

커넥터는 API에서 지속적으로 최신 데이터를 가져오고, 사용자 컨텍스트로 풍부하게 만들어, 통합 이벤트 스트림으로 Kafka에 전달합니다. 이 접근 방식은 데이터 생산자(API)와 데이터 소비자(분석, 모니터링 또는 저장 시스템)를 분리하여 확장성과 유연성을 가능하게 합니다. 새로운 사용자나 장소가 추가되면, 데이터 파이프라인이 자동으로 적응하여 다운스트림 시스템이 항상 사용자 활동에 대한 최신 뷰에 접근할 수 있도록 보장합니다.

여기서의 목표는 시스템의 모든 사용자가 방문한 모든 장소를 수집하는 통합을 구축하는 것입니다. 관련 단계는 다음과 같습니다:

- **부모 API 호출하기**

  사용자 목록 가져오기:

  ```json
  [
    { "id": "user101", "name": "Alice" },
    { "id": "user102", "name": "Bob" }
  ]
  ```

- **각 사용자에 대해 자식 API 호출하기**

  각 사용자의 ID를 사용하여 방문한 장소를 가져옵니다:

  `GET /users/user101/places`

  `GET /users/user102/places`

- **결과 집계하기**

  모든 사용자의 데이터를 Kafka로 스트리밍합니다. JSON 형식으로 사용자가 방문한 장소를 나타내는 각 레코드는 Kafka 토픽에 게시되어, 분석 엔진, 데이터 파이프라인 또는 실시간 대시보드와 같은 다운스트림 소비자가 사용할 수 있습니다. 자식 API 응답은 암묵적으로 전체 데이터를 포함하며, 이 경우 장소와 사용자 ID 데이터 포인트의 조합을 의미합니다.

부모 API와 자식 API에서 반환된 각 레코드는 Kafka 토픽에 게시됩니다. 이를 통해 다음이 가능합니다:

- 실시간 처리 및 분석
- 다운스트림 시스템과의 원활한 통합
- 중첩된 API 데이터 구조를 이벤트 스트리밍 플랫폼으로 효율적으로 수집

### Apache Flink로 체이닝된 API 데이터를 실시간으로 풍부하게 만들고 쿼리하기

위의 예시에서, API 체이닝을 통해 사용자 및 장소 데이터가 별도의 Kafka 토픽으로 수집되면, Apache Flink와 같은 도구를 사용하여 이러한 스트림을 실시간으로 조인하고 풍부하게 만들 수 있습니다. 예를 들어, 사용자 메타데이터를 방문 장소와 결합하여 사용자 활동에 대한 완전한 뷰를 생성할 수 있습니다.

### 다중 API 데이터 흐름을 위한 일반적인 Kafka 및 Flink 사용 사례

Kafka와 Flink를 활용한 API 데이터 흐름은 거의 모든 산업에 걸쳐 강력한 실시간 통합 패턴을 가능하게 합니다. 관련 API 호출을 자동으로 체이닝하고 풍부해진 결과를 Kafka로 스트리밍하는 능력은 더 깊은 인사이트와 더 반응적인 비즈니스 운영을 위한 기회를 만듭니다. 고객 360 뷰, 운영 대시보드 또는 규정 준수 보고 시스템을 구축하든, API 체이닝은 고립된 데이터 소스 간의 전통적인 장벽을 제거합니다.

다음 예시들은 조직이 실제 데이터 통합 과제를 해결하기 위해 API 체이닝을 어떻게 활용할 수 있는지 보여줍니다. 각 패턴은 API의 부모-자식 관계가 기업 전체의 분석, 머신 러닝 및 운영 인텔리전스를 지원하는 통합된 실시간 데이터 스트림으로 어떻게 변환될 수 있는지 보여줍니다.

**이커머스 주문 및 항목:** 재고, 이행 및 고객 인사이트를 위해 주문 데이터를 라인 항목으로 풍부하게 만듭니다.

**계층적 데이터 가져오기:** 조인을 지원하지 않는 시스템에서 중첩된 리소스(예: 조직 → 팀 → 구성원)를 순회합니다.

**물류, 배송 및 추적 이벤트:** 배송 메타데이터와 실시간 추적 업데이트를 모두 캡처하여 배송 진행 상황과 운영 성과를 모니터링합니다.

**금융 고객 활동 및 거래:** 보고, 청구 또는 고객 서비스를 위해 고객 금융 기록과 거래 내역을 함께 가져옵니다.

**사물인터넷(IoT) 및 디바이스 관리:** 디바이스나 센서를 관리할 때, 체이닝된 엔드포인트를 통해 추가 메타데이터, 위치 정보 또는 과거 통계를 가져와야 할 수 있습니다.

예시: `GET /devices` → `GET /devices/{id}/status` → `GET /locations/{location_id}`

## Confluent로 다중 API 데이터 흐름을 쉽게 구현하는 방법

위 시나리오와 같이 API가 계층적으로 구조화된 시스템에서, API 체이닝은 이론적으로는 매우 강력한 패턴이 될 수 있지만 실행에서는 매우 복잡할 수 있습니다. 대부분의 통합 시나리오에서 하나의 엔드포인트에서만 데이터를 가져오는 것으로는 충분하지 않습니다. 분석, 풍부화, 모니터링 또는 개인화를 위해 여러 관련 API(동일한 기본 URL을 가진)의 데이터를 결합해야 하는 경우가 많습니다.

이 로직을 직접 구축하는 것이 기술적으로 가능하지만, 그렇게 하면 루핑, 상태 기반 페이지네이션, 속도 제한, 재시도 및 오류 처리 등 여러 계층의 복잡성을 관리해야 합니다. API 체이닝은 이 모든 것을 추상화하여, 팀이 인프라 배관 작업 대신 인사이트 제공에 집중할 수 있게 합니다.

데이터 파이프라인을 풍부하게 만들거나 데이터 레이크에 공급하든, HTTP Source V2 커넥터는 구조화된 API 데이터를 Kafka로 가져오는 강력하고 마찰이 적은 방법을 제공합니다. HTTP 커넥터는 Kafka 클라이언트 애플리케이션과 Marketo, Mailchimp, ServiceNow와 같은 HTTP 기반 서비스 간의 통신을 용이하게 합니다. 이 커넥터가 Kafka와 클라우드 서비스 사이의 브릿지 역할을 하여, 애플리케이션과 연결해야 하는 모든 HTTP 기반 서비스 간의 통신을 가능하게 합니다.

API 체이닝을 안정적이고 확장 가능한 방식으로 구현하려면, 구성을 통한 계층적 API 관계를 포함하여 광범위한 HTTP 기반 수집 패턴을 지원하는 HTTP Source V2 커넥터를 도입해야 합니다.

이 커넥터는 커스텀 스크립팅이나 추가 오케스트레이션 없이 체이닝 프로세스를 단순화합니다. 대신, 간단한 구성 속성을 사용하여 하나의 API 호출이 다른 API 호출로 이어지는 방식을 선언할 수 있습니다.

### HTTP Source V2 커넥터로 API 관계 정의하기

원래의 부모-자식 API 예시로 돌아가겠습니다. api1(`/users`)과 api2(`/users/${id}/places`)를 HTTP Source V2 커넥터로 체이닝하려면, 커넥터 구성에서 `api.chaining.parent.child.relationship` 속성을 사용할 수 있습니다:

```json
{
  "kafka.api.key": "********",
  "kafka.api.secret": "************",
  "kafka.auth.mode": "KAFKA_API_KEY",
  "name": "API Chaining - Http v2 Source",
  "connector.class": "HttpSourceV2",
  "output.data.format": "AVRO",
  "tasks.max": "2",
  "apis.num": "2",
  "http.api.base.url": "https://example.com",
  "api.chaining.parent.child.relationship": "api1::api2",
  "api1.http.api.path": "/users",
  "api1.topics": "users-topic",
  "api2.http.api.path": "/users/${id}/places",
  "api2.topics": "places-topic"
}
```

### 작동 방식:

- 커넥터는 먼저 api1(`/users`)을 호출하여 모든 사용자를 조회합니다.
- 각 사용자 객체에서 `id` 필드를 추출합니다.
- 그런 다음 추출된 각 `id`에 대해 api2(`/users/${id}/places`)를 호출하며, `${id}` 플레이스홀더를 동적으로 대체합니다.
- api2의 모든 응답은 구성된 Kafka 토픽에 게시됩니다.

이 체이닝 동작은 전적으로 구성을 통해 이루어집니다. 반복, 대체 또는 순서 지정을 수동으로 처리할 필요가 없습니다. 또한 커넥터를 통해 부모 응답의 ID를 자식 API 호출에 자동으로 대체할 수 있어, 커스텀 스크립팅이나 수동 데이터 연결의 필요성이 사라집니다. 더 중요한 것은, 내부적으로 오프셋 관리를 처리하여 부모 및 자식 데이터셋이 레코드를 누락하지 않고 완전히 순회되도록 보장한다는 것입니다.

## 실제 사용 사례: 부모-자식 API 체이닝 실전

API 체이닝은 다양한 도메인에서 실시간으로 풍부해진 데이터 파이프라인을 가능하게 합니다. Confluent를 활용한 실시간 API 체이닝에 대해 배운 내용을 실제로 적용하여, 원시 API 응답을 Kafka의 의미 있고 구조화된 데이터 스트림으로 변환해 보겠습니다.

### ServiceNow 인시던트에 작업 노트 풍부하게 만들기

IT 서비스 관리에서, 티켓이 어떻게 해결되는지에 대한 가시성은 티켓 상태 자체만큼이나 중요합니다. ServiceNow에서 API를 체이닝함으로써, 각 인시던트에 연결된 작업 노트를 캡처하고 분석할 수 있습니다.

#### 운영 인사이트

- 인시던트 해결에 걸리는 시간을 분석합니다.
- 노트 수 또는 유형별로 높은 노력이 필요한 티켓을 식별합니다.
- 작업 노트의 패턴을 기반으로 반복되는 기술적 문제를 발견합니다.

#### 규정 준수 및 서비스 수준 계약(SLA) 추적

- 에이전트가 해결 절차 및 문서화 표준을 따르는지 확인합니다.
- 규제 또는 내부 규정 준수를 위한 인시던트 처리를 감사합니다.
- 지원 팀 전체의 책임성과 투명성을 개선합니다.

#### API 체인:

- api1 - 인시던트 티켓 목록을 반환하며, 각각 인시던트를 고유하게 식별하는 `sys_id`를 포함합니다.

  `GET /api/now/table/incident`

- api2 - 주어진 인시던트에 연관된 모든 작업 노트를 반환합니다.

  `GET /api/now/table/sys_journal_field?sysparm_query=element_id=${sys_id}^element=work_notes`

### 커넥터 구성:

```json
{
  "kafka.api.key": "********",
  "kafka.api.secret": "************",
  "kafka.auth.mode": "KAFKA_API_KEY",
  "name": "API Chaining - ServiceNow",
  "connector.class": "HttpSourceV2",
  "output.data.format": "AVRO",
  "tasks.max": "2",
  "apis.num": "2",
  "http.api.base.url": "https://instance-id.service-now.com",
  "auth.type": "BASIC",
  "connection.user": "admin",
  "connection.password": "**********",
  "api.chaining.parent.child.relationship": "api1::api2",
  "api1.http.api.path": "/api/now/table/incident",
  "api1.topics": "users-topic",
  "api2.http.api.path": "/api/now/table/sys_journal_field?sysparm_query=element_id=${sys_id}^element=work_notes",
  "api2.topics": "places-topic"
}
```

- api1이 모든 인시던트를 가져옵니다.
- 커넥터가 각 인시던트에서 `sys_id`를 추출합니다.
- api2가 각 `sys_id`에 대해 호출되어 관련 작업 노트를 가져옵니다.
- 이러한 노트는 가시성과 분석을 위해 Kafka로 스트리밍됩니다.

또한 Flink를 활용하여 특정 인시던트에 대한 모든 작업 노트를 찾고 인시던트 생명주기에 대한 완전한 그림을 제공할 수 있어, 지원 리더, 사이트 신뢰성 엔지니어 또는 감사자가 로그를 스캔하거나 데이터를 수동으로 연결하지 않고도 모든 티켓의 해결 단계를 즉시 추적할 수 있습니다.

인시던트와 작업 노트 데이터가 API 체이닝을 통해 Kafka로 흐르면, 인공지능과 머신 러닝을 적용하여 더 깊은 인사이트를 얻을 수 있습니다. 예를 들어, `ML_PREDICT()`와 같은 내장 함수나 다른 스트리밍 머신 러닝 프레임워크를 사용하여, 인시던트 해결 패턴을 실시간으로 분석하고, 반복되는 문제를 감지하며, SLA 위반을 예측하거나, 과거 해결을 기반으로 다음 단계를 추천할 수 있습니다. 이를 통해 IT 서비스 관리에 지능적 자동화를 도입하여, 사전 예방적 지원과 더 빠른 근본 원인 식별이 가능해집니다.

ServiceNow에서 인시던트와 저널 API를 체이닝함으로써, 팀은 지원 워크플로우에 대한 360도 뷰를 얻게 되며, 이는 SLA 개선, 인력 최적화 및 규정 준수 입증에 매우 중요합니다.

## 마무리: API 체이닝의 힘 활용하기

Kafka 및 Flink와 결합하면, API 체이닝은 함께 작동하도록 설계되지 않은 API로부터 완전하고, 맥락적이며, 연결된 데이터셋을 구성하는 강력한 방법이 될 수 있습니다.

또한 오프셋 기반이든 커서 기반이든 다양한 페이지네이션 메커니즘과 배치 처리를 지원하여 실제 API 동작의 처리를 단순화하고, 대규모 데이터셋에서 안정적이고 효율적인 데이터 조회를 보장합니다.

HubSpot에서 고객 프로필을 동기화하고 Stripe의 구매 행동을 추가하거나, 날씨 API 컨텍스트로 IoT 텔레메트리 스트림을 강화하거나, CI/CD 추적성을 위해 GitHub 커밋과 배포 로그를 결합하든, Confluent의 HTTP Source V2 커넥터를 통해 다음을 수행할 수 있습니다:

- 다단계 데이터 수집 자동화
- 중첩된 API 간의 관계 보존
- 더 깊은 분석을 위해 실시간으로 풍부해진 데이터를 Kafka로 스트리밍
- 복잡한 워크플로우를 간단한 선언적 파이프라인으로 전환
