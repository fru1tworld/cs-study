# RFC 5952 - A Recommendation for IPv6 Address Text Representation

> 2010년 8월 발행, IPv6 주소의 텍스트 표현을 위한 권고안
> RFC 4291이 허용한 여러 표현 방식 중 단 하나의 표준(정규) 형식을 규정

## 1. 개요

RFC 4291은 IPv6 주소의 텍스트 표현에 **유연성**을 부여했습니다. 그 결과 동일한 주소를 여러 가지 방식으로 표기할 수 있어, 운영·로그 검색·코드 비교 등에서 혼란이 발생했습니다.

RFC 5952는 이러한 모호함을 제거하기 위해 **단 하나의 정규(canonical) 표현 형식**을 권고합니다. 이는 RFC 4291의 표현 규칙을 대체하지 않고, 그 위에 추가적인 제약을 더하는 형태입니다.

### 같은 주소의 여러 표현 (문제 상황)

다음은 모두 동일한 IPv6 주소입니다:

```
2001:db8:0:0:1:0:0:1
2001:0db8:0:0:1:0:0:1
2001:db8::1:0:0:1
2001:db8::0:1:0:0:1
2001:0db8::1:0:0:1
2001:db8:0:0:1::1
2001:db8:0000:0:1::1
2001:DB8:0:0:1::1
```

> "출력(생성) 측의 다양성이 문제를 일으킵니다. 검색·매칭·검증을 단순화하려면 표현을 하나로 통일해야 합니다."

## 2. 용어 정의

| 용어 | 의미 |
|------|------|
| 16비트 필드 | 콜론으로 구분되는 4자리 16진수 그룹 (`x`) |
| Leading zero | 각 필드 앞쪽의 불필요한 0 (예: `0db8`의 `0`) |
| `::` | 연속된 16비트 0 필드들을 압축하는 이중 콜론 |

## 3. 권고 표현 규칙 (Section 4)

RFC 5952가 정의하는 정규 표현은 다음 규칙들을 **모두** 만족해야 합니다.

### 3.1 Leading Zero 제거 (4.1)

> "각 16비트 필드의 선행 0(leading zero)은 반드시 제거해야 합니다(MUST)."

- 단, 필드 전체가 0인 경우 최소 한 자리 `0`은 남김

| 잘못된 표현 | 올바른 표현 |
|-------------|-------------|
| `2001:0db8::0001` | `2001:db8::1` |
| `2001:0db8:0000:0000:0000:0000:0000:0001` | `2001:db8::1` |

### 3.2 0 압축 `::` 사용 (4.2)

#### (4.2.1) 가능한 한 최대로 압축

> "`::`는 반드시(MUST) 표현 가능한 최대 길이의 0 필드 연속을 압축하는 데 사용해야 합니다."

| 잘못된 표현 | 올바른 표현 |
|-------------|-------------|
| `2001:db8:0:0:0:0:2:1` | `2001:db8::2:1` |
| `2001:db8::0:1` | `2001:db8::1` |

#### (4.2.2) 단일 16비트 0 필드는 압축하지 않음

> "단 하나의 16비트 0 필드는 `::`로 축약하면 안 됩니다(MUST NOT). 대신 `0`으로 표기합니다."

- `::`로 16비트 하나를 줄여도 글자 수 이득이 없고 오히려 혼란을 유발하기 때문

| 잘못된 표현 | 올바른 표현 |
|-------------|-------------|
| `2001:db8::1:1:1:1:1` (가운데 0 하나를 ::로 표기한 경우) | `2001:db8:0:1:1:1:1:1` |

```
예) 2001:db8:0:1:1:1:1:1
    → 가운데 'db8' 뒤의 0은 16비트 한 필드뿐이므로 ::로 줄이지 않고 0으로 둔다
```

#### (4.2.3) 가장 긴 0 연속을 압축, 동률이면 앞쪽 우선

> "0 필드 연속이 여러 개일 때는 가장 긴 연속을 `::`로 압축합니다. 길이가 같으면 가장 앞쪽(왼쪽) 연속을 압축해야 합니다(MUST)."

| 주소 | 올바른 표현 |
|------|-------------|
| `2001:0:0:1:0:0:0:1` (앞 2개 vs 뒤 3개) | `2001:0:0:1::1` |
| `2001:db8:0:0:1:0:0:1` (앞 2개 = 뒤 2개, 동률) | `2001:db8::1:0:0:1` |

### 3.3 소문자 사용 (4.3)

> "16진수 a~f는 반드시 소문자로 표기해야 합니다(MUST)."

| 잘못된 표현 | 올바른 표현 |
|-------------|-------------|
| `2001:DB8::1` | `2001:db8::1` |
| `2001:DB8:0:0:0:0:0:1` | `2001:db8::1` |

## 4. 특수 주소의 텍스트 표현 (Section 5)

### IPv4-Mapped / IPv4-Translated 등 혼합 표기

IPv4 주소를 내장한 IPv6 주소(예: IPv4-Mapped)는 끝부분을 점 십진 표기로 둘 수 있습니다.

```
::ffff:192.0.2.1     (IPv4-Mapped, 권장)
::ffff:c000:201      (16진 표기, 비권장)
```

> 혼합 표기를 쓰는 주소(`::ffff:...` 등)는 가독성을 위해 끝부분을 점 십진(`d.d.d.d`)으로 표기하는 것이 권장됩니다.

## 5. 부가 표기 권고 (Section 6)

### 5.1 포트 번호와 함께 쓸 때

IPv6 주소와 포트를 함께 표기할 때는 주소를 대괄호 `[ ]`로 감쌉니다.

```
[2001:db8::1]:80
http://[2001:db8::1]:8080/
```

### 5.2 프리픽스(CIDR) 표기

```
2001:db8::/48
2001:db8:0:cd30::/60
```

- 노드 주소 + 프리픽스 길이 조합도 가능: `2001:db8:0:cd30:123:4567:89ab:cdef/60`

### 5.3 Zone ID (스코프 식별자)

Link-Local 주소처럼 인터페이스를 명시해야 할 때 `%`로 Zone ID를 붙입니다.

```
fe80::1%eth0
fe80::1%2
[fe80::1%eth0]:443   (포트와 함께)
```

## 6. 정규 표현 요약 체크리스트

| 규칙 | 요구 수준 | 내용 |
|------|-----------|------|
| Leading zero 제거 | MUST | `0db8` → `db8` |
| 최대 길이 0 압축 | MUST | 가장 긴 0 연속만 `::`로 |
| 단일 0 필드 비압축 | MUST NOT | 16비트 하나는 `0`으로 |
| 동률 시 앞쪽 우선 | MUST | 같은 길이면 왼쪽 연속 압축 |
| 소문자 사용 | MUST | `DB8` → `db8` |

### 변환 예시

```
입력:  2001:0DB8:0000:0000:0000:0000:0000:0001
                ↓ leading zero 제거
       2001:DB8:0:0:0:0:0:1
                ↓ 가장 긴 0 연속 :: 압축
       2001:DB8::1
                ↓ 소문자화
정규:  2001:db8::1
```

## 7. 요약

RFC 5952는 RFC 4291이 허용한 다양한 IPv6 텍스트 표현 중 **단 하나의 정규 형식**을 권고하여 운영·검색·비교의 일관성을 확보합니다.

- **Leading zero 제거**, **소문자 사용**으로 표기 통일
- **가장 긴 0 연속만 `::`로 압축**, 동률이면 앞쪽 우선
- **단일 16비트 0 필드는 `::`로 줄이지 않음**
- 포트는 `[addr]:port`, Zone ID는 `addr%zone`, 프리픽스는 `addr/len`

이 권고를 따르면 사람과 시스템이 IPv6 주소를 모호함 없이 일관되게 다룰 수 있습니다.
