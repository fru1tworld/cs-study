# OAuth 2.0 보안 최선의 현행 관행 (Best Current Practice for OAuth 2.0 Security)

## 초록 (Abstract)

이 문서는 OAuth 2.0에 대한 현행 최선의 보안 관행을 기술한다. OAuth 2.0이 발행된 이후 수집된 실질적인 경험을 반영하고, OAuth 2.0의 더 넓은 적용으로 인해 관련된 새로운 위협을 다루기 위해 RFC 6749, 6750, 6819에 제공된 위협 모델과 보안 조언을 갱신 및 확장한다. 또한, 보안성이 낮거나 심지어 안전하지 않다고 간주되는 일부 운영 모드를 폐기한다.

## 1. 서론 (Introduction)

[RFC6749] 및 [RFC6750]에서의 발행 이후, OAuth 2.0(이 문서에서는 간단히 "OAuth"라 지칭)은 시장에서 막대한 견인력을 얻었으며 API 보호 및 OpenID Connect [OpenID.Core]를 사용한 연합 로그인의 표준이 되었다. OAuth는 다양한 시나리오와 다양한 종류의 배포에 사용되지만, 다음과 같은 과제가 관찰될 수 있다:

*  OAuth 구현이 알려진 구현 취약점과 안티 패턴(즉, 안전하지 않다고 간주되는 잘 알려진 패턴)을 통해 공격받고 있다. 이러한 위협의 대부분은 OAuth 2.0 위협 모델 및 보안 고려사항 [RFC6819]에서 논의되지만, 지속적인 악용은 더 구체적인 권고 사항, 더 쉽게 구현할 수 있는 완화 조치, 더 깊은 방어가 필요함을 보여준다.

*  OAuth는 처음에 고려된 것보다 더 높은 보안 요구사항을 가진 환경에서 사용되고 있으며, 오픈 뱅킹, eHealth, eGovernment, 전자 서명 등이 이에 해당한다. 이러한 사용 사례는 더 엄격한 지침과 추가 보호를 요구한다.

*  OAuth는 원래 예상했던 것보다 훨씬 더 동적인 설정에서 사용되고 있으며, 보안과 관련하여 새로운 과제를 만들고 있다. 이러한 과제는 [RFC6749], [RFC6750], [RFC6819]의 원래 범위를 넘어선다.

   OAuth는 처음에 클라이언트, 인가 서버, 리소스 서버 간의 정적 관계를 가정했다. 서버의 URL은 배포 시점에 클라이언트에 알려져 있었고, 이러한 당사자들 간의 신뢰 관계에 대한 앵커를 구축했다. 클라이언트가 합법적인 서버와 통신하고 있는지에 대한 검증은 TLS 서버 인증([RFC6819]의 섹션 4.5.4 참조)에 기반했다. OAuth의 채택이 증가함에 따라, 이 단순한 모델은 해체되었고, 여러 시나리오에서 한쪽의 클라이언트와 다른 쪽의 특정 배포의 인가 및 리소스 서버 간의 관계의 동적 수립으로 대체되었다. 이러한 방식으로, 동일한 클라이언트가 다른 프로바이더의 서비스에 접근하는 데 사용될 수 있었고(이메일이나 OpenID Connect와 같은 표준 API의 경우), 또는 멀티테넌트 환경에서 특정 테넌트의 프런트엔드 역할을 할 수 있었다. OAuth 2.0 동적 클라이언트 등록 프로토콜 [RFC7591] 및 OAuth 2.0 인가 서버 메타데이터 [RFC8414]와 같은 OAuth 확장이 동적 시나리오에서의 OAuth 사용을 지원하기 위해 개발되었다.

*  기술이 변화했다. 예를 들어, 브라우저가 요청을 리다이렉트할 때 프래그먼트를 처리하는 방식이 변경되었으며, 그에 따라 암시적 그랜트의 기본 보안 모델도 변경되었다.

이 문서는 이러한 과제를 해결하기 위한 갱신된 보안 권고 사항을 제공한다. OAuth 2.0 [RFC6749] 및 OpenID Connect [OpenID.Core]와 같은 기존 명세에 정의된 것을 넘어서는 새로운 요구사항을 도입하고, 보안성이 낮거나 심지어 안전하지 않다고 간주되는 일부 운영 모드를 폐기한다. 그러나, 이 문서는 [RFC6749], [RFC6750], [RFC6819]에 제공된 보안 조언을 대체하지 않고, 그 문서들을 보완한다.

당연히, 모든 기존 생태계와 구현이 새로운 요구사항과 호환되는 것은 아니며, 이 문서에 기술된 최선의 관행을 따르면 상호운용성이 깨질 수 있다. 그럼에도 불구하고, 구현자는 가능한 한 빨리 구현과 생태계를 업그레이드하는 것이 권장된다(RECOMMENDED).

개발 중인 OAuth 2.1 [OAUTH-V2.1]은 이 문서의 보안 권고 사항을 통합할 것이다.

### 1.1. 구조 (Structure)

이 문서의 나머지 부분은 다음과 같이 구성된다: 섹션 2는 모든 OAuth 구현자를 위한 가장 중요한 최선의 관행을 요약한다. 섹션 3은 갱신된 OAuth 공격자 모델을 제시한다. 섹션 4는 실제 환경에서(작성 시점에) 발견될 수 있는 위협과 구현 문제에 대한 상세한 분석과 잠재적 대응 방안에 대한 논의이다.

### 1.2. 규약 및 용어 (Conventions and Terminology)

이 문서에서 키워드 "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", "OPTIONAL"은 BCP 14 [RFC2119] [RFC8174]에 기술된 대로, 여기에 표시된 바와 같이 모두 대문자로 표기된 경우에만 해석되어야 한다.

이 명세는 OAuth 2.0 [RFC6749]에 정의된 용어 "액세스 토큰(access token)", "인가 엔드포인트(authorization endpoint)", "인가 그랜트(authorization grant)", "인가 서버(authorization server)", "클라이언트(client)", "클라이언트 식별자(client identifier, client ID)", "보호된 리소스(protected resource)", "리프레시 토큰(refresh token)", "리소스 소유자(resource owner)", "리소스 서버(resource server)", "토큰 엔드포인트(token endpoint)"를 사용한다.

"오픈 리다이렉터(open redirector)"는 쿼리 매개변수에서 얻은 임의의 URI로 사용자의 브라우저를 전달하는 웹 서버의 엔드포인트이다.

## 2. 최선의 관행 (Best Practices)

이 섹션은 작성 시점에 최선의 관행으로 간주되는 보안 메커니즘과 조치의 핵심 집합을 기술한다. 이러한 보안 메커니즘 및 조치에 대한 세부사항(상세한 공격 설명 포함)과 덜 일반적으로 사용되는 옵션에 대한 요구사항은 섹션 4에서 제공된다.

### 2.1. 리다이렉트 기반 흐름 보호 (Protecting Redirect-Based Flows)

클라이언트 리다이렉션 URI를 사전 등록된 URI와 비교할 때, 인가 서버는 네이티브 앱의 localhost 리다이렉션 URI에서의 포트 번호를 제외하고 정확한 문자열 매칭을 활용해야(MUST) 한다(섹션 4.1.3 참조). 이 조치는 인가 코드 및 액세스 토큰의 유출 방지에 기여한다(섹션 4.1 참조). 또한 혼합 공격의 탐지에도 도움이 될 수 있다(섹션 4.4 참조).

클라이언트와 인가 서버는 섹션 4.11에 기술된 대로, 쿼리 매개변수에서 얻은 임의의 URI로 사용자의 브라우저를 전달하는 URL(오픈 리다이렉터)을 노출해서는 안 된다(MUST NOT). 오픈 리다이렉터는 인가 코드 및 액세스 토큰의 유출을 가능하게 할 수 있다.

클라이언트는 교차 사이트 요청 위조(CSRF)를 방지해야(MUST) 한다. 이 맥락에서, CSRF는 인가 서버가 아닌 악의적인 제3자에서 발생하는 리다이렉션 엔드포인트에 대한 요청을 의미한다(자세한 내용은 [RFC6819]의 섹션 4.4.1.8 참조). 인가 서버가 PKCE(Proof Key for Code Exchange) [RFC7636]를 지원하는 것을 확인한 클라이언트는 PKCE가 제공하는 CSRF 보호에 의존할 수 있다(MAY). OpenID Connect 흐름에서는 nonce 매개변수가 CSRF 보호를 제공한다. 그 외의 경우, 사용자 에이전트에 안전하게 바인딩된 state 매개변수에 포함된 일회용 CSRF 토큰이 CSRF 보호를 위해 사용되어야(MUST) 한다(섹션 4.7.1 참조).

OAuth 클라이언트가 둘 이상의 인가 서버와 상호작용할 수 있는 경우, 혼합 공격(섹션 4.4 참조)에 대한 방어가 필수(REQUIRED)이다. 이를 위해, 클라이언트는 다음을 해야(SHOULD) 한다:

*  [RFC9207]에 따라 iss 매개변수를 대응 방안으로 사용하거나,
*  인가 응답의 iss 값에 기반한 대안적 대응 방안을 사용한다([OpenID.Core]의 ID Token 내 iss 클레임이나 [OpenID.JARM] 응답 등), 해당 값을 [RFC9207]에 기술된 대로 처리한다.

이러한 옵션이 없는 경우, 클라이언트는 대신 섹션 4.4.2에 기술된 대로, 인가 엔드포인트와 토큰 엔드포인트를 식별하기 위해 개별 리다이렉션 URI를 사용할 수 있다(MAY).

사용자 자격증명을 잠재적으로 포함하는 요청을 리다이렉트하는 인가 서버는 이러한 사용자 자격증명이 실수로 전달되는 것을 방지해야(MUST) 한다(자세한 내용은 섹션 4.12 참조).

#### 2.1.1. 인가 코드 그랜트 (Authorization Code Grant)

클라이언트는 다음 옵션 중 하나를 사용하여 인가 코드 주입 공격(섹션 4.5 참조)과 인가 코드의 오용을 방지해야(MUST) 한다:

*  공개 클라이언트는 이를 위해 PKCE [RFC7636]를 사용해야(MUST) 한다. 섹션 4.5.3.1에 동기 부여가 기술되어 있다.
*  기밀 클라이언트의 경우, PKCE [RFC7636]의 사용이 권장된다(RECOMMENDED). 이는 섹션 4.5.3.1에 기술된 대로 인가 코드의 오용 및 주입에 대한 강력한 보호를 제공하기 때문이다. 또한, 부수적 효과로 섹션 4.7.1에 기술된 대로 강력한 공격자의 존재 하에서도 CSRF를 방지한다.
*  추가적인 예방 조치와 함께(섹션 4.5.3.2에 기술), 기밀 OpenID Connect [OpenID.Core] 클라이언트는 대신 nonce 매개변수와 ID Token의 해당 클레임을 사용할 수 있다(MAY).

어떤 경우든, PKCE 챌린지 또는 OpenID Connect nonce는 트랜잭션에 특화되어야 하며, 트랜잭션이 시작된 클라이언트와 사용자 에이전트에 안전하게 바인딩되어야(MUST) 한다. 인가 서버는 PKCE 챌린지 또는 OpenID Connect nonce에 상수 값이 사용되는 것을 탐지하고 방지하기 위해 합리적인 노력을 기울일 것이 권장된다.

참고: PKCE는 네이티브 앱을 보호하기 위한 메커니즘으로 설계되었지만, 이 조언은 웹 애플리케이션을 포함한 모든 종류의 OAuth 클라이언트에 적용된다.

PKCE를 사용할 때, 클라이언트는 인가 요청에서 PKCE 검증자를 노출하지 않는 PKCE 코드 챌린지 방법을 사용해야(SHOULD) 한다. 그렇지 않으면, 인가 요청을 읽을 수 있는 공격자(섹션 3의 공격자 (A4) 참조)가 PKCE가 제공하는 보안을 깨뜨릴 수 있다. 현재 S256만이 이러한 방법에 해당한다.

인가 서버는 PKCE [RFC7636]를 지원해야(MUST) 한다.

클라이언트가 인가 요청에서 유효한 PKCE code_challenge 매개변수를 전송하는 경우, 인가 서버는 토큰 엔드포인트에서 code_verifier의 올바른 사용을 강제해야(MUST) 한다.

인가 서버는 인가 요청에 code_challenge 매개변수가 있었던 경우에만 code_verifier 매개변수를 포함하는 토큰 요청을 수락하도록 하여 PKCE 다운그레이드 공격을 완화해야(MUST) 한다. 자세한 내용은 섹션 4.8.2를 참조하라.

인가 서버는 PKCE 지원을 탐지할 수 있는 방법을 제공해야(MUST) 한다. 인가 서버가 지원하는 PKCE 챌린지 방법을 포함하는 code_challenge_methods_supported 요소를 인가 서버 메타데이터 [RFC8414]에 발행하는 것이 권장된다(RECOMMENDED)(클라이언트가 이를 사용하여 PKCE 지원을 탐지할 수 있다). 인가 서버는 대신 배포별 방법으로 인가 서버의 PKCE 지원을 보장하거나 결정하는 방법을 제공할 수 있다(MAY).

#### 2.1.2. 암시적 그랜트 (Implicit Grant)

암시적 그랜트(응답 유형 token)와 인가 서버가 인가 응답에서 액세스 토큰을 발급하게 하는 기타 응답 유형은 섹션 4.1, 4.2, 4.3, 4.6에 기술된 대로 액세스 토큰 유출 및 액세스 토큰 재생에 취약하다.

더욱이, 인가 응답에서 액세스 토큰이 발급될 때 액세스 토큰을 특정 클라이언트에 바인딩하기 위한(섹션 2.2에서 권장하는 대로) 발신자 제약을 위한 표준화된 방법이 존재하지 않는다. 이는 공격자가 유출되거나 탈취된 액세스 토큰을 리소스 엔드포인트에서 사용할 수 있음을 의미한다.

이러한 문제를 방지하기 위해, 클라이언트는 액세스 토큰 주입이 인가 응답에서 방지되고 앞서 언급한 토큰 유출 벡터가 완화되지 않는 한, 암시적 그랜트(응답 유형 token)나 인가 응답에서 액세스 토큰을 발급하는 기타 응답 유형을 사용해서는 안 된다(SHOULD NOT).

클라이언트는 대신 섹션 2.1.1에 명시된 대로 응답 유형 code(즉, 인가 코드 그랜트 유형)나, code id_token 응답 유형과 같이 인가 서버가 토큰 응답에서 액세스 토큰을 발급하게 하는 기타 응답 유형을 사용해야(SHOULD) 한다. 이를 통해 인가 서버는 공격자의 재생 시도를 탐지할 수 있고, 액세스 토큰이 URL에 노출되지 않으므로 공격 표면이 일반적으로 감소한다. 또한, 인가 서버가 발급된 토큰에 발신자 제약을 적용할 수 있다(섹션 2.2 참조).

### 2.2. 토큰 재생 방지 (Token Replay Prevention)

#### 2.2.1. 액세스 토큰 (Access Tokens)

발신자 제약 액세스 토큰은 액세스 토큰의 적용 범위를 특정 발신자로 한정한다. 이 발신자는 수신자(예: 리소스 서버)에서 해당 토큰의 수락을 위한 전제 조건으로 특정 시크릿에 대한 지식을 증명할 의무가 있다.

인가 서버와 리소스 서버는 탈취되고 유출된 액세스 토큰의 오용을 방지하기 위해, OAuth 2.0용 상호 TLS [RFC8705]나 OAuth 2.0 소유 증명(DPoP) [RFC9449]과 같은(섹션 4.10.1 참조) 발신자 제약 메커니즘을 사용해야(SHOULD) 한다.

#### 2.2.2. 리프레시 토큰 (Refresh Tokens)

공개 클라이언트용 리프레시 토큰은 발신자 제약이 적용되거나 섹션 4.14에 기술된 대로 리프레시 토큰 로테이션을 사용해야(MUST) 한다. [RFC6749]는 이미 기밀 클라이언트용 리프레시 토큰은 발급된 클라이언트에 의해서만 사용될 수 있도록 의무화하고 있다.

### 2.3. 액세스 토큰 권한 제한 (Access Token Privilege Restriction)

액세스 토큰과 관련된 권한은 특정 애플리케이션이나 사용 사례에 필요한 최소한으로 제한되어야(SHOULD) 한다. 이는 클라이언트가 리소스 소유자가 인가한 권한을 초과하는 것을 방지한다. 또한, 사용자가 해당 보안 정책에 의해 인가된 권한을 초과하는 것을 방지한다. 권한 제한은 액세스 토큰 유출의 영향을 줄이는 데도 도움이 된다.

특히, 액세스 토큰은 특정 리소스 서버로, 또는 그것이 불가능한 경우 소수의 리소스 서버 집합으로 대상이 제한되어야(SHOULD) 한다. 이를 적용하기 위해, 인가 서버는 액세스 토큰을 특정 리소스 서버와 연관시키며, 모든 리소스 서버는 모든 요청에 대해 해당 요청과 함께 전송된 액세스 토큰이 해당 특정 리소스 서버를 위한 것인지 검증할 의무가 있다. 그렇지 않은 경우, 리소스 서버는 해당 요청에 대한 서비스를 거부해야(MUST) 한다. [RFC9068]에 정의된 aud 클레임은 액세스 토큰에 대상 제한을 적용하는 데 사용될 수 있다(MAY). 클라이언트와 인가 서버는 각각 [RFC6749]와 [RFC8707]에 명시된 scope 또는 resource 매개변수를 활용하여 접근하려는 리소스 서버를 결정할 수 있다(MAY).

추가적으로, 액세스 토큰은 리소스 서버나 리소스에 대한 특정 리소스 및 동작으로 제한되어야(SHOULD) 한다. 이를 적용하기 위해, 인가 서버는 액세스 토큰을 해당 리소스 및 동작과 연관시키며, 모든 리소스 서버는 모든 요청에 대해 해당 요청과 함께 전송된 액세스 토큰이 특정 리소스에 대한 특정 동작을 위한 것인지 검증할 의무가 있다. 그렇지 않은 경우, 리소스 서버는 해당 요청에 대한 서비스를 거부해야 한다. 클라이언트와 인가 서버는 [RFC6749]에 명시된 scope 매개변수와 [RFC9396]에 명시된 authorization_details를 활용하여 해당 리소스 및/또는 동작을 결정할 수 있다(MAY).

### 2.4. 리소스 소유자 비밀번호 자격증명 그랜트 (Resource Owner Password Credentials Grant)

리소스 소유자 비밀번호 자격증명 그랜트 [RFC6749]는 사용해서는 안 된다(MUST NOT). 이 그랜트 유형은 리소스 소유자의 자격증명을 클라이언트에 안전하지 않게 노출한다. 클라이언트가 선의의 것이라 하더라도, 이 그랜트의 사용은 공격 표면의 증가(즉, 인가 서버뿐만 아니라 더 많은 곳에서 자격증명이 유출될 수 있음)를 초래하며, 사용자를 인가 서버 이외의 장소에 자격증명을 입력하도록 훈련시킨다.

더 나아가, 리소스 소유자 비밀번호 자격증명 그랜트는 이중 인증 및 다단계 사용자 상호작용을 요구하는 인증 프로세스와 함께 작동하도록 설계되지 않았다. 암호화 자격증명(WebCrypto [W3C.WebCrypto], WebAuthn [W3C.WebAuthn] 참조)을 사용한 인증은 일반적으로 특정 웹 출처에 바인딩되므로 이 그랜트 유형으로는 구현이 불가능할 수 있다.

### 2.5. 클라이언트 인증 (Client Authentication)

인가 서버는 특정 배포에서 클라이언트를 위한 자격증명의 발급/등록 프로세스를 수립하고 해당 자격증명의 기밀성을 보장하는 것이 가능한 경우, 클라이언트 인증을 강제해야(SHOULD) 한다.

OAuth 2.0용 상호 TLS [RFC8705]나 [RFC7521] 및 [RFC7523]에 따른 서명된 JWT("Private Key JWT")와 같은 비대칭 암호학을 클라이언트 인증에 사용하는 것이 권장된다(RECOMMENDED). 후자는 [OpenID.Core]에서 클라이언트 인증 방법 private_key_jwt로 정의되어 있다. 비대칭 암호학을 클라이언트 인증에 사용하는 경우, 인가 서버는 민감한 대칭 키를 저장할 필요가 없으므로, 이러한 방법은 키 유출에 대해 더 강건하다.

### 2.6. 기타 권고 사항 (Other Recommendations)

OAuth 인가 서버 메타데이터 [RFC8414]의 사용은 OAuth 배포의 보안을 향상시키는 데 도움이 될 수 있다:

*  호환되는 소프트웨어 라이브러리에 의해 보안 기능 및 기타 새로운 OAuth 기능이 자동으로 활성화되도록 보장한다.
*  잘못된 구성의 가능성을 줄인다 -- 예를 들어, 잘못 구성된 엔드포인트 URL(공격자에게 속할 수 있는) 또는 잘못 구성된 보안 기능.
*  암호화 키의 로테이션을 용이하게 하고 암호학적 민첩성을 보장하는 데 도움이 될 수 있다.

따라서 인가 서버는 [RFC8414]에 따라 OAuth 인가 서버 메타데이터를 발행하고, 클라이언트는 이 인가 서버 메타데이터를 (사용 가능한 경우) 자신의 구성에 활용하는 것이 권장된다(RECOMMENDED).

섹션 4.15.1에 기술된 조건 하에서, 인가 서버는 클라이언트가 자신의 client_id나 진정한 리소스 소유자와 혼동을 일으킬 수 있는 기타 클레임에 영향을 미치는 것을 허용해서는 안 된다(SHOULD NOT).

클라이언트와 리소스 서버 간에 [BCP195]에 따른 종단 간 TLS를 사용하는 것이 권장된다(RECOMMENDED). TLS 트래픽이 중개자에서 종단되어야 하는 경우, 추가 보안 조언은 섹션 4.13을 참조하라.

인가 응답은 암호화되지 않은 네트워크 연결을 통해 전송되어서는 안 된다(MUST NOT). 이를 위해, 인가 서버는 [RFC8252]의 섹션 7.3에 기술된 대로 루프백 인터페이스 리다이렉션을 사용하는 네이티브 클라이언트를 제외하고, http 스킴을 사용하는 리다이렉션 URI를 허용해서는 안 된다(MUST NOT).

인가 응답이 HTTP 리다이렉트 대신 postMessage [WHATWG.postmessage_api]와 같은 브라우저 내 통신 기술로 전송되는 경우, 브라우저 내 메시지의 개시자와 수신자 모두 섹션 4.17에 기술된 대로 엄격하게 검증되어야(MUST) 한다.

브라우저 기반 클라이언트가 직접 접근하는 엔드포인트 -- 토큰 엔드포인트, 인가 서버 메타데이터 엔드포인트, jwks_uri 엔드포인트, 동적 클라이언트 등록 엔드포인트를 포함 -- 는 CORS(Cross-Origin Resource Sharing) [WHATWG.CORS]의 사용을 지원할 수 있다(MAY). 그러나, 인가 엔드포인트에서는 CORS가 지원되어서는 안 된다(MUST NOT). 클라이언트가 이 엔드포인트에 직접 접근하지 않고, 사용자 에이전트를 그곳으로 리다이렉트하기 때문이다.

## 3. 갱신된 OAuth 2.0 공격자 모델 (The Updated OAuth 2.0 Attacker Model)

[RFC6819]에서는 OAuth 배포가 보호되어야 하는 위협을 기술하는 위협 모델이 제시되어 있다. 그렇게 하면서, [RFC6819]는 공격자와 그들의 능력에 대해 특정 가정을 하며, 즉, 암묵적으로 공격자 모델을 수립한다. 다음에서, 이 공격자 모델은 명시적으로 만들어지고, (섹션 1에 기술된 대로) 여러 당사자를 포함하는 잠재적으로 동적인 관계를 설명하기 위해, 새로운 유형의 공격자를 포함하기 위해, 그리고 공격자 모델을 더 명확하게 정의하기 위해 갱신되고 확장된다.

이 문서의 목표는 인가 서버에서의 리소스 소유자(사용자 에이전트 사용)의 인가와 리소스 서버에서의 후속 액세스 토큰 사용이 최소한 다음 공격자들에 대해 실질적으로 가능한 한 보호되도록 하는 것이다.

(A1) 웹 공격자 - "정직한" 것들 외에 임의의 수의 네트워크 엔드포인트(브라우저 및 서버 포함)를 설정하고 운영할 수 있는 웹 공격자이다. 웹 공격자는 리소스 소유자가 방문하는 웹사이트를 설정하고, 자신의 사용자 에이전트를 운영하며, 프로토콜에 참여할 수 있다.

   특히, 웹 공격자는 인가 서버에 등록된 OAuth 클라이언트를 운영할 수 있으며, 리소스 소유자 및 다른 리소스 소유자가 ("정직한" 것들과 병행하여) 사용할 수 있는 자체 인가 및 리소스 서버를 운영할 수 있다.

   또한, 웹 공격자가 언제든지 사용자가 브라우저를 임의의 공격자가 선택한 URI로 탐색하도록 유도할 수 있다고 가정해야 한다. 실제로, 이는 예를 들어, 악성 광고를 광고 네트워크에 주입하거나 합법적으로 보이는 이메일을 보내는 등 여러 방법으로 달성될 수 있다.

   웹 공격자는 자신의 사용자 자격증명을 사용하여 새 메시지를 생성할 수 있으며, 이전에 학습한 모든 시크릿을 사용할 수 있다. 예를 들어, 웹 공격자가 잘못 구성된 리다이렉션 URI를 통해 사용자의 인가 코드를 학습하면, 웹 공격자는 해당 코드를 액세스 토큰으로 교환하려고 시도할 수 있다.

   그러나, 그들은 자신에게 대상으로 지정되지 않은 메시지를 읽거나 조작할 수는 없다(예: 공격자의 통제하에 있지 않은 인가 서버의 URL로 전송된 것).

(A2) 네트워크 공격자 - 추가적으로 프로토콜 참가자가 통신하는 네트워크에 대한 완전한 통제권을 가진 네트워크 공격자이다. 그들은 메시지를 도청, 조작, 위조할 수 있으나, 이러한 메시지가 암호학적 방법(예: TLS)에 의해 적절히 보호되는 경우는 제외된다. 네트워크 공격자는 또한 임의의 메시지를 차단할 수 있다.

웹 공격자의 예로는 인터넷 서비스 프로바이더의 고객이 있을 수 있고, 네트워크 공격자는 인터넷 서비스 프로바이더 자체, ARP 스푸핑을 사용하는 공용(Wi-Fi) 네트워크의 공격자, 또는 예를 들어 인터넷 교환점에 접근할 수 있는 국가 후원 공격자가 될 수 있다.

앞서 언급한 공격자 (A1)과 (A2)는 OAuth에 대한 형식적 분석 노력 [arXiv.1601.01229]에서 사용된 공격자 모델에 부합한다. 이것은 최소 공격자 모델이다. 구현자는 자신의 OAuth 구현 환경에서 가능한 모든 유형의 공격자를 고려해야(MUST) 한다. 예를 들어, [arXiv.1901.11520]에서는 토큰 엔드포인트에 대한 완전한 통제권을 가진 공격자를 포함하는 매우 강력한 공격자 모델이 사용된다. 이는 생태계에서 엔드포인트의 잘못된 구성의 가능한 효과를 모델링하며, 섹션 2.6에 기술된 대로 인가 서버 메타데이터를 사용하여 방지할 수 있다. 따라서 이러한 공격자는 여기에 나열되지 않는다.

그러나, OAuth에 대한 이전 공격은 다음 유형의 공격자가 특히 관련이 있음을 보여주었다:

(A3) 인가 응답의 내용을 읽을 수 있지만 수정할 수는 없는 공격자이다(즉, 인가 응답이 공격자에게 유출될 수 있다).

   이러한 공격의 예로는 오픈 리다이렉터 공격과 혼합 공격(섹션 4.4 참조)이 있으며, 이 경우 클라이언트가 공격자가 통제하는 인가 서버로 자격증명을 보내도록 속는다.

   또한, 이것은 다음을 이용하는 공격을 포함한다:

   *  리다이렉트 URI의 불충분한 검사(섹션 4.1 참조);
   *  여러 앱이 동일한 URI에 자신을 등록할 수 있는 모바일 운영 체제에 존재하는 문제; 그리고
   *  브라우저(히스토리), 프록시 서버, 운영 체제에 의해 저장/기록된 URL.

(A4) 인가 요청의 내용을 읽을 수 있지만 수정할 수는 없는 공격자이다(즉, 인가 요청이 위와 같은 방식으로 공격자에게 유출될 수 있다).

(A5) 인가 서버가 발급한 액세스 토큰을 획득할 수 있는 공격자이다. 예를 들어, 리소스 서버가 공격자에 의해 손상되거나, 잘못된 구성으로 인해 액세스 토큰이 공격자가 통제하는 리소스 서버로 전송되거나, 소셜 엔지니어링을 사용하여 리소스 소유자가 공격자가 통제하는 리소스 서버를 사용하도록 할 수 있다. 섹션 4.9.2도 참조하라.

(A3), (A4), (A5)는 일반적으로 (A1) 또는 (A2)와 함께 발생한다. 공격자들은 공통의 목표를 달성하기 위해 협력할 수 있다.

공격자 (A1) 또는 (A2)는 리소스 소유자이거나 리소스 소유자로 행동할 수 있음에 유의하라. 예를 들어, 이러한 공격자는 위에 기술된 공격에 의해 획득한 토큰이나 인가 코드를 클라이언트나 리소스 서버에서 재생하기 위해 자신의 브라우저를 사용할 수 있다.

이 문서는 공격자 (A1)에서 (A5)까지의 결과적 위협에 초점을 맞춘다.

## 4. 공격 및 완화 (Attacks and Mitigations)

이 섹션은 OAuth 구현에 대한 공격의 상세한 설명과 잠재적 대응 방안을 제공한다. [RFC6819]에서 이미 다루어진 공격 및 완화는 새로운 권고가 있는 경우를 제외하고 여기에 나열되지 않는다.

이 섹션은 또한 특정 경우와 프로토콜 옵션에 대해 (섹션 2에 정의된 것 이상의) 추가 요구사항을 정의한다.

### 4.1. 불충분한 리다이렉션 URI 검증 (Insufficient Redirection URI Validation)

일부 인가 서버는 클라이언트가 완전한 리다이렉션 URI 대신 리다이렉션 URI 패턴을 등록할 수 있도록 허용한다. 인가 서버는 런타임에 인가 엔드포인트에서의 리다이렉션 URI 매개변수 값을 등록된 패턴과 매칭한다. 이 접근 방식은 클라이언트가 트랜잭션 상태를 추가 리다이렉트 URI 매개변수에 인코딩하거나 여러 리다이렉션 URI에 대해 단일 패턴을 등록할 수 있게 한다.

이 접근 방식은 정확한 리다이렉션 URI 매칭보다 구현이 더 복잡하고 관리하기 더 오류가 발생하기 쉬운 것으로 판명되었다. 패턴 매칭 구현 또는 구체적인 구성의 결함을 악용하는 여러 성공적인 공격이 실제 환경에서 관찰되었다(예: [research.rub2] 참조). 리다이렉션 URI의 불충분한 검증은 (그랜트 및 클라이언트 유형에 따라) 클라이언트 식별 또는 인증을 효과적으로 깨뜨리며, 공격자가 인가 코드 또는 액세스 토큰을 획득할 수 있게 한다:

*  사용자 에이전트를 공격자의 통제하에 있는 URI로 직접 보내거나,
*  사용자 에이전트가 URL 프래그먼트를 처리하는 방식과 연계하여 클라이언트의 오픈 리다이렉터를 활용함으로써 OAuth 자격증명을 공격자에게 노출시킨다.

이러한 공격은 다음 하위 섹션에서 자세히 설명된다.

#### 4.1.1. 인가 코드 그랜트에 대한 리다이렉트 URI 검증 공격 (Redirect URI Validation Attacks on Authorization Code Grant)

그랜트 유형 code를 사용하는 클라이언트의 경우, 공격은 다음과 같이 작동할 수 있다:

리다이렉션 URL 패턴 https://*.somesite.example/*이 클라이언트 ID s6BhdRkqt3인 클라이언트에 등록되어 있다고 가정한다. 의도는 somesite.example의 모든 하위 도메인이 클라이언트의 유효한 리다이렉션 URI가 되도록 허용하는 것이다. 예를 들어, https://app1.somesite.example/redirect이다. 그러나, 인가 서버의 순진한 구현은 와일드카드 *를 "도메인 이름에 유효한 모든 문자"가 아닌 "모든 문자"로 해석할 수 있다. 따라서 인가 서버는 https://attacker.example/.somesite.example을 리다이렉션 URI로 허용할 수 있으나, attacker.example은 악의적인 당사자가 잠재적으로 통제하는 다른 도메인이다.

그러면 공격은 다음과 같이 수행될 수 있다:

시작하기 위해, 공격자는 사용자를 속여 자신의 브라우저에서 변조된 URL을 열도록 해야 하며, 이는 공격자의 통제하에 있는 페이지를 시작한다. 예를 들어, https://www.evil.example (섹션 3의 공격자 A1 참조).

이 URL은 합법적인 클라이언트의 클라이언트 ID를 사용하여 인가 엔드포인트에 다음 인가 요청을 개시한다(표시용 줄 바꿈):

```
GET /authorize?response_type=code&client_id=s6BhdRkqt3&state=9ad67f13
     &redirect_uri=https%3A%2F%2Fattacker.example%2F.somesite.example
     HTTP/1.1
Host: server.somesite.example
```

인가 서버는 리다이렉션 URI를 검증하고 클라이언트 s6BhdRkqt3에 등록된 리다이렉션 URL 패턴과 비교한다. 인가 요청이 처리되고 사용자에게 제시된다.

사용자가 리다이렉션 URI를 보지 못하거나 공격을 인식하지 못하면, 코드가 발급되어 즉시 공격자의 도메인으로 전송된다. 인가의 자동 승인이 활성화되어 있는 경우([RFC6749]에 따르면 공개 클라이언트에게는 권장되지 않음), 공격은 사용자 상호작용 없이도 수행될 수 있다.

공격자가 공개 클라이언트를 사칭하는 경우, 공격자는 해당 토큰 엔드포인트에서 코드를 토큰으로 교환할 수 있다.

이 공격은 기밀 클라이언트의 경우 그렇게 쉽게 작동하지 않는다. 코드 교환에는 합법적인 클라이언트의 시크릿을 사용한 인증이 필요하기 때문이다. 그러나, 공격자는 인가 코드 주입 공격을 수행하여 합법적인 기밀 클라이언트가 코드를 교환하도록 할 수 있다; 섹션 4.5를 참조하라.

리다이렉션 URI 검증 취약점은 인가 서버가 와일드카드를 적절히 처리하는 경우에도 존재할 수 있음에 유의하는 것이 중요하다. 예를 들어, 클라이언트가 리다이렉션 URL 패턴 https://*.somesite.example/*을 등록하고 인가 서버가 이를 "도메인 somesite.example에 있는 모든 호스트를 가리키는 리다이렉션 URI 허용"으로 해석한다고 가정한다. 공격자가 somesite.example에 호스트나 하위 도메인을 설정할 수 있다면, 공격자는 합법적인 클라이언트를 사칭할 수 있다. 예를 들어, 이는 하위 도메인 탈취 공격 [research.udel]에 의해 발생할 수 있으며, 오래된 CNAME 레코드(예: external-service.somesite.example)가 더 이상 존재하지 않는 외부 DNS 이름(예: customer-abc.service.example)을 가리키고 공격자에 의해 탈취될 수 있다(예: 외부 서비스에 customer-abc로 등록하여).

#### 4.1.2. 암시적 그랜트에 대한 리다이렉트 URI 검증 공격 (Redirect URI Validation Attacks on Implicit Grant)

위에서 기술한 공격은 암시적 그랜트에도 마찬가지로 작동한다. 공격자가 인가 응답을 공격자가 통제하는 URI로 보낼 수 있다면, 공격자는 액세스 토큰을 전달하는 프래그먼트에 직접 접근할 수 있다.

추가적으로, 암시적 그랜트(그리고 [OAuth.Responses]에 정의된 response_mode=fragment를 사용하는 경우 다른 그랜트도)는 또 다른 종류의 공격을 받을 수 있다. 이 공격은 Location 헤더에 프래그먼트가 포함되지 않은 경우 사용자 에이전트가 프래그먼트를 리다이렉트의 대상 URL에 다시 붙이는 사실을 활용한다([RFC9110]의 섹션 17.11 참조). 여기서 기술된 공격은 이 동작을 클라이언트의 오픈 리다이렉터(섹션 4.11.1 참조)와 결합하여 액세스 토큰을 획득한다. 이를 통해 매우 좁은 리다이렉션 URI 패턴도 우회할 수 있지만, 엄격한 URL 매칭은 우회할 수 없다.

클라이언트 s6BhdRkqt3에 대해 등록된 URL 패턴이 https://client.somesite.example/cb?*라고 가정한다. 즉, https://client.somesite.example/cb에 대한 리다이렉트에 모든 매개변수가 허용된다. 불행히도, 클라이언트는 오픈 리다이렉터를 노출한다. 이 엔드포인트는 대상 URL을 받는 redirect_to 매개변수를 지원하며, HTTP Location 헤더 리다이렉트 303을 사용하여 브라우저를 이 URL로 보낸다.

공격은 이제 다음과 같이 수행될 수 있다:

시작하기 위해, 위와 같이, 공격자는 사용자를 속여 자신의 브라우저에서 변조된 URL을 열도록 해야 한다. 예를 들어, https://www.evil.example.

이후, 웹사이트는 코드 흐름에 대한 공격과 매우 유사한 인가 요청을 개시한다. 위와 다른 점은, redirect_to=https://attacker.example을 리다이렉션 URI의 매개변수에 인코딩하여 오픈 리다이렉터를 활용하며, 응답 유형 token을 사용한다는 것이다(표시용 줄 바꿈):

```
GET /authorize?response_type=token&state=9ad67f13
    &client_id=s6BhdRkqt3
    &redirect_uri=https%3A%2F%2Fclient.somesite.example
     %2Fcb%26redirect_to%253Dhttps%253A%252F
     %252Fattacker.example%252F HTTP/1.1
Host: server.somesite.example
```

그런 다음, 리다이렉션 URI가 등록된 패턴과 일치하므로, 인가 서버는 요청을 허용하고 결과 액세스 토큰을 303 리다이렉트로 전송한다(가독성을 위해 일부 응답 매개변수 생략):

```
HTTP/1.1 303 See Other
Location: https://client.somesite.example/cb?
          redirect_to%3Dhttps%3A%2F%2Fattacker.example%2Fcb
          #access_token=2YotnFZFEjr1zCsicMWpAA&...
```

client.somesite.example에서, 요청은 오픈 리다이렉터에 도착한다. 엔드포인트는 redirect 매개변수를 읽고 URL https://attacker.example/로의 HTTP 303 Location 헤더 리다이렉트를 발행한다.

```
HTTP/1.1 303 See Other
Location: https://attacker.example/
```

client.somesite.example의 리다이렉터가 Location 헤더에 프래그먼트를 포함하지 않으므로, 사용자 에이전트는 원래 프래그먼트 #access_token=2YotnFZFEjr1zCsicMWpAA&...를 URL에 다시 붙이고 다음 URL로 탐색한다:

```
https://attacker.example/#access_token=2YotnFZFEjr1z...
```

attacker.example의 공격자 페이지는 프래그먼트에 접근하여 액세스 토큰을 획득할 수 있다.

#### 4.1.3. 대응 방안 (Countermeasures)

패턴 매칭을 올바르게 구현하고 관리하는 복잡성은 명백히 보안 문제를 야기한다. 따라서 이 문서는 정확한 리다이렉션 URI 매칭을 사용하여 필요한 로직과 구성을 단순화할 것을 권고한다. 이는 인가 서버가 두 URI가 같은지 확인해야(MUST) 함을 의미한다; 자세한 내용은 [RFC3986]의 섹션 6.2.1, Simple String Comparison을 참조하라. 유일한 예외는 localhost URI를 사용하는 네이티브 앱이다: 이 경우, 인가 서버는 [RFC8252]의 섹션 7.3에 기술된 대로 가변 포트 번호를 허용해야(MUST) 한다.

추가 권고:

*  리다이렉션 URI가 호스팅되는 웹 서버는 오픈 리다이렉터를 노출해서는 안 된다(MUST NOT)(섹션 4.11 참조).
*  브라우저는 Location 헤더의 URL에 이미 프래그먼트가 포함되지 않은 경우에만 URL 프래그먼트를 Location 리다이렉션 URL에 다시 붙인다. 따라서, 서버는 Location 헤더의 URL에 임의의 프래그먼트 식별자(예: #_)를 붙여 브라우저가 리다이렉션 URL에 프래그먼트를 다시 붙이는 것을 방지할 수 있다(MAY).
*  클라이언트는 인가 엔드포인트에서 액세스 토큰 발급을 유발하는 응답 유형 대신 인가 코드 응답 유형을 사용해야(SHOULD) 한다. 이것은 인가 서버와의 교환 프로세스를 통한 유출된 자격증명의 재사용에 대한 대응 방안과, 액세스 토큰의 발신자 제약을 통한 토큰 재생에 대한 대응 방안을 제공한다.

리다이렉션 URI를 포함하는 인가 요청의 출처와 무결성이 검증될 수 있는 경우, 예를 들어, [RFC9101]이나 클라이언트 인증과 함께 [RFC9126]을 사용하는 경우, 인가 서버는 추가 검사 없이 리다이렉션 URI를 신뢰할 수 있다(MAY).

### 4.2. Referer 헤더를 통한 자격증명 유출 (Credential Leakage via Referer Headers)

인가 요청 URI 또는 인가 응답 URI의 내용이 Referer HTTP 헤더([RFC9110]의 섹션 10.1.3 참조)를 통해, 각각 인가 서버의 또는 클라이언트의 웹사이트로부터 유출되어, 의도치 않게 공격자에게 공개될 수 있다. 가장 중요한 것은, 인가 코드나 state 값이 이런 방식으로 공개될 수 있다는 것이다. [RFC9110]의 섹션 10.1.3에 달리 명시되어 있음에도, Chromium 프로젝트 [bug.chromium]의 (현재 수정된) 문제에서 볼 수 있듯이, 브라우저 구현 문제로 인해 URI 프래그먼트에 포함된 액세스 토큰에도 동일한 일이 발생할 수 있다.

#### 4.2.1. OAuth 클라이언트로부터의 유출 (Leakage from the OAuth Client)

OAuth 클라이언트로부터의 유출은 클라이언트가 성공적인 인가 요청의 결과로 다음을 포함하는 페이지를 렌더링하는 것을 필요로 한다:

*  공격자의 통제하에 있는 다른 페이지에 대한 링크를 포함하고 사용자가 그러한 링크를 클릭하거나,
*  제3자 콘텐츠(iframe 내 광고, 이미지 등)를 포함한다. 예를 들어, 페이지에 사용자 생성 콘텐츠(블로그)가 포함된 경우.

브라우저가 공격자의 페이지로 탐색하거나 제3자 콘텐츠를 로드하자마자, 공격자는 인가 응답 URL을 수신하고 code 또는 state(그리고 잠재적으로 access_token)를 추출할 수 있다.

#### 4.2.2. 인가 서버로부터의 유출 (Leakage from the Authorization Server)

유사한 방식으로, 인가 서버의 인가 엔드포인트가 위와 같은 링크나 제3자 콘텐츠를 포함하는 경우, 공격자는 인가 요청으로부터 state를 학습할 수 있다.

#### 4.2.3. 결과 (Consequences)

Referer 헤더를 통해 유효한 code나 access token을 학습한 공격자는 섹션 4.1.1, 4.5, 4.6에 기술된 공격을 수행할 수 있다. 공격자가 state를 학습하면, state를 사용하여 달성된 CSRF 보호가 상실되어 [RFC6819]의 섹션 4.4.1.8에 기술된 CSRF 공격이 발생한다.

#### 4.2.4. 대응 방안 (Countermeasures)

OAuth 인가 응답의 결과로 렌더링되는 페이지와 인가 엔드포인트는 제3자 리소스나 외부 사이트에 대한 링크를 포함해서는 안 된다(SHOULD NOT).

다음 조치는 성공적인 공격의 가능성을 더욱 줄인다:

*  적절한 Referrer Policy [W3C.webappsec-referrer-policy]를 문서에 적용하여(referrer 메타 속성의 일부로 또는 Referrer-Policy 헤더를 설정하여) Referer 헤더를 억제한다. 예를 들어, 응답의 Referrer-Policy: no-referrer 헤더는 결과 문서에서 발생하는 모든 요청에서 Referer 헤더를 완전히 억제한다.

*  인가 엔드포인트에서 액세스 토큰 발급을 유발하는 응답 유형 대신 인가 코드를 사용한다.

*  인가 코드를 기밀 클라이언트 또는 PKCE 챌린지에 바인딩한다. 이 경우, 공격자는 코드 교환을 요청하기 위한 시크릿이 부족하다.

*  [RFC6749]의 섹션 4.1.2에 기술된 대로, 인가 코드는 토큰 엔드포인트에서의 첫 사용 후 인가 서버에 의해 무효화되어야(MUST) 한다. 예를 들어, 합법적인 클라이언트가 코드를 교환한 후 인가 서버가 코드를 무효화하면, 공격자는 나중에 이 코드를 교환하는 데 실패할 것이다.

   이것은 합법적인 클라이언트보다 먼저 공격자가 코드를 토큰으로 교환하는 경우에는 공격을 완화하지 않는다. 따라서, [RFC6749]는 코드를 두 번 교환하려는 시도가 있을 때, 인가 서버가 이전에 해당 코드에 기반하여 발급된 모든 토큰을 해지할 것을 추가로 권장한다(SHOULD).

*  state 값은 리다이렉션 엔드포인트에서의 첫 사용 후 클라이언트에 의해 무효화되어야(SHOULD) 한다. 이것이 구현되면, 공격자가 클라이언트의 웹사이트로부터 Referer 헤더를 통해 토큰을 수신하더라도, state는 이미 사용되어 클라이언트에 의해 무효화되었으므로 공격자가 다시 사용할 수 없다. (이것은 state가 인가 서버의 웹사이트에서 유출되는 경우에는 도움이 되지 않는다. 그 시점에서 state는 아직 클라이언트의 리다이렉션 엔드포인트에서 사용되지 않았기 때문이다.)

*  리다이렉트 대신 인가 응답에 form post 응답 모드를 사용한다([OAuth.Post] 참조).

### 4.3. 브라우저 히스토리를 통한 자격증명 유출 (Credential Leakage via Browser History)

인가 코드와 액세스 토큰은 브라우저의 방문한 URL 히스토리에 저장될 수 있으며, 다음에 기술된 공격을 가능하게 한다.

#### 4.3.1. 브라우저 히스토리 내의 인가 코드 (Authorization Code in Browser History)

브라우저가 프로바이더의 인가 엔드포인트로부터의 리다이렉트 결과로 client.example/redirection_endpoint?code=abcd로 탐색할 때, 인가 코드를 포함하는 URL이 브라우저의 히스토리에 저장될 수 있다. 장치에 접근할 수 있는 공격자는 코드를 획득하고 재생을 시도할 수 있다.

대응 방안:

*  [RFC6819]의 섹션 4.4.1.1 및 섹션 4.5에 기술된 대로의 인가 코드 재생 방지.
*  리다이렉트 대신 인가 응답에 form post 응답 모드를 사용([OAuth.Post] 참조).

#### 4.3.2. 브라우저 히스토리 내의 액세스 토큰 (Access Token in Browser History)

클라이언트 또는 이미 토큰을 가진 웹사이트가 의도적으로 provider.com/get_user_profile?access_token=abcdef와 같은 페이지로 탐색하면, 액세스 토큰이 브라우저 히스토리에 저장될 수 있다. [RFC6750]은 이 관행을 억제하고 헤더를 통해 토큰을 전송하도록 권고하지만, 실제로 웹사이트들은 종종 쿼리 매개변수로 액세스 토큰을 전달한다.

암시적 그랜트의 경우, client.example/redirection_endpoint#access_token=abcdef와 같은 URL도 프로바이더의 인가 엔드포인트로부터의 리다이렉트 결과로 브라우저 히스토리에 저장될 수 있다.

대응 방안:

*  클라이언트는 [RFC6750]의 섹션 2.3에 기술된 방식으로 URI 쿼리 매개변수로 액세스 토큰을 전달해서는 안 된다(MUST NOT). 인가 코드 그랜트 또는 form post 응답 모드 [OAuth.Post]와 같은 대안적 OAuth 응답 모드를 이 목적으로 사용할 수 있다.

### 4.4. 혼합 공격 (Mix-Up Attacks)

혼합 공격은 OAuth 클라이언트가 둘 이상의 인가 서버와 상호작용하고 적어도 하나의 인가 서버가 공격자의 통제하에 있는 시나리오에서 발생할 수 있다. 이는 예를 들어, 공격자가 동적 등록을 사용하여 자신의 인가 서버에 클라이언트를 등록하거나 인가 서버가 손상되는 경우에 해당할 수 있다.

공격의 목표는 손상되지 않은 인가 서버에 대한 인가 코드 또는 액세스 토큰을 획득하는 것이다. 이는 클라이언트를 속여 해당 자격증명을 손상되지 않은 인가/리소스 서버의 해당 엔드포인트에서 사용하는 대신 손상된 인가 서버(공격자)로 보내도록 함으로써 달성된다.

#### 4.4.1. 공격 설명 (Attack Description)

여기서의 설명은 [arXiv.1601.01229]를 따르며, 아래에 공격의 변형이 설명되어 있다.

전제 조건: 이 공격 변형이 작동하기 위해서는 다음이 가정된다:

*  암시적 또는 인가 코드 그랜트가 여러 인가 서버와 함께 사용되며, 그 중 하나는 "정직한"(H-AS) 것으로 간주되고 하나는 공격자에 의해 운영된다(A-AS).
*  클라이언트는 사용자가 선택한 인가 서버를 사용자의 브라우저에 바인딩된 세션에 저장하고 각 인가 서버에 대해 동일한 리다이렉션 URI를 사용한다.

다음에서는 클라이언트가 H-AS(URI: https://honest.as.example, 클라이언트 ID: 7ZGZldHQ)와 A-AS(URI: https://attacker.example, 클라이언트 ID: 666RVZJTA)에 등록되어 있다고 가정한다. 다음 예제에 표시된 URL은 공격과 관련된 매개변수만 포함하도록 표시를 위해 축약되었다.

인가 코드 그랜트에 대한 공격:

1. 사용자가 A-AS를 사용하여 그랜트를 시작하기로 선택한다(예: 클라이언트의 웹사이트에 있는 버튼을 클릭하여).

2. 클라이언트는 사용자의 세션에 사용자가 "A-AS"를 선택했다고 저장하고, URL https://attacker.example/authorize?response_type=code&client_id=666RVZJTA를 포함하는 Location 헤더로 사용자를 A-AS의 인가 엔드포인트로 리다이렉트한다.

3. 사용자의 브라우저가 공격자의 인가 엔드포인트로 탐색하면, 공격자는 즉시 브라우저를 H-AS의 인가 엔드포인트로 리다이렉트한다. 인가 요청에서, 공격자는 A-AS에서의 클라이언트 ID를 H-AS에서의 클라이언트 ID로 교체한다. 따라서, 브라우저는 https://honest.as.example/authorize?response_type=code&client_id=7ZGZldHQ를 가리키는 Location 헤더와 함께 리다이렉션(303 See Other)을 수신한다.

4. 사용자가 H-AS에서 클라이언트가 자신의 리소스에 접근하도록 인가한다. (경계심이 있는 사용자는 이 시점에서 A-AS 대신 H-AS를 사용하려 했음을 감지할 수 있다. 아래에 나열된 첫 번째 공격 변형에는 이 제한이 없다.) H-AS는 코드를 발급하고 (브라우저를 통해) 클라이언트로 전송한다.

5. 클라이언트는 여전히 코드가 A-AS에 의해 발급되었다고 가정하므로, A-AS의 토큰 엔드포인트에서 코드를 교환하려고 시도한다.

6. 따라서 공격자는 코드를 획득하고, (공개 클라이언트의 경우) 코드를 액세스 토큰으로 교환하거나 섹션 4.5에 기술된 인가 코드 주입 공격을 수행할 수 있다.

변형:

*  가로채기를 동반한 혼합: 이 변형은 공격자가 사용자의 브라우저에서 클라이언트로의 첫 번째 요청/응답 쌍(사용자가 특정 인가 서버를 선택하고 클라이언트에 의해 해당 인가 서버로 리다이렉트되는)을 가로채고 조작할 수 있는 경우에만 작동한다. 이는 섹션 3의 공격자 (A2)에 해당한다. 이 능력은 예를 들어, 사용자의 클라이언트 연결에 대한 공격자 중간자 공격의 결과일 수 있다. 공격에서, 사용자는 H-AS로 흐름을 시작한다. 공격자는 이 요청을 가로채고 사용자의 선택을 A-AS로 변경한다. 나머지 공격은 위의 단계 2부터 진행된다.
*  암시적 그랜트: 암시적 그랜트에서, 공격자는 단계 4에서 코드 대신 액세스 토큰을 수신한다. 공격자의 인가 서버는 클라이언트가 A-AS userinfo 엔드포인트([OpenID.Core]에 정의)에 대한 요청을 하거나 공격자의 리소스 서버에 대한 요청을 할 때 액세스 토큰을 수신한다(클라이언트는 A-AS로의 흐름을 완료했다고 믿으므로).
*  AS별 리다이렉트 URI: 클라이언트가 다른 인가 서버에 대해 다른 리다이렉션 URI를 사용하고, 클라이언트가 선택된 인가 서버를 사용자의 세션에 저장하지 않으며, 인가 서버가 리다이렉션 URI를 적절히 검사하지 않는 경우, 공격자는 "Cross Social-Network Request Forgery"라 불리는 공격을 실행할 수 있다. 이러한 공격은 실제로 관찰되었다. 자세한 내용은 [research.jcs_14]를 참조하라.
*  OpenID Connect: 일부 변형은 OpenID Connect를 공격하는 데 사용될 수 있다. 이러한 공격에서, 공격자는 OpenID Connect Discovery [OpenID.Discovery] 메커니즘의 기능을 오용하거나 액세스 토큰 또는 ID Token을 재생하여 혼합 공격을 수행한다. 공격은 [arXiv.1704.08539]의 부록 A와 [arXiv.1508.04324v2]의 섹션 6("Malicious Endpoints Attacks")에 자세히 기술되어 있다.

#### 4.4.2. 대응 방안 (Countermeasures)

OAuth 클라이언트가 하나의 인가 서버와만 상호작용할 수 있는 경우, 혼합 방어는 필요하지 않다. 그러나 OAuth 클라이언트가 둘 이상의 인가 서버와 상호작용하는 시나리오에서는, 클라이언트가 혼합 공격을 방지해야(MUST) 한다. 아래에서 두 가지 다른 방법이 논의된다.

두 방어 모두에 대해, 클라이언트는 각 인가 요청에 대해 인가 요청을 보낸 발행자를 저장하고 이 정보를 사용자 에이전트에 바인딩해야(MUST) 한다. 발행자는 관련 메타데이터를 통해 흐름에서 사용될 인가 엔드포인트와 토큰 엔드포인트의 조합에 대한 추상적 식별자 역할을 한다. 발행자 식별자를 사용할 수 없는 경우(예: OAuth 인가 서버 메타데이터 [RFC8414]도 OpenID Connect Discovery [OpenID.Discovery]도 사용되지 않는 경우), 이 튜플 또는 튜플 자체에 대한 다른 고유 식별자를 대신 사용할 수 있다. 표현의 간결함을 위해, 이러한 배포별 식별자는 다음에서 발행자(또는 발행자 식별자)에 포함된다.

단순히 인가 서버 URL을 저장하는 것만으로는 혼합 공격을 식별하기에 충분하지 않다는 점에 유의하는 것이 중요하다. 공격자가 손상되지 않은 인가 서버의 인가 엔드포인트 URL을 "자신의" 인가 서버 URL로 선언하되, 자신의 통제하에 있는 토큰 엔드포인트를 선언할 수 있다.

##### 4.4.2.1. 발행자 식별을 통한 혼합 방어 (Mix-Up Defense via Issuer Identification)

이 방어는 인가 서버가 인가 응답에서 발행자 식별자를 클라이언트에 전송할 것을 요구한다. 인가 응답을 수신할 때, 클라이언트는 수신된 발행자 식별자를 저장된 발행자 식별자와 비교해야(MUST) 한다. 불일치가 있는 경우, 클라이언트는 상호작용을 중단해야(MUST) 한다.

이 발행자 식별자를 클라이언트로 전달하는 여러 방법이 있다:

*  발행자 정보는 예를 들어 [RFC9207]에 정의된 별도의 응답 매개변수 iss를 통해 전달될 수 있다.
*  OpenID Connect가 사용되고 ID Token이 인가 응답에서 반환되는 경우, 클라이언트는 ID Token의 iss 클레임을 평가할 수 있다.

두 경우 모두, iss 값은 [RFC9207]에 따라 평가되어야(MUST) 한다.

이 방어는 발행자 정보를 전달하기 위해 새로운 OAuth 기능의 배포를 필요로 할 수 있지만, 혼합에 대한 강건하고 비교적 단순한 방어이다.

##### 4.4.2.2. 개별 리다이렉트 URI를 통한 혼합 방어 (Mix-Up Defense via Distinct Redirect URIs)

이 방어를 위해, 클라이언트는 상호작용하는 각 발행자에 대해 개별 리다이렉션 URI를 사용해야(MUST) 한다.

클라이언트는 인가 응답이 올바른 발행자로부터 수신되었는지를 발행자에 대한 개별 리다이렉션 URI와 인가 응답이 수신된 URI를 비교하여 확인해야(MUST) 한다. 불일치가 있는 경우, 클라이언트는 흐름을 중단해야(MUST) 한다.

이 방어는 기존 OAuth 기능 위에 구축되지만, 클라이언트가 많은 다른 발행자의 사용을 위해 한 번만 등록하는 시나리오(일부 오픈 뱅킹 체계 등)에서는 사용할 수 없으며, 클라이언트 등록과의 긴밀한 통합으로 인해 자동으로 배포하기가 더 어렵다.

더 나아가, 공격자는 "정직한" 인가 서버에 클라이언트가 공격자의 인가 서버에 할당한 리다이렉트 URI를 사용하여 새 클라이언트를 등록함으로써 이 방어가 제공하는 보호를 우회할 수 있다. 공격자는 그런 다음 위에서 기술된 대로 공격을 실행하며, 클라이언트 ID를 새로 생성한 클라이언트의 클라이언트 ID로 교체할 수 있다.

따라서 이 방어는 다른 옵션을 사용할 수 없는 경우에만 사용해야(SHOULD) 한다.

### 4.5. 인가 코드 주입 (Authorization Code Injection)

인가 응답에 포함된 인가 코드에 대한 접근 권한을 얻은 공격자(섹션 3의 공격자 (A3) 참조)는 인가 코드를 액세스 토큰으로 교환하거나 인가 코드를 달리 사용하려고 시도할 수 있다.

인가 코드가 공개 클라이언트를 위해 생성된 경우, 공격자는 인가 코드를 인가 서버의 토큰 엔드포인트로 전송하여 액세스 토큰을 획득할 수 있다. 이 공격은 [RFC6819]의 섹션 4.4.1.1에 기술되어 있다.

기밀 클라이언트의 경우, 또는 일부 특수한 상황에서, 공격자는 다음에 기술된 대로 인가 코드 주입 공격을 실행할 수 있다.

인가 코드 주입 공격에서, 공격자는 탈취한 인가 코드를 클라이언트와의 공격자 자신의 세션에 주입하려고 시도한다. 목표는 클라이언트에서 공격자의 세션을 피해자의 리소스 또는 ID와 연관시켜, 공격자에게 피해자의 리소스에 대한 적어도 제한된 접근 권한을 부여하는 것이다.

기밀 클라이언트의 클라이언트 인증을 우회하는 것 외에도, 이 공격의 다른 사용 사례로는 다음이 포함된다:

*  공격자가 이 특정 클라이언트의 특정 기능에 접근하려고 한다. 예를 들어, 공격자가 특정 앱이나 특정 웹사이트에서 피해자를 사칭하려고 한다.
*  인가 또는 리소스 서버가 공격자가 직접 접근할 수 없는 특정 네트워크로 제한되어 있다.

이러한 특수한 경우를 제외하면, 코드가 공개 클라이언트를 위해 생성된 경우 인가 코드 주입은 일반적으로 흥미롭지 않다. 위에서 기술한 대로 코드를 토큰 엔드포인트로 보내는 것이 더 단순하고 강력한 공격이기 때문이다.

#### 4.5.1. 공격 설명 (Attack Description)

인가 코드 주입 공격은 다음과 같이 작동한다:

1. 공격자가 인가 코드를 획득한다(섹션 3의 공격자 (A3) 참조). 나머지 공격에는 웹 공격자 (A1)의 능력만 필요하다.
2. 공격자의 장치에서, 공격자는 합법적인 클라이언트와의 정규 OAuth 인가 프로세스를 시작한다.
3. 합법적인 클라이언트에 대한 인가 서버의 응답에서, 공격자는 새로 생성된 인가 코드를 탈취한 인가 코드로 교체한다. 이 응답이 공격자의 장치를 통과하므로, 공격자는 이 목적을 위해 인가 응답을 가로채고 조작할 수 있는 모든 도구를 사용할 수 있다. 공격자는 네트워크를 통제할 필요가 없다.
4. 합법적인 클라이언트는 코드를 인가 서버의 토큰 엔드포인트로 전송하며, redirect_uri와 클라이언트의 client ID 및 client secret(또는 기타 클라이언트 인증 수단)을 함께 보낸다.
5. 인가 서버는 client secret, 코드가 특정 클라이언트에 발급되었는지 여부, 실제 리다이렉션 URI가 redirect_uri 매개변수와 일치하는지 여부를 확인한다([RFC6749] 참조).
6. 모든 검사가 성공하고 인가 서버는 클라이언트에 액세스 토큰 및 기타 토큰을 발급한다. 공격자는 이제 합법적인 클라이언트와의 세션을 피해자의 리소스 및/또는 ID와 연관시켰다.

#### 4.5.2. 논의 (Discussion)

분명히, 코드가 다른 클라이언트 ID(예: 공격자가 설정한 클라이언트)에 발급된 경우 단계 5의 검사는 실패할 것이다. 인가 코드가 이미 합법적인 사용자에 의해 교환되었고 일회용인 경우에도 검사가 실패할 것이다.

조작된 리다이렉션 URI를 통해 획득한 코드를 주입하려는 시도는 인가 서버가 인가 요청에 사용된 전체 리다이렉션 URI를 저장하고 redirect_uri 매개변수와 비교하는 경우에도 탐지되어야 한다.

[RFC6749]의 섹션 4.1.3은 인가 서버가 다음을 보장하도록 요구한다:

> "redirect_uri" 매개변수가 섹션 4.1.1에 기술된 대로 초기 인가 요청에 포함된 경우 "redirect_uri" 매개변수가 존재하는지 확인하고, 포함된 경우 그 값이 동일한지 확인한다.

섹션 4.5.1에 기술된 공격 시나리오에서, 합법적인 클라이언트는 인가 요청에 항상 사용하는 올바른 리다이렉션 URI를 사용할 것이다. 그러나 이 URI는 공격자가 사용한 변조된 리다이렉션 URI와 일치하지 않을 것이다(그렇지 않으면, 리다이렉트가 공격자의 페이지에 도달하지 않았을 것이다). 따라서, 인가 서버는 공격을 탐지하고 코드 교환을 거부할 것이다.

이 검사는 특정 조건이 충족되는 경우 다른 장치의 동일한 클라이언트의 다른 인스턴스에서 획득한 인가 코드를 주입하려는 시도도 탐지할 수 있다:

*  리다이렉션 URI 자체에 nonce나 다른 종류의 일회성, 비밀 데이터가 포함되고
*  클라이언트가 이 데이터를 클라이언트의 이 특정 인스턴스에 바인딩한 경우.

그러나, 이 접근 방식은 인가 엔드포인트에서 정확한 리다이렉트 URI 매칭을 강제하는 아이디어와 충돌한다. 더욱이, 프로바이더가 이 단계에서 redirect_uri 검사 요구사항을 매우 자주 무시하는 것이 관찰되었다. 아마도 명세를 읽을 때 보안에 중요하지 않은 것처럼 보이기 때문일 것이다.

다른 프로바이더는 등록된 리다이렉션 URI 패턴에 대해 redirect_uri 매개변수를 패턴 매칭만 한다. 이는 인가 서버가 모든 트랜잭션에 대해 실제 리다이렉트 URI와 해당 인가 코드 간의 링크를 저장하는 것을 절약한다. 그러나, 이러한 종류의 검사는 변조된 리다이렉션 URI가 고려되지 않으므로 명세의 의도를 분명히 충족하지 못한다. 따라서, 합법적인 클라이언트의 client_id를 사용하거나 다른 장치에서 합법적인 클라이언트를 활용하여 획득한 인가 코드를 주입하려는 모든 시도는 해당 배포에서 탐지되지 않을 것이다.

또한, [RFC6749]의 섹션 4.1.3에 정의된 요구사항이 클라이언트가 토큰 엔드포인트 호출을 위한 올바른 리다이렉션 URI를 저장하거나 재구성해야 하므로 클라이언트 구현 복잡성을 증가시키는 것으로 가정된다.

클라이언트 인증을 위한 비대칭 방법은 합법적인 클라이언트가 토큰 엔드포인트에서 인증하므로 이 공격을 중단시키지 않는다.

따라서 이 문서는 다음에 기술된 메커니즘 중 하나를 사용하여 모든 인가 코드를 특정 트랜잭션의 맥락에서 특정 장치(또는 특정 사용자 에이전트)의 특정 클라이언트 인스턴스에 바인딩할 것을 대신 권고한다.

#### 4.5.3. 대응 방안 (Countermeasures)

인가 코드를 클라이언트 인스턴스에 바인딩하기 위한 두 가지 좋은 기술적 해결책이 있으며, 다음과 같다.

##### 4.5.3.1. PKCE

[RFC7636]에 명시된 PKCE 메커니즘은 (원래 네이티브 앱을 보호하기 위해 설계되었지만) 대응 방안으로 사용될 수 있다. 공격자가 인가 코드를 주입하려고 할 때, code_verifier의 검사가 실패한다: 클라이언트는 자신의 올바른 검증자를 사용하지만, 코드는 이 검증자와 일치하지 않는 code_challenge와 연관되어 있다.

PKCE는 인가 코드 주입 공격에 대해서만 보호하는 것이 아니라 공개 클라이언트를 위해 생성된 인가 코드도 보호한다: PKCE는 공격자가 code_verifier에 대한 지식 없이 인가 서버의 토큰 엔드포인트에서 탈취한 인가 코드를 교환할 수 없도록 보장한다.

##### 4.5.3.2. Nonce

OpenID Connect의 기존 nonce 매개변수는 인가 코드 주입 공격에 대해 보호할 수 있다. nonce 값은 일회용이며 클라이언트에 의해 생성된다. 클라이언트는 이를 사용자 에이전트 세션에 바인딩하고 OpenID Provider(OP)에 대한 초기 요청과 함께 전송하도록 되어 있다. OP는 수신된 nonce 값을 토큰 엔드포인트에서의 코드 교환의 일부로 발급되는 ID Token에 넣는다. 공격자가 인가 응답에 인가 코드를 주입하면, 클라이언트 세션의 nonce 값과 토큰 엔드포인트에서 수신된 ID Token의 nonce 값이 일치하지 않으며, 공격이 탐지된다. 가정은 공격자가 (공격자가 해당 인가 코드를 탈취한) 피해자의 장치에서 사용자 에이전트 상태를 획득할 수 없다는 것이다.

이 대응 방안은 클라이언트가 토큰 엔드포인트에서 획득한 ID Token의 nonce 매개변수를 적절히 검사하고, 이 검사가 성공할 때까지 발급된 토큰을 사용하지 않는 경우에만 작동한다는 점에 유의하는 것이 중요하다. 더 정확하게, nonce 매개변수를 사용하여 코드 주입으로부터 자신을 보호하는 클라이언트는:

1. 다른 ID Token이 인가 응답에서 획득되었더라도(예: response_type=code+id_token), 토큰 엔드포인트에서 획득한 ID Token의 nonce를 검증해야(MUST) 하며,
2. 해당 검사가 성공하지 않는 한 그리고 성공할 때까지, 모든 토큰(ID Token 및 액세스 토큰)이 무시되고 다른 목적으로 사용되지 않도록 보장해야(MUST) 한다.

nonce는 공개 클라이언트의 인가 코드를 보호하지 않는다는 점에 유의하는 것이 중요하다. 공격자가 인가 코드 주입 공격을 실행할 필요가 없기 때문이다. 대신, 공격자는 탈취한 인가 코드로 토큰 엔드포인트를 직접 호출할 수 있다.

##### 4.5.3.3. 기타 해결책 (Other Solutions)

state를 코드에 바인딩하기, 암호학적 수단을 사용하여 코드에 발신자 제약을 적용하기, 또는 인스턴스별 클라이언트 자격증명과 같은 다른 해결책이 구상될 수 있지만, 지원이 부족하고 새로운 보안 요구사항을 도입한다.

PKCE는 작성 시점에 사용 가능하므로 OAuth 클라이언트에 가장 명확한 해결책이며, nonce는 OpenID Connect 클라이언트에 적합하다.

#### 4.5.4. 한계 (Limitations)

공격자가 피해자의 인가 요청에서 사용되는 nonce 또는 code_challenge 값을 수정할 수 있는 경우 위에서 기술된 대응 방안을 우회할 수 있다. 공격자는 이 값을 위 공격의 단계 2에서 자신의 세션에서 클라이언트가 선택한 것과 동일한 값으로 수정할 수 있다. (이는 피해자의 클라이언트 세션이 공격자가 클라이언트와의 세션을 시작한 후에 시작되어야 함을 요구한다.) 그런 다음 공격자가 피해자로부터 인가 코드를 캡처할 수 있다면, PKCE나 nonce가 사용되더라도 공격자는 단계 3에서 탈취한 코드를 주입할 수 있을 것이다.

이 공격은 복잡하며 공격자와 피해자의 세션 간에 긴밀한 상호작용을 요구한다. 그럼에도 불구하고, 섹션 4.1, 4.2, 4.3, 4.4, 4.11에 기술된 대로 공격자가 인가 응답의 내용을 읽는 것을 방지하는 조치가 여전히 취해져야 한다.

### 4.6. 액세스 토큰 주입 (Access Token Injection)

액세스 토큰 주입 공격에서, 공격자는 탈취한 액세스 토큰을 (공격자의 통제하에 있지 않은) 합법적인 클라이언트에 주입하려고 시도한다. 이는 일반적으로 공격자가 특정 클라이언트에서 사용자를 사칭하기 위해 유출된 액세스 토큰을 활용하려는 경우에 발생한다.

공격을 수행하기 위해, 공격자는 암시적 그랜트를 사용하여 클라이언트와 OAuth 흐름을 시작하고, 인가 서버가 발급한 액세스 토큰을 교체하거나 유출된 액세스 토큰을 포함하는 인가 서버 응답을 직접 만들어 인가 응답을 수정한다. 응답에 이 특정 트랜잭션을 위해 클라이언트가 생성한 state 값이 포함되어 있으므로, 클라이언트는 응답을 CSRF 공격으로 취급하지 않고 공격자가 주입한 액세스 토큰을 사용한다.

#### 4.6.1. 대응 방안 (Countermeasures)

순수 OAuth 흐름에서는 이러한 주입 공격을 탐지할 방법이 없다. 토큰이 트랜잭션이나 특정 사용자 에이전트에 대한 바인딩 없이 발급되기 때문이다.

OpenID Connect에서는 인가 응답에 at_hash 클레임을 포함하는 ID Token이 추가로 포함되므로 공격을 완화할 수 있다. 따라서 공격자는 응답에서 액세스 토큰뿐만 아니라 ID Token도 교체해야 한다. 공격자는 ID Token이 인증과 함께 서명되거나 암호화되므로 위조할 수 없다. 공격자는 또한 탈취된 액세스 토큰과 일치하는 유출된 ID Token을 주입할 수 없다. 유출된 ID Token의 nonce 클레임이 인가 응답에서 기대되는 것과 (매우 높은 확률로) 다른 값을 포함할 것이기 때문이다.

공격자가 리소스 엔드포인트에서 액세스 토큰을 직접 사용하는 것을 방지하기 위해 발신자 제약 액세스 토큰과 같은 추가 보호가 여전히 필요하다는 점에 유의하라.

섹션 2.1.2의 권고는 이로부터 따른다.

### 4.7. 교차 사이트 요청 위조 (Cross-Site Request Forgery)

공격자가 피해자의 장치에서 합법적인 클라이언트의 리다이렉션 URI에 요청을 주입하려고 시도할 수 있다. 예를 들어, 클라이언트가 공격자의 통제하에 있는 리소스에 접근하도록 하기 위해서이다. 이것은 교차 사이트 요청 위조(CSRF)라고 알려진 공격의 변형이다.

#### 4.7.1. 대응 방안 (Countermeasures)

오랫동안 확립된 대응 방안은 클라이언트가 CSRF 토큰이라고도 알려진 랜덤 값을 state 매개변수에 전달하여 리다이렉션 URI에 대한 요청을 기술된 대로 사용자 에이전트 세션에 연결하는 것이다. 이 대응 방안은 [RFC6819]의 섹션 5.3.5에 자세히 기술되어 있다. PKCE 또는 OpenID Connect nonce 값이 동일한 보호를 제공한다.

CSRF 보호를 위해 state 또는 nonce 대신 PKCE를 사용할 때, 다음에 유의하는 것이 중요하다:

*  클라이언트는 CSRF 보호를 위해 PKCE를 사용하기 전에 인가 서버가 PKCE를 지원하는지 확인해야(MUST) 한다. 인가 서버가 PKCE를 지원하지 않는 경우, CSRF 보호를 위해 state 또는 nonce가 사용되어야(MUST) 한다.

*  state가 애플리케이션 상태를 전달하는 데 사용되고, 그 내용의 무결성이 우려되는 경우, 클라이언트는 state를 변조 및 교체로부터 보호해야(MUST) 한다. 이는 state의 내용을 브라우저 세션에 바인딩하거나 state 값을 서명/암호화하여 달성할 수 있다. 이에 대한 하나의 예가 만료된 인터넷 드래프트 [JWT-ENCODED-STATE]에서 논의된다.

따라서 인가 서버는 PKCE 지원을 탐지할 수 있는 방법을 제공해야(MUST) 한다. [RFC8414]에 따른 인가 서버 메타데이터 사용이 권장되지만(RECOMMENDED), 인가 서버는 대신 배포별 방법으로 PKCE 지원을 보장하거나 결정하는 방법을 제공할 수 있다(MAY).

PKCE는 인가 응답을 읽을 수 있는 공격자(섹션 3의 공격자 (A3) 참조)의 존재 하에서도 CSRF 공격에 대한 강건한 보호를 제공한다. state가 사용되거나 ID Token이 인가 응답에서 반환되는 경우(예: response_type=code+id_token), 공격자는 state 값을 학습하여 위조된 인가 응답에 재생하거나, ID Token에서 nonce를 추출하여 동일한 nonce를 가진 ID Token을 발행하기 위해 인가 서버에 대한 새 요청에서 사용할 수 있다. 새 ID Token은 그런 다음 CSRF 공격에 사용될 수 있다.

### 4.8. PKCE 다운그레이드 공격 (PKCE Downgrade Attack)

PKCE를 지원하지만 모든 흐름에 대해 사용을 의무화하지 않는 인가 서버는 PKCE 다운그레이드 공격에 취약할 수 있다.

이 공격의 첫 번째 전제 조건은 특정 흐름에 대해 PKCE를 활성화 또는 비활성화하는 공격자가 통제 가능한 플래그가 인가 요청에 있다는 것이다. code_challenge 매개변수의 존재 또는 부재가 이 목적에 적합하다. 즉, 인가 서버는 이 매개변수가 인가 요청에 있으면 PKCE를 활성화하고 강제하지만, 매개변수가 없으면 PKCE를 강제하지 않는다.

이 공격의 두 번째 전제 조건은 클라이언트가 state를 전혀 사용하지 않거나(예: 클라이언트가 CSRF 방지를 위해 PKCE에 의존하므로) 클라이언트가 state를 올바르게 검사하지 않는 것이다.

대략적으로 말하면, 이 공격은 CSRF 공격의 변형이다. 공격자는 섹션 4.7에 기술된 공격과 동일한 목표를 달성한다: 공격자가 자신의 리소스에 바인딩된 인가 코드(그리고 이를 통해 액세스 토큰)를 피해자와 클라이언트 간의 세션에 주입한다.

#### 4.8.1. 공격 설명 (Attack Description)

1. 사용자가 인가 서버에서 어떤 클라이언트를 사용하여 OAuth 세션을 시작했다. 인가 요청에서, 클라이언트는 PKCE 코드 챌린지로 code_challenge=hash(abc) 매개변수를 설정했다([RFC7636]에 정의된 해시 함수와 매개변수 인코딩 사용). 클라이언트는 이제 사용자의 브라우저로부터 인가 응답을 수신하기를 기다리고 있다.
2. 공격을 수행하기 위해, 공격자는 자신의 장치를 사용하여 대상 클라이언트와 인가 흐름을 시작한다. 클라이언트는 이제 인가 요청에서 다른 PKCE 코드 챌린지, 예를 들어 code_challenge=hash(xyz)를 사용한다. 공격자는 요청을 가로채고 요청에서 전체 code_challenge 매개변수를 제거한다. 이 단계는 공격자의 장치에서 수행되므로, 공격자는 예를 들어 브라우저 디버그 도구를 사용하여 요청 내용에 대한 완전한 접근 권한을 가진다.
3. 인가 서버가 PKCE 없이 흐름을 허용하면, PKCE 코드 챌린지에 바인딩되지 않은 코드를 생성한다.
4. 공격자는 이제 사용자의 브라우저를 인가 서버와의 공격자의 세션에 대한 코드를 포함하는 인가 응답 URL로 리다이렉트한다.
5. 사용자의 브라우저가 인가 코드를 클라이언트로 전송하고, 클라이언트는 이제 인가 서버에서 액세스 토큰에 대해 코드를 교환하려고 시도한다. 클라이언트는 토큰 요청에서 PKCE 코드 검증자로 code_verifier=abc를 전송한다.
6. 인가 서버가 이 코드가 PKCE 코드 챌린지에 바인딩되지 않았음을 보고, code_verifier 매개변수의 존재나 내용을 검사하지 않는다. 사용자의 통제하에 있는 클라이언트에 (공격자의 리소스에 속하는) 액세스 토큰을 발급한다.

#### 4.8.2. 대응 방안 (Countermeasures)

state를 적절히 사용하면 이 공격을 방지할 수 있다. 그러나, 실제로 많은 OAuth 클라이언트가 state를 사용하지 않거나 올바르게 검사하지 않는 것으로 나타났다.

따라서, 인가 서버는 이 공격을 완화해야(MUST) 한다.

인가 서버의 관점에서, 위에서 기술된 공격에서 토큰 엔드포인트에 code_verifier 매개변수가 수신되지만, 인가 코드가 발급된 OAuth 흐름에 대한 인가 요청에는 code_challenge 매개변수가 없었다는 점에 유의하라.

이 사실을 이 공격을 완화하는 데 사용할 수 있다. [RFC7636]는 이미 다음을 의무화하고 있다:

*  PKCE를 지원하는 인가 서버는 인가 요청에 코드 챌린지가 포함되어 있는지 확인하고 이 정보를 발급되는 코드에 바인딩해야(MUST) 한다; 그리고
*  코드가 토큰 엔드포인트에 도착하고, 이 코드가 발급된 인가 요청에 code_challenge가 있었던 경우, 토큰 요청에 유효한 code_verifier가 있어야 한다.

이를 넘어서, PKCE 다운그레이드 공격을 방지하기 위해, 인가 서버는 인가 요청에 code_challenge가 없었던 경우 code_verifier를 포함하는 토큰 엔드포인트에 대한 요청이 거부되도록 보장해야(MUST) 한다.

PKCE의 사용을 (일반적으로 또는 특정 클라이언트에 대해) 의무화하는 인가 서버는 암묵적으로 이 보안 조치를 구현한다.

### 4.9. 리소스 서버에서의 액세스 토큰 유출 (Access Token Leakage at the Resource Server)

액세스 토큰은 특정 상황에서 리소스 서버로부터 유출될 수 있다.

#### 4.9.1. 위조 리소스 서버에 의한 액세스 토큰 피싱 (Access Token Phishing by Counterfeit Resource Server)

공격자가 자체 리소스 서버를 설정하고 클라이언트를 속여 다른 리소스 서버에 유효한 액세스 토큰을 보내도록 할 수 있다(섹션 3의 공격자 (A1) 및 (A5) 참조). 클라이언트가 이 위조 리소스 서버에 유효한 액세스 토큰을 전송하면, 공격자는 차례로 해당 토큰을 사용하여 리소스 소유자를 대신하여 다른 서비스에 접근할 수 있다.

이 공격은 클라이언트가 개발 시점에 하나의 특정 리소스 서버(및 그 URL)에 바인딩되지 않고, 런타임에 클라이언트 인스턴스에 리소스 서버 URL이 제공되는 것을 가정한다. 이러한 종류의 지연 바인딩은 클라이언트가 표준화된 API를 구현하는 서비스(예: 이메일, 캘린더, eHealth, 오픈 뱅킹)를 사용하고 사용자 또는 관리자에 의해 구성되는 상황에서 전형적이다.

#### 4.9.2. 손상된 리소스 서버 (Compromised Resource Server)

공격자가 리소스 서버를 손상시켜 해당 배포의 리소스에 접근 권한을 얻을 수 있다. 이러한 손상은 시스템에 대한 부분적 접근(예: 로그 파일)에서 해당 서버에 대한 완전한 통제까지 다양할 수 있으며, 완전한 통제의 경우 모든 통제를 우회하고 모든 리소스에 접근할 수 있다. 공격자는 또한 손상된 시스템에 보유된 다른 액세스 토큰을 획득할 수 있으며, 이는 잠재적으로 다른 리소스 서버에 접근하는 데 유효할 수 있다.

서버 시스템의 강화 및 모니터링을 통해 서버 침해를 방지하는 것은 표준 운영 절차로 간주되며, 따라서 이 문서의 범위 밖이다. 섹션 4.9는 OAuth 관련 침해의 영향과 캡처된 액세스 토큰의 재생에 초점을 맞춘다.

#### 4.9.3. 대응 방안 (Countermeasures)

구현자는 악의적인 행위자에 의한 액세스 토큰 재생에 대처하기 위해 다음 조치를 고려해야 한다:

*  섹션 4.10.1에 기술된 발신자 제약 액세스 토큰은 공격자가 다른 리소스 서버에서 액세스 토큰을 재생하는 것을 방지하기 위해 사용되어야(SHOULD) 한다. 공격자가 웹 서버 로그에 대한 읽기 전용 접근과 같이 손상된 시스템에 대한 부분적 접근만 가진 경우, 발신자 제약 액세스 토큰은 손상된 시스템에서의 재생도 방지할 수 있다.
*  섹션 4.10.2에 기술된 대상 제한은 다른 리소스 서버에서의 캡처된 액세스 토큰 재생을 방지하기 위해 사용되어야(SHOULD) 한다.
*  리소스 서버는 액세스 토큰을 다른 민감한 시크릿처럼 취급해야 하며 평문으로 저장하거나 전송해서는 안 된다(MUST NOT).

첫 번째와 두 번째 권고는 액세스 토큰이 유출되는 다른 시나리오에도 적용된다(섹션 3의 공격자 (A5) 참조).

### 4.10. 탈취된 액세스 토큰의 오용 (Misuse of Stolen Access Tokens)

액세스 토큰은 섹션 4.1, 4.2, 4.3, 4.4, 4.9에 기술된 공격을 포함하여 다양한 방법으로 공격자에 의해 탈취될 수 있다. 이러한 공격 중 일부는 해당 섹션에 기술된 특정 보안 조치에 의해 완화될 수 있다. 그러나, 일부 경우에는 이러한 조치가 충분하지 않거나 올바르게 구현되지 않는다. 따라서 인가 서버는 다음에 기술된 대로 액세스 토큰에 발신자 제약과 대상 제한이 적용되도록 보장해야(SHOULD) 한다. 아키텍처 및 성능상의 이유로 일부 배포에서는 이러한 조치의 사용이 불가능할 수 있다.

#### 4.10.1. 발신자 제약 액세스 토큰 (Sender-Constrained Access Tokens)

이름에서 알 수 있듯이, 발신자 제약 액세스 토큰은 액세스 토큰의 적용 범위를 특정 발신자로 한정한다. 이 발신자는 리소스 서버에서 해당 토큰의 수락을 위한 전제 조건으로 특정 시크릿에 대한 지식을 증명할 의무가 있다.

전형적인 흐름은 다음과 같다:

1. 인가 서버는 이 특정 토큰을 특정 클라이언트에 바인딩하는 데이터를 액세스 토큰과 연관시킨다. 바인딩은 클라이언트의 ID를 활용할 수 있지만, 대부분의 경우 인가 서버는 클라이언트에 알려진 키 자료(또는 키 자료에서 파생된 데이터)를 활용한다.
2. 이 키 자료는 어떻게든 배포되어야 한다. 인가 서버가 바인딩을 생성하기 전에 키 자료가 이미 존재하거나 인가 서버가 임시 키를 생성한다. 기존 키 자료가 배포되는 방식은 다양한 접근 방식에 따라 다르다. 예를 들어, X.509 인증서를 사용할 수 있으며, 이 경우 배포는 등록 프로세스 중에 명시적으로 발생한다. 또는, 키 자료가 TLS 레이어에서 생성 및 배포되며, 이 경우 TLS 연결 설정 중에 자동으로 발생할 수 있다.
3. 리소스 서버는 실제 소유 증명 검사를 구현해야 한다. 이는 일반적으로 애플리케이션 레벨에서 수행되며, 종종 전송 레이어(예: TLS)에서 제공하는 특정 자료에 연결된다. 리소스 서버는 또한 소유 증명의 재생이 불가능하도록 보장해야 한다.

OAuth 워킹 그룹에 의해 소유 증명을 사용하는 발신자 제약 액세스 토큰에 대한 두 가지 방법이 정의되었으며 실제로 사용되고 있다:

*  "OAuth 2.0 상호 TLS 클라이언트 인증 및 인증서 바인딩 액세스 토큰" [RFC8705]: 이 문서에 명시된 접근 방식은 클라이언트 인증과 발신자 제약 액세스 토큰 모두를 위해 상호 TLS의 사용을 허용한다. 발신자 제약 액세스 토큰의 목적을 위해, 클라이언트는 공개 키의 지문으로 리소스 서버에 식별된다. 액세스 토큰 요청의 처리 중에, 인가 서버는 TLS 스택에서 클라이언트의 공개 키를 획득하고 그 지문을 해당 액세스 토큰과 연관시킨다. 리소스 서버도 같은 방식으로 TLS 스택에서 공개 키를 획득하고 그 지문을 액세스 토큰과 연관된 지문과 비교한다.
*  "OAuth 2.0 소유 증명(DPoP)" [RFC9449]: DPoP는 액세스 토큰과 리프레시 토큰에 발신자 제약을 적용하기 위한 애플리케이션 레벨 메커니즘을 개요한다. 공개/개인 키 쌍과 애플리케이션 레벨 서명에 기반한 소유 증명을 사용한다. DPoP는 공개 클라이언트와 함께 사용될 수 있으며, 기밀 클라이언트의 경우 모든 클라이언트 인증 방법과 결합할 수 있다.

발신자 제약 토큰의 보안은 공격자가 토큰과 키 자료에 대한 접근 권한을 얻으면 약화된다는 점에 유의하라. 이것은 특히 손상된 클라이언트 소프트웨어와 교차 사이트 스크립팅 공격(클라이언트가 브라우저에서 실행되는 경우)의 경우에 해당한다. 키 자료가 하드웨어 또는 소프트웨어 보안 모듈에서 보호되거나 (TLS 스택에서와 같이) 간접적으로만 접근 가능한 경우, 발신자 제약 토큰은 적어도 클라이언트가 오프라인일 때, 즉 보안 모듈 또는 인터페이스가 공격자에게 사용 가능하지 않을 때 토큰의 사용을 방지한다. 이는 액세스 토큰뿐만 아니라 리프레시 토큰에도 적용된다(섹션 4.14 참조).

#### 4.10.2. 대상 제한 액세스 토큰 (Audience-Restricted Access Tokens)

대상 제한은 본질적으로 액세스 토큰을 특정 리소스 서버로 제한한다. 인가 서버는 액세스 토큰을 특정 리소스 서버와 연관시키며, 리소스 서버는 의도된 대상을 검증하도록 되어 있다. 액세스 토큰이 의도된 대상 검증에 실패하면, 리소스 서버는 해당 요청에 대한 서비스를 거부한다.

일반적으로, 대상 제한은 토큰 유출의 영향을 제한한다. 위조 리소스 서버의 경우, 아래에 기술된 대로 피싱된 액세스 토큰의 합법적 리소스 서버에서의 남용도 방지할 수 있다.

대상은 논리적 이름이나 물리적 주소(URL과 같은)를 사용하여 표현될 수 있다. 피싱을 방지하기 위해서는 클라이언트가 요청을 보낼 실제 URL을 사용해야 한다. 피싱의 경우, 이 URL은 위조 리소스 서버를 가리킬 것이다. 공격자가 합법적 리소스 서버(다른 URL을 가진)에서 액세스 토큰을 사용하려고 하면, 리소스 서버는 불일치(잘못된 대상)를 탐지하고 요청에 대한 서비스를 거부할 것이다.

인가 서버가 모든 리소스 서버의 URL을 알고 있는 배포에서는, 인가 서버가 알 수 없는 리소스 서버 URL에 대해 액세스 토큰 발급을 거부할 수 있다.

이것이 작동하기 위해, 클라이언트는 인가 서버에 의도된 리소스 서버를 알려줘야 한다. [RFC8707]의 메커니즘이 이 목적으로 사용될 수 있거나 정보가 scope 값에 인코딩될 수 있다([RFC6749]의 섹션 3.3).

URL 대신, 리소스 서버의 X.509 인증서의 지문을 대상 값으로 활용하는 것도 가능하다. 이 변형은 다른 CA에서 획득한 유효한 TLS 인증서를 사용하여 합법적 리소스 서버의 URL을 스푸핑하려는 시도의 탐지도 허용할 것이다. 인가 서버로부터 리소스 서버 URL을 숨기는 것이 프라이버시 이점으로 간주될 수도 있다.

대상 제한은 클라이언트 측에서 암호학을 필요로 하지 않으므로 사용하기 더 쉬운 것처럼 보일 수 있다. 그럼에도 불구하고, 모든 액세스 토큰이 특정 리소스 서버에 바인딩되므로, 여러 리소스 서버에 접근할 때 클라이언트는 개별 리소스 서버별 액세스 토큰도 획득해야 한다. ([RFC8707]에 명시된 리소스 인디케이터가 이를 달성하는 데 도움이 될 수 있다.) 반면에 OAuth 2.0용 상호 TLS [RFC8705]를 사용하면, 클라이언트가 여러 리소스 서버에서 액세스 토큰을 사용할 수 있다.

대상 제한 -- 또는 일반적으로 말해, 클라이언트가 인가 서버에 액세스 토큰을 어디에서 사용하고 싶은지에 대한 표시 -- 은 토큰 유출 방지 범위를 넘어서는 추가 이점이 있다는 점에 유의해야 한다. 이를 통해 인가 서버는 해당 서버에 맞게 특별히 발행된 형식과 내용을 가진 다른 액세스 토큰을 생성할 수 있다. 이것은 구조화된 액세스 토큰을 사용하는 배포에서 엄청난 기능적 및 프라이버시 이점을 가진다.

#### 4.10.3. 논의: 메타데이터를 통한 유출 방지 (Discussion: Preventing Leakage via Metadata)

인가 서버는 클라이언트에 액세스 토큰을 안전하게 사용할 수 있는 위치에 대한 추가 정보를 제공할 수 있다. 이 접근 방식과 이것이 권장되지 않는 이유는 다음에서 논의된다.

가장 단순한 형태로, 이것은 인가 서버가 알려진 리소스 서버의 목록을 발행하는 것을 필요로 할 것이며, 다음 예제에서 비표준 인가 서버 메타데이터 매개변수 resource_servers를 사용하여 설명된다:

```
HTTP/1.1 200 OK
Content-Type: application/json

{
  "issuer":"https://server.somesite.example",
  "authorization_endpoint":
    "https://server.somesite.example/authorize",
  "resource_servers":[
    "email.somesite.example",
    "storage.somesite.example",
    "video.somesite.example"
  ]
  ...
}
```

인가 서버는 또한 액세스 토큰이 유효한 URL을 토큰 응답에서 반환할 수 있으며, 예제와 비표준 반환 매개변수 access_token_resource_server에 의해 설명된다:

```
HTTP/1.1 200 OK
Content-Type: application/json;charset=UTF-8
Cache-Control: no-store
Pragma: no-cache

{
  "access_token":"2YotnFZFEjr1zCsicMWpAA",
  "access_token_resource_server":
    "https://hostedresource.somesite.example/path1",
...
}
```

이 완화 전략은 클라이언트가 보안 정책을 강제하고 합법적인 대상에만 액세스 토큰을 보내는 것에 의존할 것이다. OAuth 관련 보안 연구의 결과(예: [research.ubc] 및 [research.cmu] 참조)는 클라이언트 구현의 상당 부분이 state 검사와 같은 보안 통제를 적절히 구현하지 않거나 실패하는 것을 나타낸다. 따라서, 액세스 토큰 피싱을 방지하기 위해 클라이언트에 의존하는 것은 역시 실패할 가능성이 높다. 더욱이, 인가 서버와 리소스 서버 대비 클라이언트의 비율을 감안하면, 가능한 한 많은 보안 관련 로직을 해당 서버로 이동시키는 것이 더 실행 가능한 접근 방식으로 간주된다. 물론, 클라이언트도 전체 보안에 기여해야 한다. 그러나, 섹션 4.10.1 및 4.10.2에 기술된 대안적 대응 방안이 관련 당사자 간에 더 나은 균형을 제공한다.

### 4.11. 오픈 리다이렉션 (Open Redirection)

다음 공격은 인가 서버 또는 클라이언트에 오픈 리다이렉터가 있을 때 발생할 수 있다. 이러한 엔드포인트는 때때로 구현되며, 예를 들어, 사용자가 외부 웹사이트로 리다이렉트되기 전에 메시지를 표시하거나, 사용자가 중단(예: 로그인 프롬프트에 의해)되기 전에 방문하려고 했던 URL로 다시 리다이렉트하기 위해서이다.

#### 4.11.1. 클라이언트의 오픈 리다이렉터 (Client as Open Redirector)

클라이언트는 오픈 리다이렉터를 노출해서는 안 된다(MUST NOT). 공격자는 오픈 리다이렉터를 사용하여 클라이언트를 가리키는 URL을 생성하고, 섹션 4.1.2에 기술된 대로 인가 코드 및 액세스 토큰을 유출하는 데 활용할 수 있다. 또 다른 악용 사례는 클라이언트를 가리키는 것처럼 보이는 URL을 생성하는 것이다. 이는 사용자를 속여 URL을 신뢰하고 브라우저에서 따라가도록 할 수 있다. 이것은 피싱에 악용될 수 있다.

오픈 리다이렉션을 방지하기 위해, 클라이언트는 대상 URL이 허용된 경우에만 또는 요청의 출처와 무결성이 인증될 수 있는 경우에만 리다이렉트해야 한다. 오픈 리다이렉션에 대한 대응 방안은 OWASP [owasp.redir]에 기술되어 있다.

#### 4.11.2. 인가 서버의 오픈 리다이렉터 (Authorization Server as Open Redirector)

클라이언트와 마찬가지로, 공격자는 피싱 공격을 수행하기 위해 인가 서버(특히 그 URL)에 대한 사용자의 신뢰를 활용하려고 할 수 있다. OAuth 인가 서버는 정기적으로 사용자를 다른 웹사이트(클라이언트)로 리다이렉트하지만, 안전하게 해야 한다.

[RFC6749]의 섹션 4.1.2.1은 이미 잘못된 client_id와 redirect_uri 조합의 경우 인가 서버가 사용자 에이전트를 자동으로 리다이렉트하지 말아야(MUST NOT) 한다고 명시하여 오픈 리다이렉트를 방지하고 있다.

그러나, 공격자는 올바르게 등록된 리다이렉션 URI를 활용하여 피싱 공격을 수행할 수도 있다. 공격자는 예를 들어 동적 클라이언트 등록 [RFC7591]을 통해 클라이언트를 등록하고 다음 공격 중 하나를 실행할 수 있다:

1. 잘못된 scope 값 사용 등으로 의도적으로 잘못된 인가 요청을 보내, 인가 서버가 사용자 에이전트를 피싱 사이트로 리다이렉트하도록 지시한다.
2. 공격자가 통제하는 client_id와 redirect_uri로 의도적으로 유효한 인가 요청을 보낸다. 사용자가 인증한 후, 인가 서버는 사용자에게 요청에 대한 동의를 제공하도록 프롬프트한다. 사용자가 요청에 문제가 있음을 인지하고 요청을 거부하면, 인가 서버는 여전히 사용자 에이전트를 피싱 사이트로 리다이렉트한다. 이 경우, 사용자 에이전트는 사용자가 취한 동작에 관계없이 피싱 사이트로 리다이렉트된다.
3. 공격자가 통제하는 client_id와 redirect_uri로 의도적으로 유효한 자동 인증 요청(prompt=none)을 보낸다. 이 경우, 인가 서버가 자동으로 사용자 에이전트를 피싱 사이트로 리다이렉트한다.

인가 서버는 이러한 위협을 방지하기 위한 예방 조치를 취해야(MUST) 한다. 인가 서버는 항상 먼저 사용자를 인증하고, 자동 인증 사용 사례를 제외하고, 사용자를 리다이렉트하기 전에 필요할 때 자격증명을 프롬프트해야(MUST) 한다. 위험 평가에 기반하여, 인가 서버는 리다이렉션 URI를 신뢰할 수 있는지 여부를 결정해야 한다. URI 뒤의 콘텐츠의 신뢰성과 신뢰도를 평가하기 위해 내부적으로 또는 외부 서비스를 통해 수행되는 URI 분석, 리다이렉션 URI의 출처 및 기타 클라이언트 데이터를 고려할 수 있다.

인가 서버는 리다이렉션 URI를 신뢰하는 경우에만 자동으로 사용자 에이전트를 리다이렉트해야(SHOULD) 한다. URI가 신뢰되지 않는 경우, 인가 서버는 사용자에게 알리고 사용자가 올바른 결정을 내리도록 의존할 수 있다(MAY).

### 4.12. 307 리다이렉트 (307 Redirect)

인가 엔드포인트에서, 전형적인 프로토콜 흐름은 인가 서버가 사용자에게 양식에 자격증명을 입력하도록 프롬프트하고, 이 양식이 (HTTP POST 방법을 사용하여) 인가 서버로 다시 제출되는 것이다. 인가 서버는 자격증명을 확인하고, 성공하면, 사용자 에이전트를 클라이언트의 리다이렉션 엔드포인트로 리다이렉트한다.

[RFC6749]에서는 이 목적을 위해 HTTP 상태 코드 302 (Found)가 사용되지만, "사용자 에이전트를 통해 이 리다이렉션을 수행하는 데 사용할 수 있는 다른 모든 방법이 허용된다". 리다이렉션에 상태 코드 307이 대신 사용되면, 사용자 에이전트는 HTTP POST를 통해 사용자의 자격증명을 클라이언트로 전송할 것이다.

이것은 민감한 자격증명을 클라이언트에 공개한다. 클라이언트가 악의적인 경우, 인가 서버에서 사용자를 사칭하는 데 자격증명을 사용할 수 있다.

이 동작은 개발자에게 예상치 못한 것일 수 있지만 [RFC9110]의 섹션 15.4.8에 정의되어 있다. 이 상태 코드(307)는 사용자 에이전트가 POST 요청을 GET 요청으로 재작성하여 POST 요청 본문의 양식 데이터를 삭제하는 것을 요구하지 않는다.

HTTP 표준 [RFC9110]에서, 상태 코드 303만이 HTTP POST 요청을 HTTP GET 요청으로 재작성하는 것을 모호하지 않게 강제한다. 인기 있는 302를 포함한 다른 모든 상태 코드에 대해, 사용자 에이전트는 POST를 GET 요청으로 재작성하지 않는 것을 선택할 수 있으며, 이로 인해 사용자의 자격증명이 클라이언트에 공개된다. (그러나 실제로, 대부분의 사용자 에이전트는 307 리다이렉트에 대해서만 이 동작을 보여줄 것이다.)

사용자의 자격증명을 잠재적으로 포함하는 요청을 리다이렉트하는 인가 서버는 따라서 리다이렉션에 HTTP 307 상태 코드를 사용해서는 안 된다(MUST NOT). 이러한 요청에 HTTP 리다이렉션이 (JavaScript가 아닌) 사용되는 경우, 인가 서버는 HTTP 상태 코드 303 (See Other)을 사용해야(SHOULD) 한다.

### 4.13. TLS 종단 리버스 프록시 (TLS Terminating Reverse Proxies)

HTTP 애플리케이션의 일반적인 배포 아키텍처는 TLS 연결을 종단하고 수신 요청을 해당 애플리케이션 서버 노드로 디스패치하는 리버스 프록시 뒤에 애플리케이션 서버를 숨기는 것이다.

이 섹션은 OAuth와 관련된 이 배포 아키텍처의 일부 공격 각도를 강조하고 보안 통제에 대한 권고를 제공한다.

일부 상황에서, 리버스 프록시는 추가 처리를 위해 보안 관련 데이터를 업스트림 애플리케이션 서버로 전달해야 한다. 예로는 요청 발생자의 IP 주소, 토큰 바인딩 ID, 인증된 TLS 클라이언트 인증서가 있다. 이 데이터는 일반적으로 업스트림 요청에 추가된 HTTP 헤더로 전달된다. 헤더는 종종 사용자 정의, 애플리케이션별 헤더이지만, 클라이언트 인증서 및 클라이언트 인증서 체인에 대한 표준화된 헤더 필드가 [RFC9440]에 정의되어 있다.

리버스 프록시가 외부에서 전송된 모든 헤더를 통과시키는 경우, 공격자는 보안 통제를 우회하기 위해 프록시를 통해 위조된 헤더 값을 애플리케이션 서버로 직접 보내려고 시도할 수 있다. 예를 들어, 리버스 프록시가 X-Forwarded-For 헤더를 수락하고 인바운드 요청의 출처를 추가하는 것(목록으로 만드는)은 표준 관행이다. 애플리케이션 서버에서 수행되는 로직에 따라, 공격자는 단순히 허용된 IP 주소를 헤더에 추가하고 보호를 무용지물로 만들 수 있다.

따라서 리버스 프록시는 애플리케이션 서버의 보안에 관련된 모든 헤더 값의 진정성과 무결성을 보장하기 위해 모든 인바운드 요청을 정화해야(MUST) 한다.

공격자가 프록시와 애플리케이션 서버 사이의 내부 네트워크에 접근할 수 있는 경우, 공격자는 보안 통제를 우회하려고 시도할 수도 있다. 따라서, 통신 엔터티의 진정성을 보장하는 것이 필수적이다. 더 나아가, 리버스 프록시와 애플리케이션 서버 간의 통신 링크는 도청, 주입, 메시지 재생으로부터 보호되어야(MUST) 한다.

### 4.14. 리프레시 토큰 보호 (Refresh Token Protection)

리프레시 토큰은 새로운 액세스 토큰을 획득하기 위한 편리하고 사용자 친화적인 방법이다. 또한 인가 서버가 짧은 수명과 축소된 범위의 액세스 토큰을 발급할 수 있게 하여 OAuth의 보안에도 기여하며, 액세스 토큰 유출의 잠재적 영향을 줄인다.

#### 4.14.1. 논의 (Discussion)

리프레시 토큰은 특정 클라이언트에 부여된 접근의 전체 범위를 나타내며 특정 리소스로 더 이상 제약되지 않으므로, 공격자에게 매력적인 대상이다. 공격자가 리프레시 토큰을 유출하고 성공적으로 재생할 수 있다면, 공격자는 액세스 토큰을 발행하고 리소스 소유자를 대신하여 리소스 서버에 접근하는 데 사용할 수 있을 것이다.

[RFC6749]는 이미 다음을 요구하여 강건한 기본 보호를 제공한다:

*  전송 및 저장에서의 리프레시 토큰의 기밀성,
*  인가 서버와 클라이언트 간의 TLS 보호 연결을 통한 리프레시 토큰의 전송,
*  인가 서버가 리프레시 토큰의 특정 클라이언트에 대한 바인딩을 유지하고 검사하며, 가능한 경우 토큰 갱신 중 이 클라이언트의 인증, 그리고
*  리프레시 토큰이 생성, 수정, 추측될 수 없음.

[RFC6749]는 또한 리프레시 토큰 만료 및 해지와 리프레시 토큰 로테이션을 위한 해당 오류 코드와 응답 동작을 정의하여 추가적인 (구현별) 보안 조치를 위한 기반을 마련한다.

이 명세는 [RFC6749]의 범위를 넘어서는 권고 사항과 명확화를 제공한다.

#### 4.14.2. 권고 사항 (Recommendations)

인가 서버는 위험 평가에 기반하여 특정 클라이언트에 리프레시 토큰을 발급할지 여부를 결정해야(MUST) 한다. 인가 서버가 리프레시 토큰을 발급하지 않기로 결정한 경우, 클라이언트는 인가 코드 그랜트 유형과 같은 다른 그랜트 유형을 활용하여 새 액세스 토큰을 획득할 수 있다(MAY). 이러한 경우, 인가 서버는 사용자 경험을 최적화하기 위해 쿠키와 영구 그랜트를 활용할 수 있다.

리프레시 토큰이 발급되는 경우, 해당 리프레시 토큰은 리소스 소유자가 동의한 범위와 리소스 서버에 바인딩되어야(MUST) 한다. 이는 합법적인 클라이언트에 의한 권한 상승을 방지하고 리프레시 토큰 유출의 영향을 줄이기 위한 것이다.

기밀 클라이언트의 경우, [RFC6749]는 이미 리프레시 토큰이 발급된 클라이언트에 의해서만 사용될 수 있도록 요구한다.

인가 서버는 공개 클라이언트에 대해 악의적인 행위자에 의한 리프레시 토큰 재생을 탐지하기 위해 다음 방법 중 하나를 활용해야(MUST) 한다:

*  *발신자 제약 리프레시 토큰:* 인가 서버는 [RFC8705] 또는 [RFC9449]를 활용하는 등의 방법으로 리프레시 토큰을 특정 클라이언트 인스턴스에 암호학적으로 바인딩한다.

*  *리프레시 토큰 로테이션:* 인가 서버는 모든 액세스 토큰 갱신 응답과 함께 새 리프레시 토큰을 발급한다. 이전 리프레시 토큰은 무효화되지만, 관계에 대한 정보는 인가 서버에 의해 유지된다. 리프레시 토큰이 손상되고 이후 공격자와 합법적인 클라이언트 모두에 의해 사용되면, 그 중 하나는 무효화된 리프레시 토큰을 제시하게 되며, 이는 인가 서버에 침해를 알려준다. 인가 서버는 어느 당사자가 무효한 리프레시 토큰을 제출했는지 결정할 수 없지만, 활성 리프레시 토큰을 해지할 것이다. 이는 합법적인 클라이언트가 새로운 인가 그랜트를 획득하도록 강제하는 대가를 치르고 공격을 중단시킨다.

   구현 참고: 리프레시 토큰이 속하는 그랜트가 리프레시 토큰 자체에 인코딩될 수 있다. 이를 통해 인가 서버는 리프레시 토큰이 속하는 그랜트를, 그리고 확장하여 해지되어야 하는 모든 리프레시 토큰을 효율적으로 결정할 수 있다. 인가 서버는 이 경우 리프레시 토큰 값의 무결성을 보장해야(MUST) 한다. 예를 들어, 서명을 사용하여.

인가 서버는 다음과 같은 보안 이벤트의 경우 리프레시 토큰을 자동으로 해지할 수 있다(MAY):

*  비밀번호 변경 또는
*  인가 서버에서의 로그아웃.

리프레시 토큰은 클라이언트가 일정 기간 동안 비활성인 경우, 즉 리프레시 토큰이 일정 기간 동안 새 액세스 토큰을 획득하는 데 사용되지 않은 경우 만료되어야(SHOULD) 한다. 만료 시간은 인가 서버의 재량이다. 이는 전역 값이거나 클라이언트 정책 또는 리프레시 토큰과 관련된 그랜트(및 그 민감도)에 기반하여 결정될 수 있다.

### 4.15. 클라이언트의 리소스 소유자 사칭 (Client Impersonating Resource Owner)

리소스 서버는 액세스 토큰이 발급된 리소스 소유자의 ID에 기반하여, 또는 클라이언트 자격증명 그랜트에서의 클라이언트 ID에 기반하여 접근 제어 결정을 내릴 수 있다. 예를 들어, [RFC9068] (OAuth 2.0 액세스 토큰을 위한 JSON Web Token (JWT) 프로파일)은 다음과 같이 정의된 sub 클레임을 포함하는 액세스 토큰의 데이터 구조를 기술한다:

> 인가 코드 그랜트와 같이 리소스 소유자가 관여하는 그랜트를 통해 획득한 액세스 토큰의 경우, "sub"의 값은 리소스 소유자의 주체 식별자에 대응해야(SHOULD) 한다. 클라이언트 자격증명 그랜트와 같이 리소스 소유자가 관여하지 않는 그랜트를 통해 획득한 액세스 토큰의 경우, "sub"의 값은 인가 서버가 클라이언트 애플리케이션을 나타내는 데 사용하는 식별자에 대응해야(SHOULD) 한다.

두 옵션이 모두 가능한 경우, 리소스 서버는 클라이언트의 ID를 리소스 소유자의 ID로 착각할 수 있다. 예를 들어, 클라이언트가 인가 서버에 등록하는 동안 자신의 client_id를 선택할 수 있는 경우, 악의적인 클라이언트는 이를 리소스 소유자를 식별하는 값으로 설정할 수 있다(예: OpenID Connect가 사용되는 경우 sub 값). 리소스 서버가 리소스 소유자의 관여로 획득한 액세스 토큰과 그렇지 않은 것을 적절히 구별할 수 없는 경우, 클라이언트가 실수로 리소스 소유자에게 속하는 리소스에 접근할 수 있게 될 수 있다.

이 공격은 [RFC9068]을 사용하는 구현뿐만 아니라 유사한 맞춤형 해결책에도 잠재적으로 영향을 미친다.

#### 4.15.1. 대응 방안 (Countermeasures)

인가 서버는 위의 섹션 4.15에 표시된 [RFC9068]의 sub 클레임 예와 같이 클라이언트 ID와 사용자 식별자에 대한 공통 네임스페이스가 존재하는 경우, 클라이언트가 자신의 client_id나 진정한 리소스 소유자와 혼동을 일으킬 수 있는 기타 클레임에 영향을 미치는 것을 허용해서는 안 된다(SHOULD NOT). 이를 피할 수 없는 경우, 인가 서버는 리소스 서버가 두 유형의 액세스 토큰을 구별할 수 있는 다른 수단을 제공해야(MUST) 한다.

### 4.16. 클릭재킹 (Clickjacking)

[RFC6819]의 섹션 4.4.1.9에 기술된 대로, 인가 요청은 사용자 인터페이스 재장식(user interface redressing)이라고도 불리는 클릭재킹 공격에 취약하다. 이러한 공격에서, 공격자는 인가 엔드포인트 사용자 인터페이스를 무해한 컨텍스트에 삽입한다. 사용자는 해당 컨텍스트와 상호작용하고 있다고 믿으면서, 예를 들어 버튼을 클릭하면, 실제로는 대신 인가 엔드포인트 사용자 인터페이스와 상호작용한다. 반대도 달성될 수 있다: 사용자가 인가 엔드포인트와 상호작용하고 있다고 믿으면서 원래 사용자 인터페이스 위에 겹쳐진 공격자 제공 입력 필드에 실수로 비밀번호를 입력할 수 있다. 클릭재킹 공격은 사용자가 거의 알아차릴 수 없도록 설계될 수 있으며, 예를 들어 다른 요소 위에 겹쳐진 거의 보이지 않는 iframe을 사용한다.

공격자는 이 벡터를 사용하여 사용자의 인증 자격증명을 획득하고, 클라이언트에 부여된 접근 범위를 변경하며, 잠재적으로 사용자의 리소스에 접근할 수 있다.

인가 서버는 클릭재킹 공격을 방지해야(MUST) 한다. X-Frame-Options HTTP 응답 헤더 필드와 프레임 버스팅 JavaScript의 사용을 포함하여 [RFC6819]에 여러 대응 방안이 기술되어 있다. 이러한 것 외에도, 인가 서버는 콘텐츠 보안 정책(CSP) 레벨 2 [W3C.CSP-2] 이상도 사용해야(SHOULD) 한다.

효과적이려면, CSP는 인가 엔드포인트와, 해당되는 경우, 사용자를 인증하고 클라이언트를 인가하는 데 사용되는 기타 엔드포인트(예: 장치 인가 엔드포인트, 로그인 페이지, 오류 페이지 등)에서 사용되어야 한다. 이것은 CSP를 지원하는 사용자 에이전트에서 인가되지 않은 출처에 의한 프레이밍을 방지한다. 클라이언트는 리다이렉션 엔드포인트에서 사용되는 것 이외의 다른 출처에 의해 프레이밍되는 것을 허용할 수 있다(MAY). 이러한 이유로, 인가 서버는 관리자가 특정 클라이언트에 대해 허용된 출처를 구성하거나 클라이언트가 이를 동적으로 등록할 수 있도록 허용해야(SHOULD) 한다.

CSP를 사용하면 인가 서버가 단일 응답 헤더 필드에 여러 출처를 지정하고 유연한 패턴을 사용하여 이를 제약할 수 있다(자세한 내용은 [W3C.CSP-2] 참조). CSP 레벨 2는 프레임의 출처를 제한하는 정책(frame-ancestors 사용)과 HTML 페이지에서 실행이 허용되는 스크립트의 출처를 제한하는 정책(script-src 사용)을 함께 사용하여 클릭재킹에 대한 강건한 메커니즘을 제공한다. 이러한 정책의 비규범적 예가 다음 목록에 표시된다:

```
HTTP/1.1 200 OK
Content-Security-Policy: frame-ancestors https://ext.example.org:8000
Content-Security-Policy: script-src 'self'
X-Frame-Options: ALLOW-FROM https://ext.example.org:8000
...
```

일부 사용자 에이전트가 [W3C.CSP-2]를 지원하지 않으므로, 이 기술은 이러한 레거시 사용자 에이전트가 인가 서버에 의해 명시적으로 지원되지 않는 경우를 제외하고, [RFC6819]에 기술된 것을 포함한 다른 것들과 결합되어야(SHOULD) 한다. 그러한 경우에도, 추가 대응 방안이 여전히 사용되어야(SHOULD) 한다.

### 4.17. 브라우저 내 통신 흐름에 대한 공격 (Attacks on In-Browser Communication Flows)

인가 응답이 HTTP 리다이렉트 대신 postMessage [WHATWG.postmessage_api]와 같은 브라우저 내 통신 기술로 전송되는 경우, 메시지가 실수로 악의적인 출처로 전송되거나 악의적인 출처로부터 주입될 수 있다.

#### 4.17.1. 예제 (Examples)

브라우저 내 통신을 사용하는 공격의 다음 비규범적 의사 코드 예제는 [research.rub]에 기술되어 있다.

##### 4.17.1.1. 수신자 출처의 불충분한 제한 (Insufficient Limitation of Receiver Origins)

postMessage를 통해 인가 응답 또는 토큰 응답을 보낼 때, 인가 서버가 클라이언트의 출처 대신 와일드카드 출처 "*"로 응답을 보낸다. 응답이 전송되는 창이 공격자에 의해 통제되는 경우, 공격자는 응답을 읽을 수 있다.

```
window.opener.postMessage(
  {
    code: "ABC",
    state: "123"
  },
  "*" // opener 창의 모든 웹사이트가 메시지를 수신할 수 있음
)
```

##### 4.17.1.2. 불충분한 URI 검증 (Insufficient URI Validation)

postMessage를 통해 인가 응답 또는 토큰 응답을 보낼 때, 인가 서버가 수신자 출처를 리다이렉션 URI와 대조하여 검사하지 않고, 예를 들어 공격자가 제공한 출처로 응답을 보낼 수 있다. 이것은 섹션 4.1에 기술된 공격과 유사하다.

```
window.opener.postMessage(
  {
    code: "ABC",
    state: "123"
  },
  "https://attacker.example" // 공격자가 제공한 값
)
```

##### 4.17.1.3. 발신자 출처의 불충분한 검증 후 주입 (Injection after Insufficient Validation of Sender Origin)

postMessage를 통해 인가 응답 또는 토큰 응답을 기대하는 클라이언트가 메시지의 발신자 출처를 검증하지 않을 수 있다. 이를 통해 공격자가 인가 응답 또는 토큰 응답을 클라이언트에 주입할 수 있다.

악의적으로 주입된 인가 응답의 경우, 공격은 섹션 4.7에 기술된 CSRF 공격의 변형이다. 섹션 4.7에 기술된 대응 방안이 이 공격에도 적용된다.

악의적으로 주입된 토큰 응답의 경우, 섹션 4.10.1에 기술된 발신자 제약 액세스 토큰이 일부 상황에서 공격을 방지할 수 있지만, 섹션 4.17.2에 기술된 추가 대응 방안이 일반적으로 필요하다.

#### 4.17.2. 권고 사항 (Recommendations)

클라이언트 수신자 출처를 사전 등록된 출처와 비교할 때, 인가 서버는 섹션 4.1.3에 기술된 대로 정확한 문자열 매칭을 활용해야(MUST) 한다. 인가 서버는 다음의 비규범적 예에 표시된 대로 신뢰할 수 있는 클라이언트 수신자 출처로 postMessage를 전송해야(MUST) 한다:

```
window.opener.postMessage(
  {
    code: "ABC",
    state: "123"
  },
  "https://client.example" // 명시적 클라이언트 출처 사용
)
```

postMessage에서 "*"와 같은 와일드카드 출처는 사용되어서는 안 된다(MUST NOT). 공격자가 이를 사용하여 피해자의 브라우저 내 메시지를 악의적인 출처로 유출할 수 있기 때문이다. 두 조치 모두 인가 코드 및 액세스 토큰의 유출 방지에 기여한다(섹션 4.1 참조).

클라이언트는 클라이언트 수신자 엔드포인트에서의 브라우저 내 메시지 주입을 방지해야(MUST) 한다. 클라이언트는 다음의 비규범적 예에 표시된 대로, 브라우저 내 메시지의 개시자 출처를 인가 서버 출처와 정확한 문자열 매칭을 사용하여 비교해야(MUST) 한다:

```
window.addEventListener("message", (e) => {
  // 정확한 인가 서버 출처 검증
  if (e.origin === "https://honest.as.example") {
    // e.data.code와 e.data.state 처리
  }
})
```

브라우저 내 통신 흐름은 다른 통신 기술(즉, HTTP 리다이렉트 대신 postMessage)만 적용하므로, 섹션 2.1에 나열된 인가 응답을 보호하는 모든 조치가 동일하게 적용되어야(MUST) 한다.
