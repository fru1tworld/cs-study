# RFC 6797: HTTP Strict Transport Security (HSTS)

## 개요

RFC 6797은 2012년 11월에 발행되었으며, **HTTP Strict Transport Security (HSTS)** 메커니즘을 정의합니다. HSTS는 웹사이트가 **보안 연결(HTTPS)을 통해서만 접근 가능하다** 고 선언할 수 있게 하며, User Agent(브라우저)가 이 정책을 강제하도록 합니다.

### 핵심 기능
1. **URI 변환**: UA가 HSTS 호스트에 대한 비보안 URI 참조를 보안 URI 참조로 변환
2. **연결 강제**: 보안 전송 연결 시도 중 오류나 경고 발생 시 연결 종료

---

## 1. 사용 사례

### 1.1 웹 사용자
다양한 웹사이트와의 안전한 상호작용을 원하는 사용자

### 1.2 사이트 운영자
사용자와 자신을 위해 명시적으로 안전한 서비스를 제공하려는 웹사이트 운영자

---

## 2. 위협 모델

### 2.1 방어 대상 위협

| 위협 유형 | 설명 |
|----------|------|
| **수동적 네트워크 공격자** | 무료로 사용 가능한 무선 스니핑 도구를 사용한 도청. 세션 식별자/쿠키를 탈취하여 웹 세션 하이재킹 |
| **능동적 네트워크 공격자** | DNS 서버를 가장하거나 네트워크 프레임을 스푸핑. 브라우저가 사용자에게 보안 경고를 쉽게 무시하도록 허용하는 점을 악용 |
| **웹 개발 버그** | 비보안 연결을 통한 CSS 로딩 같은 단순한 실수로 전체 보안 사이트가 완전히 손상될 수 있음 |

### 2.2 방어하지 않는 위협

| 위협 유형 | 이유 |
|----------|------|
| **피싱 공격** | HSTS는 그 자체로 피싱에 대한 방어 수단이 아님 |
| **악성 소프트웨어/브라우저 취약점** | HSTS는 사용자 시스템의 신뢰성에 의존 |

---

## 3. HSTS 정책 선언

### 3.1 선언 방법

HSTS 호스트는 **보안 전송을 통해** `Strict-Transport-Security` HTTP 응답 헤더 필드를 발행하여 HSTS 정책을 선언합니다.

```http
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

### 3.2 중요 규칙

- **비보안 연결에서 수신된 STS 헤더는 무시해야 합니다**
- 정책은 도메인 이름으로 색인된 저장소에 저장됩니다
- 해당 HSTS 호스트만 자신의 정책을 업데이트하거나 삭제할 수 있습니다

---

## 4. Strict-Transport-Security 헤더

### 4.1 지시문 (Directives)

| 지시문 | 필수 여부 | 설명 |
|--------|----------|------|
| `max-age` | **필수** | STS 헤더 수신 후 UA가 호스트를 Known HSTS Host로 간주하는 기간(초) |
| `includeSubDomains` | 선택 | 이 호스트와 모든 서브도메인에 HSTS 정책 적용 |

### 4.2 예시

#### 기본 정책 (1년)
```http
Strict-Transport-Security: max-age=31536000
```

#### 서브도메인 포함 정책 (6개월)
```http
Strict-Transport-Security: max-age=15768000; includeSubDomains
```

#### 정책 삭제
```http
Strict-Transport-Security: max-age=0
```

### 4.3 max-age 값 참고

| 기간 | 초 |
|------|-----|
| 1일 | 86,400 |
| 1주 | 604,800 |
| 1개월 | 2,592,000 |
| 3개월 | 7,776,000 |
| 6개월 | 15,768,000 |
| 1년 | 31,536,000 |
| 2년 | 63,072,000 |

## 5. User Agent 처리 모델

### 5.1 헤더 필드 처리

1. **보안 전송을 통해 수신** 된 경우에만 STS 헤더 필드 처리
2. 비보안 전송을 통해 수신된 HTTP 응답의 STS 헤더 필드는 **무시**
3. 여러 STS 헤더 필드가 있는 경우 **첫 번째 것만 처리**

### 5.2 도메인 이름 매칭

매칭은 오른쪽에서 왼쪽으로 레이블별로 수행되며, ASCII 대소문자를 구분하지 않습니다.

| 매칭 유형 | 설명 |
|----------|------|
| **Superdomain Match** | Known HSTS Host의 도메인 이름이 주어진 도메인 이름의 오른쪽 부분과 레이블별로 일치 |
| **Congruent Match** | Known HSTS Host의 도메인 이름이 주어진 도메인 이름과 완전히 일치 |

### 5.3 URI 로딩 및 포트 매핑

HTTP URI를 로드할 때 UA는 도메인 이름 매칭을 확인하고 변환을 적용합니다:

| 원본 | 변환 후 |
|------|---------|
| `http://example.com` | `https://example.com` |
| `http://example.com:80` | `https://example.com:443` |
| `http://example.com:8080` | `https://example.com:8080` (비표준 포트 유지) |

### 5.4 보안 전송 오류

> "Known HSTS Host에 연결할 때, UA는 기본 보안 전송과의 오류가 있으면 연결을 종료해야 합니다. '경고', '치명적' 또는 기타 모든 오류 수준에서."

**사용자에게 보안 경고를 무시하고 진행하는 옵션을 제공하지 않습니다.**

### 5.5 Meta 요소 무시

```html
<!-- UA는 이를 무시해야 함 -->
<meta http-equiv="Strict-Transport-Security" content="max-age=31536000">
```

HSTS 정책은 반드시 HTTP 헤더에서 가져와야 합니다.

### 5.6 누락된 헤더 필드

Known HSTS Host에서 보안 채널을 통해 HTTP 응답을 수신했지만 STS 헤더 필드가 누락된 경우, UA는 해당 호스트의 max-age 값에 도달할 때까지 **계속해서 HSTS Host로 처리** 해야 합니다.

## 6. 서버 구현 조언

### 6.1 비준수 User Agent

비준수 UA는 Strict-Transport-Security 헤더 필드를 무시합니다. 웹 운영자는 HSTS 채택과 관계없이 **기존 보안 관행을 유지** 해야 합니다.

### 6.2 만료 시간 고려사항

- HSTS 만료를 인증서 유효 기간과 맞추는 것을 고려
- 기본 구성 값으로 0 사용 권장
- 예: `max-age=7776000`초 = 90일

### 6.3 자체 서명 인증서

자체 서명 인증서와 HSTS를 함께 사용하면 조직이 CA 인증서를 브라우저에 배포하지 않는 한 실패합니다. 배포 방법은 피싱 위험을 피하기 위해 신중히 고려해야 합니다.

### 6.4 includeSubDomains 영향

#### HTTP 서비스 (대체 포트/서브도메인)

CRL 배포나 OCSP 서비스를 HTTP로 제공하는 인증 기관은 다음 옵션 중 선택해야 합니다:
- `includeSubDomains` 생략
- 서비스를 HTTPS로 마이그레이션
- HTTP 서비스를 다른 도메인 이름으로 이전
- OCSP Stapling 같은 대체 배포 방법 사용

#### 서브도메인의 웹 애플리케이션

STS 헤더가 `example.com`에서만 발행되지만 UA가 일반적으로 `www.example.com`을 북마크하고 링크가 설정된 경우, 일부 사용자는 보호되지 않습니다.

**해결책**: 잘 알려진 진입점이 되는 각 HSTS 호스트 도메인 또는 서브도메인 이름에서 직접 STS 헤더 필드를 발행합니다.

## 7. User Agent 구현 조언

### 7.1 사용자 회피 불가

> "모든 경고나 오류에서 보안 연결 설정 실패는 '사용자 회피 불가'로 수행되어야 합니다. 이는 사용자에게 진행 옵션을 제공하는 대화 상자를 표시하지 않아야 함을 의미합니다."

우회를 허용하면 중간자 공격 취약점이 생성됩니다. HSTS를 선언하는 애플리케이션은 암묵적으로 이 엄격한 접근 방식에 동의합니다.

### 7.2 사용자 선언 HSTS 정책

UA는 사용자가 상호작용 전에 HSTS 호스트를 수동으로 지정할 수 있도록 허용해야 합니다. 이는 부트스트랩 MITM 취약점을 완화합니다.

### 7.3 HSTS 사전 로드 목록 (Preload List)

> "HSTS 사전 로드 목록은 웹사이트 관리자가 UA 공급업체에 의해 자신의 사이트에 대한 HSTS 정책이 미리 구성되도록 할 수 있는 기능입니다."

- 사용자 선언을 보완
- 부트스트랩 공격으로부터 보호

### 7.4 혼합 보안 컨텍스트 로드

UA는 보안 컨텍스트에서 비보안 리소스를 가져오는 것을 방지해야 합니다.

### 7.5 HSTS 정책 삭제

UA는 호스트별 정책 삭제 기능을 제공해야 하지만 신중한 UI/보안 고려가 필요합니다:
- 삭제가 일상적이 되어서는 안 됨
- 구현은 주입된 스크립트를 통한 프로그래밍적 제거를 방지해야 함

## 8. 보안 고려사항

### 8.1 기본 보안 전송

이 사양은 TLS 또는 SSL을 기본 보안 전송으로 가정합니다. 다른 프로토콜과 함께 HSTS를 사용하려면 해당 보안 속성을 평가해야 합니다.

### 8.2 비준수 User Agent

비준수 구현은 다음에 취약합니다:
- 웹사이트 버그로 인한 수동적 공격 (암호화되지 않은 쿠키, 혼합 콘텐츠)
- "클릭스루" 결정이 있는 능동적 공격 (중간자)

### 8.3 오류 없는 연결 요구사항

HSTS 정책 설정에는 오류 없는 보안 연결이 필요합니다. 이는 MITM 공격이 잘못된 정책을 설정하는 것을 방지합니다.

### 8.4 includeSubDomains의 필요성

이 지시문이 없으면 "도메인 쿠키"가 취약합니다. 공격자가 사용되지 않는 서브도메인을 등록하고 인증서를 획득하면 상위 도메인용 쿠키를 캡처할 수 있습니다.

### 8.5 서비스 거부 (DoS)

HSTS는 공격자가 정책 설정을 유발하는 경우 HTTP 전용 애플리케이션에 대한 DoS 공격을 가능하게 할 수 있습니다:
- HTTP 응답 분할 취약점 악용
- 스푸핑된 리다이렉트
- 사용자 구성 오류

`includeSubDomains` 지시문은 서브도메인이 적절한 HTTPS 지원이 없는 경우 위험을 생성합니다.

### 8.6 부트스트랩 MITM 취약점

> "부트스트랩 MITM 취약점은 사용자가 'https' URI 대신 'http' URI를 사용하여 알려지지 않은 HSTS Host에 수동으로 입력하거나 링크를 따라가는 상황에서 사용자와 HSTS Host가 마주치는 취약점입니다."

**완화 방법**:
- 사전 로드 목록
- 사용자 선언 정책

### 8.7 네트워크 시간 공격

> "능동적 네트워크 공격은 네트워크 시간 프로토콜(NTP 등)을 전복시킬 수 있어 - NTP를 신뢰하거나 실시간 시계가 없는 클라이언트에 대해 HSTS를 덜 효과적으로 만듭니다."

### 8.8 가짜 CA 인증서 피싱

공격자는 피싱(사용자가 사기 CA 인증서를 설치하도록 유도)과 DNS 캐시 포이즈닝을 결합하여 사용자를 공격자가 제어하는 사이트로 리다이렉트할 수 있습니다.

**완화 요구사항**:
- DNSSEC
- 피싱 방어
- 가짜 인증서 주입 방지

### 8.9 창의적 HSTS 저장소 조작

HSTS 호스트를 제어하는 공격자는 도메인 이름에 정보를 인코딩하여 UA가 이 데이터를 캐시하도록 할 수 있습니다. 다른 사이트가 이 캐시된 도메인 이름을 쿼리하여 브라우징 기록을 결정할 수 있습니다 - 웹 추적의 한 형태입니다.

### 8.10 국제화된 도메인 이름

IDN 처리의 복잡성은 신중한 IDNA2008 구현이 필요합니다.

## 9. 핵심 요약

| 항목 | 내용 |
|------|------|
| **목적** | 웹사이트가 HTTPS 전용 접근을 선언하고 브라우저가 이를 강제 |
| **헤더** | `Strict-Transport-Security` |
| **필수 지시문** | `max-age` (초 단위) |
| **선택 지시문** | `includeSubDomains` |
| **정책 수신** | 보안 연결(HTTPS)을 통해서만 유효 |
| **사용자 우회** | 불가 (보안 오류 시 연결 종료) |
| **부트스트랩 문제** | 첫 연결은 취약 → Preload List로 해결 |

## 10. HSTS Preload List

### 10.1 개념

브라우저 공급업체(Chrome, Firefox, Safari 등)는 HSTS 정책이 미리 로드된 도메인 목록을 브라우저에 포함합니다. 이를 통해 부트스트랩 MITM 취약점을 제거합니다.

### 10.2 Preload 등록 요구사항

1. 유효한 인증서 사용
2. 포트 80에서 443으로 리다이렉트
3. 모든 서브도메인을 HTTPS로 서비스
4. 기본 도메인에서 HSTS 헤더 제공:
   - `max-age`가 최소 1년(31536000초) 이상
   - `includeSubDomains` 지시문 포함
   - `preload` 지시문 포함

### 10.3 예시

```http
Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
```
