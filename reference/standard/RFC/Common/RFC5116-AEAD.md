# RFC 5116 - 인증 암호화를 위한 인터페이스 및 알고리즘

> 원제: An Interface and Algorithms for Authenticated Encryption
> 저자: D. McGrew (Cisco Systems, Inc.)
> 분류: Standards Track
> 발행일: 2008년 1월

## 초록

이 문서는 AEAD(Authenticated Encryption with Associated Data) 알고리즘을 기술하고, 이러한 알고리즘들을 위한 통일된 인터페이스와 레지스트리를 정의한다. 이 접근 방식은 애플리케이션에 독립적인 암호 알고리즘 스위트를 가능하게 하며, 효율성과 보안 측면의 이점을 제공하는 동시에 암호 구현의 재사용을 촉진한다.

---

## 1. 서론

### 1.1 배경

인증 암호화는 기밀성(암호화)과 메시지 인증(MAC)을 단일 암호 알고리즘으로 결합한다. AEAD를 구현하는 여러 암호 알고리즘이 정의되어 있으며, 여기에는 블록 암호 모드와 전용 알고리즘이 포함된다. AEAD는 자동화된 암호 프로토콜 분석에서 사용되는 이상적인 암호화 모델에 근접한다.

### 1.2 범위

이 문서는 AEAD를 인터페이스 명세와 IANA 레지스트리 수립을 통해 추상화로 정의한다. 레지스트리에는 네 가지 알고리즘이 초기 등록된다:

- 128비트 및 256비트 키를 사용하는 AES 갈루아/카운터 모드(GCM)
- 128비트 및 256비트 키를 사용하는 AES 카운터 및 CBC MAC 모드(CCM)

이 명세가 다루는 내용:
- AEAD 인터페이스 정의
- 사용 지침
- 알고리즘 요구사항
- 알고리즘 정의
- IANA 레지스트리 수립
- 추가 고려사항

인터페이스 추상화는 재전송 방지 서비스나 접근 제어 결정과 같은 보안 프로토콜 문제를 다루지 않는다.

### 1.3 이점

AEAD가 애플리케이션 설계자에게 제공하는 이점:
- 보안 서비스와 데이터 마샬링에 집중할 수 있다
- 독립적인 보안 분석이 가능하다
- 암호 메커니즘 설계를 몰라도 된다
- 단일 키 입력만 필요한 간소화된 인터페이스를 제공한다
- 단일 알고리즘 식별자만 필요하다

구현자에게 제공하는 이점:
- 연산과 저장 공간을 줄이는 최적화 기회가 있다
- 통일된 인터페이스로 테스트가 간소화된다
- 단일 구현으로 여러 애플리케이션을 지원할 수 있다
- 하드웨어 가속이 가능하다

### 1.4 이 문서에서 사용하는 관례

RFC 2119 키워드 관례가 적용된다: "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", "OPTIONAL".

## 2. AEAD 인터페이스

AEAD 알고리즘은 두 가지 연산을 갖는다: 인증 암호화와 인증 복호화. 입력과 출력은 옥텟 문자열로 정의된다. 구현체는 상호 운용성에 영향을 주지 않는 추가 입력을 받을 수 있다(MAY).

### 2.1 인증 암호화

**네 가지 입력 (모두 옥텟 문자열):**

1. **비밀 키 K**: 균일 랜덤 또는 의사 랜덤이어야 한다(MUST). 길이는 1~255 옥텟이며, 알고리즘별로 고정된다.

2. **논스 N**: 특정 키 값에 대해 각 호출마다 고유한 논스를 사용해야 한다(MUST). 단, 모든 논스가 영 길이인 경우는 예외이다. 고유한 논스를 생성할 수 있는 애플리케이션은 섹션 3.2에 정의된 방법을 사용해야 한다(SHOULD). 그 외의 경우 영 길이 논스를 사용해야 한다(SHOULD). 권장 길이는 12 옥텟이다. 특정 키와 함께 다른 길이를 사용할 수도 있다(MAY).

3. **평문 P**: 암호화 및 인증할 데이터. 영 길이가 가능하다(MAY).

4. **연관 데이터 A**: 인증만 하고 암호화하지 않는 데이터. 영 길이가 가능하다(MAY).

**단일 출력:**
- 암호문 C (최소 평문 이상의 길이), 또는
- 암호화를 수행할 수 없다는 표시

**길이 제한:**

| 입력 | 최소 | 최대 | 비고 |
|------|------|------|------|
| 키 | 1 옥텟 | 255 옥텟 | 알고리즘별 고정 |
| 논스 | 0 옥텟 | 알고리즘에 따름 | 12 옥텟 권장 |
| 평문 | 0 옥텟 | 제한 없음 | |
| 연관 데이터 | 0 옥텟 | 제한 없음 | |
| 암호문 | 0 옥텟 | 제한 없음 | |

**중요 제약사항:**

- 입력 길이가 허용 범위를 초과하면, 구현체는 부분 출력 없이 오류 코드를 반환해야 한다(MUST).
- 평문 P에 대해 기밀성과 인증이 제공된다.
- 영 길이 P는 알고리즘이 입력 A에 대한 MAC으로 동작하게 한다.
- 연관 데이터는 인증은 필요하지만 기밀성은 필요 없는 정보를 보호한다.
- 비밀 키 K는 다른 입력에 명시적으로 포함되어서는 안 된다(MUST NOT).
- 논스는 내부적으로 인증되므로 AD에 포함할 필요는 없지만, P나 A에 포함될 수 있다(MAY).
- 논스는 암호문과 함께 저장/전송되거나, 복호화 전에 재구성될 수 있다(MAY).
- 애플리케이션은 암호문의 구조나 형식을 가정해서는 안 된다(MUST NOT).

### 2.2 인증 복호화

**네 가지 입력:** K, N, A, C (위에서 정의한 대로)

**단일 출력:** 평문 P 또는 특수 기호 FAIL

암호문 C, 논스 N, 연관 데이터 A가 키 K에 대해 진본(authentic)이라 함은, C가 입력 K, N, P, A로 암호화 연산에 의해 생성된 경우를 말한다. 인증 복호화는 논스를 준수하는 공격자가 비밀 키를 모른 채 입력을 조작한 경우 높은 확률로 FAIL을 반환한다(안전한 AEAD를 가정).

### 2.3 데이터 형식화

AEAD 입출력에 대한 특정 인코딩은 지정되지 않는다. 애플리케이션 데이터를 형식화할 때, 암호문 C는 다른 복호화 입력에 필요한 데이터 뒤에 위치해야 한다(SHOULD) (예: 논스가 암호문보다 앞에 와야 한다). 이는 효율적인 하드웨어 구현에 유리하다.

## 3. AEAD 알고리즘 사용 지침

이 섹션은 안전한 AEAD 사용을 위한 조언을 제공한다. 논스 고유성 요구사항을 충족할 수 없는 애플리케이션은 무작위화 또는 상태 유지 알고리즘과 함께 영 길이 논스를 사용해야 한다(MUST). 그 외의 경우, 상호 운용성을 위해 12 옥텟 논스를 사용해야 한다(SHOULD).

### 3.1 논스 생성 요구사항

논스의 보안 요구사항: 고정된 키에 대해 각 인증 암호화 호출마다 각 논스 값이 고유해야 한다.

**다중 장치 시나리오:** 단일 키를 공유하는 장치들은 논스 고유성을 조율해야 한다. 간단한 접근법은 장치 식별 필드를 포함하는 논스 형식이다(섹션 3.2). 완전한 논스가 암호문과 함께 전송되는 경우, 암호화기와 복호화기 간에 형식 세부사항을 조율할 필요가 없다. 애플리케이션은 암호화 엔진에 논스 선택의 자유를 제공해야 한다(SHOULD).

**장기 키 사용:** 비밀 키가 여러 재부팅에 걸쳐 지속되는 경우, 논스에 비휘발성 저장소와 체크포인팅이 필요하다. 높은 보증 방법은 후속 논스를 사용하기 전에 저장소 확인을 기다리는 것이다. 대안으로, 시퀀스에서 여러 값 앞선 논스를 저장할 수 있다.

**일반적 접근법:** 키를 변경하면 논스 조율의 어려움을 피할 수 있다.

**알고리즘 책임:** 각 AEAD 알고리즘은 논스 재사용으로 인한 보안 저하를 기술해야 한다(SHOULD).

### 3.2 권장 논스 형식

**권장 형식:** 논스는 고정 필드(Fixed field, 초기 옥텟)와 카운터 필드(Counter field, 마지막 옥텟)로 구성된다. 고정 길이는 암호화 키별로 일정하다. 구현체는 4 옥텟 카운터 필드를 가진 12 옥텟 논스를 지원해야 한다(SHOULD).

**카운터 필드 요구사항:**
- 연속 논스는 단조 증가 시퀀스를 형성한다
- 네트워크 바이트 순서의 부호 없는 정수로 취급한다
- 주어진 암호화 장치에 대해 길이가 일정하게 유지된다
- 0에서 시작하여 1씩 증가해야 한다(SHOULD)
- 값을 건너뛸 수 있다(MAY)
- C 옥텟일 때 최대 2^(8×C)개의 논스가 가능하다

**고정 필드 요구사항:**
- 주어진 암호화 장치에 대해 일정하게 유지된다
- 여러 장치가 단일 키를 공유할 때, 각 장치는 고유한 고정 필드를 사용해야 한다(MUST)
- 논스 고유성을 보장한다
- F 옥텟일 때 최대 2^(8×F)개의 고유 암호화기가 가능하다

### 3.2.1 부분적 암묵 논스

일부 애플리케이션은 전체 논스를 전송/저장하지 않고, 복호화 전에 문맥 정보로부터 재구성하는 것을 선호한다(예: 디스크 상의 암호문 위치).

**용어:**
- **명시적 부분(Explicit part)**: 암호문과 함께 저장/전송되는 논스 부분
- **암묵적 부분(Implicit part)**: 문맥으로부터 추론되는 논스 부분

**권장 특수화:** 고정 필드를 고정-공통(Fixed-Common, 모든 논스에 공통) 및 고정-고유(Fixed-Distinct, 장치별 고유) 하위 필드로 나눈다.

**요구사항:**
- 단일 키로 암호화를 수행하는 서로 다른 장치는 고유한 고정-고유 필드를 사용해야 한다(MUST)
- 고정-고유 필드와 카운터 필드는 명시적이어야 한다(MUST)
- 고정-공통 필드는 암묵적일 수 있다(MAY)

**근거:** GCM ESP(Encapsulating Security Payload)와 CCM ESP에서 사용되는 모범 사례를 통합한 것으로, 고정 필드에 Salt 값이 포함되고 논스의 최하위 8 옥텟이 ESP 패킷에 명시적으로 포함된다.

### 3.3 AEAD 입력의 구성

AD 입력이 여러 데이터 요소로 구성될 때, 인증되지 않은 데이터에 의존하지 않고 구성 요소로 명확하게 파싱 가능해야 한다. (수학적으로: AD는 데이터 요소의 단사 함수여야 한다.) 그렇지 않으면 공격자가 하나의 데이터 세트를 다른 것으로 대체할 수 있다. 고정 폭 요소에서는 이 요구사항이 자명하게 충족된다. 가변 길이 문자열은 AD에 길이를 포함해야 한다.

마찬가지로, 평문이 여러 데이터 요소로 구성될 때도 인증되지 않은 데이터에 의존하지 않고 명확하게 파싱 가능해야 한다. 참고: AD 데이터가 평문 파싱에 사용될 수 있지만(MAY), AD에 평문 정보가 포함되면 기밀성 손실 가능성이 있다.

### 3.4 사용 예제

애플리케이션은 암호화 알고리즘 입력을 애플리케이션 데이터 측면에서 정의하고, 암호문/논스를 전달해야 한다. 가장 명확한 접근법은 각 입력을 그것을 구성하는 데이터로 표현한 후, AEAD 암호화 연산의 출력 측면에서 애플리케이션 데이터를 표현하는 것이다.

**AES-GCM ESP 예제:**

```
P = RestOfPayloadData || TFCpadding || Padding || PadLength || NextHeader
N = Salt || IV
A = SPI || SequenceNumber
```

("||"는 연결을 의미하며, 필드는 RFC 4303/RFC 4106에서 가져온 것이다)

**ESP 패킷 형식:**
```
ESP = SPI || SequenceNumber || IV || C
```

(C는 인증 태그를 포함하는 AEAD 암호문이다)

## 4. AEAD 알고리즘 명세에 대한 요구사항

**키 요구사항:**
- 고정 길이 키만 허용해야 한다(MUST)
- 특정 키 데이터 형식을 요구해서는 안 된다(MUST NOT)
- 내부 구조(예: 패리티 검사 서브키)가 있는 알고리즘은 내부적으로 처리한다

**평문 요구사항:**
- 0에서 P_MAX 옥텟까지의 평문을 허용해야 한다(MUST) (P_MAX > 0, 65,536 이상이어야 한다(SHOULD))

**연관 데이터 요구사항:**
- 0에서 A_MAX 옥텟까지의 연관 데이터를 허용해야 한다(MUST) (A_MAX > 0, 65,536 이상이어야 한다(SHOULD))

**논스 요구사항:**
- N_MIN에서 N_MAX 옥텟까지의 논스를 허용해야 한다(MUST)
- N_MAX와 N_MIN이 같을 수 있다(MAY)
- 12 옥텟 논스를 허용해야 한다(SHOULD)
- 무작위화/상태 유지 알고리즘은 N_MAX가 0일 수 있다(MAY)

**암호문 구조:**
- 출력을 어떤 방식으로든 구조화할 수 있다(MAY) (예: 인증 태그 포함)
- 효율적인 처리에 적합한 구조를 선택해야 한다(SHOULD)

**무작위화 알고리즘:**
- 랜덤 소스를 포함할 수 있다(MAY) (예: 암호문 내 내부 초기화 벡터)
- 암호화만 무작위적이고, 복호화는 결정론적이다
- N_MAX가 0일 수 있다(MAY)

**상태 유지 알고리즘:**
- 암호화 호출 간에 내부 상태를 유지할 수 있다(MAY) (예: 내부 논스 생성)
- 영 길이 애플리케이션 논스로도 보안을 가능하게 한다
- N_MAX가 0일 수 있다(MAY)

**명세 요구사항:**
- K_LEN, P_MAX, A_MAX, N_MIN, N_MAX 값을 포함해야 한다(MUST)
- C_MAX(가능한 최대 암호문 옥텟)를 명시해야 한다(MUST)
- 평문-암호문 길이 관계를 기술해야 한다(MUST)
- 이 관계는 인증 태그 길이와 같은 외부 파라미터에 의존해서는 안 된다(MUST NOT) (상호 운용성 문제를 방지하기 위해)
- 논스 재사용으로 인한 보안 저하를 기술해야 한다(SHOULD)
- 상세한 보안 분석 참조를 제공해야 한다(SHOULD)
- 명세가 특정 보안 모델을 정의할 필요는 없다 (문헌에서 여러 모델을 사용)
- 무작위화/상태 유지 알고리즘은 해당 용어를 사용하여 스스로를 기술해야 한다(SHOULD)

---

## 5. AEAD 알고리즘

네 가지 알고리즘이 명시된다: AES GCM 두 가지(128비트 및 256비트 키)와 AES CCM 두 가지(128비트 및 256비트 키).

### 5.1 AEAD_AES_128_GCM

NIST SP 800-38D에 명시된 대로 동작하며, AES-128과 16 옥텟 인증 태그를 사용한다. 암호문은 GCM 암호화 연산 출력에 인증 태그를 추가하여 형성된다. 테스트 케이스는 SP 800-38D 부록에 있다.

**파라미터:**

| 파라미터 | 값 |
|---------|-----|
| K_LEN | 16 옥텟 |
| P_MAX | 2^36 - 31 옥텟 |
| A_MAX | 2^61 - 1 옥텟 |
| N_MIN, N_MAX | 모두 12 옥텟 |
| C_MAX | 2^36 - 15 옥텟 |
| 암호문 길이 | 평문보다 정확히 16 옥텟 길다 |

**보안:** McGrew & Viega(2004)에서 분석 가능.

#### 5.1.1 논스 재사용

동일한 키에서 같은 논스를 사용한 두 GCM 암호화가 서로 다른 평문에 대해 부주의하게 발생하면 다음을 훼손한다:

- **평문 기밀성**: 공격자가 두 평문의 XOR을 재구성할 수 있다
- **모든 진본성/무결성 보호**: 공격자가 내부 해시 키를 복구하여 자명한 위조가 가능해진다

GCM은 논스 고유성이 보장될 때만 사용해야 한다. GCM의 최소 지연 설계는 후속 사용 시 취약성을 야기한다. 동일한 논스를 사용한 다중 복호화는 허용된다.

**심각한 보안 결과:** 공격자가 동일한 논스/키로 생성된 두 암호문을 관찰하면(두 암호화의 평문과 AD 값이 동일한 경우를 제외하고):
1. **기밀성 상실**: 두 평문의 비트별 XOR 재구성
2. **무결성 상실**: 내부 해시 키 복구로 자명한 위조 가능

### 5.2 AEAD_AES_256_GCM

AEAD_AES_128_GCM과 동일하나 다음이 다르다:
- K_LEN: 32 옥텟
- AES-256 GCM 사용

### 5.3 AEAD_AES_128_CCM

NIST SP 800-38C에 명시된 대로 동작하며, AES-128과 다음 파라미터를 사용하여 명세 부록 A에 따른 형식화 및 카운터 생성을 수행한다:
- 논스 길이 n: 12
- 태그 길이 t: 16
- q 값: 3

16 옥텟 인증 태그를 사용한다. 암호문은 CCM 암호화 연산 출력에 인증 태그를 추가하여 형성된다.

**파라미터:**

| 파라미터 | 값 |
|---------|-----|
| K_LEN | 16 옥텟 |
| P_MAX | 2^24 - 1 옥텟 |
| A_MAX | 2^64 - 1 옥텟 |
| N_MIN, N_MAX | 모두 12 옥텟 |
| C_MAX | 2^24 + 15 옥텟 |
| 암호문 길이 | 평문보다 정확히 16 옥텟 길다 |

**보안:** Jonsson(2002)에서 분석 가능.

#### 5.3.1 논스 재사용

동일한 키에서 같은 논스를 사용한 두 CCM 암호화의 부주의한 재사용은 메시지 보안을 훼손한다. 기밀성 상실: 공격자가 두 평문 값의 비트별 XOR을 재구성할 수 있다.

### 5.4 AEAD_AES_256_CCM

AEAD_AES_128_CCM과 동일하나 다음이 다르다:
- K_LEN: 32 옥텟
- AES-256 CCM 사용

## 6. IANA 고려사항

IANA는 알고리즘 등록을 위한 "AEAD 레지스트리"를 수립했다. 레지스트리 추가에는 독립적인 구현 상호 운용성을 가능하게 하는 충분한 세부사항을 갖춘 RFC 또는 영구적이고 쉽게 접근 가능한 참조 문서가 필요하다.

**레지스트리 항목 요소:**
- "AEAD"로 시작하는 짧은 이름 (예: "AEAD_AES_128_GCM")
- 양의 정수
- 테스트 케이스를 포함하여 알고리즘을 완전히 정의하는 명세 참조

**등록 요구사항:**
- 이름과 참조를 포함해야 한다(MUST)
- IANA가 사용 가능한 가장 작은 양수 값을 사용하여 번호를 할당한다
- IRTF Crypto Forum Research Group(cfrg@ietf.org)의 검토가 이루어져야 한다(SHOULD)

**예약 번호:** 32,768~65,535는 사적 사용을 위해 예약되어 있다. IANA는 이 번호를 할당하지 않으며, 여러 사이트가 같은 값을 비호환적으로 사용할 수 있다(MAY).

**초기 레지스트리 항목:**

| 이름 | 참조 | 숫자 ID |
|------|------|---------|
| AEAD_AES_128_GCM | 섹션 5.1 | 1 |
| AEAD_AES_256_GCM | 섹션 5.2 | 2 |
| AEAD_AES_128_CCM | 섹션 5.3 | 3 |
| AEAD_AES_256_CCM | 섹션 5.4 | 4 |

IANA 등록은 알고리즘의 보증이나 보안 평가를 구성하지 않는다.

## 7. 기타 고려사항

**무작위화 알고리즘 테스트:** 비결정론적 암호화로 인해 직접적인 테스트 케이스 검증이 불가능하다. 테스트 접근법: 결과 암호문을 복호화하고 원래 평문과 복호화 후 평문을 비교한다. 결정론적 인증 복호화 테스트와 결합한다.

**알고리즘 선정 근거:** 선정된 알고리즘은 표준 채택을 반영한다. 어떤 다른 AEAD 알고리즘이 등록되어야 하는지에 대한 질문이 남아 있다. 많은 변형이 존재하며 각각 장점이 있다. 알고리즘의 유용성과 과도한 레지스트리 성장 방지 사이의 균형이 필요하다. 현재 명세는 완전한 명세와 검증 데이터를 요구한다.

**향후 가능한 추가:** 일반적인 암호화(예: CBC)와 일반적인 MAC(예: HMAC-SHA1 또는 AES CMAC)을 결합하는 일반적으로 구성된 encrypt-then-MAC AEAD 알고리즘이 모범 사례를 반영하고 다른 AEAD 알고리즘을 지원하지 않는 암호 모듈도 활용할 수 있게 한다.

---

## 8. 보안 고려사항

이 문서는 인증 암호화 알고리즘과 사용 지침을 기술한다. 이러한 알고리즘이 애플리케이션 설계를 용이하게 하지만, 강력한 암호 보안은 여전히 어렵다. AEAD 알고리즘은 키 생성이나 키 관리를 다루지 않는다.

**논스 의존 알고리즘:** 고유한 논스를 요구하는 AEAD 알고리즘은 일부 애플리케이션/시나리오에 적합하지 않다. 애플리케이션 설계자는 섹션 3.1의 요구사항을 이해해야 한다.

**가상 머신 롤백 위험:** VM 환경에서 소프트웨어 AEAD 암호화는 VM이 이전 상태로 "롤백"되면서 부주의하게 논스를 재사용할 수 있다. 애플리케이션은 잠재적 문제를 문서화하도록 권장된다. VM 롤백 공격자는 이미 시스템 보안을 훼손했을 수 있다(예: 회계 오류). VM 롤백 위협과 완화는 활발한 연구 분야이다.

**등록 면책:** IANA AEAD 알고리즘 등록은 보안 보증으로 간주되어서는 안 된다(MUST NOT). 알고리즘의 보안 인식은 암호 분석의 진보나 무어의 법칙(연산 자원 비용 감소)을 통해 시간이 지남에 따라 저하된다.
