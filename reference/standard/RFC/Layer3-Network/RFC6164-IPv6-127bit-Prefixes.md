# RFC 6164 - Using 127-Bit IPv6 Prefixes on Inter-Router Links

> 2011년 4월 발행, 라우터 간(point-to-point) 링크에 `/127` 프리픽스 사용을 권고
> RFC 3627(`/127` 사용을 유해하다고 본 문서)과 상충하는 새로운 권고를 제시하지만, 공식적인 Updates 관계는 아니다
> 이후 RFC 6547에 의해 갱신(Updated)됨

## 1. 개요

IPv6 주소 아키텍처(RFC 4291)는 인터페이스에 기본적으로 **`/64` 프리픽스**를 권고합니다.
 그러나 두 라우터를 잇는 **point-to-point 링크**에 `/64`를 그대로 쓰면 여러 운영·보안 문제가 발생합니다.

RFC 6164는 이러한 라우터 간 링크에 한해 **`/127` 프리픽스 사용을 권고**하여, 호스트가 붙지 않는 P2P 링크에서 발생하는 문제들을 회피합니다.

> "이 문서는 라우터 간 링크에서 127비트 IPv6 프리픽스 사용의 이점을 설명하고, 안전하게 적용하는 방법을 규정합니다."

## 2. `/64` 링크가 P2P에서 일으키는 문제 (Section 2)

### 2.1 Neighbor Discovery 캐시 고갈 공격 (NDP exhaustion)

> "`/64` 링크는 2^64개의 주소를 가진다.
 공격자가 존재하지 않는 주소로 트래픽을 보내면, 라우터는 각 목적지에 대해 Neighbor Solicitation을 보내고 미해결 항목을 NS 캐시에 쌓는다."

- 라우터의 Neighbor Cache(이웃 캐시)가 미완료 항목으로 가득 차서 **정상 통신을 방해**할 수 있음
- 일종의 DoS 공격 벡터

### 2.2 Ping-Pong 문제

> "프리픽스 길이가 인터페이스 주소를 모두 포함하지 못하는 P2P 링크에서, 링크 내 미지정 주소로 향한 패킷이 두 라우터 사이를 TTL이 소진될 때까지 왕복(ping-pong)할 수 있다."

- 목적지가 자신이 아니면 상대에게 되돌려보내는 동작이 반복됨
- `/127`을 쓰면 링크에 주소가 **정확히 2개**뿐이라 이 문제가 원천 차단됨 (RFC 4443에서도 ICMP로 완화하지만, /127이 근본 해결)

### 2.3 주소 낭비

- P2P 링크엔 주소 2개만 필요한데 `/64`는 2^64개를 할당하는 셈

## 3. `/127` 권고 (Section 5)

> "라우터 간 P2P 링크를 지원하는 라우터는 `/127` 프리픽스 할당을 지원해야 한다(MUST)." (RFC 6164 §6)

- `/127`은 호스트 비트가 1비트뿐이므로 링크에 주소가 **2개**(`...0`과 `...1`)만 존재
- 운영자는 라우팅 프로토콜·인프라 주소 계획에 따라 `/127`을 자유롭게 배정 가능

### 주소 배정 예시

```
프리픽스: 2001:db8:0:1::/127
  라우터 A: 2001:db8:0:1::0
  라우터 B: 2001:db8:0:1::1
```

## 4. Subnet-Router Anycast 충돌 회피 (Section 5.1)

`/127`을 쓸 때 가장 주의할 점은 **Subnet-Router Anycast 주소**(RFC 4291)와의 충돌입니다.

> "Subnet-Router Anycast 주소는 서브넷 프리픽스 + 인터페이스 ID 전체가 0인 주소다. `/127` 링크에서는 두 주소 중 하나(`...0`)가 이 anycast 주소와 동일해진다."

### 해결 방식

- `/127` 링크에서는 **두 주소를 모두 일반 인터페이스 주소로 사용**하며, Subnet-Router Anycast 동작에 의존하지 않음
- 두 라우터 모두 `...0` 주소로 향하는 트래픽을 자신에게 온 것으로 처리(Anycast 의미를 사실상 무시)
- 따라서 P2P 링크에서는 Anycast 충돌이 실질적 문제가 되지 않음

참고: 라우터의 `/127` 지원 자체는 RECOMMENDED가 아니라 MUST 수준으로 규정되어 있다(§6).

### RFC 2526 예약 Anycast 주소 주의

> "RFC 2526이 예약한 서브넷 내 anycast 식별자(인터페이스 ID의 상위 영역)와 겹치는 주소 사용은 피해야 한다."

- `/127`은 인터페이스 ID가 1비트뿐이라 이 예약 영역과 겹치지 않으므로 안전
- (더 넓은 프리픽스를 P2P에 쓸 때는 예약 anycast 영역과 충돌하지 않도록 주의)

## 5. 다른 프리픽스 길이와의 관계 (Section 5.2)

- `/127`이 권고되지만, 운영 정책에 따라 다른 길이를 쓸 수도 있음
- 일부 운영자는 주소 계획·문서화 편의를 위해 **`/64`를 할당하되 실제로는 `/127`만 구성**하기도 함 (예약은 /64, 설정은 /127)
- 어떤 길이를 쓰든 위의 Anycast·예약 주소 충돌 주의는 동일하게 적용

## 6. EUI-64 / 자동 구성과의 관계

- `/127`은 호스트 비트가 1개뿐이므로 **EUI-64 기반 인터페이스 ID 자동 생성이 불가능** → 주소는 **수동(static) 설정**
- P2P 라우터 링크는 본래 SLAAC(상태 비저장 자동 구성)가 필요 없는 환경이므로 문제되지 않음

## 7. RFC 3627과의 관계

| 항목 | RFC 3627 (2003) | RFC 6164 (2011) |
|------|------------------|------------------|
| `/127` 사용 | **유해(harmful)하므로 피하라** | **P2P 라우터 링크에 권고** |
| 근거 | Subnet-Router Anycast 충돌 우려 | P2P에서는 충돌이 실질 문제 아님 + 보안 이점 |

> RFC 6164는 RFC 3627의 권고를 뒤집어, **P2P 라우터 링크에 한해 `/127`을 권고**합니다(다만 RFC 3627에 대한 공식 Updates 관계는 없습니다).
 이후 RFC 6547이 RFC 3627을 정식 폐기(Historic)로 만들었습니다.

## 8. 보안 고려사항 (Section 7)

- **NDP 캐시 고갈 공격 완화**: 링크에 주소가 2개뿐이라 존재하지 않는 주소로의 스캐닝/플러딩이 무의미 → 공격 표면 대폭 축소
- **Ping-Pong 차단**: 미지정 주소 자체가 없으므로 패킷 왕복 불가
- 결과적으로 `/127`은 보안 측면에서 `/64` P2P 링크보다 **더 견고**함

## 9. 요약

RFC 6164는 **호스트가 붙지 않는 라우터 간 P2P 링크**에 `/127` 프리픽스 사용을 권고합니다.

- `/64` P2P 링크의 **NDP 캐시 고갈 공격**과 **Ping-Pong 문제**를 근본적으로 해결
- 링크에 주소가 **정확히 2개**(`...0`, `...1`)만 존재
- **Subnet-Router Anycast**는 P2P 링크에서 의미가 없으므로 두 주소 모두 일반 주소로 사용
- RFC 2526 **예약 anycast 영역**과 겹치지 않아 안전
- 주소는 **수동 설정**(EUI-64 자동 생성 불가)
- 과거 `/127`을 유해하다고 본 **RFC 3627의 권고를 뒤집음** (공식 Updates 관계는 아님)
- RFC 6164 자체는 이후 **RFC 6547에 의해 갱신**(Updated)됨

호스트 링크에는 여전히 `/64`(RFC 4291)를, **라우터 간 P2P 링크에는**`/127`을 쓰는 것이 모범 사례입니다.
