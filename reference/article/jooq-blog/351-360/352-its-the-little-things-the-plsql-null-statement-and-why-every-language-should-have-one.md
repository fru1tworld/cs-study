# 작은 것들의 중요성: PL/SQL NULL 문, 그리고 모든 언어가 가져야 하는 이유

> 원문: https://blog.jooq.org/its-the-little-things-the-plsql-null-statement-and-why-every-language-should-have-one/

게시일: 2015년 4월 13일 (2022년 8월 16일 업데이트)

저자: lukaseder

---

구문은 그런 주제들 중 하나입니다. 매우 매우 매우 중요한 논쟁으로 이어지는 감정적인 주제 중 하나죠. 저는 개인적으로 PL/SQL을 좋아합니다. 극도로 장황하고 정밀합니다. 매우 강력하고 엄격한 타입 시스템을 따르도록 강제하여 여러분의 속도를 늦추는데, 이것이 실수를 피하는 데 도움이 될 가능성이 높습니다.

PL/SQL에서 제가 특히 좋아하는 것이 하나 있습니다. 빈 블록이라는 것이 존재하지 않는다는 것입니다.

Java에서는 다음과 같이 작성할 수 있습니다:

```java
// 그냥 빈 블록:
{}

// 레이블이 있는 빈 블록:
label1: {}

// 또는 사실상 빈 문장:
;
label2: ;
```

위의 것들이 순수하게 구문적 관점에서 문제가 되는 이유는 빈 블록이 의도치 않게 비어 있는 상태로 남겨졌을 수 있기 때문입니다. 빈 문장은 전혀 눈에 보이지 않을 수도 있습니다. if 문의 맥락에서 이것을 고려해 보세요:

```java
if (something) {

}

if (somethingElse) ;
```

PL/SQL에서는 이것이 불가능합니다. 빈 블록이라는 것이 존재하지 않습니다. 다음은 컴파일되지 않습니다:

```sql
BEGIN
END;
```

안 됩니다:

```
ORA-06550: line 2, column 1:
PLS-00103: Encountered the symbol "END" when expecting ...
```

또는 `IF` 문을 예로 들어봅시다. 여기서도 빈 것은 허용되지 않습니다:

```sql
IF 1 = 1 THEN
END IF;
```

작동하지 않습니다.

만약 빈 블록을 만들고 _싶다면_ (예: 나중에 작성할 코드의 자리 표시자로), 명시적으로 더미 문장을 넣어야 합니다. PL/SQL에는 그런 문장이 있습니다. `NULL` 문입니다:

```sql
BEGIN
  NULL;
END;

IF 1 = 1 THEN
  NULL;
END IF;
```

이것이 합리적입니다. 여러분은 즉시 알 수 있습니다: 좋아, 여기서는 아무 일도 일어나지 않는구나.

## 결론

장황함은 코드의 버그 수를 줄이는 데 도움이 됩니다. 사람들이 구문에 대해 덜 신경 쓸수록 (다시 말하지만, 매우 매우 매우 중요한 주제에 대한 논의를 참조하세요), 그들이 정말로 작성하려는 것에 더 집중할 수 있습니다. 우리의 자존심을 삼킵시다. 언어에 익숙해지면, 우리는 _어떤_ 언어든 받아들이게 됩니다. 그것들은 모두 결함이 있고 기이합니다. 그것은 중요하지 않습니다. 하지만 적어도, 언어는 스타일을 다르게 하는 방법에 대해 논쟁하는 것을 막아줘야 합니다.
