# "NoSQL"은 "대안적 저장 모델을 사용하는 SQL"이라고 불려야 한다

> 원문: https://blog.jooq.org/nosql-and-what-the-hype-really-went-wrong-about/

때때로 여러분은 SQL 대 NoSQL에 대한 동일한 "진실"을 말하는 이런 블로그 게시물들을 발견하게 될 것입니다:

http://onewebsql.com/blog/no-sql-do-i-really-need-it

(OneWebSQL은 jOOQ의 경쟁자입니다. 비교에 관한 이전 글을 참조하세요)

보통 이러한 블로그들은 같은 주장을 목표로 합니다:

- 성능 ("SQL"은 "NoSQL"만큼 확장될 수 "절대" "없다")
- ACID (항상 필요한 것은 아니다)
- 스키마 없음 (그냥 어떤 데이터든 저장하면 된다)

재미있는 이유로 인해, 이 모든 아이디어들은 "NoSQL"이라는 오해의 소지가 있는 용어로 이어졌는데, 이것은 어떤 사람들에게는 "SQL이 아님(no SQL)"으로, 다른 사람들에게는 "SQL만이 아님(not only SQL)"으로 해석됩니다. 그러나 SQL은 실제로 "Structured Query Language(구조화된 질의 언어)"를 의미하며, 관계형 컨텍스트를 표현하는 데 있어 매우 강력합니다. SQL은 튜플, 레코드, 테이블, 집합의 임시 생성과 이를 다른 프로젝션으로 매핑하고, 사용자 정의 집계로 축소하는 등의 작업에 잘 설계되어 있습니다. NoSQL 전도사들이 자주 사용하는 "맵/리듀스(map/reduce)"라는 용어에 주목하세요.

충분한 이유로 인해, 선도적인 NoSQL 질의 언어 중 하나인 Facebook Query Language(FQL)는 완전히 다른 데이터 모델에서 작동하지만 SQL과 매우 유사합니다. Oracle 역시 "NoSQL" 열풍에 동참하여 자체 제품을 판매하고 있습니다. 두 가지 유형의 데이터 저장소가 병합되어 ISO/IEEE 표준화된 SQL:2015(또는 그 즈음)로 질의할 수 있게 되기까지 그리 오래 걸리지 않을 것입니다. 왜냐하면 "NoSQL"의 진정한 정신은 데이터가 질의되는 방식에 있지 않기 때문입니다. 그것은 데이터가 저장되는 방식에 있습니다. NoSQL은 전적으로 데이터 저장에 관한 것입니다. 따라서 조만간 여러분은 동일한 데이터베이스에서 "전통적인" 테이블과 함께 "그래프 테이블"과 "해시맵 테이블"을 생성하고, 오늘날의 과대광고에 대해 많이 생각하지 않고도 단일 SQL 질의로 이들을 조인하게 될 것입니다.

"NoSQL"은 "대안적 저장 모델을 사용하는 SQL"이라고 불려야 하며, 순수 SQL로 질의되어야 합니다!
