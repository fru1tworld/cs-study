# jOOQ 화요일: Yalim Gerger가 Git을 Oracle에 가져오다

> 원문: https://blog.jooq.org/jooq-tuesdays-yalim-gerger-brings-git-to-oracle/

이번 블로그 포스트는 Gerger의 창업자이자 Formspider와 Gitora의 개발자인 Yalim Gerger와의 인터뷰를 다룹니다.

질문: 안녕하세요 Yalim - 당신은 Formspider를 만든 회사 Gerger의 창업자입니다. Formspider가 무엇인가요?

답변: Formspider는 Oracle PL/SQL 개발자를 위한 애플리케이션 개발 프레임워크입니다. PL/SQL 개발자들이 PL/SQL만을 프로그래밍 언어로 사용하여 고품질 비즈니스 애플리케이션을 구축할 수 있게 해줍니다. Formspider를 사용하는 데 Java나 JavaScript 기술이 필요하지 않습니다.

질문: 흥미롭군요, Java 개발자의 관점에서 봐도요! 본질적으로, 당신은 Java를 미들웨어로 사용하는 것을 완전히 우회하는 방법을 제공하고 있군요(물론 HTML, JavaScript, CSS도요).

답변: 이것은 완전히 사실은 아닙니다. 우리는 여전히 제품에서 Java를 사용합니다. Formspider에는 고객이 JEE 호환 애플리케이션 서버에 배포할 수 있는 미들티어 애플리케이션이 있습니다. 이 미들티어 애플리케이션은 클라이언트의 브라우저에서 실행되는 Formspider JavaScript 라이브러리를 데이터베이스에서 실행되는 PL/SQL에 연결하는 데 도움을 줍니다. 우리는 또한 애플리케이션에 저장된 데이터로부터 Excel 파일을 생성하기 위해 Java 라이브러리를 사용하는데, 이는 비즈니스 애플리케이션에서 흔한 사용 사례입니다. 그래서 네, 애플리케이션은 Java로 코딩되지 않습니다. 우리 고객들은 PL/SQL 개발자입니다. 하지만 우리는 제품을 개선하기 위해 Java를 사용합니다. HTML과 Javascript도 마찬가지입니다. 우리의 역할은 이러한 기술들과 그 기능들을 정말 잘 이해하고, 이를 PL/SQL 개발자들에게 직관적인 API로 제공하는 것입니다.

질문: Formspider를 통해서도 PL/SQL API에 접근하고, 자체 개발한 Java / .NET 애플리케이션을 통해서도 접근하는 고객들이 있나요?

답변: 네. Formspider로 작업하는 PL/SQL 팀과 Java 기술로 작업하는 Java 팀 둘 다 있는 고객들이 있습니다. 이것은 두 팀 간의 많은 협업을 필요로 하며, 항상 가능한 것은 아닙니다.

보통 일어나는 일은 Java/.NET 팀들이 애플리케이션을 가능한 한 데이터베이스에서 멀리 옮기려고 한다는 것입니다. 저는 방금 대형 금융 기관에서 일하는 친구와 이야기를 나눴는데, 그곳에서 객체지향 진영 사람들이 모든 PL/SQL API를 제거하려고 강하게 밀어붙이고 있었습니다. 그는 미쳐버릴 것 같았습니다. 여기에는 여러 가지 이유가 있습니다. 부분적으로는 정치적인 영역 다툼이고, 부분적으로는 데이터베이스 소프트웨어와 PL/SQL의 기능에 대한 순수한 무지입니다.

질문: 그 고통이 느껴집니다. 만능 해결책 같은 것은 없죠... 그러면 그들은 어떻게 해야 할까요? 애플리케이션은 어떻게 설계되어야 할까요? "올바른" 아키텍처가 있다고 생각하시나요?

답변: 아니요. 컨텍스트에 따라 너무 많이 달라진다고 생각합니다. 소비자 앱을 만들고 있나요, 아니면 비즈니스 앱을 만들고 있나요? 수평적 제품을 만드는 회사인가요, 아니면 특정 수직 산업에서 운영되는 비즈니스를 지원하는 IT 부서인가요? 고려해야 할 매개변수가 너무 많습니다. 너무 일반적인 표현이 될 위험을 감수하고 말하자면, 대기업을 지원하는 IT 부서는 데이터베이스에 구애받지 않는(database agnostic) 애플리케이션을 구축하려고 하면 안 됩니다. 그것은 어리석은 일입니다. 오히려 데이터베이스 소프트웨어와 사용하는 다른 소프트웨어를 최대한 활용해야 합니다. 애플리케이션의 일곱 개의 다른 레이어를 구축하는 일곱 개의 다른 조직이 우연히 협업하게 된 것처럼 행동하면 안 됩니다. 당신은 단지 하나의 조직입니다. 그렇게 행동하세요. 군더더기를 잘라내세요. 가능한 한 깊이 통합하세요. 이것이 제때에, 예산 내에서 잘 수행되는 애플리케이션을 구축하는 가장 비용 효율적인 방법입니다. 데이터베이스에 구애받지 않는 소프트웨어는 수평적 소프트웨어 회사를 위한 것입니다.

질문: 우리는 최근 듀얼 라이선싱의 주의사항에 대해 블로그에 글을 올렸는데, Data Geekery처럼 스스로를 오픈 소스 회사라고 부르는 회사에게는 유료 고객에게 소스 코드를 제공하는 것이 필수적이라고 말했습니다. 당신도 소스 코드를 제공하는 것을 봤는데 - 하지만 정확히 "오픈 소스"를 하는 것은 아니군요. 당신의 제공 방식을 어떻게 설명하시겠습니까?

답변: 그 블로그 포스트 정말 좋았습니다. jOOQ의 라이선스 방식은 훌륭하게 공정하다고 생각합니다. 즉, 데이터베이스가 무료면 무료이고, 데이터베이스에 라이선스 비용이 있으면 가격표가 있습니다. 우리의 경우, 데이터베이스에는 항상 라이선스 비용이 있습니다. 그래서 Formspider에 대한 무료 옵션은 없습니다. Oracle 커뮤니티와 그들이 익숙한 가격표에 비하면, 우리의 라이선스 비용은 너무 작아서 사실상 무료나 다름없습니다. 우리 가격표에 거부감을 느끼는 사람은 아마도 Formspider를 사용하는 것에 대해 진지하지 않은 것일 겁니다.

최고 수준의 지원 서비스에 가입한 고객은 전문 서비스 기간 동안 우리 제품의 소스 코드를 받을 수 있습니다. 이 옵션은 Formspider로 구축하는 애플리케이션에 많이 투자하는 고객에게 매력적입니다.

질문: 네, Oracle 가격표는 명성이 자자하죠... Yalim, 당신은 Oracle 전문가인 것 같습니다. 그리고 그런 사람으로서, 당신은 Gitora를 출시하려고 합니다. 그것이 무엇인가요?

답변: Gitora는 Oracle 데이터베이스를 Git과 통합하는 무료 버전 관리 도구입니다. 주로 Java 개발자들이 읽는 블로그에서 이것을 언급하는 것이 조금 부끄럽지만, 대부분의 Java 개발자들이 당연하게 여기는 매우 일반적인 버전 관리 작업들이 PL/SQL에서는 매우 어렵습니다. 여기에는 좋은 이유가 있습니다. PL/SQL에는 작업 디렉토리 개념이 없습니다. PL/SQL은 파일 기반 언어가 아닙니다. 즉, 소스 코드 단위가 개발자의 개인 파일 시스템이 아니라 Oracle 데이터베이스 내에 패키지, 프로시저, 함수로 존재하며, 모든 개발자가 전역적으로 접근할 수 있습니다. 이것이 버전 관리를 불가능하지는 않더라도 매우 어렵게 만듭니다.

질문: 그러면 사람들은 어떻게 하나요?

답변: 대부분 아무것도 안 합니다. 일일 백업이 필요할 때 코드의 이전 상태로 돌아가는 방법으로 사용됩니다.

일부 팀은 버전 관리에 연결된 하나의 작업 디렉토리를 만들고, 보통 수동으로 DDL을 추출하여 모든 PL/SQL 코드를 이 디렉토리에 저장합니다. 그것이 가장 정교한 수준입니다.

PL/SQL에서의 적절한 팀 개발과 병합은 매우 어렵고, 성공적으로 수행되는 것을 자주 보지 못했습니다. 그리고 저는 전 세계의 많은 PL/SQL 팀들과 교류해왔습니다. Gitora는 이것을 매우 쉽게 만듭니다. 데이터베이스 스키마를 작업 디렉토리로 변환합니다. Git 명령을 실행하면, 작업 디렉토리의 모든 변경 사항이 자동으로 데이터베이스 스키마에 반영됩니다.

질문: 흥미롭군요. 우리는 최근 한 고객을 위해 Microsoft Team Foundation Server 저장소에서 자동 버전 관리 및 설치를 구현하는 자체 개발 "솔루션"을 구현했습니다. 어쩌면, 우리가 Gitora로 마이그레이션해야 할까요?

답변: 그건 몰랐네요. 정말 멋지네요. PL/SQL에서 작동하고 Git 대신 TFS와 통신하는 버전 관리 도구를 만들었다면, 그것도 매우 가치 있다고 생각합니다. 본질적으로 우리는 같은 제품을 만들었지만 다른 버전 관리 제품을 사용했습니다. 그것을 공개하시길 권장합니다.

질문: 왜 안 되겠어요. 어쩌면 우리가 기여할 수도 있겠네요!

이 매우 흥미로운 통찰에 감사드립니다, Yalim!
