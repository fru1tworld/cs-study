# UI 개발자여! 합리적인 기본 정렬을 선택하라!

> 원문: https://blog.jooq.org/ui-developers-choose-sensible-default-ordering/

UI 개발자들은 테이블 열에 대한 기본 정렬 순서를 선택할 때 기술적 기본값(오름차순)이 아닌 컨텍스트에 기반하여 신중하게 선택해야 합니다.

## 문제 상황 설명

Bing 웹마스터 도구를 예로 들어보겠습니다. 트래픽 데이터를 볼 때 "검색에 노출됨(Appeared in search)" 열의 기본 내림차순 정렬은 합리적입니다. 사용자들은 최고 성과를 보인 항목을 보고 싶어하기 때문입니다. 그러나 "검색에서의 클릭(Clicks from search)"을 클릭하여 상위 트래픽을 확인하려면, 직관적이지 않은 오름차순 정렬을 뒤집기 위해 두 번째 클릭이 필요하게 됩니다. 이는 사용자에게 불필요한 좌절감을 줍니다.

## 핵심 통찰

"좋은 결정은 경험에서 나오고, 경험은 나쁜 결정을 내리는 것에서 나온다"라는 말이 여기에 적용됩니다. 정렬이 예상치 못한 결과를 생성하면 사용자들은 인지 부조화를 경험하게 되고, 실제 목표를 달성하기 위해 추가 클릭이 필요하게 됩니다.

[정렬에 대한 혼란]과 [실제 원하는 결과] 사이의 인지 부조화를 과소평가해서는 안 됩니다.

## 권장되는 기본 정렬 방식

### 오름차순이 적합한 경우:
- 이름 (알파벳순)
- 전화번호
- 행 번호
- 미래 날짜 (앞으로의 일정, 예약 등)
- 특정 집계값 (MIN 값, 백분위수)
- 기간 구성원 (일, 시간, 분 등)
- 가격

### 내림차순이 적합한 경우:
- 과거 날짜 (거래 내역, 최근 이벤트 등 - 가장 최근 것이 먼저)
- 집계값 (COUNT(*), SUM(), AVG(), MAX() 등)
- 파일 크기
- 불리언 값 (true가 false보다 먼저)
- 계좌 잔액
- 변화 지표 (상위 성과자를 먼저 보여주기)
- 성능 지표

### 정렬하지 않는 것이 좋은 경우:
- URL (의미 있는 순서가 없음)
- 일반 텍스트/CLOB (비실용적)
- 복합 데이터 (너무 복잡함)

## 핵심 요점

백엔드 개발자들은 데이터베이스가 컨텍스트가 없는 원시 데이터(예: NUMBER 또는 VARCHAR 타입)를 포함하고 있기 때문에 UI 정렬 순서를 결정할 수 없습니다. UI 정렬은 반드시 SQL 정렬과 같지 않습니다.

UI 디자이너들은 기술적 기본값이 아닌 사용자 기대와 도메인 로직에 기반하여 이러한 결정을 내려야 합니다. 데이터 타입은 데이터베이스가 결정할 수 없는 맥락적 의미를 담고 있으며, 이것이 바로 직관적인 사용자 경험을 위해 UI 계층의 책임이 필수적인 이유입니다.

기술적인 "자연 정렬(natural ordering)"과 사용자 경험 중심의 정렬 사이의 구분이 인터페이스가 직관적으로 느껴지는지 아니면 답답하게 느껴지는지를 결정합니다. UI 개발자들은 "자연 정렬"을 일급 설계 고려사항으로 다루어야 합니다.
