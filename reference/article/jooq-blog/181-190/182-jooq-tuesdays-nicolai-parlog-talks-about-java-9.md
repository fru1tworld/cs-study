# jOOQ 화요일: Nicolai Parlog가 Java 9에 대해 이야기하다

> 원문: https://blog.jooq.org/jooq-tuesdays-nicolai-parlog-talks-about-java-9/

jOOQ 화요일 시리즈에 오신 것을 환영합니다. 이 시리즈에서는 격월로 셋째 주 화요일에 SQL, Java, 오픈 소스와 같은 주제를 다루는 소프트웨어 개발 분야의 주목할 만한 인물들과의 인터뷰를 게시합니다.

오늘의 인터뷰에서는 "The Java Module System"의 저자인 Nicolai Parlog와 Java 9의 기능들과 Jigsaw 모듈 시스템에 대해 이야기를 나눕니다.

## 블로그에서 Java 전문가 그룹 토론을 연구하고 문서화하게 된 계기는 무엇인가요?

Nicolai는 Optional의 설계에 대한 호기심에서 연구를 시작했다고 설명했습니다. 그는 열악한 아카이브 인터페이스와 단편화된 대화로 인해 메일링 리스트 토론을 찾고 이해하는 것이 어렵다는 것을 발견했습니다. Brian Goetz가 관찰한 바와 같이 "Optional과 관련된 모든 것은 300개의 메시지가 필요한 것 같다"고 그는 언급했습니다. 연구 내용을 블로그 포스트로 정리한 후, Nicolai는 이러한 유형의 역사적 분석에 대한 커뮤니티의 관심을 인식했고, 진행 중인 JDK 토론을 정기적으로 요약하는 서비스 기회가 있을 수 있다고 제안했습니다.

메일링 리스트 토론은 종종 길고, 단편적이며, 따라서 다시 살펴보기가 어렵습니다. JDK 아카이브(Piperman)는 탐색하고 검색하기에 끔찍한 곳입니다. Optional에 대한 하나의 포스트를 연구하는 데 약 일주일이 걸렸습니다. 이러한 경험은 Java 커뮤니티가 전문가 그룹의 결정 과정을 이해하는 데 도움이 되는 요약의 필요성을 보여주었습니다.

## Jigsaw 외에 가장 흥미로운 Java 9 기능은 무엇인가요?

언어 기능 중에서 Nicolai는 "private 인터페이스 메서드"를 강조했습니다. 이 기능은 구현 세부 사항을 공개적으로 노출하지 않고 default 메서드 간에 코드를 공유할 수 있게 해줍니다.

API와 관련해서는 컬렉션 팩토리 메서드보다 Stream과 Optional의 개선 사항을 선호한다고 말했습니다. 그는 Java 8의 이러한 기능들을 즐겨 사용하며 그것들의 개선을 높이 평가했습니다.

JVM 기능으로는 "multi-release JAR"를 칭찬했는데, 이는 여러 Java 버전에서 우아한 API 저하(graceful degradation)를 지원합니다. 추가로 좋아하는 기능들로는 JVM 메시지 추출을 단순화하는 통합 로깅(unified logging), 메모리 소비를 10-15% 줄이는 컴팩트 문자열(compact strings), 그리고 성능을 향상시키는 인디파이드 문자열 연결(indified string concatenation)이 있습니다.

## 곧 출간될 모듈 시스템 책에서 독자들은 무엇을 얻을 수 있나요?

이 책은 기본적인 모듈 메커니즘을 넘어서는 포괄적인 내용을 제공합니다. 기능을 어떻게 사용하는지뿐만 아니라 언제, 왜 적용해야 하는지를 설명합니다. 상당 부분의 장에서는 Nicolai가 대규모 애플리케이션 마이그레이션을 수행한 경험을 바탕으로 마이그레이션 과제를 다룹니다. 추가 섹션에서는 JDeps와 같은 도구를 사용한 디버깅 기술과 일반적인 오류 메시지 처리 방법을 다룹니다.

이 책에는 메일링 리스트에서의 토론 내용도 포함되어 있어, 독자들이 모듈 시스템 설계 결정의 이유를 이해하는 데 도움이 됩니다.

## Jigsaw의 장점과 단점은 무엇인가요?

Nicolai는 강력한 캡슐화(strong encapsulation)를 가장 뛰어난 장점으로 꼽았습니다. 이를 통해 타입이 모듈 내에서만 public으로 유지될 수 있습니다. 이것은 전통적인 private-to-public 가시성 외에 추가적인 제어 계층을 만들어줍니다.

우려 사항과 관련하여 그는 기존 JAR가 모듈로 전환될 때 호환성 문제를 걱정했습니다. 특히 의존성 제외 패턴과 관련하여 그렇습니다. 그는 "transitive 의존성을 열심히 제외하는 POM들"은 JAR가 모든 의존성이 존재해야 하는 모듈이 되면 제대로 작동하지 않을 것이라고 언급했습니다.

모듈 시스템의 엄격함이 이전 Java 버전보다 메인테이너의 결정을 우회하기 더 어렵게 만들 수 있지만, 일부 해결 방법을 위한 명령줄 플래그가 존재한다고 그는 지적했습니다.

채택과 관련하여 Nicolai는 Java 8보다 더 느린 도입을 예측했지만, 개발자들이 새로운 기능과 값 타입(value types)과 같은 향후 개선 사항을 추구함에 따라 결국 마이그레이션이 이루어질 것으로 예상했습니다.

## Java 9 이후에 어떤 Java 프로젝트를 추진할 예정인가요?

Nicolai는 책 완성과 추가 마이그레이션을 통해 Java 9에 지속적으로 집중할 것이라고 밝혔습니다. 그는 마이그레이션을 위해 고용될 수 있다고 언급했습니다.

그는 Project Valhalla 이니셔티브(프리미티브 특수화와 값 타입)와 Project Amber 변경 사항을 기대하며, 2018년에 이러한 주제들을 다룰 것으로 예상한다고 말했습니다. 또한 그는 성장 중인 프로젝트로 새롭게 시작한 YouTube 채널을 홍보했습니다.
