# 오픈 소스에 더 많은 지원이 필요한 게 아니라 더 나은 비즈니스 모델이 필요하다

> 원문: https://blog.jooq.org/open-source-doesnt-need-more-support-it-needs-better-business-models/

Typesafe의 글로벌 서비스 디렉터인 Jamie Allen이 트위터에 흥미로운 관점을 게시했습니다:

> "Pivotal이 Groovy 지원을 종료하기로 한 결정은 FOSS 프로젝트에 의존하는 기업들이 해당 프로젝트를 지원해야 한다는 것을 상기시켜주는 냉엄한 현실입니다."

물론 그의 말이 맞습니다. 우리는 우리가 의존하는 FOSS 프로젝트를 지원해야 한다는 사실을 끊임없이 상기받고 있습니다.

하지만... 그게 정말 사실일까요? 정말로 오픈 소스에 더 많은 지원이 필요한 걸까요? 더 많은 기업들이 오픈 소스에 의존하니까, 그들이 더 많이 지원해야 하는 걸까요?

저는 다르게 생각합니다.

## Red Hat 예외 현상

Andreessen Horowitz의 Peter Levine은 TechCrunch에 ["Why There Will Never Be Another RedHat: The Economics Of Open Source(왜 또 다른 Red Hat은 나오지 않을 것인가: 오픈 소스의 경제학)"](http://techcrunch.com/2014/02/13/open-source-age-of-disruption/)라는 매우 흥미로운 글을 게시했습니다. 이 글에서 그는 Red Hat의 시장 가치와 수익이 Microsoft, Oracle, 심지어 Amazon에 비해 여전히 상대적으로 작다는 점을 보여줍니다. 가장 성공적인 오픈 소스 기업임에도 불구하고 말입니다.

왜 그럴까요? Red Hat이 성공한 이유는 Linux가 프로덕션 환경에서 운영할 때 지원 구독이 반드시 필요한 매우 복잡한 생태계이기 때문입니다. 금융 기관들은 24시간 지원과 1시간 이내의 응답 시간을 요구합니다. 이러한 운영상의 필요성이 구독 수익을 창출합니다. 이 지원 모델이 비즈니스의 실행 가능성을 뒷받침하는 것입니다.

## 운영 vs 개발 경제학

핵심적인 구분은 실제 벤더 수익이 어디서 발생하는가에 있습니다. 개발팀이 프로그래밍 언어를 선택하지만, 운영 부서가 프로덕션 시스템을 관리하고 지원 계약에 대한 예산 배분을 통제합니다.

중요한 작업은 이미 개발자들에 의해 오래전에 완료되었습니다. 바이너리는 빌드되어 프로덕션으로 배포되고, 그곳에서 운영팀이 인수합니다.

운영팀은 오로지 시스템의 안정성에만 관심을 가지며, 구현 언어에는 신경 쓰지 않습니다. 코드가 Groovy, Scala, Java, Ceylon, Kotlin, Fantom 중 무엇으로 실행되든 상관없습니다. 오직 JVM의 안정성과 데이터베이스 성능만이 운영팀의 관심사입니다. 지원 계약은 운영상의 의존성에서 비롯되지, 개발자의 선호도에서 비롯되지 않습니다.

## 언어 채택과 생태계 의존성

Groovy가 이 문제의 좋은 예입니다. 수만 명의 개발자가 벤더와의 상호작용 없이 Groovy를 다운로드했고, 많은 이들이 Pivotal의 후원 사실조차 알지 못했습니다. Groovy는 "함께 일하기 아름다운 언어"이지만, 강력한 브랜드가 되지는 못했습니다. 대신 "모두의 브랜드, 따라서 누구의 브랜드도 아닌 것"이 되어버렸습니다.

Pivotal은 "완전히 이해할 수 있는 이유로" 지원을 철회했습니다. Apache나 Eclipse를 모델로 한 Groovy 재단이 벤더 의존성을 줄일 수 있었을지도 모릅니다. Gradleware가 잠재적으로 Groovy를 인수하여 확실한 Groovy 회사로 자리매김할 수도 있었겠지만, 성공 여부는 불확실합니다.

## 플랫폼 vs 언어 수익 모델

성공적인 수익화에는 여러 제품을 업셀링 체인으로 통합하는 "전체론적 플랫폼 전략"이 필요합니다:

좋은 예시:
- C# → Visual Studio → SQL Server → Azure
- Java → JVM/Weblogic → Oracle Database → Oracle Commerce

그저 그런 예시:
- Kotlin → IntelliJ

문제가 있는 예시:
- Groovy → Cloud Platform ?? (불명확한 가치 제안)
- Scala → Reactive Programming ?? (약한 연결고리)
- Ceylon → RHEL ?? (언어 종속에 대한 정당화 없음)

하위 업셀링 기회가 부족한 언어들은 어려움을 겪습니다. "언어 종속에 대한 정당화가 없기" 때문입니다.

## 성공적인 오픈 소스 수익화

JetBrains, ZeroTurnaround, 그리고 저희 Data Geekery 같은 회사들은 더 작은 규모에서 실행 가능한 대안을 보여줍니다. 이 회사들은 커뮤니티 지원에 대한 낭만적인 호소에 의존하지 않고 지속 가능한 수익 모델을 개발했습니다.

저자는 이렇게 말합니다: "JetBrains, ZeroTurnaround, 또는 저희 Data Geekery 같은 회사들은 훨씬 더 작은 규모에서 그것이 가능하다는 것을 보여줍니다." 그러나 수익성을 위해서는 프로그래밍 우수성을 넘어선 정교한 비즈니스 전략이 필요합니다.

## 현재 접근 방식의 근본적인 문제

"FOSS 프로젝트에 의존하는 기업들이 해당 프로젝트를 지원해야 한다"는 Jamie Allen의 트윗은 실제 의존 관계를 뒤바꿔 놓은 것입니다. 최종 사용자들은 버려진 프로젝트를 포크할 수 있지만, 벤더들은 전적으로 커뮤니티의 선의에 의존합니다. 그러면 벤더들은 "완전히 관련 없는 제품들 간의 이상하게 들리는 업셀링"을 시도하게 됩니다. 이것은 효과가 없습니다.

벤더는 정말로 FOSS 커뮤니티의 선의와 호의에 의존합니다.

## 면죄부 판매와 유사함

소비자들은 그저 자신들의 공급망에서 작동하는 컴포넌트를 원합니다. 그것이 어떻게 만들어졌는지는 (대부분) 신경 쓰지 않습니다. 그리고 5-10년 후에도 작동할 것이라고 보장해줄 벤더를 원합니다.

저는 고객들이 면죄부를 판매하는 것과 비슷한 느낌을 받기를 원하지 않습니다. 저는 고객들이 승자처럼 느끼기를 원합니다.

## 권장되는 앞으로의 길

실행 가능한 오픈 소스 비즈니스에는 다음이 필요합니다:

- 이중 라이선싱 모델
- 합리적인 업셀링 전략
- 프리미엄 진입점으로서의 오픈 소스
- 지속 가능한 수익 메커니즘

오픈 소스를 비즈니스로 만드세요. 실행 가능한 비즈니스, 벤더에 의해 (그리고 물론 시장에 의해) 주도되는 비즈니스로 만드세요. 이중 라이선싱과 합리적인 업셀링을 포함하는 비즈니스, 벤더에게 이익이 되면 소비자에게도 이익이 되는 비즈니스, 오픈 소스를 주로 실제 비즈니스의 프리미엄 진입점으로 사용하는 비즈니스로 만드세요.

마음속으로 F(L)OSS에 대해 낭만적일 수는 있습니다, 그건 괜찮습니다. 하지만 제발 그것에 의존하지는 마세요.

저희 Data Geekery에서는 마침내 오픈 소스 라이선스에서 상용 라이선스로 전환했고 그것은 대성공이었습니다. 이제 저희가 규칙을 만듭니다. 그리고 그 규칙은 모든 사람을 훨씬 더 행복하게 만듭니다. 왜냐하면 그 규칙은 벤더가 5-10년 후에도 여전히 해당 컴포넌트를 지원하기 위해 최선을 다할 것임을 암시하기 때문입니다.
