# 좋은 풀 리퀘스트를 제공하는 방법

> 원문: https://blog.jooq.org/how-to-provide-a-good-pull-request/

저는 오늘 jOOQ 프로젝트에 훌륭한 풀 리퀘스트를 받았습니다. 그것은 모범적인 특성들을 가지고 있었습니다:

1. 리팩토링 단계들을 포함하고 있습니다 - 여러 개의 작은 리팩토링 개선 사항들이 준비 작업으로 포함되어 있습니다
2. 회귀 테스트를 포함하고 있습니다 - 수정이 기존 기능을 깨뜨리지 않는지 확인하기 위한 작은 회귀 테스트가 포함되어 있습니다
3. 실제 수정은 끝에 있습니다 - 실제 버그 수정은 준비 작업 이후에, 마지막에만 나타납니다

이렇게 많은 커밋들이 있으면, 저는 풀 리퀘스트를 쉽게 리뷰할 수 있습니다. 이것은 그것이 코드베이스에 포함되어야 할지 아닐지를 결정하기 쉽게 만들어줍니다.

## 핵심 요점

이 글은 복잡한 변경사항들을 논리적으로 순서가 정해진, 소화하기 쉬운 커밋들로 나누는 것을 권장합니다. 이러한 접근 방식은 코드 리뷰를 용이하게 하고 메인테이너들에게 수락될 가능성을 높입니다. 이 접근 방식은 리뷰어의 시간에 대한 존중을 보여주고, 철저한 검사를 통해 코드 품질을 향상시킵니다.
