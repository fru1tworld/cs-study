## jOOQ 3.2 라이선스 업데이트

3주 전, 저희는 유료 라이선스 계약을 통해 상용 데이터베이스 지원을 제공하는 새로운 듀얼 라이선스 프레임워크 하에 jOOQ 3.2를 공개하고 배포했습니다.

개발자 커뮤니티는 사용자 그룹에서 상당한 참여를 보여주었으며, 대체로 호의적인 반응이었습니다. 대부분은 라이선스 변경을 jOOQ의 지속적인 발전을 위한 긍정적인 신호로 보았지만, 일부 요소들은 정당한 질문이 필요했습니다.

중요한 우려 사항 중 하나는 "배포(distribution)" 용어와 관련하여 jOOQ 라이선스 1.0(10월 1일)의 문구에 관한 것이었습니다. 이는 jOOQ 라이선스 1.1(10월 28일)에서 수정되었으며, 라이선스 보유자가 "최종 사용자 애플리케이션과 함께 jOOQ를 배포하고 내장할 수 있음(§6.2)"을 명시적으로 허용합니다. 이는 고객에 대한 의도된 제약이 아니라 명확화를 나타냅니다. 업데이트된 라이선스는 기존 jOOQ 3.2 고객에게 즉시 적용되어 확장된 권리를 부여합니다.

라이선스 문의에 대한 FAQ가 참조용으로 게시되었습니다.

## jOOQ 3.3 전망

릴리스 3.3(2014년 초 목표)에는 다음과 같은 다양한 기능이 포함될 예정입니다:

- jOOQ Professional Edition에서 MS Access 지원
- 키셋 페이징 기능
- jOOQ 3.2+를 사용한 행 수준 보안 및 공유 스키마 멀티테넌시를 위한 SQL 변환 SPI 예제 구현
- 커뮤니티가 SPI 구현 및 추가 플러그인을 공유할 수 있는 오픈 소스 기여 플랫폼

추가 세부 정보는 GitHub 3.3.0 마일스톤에 나타나며, 사용자 그룹을 통해 참여할 수 있습니다.

## SQL 존 - 키셋 페이징

최근 뉴스레터에서는 NoSQL에 대해 논의했습니다. 그러나 SQL로 돌아가서 코드에 대한 통제력을 유지하는 것이 여전히 중요합니다. jOOQ 3.3은 "seek method"라고도 불리는 네이티브 키셋 페이징 지원을 제공할 예정입니다. 최근 블로그 게시물에서 이 메커니즘을 설명하고 대규모 데이터셋에서 상수 시간 페이징이 어떻게 작동하는지 보여줍니다. 이는 Twitter나 Facebook 스타일의 지연 데이터 로딩에 특히 유용합니다.

## SQL 존 - 유용한 SQL 기법들

주목할 만한 두 가지 SQL 기법:

- SELECT DISTINCT 쿼리에서 DENSE_RANK()를 대신 사용하여 ROW_NUMBER()를 생성하기. ROW_NUMBER()가 DISTINCT를 무효화하기 때문입니다.
- 집계된 데이터를 정규화하기 위해 재귀 SQL 사용하기 - 본질적으로 테이블의 그룹화와 집계를 해제하는 것입니다.

## SQL 존 - MySQL 파서 / jOOQ 코드 생성기

10월 10일 뉴스레터에서 GUDU Soft와 Data Geekery 간의 협력인 SQL2jOOQ를 발표했습니다. MySQL에서 jOOQ 코드로의 변환 작업이 상당히 진전되었습니다. 이 새로운 도구는 광범위한 수작업 문자열 기반 SQL을 포함하는 레거시 애플리케이션을 jOOQ로 마이그레이션하는 것을 용이하게 할 것입니다.
