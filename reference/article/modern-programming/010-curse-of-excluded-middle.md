# 배중률의 저주

> 원문: [The Curse of the Excluded Middle](https://queue.acm.org/detail.cfm?id=2611829)
> 저자: Erik Meijer | 번역일: 2026-07-10

## "대체로 함수형"인 프로그래밍은 통하지 않는다.

### Erik Meijer

소프트웨어 업계에는 "대체로 함수형(mostly functional)"인 프로그래밍을 동시성, 병렬성(매니코어), 그리고 당연히 빅데이터까지, 개발자가 마주한 문제를 해결할 은탄환으로 팔아넘기는 유행이 있다. 오늘날의 명령형 언어들은 이 흐름을 이어가며 클로저를 받아들이고 변이(mutation)와 기타 부수 효과를 제한하려 할 수 있다. 안타깝게도 "대체로 안전한(mostly secure)" 보안이 통하지 않듯, "대체로 함수형"인 프로그래밍도 통하지 않는다. 개발자는 완전히 근본주의적인 선택지도 진지하게 고려해야 한다. 모든 효과를 모나드를 사용해 타입 시스템에 명시적으로 드러내는 순수 지연 함수형 프로그래밍을 받아들이는 것이다.

10분 만에 기적을 만들어 준다는 운동 기구에 넘어가는 다이어터처럼, 개발자들은 자기 분야의 최신 위기에 대한 손쉬운 해법에 넘어갈 준비가 되어 있는 듯하다. 최근 많은 이가 "거의 함수형인 프로그래밍"과 "제한된 부수 효과"를 새로 등장한 골칫거리인 동시성과 병렬성에 맞설 완벽한 무기로 내세운다. 그 이득이 I/O 같은 흔한 연산에 숨어 있는 잠재적 효과(latent effect)의 편리함을 포기하는 대가로만 얻어진다는 사실을 받아들이려는 사람은 드물다. 운동의 이득이 시간과 땀이라는 대가를 반드시 요구한다는 사실을 다이어터가 인정하기 싫어하는 것과 같다.

"대체로 안전한"이 그렇듯 "대체로 순수한"도 희망 사항일 뿐이다. 아주 사소한 암묵적 명령형 효과 하나가 순수성의 이점을 전부 지워 버린다. 박테리아 한 마리가 무균 상태의 상처를 감염시킬 수 있는 것과 마찬가지다. 반대로 명시적이든 암묵적이든 모든 효과를 철저히 뿌리 뽑으면 프로그래밍 언어는 쓸모없어진다. 이것이 배중률의 저주다. 효과와 진지하게 대면하려면 (a) 프로그래밍이란 결국 상태를 변이시키고 기타 효과를 일으키는 일임을 받아들이되, 실용적인 이유로 효과를 최대한 길들이거나, (b) 모든 암묵적 명령형 효과를 폐지하고 타입 시스템에 완전히 명시하되, 실용적인 이유로 가끔 명시적 효과를 감추는 것을 허용하거나, 둘 중 하나를 택해야 한다.

### 문제

명령형 프로그램은 공유된 전역 상태에 암묵적 효과를 반복해서 가하는 방식으로 계산을 기술한다. 그러나 병렬/동시/분산 세계에서 단일 전역 상태는 용납할 수 없는 병목이므로, 오늘날 대부분의 프로그래밍 언어를 떠받치는 명령형 프로그래밍의 근본 가정이 무너지기 시작했다. 흔히들 믿는 것과 달리, 상태 변수를 불변으로 만드는 것만으로는 용납할 수 없는 암묵적 명령형 효과를 없애는 데 한참 못 미친다. 예외, 스레드, I/O처럼 평범한 연산도 단순한 가변 상태 못지않은 고통을 안긴다. 배열에서 20에서 30 사이의 값을 걸러내는 다음 C# 예제를 보자(Gavin Bierman에게 감사한다).

```csharp
static bool LessThanThirty(int x) {
   Console.Write("{0}? Less than 30;", x); return x < 30;
}
static bool MoreThanTwenty(int x) {
   Console.Write("{0}? More than 20;", x); return x > 20;
}

var q0 = new[]{ 1, 25, 40, 5, 23 }.Where(LessThanThirty);
var q1 = q0.Where(MoreThanTwenty);
foreach (var r in q1){ Console.WriteLine("[{0}];",r); }
```

Where는 지연 평가(또는 지연 실행)를 사용하기 때문에 q0과 q1에 쓰인 술어의 효과가 서로 뒤섞인다. 그래서 q1을 평가하면 마치 두 술어의 교집합을 구한 것처럼 20에서 30 사이의 모든 값이 출력된다.

```
1? Less than 30; 1? More Than 20; 25? Less than 30; 25? More Than 20; [25];
40? Less than 30; 5? Less than 30; 5? More Than 20; 23? Less than 30;
23? More Than 20; [23];
```

평범한 프로그래머라면 당연히 q0이 30 이상의 값을 전부 걸러낸 뒤에 q1이 시작해 20보다 작은 값을 제거하리라 기대할 것이다. 두 문장 사이의 세미콜론이 보여 주듯 프로그램이 그렇게 쓰여 있기 때문이다. 술어 사이에 상호 의존이 생기는 것은 불쾌한 충격으로 다가온다.

또 다른 예로 지연 평가와 예외를 섞어 보자. Select(map) 함수에 넘긴 클로저의 본문에서 예외가 던져지면, 지연 실행 때문에 예외는 try-catch 핸들러의 범위 안에서 던져지지 않는다. 대신 foreach 루프가 평가를 강제하는 시점에, 핸들러가 어디에도 없는 곳에서 예외가 던져진다.

```csharp
var xs = new[]{ 9, 8, 7, 6, 5, 4, 3, 2, 1, 0 };
IEnumerable<int> q;
try { q = xs.Select(x=>1/x); } catch { q = new int[]; }
foreach(var z in q){ Console.WriteLine(z): // 여기서 예외가 던져진다 }
```

효과는 다른 방식으로도 언어 기능과 충돌한다. 클로저와 disposable 패턴의 상호작용이 그렇다. 다음 예제는 파일을 읽기용으로 열고, using 블록의 렉시컬 스코프를 벗어나는 클로저에 file 변수를 캡처한다. C#에서 using 문은 블록 진입 시 초기화된 변수를 제어 흐름이 블록 끝에 도달할 때 자동으로 해제한다. 따라서 클로저가 호출되는 시점에는 파일이 이미 해제되어 있고, 예외를 일으키는 코드가 정적 소스 코드에서 위치한 곳과는 시간적으로도 공간적으로도 한참 떨어진 곳에서 뜻밖의 예외가 발생한다.

```csharp
Func<string> GetContents;
using(var file = FileSystem.OpenTextFileReader(@"my file")) {
   GetContents = ()=>file.ReadToEnd();
}
Console.WriteLine(GetContents()); // 이럴 수가! 예외 발생
```

try-catch와 using 블록의 렉시컬 스코프는 부수 효과를 일으키는 클로저의 동적 범위(dynamic extent)와 잘 어울리지 않는다.

### 토끼굴 속으로

지연 평가와 클로저를 쓴 예제가 억지스러워 보인다면, printf 디버깅 스타일로 반환값을 콘솔에 찍는 메서드를 보자. 아무리 순진해 보여도 이제 컴파일러는 공통 부분식 제거(common-subexpression elimination) 같은 단순한 최적화조차 안전하게 수행할 수 없다.

```csharp
string Ha() { var ha = "Ha"; Console.Write(ha); return ha; }

// HaHa를 출력
var haha = Ha()+Ha();

// Ha를 출력
var ha = Ha();
var haha = ha+ha;
```

그런데 잠깐, 실은 훨씬 더 심각하다. 새 객체를 만드는 것부터가 관찰 가능한 부수 효과다. 같은 인자로 호출해도 생성자는 매번 다른 결과를 반환하며, 이는 GetHashCode나 ReferenceEquals 같은 메서드로 관찰할 수 있다.

```csharp
var a = new Object();
// 예를 들어 58225482
Console.WriteLine(a.GetHashCode());

var b = new Object();
// 예를 들어 54267293
Console.WriteLine(b.GetHashCode());

Debug.Assert(a != b);
```

생성자를 순수하게 만들려면 모든 객체에 값 의미론(value semantics)을 강제해야 하고, 그러면 객체 안의 공유 가변 상태를 전부 없애야 한다. 그 순간 객체지향 프로그래밍의 본질, 즉 상태와 동작을 하나의 단위로 캡슐화한다는 개념이 사라진다.

이 예제들은 암묵적 명령형 효과가 낳는 또 다른 해로운 결과를 보여 준다. 흔한 최적화 변환의 상당수를 가로막는다는 점이다. 부수 효과는 프로그램의 전역 환경을 암묵적으로 바꾸기 때문에 효과를 격리하고 국소화하는 일이 대개 불가능하다. 그 결과 명령형 프로그램은 대체로 합성 가능하지 않으며(noncompositional), 프로그래머와 컴파일러 모두 프로그램을 추론하기 어려워진다.

이것만으로도 나쁘지만, 그렇다면 상태 변이만 폐지하면 코드가 순수해지지 않을까? 아니다! 안타깝게도 상태라는 괴물을 길들이는 데는 클래스의 모든 필드를 readonly로 만들고 지역 변수 대입을 금지하는 것만으로는 부족하다. 아래 Cω 프로그램은 코드 어디에도 대입문이나 가변 변수가 보이지 않는데도 스레드가 손쉽게 상태를 흉내 낼 수 있음을 보여 준다. private 채널 Value(T current)가 셀의 상태를 실어 나르는데, 셀 내부 스레드의 메시지 큐에 숨은 가변 상태에 얹혀 가는 것이다.

```csharp
class Cell<T> {
   Cell<T>(T init){ Value(init); }
   T Get() & async Value(T current){ return current; }
   async Set(T @new) & async Value(T old){ Value(@new); }
}
```

Cell 타입의 readonly 필드만으로 가변 Point 클래스를 정의할 수 있다.

```csharp
class Point {
   readonly Cell<Int> x = new Cell<int>(0);
   readonly Cell<int> y = new Cell<int>(0);
}
```

private 채널로 상태를 돌려 전달하는 방식은 Erlang에서 액티브 객체 또는 액터의 *토대*로 받아들여진다. 위의 가변 Cell 프로그램을 Erlang으로 옮기면 다음과 같다. 꼬리 재귀 함수 cell(Value)이 Cell의 상태를 담는다.

```erlang
new_cell(X) -> spawn(fun() -> cell(X) end).
cell(Value) ->
   receive
     {set, NewValue} -> cell(NewValue);
     {get, Pid}      -> Pid!{return, Value}, cell(Value);
     {dispose}       -> {}
   end.
set_cell(Cell, NewValue) -> Cell!{set, NewValue}.
get_cell(Cell) ->
   Cell!{get, self()},
   receive
     {return, Value} -> Value
   end.
dispose_cell(Cell)  -> Cell!{dispose}.
```

이 Erlang 액터는 언어의 패턴 매칭, 메시지 전송, 재귀 프리미티브를 사용해 사실상 동적 메서드 디스패치를 갖춘 객체를 인코딩한다. 이를 기꺼이 활용하면 가변 참조를 구현할 수 있고, Erlang 언어가 가변 상태를 직접 노출하지 않는다는 사실을 무력화하게 된다.

이 예제들은 빙산의 일각일 뿐이다. 부수 효과의 근본적인 문제는 효과를 관찰하는 통제 불가능한 방법이 아주 많다는 것이고, 더 나쁘게는 한 효과로 다른 효과를 흉내 내는 일이 흔히 가능하다는 것이다. 순수한 조합 회로에 재귀를 더하면 가변 상태를 제공하는 플립플롭을 만들 수 있다. 상태와 한정 연속(delimited continuation)만으로 어떤 효과든 시뮬레이션할 수 있다는 것,2 그리고 unchecked 예외로 연속을 시뮬레이션할 수 있다는 것이 증명되어 있다.4 효과를 다룰 때는 아무리 조심해도 지나치지 않다.

지금까지의 예제는 부수 효과가 얼마나 교묘하고 미묘한지, 가장 예상하지 못한 곳에서 어떻게 불쑥 나타나는지 보여 준다. 그렇다면 이렇게 물을지도 모른다. "조심성과 규율만 있으면, 문제를 일으키는 기능을 피해서 명령형 언어를 순수한 언어로 바꿀 수 있지 않을까?" 그러려면 언어에 내재된 효과를 *전부* 제거해야 한다. 평가(evaluation) 자체도 효과이므로 그 제어권을 컴파일러와 런타임에 넘겨야 한다. 그 결과 이런 초순수(ultra-pure) 프로그램은 사용자나 환경과 상호작용할 수 없고, 네트워크 I/O를 수행할 수 없으며, UI 이벤트에 반응하거나, 파일에서 데이터를 읽거나, 현재 시각을 얻거나, 난수를 생성할 수 없다. 이는 끔찍한 딜레마다. 순수 프로그램이 실행되었다는 흔적조차 남길 수 없다면 대체 어디에 쓸모가 있단 말인가?

상황이 암울해 보이지만, 다행히 이 구덩이에서 빠져나갈 길이 몇 가지 있다. John Hughes가 1984년 기념비적인 논문 "Why Functional Programming Matters"1에서 지적했듯, 그 길들은 모두 기능을 맹목적으로 제거하는 대신 세련되게 추가하는 방식이다.

### 근본주의 함수형 프로그래밍

순수 함수형 프로그래밍은 수학적 함수로 하는 프로그래밍이다. 값 사이의 의존 관계를 표현하는 유일한 방법이 함수에 인자를 적용하고 반환값을 거둬들이는 것이라는 뜻이다. 같은 인자로 함수를 호출하면 매번 같은 결과가 나온다. 비밀을 간직할 방법도, 나중에 꺼내 쓰려고 값을 어딘가에 숨겨 둘 방법도, "저것보다 이것을 먼저 해라"라고 직접 말할 방법도, 스레드를 띄우거나 unchecked 예외를 던지거나 그냥 콘솔에 뭔가 출력할 방법도 없다. 경직되고 근본주의적으로 보일 수 있다. 실제로 그렇다. 하지만 동시에 강력하고, 많은 것을 가능하게 한다.

근본주의 함수형 프로그래밍이 동시성 문제를 어떻게 도울 수 있는지 이해하려면, 그것이 단순히 부수 효과를 뺀 명령형 프로그래밍이 아니라는 점을 알아야 한다. 앞서 보았듯 그런 것은 쓸모가 없다. 근본주의 함수형 프로그래밍은 수학적 함수라는 근본주의적 언어를 활용해, 풍부한 효과를 모나드로 합성 가능하게 표현하고 캡슐화한다. 모나드의 직관적인 모델을 세우기 위해, 아마도 가장 단순할 제네릭 메서드를 보자.

```csharp
T Identity<T>(T me) { ... }
```

이 메서드의 타입 시그니처는 *모든 타입* T에 대해, T 타입의 인자를 받으면 T 타입의 값을 반환한다고 주장한다. 그러나 이는 진실의 전부가 아니다. 명령형 타입 시그니처는 *무신경하고(insouciant)*, 관대하고, 너무 느슨하다. 메서드의 실제 *수학적* 타입의 근사치만을 표현할 뿐이다. 실행 중에 숨어 있을 수 있는 어떤 부수 효과도 반영하지 않는다. 인자를 조급하게(eagerly) 평가한다는 사실도, 제네릭 타입 파라미터를 들여다볼 수 있다는 사실(그러면 모든 타입 T에 대해 균일하게 동작하지 않는다)도, 예외를 던지고, I/O를 수행하고, 스레드를 띄우고, 힙에 새 객체를 할당하고, 락을 잡을 수 있다는 사실도, this에 접근할 수 있다는 사실도 담지 않는다.

순수 근본주의 함수형 언어의 비법은 강박적으로 꼼꼼한 타입을 갖추고, 모나드를 사용해 모든 효과를 타입 시그니처에 명시적으로 드러내는 것이다. 즉 순수한 값의 타입과, 효과를 수반할 수 있는 "값에 대한 계산"의 타입을 구별한다.

### 모나드 비공식 입문

타입 a의 값에 대해, 그 타입의 값을 내놓을 수 있는 효과 있는 계산(effectful computation)을 함수 적용 표기 (M a)로 쓰자. 이것이 제네릭 타입에 대한 Haskell 문법이다. 이 효과 있는 계산은 타입 a의 출력 값을 생산하는 기계로 그려 볼 수 있는데, 출력 값을 계산하면서 일으키는 효과 M을 명시적으로 표시한다. 현실 세계에서 물건을 생산하는 부수 효과로 전기를 쓰고 환경을 오염시키는 공장을 떠올리면 된다. 타입 M a의 값은 타입 a의 값을 생산하겠다는 약속일 뿐이며, 아직 아무 효과도 수행되지 않았다는 점에 주의하라. 모나드 값으로 실제로 무언가를 하려면, 이런 효과 있는 계산에 작용하는 두 가지 표준 컴비네이터가 있다.

* *중위 적용 함수* (ma>>=\a->f(a)), 흔히 *bind*라고 부른다. 계산 ma를 실행해 그 효과를 드러내고, 결과 값을 a라 부르며, 이를 함수 f에 넘긴다. 이 적용의 결과 (f a)도 당연히 부수 효과를 일으킬 수 있는 계산이어야 한다. 따라서 bind의 타입 시그니처는 다음과 같다.

```haskell
(>>=):: M a -> (a -> M b) -> M b
```

* *주입 함수* (return a)는 순수한 값을 계산에 주입한다. 타입 시그니처는 return :: a -> M a 이다.

실제로는 각 효과마다 보통 *비고유 사상(nonproper morphism)*이라 불리는 것들이 따라온다. 해당 효과에 고유한 도메인 특화 연산들이다(뒤에 나오는 예제 참고).

이제 위와 같은 형태의 모든 효과 있는 계산을 추상화하고 형식화하면 *모나드*라는 대수 구조가 된다. 모나드는 효과를 만들고 전파하는 두 가지 제네릭 연산 return과 bind(>>=)를 지원한다.

```haskell
class Monad m where {
   (>>=)  :: m a -> (a -> m b) -> m b
   return :: a                 -> m a
}
```

Haskell에서 모든 모나드의 어머니는 I/O 모나드로, 전역 효과를 갖는 모든 계산을 표현한다. 그런 만큼 수많은 비고유 사상이 딸려 오는데, 콘솔 읽기/쓰기, 스레드 띄우기, 예외 던지기처럼 대부분 명령형 프로그래머에게 낯익은 연산이다. 효과가 I/O 모나드 같은 곳에 명시되고 나면, 가변 변수를 할당하고 읽고 쓰는 연산도 I/O 모나드 안으로 들어 올려야(lift) 한다.

```haskell
data IO a -- 어떻게 구현되는지 알 필요 없다

putChar :: Char -> IO ()
getChar :: IO Char

newIORef   :: a            -> IO (IORef a)
readIORef  :: IORef a      -> IO a
writeIORef :: IORef a -> a -> IO ()

forkIO :: IO a -> IO ThreadID
```

forkIO의 타입만 봐도 스레드를 띄우는 것이 부수 효과를 일으키는 연산임을 곧바로 알 수 있다. 덕분에 스레드로 가변 셀을 인코딩하는 어떤 시도든 마찬가지로 I/O 모나드 안에 갇히게 되고, 프로그래머가 실제로는 불변이 아닌 타입을 불변이라고 믿는 일을 막아 준다. 다만 forkIO의 타입이 보여 주듯, I/O 모나드가 여러 스레드에서 전역 상태를 동시에 갱신하는 것까지 막아 주지는 않는다.

모나드의 진짜 힘은 계산 자체가 그저 값이어서, 순수한 호스트 언어 안에서 일급 시민으로 주고받을 수 있다는 데서 나온다. 이 덕분에 프로그래머는 모나드 프리미티브와 비고유 사상을 이용해 새로운 추상화(도메인 특화 구조와 커스텀 제어 구조)를 만들 수 있다.

예를 들어 부수 효과를 일으키는 계산의 리스트를 받아 각각을 실행하고, 결과를 리스트로 모으는 함수를 쉽게 정의할 수 있다.

```haskell
sequence          :: Monad m => [m a] -> m [a]
sequence []       = return []
sequence (ma:mas) = ma >>= (\a -> sequence mas >>= (\as -> a:as))
```

순수 근본주의 함수형 언어는 그 자체로 실행 가능한 표시적 의미론(denotational semantics)이므로, 내부 DSL을 담기에 매우 편리한 호스트 언어다. 그런 점에서 순수 함수형 언어는 P.J. Landin이 1966년에 발표한 영향력 있는 논문 "The Next 700 Programming Languages"3의 비전을 상당 부분 실현한다.

효과를 타입 시스템에 명시하는 것의 또 다른 아름다움은 다양한 효과를 서로 구별할 수 있고 새 효과를 자유롭게 도입할 수 있다는 점이다. 예를 들어 트랜잭셔널 메모리의 모나드 (STM a)를 정의할 수 있다.6 여기에는 트랜잭션 변수를 할당하고 읽고 쓰는 비고유 사상이 들어 있다.

```haskell
newTVar   :: a           -> STM (TVar a)
readTVar  :: TVar a      -> STM a
writeTVar :: TVar a -> a -> STM ()
```

retry 연산은 현재 트랜잭션을 중단하고 어떤 트랜잭션 변수든 바뀔 때까지 기다린다. orElse 연산자로 연결하면 다른 트랜잭션 블록의 실행을 시도한다.

```haskell
retry  :: STM a
orElse :: STM a -> STM a -> STM a
```

마지막으로 atomic 함수가 트랜잭션 모나드를 일반 I/O 모나드에 주입한다. 그 시점 이후로는 그 값에 대해 더 이상 트랜잭션 연산을 수행할 수 없다.

```haskell
atomic :: STM a -> IO a
```

I/O에서 STM으로 (암묵적으로) 새어 들어오는 것이 없으므로, STM 모나드는 트랜잭션 수행에 필요한 메커니즘을 전부 담으면서도 임의의 효과를 롤백할 걱정을 하지 않아도 된다. 일반 Haskell 계산은 순수하다고 가정되기 때문이다.

부수 효과는 길들이기 어려울 뿐 아니라 뒷문으로 몰래 들어오는 일도 잦다. 순수하다고 여겨지는 Haskell에도 unsafePerformIO :: IO a->a 라는, 별것 아닌 듯 보이는 함수가 있다. 인자를 평가할 때 수반되는 부수 효과를 "잊으라"고 컴파일러에게 지시하는 함수다. 이 "함수 비슷한 것"은 선의의 부수 효과를 순수한 계산 안에 캡슐화하기 쉽게 하려는 용도지만, unsafePerformIO는 판도라의 상자를 연다. Haskell 타입 시스템을 전복시켜 언어의 보증을 망가뜨리고, 어떤 타입이든 다른 어떤 타입으로도 캐스팅할 수 있게 만들기 때문이다.

```haskell
unsafeCast :: a -> b
unsafeCast x = unsafePerformIO (do{
   writeIORef castref x; readIORef castref
})
castref = unsafePerformIO (do{ newIORef undefined })
```

이 예제들은 모나드로 부수 효과 계산을 값처럼 다루는 것의 위력을 일부 보여 주지만, 평범한 개발자가 과연 이를 감당할 수 있는가라는 질문은 남는다. 우리는 그 답이 단호한 "그렇다"라고 믿는다. 모나드가 바로 LINQ를 움직이는 원리라는 사실이 이를 뒷받침한다. LINQ의 표준 쿼리 연산자는 본질적으로 모나드 연산자다. 예를 들어 SelectMany 확장 메서드는 앞서 소개한 Haskell의 bind(>>=)와 직접 대응한다.

```
(>>=) :: m a -> (a -> m b) -> m b
M<T> SelectMany(this M<S> src, Func<S, M<T>> f)
```

Haskell의 다형성과 CLR(공용 언어 런타임), Java 등 객체지향 언어의 제네릭 사이의 큰 차이는, Haskell의 모나드 정의가 고차 카인드 다형성(higher-kinded polymorphism)에 의존한다는 점이다. 즉 모나드 클래스(인터페이스)가 타입이 아니라 타입 생성자로 매개변수화된다. 보통 제네릭은 List\<T\>나 Array\<T\>처럼 타입에 대한 매개변수화만 허용하며, M\<T\>의 컨테이너 타입 M 자체를 매개변수로 두고 M을 List나 Array로 인스턴스화하는 것은 허용하지 않는다. 그 결과 모나드를 인코딩하는 LINQ 표준 시퀀스 연산자는 제대로 된 제네릭 대신 문법적 패턴, 사실상 오버로드에 의존해야 한다.

효과에 대한 재사용 가능한 추상화를 이토록 쉽게 정의할 수 있다는 점이, 사람들이 Haskell을 세계 최고의 명령형 언어라 부르곤 하는 이유다. 어쩌면 업계가 지금 마주한 진짜 위기는 순수 근본주의 함수형 프로그래밍을 도입하는 고통이라고 여겨지는 것보다 훨씬 심각한지도 모른다. LINQ가 열광적으로 받아들여졌다는 사실은 사람들이 변화를 맞을 준비가 되었다는 신호다.

### 대안들

흔히들 근본주의적 접근은 평범한 프로그래머에게 너무 큰 패러다임 전환이며, 나아갈 길은 기존 명령형 언어에서 효과를 길들여 언어를 더 순수하게 만드는 것이라고 믿는다. 이때 옹호되는 접근은 효과를 반대 방향에서 길들이는 것이다. 모든 메서드가 주변 효과(ambient effect)를 갖는다고(I/O 모나드 안에 있다고) 가정하되, threadsafe, pure, immutable, readonly, val과 var의 구별 같은 특별한 한정자가 붙은 메서드만 효과의 *부재*를 표시한다. 언뜻 보면 개발자에게 덜 거슬리는 방식 같지만, 실은 그렇지 않다. 일단 순수한 문맥 안에 들어가면 불순한 함수를 호출할 수 없으므로, 순수 메서드에서 호출되는 모든 메서드에도 전이적으로 애노테이션을 달아야 한다. Haskell에서 순수한 식의 깊숙한 곳에 효과 하나를 추가하려면 모나드 스타일로 전면 리팩터링해야 하는 것과 마찬가지다.

순수성 애노테이션의 한 가지 문제는 확장이 불가능하다는 점이다. 즉 사용자가 새로운 "비효과(non-effect)"를 정의할 수 없다. 모나드 합성 예제에서 보았듯, 필요할 때마다 사용자가 자신의 효과와 비고유 사상을 정의할 수 있게 지원하는 것이 핵심이다.

둘째, 순수성 애노테이션은 보통 *함수*에 붙는 반면, Haskell에서 효과는 함수가 아니라 *값*에 붙는다. Haskell에서 타입 f::A->IO B의 함수는 타입 A의 값을 받아 부수 효과 계산(타입 IO B의 값으로 표현된다)을 반환하는 순수 함수다. 함수 f를 적용해도 즉각적인 효과는 전혀 일어나지 않는다. 이는 함수를 순수하다고 표시하는 것과는 사뭇 다르다. 앞서 보았듯, 효과를 값에 붙이면 프로그래머가 자신만의 제어 구조를 정의할 수 있다. 예를 들어 부수 효과 계산의 리스트를 받아 리스트를 계산하는 하나의 부수 효과 계산으로 바꾸는 식이다.

명령형 언어에서 효과를 추론하기 위한 다른 제안도 많다. 선형 타입이나 유일성 타입, 소유권 타입, 가장 최근에는 분리 논리(separation logic)5 등이다. 그러나 이들은 모두 사용자와 도구 양쪽에 높은 수준의 정교함을 요구한다. 모나드와 순수 함수형 프로그래밍의 단순한 등식 추론(equational reasoning)에 비하면 극도로 무거운 수학적 기계장치이며, 따라서 평범한 프로그래머의 삶을 실제로 단순하게 만들어 주지 못한다. 코드를 작성하고 추론하는 데 이론 컴퓨터 과학 박사 학위가 필요해서는 안 된다.

효과를 길들여 명령형 언어를 순수하게 만드는 일은, 근본주의자가 되어 순수 언어를 명령형으로 만드는 일만큼이나 개발자에게 고통스럽다.

### 결론

"대체로 함수형인 프로그래밍"이라는 발상은 실현 불가능하다. 암묵적 부수 효과를 부분적으로만 제거해서 명령형 프로그래밍 언어를 더 안전하게 만드는 것은 불가능하다. 효과 한 종류만 남겨 둬도, 방금 없애려 했던 바로 그 효과를 흉내 내기에 충분한 경우가 많다. 반대로 순수 언어에서 효과를 "잊는" 것을 허용해도 나름의 방식으로 대혼란이 벌어진다.

안타깝게도 황금 같은 중간 지대는 없으며, 우리는 고전적인 이분법 앞에 서 있다. 배중률의 저주다. (a) 순수성 애노테이션으로 효과를 길들이려 하되 코드가 여전히 근본적으로 효과적(effectful)이라는 사실을 온전히 받아들이거나, (b) 모든 효과를 타입 시스템에 명시해 순수성을 온전히 받아들이되 unsafePerformIO 같은 비함수(nonfunction)를 도입해 실용성을 챙기거나, 둘 중 하나다. 여기서 보인 예제들이 언어 설계자와 개발자들로 하여금 거울을 통과해, 근본주의 함수형 프로그래밍을 더 진지하게 바라보기 시작하게 만들기를 바란다.
