## Part Two: 데이터 통합

먼저 "데이터 통합"이 무엇을 뜻하는지, 왜 중요한지를 짚고 나서 로그와의 관계로 넘어간다. 여기서 말하는 데이터 통합이란 **조직이 가진 모든 데이터를 조직의 모든 서비스와 시스템에서 쓸 수 있도록 만드는 것**이다.

"데이터 통합"이라는 표현 자체가 널리 쓰이는 용어는 아니지만 이보다 더 적절한 표현을 찾기 어렵다. 좀 더 익숙한 용어인 ETL은 보통 관계형 데이터 웨어하우스를 채우는, 데이터 통합의 아주 제한된 일부만을 가리킨다. 이 글에서 다루는 것은 사실 ETL을 실시간 시스템과 처리 흐름까지 아우르도록 일반화한 개념에 가깝다.

![케이블 배선 사진](https://web.archive.org/web/20240105095933im_/https://content.linkedin.com/content/dam/engineering/en-us/blog/migrated/cabling.jpg)

빅데이터를 둘러싼 온갖 화려한 관심과 열기 속에서 데이터 통합이라는 화제는 거의 언급되지 않지만, 필자는 오히려 이 다소 평범해 보이는 "데이터를 쓸 수 있게 만드는" 문제야말로 조직이 집중할 수 있는 가장 값진 일 가운데 하나라고 본다.

데이터를 효과적으로 활용하는 과정은 일종의 매슬로우의 욕구 단계와 비슷한 구조를 갖는다. 피라미드의 가장 아래층은 관련된 데이터를 모두 포착해서, 이를 적용 가능한 처리 환경(화려한 실시간 질의 시스템이든, 그냥 텍스트 파일과 파이썬 스크립트든)에 함께 모아 놓는 일이다. 이 데이터는 읽고 처리하기 쉽도록 일관된 방식으로 모델링되어야 한다. 데이터를 일관되게 포착한다는 기초적인 필요가 충족된 뒤에야, MapReduce나 실시간 질의 시스템 같은 인프라로 이 데이터를 다양하게 가공하는 작업으로 넘어갈 수 있다.

당연한 얘기지만, 신뢰할 수 있고 완전한 데이터 흐름이 갖춰지지 않은 Hadoop 클러스터는 비싸고 조립하기 까다로운 공간 히터에 지나지 않는다. 데이터와 처리 인프라가 갖춰진 다음에야 좋은 데이터 모델과 잘 정의된 일관된 의미론이라는, 좀 더 정제된 문제로 관심을 옮길 수 있고, 그다음에야 더 정교한 시각화·리포팅·알고리즘 처리와 예측으로 나아갈 수 있다.

필자의 경험상 대부분의 조직은 이 피라미드의 기반부에 커다란 구멍을 갖고 있다 — 즉 신뢰할 수 있고 완전한 데이터 흐름이 없다 — 그러면서도 곧바로 고급 데이터 모델링 기법으로 건너뛰고 싶어 한다. 이는 순서가 완전히 뒤바뀐 접근이다.

그렇다면 질문은, 조직 내 모든 데이터 시스템에 걸쳐 신뢰할 수 있는 데이터 흐름을 어떻게 만들 것인가다.

### 데이터 통합을 어렵게 만드는 두 가지 흐름

이 문제를 더 어렵게 만드는 두 가지 트렌드가 있다.

**첫째, 이벤트 데이터의 폭증.** 이벤트 데이터는 "지금 무슨 상태인가"가 아니라 "무슨 일이 일어났는가"를 기록한다. 웹 시스템에서는 사용자 행동 로깅이 대표적이지만, 데이터센터 전체의 머신들을 안정적으로 운영·모니터링하는 데 필요한 머신 수준의 이벤트와 통계도 여기 포함된다. 사람들은 흔히 이를 "로그 데이터"라 부르는데, 애플리케이션 로그 파일에 자주 기록되기 때문이다. 하지만 이는 형식과 기능을 혼동하는 표현이다. 이런 데이터는 현대 웹 산업의 핵심이다. 구글의 부(富)는 결국 클릭과 노출, 즉 이벤트를 바탕으로 만들어진 연관성 파이프라인에서 나온다.

이런 흐름이 웹 기업에만 국한되는 것도 아니다. 웹 기업들이 먼저 이런 흐름을 겪는 이유는 그들이 이미 완전히 디지털화되어 있어 계측(instrumentation)이 쉽기 때문일 뿐이다. 금융 데이터는 오래전부터 이벤트 중심이었고, RFID는 물리적 사물에도 이런 추적 방식을 더한다. 전통적인 비즈니스와 활동들이 디지털화되면서 이 흐름은 계속될 것으로 보인다.

이런 이벤트 데이터는 전통적인 데이터베이스 용도의 데이터보다 몇 자릿수는 더 큰 규모를 이루는 경향이 있고, 이는 처리에 상당한 부담을 준다.

**둘째, 전문화된 데이터 시스템의 폭증.** 지난 5년 사이 OLAP, 검색, 단순 온라인 저장, 배치 처리, 그래프 분석 등 특정 용도에 특화된 시스템들이 인기를 끌고 흔히 무료로도 이용 가능해졌다.

더 다양한 종류의 더 많은 데이터를, 더 많은 시스템으로 밀어 넣고 싶다는 욕구가 합쳐지면서 거대한 데이터 통합 문제가 생겨난다.

### 로그 구조 데이터 흐름

이 문제를 다루는 데 있어 로그는 시스템 간 데이터 흐름을 위한 자연스러운 자료구조다. 처방은 아주 단순하다. **조직의 모든 데이터를 실시간 구독이 가능한 중앙 로그에 담는 것.**

각 논리적인 데이터 소스는 각자의 로그로 모델링될 수 있다. 데이터 소스는 클릭이나 페이지뷰 같은 이벤트를 남기는 애플리케이션일 수도 있고, 변경을 받아들이는 데이터베이스 테이블일 수도 있다. 각 구독 시스템은 이 로그를 가능한 한 빠르게 읽어서, 새 레코드를 자신의 저장소에 적용하고 로그 상의 자기 위치를 갱신해 나간다. 구독자는 캐시, Hadoop, 다른 사이트의 또 다른 데이터베이스, 검색 시스템 등 어떤 종류의 데이터 시스템이든 될 수 있다.

![로그 구독 다이어그램](https://web.archive.org/web/20240105095933im_/https://content.linkedin.com/content/dam/engineering/en-us/blog/migrated/log_subscription.png)

예컨대 로그 개념은 각 변경마다 하나의 논리적 시계를 제공하는데, 이를 기준으로 모든 구독자의 위치를 잴 수 있다. 각 구독자 시스템이 서로 대비해 어디까지 읽었는지를 "특정 시점"으로 나타낼 수 있으므로, 서로 다른 구독 시스템들의 상태를 추론하는 일이 훨씬 단순해진다.

좀 더 구체적으로 살펴보자. 데이터베이스 하나와 여러 대의 캐시 서버가 있는 단순한 상황을 가정하면, 로그는 이 시스템들 전체의 업데이트를 동기화하고 각 시스템이 어느 시점에 도달했는지를 추론하는 방법을 제공한다. 로그 엔트리 X번으로 어떤 레코드를 기록한 뒤 캐시에서 읽어야 한다고 하자. 만약 오래된(stale) 데이터를 보지 않도록 보장하고 싶다면, X까지 복제가 끝나지 않은 캐시로부터는 읽지 않도록만 하면 된다.

또한 로그는 데이터 생산과 데이터 소비를 비동기적으로 분리해주는 버퍼 역할도 한다. 이는 특히 서로 다른 속도로 소비하는 여러 구독자가 존재할 때 중요하다. 구독 시스템은 다운되거나 유지보수에 들어가더라도 복귀했을 때 자신이 감당할 수 있는 속도로 따라잡을 수 있다. Hadoop이나 데이터 웨어하우스 같은 배치 시스템은 시간 단위, 일 단위로만 소비해도 되지만, 실시간 질의 시스템은 초 단위로 최신 상태를 따라잡아야 할 수도 있다. 원본 데이터 소스도, 로그 자체도 다양한 목적지 시스템들에 대해 전혀 알 필요가 없으므로, 파이프라인을 전혀 바꾸지 않고도 소비자 시스템을 추가하거나 제거할 수 있다.

이 대목에서 필자는 톨스토이의 『안나 카레니나』 도입부 문장을 빗대어, "잘 동작하는 데이터 파이프라인은 모두 로그처럼 설계되어 있고, 고장 난 파이프라인은 저마다의 방식으로 고장 난다"는 취지의 농담을 남긴다.

![톨스토이 이미지](https://web.archive.org/web/20240105095933im_/https://content.linkedin.com/content/dam/engineering/en-us/blog/migrated/tolstoy.jpg)

특히 중요한 점은, 목적지 시스템이 오직 로그만 알면 되고 원본 시스템의 세부 사항은 전혀 몰라도 된다는 것이다. 소비 시스템은 데이터가 RDBMS에서 왔는지, 최신 유행의 키-값 저장소에서 왔는지, 아니면 어떤 실시간 질의 시스템도 거치지 않고 만들어졌는지 신경 쓸 필요가 없다. 사소해 보이지만 실제로는 결정적으로 중요한 지점이다.

필자는 여기서 "메시징 시스템"이나 "pub/sub" 대신 굳이 "로그"라는 표현을 쓰는데, 그 이유는 로그가 의미론에서 훨씬 더 구체적이고, 데이터 복제를 실제로 지원하는 데 필요한 실용적 구현에 훨씬 가까운 표현이기 때문이다. "발행-구독"이라는 표현은 사실상 메시지의 간접 주소 지정 이상을 함의하지 않으며, 발행-구독을 표방하는 두 메시징 시스템을 비교해보면 보장하는 내용이 서로 완전히 다른 경우가 많아 이 도메인에서는 별로 쓸모가 없다는 것이 필자의 경험이다. 로그는 지속성 보장과 강한 순서 보장을 갖춘 일종의 메시징 시스템으로 볼 수 있으며, 분산 시스템 문헌에서는 이런 통신 모델을 (다소 투박한 이름으로) atomic broadcast라 부르기도 한다.

여기서 짚어둘 점은, 로그는 어디까지나 인프라일 뿐이라는 것이다. 로그만으로 데이터 흐름을 완전히 정복했다고 할 수는 없다. 나머지 이야기는 메타데이터, 스키마, 호환성, 데이터 구조와 그 진화를 다루는 온갖 세부 사항들에 관한 것이다. 하지만 데이터 흐름의 메커니즘을 신뢰성 있고 일반적인 방식으로 다루는 방법이 갖춰지기 전까지는, 의미론적 세부 사항은 부차적인 문제다.

### LinkedIn에서의 경험

필자는 LinkedIn이 중앙 집중형 관계형 데이터베이스에서 여러 분산 시스템의 집합으로 옮겨가는 과정에서 이 데이터 통합 문제가 빠르게 부상하는 것을 직접 목격했다. 당시 LinkedIn의 주요 데이터 시스템으로는 검색, 소셜 그래프, Voldemort(키-값 저장소), Espresso(문서 저장소), 추천 엔진, OLAP 질의 엔진, Hadoop, Teradata, Ingraphs(모니터링 그래프·메트릭 서비스) 등이 있었다. 이들 각각은 자기 분야에서 고급 기능을 제공하는 특화된 분산 시스템이었다.

![LinkedIn 이미지](https://web.archive.org/web/20240105095933im_/https://content.linkedin.com/content/dam/engineering/en-us/blog/migrated/linkedin.png)

데이터 흐름에 로그를 쓴다는 아이디어 자체는 필자가 합류하기 전부터 LinkedIn 내부에 이미 있었다. 초기 인프라 중 하나로 Oracle 테이블 위에 로그 캐싱 추상화를 얹어 데이터베이스 변경 구독을 확장할 수 있게 해주는 Databus라는 서비스가 있었는데, 이는 소셜 그래프와 검색 색인에 데이터를 공급하기 위한 것이었다.

필자 개인의 관여는 2008년 무렵 키-값 저장소 개발을 마친 뒤 시작되었다. 다음 과제는 실제로 동작하는 Hadoop 환경을 구축해서 일부 추천 처리를 그리로 옮기는 것이었다. 이 분야 경험이 부족했던 팀은 데이터를 들여오고 내보내는 데는 몇 주 정도, 나머지 시간은 정교한 예측 알고리즘 구현에 쓰겠다고 순진하게 예상했다. 그렇게 길고 힘든 여정이 시작됐다.

처음에는 기존 Oracle 데이터 웨어하우스에서 데이터를 긁어오는 방식을 계획했다. 그러나 곧 Oracle에서 데이터를 빠르게 뽑아내는 것 자체가 상당한 비기(秘技)라는 것을 깨달았다. 설상가상으로 데이터 웨어하우스의 처리 방식은 Hadoop에서 계획한 프로덕션 배치 처리와 맞지 않았다 — 대부분의 처리가 되돌릴 수 없고 특정 리포팅 용도에 특화되어 있었기 때문이다. 결국 데이터 웨어하우스를 우회해 원본 데이터베이스와 로그 파일로 직접 접근하는 방식을 택했고, 마지막으로 처리 결과를 서빙용 키-값 저장소로 다시 적재하는 또 하나의 파이프라인을 구현했다.

이런 평범한 데이터 복사 작업이 결국 초기 개발 기간의 상당 부분을 차지하는 항목이 되어버렸다. 더 나쁜 것은, 어느 파이프라인에든 문제가 생기면 Hadoop 시스템 전체가 거의 무용지물이 됐다는 점이다. 나쁜 데이터에 정교한 알고리즘을 돌려봐야 더 나쁜 데이터만 만들어질 뿐이었다.

비교적 범용적으로 구축했음에도, 새로운 데이터 소스가 추가될 때마다 별도의 커스텀 설정이 필요했고, 이는 곧 수많은 오류와 장애의 원인이 되었다. Hadoop 위에 구현한 사이트 기능들이 인기를 끌면서 관심 있는 엔지니어들의 대기 목록이 길어졌고, 각 사용자는 연동하고 싶은 시스템 목록과 원하는 새 데이터 피드 목록을 저마다 갖고 있었다.

![시시포스 이미지: 고대 그리스의 ETL, 별로 달라진 게 없다](https://web.archive.org/web/20240105095933im_/https://content.linkedin.com/content/dam/engineering/en-us/blog/migrated/sisyphus.jpg)

이 과정에서 몇 가지가 서서히 분명해졌다.

첫째, 다소 지저분하긴 했지만 이미 구축된 파이프라인들은 실제로 대단히 값진 것이었다. 새로운 처리 시스템(Hadoop)에 데이터를 쓸 수 있게 만드는 과정 자체가 이전에는 어려웠던 여러 계산을 가능하게 만들었고, 이전에는 여러 전문 시스템에 갇혀 있던 데이터 조각들을 한데 모으는 것만으로도 새로운 제품과 분석이 나올 수 있었다.

둘째, 신뢰할 수 있는 데이터 적재를 위해서는 파이프라인 자체가 이를 깊이 지원해야 한다는 것이 분명해졌다. 필요한 구조를 처음부터 제대로 포착해두면, Hadoop으로의 데이터 적재를 완전히 자동화할 수 있다. 새로운 데이터 소스를 추가하거나 스키마가 바뀔 때마다 사람이 손댈 필요 없이, 데이터는 자동으로 HDFS에 나타나고 Hive 테이블은 적절한 컬럼으로 자동 생성되어야 한다는 것이다.

셋째, 여전히 데이터 커버리지가 매우 낮았다. LinkedIn이 가진 전체 데이터 중 Hadoop에서 실제로 이용 가능한 비율을 따져보면 여전히 매우 불완전했고, 새 데이터 소스 하나를 운영 가능한 상태로 만드는 데 드는 노력을 생각하면 완전한 커버리지에 도달하는 것은 결코 쉬운 일이 아니었다.

당시 방식 — 각 데이터 소스와 목적지 쌍마다 커스텀 데이터 적재를 따로 구축하는 방식 — 은 명백히 불가능한 접근이었다. 수십 개의 데이터 시스템과 저장소가 있었는데, 이들 모두를 서로 연결하려면 각 시스템 쌍 사이에 커스텀 파이프를 놓아야 했다.

![복잡하게 얽힌 파이프라인 다이어그램](https://web.archive.org/web/20240105095933im_/https://content.linkedin.com/content/dam/engineering/en-us/blog/migrated/datapipeline_complex.png)

여기서 유의할 점은, 데이터베이스나 Hadoop처럼 많은 시스템이 데이터의 소스인 동시에 목적지이기도 해서 데이터가 흔히 양방향으로 흐른다는 것이다. 즉 시스템당 사실상 두 개의 파이프라인 — 데이터를 들여오는 것과 내보내는 것 — 을 만들어야 했다. 이런 식으로는 완전한 연결에 다가갈수록 파이프라인 수가 대략 시스템 수의 제곱(O(N²))에 가깝게 늘어나므로, 이를 유지하려면 어마어마한 인력이 필요했고 애초에 운영 가능한 구조가 아니었다.

대신 필요했던 것은 이런 형태의 좀 더 범용적인 구조였다.

![단순화된 파이프라인 다이어그램](https://web.archive.org/web/20240105095933im_/https://content.linkedin.com/content/dam/engineering/en-us/blog/migrated/datapipeline_simple.png)

가능한 한, 각 소비자를 데이터 소스로부터 격리시켜야 했다. 이상적으로는 모든 것에 접근할 수 있게 해주는 단 하나의 데이터 저장소만 연동하면 충분해야 한다. 즉 새로운 데이터 시스템(소스든 목적지든)을 추가할 때 필요한 통합 작업이 각 소비자마다가 아니라 단 하나의 중앙 파이프라인에 연결하는 것으로 끝나야 한다는 아이디어다.

이 경험이 필자로 하여금 Kafka를 만드는 데 집중하게 했다. Kafka는 메시징 시스템에서 봐온 것과 데이터베이스·분산 시스템 내부에서 널리 쓰이던 로그 개념을 결합한 것이다. 목표는 모든 활동 데이터를 위한 중앙 파이프라인이자, 나아가 Hadoop에서 나오는 데이터 배포, 모니터링 데이터 등 다양한 용도로 쓸 수 있는 무언가를 만드는 것이었다.

오랫동안 Kafka는 인프라 제품으로서 다소 독특한(어떤 이는 이상하다고 할 만한) 존재였다. 데이터베이스도 아니고, 로그 파일 수집 시스템도 아니고, 전통적인 메시징 시스템도 아니었기 때문이다. 그런데 최근 Amazon이 Kinesis라는, Kafka와 매우 흡사한 서비스를 내놓았다. 파티셔닝을 다루는 방식, 데이터 보존 방식, 그리고 Kafka API에서 상위/하위 레벨 컨슈머를 나누는 다소 특이한 구조까지 유사성이 이어졌다. 필자는 이를 반가운 신호로 받아들였다. 좋은 인프라 추상화를 만들었다는 증거 중 하나는 AWS가 그것을 서비스로 제공한다는 것이기 때문이다. Amazon의 비전 역시 DynamoDB, RedShift, S3 등 자사의 여러 분산 시스템을 잇는 배관이자, EC2를 이용한 분산 스트림 처리의 기반이라는, 이 글이 말하는 것과 거의 동일한 그림으로 보인다.

### ETL과 데이터 웨어하우스와의 관계

데이터 웨어하우징에 대해서도 잠깐 짚고 넘어갈 만하다. 데이터 웨어하우스는 분석을 지원하기 위해 구조화된, 깨끗하고 통합된 데이터의 저장소가 되는 것을 목표로 한다. 훌륭한 아이디어다. 데이터 웨어하우징 방법론에 익숙하지 않은 이들을 위해 설명하면, 원본 데이터베이스에서 주기적으로 데이터를 추출하고, 이를 이해 가능한 형태로 다듬은 뒤, 중앙 데이터 웨어하우스에 적재하는 과정을 말한다. 모든 데이터의 깨끗한 사본을 담은 이 중앙 저장소를 갖는 것은 데이터 집약적인 분석과 처리에서 엄청나게 값진 자산이다. 이 방법론 자체는 Oracle이나 Teradata 같은 전통적 웨어하우스를 쓰든 Hadoop을 쓰든 크게 다르지 않다. 적재와 가공의 순서만 달라질 뿐이다.

깨끗하고 통합된 데이터를 담은 데이터 웨어하우스는 굉장한 자산이지만, 그것을 만들어내는 메커니즘 자체는 다소 시대에 뒤떨어져 있다.

![Oracle 이미지](https://web.archive.org/web/20240105095933im_/https://content.linkedin.com/content/dam/engineering/en-us/blog/migrated/oracle.jpg)

데이터 중심 조직이 마주하는 핵심 문제는, 깨끗하게 통합된 데이터가 데이터 웨어하우스라는 하나의 시스템에 결합(coupling)되어 있다는 점이다. 데이터 웨어하우스는 단순 카운팅, 집계, 필터링이 중심인 여러 종류의 리포팅과 애드혹 분석에 잘 맞는 배치 질의 인프라다. 하지만 깨끗하고 완전한 데이터의 유일한 저장소가 배치 시스템이라면, 실시간 처리·검색 색인·모니터링 시스템처럼 실시간 피드가 필요한 시스템에는 그 데이터를 쓸 수 없다는 문제가 생긴다.

필자가 보기에 ETL은 사실 두 가지 서로 다른 일을 하나로 뭉뚱그린 것이다. 첫째는 추출과 데이터 정제 과정 — 조직 곳곳의 시스템에 갇혀 있던 데이터를 해방시키고 시스템 특유의 잡음을 제거하는 일이다. 둘째는 그 데이터를 데이터 웨어하우스 질의에 맞게 재구성하는 일, 즉 관계형 DB의 타입 체계에 맞추거나 star/snowflake 스키마에 강제로 끼워 맞추거나 고성능 컬럼 포맷으로 쪼개는 일이다. 이 두 가지를 뭉뚱그리는 것이 문제다. 깨끗하고 통합된 데이터 저장소는 저지연 처리나 다른 실시간 저장 시스템의 색인 작업에도 실시간으로 이용 가능해야 한다.

이렇게 나누면 데이터 웨어하우징 ETL을 조직적으로 훨씬 더 확장 가능하게 만드는 부수적 이점도 생긴다. 데이터 웨어하우스 팀의 고질적인 문제는, 조직 내 다른 모든 팀이 생성한 데이터를 수집하고 정제하는 책임을 떠안는다는 것이다. 인센티브가 서로 어긋나 있다 — 데이터를 생산하는 쪽은 그 데이터가 웨어하우스에서 어떻게 쓰이는지 잘 모르는 경우가 많아서, 추출하기 어렵거나 사용 가능한 형태로 바꾸는 데 무겁고 확장하기 힘든 변환이 필요한 데이터를 만들어내곤 한다. 물론 중앙 팀은 조직의 성장 속도를 결코 따라잡지 못하니, 데이터 커버리지는 늘 듬성듬성하고 데이터 흐름은 취약하며 변경은 느리기만 하다.

더 나은 방식은 데이터를 추가하기 위한 명확한 API를 갖춘 중앙 파이프라인, 즉 로그를 두는 것이다. 이 파이프라인에 연동하고 깨끗하고 잘 구조화된 데이터 피드를 제공할 책임은 그 데이터를 생산하는 쪽에 있다. 즉 시스템을 설계하고 구현할 때부터 데이터를 어떻게 내보내고 중앙 파이프라인에 맞는 잘 구조화된 형태로 만들지를 함께 고민해야 한다는 뜻이다. 새로운 저장 시스템이 추가되어도 데이터 웨어하우스 팀에는 아무 영향이 없다. 이들에게는 이미 통합을 위한 단 하나의 지점이 있기 때문이다. 데이터 웨어하우스 팀은 이제 중앙 로그에서 구조화된 피드를 적재하고, 자신들의 시스템에 특화된 변환만 수행하면 되는, 훨씬 단순한 문제만 다루면 된다.

![파이프라인 소유권 다이어그램](https://web.archive.org/web/20240105095933im_/https://content.linkedin.com/content/dam/engineering/en-us/blog/migrated/pipeline_ownership.png)

이런 조직적 확장성 문제는 전통적인 데이터 웨어하우스를 넘어 추가적인 데이터 시스템을 도입하려 할 때 더욱 중요해진다. 예컨대 조직 전체 데이터셋에 대한 검색 기능을 제공하고 싶다거나, 실시간 트렌드 그래프와 알림 기능을 갖춘 초 단위 미만 모니터링을 제공하고 싶다고 하자. 이런 경우 전통적인 데이터 웨어하우스나 심지어 Hadoop 클러스터의 인프라조차 적합하지 않다. 설상가상으로, 데이터베이스 적재를 지원하기 위해 만들어진 ETL 파이프라인은 이런 다른 시스템들에 데이터를 공급하는 데 별 쓸모가 없는 경우가 많아서, 이런 인프라를 처음부터 구축하는 부담이 데이터 웨어하우스를 도입하는 것만큼이나 커져버린다. 이는 현실적으로 감당하기 어려운 일이고, 대부분의 조직이 모든 데이터에 대해 이런 역량을 쉽게 갖추지 못하는 이유를 설명해준다. 반대로 조직이 이미 균일하고 잘 구조화된 데이터 피드를 갖춰 두었다면, 어떤 새 시스템이든 전체 데이터에 완전히 접근하게 만드는 데 필요한 것은 그 파이프라인에 붙일 단 하나의 연동 작업뿐이다.

이런 아키텍처는 또한 정제나 변환 작업을 어디서 수행할지에 관해 몇 가지 선택지를 제시한다.

- 데이터 생산자가 회사 전체 로그에 데이터를 추가하기 전에 수행할 수 있다.
- 로그 위에서의 실시간 변환으로 수행할 수 있다(이 경우 새로운, 변환된 로그가 만들어진다).
- 특정 목적지 데이터 시스템으로의 적재 과정 일부로 수행할 수 있다.

가장 좋은 모델은 데이터를 발행하는 쪽이 로그에 발행하기 전에 정제를 마치는 것이다. 즉 데이터가 그것을 만든 특정 코드나 그것이 유지되던 저장 시스템의 흔적을 남기지 않는, 표준적인(canonical) 형태를 갖추도록 하는 것이다. 이런 세부 사항은 자신의 데이터를 가장 잘 아는 팀이 처리하는 것이 최선이며, 이 단계에서 적용되는 어떤 로직이든 무손실이고 되돌릴 수 있어야 한다.

실시간으로 수행 가능한 부가가치 변환은 원본 로그 피드에 대한 후처리로 수행하는 것이 좋다. 예컨대 이벤트 데이터의 세션화(sessionization)나, 일반적으로 유용한 다른 파생 필드를 추가하는 일이 여기 해당한다. 원본 로그는 여전히 그대로 남아 있고, 이 실시간 처리는 보강된 데이터를 담은 파생 로그를 만들어낸다.

마지막으로, 목적지 시스템에만 특화된 집계는 적재 과정의 일부로 수행되어야 한다. 예컨대 데이터 웨어하우스에서 분석과 리포팅을 위한 특정 star/snowflake 스키마로 데이터를 변환하는 작업이 여기 해당한다. 전통적인 ETL 과정에 가장 가깝게 대응하는 이 단계는, 이제 훨씬 더 깨끗하고 균일한 스트림들을 대상으로 이루어지므로 훨씬 단순해진다.

### 로그 파일과 이벤트

이 아키텍처의 부수적인 이점 하나는 결합이 느슨한(decoupled), 이벤트 기반 시스템을 가능하게 한다는 것이다.

웹 업계에서 흔한 활동 데이터 처리 방식은 텍스트 파일에 로그로 남긴 뒤 이를 긁어서 데이터 웨어하우스나 Hadoop으로 옮겨 집계·질의하는 것이다. 이 방식의 문제는 다른 모든 배치 ETL의 문제와 같다 — 데이터 흐름이 데이터 웨어하우스의 처리 능력과 스케줄에 결합되어 버린다는 것.

LinkedIn은 이벤트 데이터 처리를 로그 중심 방식으로 구축했다. 중앙의, 다중 구독자를 지원하는 이벤트 로그로 Kafka를 쓰며, 특정 종류의 행동이 갖는 고유한 속성을 포착하는 수백 종의 이벤트 타입을 정의해 두었다. 페이지뷰, 광고 노출, 검색부터 서비스 호출, 애플리케이션 예외까지 모두 이 범주에 들어간다.

이 방식의 장점을 이해하기 위해 간단한 이벤트 하나를 생각해보자 — 채용공고 페이지에 공고 하나를 보여주는 것. 채용공고 페이지는 원래 공고를 보여주는 데 필요한 로직만 가지면 된다. 그런데 상당히 동적인 사이트에서는 이 페이지가 공고를 보여주는 것과 무관한 로직으로 금세 뒤덮이기 쉽다. 예를 들어 다음과 같은 시스템들과 연동해야 한다고 하자.

- 오프라인 처리를 위해 이 데이터를 Hadoop과 데이터 웨어하우스로 보내야 한다.
- 콘텐츠 스크래핑 같은 시도를 탐지하기 위해 조회수를 세야 한다.
- 채용 담당자의 분석 페이지에 표시하기 위해 이 조회를 집계해야 한다.
- 같은 추천을 반복해서 보여주지 않도록 사용자별 노출 상한(impression cap)을 관리하려면 이 조회를 기록해야 한다.
- 추천 시스템이 해당 공고의 인기도를 정확히 추적하기 위해 이 조회를 기록해야 할 수도 있다.
- 그 외에도 여러 가지.

이렇게 되면 공고 하나를 보여주는 단순한 동작이 순식간에 상당히 복잡해진다. 게다가 공고가 노출되는 지점(모바일 앱 등)이 늘어날수록 이 로직을 그대로 이식해야 하고 복잡도는 계속 커진다. 더 나쁜 것은, 이렇게 연동해야 하는 시스템들이 서로 얽혀버려서 공고 표시 화면을 담당하는 사람이 다른 여러 시스템과 기능을 알아야 하고 이들이 제대로 연동되는지 신경 써야 한다는 점이다. 이는 아주 단순화된 예시일 뿐이고, 실제 애플리케이션은 이보다 훨씬 더 복잡할 수밖에 없다.

"이벤트 기반" 스타일은 이 문제를 단순화하는 접근법을 제공한다. 공고 표시 페이지는 이제 공고를 보여주고, 공고가 표시되었다는 사실과 공고·조회자·기타 유용한 속성들을 기록하기만 하면 된다. 추천 시스템, 보안 시스템, 채용 담당자 분석 시스템, 데이터 웨어하우스 같은 다른 관심 있는 시스템들은 모두 이 피드를 구독해서 각자의 처리를 수행한다. 화면을 표시하는 코드는 이런 다른 시스템들의 존재를 알 필요가 없고, 새로운 데이터 소비자가 추가되어도 바뀔 필요가 없다.

### 확장 가능한 로그 구축하기

발행자와 구독자를 분리하는 것 자체는 새로운 아이디어가 아니다. 하지만 소비자 규모의 웹사이트에서 벌어지는 모든 일을 다중 구독자용 실시간 저널로 담는 commit log를 유지하고 싶다면, 확장성이 핵심 과제가 된다. 로그를 범용 통합 메커니즘으로 쓴다는 발상은, 빠르고 저렴하며 실제 규모에서 실용적으로 동작할 만큼 확장 가능한 로그를 만들어낼 수 없다면 그저 우아한 공상에 그칠 뿐이다.

시스템 엔지니어들은 흔히 분산 로그를 느리고 무거운 추상화로 여기고(그래서 Zookeeper가 다루는 것 같은 "메타데이터" 용도에나 어울린다고 생각하는 경우가 많다), 그러나 대용량 데이터 스트림을 저널링하는 데 초점을 맞춰 신중하게 구현하면 꼭 그럴 필요는 없다. LinkedIn은 이 글을 쓰는 시점 기준으로 하루 600억 건이 넘는 고유 메시지를 Kafka로 처리하고 있으며(데이터센터 간 미러링에서 발생하는 쓰기까지 더하면 수천억 건에 달한다), 이는 로그가 결코 느린 추상화일 필요가 없다는 방증이다.

이런 규모를 지원하기 위해 Kafka는 몇 가지 기법을 사용한다.

- 로그를 파티셔닝한다.
- 읽기와 쓰기를 배칭(batching)해서 처리량을 최적화한다.
- 불필요한 데이터 복사를 피한다.

수평 확장을 가능하게 하기 위해 로그를 여러 파티션으로 쪼갠다.

![파티셔닝된 로그 다이어그램](https://web.archive.org/web/20240105095933im_/https://content.linkedin.com/content/dam/engineering/en-us/blog/migrated/partitioned_log.png)

각 파티션은 완전히 순서가 정해진 로그이지만, 파티션들 사이에는 전역적인 순서가 없다(메시지에 굳이 포함시킨 벽시계 시간이 있다면 모를까). 어떤 메시지를 어느 파티션에 넣을지는 쓰는 쪽에서 제어할 수 있으며, 대부분의 사용자는 사용자 ID 같은 키를 기준으로 파티셔닝한다. 파티셔닝 덕분에 샤드 간 조정 없이도 로그에 append할 수 있고, 시스템의 처리량은 Kafka 클러스터 크기에 비례해 선형적으로 늘어난다.

각 파티션은 설정 가능한 수만큼 복제되며, 각 복제본은 해당 파티션의 로그를 동일하게 갖고 있다. 언제든 그중 하나가 리더 역할을 맡고, 리더가 실패하면 다른 복제본 중 하나가 리더 역할을 이어받는다.

파티션 간 전역 순서가 없다는 것은 하나의 제약이지만, 실제로 큰 문제가 되지는 않았다. 실제로 로그와의 상호작용은 보통 수백, 수천 개의 서로 다른 프로세스에서 이루어지므로, 이들의 행동 전체에 대한 전역 순서를 논하는 것 자체가 별로 의미가 없다. 대신 Kafka가 보장하는 것은 각 파티션 내부의 순서가 유지된다는 것과, 단일 발신자가 특정 파티션에 append한 메시지들은 보낸 순서대로 전달된다는 것이다.

로그는 파일 시스템처럼 선형적인 읽기·쓰기 패턴에 최적화하기 좋다. 작은 규모의 읽기·쓰기를 더 크고 처리량 높은 연산으로 묶어낼 수 있는데, Kafka는 이 최적화를 적극적으로 추구한다. 배칭은 클라이언트에서 서버로 데이터를 보낼 때, 디스크에 쓸 때, 서버 간 복제 시, 컨슈머로의 데이터 전송 시, 커밋된 데이터를 확인응답할 때 등 여러 단계에서 이루어진다.

또한 Kafka는 인메모리 로그, 디스크 상의 로그, 네트워크 전송 사이에 동일하게 유지되는 단순한 바이너리 포맷을 쓴다. 이는 zero-copy 데이터 전송을 비롯한 다양한 최적화를 가능하게 한다.

이런 최적화들이 누적된 효과로, 메모리 용량을 훨씬 초과하는 데이터셋을 다루면서도 대체로 디스크나 네트워크가 지원하는 속도만큼 데이터를 읽고 쓸 수 있게 된다.

이 글의 주된 목적이 Kafka 자체를 설명하는 것은 아니므로 세부 사항은 여기까지만 다룬다. LinkedIn의 접근 방식과 Kafka의 설계에 대한 더 자세한 내용은 원문이 링크하는 별도의 자료들에서 확인할 수 있다.
