## 초록

CPU 코어가 점점 빨라지고 많아지면서, 대부분의 프로그램에서 성능을 제한하는 요인은 메모리 접근이 되었다. 하드웨어 설계자들은 CPU 캐시 같은 정교한 메모리 처리·가속 기법을 만들어 왔지만, 이런 장치는 프로그래머의 도움 없이는 최적으로 동작하지 못한다. 문제는 대부분의 프로그래머가 메모리 서브시스템의 구조도, 그것을 사용하는 비용도 잘 모른다는 점이다. 이 문서는 현대 범용(commodity) 하드웨어의 메모리 서브시스템 구조를 설명하고, CPU 캐시가 왜 생겨났는지, 어떻게 동작하는지, 프로그램이 캐시를 활용해 최적 성능을 내려면 무엇을 해야 하는지를 다룬다.

## 1장 서론

초창기 컴퓨터는 CPU, 메모리, 대용량 저장 장치, 네트워크 인터페이스가 함께 개발되어 성능 균형이 잘 맞았다. 예를 들어 메모리와 네트워크 인터페이스가 데이터를 공급하는 속도는 CPU보다 크게 느리지 않았다.

컴퓨터의 기본 구조가 안정화되고 각 서브시스템이 개별적으로 최적화되기 시작하면서 이 균형이 깨졌다. 특히 대용량 저장 장치와 메모리 서브시스템은 비용 문제 때문에 다른 부품보다 개선이 느렸다.

- **저장 장치의 느림**은 주로 소프트웨어 기법으로 해결됐다. OS가 자주 쓰일 것 같은 데이터를 메인 메모리에 캐싱하고, 저장 장치 자체에도 캐시를 붙였다. 이 문서는 저장 장치 최적화는 다루지 않는다.
- **메인 메모리의 병목**은 훨씬 풀기 어려웠고, 거의 모든 해법이 하드웨어 변경을 요구했다. 오늘날 그 변화는 주로 네 가지 형태다:
  1. RAM 하드웨어 설계(속도와 병렬성)
  2. 메모리 컨트롤러 설계
  3. CPU 캐시
  4. 장치를 위한 DMA(직접 메모리 접근)

이 문서는 주로 CPU 캐시와 메모리 컨트롤러 설계의 영향을 다루며, 그 과정에서 DMA도 살펴본다. 범용 하드웨어에 한정하며, OS 관련 내용은 Linux만 다룬다.

### 문서 구조

- 2장: RAM의 기술적 세부 사항. 몰라도 뒤를 이해할 수는 있지만 알아두면 좋다.
- 3장: CPU 캐시 동작의 상세. 이후 내용 이해에 필수적이다.
- 4장: 가상 메모리 구현 개요. 역시 필수 기반 지식.
- 5장: NUMA 시스템의 상세.
- 6장: **이 문서의 핵심.** 앞의 모든 내용을 종합해 프로그래머에게 상황별로 성능 좋은 코드를 쓰는 법을 조언한다. 급한 독자는 6장부터 읽고 필요하면 앞으로 돌아가도 된다.
- 7장: 프로그래머를 돕는 도구들.
- 8장: 가까운 미래에 기대되는 기술 전망.

문서 제목은 David Goldberg의 고전 "What Every Computer Scientist Should Know About Floating-Point Arithmetic"에 대한 오마주다. 본문에 "보통(usually)" 같은 한정어가 자주 나오는 이유는, 여기서 다루는 기술이 현실에서 수많은 변형으로 존재하고 이 문서는 가장 주류인 형태만 다루기 때문이다.

## 2장 오늘날의 범용 하드웨어

특수 하드웨어는 퇴조하고 있고, 스케일링은 수직(더 큰 기계)보다 수평(작은 범용 컴퓨터를 여러 대 연결)으로 이뤄지는 추세다. 빠르고 저렴한 네트워크 하드웨어가 보편화됐기 때문이다. 2007년 시점의 Red Hat은 향후 데이터센터의 "표준 빌딩 블록"을 소켓 최대 4개 × 쿼드코어 CPU(Intel의 경우 하이퍼스레딩 포함)로 예상했다. 즉 표준 시스템이 최대 64개의 가상 프로세서를 갖는다는 뜻이다. 최적화도 대부분 이런 기계를 겨냥한다.

### 전통적 구조: 노스브리지/사우스브리지

수년간 PC와 소형 서버는 칩셋을 노스브리지와 사우스브리지 두 부분으로 표준화했다(그림 2.1).

- 모든 CPU는 공통 버스(FSB, Front Side Bus)로 **노스브리지**에 연결된다. 노스브리지는 메모리 컨트롤러를 포함하며, 그 구현이 사용 가능한 RAM 종류(DRAM, Rambus, SDRAM 등)를 결정한다.
- **사우스브리지**(I/O 브리지)는 PCI, PCI Express, SATA, USB 등 여러 버스를 통해 장치와 통신한다.

이 구조의 결과:
- CPU 간 데이터 통신은 모두 노스브리지와 통신할 때 쓰는 같은 버스를 지나야 한다.
- RAM과의 모든 통신은 노스브리지를 지나야 한다.
- RAM은 포트가 하나뿐이다.
- CPU와 사우스브리지 장치 간 통신은 노스브리지를 경유한다.

**병목 1 — 장치의 RAM 접근**: 초기 PC에서는 브리지 쪽 장치와의 모든 통신이 CPU를 거쳐야 했다. 이를 우회하려고 DMA가 도입됐다. DMA는 장치가 노스브리지의 도움으로 CPU 개입 없이 RAM에 직접 데이터를 읽고 쓰게 한다. CPU 부담은 크게 줄지만, DMA 요청이 CPU의 RAM 접근과 노스브리지 대역폭을 두고 경쟁하는 문제가 생긴다.

**병목 2 — 노스브리지와 RAM 사이의 버스**: 옛 시스템은 모든 RAM 칩에 버스가 하나라 병렬 접근이 불가능했다. 최근 RAM(DDR2 등)은 두 개의 분리된 버스(채널)를 요구해 대역폭이 두 배가 되고, 노스브리지는 메모리 접근을 채널에 인터리빙한다. FB-DRAM 같은 더 새로운 기술은 채널을 더 늘린다.

대역폭이 제한된 상황에서는 메모리 접근을 지연이 최소화되게 스케줄링하는 것이 중요하다. 프로세서는 CPU 캐시가 있어도 메모리보다 훨씬 빠르고, 여러 하이퍼스레드·코어·프로세서가 동시에 메모리에 접근하면 대기 시간은 더 길어진다. DMA도 마찬가지다. 동시성 말고 접근 패턴 자체도 성능에 크게 영향을 준다(특히 다중 채널일 때).

### 변형 1: 외부 메모리 컨트롤러

일부 고가 시스템은 노스브리지 안에 메모리 컨트롤러를 두지 않고, 노스브리지에 여러 개의 외부 메모리 컨트롤러(MC)를 연결한다(그림 2.2, 예: 4개). 장점은 메모리 버스가 여러 개라 총 대역폭이 늘고 더 많은 메모리를 지원한다는 것. 서로 다른 메모리 뱅크에 동시 접근하면 지연도 줄어든다. 이때 1차 제약은 노스브리지 내부 대역폭이다(Intel 계열에서 이 구조가 유명했다).

### 변형 2: 통합 메모리 컨트롤러 (NUMA의 시작)

또 다른 방법은 메모리 컨트롤러를 CPU 안에 통합하고 메모리를 각 CPU에 붙이는 것이다. AMD Opteron 기반 SMP 시스템이 이를 대중화했고, Intel도 Nehalem부터 CSI(Common System Interface)로 같은 접근을 취한다(그림 2.3).

장점: 프로세서 수만큼 메모리 뱅크가 존재한다. 쿼드 CPU 기계에서 거대한 노스브리지 없이 메모리 대역폭이 4배가 된다.

단점: 모든 프로세서가 시스템의 모든 메모리에 접근할 수 있어야 하므로 메모리가 더 이상 균일(uniform)하지 않다 — 그래서 **NUMA**(Non-Uniform Memory Architecture)다.
- 로컬 메모리(그 프로세서에 붙은 메모리)는 보통 속도로 접근한다.
- 다른 프로세서에 붙은 메모리는 프로세서 간 인터커넥트를 거쳐야 한다. 그림 2.3의 구조에서 CPU1이 CPU2의 메모리에 접근하면 인터커넥트 1개, CPU4의 메모리면 2개를 건넌다.
- 원격 메모리 접근에 드는 추가 시간을 **NUMA 팩터**라 부른다. 큰 기계(IBM x445, SGI Altix 등)에서는 노드 사이 연결이 비싸 NUMA 팩터가 꽤 커질 수 있다.

2008년 말부터는 모든 SMP 기계가 NUMA일 것으로 예상됐다. 프로그램이 NUMA 기계에서 돌고 있음을 인지하는 것이 중요하며, 5장에서 자세히 다룬다.

### 2.1 RAM 종류

핵심 질문: 왜 한 기계 안에 SRAM과 DRAM이 둘 다 있는가? SRAM이 훨씬 빠르고 같은 기능을 제공하는데 왜 모든 RAM이 SRAM이 아닌가? 답은 예상대로 **비용**이다. SRAM은 생산·사용 비용이 DRAM보다 훨씬 비싸다.

#### 2.1.1 정적 RAM (SRAM)

6트랜지스터 SRAM 셀(그림 2.4)의 핵심은 교차 결합된 인버터 두 개를 이루는 트랜지스터 4개(M1~M4)로, 0과 1을 나타내는 두 개의 안정 상태를 갖는다. Vdd에 전원이 공급되는 한 상태가 유지된다. 셀 상태를 읽으려면 워드 액세스 라인 WL을 올리면 즉시 BL과 BL̄에서 읽을 수 있고, 덮어쓰려면 BL/BL̄을 원하는 값으로 설정한 뒤 WL을 올린다.

기억할 특성:
- 셀 하나에 트랜지스터 6개가 필요하다(4개짜리 변형도 있지만 단점이 있다).
- 상태 유지에 지속적인 전력이 필요하다.
- WL이 올라가면 셀 상태를 거의 즉시 읽을 수 있고, 신호가 다른 트랜지스터 제어 신호처럼 각지고(rectangular) 뚜렷하다.
- 상태가 안정적이라 리프레시가 필요 없다.

#### 2.1.2 동적 RAM (DRAM)

DRAM 셀은 훨씬 단순하다: 트랜지스터 1개 + 커패시터 1개(그림 2.5). 상태는 커패시터 C에 저장되고 트랜지스터 M이 접근을 통제한다. 읽기는 액세스 라인 AL을 올려 커패시터의 전하 유무에 따라 데이터 라인 DL에 전류가 흐르는지 보는 것이고, 쓰기는 DL을 설정한 뒤 AL을 충분히 오래 올려 커패시터를 충전/방전하는 것이다.

이 단순함에는 여러 복잡한 문제가 따른다:

1. **누설(leakage)**: 읽기가 커패시터를 방전시키고, 셀 수를 늘리려면(요즘 칩은 10^9개 이상) 커패시터 용량이 펨토패럿 수준으로 작아야 한다. 완충된 커패시터가 담는 전자는 수만 개 정도. 커패시터 저항이 커도(테라옴 수준) 전하는 짧은 시간에 새어 나간다. 그래서 DRAM 셀은 끊임없이 **리프레시**해야 한다 — 요즘 대부분의 DRAM 칩은 64ms마다. 리프레시 중에는 그 부분에 접근할 수 없고, 일부 워크로드에서는 이 오버헤드가 메모리 접근의 최대 50%를 지연시킬 수 있다.
2. **신호 증폭**: 셀에서 읽은 정보는 그대로 쓸 수 없고, 저장된 0/1을 구분하는 감지 증폭기(sense amplifier)에 데이터 라인을 연결해야 한다.
3. **파괴적 읽기**: 읽기가 커패시터 전하를 소모하므로 모든 읽기 뒤에는 재충전 작업이 따라야 한다(감지 증폭기 출력을 도로 넣어 자동 수행). 즉 읽기에 추가 에너지와, 더 중요하게는 시간이 든다.
4. **충·방전이 즉각적이지 않음**: 커패시터의 충전·방전 곡선은 지수 함수 Q(t) = Q0(1 − e^(−t/RC)), Q(t) = Q0·e^(−t/RC)를 따른다(그림 2.6). 감지 증폭기가 받는 신호가 각지지 않아, 셀 출력을 언제 신뢰할 수 있는지 보수적으로 추정해야 한다. 이 지연이 DRAM의 속도를 근본적으로 제한한다.

단순함의 장점은 **크기**다. DRAM 셀은 SRAM 셀보다 칩 면적을 훨씬 적게 차지하고, 구조가 규칙적이라 조밀하게 배치하기도 쉽다. SRAM은 셀마다 상태 유지 전력도 필요하다. 결국 (극적인) 비용 차이가 승리한다. 네트워크 라우터 같은 특수 하드웨어를 제외하면 메인 메모리는 DRAM일 수밖에 없고, 이것이 프로그래머에게 큰 함의를 갖는다.

#### 2.1.3 DRAM 접근

프로그램이 가상 주소로 메모리를 지정하면 프로세서가 물리 주소로 변환하고, 메모리 컨트롤러가 해당 RAM 칩을 선택한다. 칩 위의 개별 셀을 선택하기 위해 물리 주소의 일부가 주소 라인들로 전달된다.

메모리 컨트롤러에서 셀을 개별 주소 지정하는 것은 비현실적이다. 4GB RAM이면 주소 라인이 2^32개 필요하다. 대신 주소를 이진수로 인코딩해 적은 수의 주소 라인으로 전달하고, DRAM 칩에서 디멀티플렉스한다. N개 주소 라인의 디멀티플렉서는 출력 라인이 2^N개다. 문제는 셀 수가 커지면 이 방식이 무너진다는 것: 1Gbit 용량 칩이면 주소 라인 30개와 2^30개의 선택 라인이 필요한데, 속도를 희생하지 않는 디멀티플렉서의 크기는 입력 라인 수에 지수적으로 커진다. 게다가 주소 라인 30개에 30개의 임펄스를 동기적으로 전송하는 것은 15개보다 훨씬 어렵다(라인 길이·타이밍을 정확히 맞춰야 함).

그래서 DRAM은 셀을 행과 열로 조직한다(그림 2.7). 주소 라인 a0, a1이 **행 주소 선택(RAS)** 디멀티플렉서를 거쳐 한 행 전체의 주소 라인을 선택하고, 읽기 시 그 행 모든 셀의 내용이 **열 주소 선택(CAS)** 멀티플렉서에 공급되며, a2, a3에 따라 한 열의 내용이 데이터 핀으로 나온다. 이 과정이 여러 DRAM 칩에서 병렬로 일어나 데이터 버스 폭만큼의 비트를 만든다. 쓰기는 새 값을 데이터 버스에 올리고 RAS/CAS로 셀을 선택하면 저장된다.

또 다른 확장성 문제: 모든 RAM 칩에 주소 라인 30개를 연결하는 것도 불가능하다(핀은 귀한 자원). 메모리 컨트롤러가 8개의 RAM 모듈을 각각 30개 이상의 주소 라인으로 병렬 연결하면 주소 처리에만 240핀 이상이 필요하다. 이를 해결하려고 DRAM은 오래전부터 **주소 자체를 멀티플렉싱**한다: 주소를 두 부분으로 나눠, 먼저 행을 선택하는 비트(a0, a1)를 보내고(이 선택은 취소될 때까지 유지), 그다음 열 비트(a2, a3)를 보낸다. RAS와 CAS 신호가 언제 주소가 유효한지 알려준다. 외부 주소 라인이 절반으로 줄어드는 대신, 이 주소 멀티플렉싱은 뒤(2.2절)에서 볼 고유한 문제들을 만든다.

#### 2.1.4 결론 (2.1절 요약)

- 모든 메모리가 SRAM이 아닌 데는 이유가 있다(비용).
- 메모리 셀은 사용하려면 개별적으로 선택되어야 한다.
- 주소 라인 수가 메모리 컨트롤러·메인보드·DRAM 모듈·DRAM 칩의 비용을 직접 결정한다.
- 읽기/쓰기 결과가 나오기까지 시간이 걸린다.

SRAM은 보통 직접 주소 지정되며 CPU 캐시나 온다이(on-die) 용도로 쓰인다. SRAM 셀의 최고 속도는 들이는 노력에 따라 CPU 코어보다 약간 느린 정도에서 1~2자릿수 느린 정도까지 다양하다.

### 2.2 DRAM 접근 기술 상세

여기서는 현재 기술인 SDRAM(동기식 DRAM)과 그 후속 DDR에 집중한다(비동기 DRAM과 Rambus는 다루지 않음).

SDRAM은 이름대로 클럭에 맞춰 동작한다. 메모리 컨트롤러가 제공하는 클럭이 FSB의 속도를 결정한다. 2007년 기준 FSB 주파수는 800MHz, 1,066MHz, 1,333MHz가 쓰였고 1,600MHz가 발표됐다. 단, 실제 버스 클럭이 이만큼 높은 것은 아니다. 요즘 버스는 더블/쿼드 펌핑, 즉 사이클당 데이터를 2회 또는 4회 전송하므로, 쿼드 펌핑 200MHz 버스를 "유효" 800MHz로 광고하는 것이다.

SDRAM의 전송 단위는 64비트(8바이트)다. 따라서 FSB 전송률 = 8바이트 × 유효 버스 주파수(쿼드 펌핑 200MHz면 6.4GB/s). 하지만 이것은 결코 넘을 수 없는 버스트 속도이고, 실제 프로토콜에는 데이터를 전송하지 못하는 다운타임이 많다. 이 다운타임을 이해하고 최소화하는 것이 성능의 열쇠다.

#### 2.2.1 읽기 접근 프로토콜

그림 2.8은 DRAM 읽기 사이클을 세 단계로 보여준다. 모든 신호는 클럭(CLK) 상승 에지에서 읽는다.

1. 메모리 컨트롤러가 행 주소를 주소 버스에 올리고 RAS 신호를 내린다(RAS/CAS는 액티브 로우). RAM 칩이 해당 행을 래치하기 시작한다.
2. **t_RCD**(RAS-to-CAS Delay) 클럭 사이클 후 열 주소를 주소 버스에 올리고 CAS를 내린다. 주소의 두 부분이 같은 주소 버스로 전송되는 것을 볼 수 있다.
3. 주소 지정이 끝나도 RAM 칩은 준비 시간이 필요하다. 이 지연을 **CAS 지연(CL, CAS Latency)** 이라 한다. 그림 예시는 CL=2. CL은 메모리 컨트롤러·메인보드·DRAM 모듈 품질에 따라 다르고, 2.5 같은 반값도 있다(CL=2.5면 첫 데이터가 파란 구간의 첫 하강 에지에서 나옴).

이만큼 준비해 놓고 데이터 워드 하나만 전송하면 낭비다. 그래서 DRAM 모듈은 메모리 컨트롤러가 전송량을 지정하게 한다. 보통 2, 4, 8워드 중 선택하며, 이 덕에 새 RAS/CAS 시퀀스 없이 캐시 라인 전체를 채울 수 있다. 행 선택을 유지한 채 CAS 신호만 새로 보내는 것도 가능해서, 연속된 메모리 주소를 훨씬 빠르게 읽고 쓸 수 있다(행을 "열린" 상태로 유지). 행을 계속 열어두는 투기적 결정은 실제 애플리케이션에서 단점도 있다. 새 CAS 신호 전송은 RAM 모듈의 커맨드 레이트(Command Rate, Tx 표기; 1 또는 2, 고성능 모듈은 T1)의 제약만 받는다.

이 예시의 SDRAM은 사이클당 1워드를 내보낸다(1세대). DDR은 사이클당 2워드(상승·하강 에지 모두)를 전송해 전송 시간은 줄이지만 지연(latency)은 줄이지 못한다.

#### 2.2.2 프리차지와 활성화

그림 2.8은 전체 사이클의 일부일 뿐이다. 새 RAS 신호를 보내기 전에는 현재 래치된 행을 비활성화하고 새 행을 **프리차지**해야 한다(그림 2.9).

- 프리차지 명령은 전용 라인이 없고, Write Enable(WE)과 RAS를 동시에 내리는 식으로 발행하는 구현도 있다.
- 프리차지 발행 후 행을 선택할 수 있기까지 **t_RP**(Row Precharge time) 사이클이 걸린다.
- 그림 2.9에서 이 시간의 상당 부분이 데이터 전송과 겹치는 것은 좋지만, t_RP가 전송 시간보다 커서 다음 RAS가 한 사이클 지연된다.
- 타임라인을 이어 보면 다음 데이터 전송은 이전 전송이 끝나고 5사이클 뒤에 시작된다. 즉 데이터 버스가 7사이클 중 2사이클만 사용된다. 이걸 FSB 속도와 곱하면 800MHz 버스의 이론상 6.4GB/s가 1.8GB/s로 떨어진다. 이런 상황은 피해야 하며 6장의 기법이 이 수치를 끌어올리는 데 도움을 준다. 하지만 프로그래머 몫도 크다.

또 하나의 타이밍 값: SDRAM 모듈은 RAS 신호 후 다른 행을 프리차지하기 전까지 **t_RAS**(Active to Precharge delay)만큼 기다려야 한다. 이 값은 보통 t_RP의 2~3배로 꽤 크다. RAS 이후 CAS 하나만 따라오고 데이터 전송이 몇 사이클 만에 끝나는 경우 문제가 된다. 그림 2.9에서 첫 CAS 직전에 RAS가 있었고 t_RAS가 8사이클이라면, t_RCD + CL + t_RP 합이 7사이클뿐이므로 프리차지 명령이 1사이클 더 지연된다.

DDR 모듈 표기법 **w-x-y-z-T** (예: 2-3-2-8-T1):

| 기호 | 예 | 의미 |
|---|---|---|
| w | 2 | CAS 지연 (CL) |
| x | 3 | RAS-to-CAS 지연 (t_RCD) |
| y | 2 | RAS 프리차지 (t_RP) |
| z | 8 | Active to Precharge 지연 (t_RAS) |
| T | T1 | 커맨드 레이트 |

다른 타이밍 상수도 많지만, 실무에서는 이 다섯 가지면 모듈 성능을 판단하기에 충분하다. 컴퓨터 구입 시 FSB·SDRAM 모듈 속도와 함께 컴퓨터 속도를 결정하는 가장 중요한 요소들이다. BIOS에서 이 값들을 조정할 수 있는 경우도 있다(RAM 품질이 높으면 안정성을 해치지 않고 지연을 줄일 수 있기도 하다 — 단 자기 책임).

#### 2.2.3 리차징(리프레시)

DRAM 접근에서 흔히 간과되는 주제가 리프레시다. JEDEC 규격에 따라 DRAM 셀은 64ms마다 리프레시되어야 한다. DRAM 어레이가 8,192행이면 메모리 컨트롤러는 평균 7.8125µs마다 리프레시 명령을 발행해야 한다(명령을 큐잉할 수 있어 실제 최대 간격은 더 길 수 있다). 행이 리프레시되는 동안 그 부분에는 접근할 수 없다. 인용된 연구에 따르면 "놀랍게도 DRAM 리프레시 구성이 성능에 극적으로 영향을 줄 수 있다".

프로그래머가 리프레시 발행 시점에 대해 할 수 있는 일은 별로 없지만, 측정치를 해석할 때 이 DRAM 생명주기를 기억해야 한다. 리프레시 중인 행에서 중요한 워드를 읽어야 하면 프로세서가 꽤 오래 멈출 수 있다.

#### 2.2.4 메모리 타입

**SDR (Single Data Rate) SDRAM** (그림 2.10): 메모리 셀 어레이와 데이터 전송률이 같은 주파수 f로 동작한다. 예: 100MHz 어레이 → 100Mb/s(라인당) 전송. 전체 부품의 주파수를 올리는 것은 전력 소모(≒ 동적 용량 × 전압² × 주파수) 때문에 비싸고, 주파수를 올리면 전압도 올려야 해서 더 문제다.

**DDR1** (그림 2.11): 셀 어레이 주파수를 올리지 않고 사이클당 2비트(상승·하강 에지, "더블 펌핑")를 전송한다. 데이터 라인당 2비트를 담는 I/O 버퍼를 도입하고, 셀 어레이의 데이터 버스를 2라인으로 만들어(같은 열 주소로 두 셀을 병렬 접근) 구현한다. SDR은 주파수로 명명됐지만(PC100 = 100MHz SDR), DDR1은 마케팅상 전송 바이트 수로 명명됐다: 100MHz × 64bit × 2 = 1,600MB/s → "PC1600". 실제 개선은 2배일 뿐인데 이름은 16배 커 보인다.

**DDR2** (그림 2.12): 버스 주파수를 2배로 올린다(어레이 주파수는 유지). I/O 버퍼가 클럭 사이클마다 4비트를 받아 버스로 내보낸다. DDR2 모듈의 변경점은 DIMM의 I/O 버퍼만 고속으로 동작하게 만드는 것이라 에너지 비용이 크지 않다. FSB "유효" 주파수는 클럭의 양 에지 전송을 반영해 부풀린 수치다(133MHz 모듈 × 266MHz 버스 → FSB "533MHz").

DDR2 모듈 명명(표 2.1):

| 어레이 주파수 | 버스 주파수 | 데이터 전송률 | 이름(Rate) | 이름(FSB) |
|---|---|---|---|---|
| 133MHz | 266MHz | 4,256MB/s | PC2-4200 | DDR2-533 |
| 166MHz | 333MHz | 5,312MB/s | PC2-5300 | DDR2-667 |
| 200MHz | 400MHz | 6,400MB/s | PC2-6400 | DDR2-800 |
| 250MHz | 500MHz | 8,000MB/s | PC2-8000 | DDR2-1000 |
| 266MHz | 533MHz | 8,512MB/s | PC2-8500 | DDR2-1066 |

**DDR3** (그림 2.13): 전압을 1.8V에서 1.5V로 낮춰(전력이 전압 제곱에 비례하므로 이것만으로 30% 개선) 다이 축소·전기적 개선과 합치면 같은 주파수에서 절반의 전력, 또는 같은 전력으로 더 높은 주파수/2배 용량을 얻는다. 셀 어레이는 외부 버스의 1/4 속도로 돌고 I/O 버퍼는 8비트를 다룬다. 초기 DDR3는 CAS 지연이 DDR2보다 다소 높아, 대역폭이 지연보다 중요한 고주파 영역에서만 유리할 것으로 봤다.

DDR3 모듈 명명(표 2.2):

| 어레이 주파수 | 버스 주파수 | 데이터 전송률 | 이름(Rate) | 이름(FSB) |
|---|---|---|---|---|
| 100MHz | 400MHz | 6,400MB/s | PC3-6400 | DDR3-800 |
| 133MHz | 533MHz | 8,512MB/s | PC3-8500 | DDR3-1066 |
| 166MHz | 667MHz | 10,667MB/s | PC3-10667 | DDR3-1333 |
| 200MHz | 800MHz | 12,800MB/s | PC3-12800 | DDR3-1600 |
| 233MHz | 933MHz | 14,933MB/s | PC3-14900 | DDR3-1866 |

**DDR의 근본 문제와 FB-DRAM**: 버스 주파수가 올라가면 병렬 데이터 버스 구성이 어려워진다. DDR2 모듈은 핀이 240개인데, 데이터·주소 핀 배선을 거의 같은 길이로 맞춰야 하고, 한 버스에 모듈을 여러 개 데이지 체인으로 달면 신호가 점점 왜곡된다. DDR2는 채널당 2모듈, DDR3는 고주파에서 1모듈만 허용한다. 240핀 × 채널이면 노스브리지 하나가 2채널 이상을 감당하기 어렵다. 결과적으로 범용 메인보드는 DDR2/DDR3 모듈을 최대 4개까지만 달 수 있어 메모리 용량이 심하게 제한된다.

해법 하나는 앞서 본 CPU 통합 메모리 컨트롤러(AMD Opteron, Intel CSI)다. 프로세서당 적정량의 메모리를 연결할 수 있으면 도움이 되지만, NUMA와 그 부작용이 따라온다.

Intel의 대형 서버용 해법은 **FB-DRAM**(Fully Buffered DRAM)이다. DDR2와 같은 메모리 칩을 쓰되(그래서 생산이 저렴), 메모리 컨트롤러와의 연결을 병렬 버스 대신 **직렬 버스**로 한다(PATA→SATA, PCI/AGP→PCI Express와 같은 흐름). 직렬 버스는 훨씬 높은 주파수로 구동할 수 있어 직렬화의 손해를 상쇄하고도 남고, 완전 이중(full-duplex)이며, 차동(differential) 버스를 싸게 구현해 속도를 더 올릴 수 있다.

직렬 버스의 주요 효과:
1. 채널당 더 많은 모듈 사용 가능 (FB-DRAM은 채널당 8개)
2. 노스브리지/메모리 컨트롤러당 더 많은 채널 (6채널)
3. 완전 이중 2라인 설계
4. 차동 버스가 저렴해 속도 상승

FB-DRAM 모듈 핀은 69개(DDR2는 240개). 듀얼채널 노스브리지의 2 × 240핀 대비 6 × 69핀으로 6채널을 구동할 수 있고, 채널당 배선도 훨씬 단순해 메인보드 비용도 줄 수 있다.

| 항목 | DDR2 | FB-DRAM |
|---|---|---|
| 핀 | 240 | 69 |
| 채널 | 2 | 6 |
| 채널당 DIMM | 2 | 8 |
| 최대 메모리 (4GB 모듈 기준) | 16GB | 192GB |
| 처리량 (DDR2-800 기준) | ~10GB/s | ~40GB/s |

단점: 한 채널에 DIMM을 여러 개 달면 각 DIMM에서 신호가 (조금씩) 지연되어 지연 시간이 늘고, 직렬 버스 구동 칩이 고주파 때문에 상당한 에너지를 소모한다. 그래도 같은 주파수·같은 용량이면 FB-DRAM은 최대 4개의 DIMM이 각자 채널을 가질 수 있어 DDR2/DDR3보다 항상 빠를 수 있다. 대용량 메모리 시스템에는 범용 부품 기준 DDR에 답이 없다.

#### 2.2.5 결론 (2.2절 요약)

DRAM 접근은 결코 빠른 과정이 아니다 — 적어도 프로세서가 레지스터와 캐시에 접근하는 속도에 비하면. CPU와 메모리의 주파수 차이를 기억하라: 2.933GHz Intel Core 2와 1.066GHz FSB의 클럭비는 11:1이다(1.066GHz 버스는 쿼드 펌핑). 메모리 버스에서 1사이클 지연은 프로세서에게 11사이클 지연이다. 실제 DRAM은 대개 더 느려 지연은 더 크다.

타이밍 차트가 보여주듯 DRAM 모듈은 높은 지속 전송률이 가능하다. DRAM 행 전체를 스톨 없이 전송해 데이터 버스를 100% 점유할 수 있다(DDR이면 사이클당 64비트 워드 2개, DDR2-800 × 2채널이면 12.8GB/s).

하지만 의도적으로 그렇게 설계하지 않는 한 DRAM 접근은 순차적이지 않다. 불연속 메모리 영역을 쓰면 프리차지와 새 RAS 신호가 필요하고, 이때 속도가 떨어진다. 프리차지와 RAS를 빨리 보낼수록 실제 행 사용 시 페널티가 작아진다. 하드웨어·소프트웨어 프리페칭(6.3절)으로 타이밍에 겹침을 만들어 스톨을 줄일 수 있다. 프리페칭은 메모리 연산을 시간적으로 앞당겨 나중 시점의 경합도 줄여준다 — 한 라운드에서 생산한 데이터를 저장하는 것과 다음 라운드에 필요한 데이터를 읽는 것이 겹치는 흔한 문제를, 읽기를 앞당겨 해소하는 식이다.

### 2.3 그 밖의 메인 메모리 사용자

고성능 네트워크 카드나 대용량 저장 장치 컨트롤러는 필요한 데이터를 모두 CPU를 거쳐 나를 수 없어 DMA로 메인 메모리에 직접 읽고 쓴다(그림 2.1 구조에서는 사우스/노스브리지를 통해). USB 등 다른 버스도 DMA를 안 쓰더라도 FSB 대역폭을 소모한다(사우스브리지→노스브리지→FSB 경로).

DMA는 유익하지만 FSB 대역폭 경쟁을 심화시킨다. DMA 트래픽이 많을 때 CPU는 메인 메모리를 기다리며 평소보다 오래 스톨될 수 있다. 하드웨어를 잘 고르면 우회할 수 있다: NUMA 구조(그림 2.3)에서는 DMA의 영향을 받지 않는 노드의 메모리를 계산에 쓰거나, 노드마다 사우스브리지를 붙여 FSB 부하를 분산할 수 있다.

또 하나: 전용 비디오 RAM이 없는 저가 시스템은 메인 메모리 일부를 비디오 RAM으로 쓴다. 비디오 RAM 접근은 매우 빈번해서(1024×768, 16bpp, 60Hz면 94MB/s), 시스템 메모리는 그래픽카드 RAM과 달리 포트가 2개가 아니므로 시스템 성능과 특히 지연 시간에 큰 악영향을 준다. 성능이 중요하면 이런 시스템은 피하는 것이 좋다.
