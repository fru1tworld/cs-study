# USE 메서드

> 원문: [The USE Method](https://www.brendangregg.com/usemethod.html)
> 저자: Brendan Gregg | 정리일: 2026-07-10
>
> 원문 사이트에 별도의 허용적 라이선스가 없어, 원문의 섹션 구조를 빠짐없이 따라가는 상세 정리 노트로 작성했다. 수치·표·용어는 원문 그대로다.

![Boeing 707 Emergency Checklist (1969)](https://www.brendangregg.com/USEmethod/B707_emergency01_crop0.jpg)

USE(Utilization, Saturation, Errors) 메서드는 어떤 시스템이든 성능을 분석할 수 있는 방법론이다. 체크리스트를 만들도록 이끌며, 서버 분석에서는 이 체크리스트로 리소스 병목이나 에러를 빠르게 찾아낼 수 있다. 주어진 지표(부분적인 답)에서 출발해 거꾸로 추적하는 대신, 먼저 질문을 던지고 그 답을 찾아 나간다.

원문 사이트에는 운영체제별 USE 체크리스트가 정리되어 있다: [Linux](https://www.brendangregg.com/USEmethod/use-linux.html), [Solaris](https://www.brendangregg.com/USEmethod/use-solaris.html), [Mac OS X](https://www.brendangregg.com/USEmethod/use-macosx.html), [FreeBSD](https://www.brendangregg.com/USEmethod/use-freebsd.html), [Unix 7th](https://www.brendangregg.com/USEmethod/use-unix7th.html), [SmartOS](https://www.brendangregg.com/USEmethod/use-smartos.html). 이들에서 자동 생성한 [Rosetta Stone of Performance Checklists](https://www.brendangregg.com/USEmethod/use-rosetta.html)도 있다. 자기 환경에 맞게 도구를 추가해 커스터마이즈하면 된다.

## 도입

심각한 성능 문제가 터졌고 서버가 원인으로 의심된다. 무엇부터 확인해야 할까?

시작이 가장 어렵다. Gregg는 중요한 영역을 빠뜨리지 않으면서 흔한 성능 문제를 빠르게 해결하는 법을 가르치려고 USE 메서드를 만들었다. 비행 매뉴얼의 비상 체크리스트처럼 단순하고, 직관적이고, 완전하고, 빠르게 쓸 수 있도록 설계했다. 기업 환경, 강의(학습 도구), 그리고 클라우드 컴퓨팅 환경에서 수없이 성공적으로 쓰였다.

USE 메서드는 세 가지 지표 유형과 복잡한 시스템에 접근하는 전략으로 이루어진다. 저자의 경험으로는 서버 문제의 약 80%를 5%의 노력으로 해결하며, 서버가 아닌 시스템에도 적용할 수 있다. 다만 이것은 더 큰 방법론 도구 상자의 도구 하나로 보아야 한다. 해결하지 못하는 문제 유형도 많으며, 그런 문제에는 다른 방법과 더 긴 시간이 필요하다.

## 요약

USE 메서드는 한 문장으로 요약된다.

**모든 리소스의 사용률(utilization), 포화도(saturation), 에러(errors)를 확인한다.**

성능 조사 초기에 시스템 전반의 병목을 찾는 용도로 쓴다.

용어 정의:

- **리소스(resource)**: 서버의 모든 물리적 기능 구성 요소(CPU, 디스크, 버스, ...) [1]
- **사용률(utilization)**: 리소스가 작업을 처리하느라 바빴던 평균 시간 [2]
- **포화도(saturation)**: 리소스가 처리하지 못해 대기(주로 큐잉)하는 추가 작업의 정도
- **에러(errors)**: 에러 이벤트의 횟수

[1] 일부 소프트웨어 리소스나 소프트웨어가 부과한 제한(리소스 컨트롤)도 함께 고려하면 유용할 수 있다.

[2] 사용률에는 "사용 중인 리소스의 비율"이라는 다른 정의도 있다. 그 정의에서는 100% 사용률이 "더 이상 작업을 받을 수 없음"을 뜻하는데, 위의 "바쁨(busy)" 정의와는 다르다.

지표는 보통 다음과 같이 표현한다.

- 사용률: 시간 구간 대비 백분율. 예: "디스크 하나가 90% 사용률로 돌고 있다."
- 포화도: 큐 길이. 예: "CPU들의 평균 실행 큐(run queue) 길이가 4다."
- 에러: 스칼라 카운트. 예: "이 네트워크 인터페이스에서 late collision이 50번 발생했다."

에러는 성능을 저하시킬 수 있고, 실패가 복구 가능한 유형이면 즉시 눈에 띄지 않을 수 있으므로 반드시 조사해야 한다. 실패 후 재시도되는 연산, 이중화 풀에서 고장 난 장치가 여기에 해당한다.

## 사용률이 낮으면 포화가 없다는 뜻일까?

짧은 순간의 높은 사용률 버스트가 포화와 성능 문제를 일으킬 수 있다. 긴 구간으로 평균 낸 사용률이 *낮더라도* 그렇다. 직관에 어긋날 수 있는 지점이다.

저자가 겪은 사례: 모니터링 도구상 CPU 사용률이 80%를 넘은 적이 없는데도 고객이 CPU 포화(지연)로 고생하고 있었다. 도구는 5분 평균을 보고하고 있었고, 그 사이 CPU 사용률은 몇 초씩 100%를 찍고 있었다.

## 리소스 목록

먼저 순회할 리소스 목록이 필요하다. 서버용 일반 목록:

- CPU: 소켓, 코어, 하드웨어 스레드(가상 CPU)
- 메모리: 용량
- 네트워크 인터페이스
- 스토리지 장치: I/O, 용량
- 컨트롤러: 스토리지, 네트워크 카드
- 인터커넥트: CPU, 메모리, I/O

일부 구성 요소는 두 종류의 리소스다. 예를 들어 스토리지 장치는 서비스 요청 리소스(I/O)이면서 용량 리소스(수용량)이기도 하며, 두 유형 모두 시스템 병목이 될 수 있다. 요청 리소스는 요청을 큐에 넣었다가 처리하는 *큐잉 시스템*으로 정의할 수 있다.

하드웨어 캐시(MMU TLB/TSB, CPU 캐시 등)처럼 목록에서 뺀 물리 구성 요소도 있다. USE 메서드는 높은 사용률이나 포화에서 성능이 저하되어 병목이 되는 리소스에 가장 효과적인데, 캐시는 사용률이 높을수록 오히려 성능을 *개선*한다. 캐시 히트율 등은 USE 메서드 이후, 즉 시스템 전반의 병목을 배제한 다음 확인하면 된다. 어떤 리소스를 포함할지 애매하면 일단 포함하고 지표가 얼마나 잘 동작하는지 보라.

## 기능 블록 다이어그램

리소스를 순회하는 또 다른 방법은 시스템의 [기능 블록 다이어그램(Functional Block Diagram)](http://en.wikipedia.org/wiki/Function_block_diagram)을 찾거나 직접 그리는 것이다. 구성 요소 간 관계도 보여주므로 데이터 흐름의 병목을 찾을 때 매우 유용하다. [Sun Fire V480 Guide](http://docs.oracle.com/cd/E19095-01/sfv480.srvr/816-0904-10/816-0904-10.pdf) 82쪽의 예:

![Sun Fire V480 기능 블록 다이어그램](https://www.brendangregg.com/USEmethod/v480.png)

이런 다이어그램은 구하기 어려울 수 있다. 실제로 하드웨어를 만드는 하드웨어 엔지니어가 최고의 자료원이고, 아니면 직접 그려도 된다.

여러 버스의 사용률을 파악하는 과정에서 기능 다이어그램의 각 버스에 최대 대역폭을 표기해 두면, 측정을 한 번도 하기 전에 시스템 병목 후보를 식별할 수 있는 다이어그램이 된다. (물리 구성 요소를 바꿀 수 있는 하드웨어 제품 설계 단계에서 유용한 연습이다.)

## 인터커넥트

CPU·메모리·I/O 인터커넥트는 자주 간과된다. 다행히 시스템 병목이 되는 일은 흔치 않다. 불행히도 병목이 되면 손쓰기 어렵다(메인보드를 업그레이드하거나 부하를 줄이는 정도 — 예: "zero copy" 프로젝트는 메모리 버스 부하를 줄인다). USE 메서드를 쓰면 최소한 그동안 고려하지 않던 것, 즉 인터커넥트 성능을 인식하게 된다. 저자가 USE 메서드로 인터커넥트 문제를 찾아낸 사례는 [Analyzing the HyperTransport](http://www.brendangregg.com/blog/2009-09-22/analyzing-the-hypertransport.html)를 보라.

## 지표

리소스 목록이 있으면 지표 유형(사용률, 포화도, 에러)을 각각 대입한다. 원문에서는 아래 표의 지표 칸을 마우스 오버로 가려 두고 독자가 먼저 채워 보게 한다. 일반적인 Unix/Linux 용어 기준의 답:

| 리소스 | 유형 | 지표 |
|---|---|---|
| CPU | 사용률 | CPU 사용률(CPU별 또는 시스템 전체 평균) |
| CPU | 포화도 | 실행 큐(run-queue) 길이 또는 스케줄러 지연 |
| 메모리 용량 | 사용률 | 가용 여유 메모리(시스템 전체) |
| 메모리 용량 | 포화도 | 익명 페이징(anonymous paging) 또는 스레드 스와핑("페이지 스캐닝"도 포함 가능) |
| 네트워크 인터페이스 | 사용률 | RX/TX 처리량 / 최대 대역폭 |
| 스토리지 장치 I/O | 사용률 | 장치 busy 백분율 |
| 스토리지 장치 I/O | 포화도 | 대기 큐 길이 |
| 스토리지 장치 I/O | 에러 | 장치 에러("soft", "hard", ...) |

시간 요소는 생략했다. 이 지표들은 구간별 평균이거나 카운트다. 수집 방법도 생략했는데, 자기만의 체크리스트에는 어떤 OS 도구나 모니터링 소프트웨어를 쓰고 어떤 통계를 읽을지를 포함하라. 구할 수 없는 지표에는 "?"를 적어라. 그러면 따라가기 쉽고 빠르면서 시스템에 대해 가능한 한 완전한 체크리스트가 만들어진다.

## 더 어려운 지표

더 어려운 조합들(역시 먼저 스스로 생각해 볼 것):

| 리소스 | 유형 | 지표 |
|---|---|---|
| CPU | 에러 | 예: 정정 가능한 CPU 캐시 ECC 이벤트, 고장(faulted) CPU (OS+하드웨어가 지원하는 경우) |
| 메모리 용량 | 에러 | 예: 실패한 malloc() (보통 물리 메모리가 아니라 가상 메모리 고갈 때문이긴 하다) |
| 네트워크 | 포화도 | 포화 관련 NIC/OS 이벤트. 예: "dropped", "overruns" |
| 스토리지 컨트롤러 | 사용률 | 컨트롤러에 따라 다름. 최대 IOPS나 처리량이 있으면 현재 활동과 비교 |
| CPU 인터커넥트 | 사용률 | 포트별 처리량 / 최대 대역폭 (CPU 성능 카운터) |
| 메모리 인터커넥트 | 포화도 | 메모리 스톨 사이클, 높은 CPI (CPU 성능 카운터) |
| I/O 인터커넥트 | 사용률 | 버스 처리량 / 최대 대역폭 (하드웨어에 성능 카운터가 있을 수 있음. 예: Intel "uncore" 이벤트) |

이런 지표는 OS에 따라 측정이 더 어려워지고, 저자는 직접 소프트웨어를 짜야 할 때가 많았다(예: [Analyzing the HyperTransport](http://www.brendangregg.com/blog/2009-09-22/analyzing-the-hypertransport.html)의 "amd64htcpu" 스크립트).

모든 조합을 반복하고 각 지표의 수집 방법을 적으면 약 30개의 지표 목록이 나온다. 일부는 측정할 수 없고 일부는 측정이 까다롭다. 다행히 가장 흔한 문제는 대개 쉬운 지표(CPU 포화, 메모리 용량 포화, 네트워크 인터페이스 사용률, 디스크 사용률)에서 발견되므로 이것들을 먼저 확인하면 된다.

## 실전에서

OS의 모든 조합에서 지표를 읽는 것은 시간이 많이 든다. 특히 버스와 인터커넥트 지표까지 가면 더 그렇다. 부분 집합(CPU, 메모리 용량, 스토리지 용량, 스토리지 장치 I/O, 네트워크 인터페이스)만 확인할 시간밖에 없을 수도 있는데, 이것도 생각보다 낫다. USE 메서드 덕분에 무엇을 확인하지 않았는지 인식하게 되었기 때문이다. 한때 unknown-unknown이던 것이 이제 known-unknown이 되었다. 회사 차원에서 성능 문제의 근본 원인을 반드시 찾아야 하는 순간이 오면, 더 철저한 분석을 위한 추가 작업의 할 일 목록을 이미 갖고 있는 셈이다.

OS에 지표가 계속 추가되면서 쉽게 확인할 수 있는 지표의 범위가 넓어지기를, 그리고 성능 모니터링 소프트웨어가 USE 메서드 마법사를 추가해 이 작업을 대신해 주기를 저자는 기대한다.

## 소프트웨어 리소스

일부 소프트웨어 리소스도 비슷하게 볼 수 있다. 보통 애플리케이션 전체가 아니라 소프트웨어의 작은 구성 요소에 적용된다.

- **뮤텍스 잠금**: 사용률은 잠금이 유지된 시간, 포화도는 잠금을 기다리며 큐잉된 스레드로 정의할 수 있다.
- **스레드 풀**: 사용률은 스레드가 작업을 처리하느라 바빴던 시간, 포화도는 스레드 풀의 처리를 기다리는 요청 수.
- **프로세스/스레드 용량**: 시스템의 프로세스·스레드 수 제한 대비 현재 사용량이 사용률, 할당 대기가 포화, 할당 실패(예: "cannot fork")가 에러.
- **파일 디스크립터 용량**: 위와 같되 파일 디스크립터 대상.

이 유형에 너무 매달리지 마라. 지표가 잘 동작하면 쓰고, 아니면 소프트웨어는 다른 방법론(예: 지연 시간 기반)에 맡겨라.

## 해석 가이드

USE 메서드는 어떤 지표를 쓸지 알려준다. OS에서 읽는 법을 익힌 다음 과제는 현재 값의 해석이다. 어떤 것은 해석이 명확하고 문서화도 잘 되어 있지만, 어떤 것은 그렇지 않고 워크로드 요구사항이나 기대치에 따라 달라진다.

지표 유형별 일반 해석 지침:

- **사용률**: 100% 사용률은 대개 병목의 신호다(포화도와 그 영향을 확인해 검증하라). 높은 사용률(예: 70% 초과)도 두 가지 이유로 문제가 되기 시작할 수 있다.
  - 비교적 긴 구간(수 초~수 분)으로 측정한 사용률에서는, 이를테면 70%라는 총 사용률이 짧은 100% 버스트를 가릴 수 있다.
  - 하드 디스크 같은 일부 시스템 리소스는 작업 도중에 더 높은 우선순위의 작업을 위해서라도 중단될 수 없다. 사용률이 70%를 넘으면 큐잉 지연이 더 잦고 뚜렷해질 수 있다. 거의 언제든 중단("선점")될 수 있는 CPU와 대비된다.
- **포화도**: 0이 아닌 어떤 수준의 포화도든 문제일 수 있다. 대기 큐의 길이나 큐에서 기다린 시간으로 측정한다.
- **에러**: 0이 아닌 에러 카운터는 조사할 가치가 있다. 특히 성능이 나쁜 상태에서 계속 증가하고 있다면 더 그렇다.

부정 사례(낮은 사용률, 포화 없음, 에러 없음)의 해석은 쉽다. 그리고 이는 생각보다 유용하다 — 조사 범위를 좁히면 문제 영역에 빠르게 집중할 수 있다.

## 클라우드 컴퓨팅

클라우드 컴퓨팅 환경에서는 하나의 시스템을 공유하는 테넌트를 제한·조절(throttle)하는 소프트웨어 리소스 컨트롤이 있을 수 있다. 메모리·CPU·네트워크·스토리지 I/O에 대한 하이퍼바이저 또는 컨테이너(cgroup) 제한이 그 예다. 네트워크 처리량처럼 외부 하드웨어가 제한을 가할 수도 있다. 이런 리소스 제한 각각도 물리 리소스처럼 USE 메서드로 검토할 수 있다.

예를 들어 저자의 환경에서 "메모리 용량 사용률"은 테넌트의 메모리 사용량 대 메모리 상한(cap)이 될 수 있다. "메모리 용량 포화"는 전통적인 Unix 페이지 스캐너가 놀고 있어도 익명 페이징 활동으로 관측할 수 있다.

## 전략

USE 메서드의 흐름도. 에러는 보통 더 빠르고 해석하기 쉬우므로 약간의 최적화로서 사용률·포화도보다 먼저 확인해도 된다.

![USE 메서드 흐름도](https://www.brendangregg.com/USEmethod/usemethod_flow.png)

USE 메서드는 시스템 병목일 가능성이 높은 문제를 찾아낸다. 다만 시스템은 성능 문제를 동시에 여러 개 겪을 수 있어, 처음 찾은 것이 *하나의* 문제일지언정 *바로 그* 문제가 아닐 수 있다. 발견할 때마다 다른 방법론으로 조사한 뒤, 필요하면 USE 메서드로 돌아와 나머지 리소스를 계속 순회한다.

후속 분석 전략으로는 워크로드 특성화(workload characterization)와 드릴다운 분석이 있다. 이를 마치면 시정 조치가 "가해지는 부하 조정"인지 "리소스 자체 튜닝"인지에 대한 근거가 생긴다.

## 아폴로

USE 메서드는 서버 너머에도 적용할 수 있다. 재미있는 예로 저자는 자신이 전혀 전문성이 없고 어디서 시작할지도 모르는 시스템, 아폴로 달 착륙선(LM)의 유도 시스템을 골랐다.

첫 단계는 리소스 목록, 더 좋게는 기능 블록 다이어그램을 찾는 것이다. "Lunar Module - LM10 Through LM14 Familiarization Manual"(1969)에서 찾은 다이어그램:

![아폴로 LM 유도 시스템 블록 다이어그램](https://www.brendangregg.com/USEmethod/apollo_LM_guidance.png)

일부 구성 요소는 사용률·포화 특성이 없을 수 있다. 한 번 순회한 뒤 관련 구성 요소만 남겨 다시 그리면 된다. (저자라면 메모리의 "erasable storage" 구역, "core set area"와 "vac area" 레지스터도 추가하겠다고 한다.)

아폴로 유도 컴퓨터(AGC) 자체부터 보면, 각 지표에 대해 LM 문서들을 뒤져 본 결과:

- **AGC 사용률**: 작업(job)을 수행한 CPU 사이클 수("DUMMY JOB" 제외)를 클럭 속도(2.048 MHz)로 나눈 값으로 정의할 수 있다. 당시에도 잘 이해되던 지표로 보인다.
- **AGC 포화도**: 프로그램 상태를 저장하는 7세트의 레지스터인 "core set area"의 작업 수로 정의할 수 있다. 이는 더 높은 우선순위 작업의 인터럽트가 오면 "EXECUTIVE" 프로그램(요즘으로 치면 "커널")이 작업을 중단시킬 수 있게 해 준다. 소진되면 포화 상태에서 에러 상태로 넘어가 AGC가 1202 "EXECUTIVE OVERFLOW-NO CORE SETS" 알람을 보고한다.
- **AGC 에러**: 알람이 다수 정의되어 있다. 1202 외에 1203 "WAITLIST OVERFLOW-TOO MANY TASKS" 알람도 있는데, 이는 다른 유형의 성능 문제다. 정상적인 작업 스케줄링으로 돌아가기 전에 처리되는 시한(timed) 태스크가 너무 많은 것이다. 1202처럼 WAITLIST 길이를 포화 지표로 정의해 두면 오버플로와 에러가 나기 전에 포화를 측정할 수 있다.

우주 애호가라면 익숙할 내용: 1201("NO VAC AREAS")과 1202 알람은 [아폴로 11호 하강](http://en.wikipedia.org/wiki/Apollo_Guidance_Computer#PGNCS_trouble) 중에 실제로 발생한 것으로 유명하다. ("VAC"은 "[vector accumulator](https://www.brendangregg.com/USEmethod/AGC_core_and_vac.png)"의 줄임말로, 벡터 값을 다루는 작업용 추가 저장 공간이다. 위키백과의 "vacant"라는 설명은 틀렸을 것이라고 저자는 본다.)

아폴로 11호의 1201 알람이 있으니 워크로드 특성화 같은 다른 방법론으로 분석을 이어갈 수 있다. 워크로드는 대부분 인터럽트로 가해지며, 그중 다수가 기능 다이어그램에 보인다. LM이 하강 중인데도 사령선(CM) 추적용 랑데부 레이더가 AGC에 인터럽트로 작업을 밀어 넣고 있었다. 불필요한 작업(또는 저우선순위 작업 — 필요시 LM AGC가 즉시 중단 궤도와 CM 랑데부를 계산할 수 있도록 레이더 업데이트 일부는 바람직했을 수 있다)을 찾아낸 사례다.

더 어려운 예로 랑데부 레이더 자체를 리소스로 보면, 에러가 가장 [식별하기 쉽다](https://www.brendangregg.com/USEmethod/apollo_LM_radarsignalflow.png). "DATA NO GOOD", "NO TRACK", "SHAFT- AND TRUNNION-AXIS ERROR" 신호의 세 유형이 있다. 사용률은 더 어려운데, 하나는 구동 모터의 사용률 — 각도 명령에 응답하느라 바빴던 시간(기능 다이어그램의 "COUPLING DATA UNIT" 경유) — 로 정의할 수 있다. 구동 모터나 반환되는 레이더 데이터에 포화 특성이 있는지는 LM 문서를 더 읽어야 한다.

이 방법론 덕분에 짧은 시간 안에, 어디서 시작할지 전혀 모르는 상태에서 찾아보고 조사할 구체적인 지표가 있는 상태가 되었다.

## 다른 방법론

USE 메서드가 서버 문제의 80%를 찾는다면, 지연 시간 기반 방법론(예: [Method R](http://carymillsap.blogspot.com/2009/11/why-we-made-method-r.html))은 전체 문제의 100%에 다가갈 수 있다. 그러나 소프트웨어 내부에 익숙하지 않으면 시간이 훨씬 오래 걸릴 수 있다. 이런 방법론은 이미 그런 지식을 갖춘 데이터베이스 관리자나 애플리케이션 개발자에게 더 맞을 수 있다. USE 메서드는 OS와 하드웨어가 책임 범위이자 전문 분야인 주니어·시니어 시스템 관리자에게 더 적합하며, 다른 직군도 시스템 상태를 빠르게 점검하고 싶을 때 쓸 수 있다.

## 도구 기반 방법(Tools Method)

USE 메서드와 비교하기 위해 도구 기반 접근("Tools Method")을 기술하면:

1. 사용 가능한 성능 도구를 나열한다(필요하면 설치·구매).
2. 각 도구의 유용한 지표를 나열한다.
3. 각 지표의 가능한 해석 규칙을 나열한다.

결과는 어떤 도구를 돌리고 어떤 지표를 읽고 어떻게 해석할지 알려주는 처방적 체크리스트다. 꽤 효과적일 수 있지만, 문제는 오로지 사용 가능한(또는 알려진) 도구에만 의존한다는 점이다. 그런 도구는 시스템의 불완전한 그림만 제공할 수 있고, 사용자는 자신의 시야가 불완전하다는 것조차 모른다 — 그래서 문제는 남는다.

반면 USE 메서드는 시스템 리소스를 순회하며 물어야 할 질문의 완전한 목록을 먼저 만들고, 그 다음에 답해 줄 도구를 찾는다. 더 완전한 그림이 만들어지고, 알 수 없는 영역은 문서화되어 그 존재가 인지된다("known unknowns"). USE에 기반해 어떤 도구를 돌리고(있다면) 어떤 지표를 읽고 어떻게 해석할지의 체크리스트를 비슷하게 만들 수 있다.

또 다른 문제는 수많은 도구를 순회하는 일이 목표 — 병목 찾기 — 에서 주의를 빼앗을 수 있다는 점이다. USE 메서드는 도구와 지표가 감당 못 할 만큼 많아도 병목과 에러를 효율적으로 찾는 전략을 제공한다.

## 결론

USE 메서드는 시스템 상태를 완전하게 점검해 흔한 병목과 에러를 찾아내는 단순한 전략이다. 조사 초기에 투입해 문제 영역을 빠르게 식별하고, 필요하면 다른 방법론으로 더 깊이 파고들 수 있다. USE의 강점은 속도와 가시성이다. 모든 리소스를 고려하므로 문제를 놓칠 가능성이 낮다. 다만 특정 유형의 문제 — 병목과 에러 — 만 찾으므로 더 큰 도구 상자의 도구 하나로 보아야 한다.

스레드 기반의 상보적 방법론인 [TSA Method](https://www.brendangregg.com/tsamethod.html)도 참고하라.
