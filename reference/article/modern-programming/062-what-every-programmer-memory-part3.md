## 4장 가상 메모리

프로세서의 가상 메모리(VM) 서브시스템은 각 프로세스에 자신만 시스템을 독점하는 것처럼 보이는 가상 주소 공간을 제공한다. 가상 메모리의 일반적 장점은 다른 문헌에 잘 정리되어 있으므로, 이 장은 구현 세부사항과 그로 인한 비용에 집중한다.

가상 주소 공간은 CPU의 **MMU**(Memory Management Unit)가 구현한다. OS가 페이지 테이블 자료구조를 채워두면 나머지 변환 작업은 대부분 CPU가 스스로 처리한다. MMU에 들어가는 입력은 가상 주소이고, 32비트 시스템은 32비트 값, 64비트 시스템은 64비트 값을 쓴다. x86/x86-64는 세그먼트라는 한 단계의 간접 참조를 추가로 두지만(모든 논리 주소에 오프셋을 더하는 정도), 성능 관점에서 프로그래머가 신경 쓸 부분은 아니다.

### 4.1 가장 단순한 주소 변환

가상 주소를 물리 주소로 바꾸는 과정이 핵심이다. MMU는 페이지 단위로 주소를 재매핑할 수 있으며, 캐시 라인 주소 지정과 마찬가지로 가상 주소를 여러 부분으로 나눠 각각 다른 테이블의 인덱스로 쓴다.

가장 단순한 모델은 테이블이 한 단계뿐이다. 가상 주소의 상위 비트로 **페이지 디렉터리**의 항목을 고르고, 그 항목이 물리 페이지의 주소를 가리킨다(여러 디렉터리 항목이 같은 물리 주소를 가리킬 수도 있다). 이 페이지 디렉터리 주소와 가상 주소 하위 비트(오프셋)를 합치면 완전한 물리 주소가 나온다. 페이지 디렉터리 자료구조 자체는 메인 메모리에 저장되고, OS는 연속된 물리 메모리를 할당해 그 시작 주소를 특수 레지스터에 담아둔다.

x86의 4MB 페이지를 예로 들면, 가상 주소의 오프셋 부분이 22비트(4MB 전체를 담기에 충분)이고 나머지 10비트가 페이지 디렉터리의 1,024개 항목 중 하나를 고른다. 각 항목은 4MB 페이지의 10비트 베이스 주소를 담고 있고, 이를 오프셋과 합쳐 32비트 전체 주소를 만든다.

### 4.2 다단계 페이지 테이블

4MB 페이지는 일반적이지 않다. OS의 많은 작업이 페이지 경계 정렬을 요구하기 때문에 메모리 낭비가 크다. 32비트 머신에서 흔히 쓰는(64비트 머신에서도 자주 쓰이는) 4KB 페이지라면 가상 주소의 오프셋 부분은 12비트에 불과하고, 남은 20비트가 디렉터리 선택에 쓰인다. 2^20개 항목을 가진 단일 테이블은 비현실적이다 — 항목 하나가 4바이트뿐이어도 테이블 자체가 4MB나 되고, 프로세스마다 별도 페이지 디렉터리를 둔다면 이 테이블들만으로 시스템 물리 메모리의 상당 부분이 묶여 버린다.

해법은 **다단계 페이지 테이블**이다. 여러 레벨이 모여 거대하고 성긴(sparse) 페이지 디렉터리를 이루므로, 실제로 쓰이지 않는 주소 공간 영역은 메모리를 할당받을 필요가 없다. 오늘날 가장 복잡한 페이지 테이블 구조는 4단계로 이뤄진다. 가상 주소는 최소 다섯 부분(레벨4~1 인덱스와 오프셋)으로 나뉜다. 레벨4 디렉터리는 CPU의 특수 레지스터로 참조되고, 레벨4부터 레벨2까지의 항목은 그다음 하위 레벨 디렉터리를 가리킨다. 항목이 비어 있다고 표시되면 하위 디렉터리를 가리킬 필요가 없으므로 트리가 성기고 컴팩트하게 유지된다. 레벨1 디렉터리 항목만 실제 물리 주소(와 접근 권한 등 부가 정보)를 담는다.

물리 주소를 찾으려면 프로세서가 (보통 레지스터에 저장된) 최상위 디렉터리 주소부터 시작해, 가상 주소의 해당 인덱스로 항목을 골라 다음 레벨 디렉터리 주소를 얻고, 이를 반복해 레벨1까지 내려간다. 마지막에 얻은 값에 가상 주소의 페이지 오프셋을 더하면 물리 주소가 완성된다. 이 과정을 **페이지 트리 워크**(page tree walk)라 부른다. x86/x86-64 같은 프로세서는 이를 하드웨어로 처리하고, 다른 아키텍처는 OS의 도움을 받는다.

시스템에서 실행되는 프로세스마다 자신만의 페이지 테이블 트리가 필요할 수 있다(트리를 부분적으로 공유하는 것도 가능하지만 예외적). 그래서 트리가 차지하는 메모리를 최소화하는 것이 성능과 확장성에 좋고, 이상적으로는 가상 주소 공간에서 사용 중인 메모리를 서로 가깝게 모아두는 것이다(실제 물리 주소가 어딘지는 상관없다). 작은 프로그램이라면 레벨2·3·4 각각 디렉터리 하나씩과 레벨1 디렉터리 몇 개로 충분할 수 있다. x86-64에서 4KB 페이지와 디렉터리당 512항목을 가정하면 레벨별 디렉터리 하나씩(총 4개)으로 2MB를 주소 지정할 수 있고, 레벨2~4 디렉터리 하나와 레벨1 디렉터리 512개로 1GB의 연속 메모리를 커버할 수 있다.

다만 모든 메모리를 연속으로 할당할 수 있다는 가정은 지나치게 단순하다. 유연성을 위해 스택과 힙 영역은 대개 주소 공간의 정반대 끝에 배치되어(각자 필요한 만큼 자라날 수 있도록) 최소 두 개의 레벨2 디렉터리가 필요해진다. 게다가 보안 목적으로 실행 파일의 각 부분(코드, 데이터, 힙, 스택, DSO 즉 공유 라이브러리)이 무작위화된 주소에 매핑되면서 프로세스의 메모리 영역이 가상 주소 공간 전체에 흩어지게 된다. 랜덤화 비트 수를 제한해 범위를 좁힐 수는 있지만, 대부분의 경우 레벨2·3 디렉터리를 하나 둘로 압축하기는 어렵다. 보안보다 성능이 훨씬 중요하다면 랜덤화를 끌 수 있고, 이 경우 OS는 대개 모든 DSO를 가상 메모리에서 연속적으로 적재한다.

### 4.3 페이지 테이블 접근 최적화

페이지 테이블 자료구조는 모두 메인 메모리에 있고, OS가 여기서 테이블을 만들고 갱신한다. 프로세스 생성이나 페이지 테이블 변경 시 CPU에 통지된다. 가상 주소를 물리 주소로 바꿀 때마다 최소한 레벨마다 디렉터리 하나씩을 거치므로, 프로그램의 단일 메모리 접근 하나에 최대 4번의 메모리 접근이 필요할 수 있어 매우 느리다. 디렉터리 항목을 일반 데이터처럼 취급해 L1d·L2 등에 캐싱하는 것도 가능하지만, 그래도 너무 느리다 — 4단계 페이지 테이블이라면 각 절대 주소 계산에 (이전 조회 결과에 의존해 병렬화도 안 되는) L1d 접근이 4번 필요해 최소 12사이클, 여기에 L1d 미스 확률까지 더해지면 파이프라인이 도저히 숨길 수 없는 비용이 된다.

그래서 CPU 설계자들은 디렉터리 항목을 그냥 캐싱하는 대신, **물리 페이지 주소 계산 결과 자체**를 캐싱하는 방식을 오래전부터 써왔다. 가상 주소의 페이지 오프셋 부분은 이 계산에 관여하지 않으므로, 나머지 가상 주소 부분만 캐시 태그로 쓰인다(페이지 크기에 따라 수백~수천 개의 명령어/데이터 객체가 같은 태그, 즉 같은 물리 주소 프리픽스를 공유하게 됨). 이 계산 결과를 담는 캐시가 바로 **TLB**(Translation Look-Aside Buffer)다.

TLB는 매우 빨라야 하므로 보통 작은 캐시이고, 다른 캐시들처럼 다단계(상위 레벨일수록 크고 느림)로 구현된다. L1TLB의 작은 크기는 완전 연관(fully associative) + LRU 축출 정책으로 보완하는 경우가 많다. 최근에는 이 캐시가 커지면서 세트 연관 방식으로 바뀌어, 가장 오래된 항목이 항상 축출 대상은 아니게 되었다. TLB 태그가 가상 주소의 일부이므로, 태그가 일치하면 캐시된 값에 가상 주소의 페이지 오프셋을 더해 물리 주소를 빠르게 계산한다 — 절대 주소를 쓰는 모든 명령어, 그리고 물리 주소를 키로 쓰는 일부 L2 조회에서 이 물리 주소가 즉시 필요하므로 매우 빨라야 한다. TLB 조회가 실패(미스)하면 프로세서는 상당히 비용이 큰 페이지 트리 워크를 수행해야 한다.

소프트웨어·하드웨어 프리페칭이 다른 페이지의 주소를 미리 가져올 수도 있지만, 하드웨어 프리페칭이 유효하지 않은 페이지 트리 워크를 유발할 수 있으므로 TLB 항목의 하드웨어 프리페칭은 허용되지 않는다. 그래서 TLB 프리페칭은 프로그래머가 명시적 프리페치 명령으로만 할 수 있다. TLB도 명령어용(ITLB)과 데이터용(DTLB)으로 나뉘며, L2TLB 같은 상위 레벨은 대개 통합(unified) 캐시다.

#### 4.3.1 TLB 사용의 주의점

TLB는 **프로세서 코어 전역 자원**이다. 코어에서 실행되는 모든 스레드와 프로세스가 같은 TLB를 쓴다. 가상→물리 주소 변환은 어떤 페이지 테이블 트리가 설치되어 있느냐에 좌우되므로, 페이지 테이블이 바뀌면 CPU는 캐시된 항목을 무작정 재사용할 수 없다. 프로세스마다(단, 같은 프로세스의 스레드끼리는 아님) 서로 다른 페이지 테이블 트리를 갖고, 커널과 (있다면) VMM(하이퍼바이저)도 마찬가지다. 프로세스의 주소 공간 레이아웃 자체가 바뀔 수도 있다. 이 문제를 다루는 두 가지 방법:

- **페이지 테이블 트리가 바뀔 때마다 TLB를 통째로 플러시**한다. 대부분의 OS에서 스레드/프로세스 전환에는 커널 코드 실행이 수반되므로, TLB 플러시는 커널 주소 공간을 벗어나거나(때로는 들어갈 때도) 발생하며, 가상화 환경이면 VMM 호출 시점에도 발생한다.
- **TLB 항목의 태그를 확장**해 그 항목이 속한 페이지 테이블 트리를 고유하게 식별하도록 한다.

전자(전체 플러시)는 효과적이지만 비용이 크다. 시스템 콜 하나가 수천 개 명령어만 실행하며 페이지 몇 개만 건드리더라도, 전체 플러시는 그 몇 개보다 훨씬 많은 항목을 불필요하게 지운다. Core2 아키텍처의 ITLB 128항목, DTLB 256항목 기준으로, 전체 플러시는 각각 100개, 200개 이상의 항목을 불필요하게 날린다. 시스템 콜이 같은 프로세스로 복귀하면 그 항목들은 다시 쓰일 수 있었을 텐데 이미 사라진 뒤다. 커널이나 VMM에서 자주 쓰이는 코드도 마찬가지로, 페이지 테이블 자체는 거의 바뀌지 않는데도 커널에 진입할 때마다 TLB를 처음부터 다시 채워야 한다 — 이것이 오늘날 프로세서의 TLB 캐시가 더 커지지 않는 이유이기도 하다(프로그램이 이 많은 항목을 다 채울 만큼 오래 실행되지 않을 가능성이 높으므로).

CPU 설계자들이 이를 놓치지 않았다. 한 가지 최적화는 **개별 TLB 항목만 선택적으로 무효화**하는 것이다(예: 커널 코드/데이터가 특정 주소 범위에 속한다면 그 범위에 해당하는 페이지만 축출 — 태그 비교만 하면 되므로 비용이 크지 않다. `munmap` 호출로 주소 공간 일부가 바뀔 때도 유용).

더 나은 해법은 **TLB 태그 확장**이다. 가상 주소의 일부에 더해 페이지 테이블 트리(즉 주소 공간)를 고유하게 식별하는 값을 추가하면 TLB를 전혀 플러시하지 않아도 된다. 문제는 TLB 태그에 쓸 수 있는 비트 수는 심하게 제한적인 데 반해 주소 공간 수는 그렇지 않다는 것 — 그래서 식별자 재사용이 필요하고, 재사용이 일어나면 해당 식별자를 가진 항목들만(운 좋으면 훨씬 작은 집합) 부분적으로 플러시해야 한다.

이 확장 태깅은 가상화뿐 아니라 여러 프로세스가 도는 일반 환경에서도 유리하다. 각 프로세스의 메모리 사용(따라서 TLB 항목 사용)이 제한적이라면, 다시 스케줄링될 때 최근 쓰인 TLB 항목이 아직 남아있을 가능성이 크다. 추가 이점 두 가지: (1) 커널·VMM 같은 특수 주소 공간에 짧게 들어갔다가 나오는 경우, 태그가 있으면 호출한 주소 공간의 캐시된 변환이 보존된다. (2) 같은 프로세스의 두 스레드 사이를 전환할 때는 TLB 플러시가 아예 필요 없다(태그가 없으면 커널 진입이 첫 스레드의 TLB 항목을 지워버림).

일부 프로세서는 이미 이런 확장 태그를 구현했다. AMD는 Pacifica 가상화 확장에서 1비트 **ASID**(Address Space ID)를 도입해 VMM과 게스트 도메인의 주소 공간을 구분한다(덕분에 VMM 진입/복귀 시마다 게스트나 VMM의 TLB 항목을 플러시하지 않아도 됨). 인텔은 같은 목적의 VPID(Virtual Processor ID)를 더 많은 비트로 제공하지만, 게스트 도메인당 고정값이라 그 안의 개별 프로세스 구분에는 쓸 수 없다.

#### 4.3.2 TLB 성능에 영향을 주는 요인

가장 큰 요인은 **페이지 크기**다. 페이지가 클수록 더 많은 명령어/데이터 객체가 한 페이지에 들어가므로 필요한 주소 변환 수, 곧 필요한 TLB 항목 수가 줄어든다. 오늘날 대부분의 아키텍처는 여러 페이지 크기를 지원하며 일부는 동시 사용도 가능하다(x86/x86-64는 기본 4KB 외에 4MB·2MB 페이지도 지원, IA-64와 PowerPC는 64KB 등을 기본 페이지 크기로 허용).

큰 페이지 사용에는 대가도 따른다. 큰 페이지에 쓰일 메모리 영역은 물리 메모리에서 연속적이어야 한다. 물리 메모리 관리 단위 자체를 가상 메모리 페이지 크기로 올리면 낭비가 커진다(실행 파일 로딩 등 각종 메모리 연산이 페이지 경계 정렬을 요구하므로, 평균적으로 매핑마다 페이지 크기의 절반이 낭비됨). 그래서 x86-64에서 물리 메모리 할당 단위를 2MB로 올리는 것은 비현실적이다. 결국 큰 페이지도 여러 개의 작은 페이지로 구성되고, 이 작은 페이지들은 물리적으로 연속이어야 한다 — 4KB 단위 페이지로 2MB(즉 연속된 512개 페이지)의 연속 물리 메모리를 확보하는 것은, 시스템이 한동안 돌며 물리 메모리가 파편화된 뒤에는 극히 어렵거나 불가능할 수 있다.

그래서 리눅스에서는 이런 큰 페이지를 전용 `hugetlbfs` 파일시스템을 통해 시스템 시작 시점에 미리 할당해야 한다. 고정된 수의 물리 페이지가 큰 가상 페이지 전용으로 예약되며, 늘 쓰이지 않을 수도 있는 자원을 묶어두는 셈이고 풀 크기를 늘리려면 보통 재시작이 필요하다. 그럼에도 성능이 최우선이고 자원이 충분하며 번거로운 설정이 큰 걸림돌이 아닌 상황(예: 데이터베이스 서버)에서는 huge page가 정답이다.

기본 가상 페이지 크기 자체를 늘리는 것도(선택적 huge page와 별개로) 문제가 있다. 메모리 매핑 연산(실행 파일 로딩 등)이 이 페이지 크기에 맞아야 하고 더 작은 매핑은 불가능해진다. 대부분의 아키텍처에서 실행 파일 각 부분의 위치는 고정된 관계를 갖는데, 빌드 시 고려된 것보다 페이지 크기가 커지면 로드 자체가 실패할 수 있다. ELF 실행 파일의 정렬 요구사항은 프로그램 헤더에 인코딩되어 있으며(`eu-readelf -l` 등으로 확인 가능), x86-64 바이너리에서는 이 값이 보통 0x200000(2,097,152 = 2MB, 해당 프로세서가 지원하는 최대 페이지 크기)로 나타난다.

큰 페이지 크기의 또 다른 효과는 페이지 테이블 트리의 레벨 수가 줄어든다는 것이다. 가상 주소에서 페이지 오프셋에 해당하는 부분이 늘어나므로 페이지 디렉터리로 처리해야 할 남은 비트가 줄어, TLB 미스 시 필요한 작업량이 줄어든다. 큰 페이지 크기 사용 외에도, 동시에 쓰이는 데이터를 더 적은 수의 페이지에 몰아 배치하면(캐시 사용 최적화와 비슷한 발상) TLB 항목 수를 줄일 수 있다. TLB 항목 수 자체가 워낙 적기 때문에 이는 중요한 최적화가 될 수 있다.

### 4.4 가상화의 영향

OS 이미지 가상화가 점점 보편화되면서 메모리 처리에 한 계층이 더 추가된다(프로세스 가상화나 jail·컨테이너류는 OS가 하나뿐이므로 이 범주가 아니다). Xen이나 KVM 같은 기술은 프로세서의 도움을 받거나 받지 않고 독립적인 OS 이미지 실행을 가능하게 하는데, 이 경우 물리 메모리 접근을 직접 통제하는 소프트웨어는 단 하나(VMM)가 된다.

**Xen 모델**: Xen VMM이 그 역할을 한다. 메모리·프로세서 외의 하드웨어는 VMM이 직접 제어하지 않고, 특권을 가진 Dom0 도메인이 제어한다(현재는 Dom0과 비특권 DomU 커널이 메모리 처리 면에서는 거의 같은 커널이다). VMM은 Dom0·DomU 커널에 물리 메모리를 내어주고, 이 커널들이 마치 하드웨어 위에서 직접 도는 것처럼 평소 방식대로 메모리를 관리한다. 도메인 간 완전한 분리를 위해, Dom0·DomU 커널의 메모리 처리는 물리 메모리에 무제한 접근할 수 없다 — VMM은 개별 물리 페이지를 그냥 나눠주고 게스트 OS가 알아서 주소를 매기게 두지 않는다(그러면 오류가 있거나 악의적인 게스트 도메인에 대한 보호가 전혀 없어지므로). 대신 VMM은 게스트 도메인마다 자신만의 페이지 테이블 트리를 만들어 이를 통해 메모리를 내어준다.

이 접근 제어는 Xen의 반가상화(para-)든 완전가상화(하드웨어 지원)든 동일하게 활용된다. 게스트 OS가 자신의 페이지 테이블을 수정할 때마다 VMM이 호출되고, VMM은 갱신된 정보로 자신의 **섀도 페이지 테이블**(실제 하드웨어가 쓰는 테이블)을 갱신한다. 이 과정은 페이지 테이블 트리를 수정할 때마다 VMM 호출이 필요하므로 상당히 비싸다. 이 추가 비용에 대응해 프로세서들은 섀도 페이지 테이블 생성 자체를 피하는 기능을 도입하고 있다 — 인텔의 **EPT**(Extended Page Tables), AMD의 **NPT**(Nested Page Tables)가 그것이다. 두 기술 모두 게스트 OS의 페이지 테이블이 "게스트 가상 주소"에서 "호스트 가상 주소"를 만들고, 이 호스트 가상 주소를 도메인별 EPT/NPT 트리로 다시 물리 주소로 변환한다. 이렇게 하면 메모리 처리 관련 VMM 진입 대부분이 사라져 가상화 없는 경우에 가까운 속도를 얻을 수 있고, 프로세스별이 아니라 도메인별로 페이지 테이블 트리 하나만 유지하면 되므로 VMM의 메모리 사용량도 줄어든다. 이 추가 주소 변환 단계의 결과도 TLB에 저장된다 — 즉 TLB는 가상-물리 주소만이 아니라 조회의 최종 결과 전체를 담는다. AMD의 Pacifica 확장이 도입한 ASID가 매 진입마다 TLB 플러시를 피하는 데 쓰인다(초기 버전은 VMM과 게스트 OS를 구분할 만큼인 1비트뿐). 인텔의 VPID도 같은 목적이지만 게스트 도메인당 고정값이라 그 안의 개별 프로세스 구분에는 못 쓴다.

**KVM 모델**: Xen 방식은 주소 공간 변경마다 작업량이 크다는 문제 외에도, VMM 기반 가상화 자체가 메모리 처리 계층을 두 겹으로 만든다는 근본 문제가 있다(NUMA 같은 복잡성까지 고려하면 메모리 처리는 원래도 어려운 일이다). Xen의 별도 VMM 접근 방식은 "메모리 영역 발견" 같은 사소해 보이는 것까지 포함한 메모리 관리 구현 전체를 VMM에 중복 구현해야 해서 최적의 처리가 어렵다. KVM 리눅스 커널 확장은 이 문제를 다르게 푼다 — 하드웨어를 직접 제어하는 별도 VMM 대신, 평범한 리눅스 커널이 이 기능을 떠맡는다. 게스트 도메인은 "게스트 모드"라는 이름으로 일반 사용자 레벨 프로세스와 나란히 돈다. 반가상화든 완전가상화든 가상화 기능은 KVM VMM이 제어하는데, 이것도 커널이 제공하는 특수 KVM 디바이스로 게스트 도메인을 제어하는 또 하나의 사용자 레벨 프로세스일 뿐이다. Xen 대비 이점은, 게스트 OS 사용 시 여전히 메모리 처리기가 두 개 동작하지만 그 구현은 리눅스 커널 하나뿐이라는 것 — 리눅스 게스트의 메모리 처리기가 베어메탈 위에서 도는 바깥쪽 리눅스 커널의 메모리 처리기와 같은 가정을 하므로 작업량도 적고 버그도 적으며 두 메모리 처리기가 맞닿는 지점의 마찰도 적다.

전체적으로, 가상화 환경에서는 캐시 미스(명령어·데이터·TLB를 막론하고)의 비용이 비가상화 환경보다 더 크다는 점을 프로그래머가 인지해야 한다. 이 작업량을 줄이는 최적화는 가상화 환경에서 더 큰 보상을 준다. EPT·NPT 같은 기술로 프로세서 설계자들이 시간이 지나며 그 격차를 계속 줄여가겠지만, 완전히 사라지지는 않을 것이다.

## 5장 NUMA 지원

2장에서 일부 머신은 물리 메모리의 특정 영역에 대한 접근 비용이 접근이 시작된 위치에 따라 달라진다는 것을 봤다. 이런 하드웨어는 OS와 애플리케이션 모두의 특별한 배려가 필요하다.

### 5.1 NUMA 하드웨어

비균일 메모리 아키텍처는 점점 흔해지고 있다. 가장 단순한 형태의 NUMA는 프로세서가 로컬 메모리를 갖고, 이것이 다른 프로세서에 붙은 메모리보다 접근 비용이 싼 구조다. 이때 NUMA 팩터는 크지 않다.

대형 머신에서는 NUMA가 특히 더 필요해진다. 범용 하드웨어에서는 모든 프로세서가 같은 노스브리지를 공유하므로(AMD Opteron NUMA 노드는 별개 문제이니 일단 제외) 노스브리지가 심각한 병목이 된다. 대형 머신은 노스브리지 대신 커스텀 하드웨어를 쓸 수 있지만, 메모리 칩 자체가 멀티포트(여러 버스에서 동시 접근 가능)가 아닌 한 여전히 병목이 남는다 — 멀티포트 RAM은 만들고 지원하기가 복잡하고 비싸서 거의 쓰이지 않는다.

그다음 단계 복잡도는 AMD가 쓰는 방식으로, 인터커넥트 메커니즘(AMD는 Digital에서 라이선스한 HyperTransport)이 RAM에 직접 붙지 않은 프로세서에도 접근 경로를 제공한다. 이런 식으로 만들 수 있는 구조의 크기는, 노드 간 최대 거리인 지름(diameter)을 임의로 늘리지 않는 한 제한적이다.

노드 연결에 효율적인 토폴로지가 **하이퍼큐브**다. 노드당 인터커넥트 인터페이스가 C개면 노드 수는 2^C개로 제한되며, 하이퍼큐브는 2^n개 CPU와 n개 인터커넥트를 가진 모든 시스템 중 지름이 가장 작다. AMD 1세대 Opteron은 프로세서당 HyperTransport 링크 3개를 가졌는데, 적어도 하나의 프로세서는 사우스브리지에 링크 하나를 써야 하므로 실질적으로 C=2인 하이퍼큐브(직접·효율적으로 구현 가능한 최소 지름)가 된다. 다음 세대에서 링크가 4개가 되면 C=3 하이퍼큐브가 가능해진다.

그렇다고 더 많은 프로세서를 모으는 게 불가능한 건 아니다. 더 큰 프로세서 집합을 지원하는 크로스바(예: Newisys의 Horus)를 개발한 회사들도 있지만, 이런 크로스바는 NUMA 팩터를 키우고 어느 프로세서 수 이상에서는 효과가 떨어진다.

그다음 단계는 CPU 그룹을 연결해 공유 메모리를 구현하는 것으로, 모두 특수 하드웨어가 필요하며 범용 시스템이 전혀 아니다. 여러 복잡도 수준이 있는데, IBM x445 같은 시스템은 그나마 범용 머신에 가깝다 — 통상적인 4U 8-way x86/x86-64 머신으로 구매할 수 있고, 이런 머신 2대(나중엔 최대 4대)를 연결해 공유 메모리를 가진 단일 머신처럼 동작시킬 수 있다(다만 이 인터커넥트가 상당한 NUMA 팩터를 만들어 OS와 애플리케이션 모두 이를 고려해야 함). 스펙트럼 반대쪽 끝에는 SGI Altix처럼 애초에 상호 연결을 염두에 두고 설계된 머신이 있다 — SGI의 NUMAlink 인터커넥트는 매우 빠르면서도 지연이 낮아, HPC(고성능 컴퓨팅), 특히 MPI를 쓰는 환경의 요구사항 두 가지를 모두 만족한다. 대신 이런 정교함과 특수성은 매우 비싸다. NUMA 팩터를 상당히 낮게 유지할 수는 있지만, CPU 수가 수천 개에 이르고 인터커넥트 용량이 제한적이라 워크로드에 따라 NUMA 팩터가 동적으로 변하며 감당 못 할 수준까지 오를 수도 있다.

더 흔한 방식은 여러 범용 머신을 고속 네트워킹으로 연결해 클러스터를 이루는 것이다. 다만 이는 공유 주소 공간을 구현하지 않으므로 NUMA 머신이 아니며 이 장의 논의 대상도 아니다.

### 5.2 NUMA를 위한 OS 지원

NUMA 머신을 지원하려면 OS가 메모리의 분산된 특성을 고려해야 한다. 예를 들어 프로세스가 특정 프로세서에서 돈다면, 그 프로세스 주소 공간에 배정되는 물리 RAM은 이상적으로 로컬 메모리에서 와야 한다. 그렇지 않으면 코드·데이터를 위한 모든 명령어가 원격 메모리에 접근해야 한다.

NUMA 머신에만 있는 특수 사례도 고려해야 한다. DSO의 텍스트 세그먼트는 보통 머신의 물리 RAM에 정확히 한 번만 존재한다. 그런데 이 DSO를 모든 CPU의 프로세스·스레드가 쓴다면(libc 같은 기본 런타임 라이브러리가 대표적), 소수 프로세서를 제외한 나머지 전부가 원격 접근을 해야 한다는 뜻이다. 이상적으로는 OS가 이런 DSO를 각 프로세서의 물리 RAM에 "미러링"해 로컬 사본을 쓰게 해야 하지만, 이는 최적화일 뿐 필수는 아니고 대체로 구현하기 어렵다(지원되지 않거나 제한적으로만 지원됨).

상황을 악화시키지 않으려면 OS는 프로세스·스레드를 한 노드에서 다른 노드로 마이그레이션하지 않아야 한다. 일반 멀티프로세서 머신에서도 OS는 마이그레이션을 피하려 하는데(캐시 콘텐츠 손실 때문), NUMA 환경에서는 새 프로세서 선택이 더 제한적이다 — 새로 고를 프로세서가 기존 프로세서보다 그 프로세스가 쓰는 메모리에 더 비싸게 접근하면 안 되므로 대상 목록이 좁아진다. 조건에 맞는 여유 프로세서가 없으면 OS는 어쩔 수 없이 더 비싼 메모리 접근 비용을 가진 프로세서로 마이그레이션할 수밖에 없다. 이때 두 가지 대응이 가능하다: 상황이 일시적이길 바라며 나중에 더 적합한 프로세서로 되돌리거나, 프로세스의 메모리 자체를 새 프로세서에 가까운 물리 페이지로 옮기는 것이다(후자는 상당히 비싼 작업 — 많은 메모리를 복사해야 하고 그동안 프로세스를 잠시 멈춰 이전 페이지 수정 사항을 올바르게 반영해야 함).

일반적으로 NUMA 머신의 모든 프로세스가 같은 양의 메모리를 쓴다고 가정할 수 없어서, 프로세스를 프로세서에 분산해도 메모리 사용까지 균등하게 분산되리라는 보장은 없다(HPC 분야를 제외하면 흔히 메모리 사용은 매우 불균등하다). 요청이 시작된 프로세서에 항상 로컬로 메모리를 할당한다면, 언젠가는 큰 프로세스가 도는 노드의 로컬 메모리가 고갈되는 문제에 부딪힌다. 이런 심각한 문제에 대응해, 기본적으로 메모리는 로컬 노드에만 배타적으로 할당되지 않는다 — 시스템 전체 메모리를 활용하기 위한 기본 전략은 메모리를 스트라이핑하는 것이다. 이는 시스템 전체 메모리의 균등한 사용을 보장하고, 부수 효과로 모든 사용 메모리에 대한 평균 접근 비용이 프로세서를 옮겨도 크게 변하지 않으므로 프로세스를 자유롭게 마이그레이션할 수 있게 된다. NUMA 팩터가 작다면 스트라이핑도 받아들일 만하지만 여전히 최적은 아니다(5.4절 데이터 참고). 이런 기본 전략을 그대로 두는 것이 부담이 되는 프로세스를 위해, 리눅스는 프로세스 단위로 메모리 할당 전략을 선택할 수 있게 하며(자식 프로세스에도 적용 가능), 관련 인터페이스는 6장에서 다룬다.

### 5.3 공개되는 정보

커널은 `sysfs`(`/sys/devices/system/cpu/cpu*/cache`)를 통해 프로세서 캐시 정보를 공개한다. 각 `index*` 서브디렉터리의 `type`, `level`, `shared_cpu_map` 파일이 캐시 토폴로지 파악에 중요하다. 예컨대 인텔 Core 2 QX6700에서는 각 코어가 L1i·L1d·L2 세 캐시를 갖고, L1 계열은 코어별 전용(비트마스크에 비트 하나만 설정)이지만 L2는 cpu0·cpu1이 하나를, cpu2·cpu3가 다른 하나를 공유하는 구조로 나타난다.

프로세서 토폴로지 정보(`/sys/devices/system/cpu/cpu*/topology`)의 `physical_package_id`, `core_id`, `core_siblings`, `thread_siblings` 값을 캐시 정보와 함께 보면, 예를 들어 4소켓 듀얼코어 Opteron 머신에서는 하이퍼스레드가 없고(총 8개 CPU가 물리 프로세서 4개 × 코어 2개), 어떤 코어도 캐시를 공유하지 않는다는 것을 확인할 수 있다.

NUMA의 성격 자체는 이 정보만으로는 알 수 없다. NUMA 노드별 정보는 `/sys/devices/system/node/node*` 아래에 있으며, 각 노드 디렉터리의 `cpumap`과 `distance` 파일이 중요하다. 예시 Opteron 머신에서는 4개 노드 각각이 프로세서 하나에 대응하고, `distance` 값은 로컬 접근이 10, 원격 접근이 모두 20으로 나타난다(이 데이터가 실제로는 부정확할 수 있다는 지적도 있다 — HyperTransport 링크 3개 중 하나는 사우스브리지에 쓰여야 하므로 최소 한 쌍의 노드는 더 큰 거리를 가져야 정상인데 여기선 그렇지 않게 나온다).

이런 정보는 존재하지만 다루기 번거로우며, 이를 쉽게 다루도록 돕는 인터페이스는 6.5절에서 소개된다. 프로세스별 메모리 배치 현황은 `/proc/PID/numa_maps` 의사 파일로 볼 수 있다 — 메모리 매핑 파일, COW(Copy-On-Write) 페이지, 익명 메모리가 시스템의 각 노드에 어떻게 분산되어 있는지 `N0`~`N3` 값으로 보여준다(예시에서는 프로그램 자체와 dirty 페이지는 실행 노드에, `ld.so`·`libc`의 읽기 전용 매핑이나 공유 파일은 다른 노드에 할당된 패턴이 보인다).

### 5.4 원격 접근 비용

거리(distance) 값은 실제로 의미가 있다. AMD가 문서화한 4소켓 머신의 NUMA 비용 자료에 따르면(쓰기 연산 기준), 쓰기가 읽기보다 느린 것은 당연하고, 흥미로운 부분은 1-hop과 2-hop 접근의 비용이다. 두 가지 1-hop 케이스는 비용이 약간씩 다르며, 핵심적으로 기억할 수치는 **2-hop 읽기·쓰기가 0-hop 읽기보다 각각 30%, 49% 느리고**, 2-hop 쓰기는 0-hop 쓰기보다 32%, 1-hop 쓰기보다 17% 느리다는 것이다. 프로세서와 메모리 노드의 상대적 위치가 큰 차이를 만든다. AMD의 다음 세대 프로세서가 프로세서당 4개의 코히어런트 HyperTransport 링크를 갖게 되면 4소켓 머신의 지름이 1이 되지만, 8소켓으로 늘어나면 하이퍼큐브 지름이 다시 3이 되어 문제가 재발한다.

실제 세계에서의 효과를 보려면 3.5.1절과 같은 방식으로 대역폭을 측정하되, 이번엔 메모리가 1홉 떨어진 원격 노드에 있는 경우를 로컬 메모리와 비교한다. 중요한 결과는 **읽기 연산이 항상 약 20% 느리다**는 것이다(원문 자체 측정으로는 20%, AMD 문서 수치인 9%보다 크게 나왔는데, 이는 후자가 끊김 없는 순수 읽기/쓰기 조건을 가정한 수치이거나 구형 프로세서 리비전 기준일 가능성이 있다고 지적한다). 실행 시 L2 캐시 미스가 나면 각 캐시 라인마다 이 추가 비용이 붙으므로, 캐시 크기를 넘는 대규모 워크로드의 모든 측정 비용은 메모리가 원격일 경우 9~30% 더 늘려 잡아야 한다.
