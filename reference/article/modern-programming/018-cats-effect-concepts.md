# Cats Effect: 개념

> 원문: [Concepts · Cats Effect](https://typelevel.org/cats-effect/docs/concepts)
> 저자: Typelevel (Cats Effect 공식 문서) | 번역일: 2026-07-10

Cats Effect는 수많은 개념을 도입한다. 각각은 매우 일반적이고 개별적으로는 단순하지만, 이 분야가 처음이라면 낯설고 위압적으로 느껴질 수 있다. 여기에는 개념과 상황을 설명하는 용어 체계도 따라오는데, 이는 다른 비동기 프레임워크(Akka나 Vert.x 등)에서 쓰는 용어와 상당히 다를 수 있다. 이 페이지는 Cats Effect 런타임을 높은 수준에서 설명하면서, 세부 사항을 논의할 때 흔히 전제되는 개념과 용어를 소개한다.

전반적인 방향을 잡자면, Cats Effect는 JVM 위의 다른 비동기 런타임과 같은 개념적 공간에 있다고 보면 된다. 그런 런타임의 흔한 예로는 다음이 있다.

- Akka
  - Akka는 akka-http나 akka-cluster 같은 것까지 아우르는, 좀 더 총체적인 *프레임워크*로 여겨지는 경향이 있다. Akka에서 Cats Effect에 가장 직접적으로 대응하는 요소는 `Actor`와 `Future`(사실 `Future`는 Scala 표준 라이브러리의 일부다)다.
- Netty
  - Netty의 핵심 런타임(`EventLoopGroup`과 각종 `Handler`)은 비교적 기초적이고 대부분 자체 NIO 래퍼를 뒷받침하는 용도다. Cats Effect는 NIO 래퍼를 제공하지 않지만 [Fs2](https://fs2.io)가 제공한다. 반대로 Cats Effect의 런타임은 훨씬 견고하다.
- Tokio
- RxJava
- Vert.x

위 라이브러리들은 모두 Cats Effect와 상당한 차이와 개념적 불일치가 있다는 점에 유의하라. Tokio와 Vert.x가 아마 가장 가까운 상대이겠지만, 둘 다 더 수직 통합적이다(프레임워크 지향 접근을 취한다는 뜻이다). 반면 Cats Effect는 폭넓고 조합 가능한 생태계를 가능하게 하는 라이브러리로 스스로를 명확히 규정한다. 게다가 Cats Effect는 이들 생태계에는 아예 존재하지 않는 기능을 여러 곳에서 정의한다. 예를 들어 Akka(`Future` 기준), Netty, Vert.x 모두 비동기 취소(asynchronous cancelation, "인터럽션"이라고도 부른다)를 전혀 지원하지 않는다. 그래서 타임아웃이나 동시적 에러 처리 같은 기본 기능이 실행 중인 애플리케이션에서 리소스 누수로 이어질 수 있다. 기능 불일치의 또 다른 예는 파이버를 인지하는 work-stealing 런타임인데, 이는 Tokio에 있고 Akka에도 (그보다 덜한 정도로) 있지만, Netty나 Vert.x에는 없다. 마지막 예로, 비동기 트레이싱은 Vert.x와 Akka에 제한적으로 존재하고 다른 프레임워크에는 아예 없으며, 이 둘 중 어느 쪽도 (Cats Effect와 달리) 프로덕션에서 쓸 만큼 성능이 좋은 버전을 제공하지 않는다.

이런 차이에도 불구하고, Cats Effect가 근본적으로는 이 분야의 다른 프레임워크와 같은 문제를 상당수 해결한다는 점을 이해하면 대체로 도움이 된다. Cats Effect는 JVM과 JavaScript 위에서 복잡한 고성능 비동기·병렬 소프트웨어를 쉽게 만들고 확장할 수 있게 해주는 기반 런타임 계층이다.

## 파이버(Fibers)

파이버는 Cats Effect의 근본 추상화다. 이 용어는 의도적으로 `Thread`를 연상시키는데, 파이버는 말 그대로 경량 스레드("그린 스레드"나 "코루틴"이라고도 자주 불린다)이기 때문이다. 스레드와 마찬가지로 파이버는 궁극적으로 하부 하드웨어가 그 순서대로 평가하게 될 일련의 동작을 나타낸다. 파이버가 스레드와 갈라지는 지점은 발자국(footprint)과 추상화 수준이다.

파이버는 매우 가볍다. Cats Effect의 `IO` 런타임은 파이버 하나를 대략 150바이트로 구현한다. 즉 한 프로세스 안에서 말 그대로 수천만 개의 파이버를 문제없이 만들 수 있고, 주된 제약 요인은 그저 메모리다. 예를 들어 Cats Effect로 정의한 클라이언트/서버 애플리케이션은 들어오는 요청마다 새 파이버를 만든다. Java의 `ServerSocket`으로 짠 소박한 서버가 요청마다 새 `Thread`를 만드는 것과 비슷하다(다만 파이버로 이렇게 하는 것은 안전하고 빠르다!). 파이버는 워낙 가볍기 때문에 새 파이버를 만들고 시작하는 행위 자체가 극히 빠르고, 그래서 편할 때마다 아주 짧게 쓰고 버리는 "일회용" 파이버를 만드는 것도 가능하다. Cats Effect의 많은 함수가 내부적으로 파이버로 구현되어 있으며, 심지어 병렬성과 무관한 함수(`memoize` 같은 것)도 그렇다.

이 속성만으로도 파이버는 충분히 유용한 도구가 되지만, Cats Effect는 여기서 한발 더 나아가 모든 파이버에 대해 비동기 콜백, 리소스 처리, 취소(인터럽션)를 일급으로 지원한다. 특히 비동기 지원의 효과가 깊다. 파이버의 개별 "단계"(스레드의 문장(statement)과 비슷하다)는 값이나 에러를 낼 때까지 실행되는 동기 방식일 수도 있고, 나중에 외부에서 호출될 수 있는 콜백을 등록하는 비동기 방식일 수도 있는데, 이 두 종류의 단계 사이에 근본적인 차이가 없다는 뜻이기 때문이다. 둘 다 그냥 파이버의 일부다. 그래서 비동기적이고 콜백 지향적인 동작 사이를 누비는 비즈니스 로직을 정의하는 것이, 같은 로직을 고전적인 블로킹 제어 흐름으로 정의하는 것만큼이나 쉽다.

달리 말하면, 파이버에서는 콜백과 `return` 사이에 차이가 없다.

스레드의 각 단계는 문장이고, 그 문장들은 텍스트 파일에 특정 순서로 적고 세미콜론(`;`) 연산자로 결합해 순차적으로 정의한다. 파이버의 각 단계는 이펙트이고, 그 이펙트들은 `flatMap` 함수로 명시적으로 조합해 순차적으로 정의한다. 예를 들어:

```scala
import cats.effect._

IO.println("Hello") flatMap { _ => 
  IO.println("World")
}
```

같은 것을 `for` 컴프리헨션으로 표현하는 것도 아주 흔하다.

```scala
for {
  _ <- IO.println("Hello")
  _ <- IO.println("World")
} yield ()
```

`flatMap`은 `;` 같은 마법 문법이 아니라 여느 메서드와 다를 바 없는 메서드이므로, 이펙트를 조합하는 흔한 패턴을 담아내는 편의 문법과 고수준 추상화를 만들 수 있다. 예를 들어 두 이펙트를 이어 붙이되 첫 번째 결과를 무시하는 패턴은 전용 연산자 `>>`가 있을 만큼 흔하다.

```scala
IO.println("Hello") >> IO.println("World")
```

(이 연산자는 `*>`와 거의 동의어이며, `*>`도 흔히 쓰이고 같은 의미다)

Scala의 매크로를 사용해 `flatMap` 위에 우리만의 편의 문법을 만들 수도 있다. 예를 들어 [typelevel/cats-effect-cps](https://github.com/typelevel/cats-effect-cps)를 쓰면 JavaScript나 Rust 같은 다른 언어의 `async`/`await` 문법을 연상시키는 방식으로 파이버를 정의할 수 있다.

```scala
import cats.effect.cps._

async[IO] {
  IO.println("Hello").await
  IO.println("World").await
}
```

이 모든 것이 파이버다. 정확히는 파이버의 일부에 대한 정의인데, 문장 한 쌍이 스레드의 일부를 정의하는 것과 마찬가지다. 사실 `IO.println` 자체도 `flatMap`과 다른 연산들로 정의되어 있으므로 역시 파이버의 일부다. 파이버의 각 단계는 이펙트이고, 단계들을 조합하면 더 큰 이펙트가 만들어지며, 이것을 계속 이어서 조합할 수 있다.

모든 애플리케이션에는 "메인 파이버"가 있다. 이는 프로세스 안에서 제어 흐름이 시작되는 지점이라는 점에서 "메인 스레드" 개념과 아주 비슷하다. Cats Effect에서는 관례적으로 이 메인 파이버를 `IOApp`으로, 구체적으로는 `run` 메서드가 반환하는 이펙트로 정의한다.

```scala
object Hi extends IOApp.Simple {
  val run = IO.println("Hello") >> IO.println("World")
}
```

이 예에서 `Hi`는 메인 클래스다. 외부에서 실행하면(예: `java` 명령, ScalaJS라면 `node`) 메인 파이버를 시작해 그 파이버가 완료될 때까지 실행하고, 그 시점에 프로세스가 종료된다.

한 파이버가 다른 파이버를 `start`하면, 보통 첫 번째 파이버를 두 번째 파이버의 "부모"라고 부른다. 이 관계는 직접적인 계층 구조는 아니어서, 부모가 자식보다 먼저 종료되어도 아무런 비일관성이 생기지 않는다. 그러나 파이버 모델의 어떤 속성들은 이 부모/자식 관계를 염두에 두면 훨씬 잘 이해된다. 예를 들어 파이버는 언제나 (`handleErrorWith`나 `attempt` 같은 것으로) 에러를 관찰하고 복구할 수 있다. 이는 개념적으로 `try`/`catch`와 비슷하다. 파이버는 자신의 취소도 관찰할 수 있지만(아래 참고) 거기서 복구할 수는 없다. 즉 취소된 뒤에는 실행을 계속할 수 없다. 부모 파이버는 (`cancel` 메서드로) 자식의 취소를 개시할 수 있고, 그 자식의 최종 결과(`Canceled`일 수도 있다)를 관찰할 수 있으며, 자식이 종료된 뒤에도 실행을 계속할 수 있다.

### 취소(Cancelation)

기본적으로 파이버는 실행 중 어느 지점에서든 취소할 수 있다. 덕분에 애플리케이션 안에서 더 이상 필요 없는 계산을 즉시 중단하고 그 리소스를 가능한 한 일찍 안전하게 해제할 수 있다.

실무에서 파이버 취소는 대개 두 상황 중 하나에 대한 반응으로 일어난다. 타임아웃과 동시적 에러다. 전자의 예:

```scala
import scala.concurrent.duration._

lazy val loop: IO[Unit] = IO.println("Hello, World!") >> loop

loop.timeout(5.seconds)   // => IO[Unit]
```

위 코드는 `loop`으로 정의된 파이버를 시작하는 `IO`를 만든다. 이 파이버는 표준 출력에 "Hello, World!"를 무한히 찍는다. 그냥 두면 `loop` 파이버는 영원히 돈다. 하지만 `timeout` 함수는 5초를 기다린 뒤 그 파이버에 `cancel`을 호출해 실행을 중단시키고, 파이버가 그 시점에 쥐고 있던 리소스(이 경우에는 없음)를 해제한다.

이는 Java 표준 라이브러리의 `Thread#interrupt` 메서드와 개념적으로 아주 비슷하다. 비슷하긴 하지만, 취소를 훨씬 견고하고 신뢰할 수 있으며 훨씬 안전하게 만드는 중요한 차이가 몇 가지 있다.

첫째, 취소는 협력적이다. 한 파이버가 다른 파이버에 `cancel`을 호출하면, 이는 사실상 대상 파이버에 대한 요청이다. 대상 파이버가 어떤 이유로든 그 순간 취소될 수 없다면, 취소를 요청한 파이버는 취소가 가능해질 때까지 비동기적으로 기다린다. 취소가 시작되면 대상 파이버는 제어를 취소자에게 돌려주기 전에 자신의 파이널라이저(보통 파일 핸들 같은 리소스 해제에 쓰인다)를 모두 실행한다. 반면 `interrupt`는 대상 `Thread`가 실제로 인터럽트되지 않았더라도 항상 즉시 반환한다.

둘째, 스코프가 지정된 구역 안에서 취소를 억제할 수 있다. 파이버가 원자적으로 실행되어야 하는 일련의 동작(즉 모든 동작이 실행되거나 아무것도 실행되지 않아야 하는 것)을 수행 중이라면, `IO.uncancelable` 메서드로 해당 스코프 안에서 취소 신호를 가려서, 파이버가 임계 구역을 마칠 때까지 취소가 유예되도록 보장할 수 있다. 이는 복합적인 리소스 획득과 함께 흔히 쓰인다. 파이버가 "중간에" 취소되면 희소한 리소스가 누수될 수 있기 때문이다. 이 점은 억제가 불가능한 `Thread#interrupt`와 크게 다르다.

마지막으로, 파이버 모델은 취소를 둘러싼 제어력과 보장이 훨씬 강하기 때문에, 대상 파이버 안에서 취소의 세분성(granularity)을 극적으로 높이는 것이 가능하고 안전하다. 구체적으로 파이버의 모든 단계에는 취소 검사가 들어 있다. JVM이 모든 `;`마다 인터럽션 플래그를 검사한다면 `interrupt`가 했을 법한 일과 비슷하다. 위 예제의 `loop` 파이버가 스레드가 전혀 블록되지 않는데도 취소되는 것이 바로 이 원리다. `while` 루프에 `Thread#interrupt`를 써본 사람이라면 이 차이가 얼마나 중요한지 안다. Cats Effect에서는 이런 종류의 파이버 취소가 가능하고, 실제로 아주 흔하다.

## 용어(Terminology)

대부분의 명령형 언어권(Scala 포함)에서는 "비동기(asynchronous)", "동시(concurrent)", "병렬(parallel)"이라는 용어를 뒤섞어 쓰는 일이 아주 흔하다. 특히 "비동기"는 "병렬"과 동의어로 받아들여지곤 하는데, 실제로는 세 용어 모두 아주 뚜렷한 정의가 있고, Cats Effect 같은 프레임워크에서는 그 차이가 중요해진다.

### 비동기(Asynchronous)

"비동기"는 "동기(synchronous)"의 반대말로, 주어진 이펙트가 값을 만들어내는 방식과 관련된 개념이다. 동기 이펙트는 `apply`(또는 `delay`, `blocking`, `interruptible`, `interruptibleMany`)로 정의하며, `return`으로 결과를 내거나 `throw`로 에러를 던진다. 우리에게 익숙한 "순차적" 유형의 이펙트다.

```scala
IO(Thread.sleep(500))   // => IO[Unit]
```

(참고: `Thread.sleep`은 하부 `Thread`를 블록하므로 위 코드는 사실 `IO.interruptible`로 정의해야 하지만, 요점을 보여주기 위해 `apply`를 사용했다)

비동기 이펙트는 `async`(또는 `async_`)로 정의하며 콜백으로 결과를 만든다. 성공 결과는 `Right`로, 에러는 `Left`로 감싼다.

```scala
import java.util.concurrent.{Executors, TimeUnit}

val scheduler = Executors.newScheduledThreadPool(1)

IO.async_[Unit] { cb =>
  scheduler.schedule(new Runnable {
    def run = cb(Right(()))
  }, 500, TimeUnit.MILLISECONDS)

  ()
}
// => IO[Unit]
```

여기 보인 `Thread.sleep` 이펙트와 `schedule` 이펙트는 의미가 같다. 둘 다 500밀리초를 지연한 뒤 파이버의 다음 단계가 진행되게 한다. 차이는 정의 방식이다. `Thread.sleep`은 동기이고 `schedule`은 비동기다. 이 차이가 함의하는 바는 의외로 깊다. `Thread.sleep`은 지연이 끝날 때까지 JVM 수준의 제어 흐름을 반환하지 않으므로, 지연 시간 내내 희소한 리소스(하부 커널 `Thread`)를 사실상 낭비하고, 그동안 다른 작업이 그 리소스를 더 효율적으로 쓰지 못하게 막는다. 반대로 `schedule`은 실행 즉시 반환하며, 지정한 시간이 지나면 나중에 콜백을 호출할 뿐이다. 즉 하부 커널 `Thread`가 낭비되지 않고, 그 사이에 다른 작업을 평가하는 데 재활용될 수 있다.

비동기 이펙트는 (네트워크 I/O나 타이머처럼 적용 가능한 곳에서라면) 동기 이펙트보다 훨씬 효율적이지만, 콜백과 이벤트 리스너를 직접 관리해야 해서 실제 애플리케이션에서는 다루기 어렵다고 여겨진다. 파이버는 비동기 이펙트를 기본 지원하기 때문에 이 단점을 완전히 없애준다. 위의 두 예제 모두 해당 이펙트는 그저 `IO[Unit]` 타입의 값이며, 밖에서 보면 두 이펙트는 동일하게 동작한다. 따라서 `return`/`throw`와 콜백의 차이는 전적으로 정의 지점 안에 캡슐화되고, 애플리케이션의 나머지 제어 흐름은 그 차이를 전혀 모른 채 지나간다. 이것이 파이버가 가진 힘의 큰 부분이다.

"비동기"의 정의에는 "병렬"이나 "동시적(simultaneous)"을 함의하는 요소가 전혀 없고, "순차적"의 의미를 부정하지도 않는다는 점이 매우 중요하다(기억하라: 모든 파이버는 이펙트의 시퀀스다). "비동기"는 단지 "`return`/`throw`가 아니라 콜백으로 값이나 에러를 만든다"는 뜻일 뿐이다. 이는 파이버가 관리하는 이펙트의 구현 세부 사항이지, 설계의 중심에 두어야 할 더 큰 근본 패턴이 아니다.

### 동시(Concurrent)

Cats Effect의 언어에서 "동시"는 일반적으로 제어 흐름이 서로 독립적이라고 정의된 둘 이상의 동작을 가리킨다. "순차적"의 반대이며, 정확히는 "순차적"이라면 "동시"일 수 없다는 뜻이다. 결정적으로, "동시"인 것들은 하부 런타임이 최적이라고 판단하면 순차적으로 평가될 수 있는 반면, 순차적인 동작은 항상 하나씩 차례로 평가된다.

동시성은 비동기 실행과 자주 혼동되는데, 실무에서 동시성의 구현이 보통 어떤 비동기 평가 메커니즘에 의존하기 때문이다. 그러나 위에서 말했듯 비동기성은 어디까지나 구현 세부 사항이며, 동시 대 순차라는 의미론에 대해서는 아무것도 말해주지 않는다.

Cats Effect에는 동시 이펙트를 정의하는 메커니즘이 많다. 가장 단순한 것 중 하나가 `parTupled`인데, 독립적인 이펙트 한 쌍을 평가해 결과의 튜플을 만든다.

```scala
(callServiceA(params1), callServiceB(params2)).parTupled   // => IO[(Response, Response)]
```

모든 동시성 지원이 그렇듯, `parTupled`는 두 이펙트(`callServiceA(params1)`와 `callServiceB(params2)`)가 독립적이고 병렬로 평가될 수 있다고 하부 런타임에 선언하는 수단이다. Cats Effect는 두 이펙트가 병렬로 평가될 수 있다고 절대 마음대로 가정하지 않는다.

Cats Effect의 모든 동시성은 파이버를 만들고 다루는 하부 프리미티브 `start`와 `join`으로 구현된다. 이 동시성 프리미티브들은 `Thread`의 같은 이름 연산과 매우 비슷하지만, Cats Effect의 대부분이 그렇듯 훨씬 빠르고 안전하다.

#### 구조적 동시성(Structured Concurrency)

격식을 갖춰 말하면, 구조적 동시성은 모든 동시 연산이 닫힌 계층 구조를 이루어야 하는 제어 흐름의 한 형태다. 개념적으로는, 어떤 동작들을 동시에 실행하도록 분기(fork)한 연산은 다음으로 넘어가기 전에 그 동작들이 완료되었음을 반드시 보장해야 한다는 뜻이다. 나아가 동시 연산의 결과는 그 연산이 완료된 뒤에만, 그리고 계층 구조상의 부모에게만 공개되어야 한다. 위의 `parTupled`가 그 단순한 예다. `IO[(Response, Response)]`라는 결과는 두 서비스 호출이 모두 완료될 때까지 얻을 수 없고, 그 응답들은 결과 튜플 안에서만 접근할 수 있다.

Cats Effect에는 구조적 동시성 도구가 많은데, 대표적으로 `parTupled`, `parMapN`, `parTraverse`가 있다. 그 외에도 `background`, `Supervisor`, `Dispatcher` 같은 더 견고한 구조적 동시성 연산자를 여럿 제공한다. 또한 특히 [Fs2](https://fs2.io) 같은 고수준 프레임워크의 도움으로, 구조적 동시성이 예외가 아니라 규칙인 생태계를 길러 왔다. 다만 구조적 동시성은 제약이 클 수 있어서, Cats Effect는 필요할 때 비구조적 동시성을 막지 않는다.

구체적으로, 호출자가 완료를 기다리도록 강제받지 않고도 파이버를 `start`할 수 있다. 이런 저수준의 유연성이 필요한 경우가 있긴 하지만, 파이버 "누수"(파이버를 `start`해 놓고 런타임 밖의 참조를 전부 버리는 것)를 초래할 수 있어 다소 위험하기도 하다. 일반적으로는 `background`나 `Supervisor` 같은 구조적(그러면서도 아주 유연한) 도구에 의존하는 편이 낫다.

또한 Cats Effect는 공유되는 동시 상태를 모델링하는 범용 도구 한 쌍, `Ref`와 `Deferred`를 제공한다. 이 메커니즘들은 본질적으로 비구조적이어서, 불필요하게 따라가기 어려운 비즈니스 로직을 낳을 수 있다. 하지만 강력한 도구들이 대개 그렇듯 이들에게도 어울리는 때와 장소가 있다. 이 도구들로 `Queue`, `Semaphore` 같은 강력한 고수준 추상화를 만들 수 있다.

요컨대 Cats Effect는 구조적 동시성을 권장하고 이를 위한 유연한 도구를 다수 제공하지만, `start`나 `Ref` 같은 비구조적 동시성 조합을 막지는 않는다.

### 병렬(Parallel)

비동기 실행과 마찬가지로, 병렬성은 런타임의 구현 세부 사항이다. 두 가지가 병렬로 평가된다는 것은 하부 런타임과 하드웨어가 관련 계산들을 하부 프로세서들 위에 동시에(simultaneously) 스케줄링할 자유가 있다는 뜻이다. 이 개념은 동시성과 밀접하게 연관되는데, 동시성이란 Cats Effect 사용자가 어떤 것들이 병렬로 실행될 수 있다고 런타임에 선언하는 방법이기 때문이다.

동시성과 병렬성의 구분은 동시 이펙트가 병렬로 평가되지 않는 상황을 살펴보면 대체로 이해하기 쉽다. 명백한 상황 하나는 애플리케이션이 JVM이 아니라 JavaScript 위에서 돌아갈 때다. JavaScript는 싱글 스레드 언어이므로, 동시라고 정의되어 있어도 그 무엇도 병렬로 평가될 수 없다. 그렇다고 JavaScript에서 동시성이 쓸모없다는 뜻은 아니다. B를 실행하기 위해 A가 끝나기를 기다릴 필요가 없다는 사실을 런타임이 아는 것은 여전히 도움이 된다. 다만 하부 하드웨어에서는 모든 것이 순차적으로 평가된다는 뜻이다.

동시 이펙트가 병렬로 평가되지 않는 더 복잡한 예는 파이버 수가 런타임 안의 하부 `Thread` 수를 초과할 때 일어날 수 있다. 일반적으로 Cats Effect의 런타임은 하부 스레드 수를 하드웨어가 제공하는 물리 스레드 수에 맞추려고 하는 반면, 파이버 수는 수천만 개(가용 메모리가 큰 시스템에서는 그 이상)까지 늘어날 수 있다. 실제 캐리어(carrier) 스레드는 소수뿐이므로, 런타임은 동시 파이버 중 일부를 같은 하부 캐리어 스레드에 스케줄링하고, 그 파이버들은 병렬이 아니라 직렬로 실행된다.

또 하나의 구현 세부 사항으로, 하부 런타임에 실행 스레드가 하나뿐일 때조차(JavaScript처럼) 파이버가 자기 캐리어 스레드를 "독차지"하지 못하게 되어 있다는 점을 짚어둘 만하다. 파이버가 비동기 이펙트를 시퀀싱할 때마다 큐의 다음 파이버에 스레드를 양보한다. 또한 어떤 파이버가 양보 없이 순차 이펙트를 길게 이어가면, 런타임이 이 상황을 감지해 인위적인 양보를 삽입함으로써 대기 중인 다른 파이버들이 진행할 기회를 얻게 한다. 이것이 공정성(fairness)의 중요한 요소 중 하나다.

## 이펙트(Effects)

이펙트는 평가가 일어날 때 수행될 동작(하나 또는 여럿)에 대한 기술(description)이다. 아주 흔한 이펙트의 하나가 `IO`다.

```scala
val printer: IO[Unit] = IO.println("Hello, World")
val printAndRead: IO[String] = IO.print("Enter your name: ") >> IO.readLine

def foo(str: String): IO[String] = ???
```

위 코드에서 `printer`와 `printAndRead`는 둘 다 이펙트다. 평가될 때 수행될 동작(`printAndRead`의 경우 복수의 동작)을 기술한다. `foo`는 이펙트를 반환하는 함수의 예다. 이런 함수를 줄여서 "이펙트풀(effectful)"하다고 부르곤 한다. 즉 `foo`는 이펙트풀 함수다.

이는 `foo`가 이펙트를 수행하는 함수라고 말하는 것과는 아주 다르다. `printer` 이펙트가 실제로 출력하는 행위와 아주 다른 것과 같은 이치다. 다음과 같이 써 보면 이 점이 깔끔하게 드러난다.

```scala
printer
printer
printer
```

이 코드를 평가하면 "Hello, World"는 정확히 0번 출력된다. `printer`는 그저 기술적인(descriptive) 값일 뿐이라 그 자체로는 아무것도 하지 않기 때문이다. Cats Effect의 핵심은 이펙트를 값으로 표현할 수 있게 하는 것이다.

특기할 점은, `Future`는 이렇게 할 수 없다는 것이다.

```scala
val printer: Future[Unit] = Future(println("Hello, World"))

printer
printer
printer
```

이 코드는 "Hello, World"를 정확히 한 번 출력한다. 즉 `printer`는 동작의 기술이 아니라 (이미 우리 통제 밖에서 실행된) 그 동작의 결과를 나타낸다. 결정적으로, 위 코드에서 `val`을 `def`로 바꾸면 세 번 출력되는데, `Future`를 다룰 때 이것이 생각보다 훨씬 자주 버그와 예상치 못한 동작의 원인이 된다.

Cats Effect를 고급으로 사용할 때는 단순한 `IO`가 아닌 이펙트 타입을 쓰는 것도 흔하다.

```scala
import cats.Monad
import cats.effect.std.Console
import cats.syntax.all._

def example[F[_]: Monad: Console](str: String): F[String] = {
  val printer: F[Unit] = Console[F].println(str)
  (printer >> printer).as(str)
}
```

위에서 `example`은 이펙트풀 함수이고, `printer`는 이펙트다(`printer >> printer`와 `(printer >> printer).as(str)`도 마찬가지다). 여기서 이펙트 타입은 `F`인데, `IO`일 수도 있지만 더 흥미로운 무언가일 수도 있다! `example`의 호출자는 호출 지점에서 이펙트를 자유롭게 고를 수 있다. 예컨대 `example[IO]("Hello, World")`라고 쓰면 `IO[String]`이 반환된다. Cats Effect 생태계의 조합 가능성 상당 부분이 물밑에서 이 기법으로 달성된다.

### 부수 효과(Side-Effects)

어떤 코드를 실행했을 때 값을 반환하는 것 이외의 변화가 일어나면, 보통 그 코드에 "부수 효과가 있다"고 말한다. 더 직관적으로는, 두 번 이상 실행되는지 여부나 언제 실행되는지가 중요한 코드에는 거의 항상 부수 효과가 있다. 그 고전적인 예가 `System.out.println`이다.

```scala
def double(x: Int): Int = {
  System.out.println("Hello, World")
  x + x
}
```

`double` 함수는 `Int`를 받아 그 값을 자기 자신과 더한 값을 반환한다... 그리고 표준 출력에 "Hello, World"를 찍는다. "부수 효과(side-effect)"에서 "부수(side)"란 바로 이 뜻이다. "곁다리로" 다른 무언가가 일어난다는 것이다. 로깅, `var`의 값 변경, 네트워크 호출 등등에 대해서도 같은 말을 할 수 있다.

결정적으로, 부수 효과는 이펙트와 같은 것이 아니다. 이펙트는 어떤 동작의 기술이고, 그 동작이 실행될 때 부수 효과를 수행할 수 있다. 이펙트가 동작의 기술에 불과하다는 사실이야말로 이펙트를 훨씬 안전하고 통제 가능하게 만드는 요인이다. 코드에 부수 효과가 들어 있으면 그 동작은 그냥 일어나 버린다. 병렬로 평가하게 하거나, 다른 스레드 풀에서, 혹은 스케줄에 따라 평가하게 하거나, 실패 시 재시도하게 만들 수 없다. 이펙트는 어떤 동작을 취할지에 대한 기술일 뿐이므로, 구체적인 사용 사례의 필요에 맞게 그것이 결국 어떻게 실행될지에 대한 의미론을 자유롭게 바꿀 수 있다.

Cats Effect에서 부수 효과를 담은 코드는 항상 "특수한" 생성자 중 하나로 감싸야 한다. 구체적으로:

- 동기(`return`하거나 `throw`함)
  - `IO(...)` 또는 `IO.delay(...)`
  - `IO.blocking(...)`
  - `IO.interruptible(...)`
  - `IO.interruptibleMany(...)`
- 비동기(콜백을 호출함)
  - `IO.async` 또는 `IO.async_`

부수 효과를 일으키는 코드를 이 생성자들 중 하나로 감싸면, 코드 자체는 여전히 부수 효과를 담고 있지만, 생성자의 렉시컬 스코프 밖에서는 전체(즉 `IO(...)`까지 포함해서)를 부수 효과가 아니라 이펙트로서 추론할 수 있다.

예를 들어 앞의 `System.out.println` 부수 효과 코드를 감싸서 이펙트 값으로 변환할 수 있다.

```scala
val wrapped: IO[Unit] = IO(System.out.println("Hello, World"))
```

부수 효과를 일으키는 로직을 항상 이펙트 생성자로 감싼다는 이 경험칙을 엄격히 지키면 함수형 프로그래밍의 모든 힘과 조합 가능성이 열린다. 또한 Cats Effect가 코드 조각들을 언제 평가할지에 대해 더 공격적인 가정을 세워 CPU와 캐시 리소스를 더 잘 활용할 수 있게 되므로, 애플리케이션의 스케줄링과 최적화도 더 효과적으로 수행할 수 있다.
