# 함수형 프로그래밍을 위한 모나드

> 원문: [Monads for functional programming](https://homepages.inf.ed.ac.uk/wadler/papers/marktoberdorf/baastad.pdf)
> 저자: Philip Wadler (University of Glasgow) | 번역일: 2026-07-10

**초록.** 모나드로 함수형 프로그램을 구조화하는 방법을 설명한다. 모나드는 전역 상태, 예외 처리, 출력, 비결정성처럼 다른 언어에 있는 효과를 흉내 내기 위한 편리한 틀을 제공한다. 세 가지 사례 연구를 자세히 살펴본다. 모나드가 단순한 평가기의 수정을 어떻게 쉽게 만드는지, 모나드가 제자리 갱신이 가능한 배열 데이터 타입의 기반으로 어떻게 쓰이는지, 그리고 모나드로 파서를 어떻게 만드는지다.

## 1 서론

순수할 것인가, 비순수할 것인가?

함수형 프로그래밍 커뮤니티는 두 진영으로 나뉜다. Miranda(주1)와 Haskell 같은 *순수* 언어는 순전히 람다 계산 그 자체다. Scheme과 Standard ML 같은 *비순수* 언어는 람다 계산에 대입, 예외, 컨티뉴에이션 같은 여러 가지 *효과*를 덧붙인다. 순수 언어는 추론하기 쉽고 지연 평가의 혜택을 볼 수 있으며, 비순수 언어는 효율 면에서 유리하고 때로는 더 간결한 표현이 가능하다.

이론 전산학, 특히 타입 이론과 범주론 분야의 최근 발전은 순수 진영과 비순수 진영의 장점을 통합할 수 있는 새로운 접근을 제시했다. 이 글은 그중 하나, 곧 *모나드*를 이용해 순수 함수형 언어에 비순수 효과를 통합하는 방법을 다룬다.

범주론에서 나온 모나드 개념은 Moggi가 프로그래밍 언어의 표시적 의미론을 구조화하는 데 적용한 바 있다[13, 14]. 같은 기법을 함수형 프로그램을 구조화하는 데에도 적용할 수 있다[21, 23].

모나드의 응용을 세 가지 사례 연구로 보인다. 2장에서는 단순한 평가기를 수정하기 쉽게 구조화하는 과정을 통해 모나드를 소개한다. 3장에서는 모나드가 만족하는 법칙을 설명한다. 4장에서는 순수 함수형 언어에서 갱신 가능한 상태를 제공한다는 오래된 문제에 모나드가 어떤 새로운 해법을 주는지 보인다. 5장에서는 재귀 하강 파서를 만드는 문제에 모나드를 적용한다. 이는 그 자체로도 흥미롭고, 컴퓨팅의 중심 개념 두 가지인 순차와 선택의 패러다임을 제공한다는 점에서도 흥미롭다.

여기서 제시하는 구조화 기법이 범주론의 통찰 없이 발견됐을지는 의심스럽다. 하지만 일단 발견되고 나면 범주론적인 것을 전혀 언급하지 않고도 쉽게 표현할 수 있다. 이 글을 읽는 데 범주론 지식은 필요 없다.

예제는 Haskell로 제시하지만 Haskell 지식도 필요 없다. 필요한 것은 순수·비순수 함수형 프로그래밍의 기초에 대한 대략적인 친숙함뿐이다. 일반적인 배경 지식은 [3, 12]를 보라. 언급되는 언어는 Haskell [4], Miranda [20], Standard ML [11], Scheme [17]이다.

(원문 각주: 이 글은 J. Jeuring과 E. Meijer가 엮은 *Advanced Functional Programming*, 1995년 5월 Båstad Spring School 논문집, Springer Verlag LNCS 925에 실렸다. 이전 판은 M. Broy가 엮은 *Program Design Calculi*, 1992년 7월 30일–8월 8일 Marktoberdorf Summer School 논문집에 실렸다. 2001년 8월에 몇 가지 오류를 고쳤으며, 이를 지적해 준 Dan Friedman에게 감사한다.)

(주1: Miranda는 Research Software Limited의 상표다.)

## 2 모나드 평가하기

순수 함수형 언어에는 이런 장점이 있다. 모든 데이터 흐름이 명시적이다. 그리고 이런 단점이 있다. 때때로 그것이 고통스러울 만큼 명시적이다.

순수 함수형 언어의 프로그램은 등식의 집합으로 작성한다. 명시적 데이터 흐름은 식의 값이 자유 변수에만 의존하도록 보장한다. 따라서 같은 것을 같은 것으로 치환하는 일이 언제나 타당하며, 이 덕분에 이런 프로그램은 추론하기가 특히 쉽다. 명시적 데이터 흐름은 또한 계산 순서가 무관하도록 보장하므로, 이런 프로그램은 지연 평가에 적합하다.

명시적 데이터 흐름이 축복이자 저주가 되는 지점은 모듈성이다. 한편으로 그것은 모듈성의 극치다. 들어오는 데이터와 나가는 데이터가 모두 드러나 접근 가능하므로 유연성이 최대가 된다. 다른 한편으로 그것은 모듈성의 밑바닥이다. 알고리즘의 본질이, 데이터를 생성 지점에서 사용 지점까지 실어 나르는 데 필요한 배관 아래 파묻힐 수 있다.

내가 순수 함수형 언어로 평가기를 하나 작성한다고 하자.

- 여기에 오류 처리를 추가하려면 재귀 호출마다 오류를 검사하고 적절히 처리하도록 수정해야 한다. 예외가 있는 비순수 언어를 썼다면 그런 재구성은 필요 없었을 것이다.
- 수행한 연산 횟수를 세는 기능을 추가하려면 재귀 호출마다 그 횟수를 적절히 넘겨주도록 수정해야 한다. 증가시킬 수 있는 전역 변수가 있는 비순수 언어를 썼다면 그런 재구성은 필요 없었을 것이다.
- 실행 추적을 추가하려면 재귀 호출마다 그 추적을 적절히 넘겨주도록 수정해야 한다. 출력을 부수 효과로 수행하는 비순수 언어를 썼다면 그런 재구성은 필요 없었을 것이다.

아니면 *모나드*를 쓸 수도 있다.

이 글은 위에서 말한 변경이 간단해지도록 모나드로 평가기를 구조화하는 방법을 보인다. 어느 경우든 필요한 것은 모나드를 다시 정의하고 몇 군데를 국소적으로 고치는 일뿐이다.

이 프로그래밍 스타일은 비순수 언어의 여러 기능이 주는 유연성을 어느 정도 되찾아 준다. 대응하는 비순수 기능이 없는 경우에도 적용될 수 있다. 명시적 데이터가 주는 유연성과 암묵적 배관이 주는 간결함 사이의 긴장을 없애 주지는 못하지만, 어느 정도 완화해 준다.

이 기법은 평가기뿐 아니라 광범위한 함수형 프로그램에 적용된다. Glasgow는 여러 해 동안 함수형 언어 Haskell의 컴파일러를 만들어 왔다. 이 컴파일러는 그 자체가 Haskell로 작성됐으며 여기서 설명하는 구조화 기법을 사용한다. 이 논문은 수십 줄짜리 프로그램에서 모나드를 쓰는 모습을 보여 주지만, 우리는 그보다 세 자릿수 더 큰 프로그램에서 모나드를 써 본 경험이 있다.

먼저 단순한 항을 다루는 기본 평가기에서 출발해, 예외·상태·출력을 흉내 내는 변형들을 살펴본다. 이 변형들의 공통점을 분석해 모나드 개념을 추상해 내고, 각 변형이 모나드 틀에 어떻게 들어맞는지 보인다.

### 2.1 변형 0: 기본 평가기

평가기는 항(term)에 작용한다. 설명을 위해 항은 지나치다 싶을 만큼 단순하게 잡았다.

```haskell
data Term = Con Int | Div Term Term
```

항은 정수 `a`를 담은 상수 `Con a`이거나, 항 `t`와 `u`의 몫인 `Div t u`다.

기본 평가기는 단순함 그 자체다.

```haskell
eval           :: Term → Int
eval (Con a)   = a
eval (Div t u) = eval t ÷ eval u
```

함수 `eval`은 항을 정수로 바꾼다. 항이 상수면 그 상수를 반환한다. 항이 몫이면 부분 항들을 평가한 뒤 몫을 계산한다. 정수 나눗셈은 `÷`로 표기한다.

다음이 계속 쓰일 예제다.

```haskell
answer, error :: Term
answer        = (Div (Div (Con 1972) (Con 2)) (Con 23))
error         = (Div (Con 1) (Con 0))
```

`eval answer`를 계산하면 `((1972 ÷ 2) ÷ 23)`의 값인 `42`가 나온다. 기본 평가기에는 오류 처리가 없으므로 `eval error`의 결과는 정의되지 않는다.

### 2.2 변형 1: 예외

두 번째 예제가 의미 있는 오류 메시지를 반환하도록 오류 검사를 추가하고 싶다고 하자. 비순수 언어에서는 예외를 쓰면 간단히 해결된다.

순수 언어에서는 예외를 던질 수 있는 계산을 나타내는 타입을 도입해 예외 처리를 흉내 낼 수 있다.

```haskell
data M a       = Raise Exception | Return a
type Exception = String
```

타입 `M a`의 값은 예외 `e`를 담은 `Raise e`이거나, 타입 `a`의 값을 담은 `Return a`다. 관례상 `a`는 `M a`에서처럼 타입 변수로도 쓰고, `Return a`에서처럼 그 타입의 값을 가리키는 변수로도 쓴다.

(‘data’ 선언과 ‘type’ 선언의 차이를 짚고 넘어가자. ‘data’ 선언은 새 데이터 타입(여기서는 `M`)과 그 타입의 값을 만드는 새 생성자들(여기서는 `Raise`와 `Return`)을 도입한다. ‘type’ 선언은 기존 타입에 새 이름을 붙인다. 여기서는 `Exception`이 `String`의 다른 이름이 된다.)

평가기를 이 표현에 맞게 고치는 일은 단순하지만 지루하다.

```haskell
eval           :: Term → M Int
eval (Con a)   = Return a
eval (Div t u) = case eval t of
                   Raise e  → Raise e
                   Return a →
                     case eval u of
                       Raise e  → Raise e
                       Return b →
                         if b == 0
                           then Raise "divide by zero"
                           else Return (a ÷ b)
```

평가기를 호출할 때마다 결과의 형태를 검사해야 한다. 예외가 던져졌으면 다시 던지고, 값이 반환됐으면 그 값을 처리한다. 새 평가기를 `answer`에 적용하면 `(Return 42)`가 나오고, `error`에 적용하면 `(Raise "divide by zero")`가 나온다.

### 2.3 변형 2: 상태

오류는 잠시 잊고, 평가 도중 수행된 나눗셈 횟수를 세고 싶다고 하자. 비순수 언어에서는 상태를 쓰면 간단히 해결된다. 어떤 변수를 처음에 0으로 놓고 나눗셈이 일어날 때마다 1씩 증가시키면 된다.

순수 언어에서는 상태에 작용하는 계산을 나타내는 타입을 도입해 상태를 흉내 낼 수 있다.

```haskell
type M a   = State → (a, State)
type State = Int
```

이제 타입 `M a`의 값은 초기 상태를 받아 계산된 값과 최종 상태의 쌍을 반환하는 함수다.

역시 평가기를 이 표현에 맞게 고치는 일은 단순하지만 지루하다.

```haskell
eval             :: Term → M Int
eval (Con a) x   = (a, x)
eval (Div t u) x = let (a, y) = eval t x in
                   let (b, z) = eval u y in
                   (a ÷ b, z + 1)
```

평가기를 호출할 때마다 이전 상태를 넘겨주고, 결과에서 새 상태를 꺼내 적절히 전달해야 한다. `eval answer 0`을 계산하면 `(42, 2)`가 나온다. 초기 상태 0에서 답은 42이고 최종 상태는 2이므로, 나눗셈이 두 번 수행됐음을 알 수 있다.

### 2.4 변형 3: 출력

마지막으로, 실행 추적을 표시하고 싶다고 하자. 비순수 언어에서는 적절한 지점에 출력 명령을 끼워 넣으면 간단히 해결된다.

순수 언어에서는 출력을 생성하는 계산을 나타내는 타입을 도입해 출력을 흉내 낼 수 있다.

```haskell
type M a    = (Output, a)
type Output = String
```

이제 타입 `M a`의 값은 생성된 출력과 계산된 값의 쌍이다.

또다시, 평가기를 이 표현에 맞게 고치는 일은 단순하지만 지루하다.

```haskell
eval           :: Term → M Int
eval (Con a)   = (line (Con a) a, a)
eval (Div t u) = let (x, a) = eval t in
                 let (y, b) = eval u in
                 (x ++ y ++ line (Div t u) (a ÷ b), a ÷ b)

line           :: Term → Int → Output
line t a       = "eval (" ++ showterm t ++ ") ⇐ " ++ showint a ++ "↵"
```

평가기를 호출할 때마다 출력들을 모아 감싸는 호출의 출력으로 조립해야 한다. 함수 `line`은 출력 한 줄을 생성한다. 여기서 `showterm`과 `showint`는 항과 정수를 문자열로 바꾸고, `++`는 문자열을 이어 붙이며, `↵`는 개행 문자 하나로 이뤄진 문자열을 나타낸다.

`eval answer`를 계산하면 쌍 `(x, 42)`가 반환된다. 여기서 `x`는 다음 문자열이다.

```text
eval (Con 1972) ⇐ 1972
eval (Con 2) ⇐ 2
eval (Div (Con 1972) (Con 2)) ⇐ 986
eval (Con 23) ⇐ 23
eval (Div (Div (Con 1972) (Con 2)) (Con 23)) ⇐ 42
```

이는 계산의 추적을 나타낸다.

지금까지의 논의만 보면 비순수 언어의 프로그램이 순수 언어의 프로그램보다 수정하기 쉬워 보일 수 있다. 하지만 때로는 그 반대다. 앞의 프로그램을 고쳐 실행 추적을 역순으로 표시하고 싶다고 하자.

```text
eval (Div (Div (Con 1972) (Con 2)) (Con 23)) ⇐ 42
eval (Con 23) ⇐ 23
eval (Div (Con 1972) (Con 2)) ⇐ 986
eval (Con 2) ⇐ 2
eval (Con 1972) ⇐ 1972
```

순수 프로그램에서는 이것이 단순함 그 자체다. 다음 항을

```haskell
x ++ y ++ line (Div t u) (a ÷ b)
```

다음 항으로 바꾸기만 하면 된다.

```haskell
line (Div t u) (a ÷ b) ++ y ++ x
```

비순수 프로그램을 고쳐 같은 효과를 내기는 그렇게 쉽지 않다. 문제는 출력이 계산의 부수 효과로 일어나는데, 이제는 어떤 항의 계산 결과를 그 계산이 생성한 출력보다 *먼저* 표시하고 싶다는 점이다. 여러 방법으로 달성할 수는 있지만, 어느 방법이든 비순수 프로그램을 상당히 뜯어고쳐야 한다.

### 2.5 모나드 평가기

인터프리터의 각 변형에는 비슷한 구조가 있고, 이를 추상하면 *모나드* 개념이 나온다.

각 변형에서 우리는 계산의 타입을 도입했다. `M`은 차례대로 예외를 던질 수 있는 계산, 상태에 작용하는 계산, 출력을 생성하는 계산을 나타냈다. 지금쯤이면 독자도 `M`이 *모나드*(monad)를 뜻한다고 짐작했을 것이다.

원래 평가기의 타입은 `Term → Int`였고, 각 변형에서는 `Term → M Int` 형태가 됐다. 일반적으로 타입 `a → b`의 함수가 타입 `a → M b`의 함수로 바뀐다. 이는 타입 `a`의 인자를 받아 타입 `b`의 결과를 반환하되, `M`이 포착하는 부가 효과가 있을 수 있는 함수라고 읽을 수 있다. 그 효과는 상태에 작용하는 것일 수도, 출력을 생성하는 것일 수도, 예외를 던지는 것일 수도, 그 밖의 무엇일 수도 있다.

타입 `M`에는 어떤 연산이 필요할까? 예제들을 살펴보면 두 가지가 드러난다. 첫째, 값을 받아 그 값을 반환하기만 하고 다른 일은 하지 않는 계산으로 바꾸는 방법이 필요하다.

```haskell
unit :: a → M a
```

둘째, 타입 `a → M b`의 함수를 타입 `M a`의 계산에 적용하는 방법이 필요하다. 인자가 함수보다 앞에 오는 순서로 쓰는 편이 편리하다.

```haskell
(⋆) :: M a → (a → M b) → M b
```

모나드는 타입 생성자 `M`과 위의 다형 타입을 갖는 두 연산으로 이뤄진 삼중항 `(M, unit, ⋆)`이다. 이 연산들은 3장에서 제시하는 세 법칙을 만족해야 한다.

우리는 종종 다음 형태의 식을 쓸 것이다.

```haskell
m ⋆ λa. n
```

여기서 `m`과 `n`은 식이고 `a`는 변수다. `λa. n` 형태는 람다식이며, 묶인 변수 `a`의 유효 범위는 식 `n`이다. 위 식은 이렇게 읽는다. 계산 `m`을 수행하고, 그 결과 값을 `a`에 묶은 뒤, 계산 `n`을 수행한다. 타입이 유용한 길잡이가 된다. `(⋆)`의 타입에서, 식 `m`은 타입 `M a`, 변수 `a`는 타입 `a`, 식 `n`은 타입 `M b`, 람다식 `λa. n`은 타입 `a → M b`, 전체 식은 타입 `M b`임을 알 수 있다.

위 식은 다음 식과 유사하다.

```haskell
let a = m in n
```

비순수 언어에서 이 식도 같은 뜻으로 읽힌다. 계산 `m`을 수행하고, 그 결과 값을 `a`에 묶은 뒤, 계산 `n`을 수행해 그 값을 반환한다. 다만 여기서는 타입이 값과 계산을 구별해 주지 않는다. 식 `m`은 타입 `a`, 변수 `a`는 타입 `a`, 식 `n`은 타입 `b`, 전체 식은 타입 `b`다. ‘let’과의 유비는 `⋆`의 인자 순서를 그렇게 정한 이유를 설명해 준다. 계산 `m`이 계산 `n`보다 먼저 수행되므로, 인자 `m`이 함수 `λa. n`보다 앞에 오는 것이 편리하다.

평가기는 이 추상화를 써서 쉽게 다시 쓸 수 있다.

```haskell
eval           :: Term → M Int
eval (Con a)   = unit a
eval (Div t u) = eval t ⋆ λa. eval u ⋆ λb. unit (a ÷ b)
```

우선순위를 짚고 넘어가자. 람다 추상은 가장 느슨하게, 함수 적용은 가장 단단하게 묶이므로, 마지막 등식은 다음과 같다.

```haskell
eval (Div t u) = ((eval t) ⋆ (λa. ((eval u) ⋆ (λb. (unit (a ÷ b))))))
```

타입 `Term → M Int`는 평가기가 항을 받아 정수를 내놓는 계산을 수행함을 나타낸다. `(Con a)`를 계산하려면 그냥 `a`를 반환한다. `(Div t u)`를 계산하려면 먼저 `t`를 계산해 결과를 `a`에 묶고, 다음 `u`를 계산해 결과를 `b`에 묶은 뒤, `a ÷ b`를 반환한다.

새 평가기는 원래의 기본 평가기보다 조금 복잡하지만 훨씬 유연하다. 위의 변형들 각각은 `M`, `unit`, `⋆`의 정의를 바꾸고 한두 군데만 국소적으로 고치면 얻을 수 있다. 이런 간단한 변경을 위해 평가기 전체를 다시 쓸 필요가 더는 없다.

### 2.6 변형 0 다시 보기: 기본 평가기

가장 단순한 모나드에서 계산은 값과 다를 것이 없다.

```haskell
type M a = a

unit   :: a → I a
unit a = a

(⋆)   :: M a → (a → M b) → M b
a ⋆ k = k a
```

이를 항등 모나드라고 부른다. `M`은 타입에 대한 항등 함수이고, `unit`은 항등 함수이며, `⋆`는 그저 함수 적용이다.

`M`, `unit`, `⋆`를 위와 같이 잡고 2.5절의 모나드 평가기를 단순화하면 2.1절의 기본 평가기가 나온다.

### 2.7 변형 1 다시 보기: 예외

예외 모나드에서 계산은 예외를 던지거나 값을 반환한다.

```haskell
data M a       = Raise Exception | Return a
type Exception = String

unit   :: a → M a
unit a = Return a

(⋆)   :: M a → (a → M b) → M b
m ⋆ k = case m of
          Raise e  → Raise e
          Return a → k a

raise   :: Exception → M a
raise e = Raise e
```

`unit a` 호출은 값 `a`를 그대로 반환한다. `m ⋆ k` 호출은 계산 `m`의 결과를 살펴서, 예외면 다시 던지고, 아니면 반환된 값에 함수 `k`를 적용한다. 항등 모나드의 `⋆`가 함수 적용이었듯이, 예외 모나드의 `⋆`는 일종의 엄격한(strict) 함수 적용으로 볼 수 있다. 마지막으로 `raise e` 호출은 예외 `e`를 던진다.

모나드 평가기에 오류 처리를 추가하려면 모나드를 위와 같이 잡는다. 그런 다음 `unit (a ÷ b)`를 다음으로 바꾸기만 하면 된다.

```haskell
if b == 0
  then raise "divide by zero"
  else unit (a ÷ b)
```

이는 비순수 언어에서 필요한 변경과 맞먹는 수준이다.

예상대로 이 평가기는 2.2절의 예외가 있는 평가기와 동등하다. 실제로 이 절의 `unit`과 `⋆`의 정의를 펼쳐서 단순화하면 그 절의 평가기가 나온다.

### 2.8 변형 2 다시 보기: 상태

상태 모나드에서 계산은 초기 상태를 받아 값과 최종 상태의 쌍을 반환한다.

```haskell
type M a   = State → (a, State)
type State = Int

unit   :: a → M a
unit a = λx. (a, x)

(⋆)   :: M a → (a → M b) → M b
m ⋆ k = λx. let (a, y) = m x in
            let (b, z) = k a y in
            (b, z)

tick :: M ()
tick = λx. ((), x + 1)
```

`unit a` 호출은 초기 상태 `x`를 받아 값 `a`와 최종 상태 `x`를 반환하는 계산을 돌려준다. 즉 `a`를 반환하고 상태는 그대로 둔다. `m ⋆ k` 호출은 초기 상태 `x`에서 계산 `m`을 수행해 값 `a`와 중간 상태 `y`를 얻고, 상태 `y`에서 계산 `k a`를 수행해 값 `b`와 최종 상태 `z`를 얻는다. `tick` 호출은 상태를 증가시키고 빈 값 `()`을 반환한다. `()`의 타입 역시 `()`으로 쓴다.

비순수 언어에서 `tick` 같은 연산은 타입 `() → ()`의 함수로 표현될 것이다. 군더더기 인자 `()`은 함수가 적용될 때까지 효과를 지연하기 위해 필요하고, 출력 타입이 `()`이므로 이 함수의 목적이 부수 효과에 있다고 짐작할 수 있다. 반면 여기서 `tick`의 타입은 `M ()`이다. 군더더기 인자가 필요 없고, `M`이 드러나 있어 어떤 종류의 효과가 일어날 수 있는지 명시적으로 알려 준다.

모나드 평가기에 실행 횟수 세기를 추가하려면 모나드를 위와 같이 잡는다. 그런 다음 `unit (a ÷ b)`를 다음으로 바꾸기만 하면 된다.

```haskell
tick ⋆ λ(). unit (a ÷ b)
```

(여기서 `λ(). e`는 `e`에 나타나지 않는 새 변수 `x :: ()`에 대한 `λx. e`와 같다. 람다식이 묶는 값이 반드시 `()`이어야 함을 나타낸다.) 역시 비순수 언어에서 필요한 변경과 맞먹는 수준이다. 단순화하면 2.3절의 상태가 있는 평가기가 나온다.

### 2.9 변형 3 다시 보기: 출력

출력 모나드에서 계산은 생성된 출력과 반환된 값의 쌍으로 이뤄진다.

```haskell
type M a    = (Output, a)
type Output = String

unit   :: a → M a
unit a = ("", a)

(⋆)   :: M a → (a → M b) → M b
m ⋆ k = let (x, a) = m in
        let (y, b) = k a in
        (x ++ y, b)

out   :: Output → M ()
out x = (x, ())
```

`unit a` 호출은 출력 없음과 `a`의 쌍을 반환한다. `m ⋆ k` 호출은 계산 `m`에서 출력 `x`와 값 `a`를 꺼내고, 계산 `k a`에서 출력 `y`와 값 `b`를 꺼낸 뒤, `x`와 `y`를 이어 붙인 출력과 값 `b`의 쌍을 반환한다. `out x` 호출은 출력이 `x`이고 값이 빈 값 `()`인 계산을 반환한다.

모나드 평가기에 실행 추적을 추가하려면 모나드를 위와 같이 잡는다. 그런 다음 `Con a` 절에서 `unit a`를 다음으로 바꾸고,

```haskell
out (line (Con a) a) ⋆ λ(). unit a
```

`Div t u` 절에서 `unit (a ÷ b)`를 다음으로 바꾼다.

```haskell
out (line (Div t u) (a ÷ b)) ⋆ λ(). unit (a ÷ b)
```

또다시, 비순수 언어에서 필요한 변경과 맞먹는 수준이다. 단순화하면 2.4절의 출력이 있는 평가기가 나온다.

출력을 역순으로 얻으려면 `⋆`의 정의에서 `x ++ y`를 `y ++ x`로 바꾸기만 하면 된다. 이는 순수 프로그램에서 필요한 변경과 맞먹으며, 비순수 프로그램에서 필요한 변경보다는 훨씬 간단하다.

순수 버전과 비순수 버전의 차이 하나로, 비순수 버전은 계산하는 대로 출력을 표시하지만 순수 버전은 전체 계산이 끝날 때까지 아무것도 표시하지 않으리라 생각할 수 있다. 사실 순수 언어가 지연 평가를 하면 계산이 진행됨에 따라 출력이 점진적으로 표시된다. 게다가 출력 순서를 뒤집어도 이런 점진적 표시가 그대로 일어나는데, 비순수 언어에서 그렇게 만들기는 훨씬 어렵다. 실제로 비순수 언어에서 이를 달성하는 가장 쉬운 방법은 지연 평가를 흉내 내는 것일지도 모른다.

## 3 모나드 법칙

모나드의 연산들은 세 가지 법칙을 만족한다.

- **왼쪽 단위원(Left unit).** 값 `a`를 계산하고, 그 결과를 `b`에 묶은 뒤, `n`을 계산한다. 그 결과는 `n`에서 변수 `b`를 값 `a`로 치환한 것과 같다.

  ```haskell
  unit a ⋆ λb. n = n[a/b]
  ```

- **오른쪽 단위원(Right unit).** `m`을 계산하고, 그 결과를 `a`에 묶은 뒤, `a`를 반환한다. 그 결과는 `m`과 같다.

  ```haskell
  m ⋆ λa. unit a = m
  ```

- **결합법칙(Associative).** `m`을 계산해 결과를 `a`에 묶고, `n`을 계산해 결과를 `b`에 묶고, `o`를 계산한다. 이런 계산에서 괄호를 어디에 치든 무관하다.

  ```haskell
  m ⋆ (λa. n ⋆ λb. o) = (m ⋆ λa. n) ⋆ λb. o
  ```

  변수 `a`의 유효 범위가 왼쪽에서는 `o`를 포함하지만 오른쪽에서는 포함하지 않으므로, 이 법칙은 `a`가 `o`에 자유 변수로 나타나지 않을 때만 성립한다.

왼쪽·오른쪽 단위원이 있고 결합법칙을 만족하는 이항 연산을 모노이드(monoid)라 부른다. 모나드가 모노이드와 다른 점은 오른쪽 피연산자에 묶기(binding) 연산이 들어간다는 것이다.

이 법칙들의 유용함을 보이기 위해, 덧셈이 결합법칙을 만족함을 증명해 보자. 나눗셈 대신 덧셈에 기반한 평가기의 변형을 생각한다.

```haskell
data Term = Con Int | Add Term Term

eval           :: Term → M Int
eval (Con a)   = unit a
eval (Add t u) = eval t ⋆ λa. eval u ⋆ λb. unit (a + b)
```

다음 두 항의 평가가

```haskell
Add t (Add u v)   와   Add (Add t u) v
```

같은 결과를 계산함을 보인다. 왼쪽 항을 단순화하면 다음과 같다.

```text
  eval (Add t (Add u v))
= { eval의 정의 }
  eval t ⋆ λa. eval (Add u v) ⋆ λx. unit (a + x)
= { eval의 정의 }
  eval t ⋆ λa. (eval u ⋆ λb. eval v ⋆ λc. unit (b + c)) ⋆ λx. unit (a + x)
= { 결합법칙 }
  eval t ⋆ λa. eval u ⋆ λb. eval v ⋆ λc. unit (b + c) ⋆ λx. unit (a + x)
= { 왼쪽 단위원 }
  eval t ⋆ λa. eval u ⋆ λb. eval v ⋆ λc. unit (a + (b + c))
```

오른쪽 항도 비슷하게 단순화한다.

```text
  eval (Add (Add t u) v)
= { 위와 같은 방식 }
  eval t ⋆ λa. eval u ⋆ λb. eval v ⋆ λc. unit ((a + b) + c)
```

정수 덧셈의 결합법칙에 의해 결과가 따라 나온다. 이 증명은 사소하다. 하지만 모나드 법칙이 없었다면 불가능했을 것이다.

이 증명은 어떤 모나드에서도 성립한다. 예외, 상태, 출력 모두 그렇다. 단, 코드가 위와 같다는 가정하에서다. 코드를 수정하면 증명도 수정해야 한다. 2.3절은 `tick` 호출을 추가해 프로그램을 수정했다. 이 경우에도 결합법칙은 여전히 성립하는데, 다음 법칙으로 보일 수 있다.

```haskell
tick ⋆ λ(). m = m ⋆ λ(). tick
```

이 법칙은 `m` 안에서 상태에 작용하는 것이 `tick`뿐일 때 성립한다. 2.4절은 `line` 호출을 추가해 프로그램을 수정했다. 이 경우 덧셈은 더 이상 결합적이지 않다. 괄호를 바꾸면 추적이 달라진다는 뜻에서 그렇다. 다만 계산되는 값 자체는 여전히 같다.

또 다른 예로, 모든 모나드에서 다음 연산을 정의할 수 있다.

```haskell
map     :: (a → b) → (M a → M b)
map f m = m ⋆ λa. unit (f a)

join    :: M (M a) → M a
join z  = z ⋆ λm. m
```

`map` 연산은 계산이 내놓는 결과에 함수를 적용할 뿐이다. `map f m`을 계산하려면 먼저 `m`을 계산해 결과를 `a`에 묶고 `f a`를 반환한다. `join` 연산은 좀 더 까다롭다. `z`를 그 자신이 계산을 내놓는 계산이라 하자. `join z`를 계산하려면 먼저 `z`를 계산해 결과를 `m`에 묶은 뒤, 계산 `m`처럼 행동한다. 즉 `join`은 머리가 어지러운 이중 계산 층을 평범한 단일 계산 층으로 납작하게 만든다. 5.1절에서 보겠지만 리스트는 모나드를 이루며, 이 모나드에서 `map`은 리스트의 각 원소에 함수를 적용하고 `join`은 리스트의 리스트를 이어 붙인다.

항등 함수를 `id`(`id x = x`), 함수 합성을 `(·)`(`(f · g) x = f (g x)`)로 쓰면 다음과 같은 여러 법칙을 정식화할 수 있다.

```haskell
map id          = id
map (f · g)     = map f · map g
map f · unit    = unit · f
map f · join    = join · map (map f)
join · unit     = id
join · map unit = id
join · map join = join · join
m ⋆ k           = join (map k m)
```

각각의 증명은 `map`과 `join`의 정의와 세 모나드 법칙에서 곧바로 따라 나온다.

모나드는 `unit`과 `⋆`가 아니라 `unit`, `join`, `map`으로 정의되는 경우도 많다[10, 13]. 그 경우 세 모나드 법칙은 위 여덟 법칙 중 앞의 일곱으로 대체된다. `⋆`를 여덟 번째 법칙으로 정의하면 세 모나드 법칙이 따라 나온다. 따라서 두 정의는 동등하다.

## 4 상태

배열은 현재의 컴퓨터 아키텍처와 잘 맞기 때문에 컴퓨팅에서 중심 역할을 한다. 프로그램에는 `x[i]` 같은 배열 조회와 `x[i] := v` 같은 배열 갱신이 널려 있다. 이 연산들이 인기 있는 이유는 배열 조회는 인덱스 접근(indexed fetch) 명령어 하나로, 배열 갱신은 인덱스 저장(indexed store) 명령어 하나로 구현되기 때문이다.

함수형 언어에 배열을 추가하기는 쉽고, 효율적인 배열 조회를 제공하기도 쉽다. 반면 효율적인 배열 갱신을 어떻게 제공할 것인가는 오랜 역사를 가진 문제다. 모나드는 이 오래된 문제에 새로운 답을 준다.

프로그램을 배열 갱신에 기반해 짜는 것이 바람직한가 역시 오랜 역사를 가진 문제다. 배열에 기반한 알고리즘과 아키텍처에 이미 엄청난 노력이 투입돼 왔으므로, 이 논쟁은 비켜 가고 답이 ‘그렇다’라고 그냥 가정하겠다.

앞 장에서 모나드를 쓴 방식과 이 장에서 쓰는 방식 사이에는 중요한 차이가 있다. 앞 장은 모나드가 기존 언어 기능을 더 효과적으로 쓰도록 돕는다는 것을 보였고, 이 장은 모나드가 새로운 언어 기능을 정의하는 데 어떻게 도움이 되는지 보인다. 프로그래밍 언어를 바꿀 필요는 없지만, 구현이 새로운 추상 데이터 타입을(아마도 표준 프렐류드의 일부로) 제공해야 한다.

여기서는 프로그램 내부의 상태를 다루는 데 모나드를 쓰지만, 같은 기법으로 외부 상태도 다룰 수 있다. 입출력을 수행하거나 다른 프로그래밍 언어와 통신하는 것이다. Haskell의 Glasgow 구현은 모나드에 기반한 설계로 입출력과 명령형 언어 C와의 연동을 제공한다[15]. 이 설계는 Haskell 표준 1.3 버전에 채택됐다.

### 4.1 배열

`Arr`를 타입 `Ix`의 인덱스를 받아 타입 `Val`의 값을 내놓는 배열의 타입이라 하자. 이 타입의 핵심 연산은 다음과 같다.

```haskell
newarray :: Val → Arr
index    :: Ix → Arr → Val
update   :: Ix → Val → Arr → Arr
```

`newarray v` 호출은 모든 항목이 `v`로 설정된 배열을 반환한다. `index i x` 호출은 배열 `x`의 인덱스 `i`에 있는 값을 반환한다. `update i v x` 호출은 인덱스 `i`의 값이 `v`이고 나머지는 `x`와 동일한 배열을 반환한다. 이 연산들의 행동은 다음 법칙으로 명세된다.

```haskell
index i (newarray v)  = v
index i (update i v x) = v
index i (update j v x) = index i x,  단 i ≠ j
```

실제로는 이 연산들이 더 복잡할 것이다. 예컨대 인덱스 범위를 지정하는 방법이 필요하다. 하지만 요점을 설명하기에는 위 정도로 충분하다.

`update` 연산을 효율적으로 구현하는 방법은 배열의 해당 항목을 덮어쓰는 것이다. 그러나 순수 함수형 언어에서 이것이 안전하려면 갱신 시점에 그 배열을 가리키는 다른 포인터가 존재하지 않아야 한다. 이 성질을 만족하는 배열을, Schmidt [18]를 따라 단일 스레드(single threaded) 배열이라 부른다.

단순한 명령형 언어의 인터프리터를 만드는 문제를 생각해 보자. 이 언어의 추상 구문은 다음 데이터 타입으로 표현된다.

```haskell
data Term = Var Id | Con Int | Add Term Term
data Comm = Asgn Id Term | Seq Comm Comm | If Term Comm Comm
data Prog = Prog Comm Term
```

여기서 `Id`는 명시하지 않은 식별자 타입이다. 항은 변수, 상수, 또는 두 항의 합이다. 명령은 대입, 두 명령의 순차 실행, 또는 조건문이다. 프로그램은 명령 하나와 항 하나로 이뤄진다.

실행의 현재 상태는 인덱스가 식별자이고 대응 값이 정수인 배열로 모델링한다.

```haskell
type State = Arr
type Ix    = Id
type Val   = Int
```

인터프리터는 다음과 같다.

```haskell
eval             :: Term → State → Int
eval (Var i) x   = index i x
eval (Con a) x   = a
eval (Add t u) x = eval t x + eval u x

exec              :: Comm → State → State
exec (Asgn i t) x = update i (eval t x) x
exec (Seq c d) x  = exec d (exec c x)
exec (If t c d) x = if eval t x == 0 then exec c x else exec d x

elab            :: Prog → Int
elab (Prog c t) = eval t (exec c (newarray 0))
```

이는 표시적 의미론과 매우 닮았다. 항 평가기는 항과 상태를 받아 정수를 반환한다. 변수 평가는 상태를 인덱싱하는 것으로 구현된다. 명령 실행기는 명령과 초기 상태를 받아 최종 상태를 반환한다. 대입은 상태를 갱신하는 것으로 구현된다. 프로그램 상세화기(elaborator)는 프로그램을 받아 정수를 반환한다. 모든 식별자가 0에 대응되는 초기 상태에서 명령을 실행한 뒤, 그 결과 상태에서 주어진 식을 평가해 값을 반환한다.

이 인터프리터의 상태는 단일 스레드다. 실행의 어떤 순간에도 상태를 가리키는 포인터는 하나뿐이므로, 상태를 제자리에서 갱신해도 안전하다. 이것이 성립하려면 `update` 연산이 새 값을 배열에 넣기 전에 먼저 평가해야 한다. 그러지 않으면 배열이 자신을 가리키는 포인터를 담은 클로저를 품게 되어 단일 스레드 성질이 깨질 수 있다. 의미론 용어로는 `update`가 세 인자 모두에 대해 엄격하다(strict)고 말한다.

주어진 함수형 프로그램이 배열을 단일 스레드 방식으로 쓰는지 판정하는 분석을 여러 연구자가 제안했고, 그런 분석을 최적화 컴파일러에 넣으려는 의도였다. 이 분석들 대부분은 추상 해석(abstract interpretation)에 기반한다[1]. 이 분야에서 어느 정도 성과가 있었지만, 이 분석들은 감당하기 어려울 만큼 비용이 큰 경향이 있다[2, 7].

설령 그런 분석이 실용적이더라도 그것을 쓰는 것이 현명하지 않을 수 있다. `update`의 최적화 여부는 프로그램의 시간·공간 사용량을 열 배 이상 좌우할 수 있다. 프로그램이 충분히 빠르게, 주어진 공간 안에서 돌아가리라는 것을 알려면 프로그래머는 그 최적화가 반드시 일어난다는 보장을 받아야 한다. 배열이 단일 스레드여야 함을 최적화 컴파일러의 변덕에 맡기기보다 프로그래머가 명시적으로 표시하는 편이 나을 수 있다.

배열이 단일 스레드임을 표시하는 기법 역시 여러 연구자가 제안했다. 이 기법들 대부분은 타입 시스템에 기반한다[6, 19, 22]. 유망한 분야지만 이 타입 시스템들의 복잡도는 여전히 만만치 않다.

다음 절은 배열이 단일 스레드로 쓰이리라는 의도를 명시적으로 표시하는 또 다른 방법을 제시한다. 당연히 모나드에 기반한다. 이 방법의 장점은 추상 데이터 타입이라는 아이디어만 사용해 기존 타입 시스템 위에서 그대로 동작한다는 것이다.

### 4.2 배열 변환기

배열 변환기(array transformer) 모나드는 상태를 배열로 잡은 상태 변환기 모나드일 뿐이다. `M`, `unit`, `⋆`의 정의는 전과 같다.

```haskell
type M a   = State → (a, State)
type State = Arr

unit   :: a → M a
unit a = λx. (a, x)

(⋆)   :: M a → (a → M b) → M b
m ⋆ k = λx. let (a, y) = m x in
            let (b, z) = k a y in
            (b, z)
```

앞에서는 상태가 정수였고 상태에 작용하는 부가 연산 `tick`이 있었다. 이제 상태는 배열이고, 배열 생성·인덱싱·갱신에 대응하는 부가 연산들이 있다.

```haskell
block     :: Val → M a → a
block v m = let (a, x) = m (newarray v) in a

fetch   :: Ix → M Val
fetch i = λx. (index i x, x)

assign     :: Ix → Val → M ()
assign i v = λx. ((), update i v x)
```

`block v m` 호출은 모든 위치가 `v`로 초기화된 새 배열을 만들고, 이 초기 상태에 모나드 `m`을 적용해 값 `a`와 최종 상태 `x`를 얻은 뒤, 배열을 해제하고 `a`를 반환한다. `fetch i` 호출은 현재 상태의 인덱스 `i`에 있는 값을 반환하고 상태는 그대로 둔다. `assign i v` 호출은 빈 값 `()`을 반환하고, 인덱스 `i`가 값 `v`를 담도록 상태를 갱신한다.

조금만 생각해 보면 이 연산들이 실제로 단일 스레드임을 알 수 있다. 배열을 복제할 수 있는 유일한 연산은 `fetch`인데, 이는 다음과 같이 구현할 수 있다. 먼저 배열에서 주어진 인덱스의 항목을 가져온 다음, 이 값과 배열 포인터로 이뤄진 쌍을 반환한다. 의미론 용어로 말하면 `fetch`는 배열과 인덱스에 대해서는 엄격하지만 그 인덱스에 놓인 값에 대해서는 엄격하지 않고, `assign`은 배열과 인덱스에 대해서는 엄격하지만 대입되는 값에 대해서는 엄격하지 않다.

(이는 앞 절과 다르다. 앞 절에서는 인터프리터가 단일 스레드이기 위해 `update`가 주어진 값에 대해 엄격해야 했다. 이 절에서는 곧 보겠지만 그 엄격성이 사라지는 대신 항 평가에 불필요한 순차성이 도입된다. 다음 절에서는 그 불필요한 순차성이 사라지는 대신 엄격성이 다시 도입된다.)

이제 `M`을 `unit`, `⋆`, `block`, `fetch`, `assign` 다섯 연산을 지원하는 추상 데이터 타입으로 만들 수 있다. `block` 연산이 중심 역할을 하는데, 결과 타입에 `M`이 없는 유일한 연산이기 때문이다. `block`이 없다면 `M`을 쓰면서 출력 타입에 `M`이 없는 프로그램을 작성할 방법이 없다.

`M`을 추상 데이터 타입으로 만들면 단일 스레드 성질이 보존됨이 보장되고, 따라서 대입을 제자리 갱신으로 구현해도 안전하다. 이 목적에는 데이터 추상화가 필수다. 그러지 않으면 다음처럼 단일 스레드 성질을 위반하는 프로그램을 쓸 수 있다.

```haskell
λx. (assign i v x, assign i w x)
```

이제 인터프리터를 다음과 같이 다시 쓸 수 있다.

```haskell
eval           :: Term → M Int
eval (Var i)   = fetch i
eval (Con a)   = unit a
eval (Add t u) = eval t ⋆ λa. eval u ⋆ λb. unit (a + b)

exec            :: Comm → M ()
exec (Asgn i t) = eval t ⋆ λa. assign i a
exec (Seq c d)  = exec c ⋆ λ(). exec d ⋆ λ(). unit ()
exec (If t c d) = eval t ⋆ λa.
                    if a == 0 then exec c else exec d

elab            :: Prog → Int
elab (Prog c t) = block 0 (exec c ⋆ λ(). eval t ⋆ λa. unit a)
```

타입을 보면 항의 평가는 정수를 반환하며 상태에 접근하거나 상태를 수정할 수 있고, 명령의 실행은 아무것도 반환하지 않으며 상태에 접근하거나 상태를 수정할 수 있음을 알 수 있다. 실제로는 평가가 상태에 접근만 하고 절대 바꾸지 않는데, 이를 표시할 수 있는 더 정교한 체계를 곧 살펴본다.

`M`의 추상 데이터 타입은 제자리 갱신이 안전함을 보장한다. 특별한 분석 기법이 전혀 필요 없다. 모나드 인터프리터가 원래 인터프리터에서 어떻게 유도되는지는 쉽게 알 수 있고, (앞서 준 정의를 쓰면) 둘의 동등성 증명도 단순하다.

다시 쓴 인터프리터는 이전 버전보다 약간 길지만, 어쩌면 약간 더 읽기 쉽다. 예컨대 `(Seq c d)`의 실행은 이렇게 읽힌다. `c`의 실행을 계산하고, 다음 `d`의 실행을 계산한 뒤, 아무것도 반환하지 않는다. `exec d`가 `exec c`의 왼쪽에 나타난다는 불안한 성질을 가진 이전 버전과 비교해 보라.

이 프로그램의 한 가지 단점은 순차성이 지나치게 많다는 것이다. `(Add t u)`의 평가는 이렇게 읽힌다. `t`의 평가를 계산해 결과를 `a`에 묶고, 다음 `u`의 평가를 계산해 결과를 `b`에 묶은 뒤, `a + b`를 반환한다. 원래 프로그램에는 없던 불필요한 순서가 `t`와 `u`의 평가에 강제된다는 점에서 아쉽다. `eval`이 상태에 의존하기는 해도 상태를 바꾸지는 않으므로 이 순서는 중요하지 않다. 이를 바로잡기 위해 상태 변환기 모나드 `M`에 상태 판독기(state reader) 모나드 `M′`를 추가한다.

### 4.3 배열 판독기

배열 변환기 모나드는 초기 배열을 받아 값과 최종 배열을 반환한다는 점을 상기하자.

```haskell
type M a   = State → (a, State)
type State = Arr
```

이에 대응하는 배열 판독기(array reader) 모나드는 배열을 받아 값을 반환한다. 배열은 원래 배열과 동일하다고 가정하므로 반환하지 않는다.

```haskell
type M′ a = State → a

unit′   :: a → M′ a
unit′ a = λx. a

(⋆′)   :: M′ a → (a → M′ b) → M′ b
m ⋆′ k = λx. let a = m x in k a x

fetch′   :: Ix → M′ Val
fetch′ i = λx. index i x
```

`unit′ a` 호출은 주어진 상태 `x`를 무시하고 `a`를 반환한다. `m ⋆′ k` 호출은 주어진 상태 `x`에서 계산 `m`을 수행해 값 `a`를 얻고, 같은 상태 `x`에서 계산 `k a`를 수행한다. 즉 `unit′`는 상태를 버리고 `⋆′`는 상태를 복제한다. `fetch′ i` 호출은 주어진 상태 `x`의 인덱스 `i`에 있는 값을 반환한다.

상태를 읽기만 하는 계산은 상태를 읽고 쓸 수 있는 계산의 부분집합임이 분명하다. 따라서 모나드 `M′`의 계산을 모나드 `M`의 계산으로 강제 변환(coerce)하는 방법이 있어야 한다.

```haskell
coerce   :: M′ a → M a
coerce m = λx. let a = m x in (a, x)
```

`coerce m` 호출은 초기 상태 `x`에서 계산 `m`을 수행해 `a`를 얻고, `a`와 상태 `x`의 쌍을 반환한다. 함수 `coerce`는 곧 논의할 여러 수학적 성질을 갖는다.

이번에도 적절히 구현하면 이 연산들은 단일 스레드 성질을 유지한다. `⋆′`와 `coerce`의 정의는 둘 다 중간 값 `a`에 대해 엄격해야 한다. 그러면 `coerce m`이 상태 `x`에서 수행될 때, 쌍 `(a, x)`가 반환되기 전에 `m x`의 계산이 상태 `x`를 가리키는 포인터가 남아 있지 않은 형태 `a`로 환원됨이 보장된다. 따라서 상태가 갱신되는 시점에는 언제나 상태를 가리키는 포인터가 하나뿐이다.

다음 법칙을 만족하는 모나드를 가환(commutative) 모나드라 한다.

```haskell
m ⋆ λa. n ⋆ λb. o = n ⋆ λb. m ⋆ λa. o
```

변수 `a`의 유효 범위가 오른쪽에서는 `n`을 포함하고 왼쪽에서는 포함하지 않으므로, 이 법칙은 `a`가 `n`에 자유 변수로 나타나지 않을 때만 유효하다. 마찬가지로 `b`도 `m`에 자유 변수로 나타나지 않아야 한다. 가환 모나드에서는 계산 순서가 중요하지 않다.

상태 판독기 모나드는 가환이고, 상태 변환기 모나드는 그렇지 않다. 그러므로 상태 판독기 모나드의 계산에는 불필요한 순서가 강제되지 않는다. 특히 `m ⋆′ k` 호출은 `m`과 `k a`를 병렬로 계산하도록 안전하게 구현할 수 있다. 다만 최종 결과는 여전히 `a`에 대해 엄격해야 한다. 예컨대 GRIP 프로세서에서 쓰는 어노테이션으로 `⋆′`를 다음과 같이 정의할 수 있다.

```haskell
m ⋆′ k = λx. let a = m x in
             let b = k a x in
             par a (par b (seq a b))
```

두 번의 `par` 호출이 `a`와 `b`의 병렬 계산을 촉발하고(spark), `seq` 호출은 `a`가 바닥(bottom)이 아닌 값으로 환원되기를 기다린 뒤 `b`를 반환한다.

이 연산들은 두 추상 데이터 타입 `M`과 `M′`로 묶을 수 있으며, `unit`, `⋆`, `unit′`, `⋆′`, `block`, `assign`, `fetch′`, `coerce` 여덟 연산을 지원한다. 추상화가 단일 스레드 성질을 보장하므로 `assign`은 제자리 갱신으로 구현할 수 있다.

인터프리터를 다시 한번 고쳐 쓸 수 있다.

```haskell
eval           :: Term → M′ Int
eval (Var i)   = fetch′ i
eval (Con a)   = unit′ a
eval (Add t u) = eval t ⋆′ λa. eval u ⋆′ λb. unit′ (a + b)

exec            :: Comm → M ()
exec (Asgn i t) = coerce (eval t) ⋆ λa. assign i a
exec (Seq c d)  = exec c ⋆ λ(). exec d ⋆ λ(). unit ()
exec (If t c d) = coerce (eval t) ⋆ λa.
                    if a == 0 then exec c else exec d

elab            :: Prog → Int
elab (Prog c t) = block 0 (exec c ⋆ λ(). coerce (eval t) ⋆ λa. unit a)
```

이전 버전과의 차이는 `eval`이 `M`이 아니라 `M′`로 작성됐고, 다른 두 함수에서 `eval` 호출을 `coerce` 호출이 감싼다는 점이다. 새 타입은 `eval`이 상태에 의존하되 바꾸지는 않으며, `exec`는 상태에 의존할 수도 있고 바꿀 수도 있음을 분명히 드러낸다.

이전 버전의 과도한 순차성은 사라졌다. `(Add t u)`의 평가에서 두 부분식은 어느 순서로든, 또는 동시에 평가해도 된다.

모나드 `M′`에서 모나드 `M`으로의 모나드 사상(monad morphism)이란 모나드 구조를 보존하는 함수 `h :: M′ a → M a`다.

```haskell
h (unit′ a)     = unit a
h (m ⋆′ λa. n) = (h m) ⋆ λa. (h n)
```

어떤 목적을 위해 여러 모나드의 조합을 쓰고 싶은 경우가 자주 있는데, 이때 한 모나드에서 다른 모나드로 변환하는 핵심 역할을 모나드 사상이 맡는다[9].

특히 `coerce`는 모나드 사상이며, 이로부터 두 버전의 인터프리터가 동등하다는 사실이 즉시 따라 나온다.

### 4.4 결론

함수형 언어에서 제자리 배열 갱신을 어떻게 제공할 것인가는 오래된 문제다. 이 장은 두 추상 데이터 타입과 그 사이의 여덟 연산으로 이뤄진 새로운 해법을 제시했다. 이 타입들의 구현을(아마도 표준 프렐류드의 일부로) 제공하는 것 외에 프로그래밍 언어를 바꿀 필요는 전혀 없다. 이제껏 제안된 훨씬 정교한 해법들이 수두룩하다는 점을 생각하면, 이렇게 단순한 해법의 발견은 놀라움으로 다가온다.

같은 해법을 컨티뉴에이션 전달 스타일(continuation passing style)로 표현하는 다른 방법을 이후 Hudak이 제안했다[8]. 하지만 Hudak의 해법은 모나드 해법에서 영감을 받은 것이며, 모나드 해법이 여전히 몇 가지 작은 장점을 갖는 것으로 보인다[15].

이 해법이 왜 20년 전에 발견되지 않았을까? 한 가지 가능한 이유는 이 데이터 타입들이 고차 함수를 본질적인 방식으로 사용한다는 점이다. 배열의 통상적인 공리화는 1차 함수만 다루므로, 고차 함수에 기반한 추상 데이터 타입을 찾아볼 생각을 아무도 하지 못했을 것이다. 모나드가 이 해법의 발견으로 이어졌다는 사실은 모나드에 유리한 점수로 쳐 줘야 한다.

## 5 파서

파서는 이론 전산학의 위대한 성공 사례다. BNF 형식론은 프로그래밍 언어의 구문을 간결하고 정밀하게 기술하는 방법을 제공한다. 수학적 검사로 BNF 문법이 모호한지, 공허한지 판정할 수 있다. 변환을 통해 파싱하기 더 쉬운 동등한 문법을 만들 수 있다. 컴파일러-컴파일러는 문법의 고수준 명세를 효율적인 프로그램으로 바꿔 준다.

이 장은 모나드가 재귀 하강 파서(recursive descent parser)를 만드는 단순한 틀을 제공함을 보인다. 이는 그 자체로도 흥미롭고, 파싱의 기본 구조인 순차(sequencing)와 선택(alternation)이 컴퓨팅 전반의 근본이라는 점에서도 흥미롭다. 또한 모나드가 백트래킹(즉 천사적 비결정성, angelic non-determinism)을 어떻게 모델링할 수 있는지 보여 주는 예이기도 하다.

### 5.1 리스트

우리의 파서 표현은 리스트에 의존한다. 리스트는 함수형 프로그래밍 어디에나 있으며, 지금까지 리스트를 거의 언급하지 않고 버텨 왔다는 사실이 오히려 놀랍다. 사실 리스트는 변장한 채 이미 등장했다. 문자열이 곧 문자의 리스트이기 때문이다.

표기법을 복습하자. 원소가 모두 타입 `a`인 리스트의 타입을 `[a]`로 쓰고, ‘cons’를 `:`로 쓴다. 따라서 `[1, 2, 3] = 1 : 2 : 3 : []`이고, 둘 다 타입은 `[Int]`다. 문자열은 문자의 리스트이므로 `String`과 `[Char]`는 동등하며, `"monad"`는 `['m', 'o', 'n', 'a', 'd']`의 줄임 표기일 뿐이다.

리스트가 모나드를 이룬다는 것은 그리 놀랍지 않을 것이다.

```haskell
unit   :: a → [a]
unit a = [a]

(⋆)         :: [a] → (a → [b]) → [b]
[] ⋆ k      = []
(a : x) ⋆ k = k a ++ (x ⋆ k)
```

`unit a` 호출은 `a` 하나를 담은 단위 리스트를 만들 뿐이다. `m ⋆ k` 호출은 리스트 `m`의 각 원소에 `k`를 적용하고, 그 결과 리스트들을 이어 붙인다.

모나드가 효과를 캡슐화하고 리스트가 모나드를 이룬다면, 리스트는 어떤 효과에 대응할까? 실제로 대응하는 효과가 있으며, 그것은 바로 선택(choice)이다. 타입 `[a]`의 계산을 리스트의 각 원소마다 하나씩, 값들의 선택지를 제시하는 것으로 생각할 수 있다. 타입 `a → b`의 함수에 대응하는 모나드적 대상은 타입 `a → [b]`의 함수다. 이는 각 인자에 대해 결과의 선택지를 제시하므로 관계(relation)에 대응한다. 연산 `unit`은 각 인자를 자기 자신에만 대응시키는 항등 관계에 대응한다. `k :: a → [b]`이고 `h :: b → [c]`이면,

```haskell
λa. k a ⋆ λb. h b :: a → [c]
```

는 `k`와 `h`의 관계 합성에 대응한다.

리스트 조건제시법(list comprehension) 표기는 리스트를 다루는 편리한 방법을 제공한다. 그 행동은 집합 조건제시법과 유사하지만 순서가 유의미하다는 점이 다르다. 예를 들면 다음과 같다.

```haskell
[ sqr a | a ← [1, 2, 3] ]           = [1, 4, 9]
[ (a, b) | a ← [1, 2], b ← "list" ] = [(1, 'l'), (1, 'i'), (1, 's'), (1, 't'),
                                       (2, 'l'), (2, 'i'), (2, 's'), (2, 't')]
```

리스트 조건제시법 표기는 모나드 연산으로 깔끔하게 번역된다.

```haskell
[ t | x ← u ]        = u ⋆ λx. unit t
[ t | x ← u, y ← v ] = u ⋆ λx. v ⋆ λy. unit t
```

여기서 `t`는 식, `x`와 `y`는 변수(더 일반적으로는 패턴), `u`와 `v`는 리스트로 평가되는 식이다. 조건제시법과 모나드의 관계는 다른 곳에서 자세히 다뤘다[21].

### 5.2 파서 표현하기

파서는 상태 변환기와 비슷한 방식으로 표현한다.

```haskell
type M a   = State → [(a, State)]
type State = String
```

즉 타입 `a`에 대한 파서는 파싱할 문자열을 나타내는 상태를 받아, 문자열에서 파싱해 낸 타입 `a`의 값과 아직 파싱하지 않은 나머지 문자열을 나타내는 상태의 쌍들로 이뤄진 리스트를 반환한다. 이 리스트는 입력 상태에 대해 가능한 모든 대안적 파스를 나타낸다. 파스가 없으면 빈 리스트, 파스가 하나면 원소 하나, 서로 다른 파스가 둘 가능하면 원소 둘, 이런 식이다.

앞에서 다룬 타입의 트리를 반환하는, 산술식에 대한 단순한 파서를 생각해 보자.

```haskell
data Term = Con Int | Div Term Term
```

이런 항에 대한 파서가 있다고 하자.

```haskell
term :: M Term
```

사용 예는 다음과 같다.

```haskell
term "23"                = [(Con 23, "")]
term "23 and more"       = [(Con 23, " and more")]
term "not a term"        = []
term "((1972÷2)÷23)"     = [(Div (Div (Con 1972) (Con 2)) (Con 23), "")]
```

모든 입력 `x`에 대해 가능한 파스의 리스트 `m x`가 비어 있거나 정확히 한 원소를 가지면, 파서 `m`은 모호하지 않다(unambiguous)고 한다. 예컨대 `term`은 모호하지 않다. 모호한 파서는 둘 이상의 대안적 파싱을 담은 리스트를 반환할 수 있다.

### 5.3 항목 하나 파싱하기

가장 기본적인 파서는 입력의 첫 항목을 반환하고, 입력이 소진됐으면 실패한다.

```haskell
item         :: M Char
item []      = []
item (a : x) = [(a, x)]
```

두 가지 예다.

```haskell
item ""      = []
item "monad" = [('m', "onad")]
```

`item`은 분명히 모호하지 않다.

### 5.4 순차

파서를 모나드로 만들려면 연산 `unit`과 `⋆`가 필요하다.

```haskell
unit     :: a → M a
unit a x = [(a, x)]

(⋆)       :: M a → (a → M b) → M b
(m ⋆ k) x = [ (b, z) | (a, y) ← m x, (b, z) ← k a y ]
```

파서 `unit a`는 입력 `x`를 받아, 값 `a`와 나머지 입력 `x`의 쌍인 파스 하나를 내놓는다. 파서 `m ⋆ k`는 입력 `x`를 받는다. 파서 `m`을 입력 `x`에 적용해 각 파스마다 값 `a`와 나머지 입력 `y`를 얻고, 파서 `k a`를 입력 `y`에 적용해 각 파스마다 값 `b`와 최종 나머지 입력 `z`를 얻는다.

즉 `unit`은 입력을 소비하지 않는 빈 파서에 대응하고, `⋆`는 파서의 순차 결합에 대응한다.

항목 둘은 다음과 같이 파싱할 수 있다.

```haskell
twoItems :: M (Char, Char)
twoItems = item ⋆ λa. item ⋆ λb. unit (a, b)
```

두 가지 예다.

```haskell
twoItems "m"     = []
twoItems "monad" = [(('m', 'o'), "nad")]
```

파스는 리스트에 항목이 둘 이상 있을 때만 성공한다.

세 모나드 법칙은 빈 파서가 순차 결합의 단위원이며 순차 결합이 결합적임을 표현한다.

```haskell
unit a ⋆ λb. n          = n[a/b]
m ⋆ λa. unit a          = m
m ⋆ (λa. n ⋆ λb. o)     = (m ⋆ λa. n) ⋆ λb. o
```

`m`이 모호하지 않고 모든 `a`에 대해 `k a`가 모호하지 않으면, `m ⋆ k`도 모호하지 않다.

### 5.5 선택

파서는 선택(alternation)으로도 결합할 수 있다.

```haskell
zero   :: M a
zero x = []

(⊕)       :: M a → M a → M a
(m ⊕ n) x = m x ++ n x
```

파서 `zero`는 입력 `x`를 받아 언제나 실패한다. 파서 `m ⊕ n`은 입력 `x`를 받아, `m`을 입력 `x`에 적용한 모든 파스와 `n`을 같은 입력 `x`에 적용한 모든 파스를 내놓는다.

입력에서 항목을 하나 또는 둘 파싱하는 파서는 다음과 같다.

```haskell
oneOrTwoItems :: M String
oneOrTwoItems =   (item ⋆ λa. unit [a])
                ⊕ (item ⋆ λa. item ⋆ λb. unit [a, b])
```

세 가지 예다.

```haskell
oneOrTwoItems ""      = []
oneOrTwoItems "m"     = [("m", "")]
oneOrTwoItems "monad" = [("m", "onad"), ("mo", "nad")]
```

마지막 예는 대안적 파스 둘을 내놓는다. 선택이 모호한 파서를 만들 수 있음을 보여 준다.

언제나 실패하는 파서는 선택의 단위원이고, 선택은 결합적이다.

```haskell
zero ⊕ n    = n
m ⊕ zero    = m
m ⊕ (n ⊕ o) = (m ⊕ n) ⊕ o
```

나아가 `zero`는 실제로 `⋆`의 영원(zero)이며, `⋆`는 `⊕`에 대해 분배된다.

```haskell
zero ⋆ k     = zero
m ⋆ λa. zero = zero
(m ⊕ n) ⋆ k  = (m ⋆ k) ⊕ (n ⋆ k)
```

`⋆`가 `⊕`에 대해 오른쪽으로 분배되지 않는 것은 오직 우리가 대안적 파스를 순서 있는 리스트로 표현하기 때문이다. 순서 없는 백(bag)을 썼다면 `m ⋆ λa. (k a ⊕ h a) = (m ⋆ k) ⊕ (m ⋆ h)`도 성립했을 것이다. 모호하지 않은 파서는 길이가 많아야 1인 리스트를 내놓으므로 순서가 무의미하고, 따라서 어느 한쪽이 모호하지 않을 때는(그러면 양쪽 다 모호하지 않다) 이 법칙도 성립한다.

### 5.6 필터링

파서를 술어와 결합해 필터링할 수 있다.

```haskell
(▷)   :: M a → (a → Bool) → M a
m ▷ p = m ⋆ λa. if p a then unit a else zero
```

파서 `m`과 값에 대한 술어 `p`가 주어지면, 파서 `m ▷ p`는 파서 `m`을 적용해 값 `a`를 얻는다. `p a`가 성립하면 값 `a`로 성공하고, 아니면 실패한다. 필터링이 앞서 정의한 연산자들만으로 작성되고 상태를 직접 참조할 필요가 없다는 점에 주목하라.

`isLetter`와 `isDigit`을 이름 그대로의 술어라 하자. 다음은 두 파서다.

```haskell
letter :: M Char
letter = item ▷ isLetter

digit :: M Int
digit = (item ▷ isDigit) ⋆ λa. unit (ord a − ord '0')
```

첫 번째는 다음 입력 항목이 글자일 때만 성공하고, 두 번째는 숫자일 때만 성공한다. 두 번째는 문자를 ASCII 코드로 바꾸는 `ord :: Char → Int`를 사용해 숫자를 해당 값으로 변환하기도 한다. `▷`가 `⋆`보다 우선순위가 높다고 가정하면 두 번째 정의에서 괄호 일부를 뺄 수 있다.

리터럴에 대한 파서는 지정된 문자 하나를 인식한다.

```haskell
lit   :: Char → M Char
lit c = item ▷ (λa. a == c)
```

파서 `lit c`는 입력이 문자 `c`로 시작하면 성공하고, 아니면 실패한다.

```haskell
lit 'm' "monad" = [('m', "onad")]
lit 'm' "parse" = []
```

앞의 법칙들로부터, 필터링이 `zero`를 보존하고 선택에 대해 분배됨이 따라 나온다.

```haskell
zero ▷ p    = zero
(m ⊕ n) ▷ p = (m ▷ p) ⊕ (n ▷ p)
```

`m`이 모호하지 않은 파서면 `m ▷ p`도 그렇다.

### 5.7 반복

파서 하나를 반복해 파싱된 값들의 리스트를 얻을 수 있다.

```haskell
iterate   :: M a → M [a]
iterate m =   (m ⋆ λa. iterate m ⋆ λx. unit (a : x))
            ⊕ unit []
```

파서 `m`이 주어지면, 파서 `iterate m`은 파서 `m`을 0회 이상 연달아 적용하고 파싱된 모든 값의 리스트를 반환한다. 대안적 파스들의 리스트에서 가장 긴 파스가 먼저 온다.

예를 하나 보자.

```haskell
iterate digit "23 and more" = [([2, 3], " and more"),
                               ([2], "3 and more"),
                               ([], "23 and more")]
```

수(number)를 파싱하는 한 가지 방법은 다음과 같다.

```haskell
number :: M Int
number = digit ⋆ λa. iterate digit ⋆ λx. unit (asNumber (a : x))
```

여기서 `asNumber`는 한 자리 이상의 숫자 리스트를 받아 해당하는 수를 반환한다. 예를 하나 보자.

```haskell
number "23 and more" = [(23, " and more"),
                        (2, "3 and more")]
```

두 가지 파스가 나온다. 하나는 두 자리를 모두 파싱한 것이고, 다른 하나는 한 자리만 파싱한 것이다. 수는 최소 한 자리 숫자를 포함하도록 정의됐으므로 0자리 파스는 없다.

이 마지막 예가 보여 주듯, 반복자가 가능한 가장 긴 파스만 내놓도록 설계하는 편이 더 자연스러운 경우가 많다. 다음 절에서 그 방법을 설명한다.

### 5.8 편향 선택

`m ⊕ n`으로 쓰는 선택은 `m`이 내놓는 모든 파스에 이어 `n`이 내놓는 모든 파스를 내놓는다. 어떤 목적에는 둘 중 하나만 고르는 편이 더 합리적이다. `m`의 파스가 하나라도 있으면 `m`의 모든 파스를, 없으면 `n`의 모든 파스를 내놓는 것이다. 이를 편향 선택(biased choice)이라 부른다.

```haskell
(⊘)       :: M a → M a → M a
(m ⊘ n) x = if m x ≠ [] then m x else n x
```

`m ⊘ n`으로 쓰는 편향 선택은 `m`과 같은 파스들을 내놓되, `m`이 파스를 하나도 내놓지 못하면 `n`과 같은 파스들을 내놓는다.

편향 선택으로 다시 쓴 반복은 다음과 같다.

```haskell
reiterate   :: M a → M [a]
reiterate m =   (m ⋆ λa. reiterate m ⋆ λx. unit (a : x))
              ⊘ unit []
```

유일한 차이는 `⊕`를 `⊘`로 바꾼 것이다. 가능한 모든 파스를 가장 긴 것부터 리스트로 내놓는 대신, 가능한 가장 긴 파스 하나만 내놓는다.

앞의 예를 다시 보자.

```haskell
reiterate digit "23 and more" = [([2, 3], " and more")]
```

이후로 `number`는 `reiterate`로 다시 작성된 것으로 간주한다.

```haskell
number :: M Int
number = digit ⋆ λa. reiterate digit ⋆ λx. unit (asNumber (a : x))
```

모호한 파서로 가능성의 공간을 탐색할 수 있음을 살짝 드러내는 예를 하나 보자. `reiterate`를 이용해 문자열에서 항목을 하나 또는 둘씩 0회 이상 가져오는 모든 방법을 찾는다.

```haskell
reiterate oneOrTwoItems "many" = [(["m", "a", "n", "y"], ""),
                                  (["m", "a", "ny"], ""),
                                  (["m", "an", "y"], ""),
                                  (["ma", "n", "y"], ""),
                                  (["ma", "ny"], "")]
```

이는 (`oneOrTwoItems`의) 선택과 (`reiterate`의) 편향 선택을 결합한 것이다. 가능한 파스가 여럿 있지만, 각 파스에서 `oneOrTwoItems`는 입력이 모두 소비될 때까지 적용됐다. 다소 억지스러운 예이긴 해도, 비슷한 기법으로 1달러를 5센트, 10센트, 25센트 동전으로 바꾸는 모든 방법을 찾을 수 있다.

`m`과 `n`이 모호하지 않으면 `m ⊘ n`과 `reiterate m`도 모호하지 않다. 모호하지 않은 파서에 대해서는 순차 결합이 편향 선택에 대해 오른쪽으로 분배된다.

```haskell
(m ⋆ k) ⊘ (m ⋆ h) = m ⋆ λa. k a ⊘ h a
```

단 `m`이 모호하지 않을 때에 한한다. 선택과 달리, 순차 결합은 모호하지 않은 파서에 대해서도 편향 선택에 대해 왼쪽으로는 분배되지 않는다.

### 5.9 항 파서

이제 처음에 언급했던 항 파서를 작성할 수 있다. 다음은 완전히 괄호 친 항에 대한 문법을 BNF로 표현한 것이다.

```text
term ::= number | '(' term '÷' term ')'
```

이는 우리 표기법으로 다음과 같이 곧바로 번역된다. BNF와 달리 우리 표기법은 반환값을 정확히 어떻게 만들지 명시한다는 점에 주목하라.

```haskell
term :: M Term
term = (number      ⋆ λa.
          unit (Con a))
       ⊕ (lit '('   ⋆ λ_.
          term      ⋆ λt.
          lit '÷'   ⋆ λ_.
          term      ⋆ λu.
          lit ')'   ⋆ λ_.
          unit (Div t u))
```

(여기서 `λ_. e`는 `e`에 나타나지 않는 새 변수 `x`에 대한 `λx. e`와 같다. 람다식이 묶는 값에 관심이 없음을 나타낸다.) 이 파서의 사용 예는 앞에서 이미 나왔다.

위 파서는 선택으로 작성됐지만, 모호하지 않으므로 편향 선택으로 작성해도 무방하다. 다음 절의 모든 파서도 마찬가지다.

### 5.10 좌재귀

위 파서는 완전히 괄호 친 항에만 동작한다. 괄호 없는 항을 허용한다면 연산자 `÷`는 왼쪽으로 결합해야 한다. 이런 문법을 BNF로 표현하는 통상적인 방법은 다음과 같다.

```text
term   ::= term '÷' factor | factor
factor ::= number | '(' term ')'
```

이를 우리 표기법으로 번역하면 다음과 같다.

```haskell
term :: M Term
term = (term        ⋆ λt.
          lit '÷'   ⋆ λ_.
          factor    ⋆ λu.
          unit (Div t u))
       ⊕ factor

factor :: M Term
factor = (number    ⋆ λa.
            unit (Con a))
         ⊕ (lit '(' ⋆ λ_.
            term    ⋆ λt.
            lit ')' ⋆ λ_.
            unit t)
```

`factor`에는 문제가 없지만 `term`은 적용하는 순간 무한 루프에 빠진다. 문제는 `term`의 첫 단계가 `term`을 적용하는 것이어서 무한 퇴행으로 이어진다는 점이다. 이를 좌재귀(left recursion) 문제라고 한다. 함수형이든 아니든 모든 재귀 하강 파서가 겪는 어려움이다.

해법은 `term`의 문법을 다음과 같은 동등한 형태로 다시 쓰는 것이다.

```text
term  ::= factor term′
term′ ::= '÷' factor term′ | unit
```

여기서 `unit`은 늘 그렇듯 빈 파서를 뜻한다. 이는 우리 표기법으로 곧바로 번역된다.

```haskell
term :: M Term
term = factor ⋆ λt. term′ t

term′   :: Term → M Term
term′ t = (lit '÷'   ⋆ λ_.
           factor    ⋆ λu.
           term′ (Div t u))
          ⊕ unit t
```

여기서 `term′`는 항의 나머지를 파싱한다. 지금까지 파싱한 항에 해당하는 인자를 받는다.

이는 원하는 효과를 낸다.

```haskell
term "1972÷2÷23"   = [(Div (Div (Con 1972) (Con 2)) (Con 23), "")]
term "1972÷(2÷23)" = [(Div (Con 1972) (Div (Con 2) (Con 23)), "")]
```

일반적으로 좌재귀 정의

```haskell
m = (m ⋆ k) ⊕ n
```

는 다음과 같이 다시 쓸 수 있다.

```haskell
m = n ⋆ (closure k)
```

여기서

```haskell
closure     :: (a → M a) → (a → M a)
closure k a = (k a ⋆ closure k) ⊕ unit a
```

이고, `m :: M a`, `n :: M a`, `k :: a → M a`다.

### 5.11 지연 평가 개선하기

프로그램은 흔히 입력인 문자 리스트에서 출력인 또 다른 문자 리스트로 가는 함수로 표현된다. 지연 평가에서는 보통 입력의 일부만 읽어도 출력 리스트의 첫 부분이 생산된다. 이 ‘온라인’ 동작은 어떤 용도에서는 필수다.

일반적으로 파서에 그런 동작을 기대하는 것은 무리다. 입력이 전부 읽히기 전까지는 입력이 성공적으로 파싱될지 알 수 없는 것이 보통이기 때문이다. 하지만 특정한 특수 경우에는 더 잘할 수 있으리라 기대할 만하다.

`m`의 인스턴스로 시작하는 문자열에 `reiterate m`을 적용하는 경우를 생각해 보자. 이 경우 파스는 실패할 수 없다. 문자열의 나머지가 무엇이든, 내놓는 파스는 그 파싱된 값으로 시작하는 리스트일 것이라 기대할 수 있다. 지연 평가라면 나머지 입력을 읽기 전에 첫 숫자에 해당하는 출력을 만들어 낼 수 있으리라 기대할 만하다.

하지만 실제로는 그렇게 되지 않는다. 파서는 출력을 하나라도 생성하기 전에 입력 전체를 읽는다. 필요한 것은 파서 `reiterate m`이 언제나 성공한다는 사실을 인코딩할 방법이다. (입력의 시작이 `m`과 일치하지 않더라도 값으로 빈 리스트를 내놓기 때문이다.) 이를 제공하는 것이 함수 `guarantee`다.

```haskell
guarantee     :: M a → M a
guarantee m x = let u = m x in (fst (head u), snd (head u)) : tail u
```

여기서 `fst (a, b) = a`, `snd (a, b) = b`, `head (a : x) = a`, `tail (a : x) = x`다.

`guarantee`를 넣은 `reiterate`는 다음과 같다.

```haskell
reiterate   :: M a → M [a]
reiterate m = guarantee (  (m ⋆ λa. reiterate m ⋆ λx. unit (a : x))
                         ⊘ unit [])
```

이렇게 하면 `reiterate m`과 그 모든 재귀 호출이 답이 최소 하나 있는 리스트를 반환함이 보장된다. 그 결과 지연 평가 아래의 동작이 크게 개선된다.

위 설명은 매우 조작적(operational)인데, 표시적 의미론이 유용한 대안적 관점을 제공한다는 점도 짚어 둘 만하다. 종료하지 않는 프로그램을 `⊥`로 나타내자. 옛 정의에서는 다음이 성립함을 확인할 수 있다.

```haskell
reiterate digit ('1' : ⊥) = ⊥
```

반면 새 정의에서는 다음이 성립한다.

```haskell
reiterate digit ('1' : ⊥) = (('1' : ⊥), ⊥) : ⊥
```

즉 입력이 문자 `'1'`로 시작하지만 나머지 입력은 알 수 없을 때, 옛 정의에서는 출력에 대해 아무것도 알 수 없지만, 새 정의에서는 출력이 최소 하나의 파스를 내놓으며 그 값이 문자 `'1'`로 시작하는 리스트라는 것을 알 수 있다.

다른 파서들도 `guarantee`를 신중하게 사용하면 혜택을 볼 수 있으며, 특히 `iterate`도 `reiterate`처럼 수정할 수 있다.

### 5.12 결론

모나드가 재귀 하강 파서를 구조화하는 유용한 틀을 제공함을 살펴봤다. 빈 파서와 순차 결합은 `unit`과 `⋆`에 곧바로 대응하고, 모나드 법칙은 순차 결합이 결합적이며 빈 파서를 단위원으로 가짐을 반영한다. 실패하는 파서와 선택은 `zero`와 `⊕`에 대응하고, 이들이 만족하는 법칙은 선택이 결합적이며 실패하는 파서를 단위원으로 가진다는 것, 그리고 순차 결합이 선택에 대해 분배된다는 것을 반영한다.

순차와 선택은 파서뿐 아니라 컴퓨팅의 상당 부분에서 근본적이다. 모나드가 순차를 포착한다면, 순차와 선택을 함께 포착하는 것은 무엇인가라고 묻는 것이 자연스럽다. `unit`, `⋆`, `zero`, `⊕`에 위 법칙들을 더한 것이 그런 구조를 제공하는지도 모른다. 더 많은 실험이 필요하다. 한 가지 희망적인 조짐은, 파서 모나드를 살짝 변형하면 Dijkstra의 가드 명령 언어(guarded command language)의 그럴듯한 모델이 나온다는 것이다.
