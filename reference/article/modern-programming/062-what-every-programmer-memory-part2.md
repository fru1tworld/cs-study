## 3장 CPU 캐시

25년 전만 해도 CPU 코어 클럭과 메모리 버스 클럭은 비슷한 수준이었고 메모리 접근은 레지스터 접근보다 약간 느린 정도였다. 90년대 초 CPU 클럭이 급격히 올라가면서 이 균형이 깨졌다. RAM을 CPU만큼 빠르게 만들 수는 있지만 경제적이지 않다 — 그런 RAM은 지금 쓰는 DRAM보다 몇 자릿수 비싸다. 그래서 "작고 아주 빠른 RAM"과 "크고 상대적으로 빠른 RAM" 중 하나를 골라야 한다면, 작업 집합(working set)이 작은 RAM의 용량을 넘어서는 순간부터는 큰 쪽이 항상 이긴다. 보조 저장 장치(디스크)로의 스와핑 비용이 DRAM 접근보다 몇 자릿수 더 크기 때문이다.

다행히 이것은 양자택일 문제가 아니다. 대량의 DRAM에 소량의 고속 SRAM을 더할 수 있다. 이론적으로는 프로세서 주소 공간의 일부를 SRAM 전용으로 떼어 놓고 OS가 SRAM에 어떤 데이터를 둘지 관리하게 만드는 방법도 가능하지만(사실상 레지스터 집합의 확장), 실제로는 채택되지 않았다. 프로세스마다 이 메모리 영역의 가상 주소 매핑, 소프트웨어 차원의 할당 관리, 모듈 간 동기화 비용까지 얹히면서 SRAM이 주는 이득을 관리 오버헤드가 다 잡아먹기 때문이다.

그래서 SRAM은 OS나 사용자 프로그램이 직접 통제하는 자원이 아니라, 프로세서가 투명하게(transparently) 관리하는 **캐시**가 되었다. 캐시는 곧 사용될 가능성이 높은 메인 메모리 데이터의 임시 사본을 담아둔다. 이것이 가능한 이유는 프로그램 코드와 데이터에 **시간적 지역성**(temporal locality)과 **공간적 지역성**(spatial locality)이 있기 때문이다 — 짧은 시간 동안 같은 코드/데이터가 반복해서 쓰이거나(시간적), 가까운 위치의 메모리가 함께 쓰이는 경향(공간적)이 있다. 코드는 대개 루프 구조 때문에 공간적 지역성이 강하고, 데이터도 작업 집합이 어느 정도 제한된다면 지역성을 활용할 수 있다.

간단한 계산으로 캐시의 효과를 가늠해 보자. 메인 메모리 접근에 200사이클, 캐시 접근에 15사이클이 걸린다고 하면, 100개의 데이터 요소를 각각 100번씩 쓰는 코드는 캐시가 없으면 200만 사이클, 모든 데이터가 캐시된다면 168,500사이클이 걸린다 — 91.5% 개선이다.

다만 캐시(SRAM) 크기는 메인 메모리(DRAM)보다 훨씬 작다. 저자의 경험상 대략 메인 메모리의 1/1000 수준이었다(2007년 기준 4MB 캐시, 4GB 메인 메모리). 작업 집합이 캐시보다 작으면 문제없지만, 컴퓨터가 큰 메인 메모리를 갖는 이유가 있듯 작업 집합은 결국 캐시보다 커지기 마련이다. 특히 여러 프로세스가 동시에 돌아가는 시스템에서는 작업 집합 크기가 개별 프로세스와 커널의 작업 집합을 모두 합친 크기가 된다. 그래서 캐시에 무엇을 남기고 무엇을 밀어낼지 결정하는 좋은 전략, 그리고 필요하기 전에 미리 데이터를 가져다 놓는 **프리페칭(prefetching)** 기법이 중요해진다.

### 3.1 큰 그림에서 본 CPU 캐시

가장 단순한 캐시 구성에서는 CPU 코어가 메인 메모리에 직접 연결되지 않고 모든 적재·저장이 캐시를 거친다. CPU 코어와 캐시 사이는 전용 고속 연결이고, 캐시와 메인 메모리는 시스템 버스(FSB)로 연결된다.

폰노이만 구조를 오래 써왔음에도, 코드용 캐시와 데이터용 캐시를 분리하는 편이 유리하다는 것이 경험적으로 확인되었다. 인텔은 1993년부터 L1을 데이터(L1d)와 명령어(L1i)로 분리해 왔다. 코드와 데이터가 필요로 하는 메모리 영역이 서로 거의 독립적이기 때문이고, 최근에는 디코딩된 명령어를 캐싱해 파이프라인 스톨을 줄이는 이점도 생겼다.

캐시와 메인 메모리의 속도 차이가 벌어지면서 L2, 심지어 L3까지 등장했다 — L1보다 크고 느린 계층을 추가하는 편이 L1 자체를 무작정 키우는 것보다 경제적이기 때문이다. 코어 수가 늘어나면서 캐시 계층은 더 늘어날 수 있다. 표기법은 L1d(레벨1 데이터), L1i(레벨1 명령어) 등이다. 실제로는 데이터 흐름이 반드시 상위 캐시 전부를 거쳐가는 것은 아니며, 이 구성은 CPU 설계자의 재량이고 프로그래머에게는 보이지 않는다.

**코어 vs 스레드**: 별도의 코어는 (거의) 모든 하드웨어 자원을 독립적으로 갖고 서로 완전히 독립적으로 실행될 수 있다(공유 자원을 동시에 쓸 때만 제약). 반면 하이퍼스레드는 프로세서 자원 대부분을 공유하고 레지스터 정도만 개별로 갖는다(그나마도 일부는 공유). 결과적으로 "멀티 프로세서 × 멀티 코어 × 멀티 스레드"의 완전한 그림에서는, 하이퍼스레드들이 L1을 공유하고, 같은 프로세서 내의 코어들이 상위 캐시를 공유하며, 서로 다른 프로세서는 어떤 캐시도 공유하지 않는다. 이는 멀티프로세스·멀티스레드 애플리케이션의 캐시 효과를 논할 때 중요해진다.

### 3.2 상위 수준에서 본 캐시 동작

기본적으로 CPU 코어가 읽거나 쓰는 모든 데이터는 캐시에 저장된다(캐시 불가능한 메모리 영역도 있지만 이는 OS 구현자의 관심사고 프로그래머에게는 보이지 않는다). CPU가 데이터 워드를 필요로 하면 먼저 캐시를 검색한다. 캐시는 메인 메모리 전체를 담을 수 없으므로, 모든 메모리 주소가 캐시 가능하다는 전제 아래 각 캐시 항목은 메인 메모리 주소로 **태그**된다.

태그에 별도 공간이 필요하므로 캐시의 저장 단위를 워드로 잡는 것은 비효율적이다(32비트 x86에서 태그 자체가 32비트 이상 필요할 수 있음). 게다가 공간적 지역성 원리상 이웃한 메모리를 함께 적재하는 편이 유리하고, RAM 모듈도 새 CAS/RAS 신호 없이 여러 워드를 연달아 옮길 때 더 효율적이다(2장 참고). 그래서 캐시는 단일 워드가 아니라 여러 워드로 이뤄진 **캐시 라인**(옛날엔 32바이트, 지금은 보통 64바이트) 단위로 저장한다. 버스 폭이 64비트라면 캐시 라인 하나에 8번의 전송이 필요하다.

메모리 주소는 오프셋(O), 캐시 세트(S), 태그(T) 세 부분으로 나뉜다:

```
| Tag (T) | Cache Set (S) | Offset (O) |
```

캐시 라인 크기가 2^O면 하위 O비트가 라인 내 오프셋, 다음 S비트가 캐시 세트를 고르며(세트가 왜 필요한지는 3.3.1절에서), 나머지 상위 T비트가 태그로 쓰인다. 세트를 고르는 S비트는 저장할 필요가 없다(같은 세트 안의 모든 라인에 공통이므로).

명령어가 메모리를 수정할 때도 프로세서는 항상 전체 캐시 라인을 먼저 적재해야 한다(단일 명령어가 라인 전체를 한 번에 수정하는 일은 없기 때문 — 예외는 6.1절의 write-combining). 아직 메인 메모리에 반영되지 않은, 쓰기가 일어난 라인은 "**더티(dirty)**" 상태다. 새 데이터를 적재하려면 거의 항상 캐시에 자리를 먼저 비워야 한다. L1d에서의 축출은 L2로 내려가고(같은 라인 크기 사용), 이는 다시 L3, 최종적으로 메인 메모리로 이어질 수 있으며 단계가 내려갈수록 비용이 커진다. 이 방식은 AMD·VIA가 선호하는 **배타적(exclusive)** 캐시 모델이다. 반면 인텔은 **포함적(inclusive)** 캐시를 쓴다 — L1d의 모든 라인이 L2에도 존재해서 L1d 축출이 훨씬 빠르다(같은 내용을 두 곳에 두는 낭비는 L2가 충분히 크면 감내할 만하다). 배타적 캐시의 장점은 새 라인 적재 시 L1d만 건드리면 되고 L2를 건드리지 않아도 된다는 것.

**캐시 일관성(coherency)**: SMP 시스템에서는 모든 프로세서가 항상 동일한 메모리 상태를 봐야 한다. 어느 한 프로세서가 자기 캐시만 보고 메인 메모리에서 읽었다면 다른 프로세서의 더티 캐시 라인 내용을 놓칠 것이다. 그래서 프로세서는 다른 프로세서가 특정 캐시 라인을 읽거나 쓰려는 시도를 감지(**스누핑**)한다. 쓰기 접근이 감지되고 자신이 그 라인의 깨끗한(clean) 사본을 갖고 있었다면 그 라인은 무효화(invalid)된다(읽기 접근은 무효화를 유발하지 않음 — 깨끗한 사본은 여러 곳에 있어도 무방). 더 정교한 구현에서는, 어떤 프로세서가 더티 라인을 갖고 있는데 다른 프로세서가 그 라인을 읽거나 쓰려 하면 메인 메모리가 이미 낡은 상태이므로 메인 메모리를 거치지 않고 스누핑을 통해 직접 라인 내용을 건네준다(구현에 따라 메모리 컨트롤러가 이 전송을 감지해 메인 메모리도 갱신). 이때 쓰기라면 원래 프로세서는 자기 사본을 무효화한다.

이 모든 것이 요약되는 두 규칙:
- 더티 캐시 라인은 다른 어떤 프로세서의 캐시에도 존재하지 않는다.
- 같은 캐시 라인의 깨끗한 사본은 임의로 많은 캐시에 동시에 존재할 수 있다.

**캐시 히트/미스 비용**: 인텔이 Pentium M 기준으로 제시한 수치는 대략 레지스터 ≤1사이클, L1d ~3사이클, L2 ~14사이클, 메인 메모리 ~240사이클이다. L2(온다이) 접근 시간의 상당 부분은 배선 지연(wire delay)에서 온다는 점이 흥미로운데, 이는 공정 미세화(예: Merom 60nm → Penryn 45nm)로만 개선 가능한 물리적 한계다. 다행히 파이프라인 덕분에 이 비용 전체를 매번 지불하지 않아도 된다 — 메모리 로드가 파이프라인 초반에 시작될 수 있으면 다른 작업과 겹쳐 비용을 숨길 수 있다(L1d는 대개, 파이프라인이 긴 프로세서라면 L2도 가능). 쓰기 연산도 섀도 레지스터를 활용해 실제 메모리 반영을 늦추는 트릭이 가능하다.

원문은 임의 접근(랜덤 워크) 벤치마크로 작업 집합 크기에 따른 사이클/원소 그래프(그림 3.4)를 보여주는데, L1d·L2 두 캐시만 있는(L3 없는) 프로세서에서 뚜렷한 세 단계 평탄 구간이 나타난다 — L1d(2^13바이트 추정) 안에서는 10사이클 미만, L2(2^20바이트 추정) 초과 전까지는 약 28사이클, L2를 넘어서면 480사이클 이상으로 급증한다(쓰기가 섞이면 더티 라인 반영 비용까지 더해짐). 이 격차는 몇 퍼센트가 아니라 자릿수 단위이므로, 6장에서 다룰 캐시 활용 코딩 기법을 배워야 할 충분한 이유가 된다.

### 3.3 CPU 캐시 구현 상세

캐시 구현자의 근본 문제는, 메인 메모리 전체가 잠재적으로 캐시 대상이지만 캐시는 그보다 훨씬 작다(1:1000 비율도 드물지 않음)는 것이다.

#### 3.3.1 연관성(Associativity)

**완전 연관(fully associative)** 캐시는 어떤 캐시 라인이든 어떤 메모리 위치든 담을 수 있는 구조다(그림 3.5). 이를 위해선 매 접근마다 모든 캐시 라인의 태그와 요청 주소의 태그를 비교해야 한다. 4MB/64B 캐시라면 항목이 65,536개나 되어, 몇 사이클 안에 이 많은 항목 중 하나를 골라내는 로직은 현실적으로 구현 불가능하다. 그래서 완전 연관 캐시는 TLB처럼 아주 작은 캐시(수십 개 항목)에서만 실용적이다.

**직접 매핑(direct-mapped)** 캐시는 반대 극단이다. 각 태그가 정확히 하나의 캐시 항목에만 매핑된다. 4MB/64B 캐시에서 주소의 6~21비트(16비트)로 65,536개 항목을 직접 인덱싱할 수 있다(그림 3.6). 이 구조는 비교기(comparator)가 하나뿐이라 빠르고 상대적으로 구현이 쉽다(멀티플렉서 트랜지스터 수는 캐시 라인 수 N에 대해 O(log N)로 완만하게 증가). 하지만 프로그램이 사용하는 주소가 direct-mapping에 쓰이는 비트에 대해 고르게 분포하지 않으면(실제로 흔한 일이다), 일부 캐시 항목은 계속 축출되는 반면 다른 항목은 거의 비어 있는 문제가 생긴다.

**세트 연관(set-associative)** 캐시(그림 3.7)는 두 방식의 장점을 결합한다. 태그·데이터 저장 공간을 세트로 나누고 주소로 세트를 고르는 것은 direct-mapped와 비슷하지만, 각 세트 값마다 하나가 아니라 소수의 값(way)을 캐시해 두고 세트 내 모든 항목의 태그를 병렬로 비교한다(완전 연관과 비슷). 그 결과 특정 세트에 몰리는 나쁜(혹은 의도적인) 주소 선택에도 잘 버티면서, 구현 가능한 비교기 수로 캐시 크기를 제한받지 않는다. 캐시가 커지면 늘어나는 건 열(column) 수뿐이고, 행(row, 즉 비교기) 수는 연관도(associativity)가 늘 때만 증가한다. 오늘날 프로세서는 L2에서 최대 24-way, L1에서는 보통 8-way 정도의 연관도를 쓴다.

4MB/64B 캐시에 8-way 연관을 적용하면 세트가 8,192개이고 태그는 13비트만 쓰면 된다. 세트 안 8개 태그를 비교하는 것은 충분히 빠르게 할 수 있다. 다음 관계가 성립한다:

```
캐시 크기 = 캐시 라인 크기 × 연관도 × 세트 개수
O = log2(캐시 라인 크기)
S = log2(세트 개수)
```

gcc 실행을 대상으로 캐시 크기·라인 크기·연관도를 바꿔가며 L2 캐시 미스를 측정한 표(원문 표 3.1)를 보면, 연관도를 늘리는 것이 미스를 상당히 줄여준다는 것을 확인할 수 있다. 예컨대 8MB 캐시에서 direct-mapped를 2-way로만 바꿔도 캐시 미스가 거의 44% 줄어든다. 다만 "연관도를 올리는 것이 캐시 크기를 두 배로 늘리는 것과 같은 효과"라는 통설은 극단적인 경우(4MB→8MB로 넘어가는 지점 등)에만 맞고, 일반적으로 연관도를 계속 두 배로 늘려도 이득은 점점 줄어든다.

단일 스레드 워크로드에서는 연관도를 8 이상으로 올려도 효과가 크지 않은 편이지만, 하이퍼스레딩(L1 공유)이나 멀티코어(L2 공유)가 도입되면 사실상 두 개(쿼드코어면 네 개) 프로그램이 같은 캐시를 두고 경쟁하는 셈이어서 실질 연관도가 반토막(또는 1/4)난다. 그래서 코어 수가 늘수록 공유 캐시의 연관도도 함께 늘어야 하고, 그것도 한계(16-way 정도부터 어려움)에 부딪히면 공유 L3 캐시 같은 더 상위 계층으로 문제를 옮기게 된다.

캐시 크기 증가의 효과는 작업 집합 크기와 떼어놓고 해석할 수 없다. 원문 실험에서 프로그램의 최대 작업 집합은 5.6MB였는데, 이 경우 16MB 캐시도 완전한 direct-mapping이라면 충돌이 생길 수 있지만(2-way 16MB가 direct-mapped 16MB보다 나은 이유), 32MB 캐시로 더 키워봐야 이 워크로드에서는 이득이 미미할 것으로 추정된다. 다만 작업 집합 자체가 시간이 지나며 커지는 경향이 있으므로, 컴퓨터를 구매할 때 캐시 크기를 정하려면 실제 워크로드의 작업 집합 크기를 측정해 보는 것이 좋다.

#### 3.3.2 캐시 효과 측정

측정 프로그램은 임의 크기의 작업 집합을 시뮬레이션하며 읽기/쓰기, 순차/랜덤 접근을 조합해 테스트한다. 구조는 다음과 같다(개념 재현):

```c
struct l {
    struct l *n;      // 다음 원소를 가리키는 포인터
    long int pad[NPAD]; // 페이로드 크기 조절용
};
```

원소들은 원형 단일 연결 리스트로 묶이고, `n` 포인터를 순차적으로(메모리 배치 순서대로) 혹은 무작위 순서로 따라간다. NPAD 값으로 원소 하나의 크기를 조절해 캐시 라인 채움 정도를 실험할 수 있다.

**순차 접근(단일 스레드)**: NPAD=0(8바이트 원소, 64비트 기준)일 때 사이클/원소 그래프(그림 3.10)는 세 구간을 보인다 — 2^14바이트까지, 2^15~2^20바이트, 2^21바이트 이상. 이는 각각 L1d(16KB)·L2(1MB) 크기와 맞아떨어진다(경계가 날카롭지 않은 건 시스템의 다른 부분도 같은 캐시를 쓰기 때문). 흥미로운 점은 L2 접근이 이론상 14사이클 이상 걸려야 할 것 같은데 실측은 9사이클 수준에 그친다는 것 — 프로세서가 순차 접근을 예상하고 다음 캐시 라인을 미리 가져오는 **하드웨어 프리페칭** 덕분이다. 메인 메모리 접근이 200+사이클인데도 프리페칭 덕에 9사이클 수준을 유지할 수 있다는 것이 프리페칭 효과의 크기를 잘 보여준다.

NPAD을 늘려 원소 크기(0, 56, 120, 248바이트)를 다르게 한 실험(그림 3.11)은 L1d 구간에서는 원소 크기와 무관하게 거의 평탄하지만, L2 구간에서는 NPAD=0을 제외한 모든 크기가 약 28사이클(L2 접근 시간)로 올라간다 — NPAD=7만 되어도 매 반복마다 새 캐시 라인이 필요해져서 L2→L1d 프리페칭이 사실상 무력화되기 때문이다. 작업 집합이 L2보다 커지는 구간에서는 원소 크기별 편차가 훨씬 커지는데, 이는 하드웨어 프리페처가 **페이지 경계를 넘어가지 못한다**는 제약 때문이다(다음 페이지가 존재하는지, 상주하는지 모르는 채로 프로세서 스스로 페이지 폴트를 유발할 수는 없으므로).

TLB 영향을 보려면 원소를 촘촘히 배치한 경우와, 각 원소를 별도 페이지에 하나씩 두는 경우를 비교한다(그림 3.12). 원소별 별도 페이지 배치에서는 작업 집합이 2^13바이트를 넘는 순간 거대한 스파이크가 생기는데, 이는 TLB 캐시가 흘러넘치는(overflow) 지점이다(원소 크기 64바이트 가정 시 TLB 항목 64개 계산과 일치). 가상→물리 주소 변환에 드는 사이클은 매우 크고, 이 비용은 L2·메인 메모리 접근 비용에 **추가로** 더해진다 — 그래서 NPAD=31의 원소당 총비용이 이론상의 RAM 접근 시간보다 높게 나오는 이유 중 하나가 TLB 미스다.

리스트 원소를 수정하는 실험(그림 3.13, `Inc`는 pad[0] 증가, `Addnext0`은 다음 원소의 pad[0]을 더함)에서는 직관과 달리 "일이 더 많아 보이는" Addnext0이 일부 구간에서 오히려 Inc보다 빠르다. 다음 원소를 미리 읽는 동작이 사실상 강제 프리페치가 되어, 다음 리스트 원소가 이미 L1d에 있게 되기 때문이다. 다만 메모리를 수정하는 두 테스트는 L2 축출 시 더티 라인을 그냥 버릴 수 없고 메모리에 기록해야 하므로 FSB 대역폭을 반씩 나눠 쓰는 셈이 되어, Addnext0이 L2를 벗어나는 시점부터는 Follow의 두 배(28사이클 vs 14사이클)로 벌어진다.

캐시 크기 자체의 영향도 확인할 수 있다(그림 3.14): L1d 32KB/L2 1MB, L1d 16KB/L2 512KB/L3 2MB, Core2의 L1d 32KB/L2 4MB 세 시스템을 비교하면, 마지막 레벨 캐시가 클수록 곡선이 낮은 구간(L2 접근 비용 수준)에 더 오래 머물고, 작업 집합이 220바이트일 때 두 번째 프로세서가 첫 번째보다 두 배 빠르다. 워크로드가 마지막 레벨 캐시 크기에 맞도록 조절될 수 있다면, 더 큰 캐시를 가진 프로세서에 웃돈을 지불할 가치가 충분하다.

**랜덤 접근(단일 스레드)**: 접근 패턴이 예측 불가능하면 순차 접근에서 보던 프리페칭 효과가 사라진다. 랜덤 접근 곡선(그림 3.15)은 순차 접근과 달리 평탄 구간(plateau) 없이 계속 상승하며, 실측 사이클이 이론상 메인 메모리 접근 시간(200~300사이클)보다도 높은 450사이클 이상까지 올라간다 — 프리페칭이 오히려 낭비로 작용하는 셈이다. L2 접근/미스 비율(그림 3.16, 표 3.2)을 보면 작업 집합이 L2보다 커질수록 미스율이 거의 100%에 근접하도록 계속 오른다. 랜덤 접근에서 작업 집합이 두 배로 늘 때마다 원소당 접근 시간도 두 배 이상으로 뛰는데, 이는 TLB 미스율 상승이 원인이다. 리스트를 통째로 무작위화하는 대신 일정 개수 페이지 단위 블록으로 나눠 무작위화하면(그림 3.17) TLB에 필요한 항목 수가 줄어 성능이 개선되며, 큰 블록으로 갈수록 "전체 무작위화" 곡선에 점근한다.

#### 3.3.3 쓰기 동작

캐시 일관성은 코드 수준에서는 완전히 투명해야 한다(커널 코드는 예외적으로 캐시 플러시가 필요할 때가 있다). 쓰기 정책은 두 가지다:

- **write-through**: 캐시 라인에 쓰면 즉시 메인 메모리에도 반영한다. 항상 메모리와 캐시가 일치하므로 라인을 교체할 때 그냥 버리면 되지만, 로컬 변수를 반복해서 수정하는 코드조차 FSB 트래픽을 계속 만들어내서 느리다.
- **write-back**: 캐시 라인을 즉시 반영하지 않고 더티로만 표시해 두었다가, 나중에 캐시에서 축출될 때 실제로 메모리에 기록한다. FSB의 여유 대역폭을 이용해 미리 기록해 둘 수도 있어 대체로 write-through보다 성능이 좋고, 준수한 프로세서 대부분이 이 방식을 쓴다.

문제는 둘 이상의 프로세서(혹은 코어·하이퍼스레드)가 같은 메모리에 접근할 때 발생한다. 한 프로세서에 더티인 라인을 다른 프로세서가 읽으려 하면, 단순히 메인 메모리에서 읽어올 수 없다. 이때 필요한 프로토콜이 **MESI**(Modified, Exclusive, Shared, Invalid)다:

- **Modified**: 로컬 프로세서가 수정했고, 어떤 캐시에도 이 라인의 유일한 사본이다.
- **Exclusive**: 수정되지 않았고, 다른 어떤 프로세서의 캐시에도 없다고 알려져 있다.
- **Shared**: 수정되지 않았고, 다른 프로세서 캐시에도 있을 수 있다.
- **Invalid**: 무효(사용되지 않음).

상태 전이는 주로 스누핑으로 이뤄진다. Modified 상태 라인을 다른 프로세서가 읽으려 하면 첫 프로세서가 내용을 건네고 Shared로 전환한다(메모리 컨트롤러도 이를 받아 메인 메모리에 기록해야 Shared 표시가 성립). 다른 프로세서가 쓰려고 하면 첫 프로세서는 내용을 건네고 자신은 Invalid로 전환한다 — 이것이 악명 높은 **RFO**(Request For Ownership)다. Shared 라인에 로컬에서 쓰려면 다른 프로세서들에 RFO 메시지를 보내 그쪽 사본을 모두 Invalid로 만들어야 한다. Exclusive는 Shared와 거의 같지만, 로컬 쓰기를 버스에 알릴 필요가 없다는 점이 다르다 — 그래서 프로세서는 되도록 Shared보다 Exclusive를 유지하려 한다(E→M 전이가 S→M 전이보다 훨씬 빠르기 때문).

RFO 메시지가 필요한 대표적 상황은 (1) 스레드가 다른 프로세서로 마이그레이션될 때(캐시 라인 전체를 새 프로세서로 옮겨야 함), (2) 서로 다른 프로세서가 정말로 같은 캐시 라인이 필요할 때다. 멀티스레드·멀티프로세스 프로그램은 동기화를 위해 어느 정도 메모리 공유가 불가피하지만, 불필요한 RFO 트래픽은 최대한 줄여야 한다. 캐시 일관성 프로토콜 메시지는 시스템의 모든 프로세서에 전파되어야 하고, MESI 전이는 모든 프로세서가 응답할 기회를 가질 때까지 완료되지 않으므로, 가장 느린 응답 시간이 전체 프로토콜 속도를 좌우한다.

FSB 자체도 공유 자원이라는 점을 기억해야 한다. 한 프로세서가 버스를 포화시킬 수 있다면(흔한 일이다) 두세 개 프로세서가 같은 버스를 나눠 쓸 때 프로세서당 가용 대역폭은 더 줄어든다. AMD처럼 프로세서마다 로컬 메모리가 있는 구조에서도 멀티스레드/멀티프로세스 프로그램은 결국 어느 시점엔 같은 메모리 영역을 두고 동기화해야 하므로 이 제약에서 완전히 자유롭지 않다.

**멀티스레드 접근**: 순차 읽기 전용 테스트(그림 3.19, NPAD=15)에서 메모리 수정이 전혀 없고 RFO도 필요 없는데도, 2스레드에서 최대 18%, 4스레드에서 최대 34%의 성능 저하가 관측된다 — 캐시 라인 전송이 필요 없는데도 이 정도 손실이 나는 건 순전히 "프로세서→메모리 컨트롤러" 혹은 "메모리 컨트롤러→메모리 모듈" 버스 대역폭 병목 때문이다. 메모리를 수정하는 순차 증가 테스트(그림 3.20)에서는 2스레드 약 18%, 4스레드 무려 93% 페널티가 나온다 — 프리페치 트래픽과 write-back 트래픽이 합쳐져 버스를 거의 포화시키기 때문이다. 랜덤 Addnextlast 테스트(그림 3.21)에서는 원소 하나 처리에 최대 약 1,500사이클까지 치솟는다.

효율성 표(표 3.3, 재구성):

| 스레드 수 | 순차 읽기 | 순차 증가 | 랜덤 덧셈 |
|---|---|---|---|
| 2 | 1.69 | 1.69 | 1.54 |
| 4 | 2.98 | 2.07 | 1.65 |

2스레드는 이론상 한계인 2배에 어느 정도 근접하지만, 4스레드는 4배는커녕 2스레드 대비 추가 이득이 미미하다(속도-업 그래프인 그림 3.22 참고). 이 때문에 같은 메모리 컨트롤러를 공유하는 4소켓 초과 메인보드를 찾기 어렵고, 더 많은 프로세서가 필요하면 5장에서 다룰 NUMA 구조로 넘어가야 한다.

**하이퍼스레드 특수 사례**: 하이퍼스레드는 레지스터 집합만 빼고 거의 모든 처리 자원을 공유하므로 진짜 동시 실행이 아니다. 두 하이퍼스레드가 한 코어에서 돌 때 이득을 보려면 두 스레드의 합산 실행 시간이 단일 스레드 실행 시간보다 짧아야 한다. 단순 모델로 최소 요구 캐시 히트율을 계산할 수 있다:

```
Texe = N(1 − Fmem)·Tproc + Fmem·(Ghit·Tcache + (1 − Ghit)·Tmiss)
```

(N: 명령어 수, Fmem: 메모리 접근 비율, Ghit: 캐시 히트율, Tproc/Tcache/Tmiss: 각각 처리·캐시 히트·캐시 미스 사이클 수)

이 식을 풀면(그림 3.23), 단일 스레드 히트율이 낮을수록(예: 60%면 듀얼스레드는 10%만 넘어도 이득) 하이퍼스레딩 이득을 보기 쉽고, 단일 스레드 히트율이 이미 아주 높으면(95%) 듀얼스레드에서도 80% 이상을 유지해야 해서 훨씬 어렵다. 문제는 두 하이퍼스레드가 같은 L1d(및 L2 등)를 나눠 쓰므로 실질 캐시 크기가 반토막 난다는 점 — 작업 집합이 겹치지 않는 두 스레드라면 95% 히트율이 그대로 반토막 나 요구치인 80%에 못 미칠 수 있다. 그래서 하이퍼스레드는 제한된 상황에서만 유용하며, 워크로드에 따라 BIOS에서 꺼두는 편이 나을 수도 있다(단, SMT는 병렬 코드의 특정 버그를 잡아내는 디버깅 용도로는 의외로 유용하다는 각주도 있다).

#### 3.3.4 멀티프로세서 지원 (계속)

**가상 vs 물리 주소 태깅**: 캐시 태그에 가상 주소를 쓸지 물리 주소를 쓸지는 설계 선택이다. 가상 주소를 쓰면 MMU 변환을 기다리지 않고 파이프라인 이른 단계에서 캐시 조회를 시작할 수 있어 접근 비용을 더 많이 숨길 수 있다. 그래서 오늘날 L1(작고 필요시 통째로 비우는 비용이 크지 않음)은 대체로 가상 주소 태깅을 쓰고, L2 이상의 크고 느린 캐시는 물리 주소 태깅을 쓴다(변환이 끝날 시간적 여유가 있고, 큰 캐시를 자주 통째로 비우는 건 비용이 크기 때문). 프로그래머가 신경 쓸 부분은 거의 없지만, 가능하면 같은 물리 메모리를 한 프로세스 안에서 여러 가상 주소에 매핑하지 않는 것이 좋다.

**교체 전략**: 대부분의 캐시는 LRU(Least Recently Used)를 기본으로 쓰며, 이는 대체로 좋은 기본 전략이다. 연관도가 커질수록 LRU 리스트 유지 비용도 커져서 다른 전략이 채택될 수도 있다. 캐시가 물리 주소로 태깅되어 있으면 프로그래머는 가상 주소가 캐시 세트와 어떻게 연결되는지 알 방법이 없으므로 이 영역은 사실상 OS(그리고 가상화가 개입되면 하이퍼바이저)의 몫이다. 프로그래머 입장에서 할 수 있는 것은 논리 메모리 페이지를 있는 그대로 활용하는 것과, 가능한 한 큰 페이지 크기를 써서 물리 주소를 다양화하는 정도다(큰 페이지의 다른 이점은 4장에서).

### 3.4 명령어 캐시

명령어도 캐시된다. 다만 데이터 캐시보다 덜 골치 아프다. 실행되는 코드량은 문제의 복잡도에 좌우되고 대체로 고정적이며, 코드는 (사람이 직접 짜는 데이터 접근 패턴과 달리) 컴파일러가 좋은 코드 생성 규칙을 알고 생성하며, 프로그램 흐름은 데이터 접근 패턴보다 훨씬 예측하기 쉽다(분기 예측이 잘 통함). 코드는 대체로 공간·시간 지역성도 좋다.

CPU 코어 클럭이 급격히 빨라지면서 캐시(1레벨 포함)와의 속도 차가 벌어졌고, 그래서 CPU는 파이프라인화되었다(명령어 실행이 디코드→준비→실행 단계로 나뉨). 파이프라인은 매우 길 수 있다(인텔 넷버스트 아키텍처는 20단계 이상). 파이프라인이 스톨되면(다음 명령어 위치를 예측하지 못하거나 로드에 시간이 걸릴 때) 다시 속도를 올리기까지 시간이 걸리므로, CPU 설계자들은 분기 예측에 상당한 칩 면적과 노력을 들인다.

CISC 프로세서(특히 x86/x86-64)는 디코딩 단계 자체에 시간이 걸리므로, 최근 프로세서는 L1i에 원본 바이트 시퀀스 대신 **디코딩된 명령어**를 캐시한다 — 이를 "**트레이스 캐시**"라 부른다. 트레이스 캐시는 캐시 히트 시 파이프라인 앞단 단계를 건너뛸 수 있게 해주어 스톨 이후 특히 유리하다. L2 이상은 코드·데이터를 함께 담는 통합 캐시이므로 여기서는 바이트 시퀀스 형태로 캐시된다.

프로그래머가 할 수 있는 규칙은 많지 않지만 핵심은 두 가지다: (1) 가능한 한 작은 코드를 생성한다(소프트웨어 파이프라이닝 등을 위해 코드가 커지는 예외는 있다), (2) 코드 배치나 명시적 프리페치로 프로세서의 프리페칭 판단을 돕는다. 이는 대부분 컴파일러의 코드 생성으로 강제되고, 프로그래머가 할 수 있는 세부 사항은 6장에서 다룬다.

**자기 수정 코드(SMC)**: 메모리가 귀했던 시절엔 프로그램 자체를 실행 중에 바꿔 코드 크기를 줄이는 기법이 흔했다. SMC는 요즘도 성능 목적이나 보안 익스플로잇에서 가끔 보이지만 일반적으로 피해야 한다. 변경된 코드는 트레이스 캐시(디코딩된 명령어 캐시)에 남아있을 수 없고, 트레이스 캐시가 안 쓰이더라도 이미 파이프라인에 들어간 다음 명령어가 바뀌면 프로세서가 많은 작업을 버리고 다시 시작해야 한다(경우에 따라 프로세서 상태 대부분을 버려야 할 수도 있다). 또한 프로세서는(사실상 항상 참인 가정이므로) 코드 페이지가 불변이라고 전제하고 L1i에 MESI 대신 단순화된 SI 프로토콜을 쓰는데, 수정이 감지되면 비관적인 가정을 여럿 해야 한다. 리눅스에서는 툴체인이 기본적으로 코드 페이지를 쓰기 금지로 만들기 때문에, SMC가 있는 프로그램은 링크 시점에 별도 조치가 필요해 비교적 쉽게 식별할 수 있고, 최신 x86/x86-64 프로세서는 SMC 사용을 세는 전용 성능 카운터도 제공한다.

### 3.5 캐시 미스 요인

캐시 미스의 실제 비용과 완화 방법을 좀 더 구체적으로 살펴본다.

#### 3.5.1 캐시와 메모리 대역폭

SSE 명령어로 16바이트씩 적재/저장하며 작업 집합을 1KB부터 512MB까지 늘려가는 벤치마크로 대역폭(바이트/사이클)을 측정한 결과, 프로세서 세대·벤더에 따라 편차가 상당히 크다.

- **인텔 넷버스트(64비트)**: L1d 안에서는 읽기가 이론적 최대인 16바이트/사이클(movaps 하나로 16바이트)에 도달한다. L1d를 벗어나면 6바이트/사이클 미만으로 급락하고, DTLB가 소진되는 2^18바이트 지점에서 계단이 하나 더 생긴다. 반면 **쓰기 성능은 작업 집합 크기와 무관하게 4바이트/사이클을 넘지 못한다** — 이는 L1d가 Write-Through 모드라서 L2 속도에 제한받기 때문으로 추정된다. L2도 부족해지면 쓰기·복사 성능은 0.5바이트/사이클까지 떨어져 읽기 대비 10배 느려진다. 하이퍼스레드 두 개를 동시에 쓰면(자원을 절반씩 나눠 가지므로) 캐시·대역폭이 반토막 나서 최악의 활용 사례가 된다.
- **인텔 Core 2(듀얼코어, L2 공유, P4보다 4배 큰 L2)**: 읽기 성능은 작업 집합 전 구간에서 거의 이론적 최대(16바이트/사이클)에 근접하며 이는 L1d로의 프리페칭까지 효과적으로 이뤄진다는 뜻이다. Write-Through가 아니라 데이터를 L1d에 쓰고 필요할 때만 축출하는 방식이라 쓰기 성능도 최대치에 가깝다가, L2 부족 구간에서는 읽기·쓰기 격차가 최대 20배까지 벌어진다. 같은 프로세서의 두 코어(스레드 2개)가 같은 메모리를 두고 경쟁하면 L1d 안에 들어갈 크기라도 메인 메모리에서 읽는 것과 같은 성능으로 떨어지는데, 이는 RFO 메시지 때문이다. L1d를 넘어서면 오히려 공유 L2에서 만족되는 미스가 많아져 성능이 50% 개선되는 역설적인 구간도 나타난다(L1d에서 아직 플러시되지 않은 수정 항목에 대해서만 RFO가 필요해지므로).
- **AMD 패밀리 10h Opteron(L1d 64KB, L2 512KB, 공유 L3 2MB)**: L1d 안에서는 사이클당 2개 명령어 처리가 가능해 읽기 32바이트/사이클 이상, 쓰기도 18.7바이트/사이클로 높다. 다만 읽기 곡선은 빠르게 평탄해져 2.3바이트/사이클로 낮아진다(이 프로세서는 데이터 프리페치를 효율적으로 하지 않는 것으로 보임). 쓰기 곡선은 캐시 계층을 그대로 따라가며(L1d 피크 → L2 6바이트/사이클 → L3 2.8바이트/사이클 → 그 이상은 0.5바이트/사이클) 하락한다. 두 스레드가 공유 L3를 거쳐 데이터를 주고받는 경우 L3 접근보다도 훨씬 비싸지는데, 이는 L3를 통한 스레드 간 공유 메커니즘 자체가 비효율적임을 시사한다(Core 2가 공유 L2 속도로 처리하는 구간에서 Opteron은 훨씬 좁은 구간에서만 L3 속도에 도달).

#### 3.5.2 크리티컬 워드 로드

메모리는 캐시 라인 크기보다 작은 단위(64비트)로 전송된다. 64바이트/128바이트 캐시 라인이면 8~16번의 전송이 필요하다. DRAM 칩은 이를 버스트 모드로 전송할 수 있어 프로세서 프리페칭 목적에는 이 방식이 최선이다.

하지만 캐시 미스(최초 사용이라 발생하는 필수 미스든, 캐시 용량 부족으로 인한 용량 미스든)가 발생했을 때 프로그램에 실제로 필요한 워드가 캐시 라인의 첫 워드가 아닐 수 있다. 버스트 모드에 더블 데이터 레이트를 쓰더라도 64비트 블록들은 눈에 띄게 다른 시점에 도착한다(블록마다 4사이클 이상의 시차). 필요한 워드가 라인의 8번째라면, 첫 워드 도착 후에도 추가로 30사이클 이상을 더 기다려야 할 수 있다.

이를 완화하는 기법이 **Critical Word First & Early Restart**다. 메모리 컨트롤러는 캐시 라인 내 워드를 원하는 순서로 요청할 수 있는데, 프로세서가 정말 기다리는 "크리티컬 워드"를 먼저 요청하도록 지시하면, 그 워드가 도착하는 즉시(나머지 라인이 아직 도착 중이고 캐시가 아직 일관 상태가 아니더라도) 프로그램이 진행할 수 있다. 다만 프로세서가 미리 프리페치를 걸어둔 상태라면 크리티컬 워드가 무엇인지 알 수 없어 이 최적화가 적용되지 못하는 경우도 있다.

이 최적화가 적용되어도 크리티컬 워드가 캐시 라인의 어디에 있는지는 여전히 영향을 준다. 크리티컬 워드가 라인 끝에 있을 때가 시작에 있을 때보다(L2를 벗어난 구간에서) 약 0.7% 느리다는 실측 결과가 있다(순차 접근에서 좀 더 두드러짐 — 다음 라인 프리페칭 문제와 연결되는 것으로 보인다).

#### 3.5.3 캐시 배치

하이퍼스레드·코어·프로세서에서 캐시가 어떻게 배치되는지는 프로그래머가 직접 통제할 수 없지만, 스레드를 어느 CPU에서 실행할지는 프로그래머가 정할 수 있으므로 이 관계를 아는 것이 중요하다.

하이퍼스레드는 정의상 레지스터를 뺀 모든 것(L1 포함)을 공유한다. 코어별로는 최소 L1은 개별로 갖되, 초기 멀티코어는 공유 캐시가 전혀 없었고, 이후 인텔 듀얼코어는 공유 L2를 도입했으며(쿼드코어는 두 코어씩 짝지어 별도 L2를 쓰고 그 위 상위 캐시는 없는 경우도 있었다), AMD 패밀리 10h는 개별 L2에 공유 L3를 더했다.

공유 캐시가 없는 설계는 코어들의 작업 집합이 겹치지 않는(당시 흔했던 단일 스레드 프로그램 위주 환경) 경우에 유리하다. 완전 공유(L1 제외 전부)는 두 코어의 작업 집합이 겹칠 때 총 가용 캐시가 늘어나는 효과가 있고, 겹치지 않을 때는 "스마트 캐시" 관리로 한 코어가 캐시를 독점하지 못하게 막아준다 — 다만 실측해 보면 이 균형 조정(rebalancing)에 다소 문제가 있다.

실험: 한 프로세스가 SSE로 2MB(해당 Core 2의 L2 절반) 블록을 계속 읽거나 쓰며 한 코어에 고정되고, 다른 코어에는 가변 크기의 작업 집합을 읽거나 쓰는 두 번째 프로세스를 고정한다(그림 3.31). 두 L2가 완전히 분리돼 있다면 2^20~2^23바이트 구간에서 성능 저하가 나타나야 하는데, 실제로는 백그라운드 프로세스가 쓰기를 할 때 1MB에 못 미치는 지점부터 이미 성능이 나빠지기 시작한다. 두 프로세스는 메모리를 공유하지 않으므로 RFO 문제가 아니라 순수한 캐시 축출 문제다 — "스마트 캐시" 관리 탓에 코어당 실제 체감 캐시 용량이 가용한 2MB보다 1MB에 더 가깝게 느껴진다는 뜻이다.

이런 구조(공유 L2가 두 코어 쌍씩 나뉜 쿼드코어)는 사실 상위 캐시 계층이 추가되기 전의 임시방편이었다 — 두 코어가 같은 버스(바깥에서 보면 FSB)로 통신하므로 별도 소켓의 듀얼코어 대비 특별한 이점이 없다. 앞으로 멀티코어 캐시 설계는 더 많은 계층으로 향할 것으로 보이며(AMD 10h가 그 시작), 저수준 캐시를 코어 일부끼리만 공유하는 흐름이 이어질지는 지켜봐야 한다. 추가 계층이 필요한 이유는 고속·고빈도 캐시(L1, L2)를 너무 많은 코어가 나눠 쓰면 성능이 나빠지고, 그러려면 캐시 크기·연관도 모두 코어 수에 맞춰 커져야 하기 때문이다 — 상대적으로 느리지만 자주 쓰이지 않는 큰 L3와 적당한 크기의 L2를 조합하는 편이 합리적인 절충안이다.

이 모든 설계 차이는 프로그래머에게 스케줄링 결정의 복잡도로 다가온다. 최선의 성능을 내려면 워크로드와 머신 아키텍처의 세부사항을 알아야 하는데, 다행히 머신 아키텍처를 파악할 수 있는 인터페이스가 존재하며 이후 장에서 소개한다.

#### 3.5.4 FSB의 영향

FSB는 시스템 성능에서 중심적 역할을 한다. 캐시 콘텐츠는 결국 메모리로의 연결이 허용하는 속도만큼만 저장·적재될 수 있다. 메모리 모듈 속도만 다른 두 대의 Core 2 머신(667MHz vs 800MHz DDR2, 20% 차이)에서 Addnext0 테스트(NPAD=7, 64비트)를 돌려보면(그림 3.32), 작업 집합이 커서 FSB가 정말 부하를 받는 구간에서 최대 18.2%의 성능 향상이 나타난다 — 이론적 최대치에 근접한 수치다. 캐시 안에 들어가는 작업 집합에서는 FSB 속도가 중요하지 않지만(이 프로세서들은 4MB L2를 가짐), 시스템의 실질 작업 집합은 동시 실행 중인 모든 프로세스의 메모리를 합친 것이므로 개별 프로그램은 작아도 시스템 전체로는 4MB를 넘기기 쉽다. 2007년 당시 일부 인텔 프로세서는 1,333MHz FSB(60% 추가 향상 가능)를 지원했고, 이후로도 속도는 계속 올라갈 전망이었다. 작업 집합이 크고 속도가 중요하다면 빠른 RAM과 높은 FSB 속도에 투자할 가치가 있지만, 프로세서가 지원하는 FSB 속도를 메인보드/노스브리지가 실제로 지원하는지는 반드시 확인해야 한다.
