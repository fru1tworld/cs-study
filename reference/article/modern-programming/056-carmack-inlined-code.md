# 존 카맥의 인라인 코드론 (John Carmack on Inlined Code)

> 원문: [John Carmack on Inlined Code](http://number-none.com/blow/john_carmack_on_inlined_code.html)
> 저자: John Carmack (Jonathan Blow의 사이트에 게재) | 이메일 작성일: 2007-03-13, 서문 2014-09-26 | 정리일: 2026-07-10

원문은 카맥이 id Software 사내에 보낸 이메일 전문과, 7년 뒤 덧붙인 서문으로 구성된 저작권 유보 문서다. 아래는 그 구조를 그대로 따라가며 내용을 상세히 재서술한 학습 노트다. 코드 예제는 원문 그대로 수록했다.

## 2014년 서문 (John Carmack, 2014-09-26)

이메일을 쓴 뒤 몇 년이 지나면서 카맥은 순수 함수형 프로그래밍에 훨씬 더 우호적이 되었다고 밝힌다. C/C++에서 합리적인 범위 안에서라도 그렇다는 것이다. (관련 글: [In-depth: Functional programming in C++](http://gamasutra.com/view/news/169296/Indepth_Functional_programming_in_C.php))

인라이닝이 겨냥하는 진짜 적은 **예기치 못한 의존성과 상태 변경**(mutation)인데, 함수형 프로그래밍은 이를 더 직접적이고 완전하게 해결한다. 다만 상태 변경을 많이 해야 한다면, 그것이 전부 인라인으로 한자리에서 일어나게 하는 데에도 장점이 있다. 자신이 저지르고 있는 일의 "끔찍함 전체"를 끊임없이 자각하게 되기 때문이다. 감당하기 버거워지는 시점이 오면, 블록들을 순수 함수로 분리해 내면 된다(그리고 다시 불순해지도록 방치하지 말 것).

이메일 본문의 논지를 확인해 주는 사례도 하나 덧붙인다. 몇 년 뒤 Doom 3 BFG Edition 작업 중, 이메일에서 정확히 예측했던 "입력 샘플링이 한 프레임 늦어지는" 버그가 실제로 발생했고 하마터면 그대로 출시될 뻔했다. 레이턴시와 반응성을 그토록 강조해 온 자신이 완전히 불필요한 한 프레임의 레이턴시를 안고 타이틀을 출시할 뻔했으니, 식은땀이 나는 순간이었다고 회고한다.

한 가지 단서도 단다. "항상 실행하고, 결과를 억제하거나 무시하라"는 전략은 고신뢰성 시스템에는 아주 좋은 생각이지만, 모바일처럼 전력·발열 제약이 있는 환경에는 덜 적합하다.

## 이메일 본문 (2007-03-13, 제목: "inlining code")

보낸 사람: John Carmack &lt;johnc@idsoftware.com&gt;

### 배경: 항공우주 소프트웨어에서 얻은 힌트

카맥은 이것이 코딩 스타일에 관한 특이한 메일이 될 것이라고 운을 뗀다. 지시를 내리려는 것이 아니라 진지하게 고민해 보자는 제안이며, 연산자 주변 공백이나 중괄호 스타일 같은 사소한 문제가 아니라 코드의 큰 구조에 관한 이야기다. 은탄환은 없지만, 생산성이 몇 퍼센트만 좋아져도 개발 기간에서 몇 달을 줄일 수 있다.

1년 전 그는 항공우주용 초고신뢰성 소프트웨어에 관한 논의에 참여했다. 그런 자리에서 그는 늘 스레드와 RTOS를 쓰자는 사람들에 맞서, 원시적인 비디오 게임처럼 생긴 단순한 폴링 루프가 훨씬 명확하고 효과적이라고 주장해 왔다. 그런데 이번에는 새로운 관점을 접했다. 인용된 Henry Spencer의 회고에 따르면, 사브 그리펜(Saab Gripen) 전투기의 플라이바이와이어 비행 소프트웨어는 한 걸음 더 나아가 **서브루틴 호출과 후방 분기(backward branch)를 금지**했다(메인 루프 맨 아래의 분기 하나만 예외). 제어 흐름은 오직 앞으로만 갔다. 앞쪽 코드가 뒤쪽 코드에 "할 일 메모"를 남겨야 할 때도 있었지만, 모든 데이터가 정적으로 할당되어 있어 그 변수들만 관찰하면 소프트웨어가 하는 일 대부분이 훤히 보였고, 테스트에 유리했다. 그리고 그 코드의 "비행 승인" 버전에서는 버그가 단 하나도 발견된 적이 없다고 한다.

카맥은 항공우주 업계의 관행 상당수는 누구도 흉내 내면 안 되고 자기 파괴적이기까지 하다고 단서를 단다. 우주왕복선 소프트웨어 개발 프로세스에 관한 유명한 기사들처럼, 모두가 그렇게 "신중하게" 개발했다면 세상은 수십 년 뒤처져 PC도 인터넷도 없었을 것이다.

하지만 이 일화만은 실용적 가치가 있어 보여 직접 실험했다. Armadillo 로켓의 비행 제어 코드는 수천 줄 규모라, 메인 tic 함수에서 시작해 모든 서브루틴을 인라인해 봤다. (문자 그대로의) 추락을 일으킬 숨은 버그를 찾았다고까지는 못 하지만, 여러 번 중복 설정되는 변수 몇 개와 수상쩍은 제어 흐름 몇 군데를 발견했고, 최종 코드는 더 작고 깨끗해졌다.

그 스타일로 한동안 살아 보니 단점을 찾지 못했고, id의 자기 코드에도 이 접근을 적용하기 시작했다. 코드를 조직하는 방식에는 대략 다음 선택지가 있다.

```cpp
------- 스타일 A:

void MinorFunction1( void ) {
}

void MinorFunction2( void ) {
}

void MinorFunction3( void ) {
}

void MajorFunction( void ) {
        MinorFunction1();
        MinorFunction2();
        MinorFunction3();
}

--------- 스타일 B:

void MajorFunction( void ) {
        MinorFunction1();
        MinorFunction2();
        MinorFunction3();
}

void MinorFunction1( void ) {
}

void MinorFunction2( void ) {
}

void MinorFunction3( void ) {
}

---------- 스타일 C:

void MajorFunction( void ) {
        // MinorFunction1

        // MinorFunction2

        // MinorFunction3

}
```

카맥은 역사적으로 프로토타입 선언을 생략할 수 있는 "스타일 A"를 써 왔고, "스타일 B"를 선호하는 사람도 있지만 둘의 차이는 대수롭지 않다. Michael Abrash는 "스타일 C"로 코드를 썼는데, 카맥은 예전에 가독성이 좋아진다고 생각해 Abrash의 코드를 오히려 "스타일 A"로 변환한 적도 있다.

이제 와서 보니 "스타일 C"에는 분명한 장점이 있다고 생각한다. 다만 그 장점은 정량화되는 개별 사실이 아니라 개발 프로세스 차원의 것이고, 통념과 상당히 어긋나므로 명확한 논거를 제시하겠다고 한다. 도그마는 아니며, 어디에 적절하고 어디에 부적절한지를 따져 보는 것 자체가 가치 있다.

중요한 전제: **함수 호출을 피하는 것 자체가 성능에 직접 도움이 된다는 주장은 전혀 아니다.**

### 논거 1: 실제로 실행되는 코드 전체에 대한 자각

카맥이 가끔 하는 훈련이 있다. `common->Frame()`, `game->Frame()`, `renderer->EndFrame()` 같은 주요 지점에서 시작해 게임의 "한 프레임을 스텝 실행"하며 모든 함수 안으로 들어가 코드 커버리지 전체를 걸어 보는 것이다. 보통 프레임 끝에 도달하기 한참 전에 우울해진다. 실제로 실행되는 모든 코드를 자각하는 것은 중요한데, 성능과 안정성에 영향을 주는데도 디버깅할 때 늘 건너뛰기만 하는 거대한 코드 블록이 생기기 너무 쉽다.

연산자 오버로딩, 암묵적 생성자 등을 가진 C++는 이 목표에 우호적이지 않다. 유연성이라는 이름으로 행해지는 많은 것들이 다소 잘못된 방향이며 수많은 개발 문제의 뿌리에 있다. 또 게임은 실시간 tic 구조가 계속 반복된다는 점에서, 콘텐츠 제작 도구나 트랜잭션 처리기와는 다른 프로그래밍 스타일을 요구하는 목표와 제약을 갖는다.

### 논거 2: 프레임 루프의 바깥층에서 일어나게 하라

프레임마다 한 번 일어날 일이라면, 어떤 이유로든 건너뛰어질 수 있는 함수 사슬 깊숙이 묻혀 있는 것보다 프레임 루프의 가장 바깥 부분에서 일어나는 편이 낫다. 예로 id의 `usercmd_t` 생성 코드는 asyncServer 안쪽에 묻혀 있는데, 사실 메인 common 루프에 있어야 한다. 하드웨어 대 소프트웨어 설계라는 주제와도 연결된다. **연산을 조건부로 수행하려 하기보다, 일단 수행한 뒤 결과의 일부 또는 전부를 억제하거나 무시하는 편이 나을 때가 많다.** `usercmd_t` 생성이 그렇다(게임 오버와 바인드셋 상호작용에 얽힌 다른 난장판들도 그렇다). "필요할 때만" 생성되기 때문이다.

전통적으로 게임의 성능을 측정하고 최적화해 온 방식은 조건부 연산을 부추겼다. 특정 상태에서는 이 연산이 필요 없다는 것을 알아채고 건너뛰는 식이다. 데모 타이밍 수치는 좋아지지만, 비싼 연산을 건너뛸 때 다른 곳에서 필요한 상태 갱신까지 함께 건너뛰는 바람에 어마어마한 버그가 생긴다.

최적화가 정말 필요한 성능 집약적 작업이야 여전히 있지만, 이 스타일은 성능 이득이 무시할 만한 곳에서도 습관처럼 적용되고 버그만 떠안는다. 60Hz 게임으로 확정한 지금은 평균 성능보다 최악 경우(worst case) 성능이 더 중요하므로, 변동이 심한 성능은 더욱 나쁘게 봐야 한다.

연산이 여러 서브시스템 깊숙이 중첩되어 수행되고 시간이 흐르며 코드가 진화하면, 한 프레임짜리 동작 레이턴시가 스며들기 아주 쉽다. 이는 입력 품질이 미묘하게 떨어지는 정도로 숨어 있을 수도 있고, 이동 중에 모델이 부착점보다 한 박자 늦게 끌려가는 식으로 노골적으로 드러날 수도 있다. 모든 것이 2000줄짜리 함수 하나에 펼쳐져 있으면 어느 부분이 먼저 실행되는지 명백하고, 뒤쪽 구간이 프레임 렌더링 전에 실행된다는 것을 확신할 수 있다.

### 논거 3: 다른 곳에서 호출할 수 없다는 것 자체가 이점

인라이닝에는 실행 코드에 대한 자각 외에, **그 함수를 다른 곳에서 호출하는 것이 불가능해진다**는 이점도 있다. 우스꽝스럽게 들리지만 요점이 있다. 코드베이스가 수년간 자라다 보면, 지름길을 택해 "필요하다고 생각되는 일만 하는" 함수를 직접 호출할 기회가 널려 있게 된다. `PartialUpdateA()`와 `PartialUpdateB()`를 호출하는 `FullUpdate()`가 있는데, 어떤 경우에는 `PartialUpdateB()`만 하면 된다고 판단(혹은 착각)하고 나머지 작업을 아껴 효율적으로 군다. 여기서 수많은 버그가 나온다. **대부분의 버그는 실행 상태가 자신이 생각하는 것과 정확히 일치하지 않아서 생긴다.**

입력 인자만 읽고 영구 상태를 조회하거나 수정하지 않고 값만 반환하는 엄밀한 함수형 함수는 이런 오류에서 안전하고, 형식적으로 논증할 수 있어 상아탑의 좋은 주제가 되지만, 실제 코드 중 이 범주에 드는 것은 아주 적다. 카맥은 순수 함수형 프로그래밍을 대규모로 쓰는 것이 실용적 개발 계획이라고는 보지 않는다. 코드가 매우 난해해지고 비효율이 극심해지기 때문이다. 하지만 함수가 전역 상태 한두 조각만 참조한다면 그것을 인자로 넘기는 쪽을 고려하는 것이 현명하다. C에 전역 참조를 금지하는 "functional" 키워드가 있었으면 좋겠다고도 한다.

const 매개변수와 const 함수는 부수효과 버그를 피하는 데 도움이 되지만, 함수는 여전히 전역 실행 환경의 변화에 취약하다. 더 많은 매개변수와 함수를 const로 만들어 보는 것은 좋은 훈련이며, 어느 시점에 좌절해서 캐스팅으로 걷어내며 끝나곤 한다. 그 좌절은 대개, 즉시 눈에 띄지 않던 온갖 상태 변경 지점 — 버그가 번식하는 자리 — 을 발견했기 때문이다.

C++ 객체 메서드는 반환 시 암묵적 덮어쓰기 대입이 일어나는 "거의 함수형"으로 볼 수도 있지만, 변수가 많은 큰 객체에서는 메서드가 무엇을 수정하는지 잘 보이지 않고, 그 함수가 decl 파싱 같은 끔찍하게 전역적인 일을 하러 달려가지 않는다는 보장도 없다.

**문제를 일으킬 가능성이 가장 낮은 함수는 존재하지 않는 함수이며, 그것이 인라이닝의 이점이다.** 함수가 한 곳에서만 호출된다면 결정은 꽤 단순하다.

### 코드 중복에 관하여

거의 모든 경우에 코드 중복은, 함수가 다른 상황에서 호출되며 생기는 2차적 문제들보다 더 큰 악이다. 그래서 함수를 피하려고 코드를 복제하는 것은 거의 권하지 않는다. 다만 많은 경우, 연산을 플래그로 표시해 두었다가 적절히 통제된 시점에 수행하게 하면 함수 없이도 해결된다. 예컨대 체력에 대한 검사를 플레이어 think 코드 한 곳에 두는 식이다. (역주: 원문 문장이 이 지점에서 끊겨 있다.)

코드 중복 이야기가 나온 김에: 카맥은 한동안 자신이 고친 버그(다 됐다고 생각한 뒤에 드러난 문제)를 전부 추적해 봤는데, 복사-붙여넣기-수정이 즉시 눈에 띄지 않는 미묘한 버그를 얼마나 자주 만드는지에 꽤 놀랐다. 서너 개짜리 벡터 연산이라면 흔히 이렇게 붙여넣고 몇 글자만 고치곤 했다.

```cpp
v[0] = HF_MANTISSA(*(halfFloat_t *)((byte *)data + i*bytePitch+j*8+0));
v[1] = HF_MANTISSA(*(halfFloat_t *)((byte *)data + i*bytePitch+j*8+1));
v[2] = HF_MANTISSA(*(halfFloat_t *)((byte *)data + i*bytePitch+j*8+2));
v[3] = HF_MANTISSA(*(halfFloat_t *)((byte *)data + i*bytePitch+j*8+3));
```

이제는 모든 것에 명시적 루프를 강력히 권하고, 컴파일러가 알아서 잘 언롤하기를 바란다. 자기 버그의 상당수가 이런 것과 관련돼 있었고, 흔히 `_X`, `_Y`나 `_WIDTH`, `_HEIGHT` 같은 개별 변수를 쓰는 2차원 경우조차 다시 생각하는 중이다. 두 원소짜리 배열보다 그쪽이 훨씬 읽기 쉽다고 느끼지만, 그것이 얼마나 자주 실수를 낳았는지에 대한 자신의 데이터를 반박하기는 어렵다.

### 실무적인 문제들

큰 함수 안에서 작은 함수들의 경계를 큼직한 주석 블록으로 구분해 두면 빠르게 훑기 좋고, 그 구간을 이름 없는 중괄호 블록으로 감싸 지역 변수의 스코프를 한정하고 에디터에서 접을 수 있게 하는 것도 유용하다. 함수를 한두 페이지보다 크게 만들지 말라는 경험칙들이 있다는 것을 알지만, 카맥은 이제 그에 명시적으로 동의하지 않는다. **많은 연산이 순차적으로 일어나야 한다면, 그 코드도 순차적으로 이어져야 한다.**

여러 페이지의 코드를 조건문이나 루프 안에 넣는 것은 가독성과 자각 측면에서 단점이 있으므로, 그런 코드는 별도 함수로 두는 것이 여전히 정당화될 수 있다. 다만 그 코드를 실행이 조건부가 아닌 다른 자리로 옮기거나, 항상 실행하되 아주 작은 조건 블록으로 결과를 억제하는 것이 가능한 경우도 있다. 실행 후 억제(execute-and-inhibit) 스타일은 보통 절대 시간은 더 들지만, 프레임 시간의 변동성을 줄이고 버그의 한 부류를 통째로 없앤다.

코드 인라이닝은 모듈성 및 OOP의 보호 장치와 금세 충돌하므로 좋은 판단이 필요하다. 모듈성의 요체는 디테일을 숨기는 것인데, 여기서 주장하는 것은 디테일에 대한 자각을 높이는 것이기 때문이다. 소스 파일 다중 체크아웃 증가, 로컬 데이터가 마스터 프리컴파일 헤더에 더 많이 들어가 전체 리빌드가 잦아지는 문제 같은 실무 요인도 저울질해야 한다. 현재 카맥의 기울기는 이렇다. 코드를 합치는 합리적 분기점으로 무거운(heavyweight) 객체를 쓰고, 중간 크기 헬퍼 객체의 사용은 줄이고, 아주 가벼운 객체는 꼭 존재해야 한다면 최대한 순수 함수형으로 만든다.

### 요약

- 함수가 한 곳에서만 호출된다면 인라인을 고려하라.
- 함수가 여러 곳에서 호출된다면, 플래그 등을 써서 작업이 한 곳에서 이루어지도록 재배치할 수 있는지 보고, 그것을 인라인하라.
- 한 함수의 변형이 여러 개 있다면, 매개변수를 늘리고(기본값을 줄 수도 있는) 단일 함수로 합치는 것을 고려하라.
- 작업이 거의 순수 함수형에 가깝고 전역 상태 참조가 적다면, 완전히 함수형으로 만들어 보라.
- 함수를 정말로 여러 곳에서 써야 한다면 매개변수와 함수 양쪽에 const를 쓰려고 노력하라.
- 제어 흐름의 복잡도와 "if 아래의 면적"을 최소화하라. 불필요한 작업을 "최적으로" 회피하는 것보다 일관된 실행 경로와 실행 시간을 우선하라.
