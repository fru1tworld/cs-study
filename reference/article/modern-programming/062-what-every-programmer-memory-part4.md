## 6장 프로그래머가 할 수 있는 일

앞 장들에서 봤듯 메모리 관련 연산만으로도 프로그램 성능에 영향을 줄 기회는 매우 많다. 이 장은 물리 RAM 접근과 L1 캐시라는 가장 낮은 수준부터 시작해, 메모리 처리에 영향을 주는 OS 기능까지 아래에서 위로 훑는다.

### 6.1 캐시 우회하기

데이터를 생성만 하고 (즉시) 다시 쓰지 않는다면, 메모리 저장 연산이 항상 캐시 라인 전체를 먼저 읽고 나서 수정한다는 사실이 오히려 손해가 된다. 이 동작은 곧 다시 쓰일 데이터를 캐시에서 밀어내고 당분간 안 쓰일 데이터를 남긴다. 행렬처럼 채워 넣은 뒤 나중에 쓰는 대형 자료구조에서 특히 그렇다 — 행렬의 마지막 원소를 채우기도 전에 이미 크기 때문에 첫 원소들이 밀려나 쓰기 캐싱 자체가 무의미해진다.

이런 상황을 위해 프로세서는 **비시간적(non-temporal) 쓰기 연산**을 지원한다. "비시간적"이란 그 데이터가 곧 재사용되지 않을 것이므로 캐싱할 이유가 없다는 뜻이다. 이 연산은 캐시 라인을 읽고 수정하는 대신 새 내용을 메모리에 직접 쓴다. 비싸 보일 수 있지만 꼭 그렇지는 않다 — 프로세서가 write-combining(3.3.3절)을 활용해 캐시 라인 전체를 채우려 시도하고, 성공하면 메모리 읽기 자체가 필요 없다. x86/x86-64에서 gcc가 제공하는 대표 인트린식은 다음과 같다:

```c
#include <emmintrin.h>
void _mm_stream_si32(int *p, int a);
void _mm_stream_si128(int *p, __m128i a);
void _mm_stream_pd(double *p, __m128d a);
```

이 명령어들은 대량의 데이터를 한 번에 처리할 때 가장 효율적이다 — 메모리에서 읽어 한두 단계 처리하고 다시 메모리에 쓰는 흐름이라 "스트림(stream)"이라는 이름이 붙었다. 주소는 8바이트 또는 16바이트로 정렬되어야 한다. A.1절의 행렬 곱셈 코드에서는 쓰인 값이 곧바로 재사용되기 때문에 이 명령어를 쓰지 않는다 — 스트림 명령어가 오히려 부적합한 사례다.

프로세서의 write-combining 버퍼는 캐시 라인 일부 쓰기 요청을 그리 오래 담아두지 못한다. 그래서 한 캐시 라인을 수정하는 모든 명령어를 연달아 실행해야 write-combining이 실제로 일어난다. 예시:

```c
#include <emmintrin.h>
void setbytes(char *p, int c)
{
  __m128i i = _mm_set_epi8(c, c, c, c, c, c, c, c,
                           c, c, c, c, c, c, c, c);
  _mm_stream_si128((__m128i *)&p[0], i);
  _mm_stream_si128((__m128i *)&p[16], i);
  _mm_stream_si128((__m128i *)&p[32], i);
  _mm_stream_si128((__m128i *)&p[48], i);
}
```

`p`가 적절히 정렬돼 있다면, 이 함수 호출은 해당 캐시 라인의 모든 바이트를 `c`로 설정한다. write-combining 로직은 연속으로 생성된 네 개의 `movntdq` 명령어를 보고, 마지막 명령어가 실행된 뒤에야 메모리 쓰기 명령을 발행한다. 즉 이 코드는 쓰기 전 캐시 라인 읽기를 피할 뿐 아니라, 곧 필요하지 않을 데이터로 캐시를 오염시키는 것도 피한다. 대용량 블록에 대해 C 런타임의 `memset` 함수가 실제로 이런 방식을 쓴다.

PowerPC의 `dcbz` 명령어처럼 특화된 해법을 제공하는 아키텍처도 있다 — 캐시 라인 전체를 0으로 지우되(결과를 위한 캐시 라인은 여전히 할당되므로 완전한 캐시 우회는 아니고, 전부 0으로만 설정 가능하다는 제약도 있지만) write-combining 로직 없이도 결과를 얻는다.

**행렬 초기화 실험**: 2차원 배열로 조직된 3000×3000 행렬을 초기화하는 시간을 측정한다. IA-32에서는 "비시간적 힌트"를 쓴 저장과 일반 저장을 비교한다. 행 우선(row-major, 행 인덱스를 안쪽 루프에서 증가) 순서와 열 우선(안쪽 루프에서 열 인덱스 증가) 순서 두 가지로 테스트한 결과(원문 표 6.1, 초 단위):

| | 행(순차) | 열(비순차) |
|---|---|---|
| 일반 저장 | 0.048s | 0.127s |
| 비시간적 저장 | 0.048s | 0.160s |

일반 저장에서 순차 접근(약 750MB/s)이 사실상 랜덤에 가까운 열 우선 접근(약 280MB/s)보다 훨씬 빠른 것은 예상대로다 — 행렬이 워낙 커서 캐시가 사실상 무력화된다. 흥미로운 부분은 캐시를 우회하는 비시간적 쓰기 쪽이다. 순차 접근에서는 캐시를 쓰는 경우와 똑같이 빠른데, 이는 앞서 설명한 write-combining 덕분이다(비시간적 쓰기의 메모리 순서 규칙이 완화되어 있어 — x86/x86-64는 `sfence` 명령어로 명시적 메모리 배리어를 넣어야 함 — 프로세서가 데이터를 더 자유롭게 반영할 수 있어 가용 대역폭을 최대한 활용한다). 반면 열 우선(비순차) 접근에서는 write-combining이 불가능해(각 메모리 셀을 개별적으로 주소 지정해야 함) 캐시된 경우보다 25% 나쁜 결과(0.16s, 약 225MB/s)가 나온다.

읽기 쪽은 최근까지 비시간적 힌트를 준 프리페치 명령 정도만 지원되었고, write-combining에 대응하는 것이 없어 메모리 매핑 I/O처럼 캐시 불가능한 메모리에서는 특히 불리했다. 인텔은 SSE4.1에서 `movntdqa`로 NTA 로드를 도입했다 — 소수의 스트리밍 로드 버퍼(각각 캐시 라인 하나 크기)를 이용해, 같은 캐시 라인에 대한 이후 접근이 저비용으로 이 버퍼에서 처리되고 대체로 실제 캐시에는 적재되지 않는다. 이 실험에서 얻을 교훈은, 현대 CPU가 순차적이기만 하면 캐시 없는 쓰기(그리고 최근엔 읽기까지)를 잘 최적화한다는 것 — 한 번만 쓰이는 대형 자료구조를 다룰 때 매우 유용하다. 반대로 캐시는 랜덤 접근 비용의 일부만 가려줄 뿐 전부를 가려주지 못한다(이 예시에서 랜덤 접근이 70% 느림) — 가능하면 랜덤 접근 자체를 피해야 한다.

### 6.2 캐시 접근

성능 개선의 최우선 대상은 L1 캐시다. L1 최적화는 다른 레벨에도 자연히 영향을 준다. 모든 메모리 접근 최적화의 공통 주제는 **지역성(시간적·공간적)을 개선하고 코드·데이터를 정렬**하는 것이다.

#### 6.2.1 L1 데이터 캐시 접근 최적화

**행렬 곱셈 예제**: N×N 정방 행렬 `A`, `B`의 곱 `AB`는 `(AB)ij = Σk aik·bkj`다. 단순한 C 구현:

```c
for (i = 0; i < N; ++i)
  for (j = 0; j < N; ++j)
    for (k = 0; k < N; ++k)
      res[i][j] += mul1[i][k] * mul2[k][j];
```

`mul1`은 순차 접근되지만 안쪽 루프가 `mul2`의 행 번호를 계속 바꾸며 진행하므로, `mul2`는 그림 6.1이 보여주는 "나쁜" 접근 패턴(열 우선)을 겪는다. 한 라운드의 안쪽 루프가 세 행렬 각각에 대해 (Core 2 기준 64바이트 캐시 라인 1,000개, 즉 32KB L1d보다 훨씬 큰 양)을 요구하게 된다.

**첫 번째 개선: 전치(transpose)**. 곱셈의 덧셈 순서는 결과에 영향을 주지 않으므로(오버플로/언더플로/반올림 효과는 논외), `mul2`를 미리 전치해 두면 `(AB)ij = Σk aik·(bT)jk`가 되어 두 입력 모두 순차 접근이 가능해진다:

```c
double tmp[N][N];
for (i = 0; i < N; ++i)
  for (j = 0; j < N; ++j)
    tmp[i][j] = mul2[j][i];
for (i = 0; i < N; ++i)
  for (j = 0; j < N; ++j)
    for (k = 0; k < N; ++k)
      res[i][j] += mul1[i][k] * tmp[j][k];
```

Intel Core 2(2,666MHz) 측정 결과(사이클 수): 원본 16,765,297,870 사이클 대비 전치 버전 3,922,373,010 사이클로, **76.6% 속도 향상**이다. 전치 복사에 드는 추가 메모리 접근 비용을 감안해도, 열 단위 1,000회의 비순차 접근이 주던 손해가 훨씬 크다는 뜻이다.

**두 번째 개선: 서브 행렬(캐시 라인 블로킹) + 복사 제거**. 항상 별도 복사본을 만들 여유가 있는 건 아니다(행렬이 너무 크거나 메모리가 부족할 수 있음). 원본 코드의 `mul2` 접근 순서를 보면 (0,0),(1,0),...,(N-1,0),(0,1),(1,1)... 순인데, (0,0)과 (0,1)은 같은 캐시 라인에 있어도 안쪽 루프 한 바퀴가 끝날 즈음엔 이미 그 라인이 축출된 뒤다. 안쪽 루프에서 중간 루프 두 번을 함께 처리하면(같은 캐시 라인의 두 double 값을 활용) L1d 미스율을 절반으로 줄일 수 있고, 캐시 라인 크기(Core 2 기준 64바이트, `sysconf(_SC_LEVEL1_DCACHE_LINESIZE)` 또는 `getconf LEVEL1_DCACHE_LINESIZE`로 조회 가능)만큼 완전히 활용하려면 중간 루프를 8번 언롤해야 하고, `res` 행렬 활용까지 고려하면(한 번에 8개 결과를 씀) 바깥 루프도 8번 언롤해야 한다. 이를 반영한 6중 루프 코드:

```c
#define SM (CLS / sizeof (double))
for (i = 0; i < N; i += SM)
  for (j = 0; j < N; j += SM)
    for (k = 0; k < N; k += SM)
      for (i2 = 0, rres = &res[i][j], rmul1 = &mul1[i][k];
           i2 < SM; ++i2, rres += N, rmul1 += N)
        for (k2 = 0, rmul2 = &mul2[k][j]; k2 < SM; ++k2, rmul2 += N)
          for (j2 = 0; j2 < SM; ++j2)
            rres[j2] += rmul1[k2] * rmul2[j2];
```

바깥 세 겹 루프는 `SM`(캐시 라인 크기 ÷ `sizeof(double)`) 간격으로 반복해 전체 곱셈을 더 작은 문제로 쪼개고, 안쪽 세 겹 루프가 나머지 인덱스를 처리한다(`k2`, `j2` 순서가 바뀐 이유는 실제 계산에서 `k2`에 의존하는 식은 하나뿐이지만 `j2`에 의존하는 식은 둘이기 때문). `rres`, `rmul1`, `rmul2` 같은 보조 변수는 gcc의 배열 인덱싱 최적화 한계(C/C++의 기본 앨리어싱 규칙상 `restrict`를 안 쓰면 포인터 접근은 모두 앨리어싱 가능성이 있다고 취급됨) 때문에 공통 표현식을 안쪽 루프 밖으로 최대한 끌어올리려 도입한 것이다(포트란이 수치 프로그래밍에서 여전히 선호되는 이유이기도 하다).

전체 결과(원문 표 6.2, 상대 비율):

| | 원본 | 전치 | 서브 행렬 | 벡터화 |
|---|---|---|---|---|
| 사이클 | 16,765,297,870 | 3,922,373,010 | 2,895,041,480 | 1,588,711,750 |
| 상대 비율 | 100% | 23.4% | 17.3% | 9.47% |

서브 행렬 방식은 복사를 피하면서(추가 메모리 불필요, 결과 행렬만 메모리에 들어가면 임의로 큰 입력 행렬도 처리 가능) 전치보다도 6.1%p 더 개선된다. 마지막으로 SSE2 같은 벡터화 명령어(더블 두 개를 한 연산으로 처리)를 쓰면 원본 대비 추가로 7.3%p 개선되어 최종적으로 원본의 약 10% 시간(318 MFLOPS → 3.35 GFLOPS)에 도달한다 — 단, 이 마지막 버전에서도 `mul2`에 대한 프리페칭은 여전히 작동하지 않는데, 이는 전치 없이는 해결할 수 없는 부분이다. 그럼에도 2.66GHz 단일 스레드 코드로 3.19 GFLOPS라면 나쁘지 않은 결과다(전체 코드는 A.1절 참고).

이 예제에서 최적화한 것은 결국 **적재한 캐시 라인의 활용**이다 — 라인의 모든 바이트가 항상 쓰이도록, 그리고 그 라인이 축출되기 전에 쓰이도록 만든 것뿐이다.

**구조체 크기와 캐시 라인**: 더 흔한 상황은 하나 이상의 캐시 라인을 채우는 자료구조에서 몇몇 멤버만 자주 쓰는 경우다(3.11절에서 이미 큰 구조체 크기의 영향을 봤다). `pahole` 도구는 바이너리에 정의된 자료구조의 캐시 라인 배치를 보여준다. 예:

```c
struct foo {
    int a;
    long fill[7];
    int b;
};
```

64비트 머신에서 `pahole` 출력은 `a` 다음 4바이트 구멍(정렬 요구사항 때문), `fill`이 캐시 라인 1개를 채움, `b`가 두 번째 캐시 라인에 위치해 구조체가 총 72바이트에 캐시 라인 2개를 차지한다는 것을 보여준다. `b`를 `a` 뒤의 4바이트 구멍으로 옮기면 구멍이 사라지고 구조체 전체가 캐시 라인 하나에 들어간다 — `pahole --reorganize`로 이런 최적화를 자동으로 얻을 수도 있다.

일반 원칙 두 가지:

1. 크리티컬 워드가 될 가능성이 높은 멤버를 구조체 맨 앞으로 옮긴다.
2. 접근 순서가 상황에 따라 정해지는 게 아니라면, 구조체에 정의된 순서대로 멤버에 접근한다.

작은 구조체는 접근 빈도 순으로 멤버를 배열하고, 큰 구조체는 캐시 라인 크기 블록 단위로 이 규칙을 적용한다. 다만 객체 자체가 캐시 라인 경계에 정렬되어 있지 않다면 이런 재배열은 무의미하다 — 정렬은 데이터 타입의 정렬 요구사항으로 정해지고(구조체는 그 안의 가장 큰 정렬 요구사항을 따름), 대개 캐시 라인 크기보다 작다. 필요한 정렬을 보장하려면:

- 동적 할당은 `posix_memalign(void **memptr, size_t align, size_t size)`를 쓴다(일반 `malloc`은 가장 까다로운 표준 타입, 보통 `long double` 정렬만 보장).
- 컴파일러가 할당하는 전역/자동 변수는 `__attribute((aligned(64)))`를 붙인다(배열은 각 원소 크기가 정렬값의 배수여야 첫 원소 외의 정렬도 보장됨).
- 사용자 정의 타입 자체에 `__attribute((aligned(64)))`를 붙이면 그 타입의 모든 객체(배열 포함)가 자동으로 정렬된다.

**정렬되지 않은 접근의 비용**: 멀티미디어 확장 명령어는 거의 항상 정렬된 접근을 요구한다(16바이트 접근이면 16바이트 정렬). x86/x86-64는 정렬 안 된 접근도 처리하는 변형 명령어를 제공하지만 더 느리다. 원문의 실험(그림 6.4, 리스트 원소를 증가시키는 익숙한 테스트를 정렬/비정렬 버전으로 비교)에 따르면, L2 캐시에 들어가는 작업 집합 크기에서 순차 접근의 정렬 안 된 접근 페널티가 약 300%에 달한다(증가 연산 일부가 캐시 라인 두 개를 건드리게 되고, L1-L2 연결이 정체되기 때문). 아주 큰 작업 집합에서도 여전히 20~30%의 손해가 남는다. 아키텍처가 정렬 안 된 접근을 지원한다고 해서 "정렬된 접근만큼 좋다"고 오해해서는 안 된다.

이런 정렬 요구사항 때문에 컴파일러는 스택 프레임 정렬도 신경 써야 한다(호출 규약이 대개 "호출하는 쪽이 스택을 정렬해야 한다"는 두 번째 방식을 택함). 가변 길이 배열(VLA)이나 `alloca`를 쓰면 스택 프레임 크기를 컴파일 타임에 알 수 없어 동적 정렬 제어가 필요해지고 코드가 약간 느려진다. 멀티미디어 확장만 엄격한 정렬을 요구하는 아키텍처라면, 그런 명령어를 안 쓰는 말단 함수(tail function, 다른 함수를 호출하지 않는 함수)에 한해 정렬 요구를 완화할 수도 있다 — x86 바이너리라면 `-mpreferred-stack-boundary=N` 옵션으로 스택 정렬 요구를 2^N바이트로 낮출 수 있다(단, x86-64 ABI는 부동소수점 인자를 SSE 레지스터로 전달하므로 이 옵션의 적용 범위가 제한적).

**자료구조 분리**: 배열 원소가 구조체라면 구조체 정의 전체가 성능에 영향을 준다. 예:

```c
struct order {
   double price;
   bool paid;
   const char *buyer[5];
   long buyer_id;
};
```

이런 레코드가 큰 배열로 저장되고, 미결제 청구서의 예상 금액을 합산하는 작업이 자주 돈다면, `buyer`와 `buyer_id` 필드까지 불필요하게 캐시에 함께 실려 최대 5배 성능 손해가 날 수 있다. `price`·`paid`를 별도 구조체로 분리하고 나머지를 다른 곳에 두는 편이 훨씬 낫다 — 프로그램 복잡도는 올라가지만 성능 이득이 그 비용을 정당화할 수 있다.

**연관도와 충돌 미스**: L1d는 완전 연관이 되기엔 너무 크고 L2만큼 연관도가 높지도 않다. 작업 집합의 많은 객체가 같은 캐시 세트에 몰리면(**충돌 미스**), 캐시 상당 부분이 비어 있어도 지연이 발생할 수 있다. L1d는 가상 주소로 인덱싱되므로 이는 프로그래머가 어느 정도 통제할 수 있는 영역이다 — 함께 쓰이는 변수를 함께 저장하면 같은 세트에 몰릴 확률이 줄어든다. 원문 실험(그림 6.5, NPAD=15 Follow 테스트, 리스트 원소 간 거리를 바꿔가며 측정)에서, 거리가 4096바이트의 배수(64원소)이고 리스트 길이가 8을 넘으면(연관도를 넘으면) 원소당 사이클이 급증한다 — 이로부터 테스트 대상 프로세서의 L1d가 연관도 8, 총 32KB임을 역산할 수 있다. 2의 거듭제곱 경계에 데이터를 배치하는 것은 실무에서 흔하지만, 바로 이런 상황을 만들기 쉽다는 점을 유의해야 한다. AMD 프로세서는 L1d를 여러 뱅크로 구현해 사이클당 두 데이터 워드를 받을 수 있는데(단, 서로 다른 뱅크이거나 같은 인덱스의 같은 뱅크일 때) 함께 쓰이는 변수를 함께 저장하면 이 조건을 만족할 확률도 높아진다.

#### 6.2.2 L1 명령어 캐시 접근 최적화

L1i를 잘 쓰려면 L1d와 비슷한 기법이 필요하지만, 프로그래머가 어셈블리를 직접 쓰지 않는 한 컴파일러의 코드 생성을 유도하는 간접적 방식뿐이다. 코드는 점프 사이 구간에서는 선형이라 프로세서가 효율적으로 프리페치할 수 있지만, 점프는 (1) 목표가 정적으로 정해지지 않을 수 있고 (2) 정적이어도 모든 캐시를 놓치면 메모리 페치에 오래 걸릴 수 있어 이 그림을 흐트러뜨린다. 그래서 오늘날 프로세서는 분기 예측(BP)에 상당한 노력을 쏟는다.

L1i를 잘 활용하려면: (1) 코드 풋프린트를 최대한 줄인다(루프 언롤링·인라이닝과 균형을 맞춰야 함), (2) 실행이 버블 없이 선형적이어야 한다, (3) 의미 있는 곳에서 코드를 정렬한다.

**컴파일러 옵션**: `-O2`/`-O3`의 루프 최적화·함수 인라이닝은 대체로 좋지만 코드 크기를 늘려 L1i(및 L2 이상)에 부담을 줄 수 있다. `-Os`는 코드 크기를 우선한다. `-finline-limit`으로 인라이닝될 함수 크기 상한을 조절할 수 있다. 한 함수가 여러 곳에서 호출되면 모든 호출처에 인라이닝하는 것은 코드 크기 폭증으로 이어질 수 있다(반대로 한 번만 호출되는 함수는 인라이닝이 값 범위 전파(value range propagation) 등 추가 최적화 기회를 열어주므로 거의 항상 이득). `always_inline`, `noinline` 함수 속성으로 개별 함수 단위 제어도 가능하다.

**분기 레이아웃**: 조건이 한쪽으로 치우친(lopsided) 조건부 실행은 정적 분기 예측 오류와 파이프라인 버블의 소지가 있다. gcc는 `__builtin_expect(long EXP, long C)`로 이를 보완하며, 흔히 다음 매크로로 감싸 쓴다:

```c
#define unlikely(expr) __builtin_expect(!!(expr), 0)
#define likely(expr)   __builtin_expect(!!(expr), 1)
```

`-freorder-blocks`(`-O2`에서 기본 활성화, `-Os`에서는 비활성)와 함께 쓰면 드물게 실행되는 코드 블록을 주 실행 경로 밖으로 옮겨, 자주 실행되는 코드가 메모리에서 선형으로 이어지게 만든다. gcc 프로파일 기반 재컴파일(7장에서 다룸)로도 같은 효과를 얻을 수 있다.

작은 루프의 이점도 있다 — Intel Core 2 프론트엔드의 **LSD**(Loop Stream Detector)는 명령어 18개 이하(서브루틴 호출 없음), 디코더 페치 최대 4회(16바이트씩), 분기 명령어 최대 4개, 실행 횟수 64회 초과인 루프를 명령어 큐에 "고정"해 재실행 시 더 빠르게 쓸 수 있게 한다.

**코드 정렬**: 명령어 하나하나를 정렬하는 건 의미가 없고, 다음 위치에서만 유효하다: (1) 함수 시작, (2) 점프로만 도달하는 기본 블록의 시작, (3) 어느 정도는 루프의 시작. 앞 두 경우는 비용이 거의 없지만(어차피 실행이 새 위치로 넘어가는 시점), 루프 시작 정렬은 이전 명령어와의 간격을 no-op이나 무조건 점프로 메워야 해서 실행 시점 비용이 생긴다(루프가 자주 실행되지 않으면 오히려 손해일 수 있음). 관련 gcc 옵션: `-falign-functions=N`, `-falign-jumps=N`, `-falign-loops=N`(정렬 요구 자체를 끄려면 N=1 또는 `-fno-align-functions`). `-falign-labels`는 모든 레이블(사실상 모든 기본 블록의 시작)을 정렬하는데, 예외적인 경우를 빼면 대개 코드를 느리게 만들어 권장되지 않는다.

#### 6.2.3 L2 이상 캐시 접근 최적화

L1에 적용되는 최적화는 L2 이상에도 그대로 적용되지만, 두 가지가 추가로 고려해야 할 사항이다: (1) 마지막 레벨 캐시의 미스는 항상 매우 비싸다(L1 미스는 대개 L2 이상에서 히트하지만 마지막 레벨 캐시는 폴백이 없음), (2) L2 이상은 여러 코어·하이퍼스레드가 공유하는 경우가 많아 실행 단위당 실질 캐시 크기는 전체보다 작다.

작업 집합을 캐시 크기에 맞추는 전략(50쪽의 최적화된 행렬 곱셈과 비슷한 발상, 다만 다루는 블록이 더 커도 됨)이 유효하다. L1 캐시 라인 크기는 프로세서 세대 간 큰 차이가 없어 하드코딩해도 무방하지만, 상위 레벨 캐시 크기는 8배 이상 차이 날 수 있어 하드코딩이 부적절하다. 최소 크기 기준으로 최적화하면 캐시 활용의 87% 이상을 버리는 셈이고, 최대 크기 기준이면 작은 캐시 머신에서 성능이 나쁘다 — 코드가 런타임에 캐시 크기에 맞춰 조정되어야 한다.

공유 캐시를 다른 코어/스레드와 나누는 경우, `/sys/devices/system/cpu/cpu*/cache` 아래 `size`와 `shared_cpu_map`을 읽어(마지막 레벨 캐시는 `level` 파일 값이 가장 큰 디렉터리) 크기를 공유하는 CPU 수로 나누면 안전한 하한을 계산할 수 있다. 다른 스레드/프로세스의 캐시 사용 패턴을 알고 있다면 이보다 더 많이 써도 되지만, 이는 상황에 따라 프로그래머나 사용자가 판단해야 한다.

#### 6.2.4 TLB 사용 최적화

TLB 최적화는 두 가지다: (1) 프로그램이 쓰는 페이지 수를 줄여 TLB 미스 자체를 줄인다, (2) 상위 레벨 디렉터리 테이블 수를 줄여 TLB 조회 자체를 저렴하게 만든다(테이블이 적을수록 메모리 사용도 줄어 디렉터리 조회의 캐시 히트율도 오른다).

첫 번째 최적화는 페이지 폴트 최소화(7.5절에서 다룸)와도 밀접하다. 페이지 폴트는 대개 일회성 비용이지만 TLB 미스는 TLB가 작고 자주 플러시되는 한 반복적으로 발생하는 비용이라, 프로그램이 충분히 오래 돌고 특정 부분이 충분히 자주 실행된다면 TLB 미스 누적 비용이 페이지 폴트 비용을 넘어설 수 있다.

두 번째 최적화는 통제가 더 어렵다. 필요한 페이지 디렉터리 수는 프로세스 가상 주소 공간에서 쓰이는 주소 범위의 분포에 좌우된다 — 위치가 넓게 흩어질수록 디렉터리가 더 필요하다. **ASLR**(Address Space Layout Randomization)이 정확히 이런 상황을 만든다(스택, DSO, 힙, 실행 파일 로드 주소를 런타임에 무작위화해 공격자의 함수·변수 주소 추측을 방해). 최고 성능이 정말 중요할 때만 ASLR을 꺼야 하고, 그 외에는 추가 디렉터리 비용이 낮아 대부분 상황에서 감내할 만하다. 프로그래머가 직접 영향을 주는 경우는 `mmap`에 `MAP_FIXED`로 주소 공간을 명시적으로 요청할 때뿐이다(위험하고 흔치 않음) — 주소를 자유롭게 고를 수 있다면 마지막 레벨 페이지 디렉터리의 경계를 알고 그에 맞춰 주소를 선택해야 한다.

### 6.3 프리페칭

프리페칭의 목적은 메모리 접근 지연을 감추는 것이다. 오늘날 프로세서의 명령어 파이프라인과 비순차(OOO) 실행이 어느 정도 지연을 감춰주지만, 이는 잘해야 캐시를 히트하는 접근에 한정된다. 메인 메모리 접근의 지연을 다 가리려면 명령어 큐가 터무니없이 길어야 한다.

#### 6.3.1 하드웨어 프리페칭

CPU가 하드웨어 프리페칭을 시작하는 트리거는 보통 특정 패턴의 캐시 미스 두 번 이상의 연속이다(옛날 구현은 인접 캐시 라인만 인식했지만, 요즘 하드웨어는 일정 간격(stride)을 건너뛰는 패턴도 인식). 캐시 미스 하나만으로 프리페치를 시작하면 전역 변수 같은 랜덤 접근 패턴에서 FSB 대역폭을 낭비하게 되므로 최소 두 번의 미스가 필요하다. 오늘날 CPU는 상위 레벨 캐시에서 8~16개의 서로 다른 스트림을 동시에 추적할 수 있지만, 이 프리페치 유닛은 같은 캐시를 쓰는 다른 코어·하이퍼스레드와 공유되므로 실제 가용 스트림 수는 금방 줄어든다.

하드웨어 프리페칭의 근본적 약점은 **페이지 경계를 넘지 못한다**는 것이다 — 프로세서는 데맨드 페이징을 지원하므로, 프리페처가 페이지 경계를 넘어 접근하면 OS 이벤트(페이지 폴트)를 유발할 수 있는데 프리페처는 프로그램·OS의 의미를 모르므로 실제로는 결코 요청되지 않을 페이지까지 건드릴 위험이 있다. 그래서 프리페처가 아무리 패턴을 잘 예측해도 페이지 경계에서는 명시적 프리페치나 새 페이지 읽기 없이는 캐시 미스를 겪게 된다 — 6.2절에서 설명한 데이터 배치 최적화(무관한 데이터를 멀리 두어 캐시 오염 최소화)가 여기서도 중요한 이유다. 페이지 크기 제약 때문에 오늘날 프로세서의 프리페치 패턴 인식 로직은 그리 정교하지 않다(보통 512바이트 윈도 정도까지만 의미가 있음). 비선형 접근 패턴은 인식하지 않는다(진짜 랜덤이거나 충분히 비반복적이라 인식 자체가 무의미하기 때문).

의도치 않게 하드웨어 프리페칭이 트리거되는 경우 대응이 마땅치 않다. 인텔 프로세서에서는 MSR(`IA32_MISC_ENABLE` 비트 9, 인접 캐시 라인 프리페치만 끄려면 비트 19)로 프로세서 전체의 하드웨어 프리페칭을 부분적/전체적으로 끌 수 있다(특권 연산이라 보통 커널에서 처리).

#### 6.3.2 소프트웨어 프리페칭

하드웨어 프리페칭은 프로그램 수정이 필요 없다는 장점이 있지만, 접근 패턴이 단순해야 하고 페이지 경계를 못 넘는다는 한계가 있다. 소프트웨어 프리페칭은 소스 코드에 특수 명령어를 명시적으로 삽입해야 한다. x86/x86-64에서 인텔의 컴파일러 인트린식 관례:

```c
#include <xmmintrin.h>
enum _mm_hint {
   _MM_HINT_T0 = 3,
   _MM_HINT_T1 = 2,
   _MM_HINT_T2 = 1,
   _MM_HINT_NTA = 0
};
void _mm_prefetch(void *p, enum _mm_hint h);
```

대부분의 x86/x86-64 프로세서는 잘못된 포인터로 인한 오류를 무시하므로 프로그래머 입장에서 다루기 쉽다. 힌트의 정확한 의미는 구현마다 조금씩 다르지만 대체로: `_MM_HINT_T0`은 (포함적 캐시에서는) 모든 레벨로, (배타적 캐시에서는) 최하위 레벨로 데이터를 가져온다. `_MM_HINT_T1`은 L2까지만(L1d 제외), `_MM_HINT_T2`는 L3가 있다면 비슷하게 동작한다. `_MM_HINT_NTA`(non-temporal aligned)는 6.1절에서 다룬 비시간적 힌트로, 짧게만 쓰일 데이터라 캐시 오염을 최대한 피하라고 지시한다(포함적 캐시 구현에서 하위 캐시로의 적재를 건너뛰고, 축출 시에도 상위 캐시를 거치지 않고 메모리로 직접 씀).

**측정 실험**: 리스트 원소마다 약 160사이클을 실제로 소비하는 포인터 체이싱 테스트(랜덤 배치)에서, 마지막 레벨 캐시(이 머신은 L2 512KB)를 넘는 작업 집합에서 프리페치가 50~60사이클, 최대 8%를 절약한다(원소마다 두 캐시 라인, NPAD=31 기준, 5개 원소 앞서 프리페치). 전체 페널티를 다 가리지는 못해도 확실히 도움이 된다. AMD 패밀리 10h Opteron의 `prefetchw` 명령어는 캐시 라인을 즉시 'M'(Modified) 상태로 프리페치해, 이후 쓰기가 뒤따를 경우(특히 경합이 있는 캐시 라인에서) 상태 전이 비용을 줄여준다.

프리페치는 이보다 큰 이득을 줄 수도 있지만 제대로 하기가 까다롭다. 성능 카운터(하드웨어/소프트웨어 프리페치, 유효 소프트웨어 프리페치, 각 레벨 캐시 미스 등)를 활용해 캐시 미스가 큰 곳을 찾고, 한 번에 한 곳씩 프리페치 명령을 추가한 뒤 "유효한 프리페치" 카운터가 실제로 늘어나는지 확인하며 진행해야 한다. gcc는 배열을 순회하는 루프에 `-fprefetch-loop-arrays`로 자동 프리페치를 넣어줄 수 있지만, 작은 배열이거나 크기를 컴파일 타임에 모르면 오히려 나빠질 수 있다고 매뉴얼이 경고한다.

#### 6.3.3 특수한 형태의 프리페치: 추측 실행

OOO 실행은 서로 충돌하지 않는 명령어를 재배치할 수 있다. IA-64의 **추측 로드**(speculative load)는 이 아이디어를 확장한다 — `ld8.a`(추측 로드)로 미리 로드를 시작해 두고, 나중에 `ld8.c.clr`로 (그 사이 값이 덮어써지지 않았다면) 이미 로드된 값을 그대로 쓰거나, 충돌이 있었다면 재로드한다. 프리페칭과 사실상 동등한 효과를 내면서, 값을 캐시 라인이 아니라 레지스터로 직접 가져온다는 (경우에 따라 큰) 이점이 있다(스레드가 스케줄에서 밀려나는 등으로 캐시 라인이 다시 축출될 걱정이 없음). 레지스터가 적은 프로세서에서는 큰 의미가 없을 수 있지만, 지원된다면 활용할 가치가 있다.

#### 6.3.4 헬퍼 스레드

소프트웨어 프리페칭을 시도하면 코드 복잡도 문제에 자주 부딪힌다 — 한 루프 안에 정상 순회와 프리페치용 순회 두 개를 함께 구현해야 하고, 얼마나 앞서 내다볼지(너무 적으면 늦고 너무 많으면 이미 축출됨) 정해야 하며, 프리페치 명령어 자체의 디코딩 비용과 루프 코드 크기 증가(L1i 효율 저하)도 감수해야 한다.

대안은 정상 연산과 프리페치를 완전히 분리된 **두 개의 스레드**로 만드는 것이다. 두 스레드를 같은 코어의 하이퍼스레드로 스케줄하면 프리페치가 L2(심지어 L1d)까지 채워줄 수 있다. 하이퍼스레드는 독립적인 코드를 실행할 때 캐시 공유가 문제였지만, 한 스레드가 순수 프리페치 헬퍼 역할이라면 오히려 바람직한 효과다 — 최하위 레벨 캐시가 미리 채워지고, 헬퍼 스레드가 대부분 유휴 상태이거나 메모리를 기다리므로 다른 하이퍼스레드의 정상 동작을 크게 방해하지 않는다.

원문 실험(그림 6.8, NPAD=31, 헬퍼 스레드가 리스트를 약 100원소 앞서가며 각 원소의 캐시 라인을 실제로 읽음): 작업 집합이 L2에 들어가는 구간에서는 헬퍼 스레드가 시스템 자원만 쓰고 기여는 없어 10~60% 성능이 나빠지지만, L2를 넘어서면 헬퍼 스레드가 실행 시간을 약 25% 줄여준다 — 메인 스레드의 산술 연산과 헬퍼 스레드의 메모리 로드가 서로 보완적이라 자원 충돌이 최소화되는 시너지 효과다. 어느 CPU들이 하이퍼스레드 관계인지는 libNUMA의 `NUMA_cpu_level_mask` 인터페이스(부록 D)로 확인할 수 있고, 스레드 어피니티(6.4.3절)로 두 스레드를 같은 코어의 하이퍼스레드에 고정한다. 헬퍼 스레드가 너무 앞서가지 않도록 futex나 POSIX 스레드 동기화로 제어해야 한다.

#### 6.3.5 다이렉트 캐시 액세스(DCA)

네트워크 인터페이스(NIC)나 디스크 컨트롤러 같은 하드웨어는 CPU 개입 없이 DMA로 메모리에 직접 쓴다. 이 경우 프로세서는 언제 데이터가 도착할지, 어디에 저장될지 몰라 프리페치할 방법이 없어 패킷 헤더를 읽을 때 캐시 미스가 난다. 인텔은 칩셋·CPU에 **DCA(Direct Cache Access)** 기술을 추가해 이를 완화한다 — NIC가 DMA 전송을 시작할 때 패킷 헤더가 프로세서 캐시로 들어가야 한다는 정보를 함께 보내면, 노스브리지가 FSB로 데이터를 전송할 때 특수 DCA 플래그를 함께 보낸다. 프로세서는 항상 FSB를 스누핑하다가 이 플래그를 인식하면 해당 데이터를 최하위 캐시로 적재하려 시도한다(힌트일 뿐이므로 무시해도 무방). 초당 수만 개의 패킷을 처리하는 상황에서, 패킷당 수백 사이클의 절감이 누적되면 특히 지연 측면에서 상당한 이득이 된다. 다만 이 최적화는 I/O 하드웨어·칩셋·CPU가 함께 통합되어야 가능하므로, 필요하다면 플랫폼 선택에 신중해야 한다.

### 6.4 멀티스레드 최적화

멀티스레딩에서 캐시 사용의 세 측면이 중요하다: **동시성(concurrency)**, **원자성(atomicity)**, **대역폭(bandwidth)**. 멀티프로세스에도 적용되지만 프로세스는 대체로 독립적이라 최적화 여지가 좁으므로 여기서는 멀티스레드에 집중한다.

#### 6.4.1 동시성 최적화

멀티스레드 애플리케이션이 공통 데이터를 쓸 때, 일반적인 캐시 최적화(풋프린트를 작게 유지)와 상충하는 문제가 있다. 여러 스레드가 같은 메모리 위치에 쓰면 각 코어의 L1d에서 그 캐시 라인이 'E'(Exclusive) 상태여야 하므로 RFO 메시지가 대량 발생한다 — 최악의 경우 쓰기 접근마다 하나씩. 이는 스레드들이 서로 다른(독립적으로 보이는) 메모리 위치를 쓸 때도 나타날 수 있는데, 이것이 **거짓 공유**(false sharing)다.

원문 실험(그림 6.10, 4개 P4 프로세서, 스레드마다 메모리 위치를 5억 번 증가): 스레드별로 별도 캐시 라인을 쓰면 스레드 수에 따라 페널티 없이 잘 스케일되지만, 모든 스레드가 같은 캐시 라인의 위치를 쓰면 오버헤드가 스레드 2/3/4개일 때 각각 390%, 734%, 1,147%에 달한다 — 캐시 라인이 쓰기를 마친 프로세서에서 곧바로 다른 프로세서로 넘어가야 하므로, 그 순간 캐시 라인을 갖지 못한 모든 프로세서가 지연되기 때문이다. 흥미롭게도 쿼드코어 단일 프로세서(Intel Core 2 QX6700, 별도 L2 두 개)에서는 이런 스케일링 문제가 나타나지 않는다(코어 수가 늘어도 오버헤드가 증가하지 않음) — 다만 여러 프로세서를 쓰는 SMP 머신은 여전히 흔하므로 실제 SMP 하드웨어에서 검증이 필요하다.

가장 단순한 "해결책"은 모든 변수를 각자 캐시 라인에 두는 것이지만, 이는 애플리케이션 풋프린트를 크게 늘려 앞서 말한 최적화와 충돌한다. 더 영리한 접근은 변수를 용도별로 구분하는 것이다:

1. **읽기 전용(초기화 후 불변) 변수**: RFO는 쓰기에만 필요하므로 상수는 캐시에서 'S'(Shared)로 자유롭게 공유될 수 있다 — 특별한 처리 없이 함께 묶어도 된다. `const`로 올바르게 표시하면 툴체인이 자동으로 `.rodata`/`.data.rel.ro` 섹션으로 옮겨준다. `const`로 표시할 수 없는 경우 특수 섹션 지정으로 배치를 통제할 수 있다(예: `__attribute__((section(".data.ro")))`).
2. **한 스레드만 쓰는 변수**: `__thread` 키워드로 스레드-로컬 변수(TLS)로 선언한다. 거짓 공유가 근본적으로 발생하지 않지만, 스레드 생성 시 TLS 설정 비용과 TLS 변수 주소 지정 비용이 더 크다는 대가가 있고, 스레드 간 값 이전이 자동이 아니며, DSO에서 TLS 변수 하나만 써도 그 DSO의 모든 TLS 변수 메모리가 할당되는 등 자원 낭비 가능성도 있다.
3. **여러 스레드가 자주 함께 쓰는 변수**: 구조체로 묶어 같은 캐시 라인에 두되, 캐시 라인 크기에 맞춰 패딩을 추가해 다른 무관한(자주 쓰이는) 변수와 한 라인을 나눠 쓰지 않게 한다.

일반 권장사항 정리: (1) 최소한 읽기 전용/읽기-쓰기(필요하면 "주로 읽기"까지 세 번째 범주로)를 분리한다, (2) 함께 쓰이는 읽기-쓰기 변수는 구조체로 묶는다, (3) 여러 스레드가 자주 쓰는 읽기-쓰기 변수는 전용 캐시 라인에 둔다(패딩 포함), (4) 스레드마다 독립적으로만 쓰이는 변수는 TLS로 옮긴다.

**원자적 증가 구현 비교**: gcc 내장 원자 연산(`__sync_*`)으로 원자적 증가를 구현하는 세 가지 방식(그림 6.12: `exchange add`, `add and fetch`, CAS 루프)을 4개 스레드가 각각 100만 번 증가시키는 벤치마크로 비교하면(x86/x86-64):

| Exchange Add | Add Fetch | CAS |
|---|---|---|
| 0.23s | 0.21s | 0.73s |

CAS 방식이 훨씬 느리다 — 메모리 연산이 두 번이고, CAS 자체가 조건부 연산이라 더 복잡하며, 동시 접근으로 CAS가 실패하면 루프를 반복해야 한다. CAS가 여러 아키텍처에서 통용되는 원자 연산이라는 이유로 다른 원자 연산 전부를 CAS로 통일해 구현하려는 유혹이 있지만(실제로 그렇게 하는 경우가 있지만), 이 수치가 보여주듯 결과가 최적과 거리가 멀 수 있다. 두 스레드가 각자 코어에서 도는 짧은 실행 구간만 봐도, 캐시 라인 상태가 최소 세 번 바뀌고 그중 둘은 RFO다(게다가 두 번째 CAS는 실패해 전체 연산을 반복해야 함). 반면 원자적 산술 연산은 프로세서가 로드-스토어를 하나로 묶어 그 사이의 캐시 라인 요청을 블록할 수 있어, 루프 반복마다 최대 RFO 하나로 끝난다. 요컨대 원자적 산술·논리 연산을 활용할 수 있는 수준에서 머신 추상화를 정의하는 것이 중요하며, CAS를 통일 메커니즘으로 무비판적으로 쓰지 말아야 한다.

x86/x86-64는 같은 명령어를 원자적/비원자적 양쪽으로 다 쓸 수 있다는 점이 독특하다 — `lock` 접두사를 붙이면 원자적이 된다. 이는 원자성이 필요 없는 상황에서 비용을 피할 문을 열어준다(라이브러리 코드처럼 필요할 때만 스레드 안전해야 하는 경우에 유용). 트릭은 `lock` 접두사를 건너뛰는 것이다:

```asm
cmpl $0, multiple_threads
je   1f
lock
1:   add $1, some_var
```

`multiple_threads`가 0이면(단일 스레드) `lock` 없이 실행되고, 아니면 `lock`과 함께 실행된다. 조건부 점프 추가가 얼핏 역효과처럼 보이지만(분기 예측이 틀리면 비용), 원자 연산 자체가 200사이클 안팎이라 손익분기점이 상당히 낮다 — 단일 스레드로 도는 경우가 많다면 이 기법이 확실히 빠르다. 다만 이 기법은 gcc의 `__sync_*` 내장 함수로는 표현할 수 없어 인라인 어셈블리가 필요하다.

#### 6.4.2 원자성 관련 배경

여러 스레드가 동시에 같은 메모리 위치를 수정하면 프로세서는 특정 결과를 보장하지 않는다 — 99.999%의 경우에 불필요한 비용을 피하기 위한 의도적 설계다. 동시 접근이 실제로 필요한 상황을 위해 프로세서는 원자 연산을 제공한다. 주요 범주:

- **비트 테스트**: 비트를 원자적으로 설정/해제하고 이전 값을 반환.
- **LL/SC(Load Lock/Store Conditional)**: 특수 로드로 트랜잭션을 시작하고, 그사이 값이 바뀌지 않았을 때만 최종 저장이 성공한다.
- **CAS(Compare-and-Swap)**: 현재 값이 주어진 값과 같을 때만 새 값을 쓰는 삼항 연산.
- **원자적 산술/논리 연산**: x86/x86-64에서만 제공(RISC 아키텍처는 미제공).

한 아키텍처는 LL/SC와 CAS 중 하나만 지원하며 둘은 표현력이 대체로 동등하다(CAS가 최근 선호되는 방식). 대부분의 프로세서에서 원자 연산은 그 자체로 항상 원자적이며, 이를 피하려면 원자성이 필요 없는 경우를 위한 완전히 별도의 코드 경로를 둬야 한다(코드 증가, 조건문, 추가 분기 필요) — 앞서 본 x86/x86-64의 `lock` 접두사 스킵 트릭이 이 비용을 피하는 유일한 예외적 수단이다.

#### 6.4.3 대역폭 고려사항

캐시 라인 경합이 없어도, 각 프로세서에는 코어·하이퍼스레드가 공유하는 메모리 대역폭 상한이 있다. 코어가 완벽한 조건에서 최고 속도로 돌아도 메모리 연결이 모든 로드/스토어 요청을 대기 없이 처리할 수 없는 시점에서, 이를 노스브리지 연결을 공유하는 코어·하이퍼스레드·프로세서 수만큼 나누면 병렬성 자체가 문제가 된다. 3.32절(FSB 속도 영향)에서 봤듯 FSB 속도를 올리는 것이 도움이 되고, 코어 수가 늘수록 FSB 속도도 함께 오를 것으로 예상되지만, 작업 집합이 크고 충분히 최적화된 프로그램이라면 결코 충분하지 않을 수 있다. Core 2의 `NUS_BNR_DRV` 같은 성능 카운터로 버스가 준비되지 않아 대기하는 사이클 수(즉 버스 혼잡도)를 관측할 수 있다 — 버스 사용률이 이미 1.0에 근접했다면 확장성 여지는 거의 없다.

대역폭 문제가 확인되면, 캐시 미스를 줄이는 코드 최적화 외에 남은 방법은 **스레드를 사용 가능한 코어에 더 잘 배치**하는 것이다. 기본적으로 커널 스케줄러는 스레드가 도는 데이터에 대한 통찰이 전혀 없다. 그림 6.13/6.14가 보여주는 대표적 비효율은, 서로 다른 프로세서(캐시 도메인)에서 도는 두 스레드가 같은 데이터셋을 쓰는 경우다 — 각 데이터셋을 메모리에서 두 번씩 읽어야 한다. 같은 데이터를 쓰는 스레드들을 같은 캐시 도메인에 모으면 데이터셋을 한 번만 읽으면 된다.

이를 프로그래머가 통제하는 유일한 커널 인터페이스가 **스레드 어피니티**다. 관련 인터페이스(모두 `<sched.h>`, `_GNU_SOURCE` 필요):

```c
int sched_setaffinity(pid_t pid, size_t size, const cpu_set_t *cpuset);
int sched_getaffinity(pid_t pid, size_t size, cpu_set_t *cpuset);
```

`cpu_set_t`와 이를 다루는 매크로들(`CPU_ZERO`, `CPU_SET`, `CPU_CLR`, `CPU_ISSET`, `CPU_COUNT`, 동적 크기용 `CPU_*_S` 계열, 논리 연산용 `CPU_EQUAL`/`CPU_AND`/`CPU_OR`/`CPU_XOR`)도 함께 제공된다. 멀티스레드 프로그램에서는(POSIX 프로세스 ID 개념이 스레드에는 없으므로) `<pthread.h>`의 `pthread_setaffinity_np`/`pthread_getaffinity_np`(스레드 핸들 기반), 그리고 스레드 생성 전에 미리 어피니티를 지정할 수 있는 `pthread_attr_setaffinity_np`/`pthread_attr_getaffinity_np`를 쓴다. 현재 실행 중인 CPU는 `sched_getcpu()`로 확인할 수 있지만 결과 반환 시점과 사용자 레벨 복귀 시점 사이에 스레드가 이동했을 수 있어 100% 정확하다고 가정할 수 없다.

반대 상황(두 스레드의 작업 집합이 겹치지 않는 경우)도 있다 — 이때는 같은 코어에서 돌면 서로의 캐시 활용을 갉아먹고 대역폭도 절반씩 나뉘므로, 오히려 같은 코어에 스케줄되지 않도록 어피니티를 설정해야 한다. 즉 최적화하려는 상황이 캐시 공유가 도움이 되는 경우인지 방해가 되는 경우인지부터 판단해야 한다. 대역폭을 위한 캐시 공유 최적화는 사실상 NUMA 프로그래밍의 한 측면이며(다음 절), "메모리"의 개념을 캐시까지 확장해서 봐야 한다. 초기 듀얼코어는 L2 공유가 없었고, Core 2 QX6700/QX6800 쿼드코어는 별도 L2 두 쌍을 쓰며, 코어 수가 늘수록 캐시 계층은 더 늘어날 전망이다 — 캐시들이 자체 계층을 이루므로 코어에 스레드를 배치하는 문제는 NUMA가 다루는 문제와 통합해서 볼 수 있다.

### 6.5 NUMA 프로그래밍

지금까지의 캐시 최적화는 NUMA에도 그대로 적용된다. 차이는 그 아래 수준에서 시작된다 — NUMA는 주소 공간의 부분마다 접근 비용이 다르다는 것을 도입한다. 균일 메모리 접근(UMA) 환경에서는 페이지 폴트 최소화(7.5절) 정도가 최적화의 전부이지만(모든 페이지가 동등하다), NUMA는 페이지의 메모리 지역성 최적화 중요성을 더한다. 인텔 CSI(x86/x86-64/IA-64용)와 AMD Opteron 모두 NUMA를 쓰므로 대부분의 SMP 머신에서 NUMA는 피할 수 없다. 코어당 코어 수가 늘면서 SMP 시스템 자체의 사용은 줄어들 수 있지만(데이터센터나 CPU 사용량이 매우 높은 환경 외에는), 그렇다고 프로그래머가 NUMA를 무시해도 된다는 뜻은 아니다 — NUMA 개념은 프로세서 캐시로도 자연스럽게 확장된다. 같은 캐시를 쓰는 코어의 두 스레드는 캐시를 공유하지 않는 코어의 두 스레드보다 더 빠르게 협업한다(초기 듀얼코어의 L2 미공유, Core 2 QX6700/6800의 별도 L2 두 개가 실제 사례). 캐시도 자체 계층을 이루므로, 코어에 스레드를 배치하는 문제는 NUMA가 다루는 문제와 통합해서 다룰 수 있다 — SMP가 아닌 머신만 쓰는 사람도 이 절을 읽을 가치가 있는 이유다.

리눅스 커널의 NUMA 관련 `sysfs` 정보(5.3절)를 직접 다루기는 번거로워서(원문 시점 기준 리눅스의 `libnuma`는 이 목적에 부족했고, 저자가 더 나은 버전을 준비 중이라고 언급), `<numaif.h>`의 저수준 시스템 콜 인터페이스가 제공된다:

- `mbind`: 지정한 메모리 페이지의 바인딩 선택
- `set_mempolicy` / `get_mempolicy`: 기본 메모리 바인딩 정책 설정/조회
- `migrate_pages`: 프로세스의 모든 페이지를 다른 노드 집합으로 마이그레이션
- `move_pages`: 선택한 페이지를 특정 노드로 이동하거나 페이지의 노드 정보 조회

#### 6.5.1 메모리 정책

메모리 정책의 목적은 기존 코드가 큰 수정 없이도 NUMA 환경에서 어느 정도 잘 동작하게 하는 것이다. 정책은 자식 프로세스에 상속되어 `numactl` 도구로 특정 정책을 주고 프로그램을 시작할 수 있다. 리눅스가 지원하는 정책:

- `MPOL_BIND`: 지정된 노드 집합에서만 할당(불가능하면 실패).
- `MPOL_PREFERRED`: 지정된 노드 집합을 우선 시도하되, 실패하면 다른 노드도 고려.
- `MPOL_INTERLEAVE`: 지정된 노드들에서 균등하게 할당(VMA 기반 정책은 가상 메모리 영역 내 오프셋으로, 태스크 기반 정책은 프리러닝 카운터로 노드 선택).
- `MPOL_DEFAULT`: 해당 영역의 기본값에 따라 할당.

정책은 계층을 이룬다(VMA 정책 > 태스크 정책 > 시스템 기본 정책 순으로 적용, 공유 메모리 세그먼트는 별도의 ShMem 정책). 시스템 기본값은 메모리를 요청한 스레드에 로컬로 할당하는 것이고(멀티스레드 프로세스라면 최초 실행 스레드의 노드가 "홈" 노드), 앞의 시스템 콜들로 다른 정책을 선택할 수 있다.

#### 6.5.2 정책 지정

`set_mempolicy(int mode, unsigned long *nodemask, unsigned long maxnode)`는 현재 스레드(태스크)의 정책만 바꾼다(전체 프로세스가 아님). `mode`가 `MPOL_DEFAULT`면 `nodemask`는 무시되고, `MPOL_PREFERRED`에 널 포인터를 주면 로컬 노드가 선택되며 그렇지 않으면 `nodemask`에서 가장 낮은 노드 번호가 쓰인다. 정책 변경은 **이미 할당된 메모리에는 영향을 주지 않는다** — 페이지가 자동으로 마이그레이션되지 않고 향후 할당에만 적용된다. `mmap`으로 예약한 주소 공간 영역은 보통 그 시점에 자동으로 할당되지 않고 첫 읽기/쓰기에서 실제 페이지가 할당되므로, 접근 사이에 정책이 바뀌면 겉보기에 균일한 주소 공간 영역이 여러 메모리 노드에 흩어질 수 있다는 점을 유의해야 한다.

#### 6.5.3 스와핑과 정책

물리 메모리가 부족해지면 시스템은 깨끗한 페이지는 버리고 더티 페이지는 스왑에 저장한다. 리눅스 스왑 구현은 페이지를 스왑에 쓸 때 노드 정보를 버리므로, 나중에 다시 페이지인될 때 노드는 처음부터 다시 정해진다(대개 실행 중인 프로세서에 가까운 노드가 되겠지만 이전과 다를 수 있음). 이는 노드 연관 정보를 페이지의 영속적 속성으로 프로그램이 저장해선 안 된다는 뜻이다 — 다른 프로세스와 공유되는 페이지라면 (`mbind` 등을 통한) 명시적 요청이나 커널의 자체 판단(한 노드의 공간이 부족할 때)으로도 연관이 바뀔 수 있다. 정확한 정보가 필요하면 항상 `get_mempolicy`(6.5.5절)를 써야 한다.

#### 6.5.4 VMA 정책

주소 범위에 VMA 정책을 지정하려면 `mbind(void *start, unsigned long len, int mode, unsigned long *nodemask, unsigned long maxnode, unsigned flags)`를 쓴다. `start`는 페이지 정렬이어야 하고 `len`은 다음 페이지 크기로 반올림된다. `flags`가 0이면 기본 동작(기존 매핑에는 영향 없이 VMA 정책만 설정)이고, 다음 플래그로 동작을 바꿀 수 있다:

- `MPOL_MF_STRICT`: 주소 범위의 모든 페이지가 `nodemask`가 지정한 노드에 있지 않으면 호출 실패(변경은 없음, 확인 용도).
- `MPOL_MF_MOVE`: 지정 노드 집합 밖에 할당된 페이지를 이동 시도(기본적으로 현재 프로세스의 페이지 테이블만으로 참조되는 페이지만 대상).
- `MPOL_MF_MOVEALL`: `MOVE`와 같지만 현재 프로세스뿐 아니라 모든 페이지를 이동 시도 — 다른 프로세스(심지어 다른 사용자 소유)의 메모리 접근에까지 영향을 줄 수 있어 특권 연산(`CAP_NICE` 필요)이다. `MOVE`/`MOVEALL` 지원은 리눅스 2.6.16부터다.

새로 예약한 주소 범위에 아직 페이지가 할당되기 전이라면 플래그 없이 `mbind`를 호출하는 것만으로 충분하다(향후 할당에 그대로 적용됨).

#### 6.5.5 노드 정보 조회

`get_mempolicy(int *policy, const unsigned long *nmask, unsigned long maxnode, void *addr, int flags)`로 주어진 주소의 정책 상태를 다양하게 조회할 수 있다. `flags`가 0이면 `addr`에 적용되는 정책(VMA 정책 우선, 없으면 태스크·시스템 기본 정책)과 그 노드마스크를 얻는다. `MPOL_F_NODE`를 설정하고 정책이 `MPOL_INTERLEAVE`라면 다음 할당이 일어날 노드 인덱스를 얻을 수 있어(새로 할당될 메모리에서 작업할 스레드의 어피니티를 미리 정하는 데 유용), `MPOL_F_ADDR`를 설정하면 `addr`을 포함하는 페이지가 실제로 할당된 메모리 노드 인덱스를 얻는다(페이지 마이그레이션 결정 등에 활용).

스레드가 도는 CPU(따라서 메모리 노드)는 메모리 할당보다 훨씬 변동성이 크다(명시적 요청 없이는 극단적 상황에서만 마이그레이션되지만, CPU 부하 재분배로 다른 CPU에 배정될 수는 있음). libNUMA는 주어진 가상 주소 범위의 노드 정보를 조회하는 `NUMA_mem_get_node_idx`, `NUMA_mem_get_node_mask`, 그리고 CPU 집합과 메모리 노드 집합을 상호 변환하는 `NUMA_cpu_to_memnode`, `NUMA_memnode_to_cpu`를 제공해 `get_mempolicy`보다 다루기 쉽다.

#### 6.5.6 CPU·노드 집합

소스가 없거나 코드 수정이 지나치게 비쌀 때, 혹은 관리자가 사용자·프로세스의 자원 사용을 제한하려 할 때는 리눅스 커널의 **CPU 집합**(특수 파일시스템, `mount -t cpuset none /dev/cpuset`)을 쓴다. 이름과 달리 메모리 노드도 함께 다룬다. 새 디렉터리를 만들면 새 CPU 집합이 생기고 부모의 설정을 상속하며, `cpus`·`mems` 의사 파일로 CPU·메모리 노드를 지정할 수 있다. 프로세스가 특정 CPU 집합에 속하면 그 설정이 어피니티·메모리 정책 비트마스크의 마스크로 작동해, 프로세스는 자신이 속한 집합의 `cpus`/`mems`에 없는 CPU·노드를 선택할 수 없다. 프로세스 ID를 해당 집합의 `tasks` 파일에 쓰는 것만으로 이동이 가능해, CPU·메모리 노드가 많은 머신에서 특히 효율적인 방법이다.

#### 6.5.7 명시적 NUMA 최적화

모든 노드의 모든 스레드가 같은 메모리 영역에 접근해야 한다면 로컬 메모리·어피니티 규칙만으로는 부족하다. 데이터가 **읽기 전용**이라면 해법은 간단하다 — 노드마다 복제본을 두어 노드 간 접근 자체를 없애는 것이다:

```c
void *local_data(void) {
  static void *data[NNODES];
  int node = NUMA_memnode_self_current_idx();
  if (node == -1) node = 0;
  if (data[node] == NULL) data[node] = allocate_data();
  return data[node];
}
void worker(void) {
  void *data = local_data();
  for (...) compute using data
}
```

`NUMA_memnode_system_count`(libNUMA)로 노드 수를 얻어 배열 크기를 정한다. `getcpu` 호출 이후 스레드가 다른 메모리 노드에 연결된 CPU로 옮겨져도 큰 문제는 아니다 — `worker`가 다른 노드의 메모리에 접근하게 되어 느려질 뿐이고, 커널은 근거 없는 재조정을 피하려 하므로 이런 전이는 드물고 당분간 반복되지 않는다.

**쓰기 가능한** 메모리 영역은 단순 복제가 안 통한다. 결과를 누적하는 용도라면, 노드마다 별도 영역에서 누적한 뒤 작업이 끝나면 합산하는 방식이 가능하다(누적이 상태 비의존적이어야 함). 쓰기 가능 영역에 대한 접근이 상당히 많다면, 커널에 페이지를 로컬 노드로 마이그레이션하도록 강제하는 것도 방법이다(단, 마이그레이션은 복사 연산이라 저렴하지 않으므로 이 비용이 상각될 만큼 접근이 많아야 함).

#### 6.5.8 전체 대역폭 활용

그림 5.4(원격 메모리, 1홉 거리) 수치가 보여주듯, 캐시가 무력화된 상황에서 원격 메모리 접근은 로컬 메모리 접근보다 측정 가능할 만큼 느리지 않을 수 있다. 이는 곧 로컬 메모리로의 대역폭을 아끼기 위해, 다시 읽을 필요 없는 데이터를 다른 프로세서에 붙은 메모리에 쓰는 전략이 가능하다는 뜻이다 — DRAM 모듈 연결의 대역폭과 인터커넥트 대역폭이 대체로 독립적이므로 병렬로 쓰면 전체 성능이 향상될 수 있다. 다만 이것이 실제로 가능한지는 여러 요인에 달려 있다: 캐시가 정말 무력화된 상태인지 확실히 해야 하고(그렇지 않으면 원격 접근 손해가 측정 가능해짐), 원격 노드 자신도 메모리 대역폭이 필요할 수 있다는 점을 미리 자세히 검토해야 한다. 이론적으로는 최대 4개의 다른 프로세서와 직접 연결될 수 있는 AMD 패밀리 10h Opteron에서, 이 추가 대역폭 전체를 (적절한 프리페치, 특히 `prefetchw`와 함께) 활용하면 시스템의 나머지 부분이 협조하는 한 개선을 얻을 수 있다.
