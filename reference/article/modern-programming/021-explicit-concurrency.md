# 명시적 동시성 (정리 노트)

> 원문: [Explicit concurrency](https://elizarov.medium.com/explicit-concurrency-67a8e8fd9b25)
> 저자: Roman Elizarov | 정리일: 2026-07-10
>
> 저작권이 있는 글이므로 전문 번역 대신 섹션 구조를 따라 내용을 재서술했다. 원문 표현 그대로가 필요하면 원문을 참고할 것.

## 도입: 동시성은 필수, 그러나 사고방식은 순차적

현대 애플리케이션(모바일, 웹, 서버)은 동시성 없이 성립하지 않는다. UI는 네트워크를 기다리는 동안에도 반응해야 하고, 서버는 수많은 요청을 동시에 처리해야 한다. 반면 병렬성(여러 CPU에서 동시에 계산)은 필수가 아닌 경우가 많다 — 동시성과 병렬성은 다른 개념이다.

문제는 프로그래머 대부분이 순차적 사고로 훈련되어 있고, 순차 코드가 추론하기도 훨씬 쉽다는 것이다. Kotlin 코루틴의 설계 철학은 이 현실을 받아들인다. **기본값은 순차 실행이고, 동시 실행은 명시적으로 요청해야 한다.**

가장 단순한 순차 코드부터 보자.

```kotlin
foo(bar())
```

이 표현식은 실행 순서가 명확하다.

```kotlin
val x = bar()  // 먼저 bar를 호출하고
foo(x)         // 그다음 foo를 호출한다
```

중요한 것은 `bar`와 `foo`가 suspend 함수여도 이 순서가 그대로라는 점이다. suspend 함수는 스레드를 블로킹하지 않고 일시 중단할 뿐, 호출 관점에서는 여전히 "끝나야 다음으로 넘어가는" 순차 함수다. 즉 코루틴을 쓴다고 코드가 저절로 동시적이 되지 않는다. 이것이 코루틴 코드가 읽기 쉬운 이유다.

## 동시 실행은 빌더로 명시한다

`bar`와 `foo`를 백그라운드에서 동시에 실행하고 싶다면 코루틴 빌더로 명시적으로 감싼다.

```kotlin
launch {
    foo(bar())
}
```

`launch`는 코루틴을 시작만 하고 즉시 반환한다. 이 지점부터 코드는 동시적으로 흐른다. 그리고 구조적 동시성 덕분에 `launch`는 `CoroutineScope` 안에서만 호출할 수 있으므로, 이 동시 실행의 수명은 그 스코프에 묶인다.

이 패턴에 이름을 붙여 재사용하려면 어떻게 선언해야 할까? 원문이 제안하는 형태는 `CoroutineScope`의 확장 함수다.

```kotlin
fun CoroutineScope.launchFooBar() = launch {
    foo(bar())
}
```

## 핵심 관례: 시그니처가 실행 방식을 말한다

여기서 이 글의 중심 관례가 나온다.

> **`CoroutineScope`의 확장으로 선언된 함수는 즉시 반환하되, 동작은 호출자의 스코프 안에서 동시적으로 수행한다.**

suspend 함수의 관례(호출자 스레드를 블로킹하지 않되, 작업이 끝나야 반환한다)와 짝을 이루는 규칙이다. 두 관례를 합치면 함수 시그니처만 보고 실행 방식을 판별할 수 있다.

- `suspend fun obfuscate(data: Data)` — **순차**. 호출하면 난독화가 끝날 때까지 (스레드 블로킹 없이) 기다린 뒤 반환한다.
- `fun CoroutineScope.obfuscate(data: Data)` — **동시**. 즉시 반환하고, 난독화는 호출자의 스코프 안에서 백그라운드로 진행된다. 스코프가 취소되면 이 작업도 취소된다.

반대로 두 형태를 섞은 시그니처는 안티패턴이다.

```kotlin
suspend fun CoroutineScope.obfuscate(data: Data)
```

이 함수는 순차인지 동시인지 시그니처로 알 수 없다. suspend라서 기다렸다 반환할 것 같기도 하고, CoroutineScope 확장이라서 백그라운드 작업을 띄울 것 같기도 하다. 호출자는 문서를 읽거나 구현을 뜯어보기 전까지 추론할 수 없다. 원문의 처방: 이런 모호한 선언을 피하고 둘 중 하나를 명확히 선택하라. 그러면 시그니처 자체가 실행 방식의 문서가 된다.

기본 선택은 suspend 함수 쪽이다. 순차 실행이 추론하기 쉽고, 동시성이 필요한지는 호출자가 가장 잘 안다. 호출자는 필요할 때 언제든 `launch`나 `async`로 감싸 동시성을 얻을 수 있다. 반대로 함수가 제멋대로 동시 실행을 시작해 버리면 호출자가 그것을 되돌릴 방법이 없다.

## Global scope

동시 작업을 띄우는 손쉬운 우회로로 `GlobalScope.launch { ... }`가 있다. 함수가 스코프를 받지 않고도 내부에서 몰래 동시 작업을 시작할 수 있게 해 준다. 원문은 이를 백그라운드 스레드 풀에 작업을 던져 넣는 옛 습관에 비유하며 경고한다.

- 그렇게 띄운 작업은 아무도 추적하지 않는다. 취소할 수도, 완료를 기다릴 수도 없다.
- 참조가 살아 있는 동안 리소스와 메모리를 붙들고 있으므로 누수의 온상이 된다.
- 호출자는 함수가 뒤에서 무언가를 계속 돌리고 있다는 사실 자체를 모른다. 동시성이 암묵적이 된다.

동시 작업이 정말 필요하면 호출자의 스코프를 받아서(즉 `CoroutineScope` 확장 함수로 선언해서) 그 안에서 띄워야 한다. 그래야 작업의 수명이 호출자의 수명에 묶이고, 구조적 동시성이 유지되며, 시그니처에 동시성이 드러난다.

## 요약

- suspend 함수든 아니든, 그냥 호출하면 순차 실행이다. 이것이 기본값이자 가장 추론하기 쉬운 형태다.
- 동시 실행은 `launch`/`async` 같은 빌더로만, 스코프 안에서만 시작된다. 코드에 항상 드러난다.
- 함수 시그니처 관례: `suspend fun` = 끝날 때까지 기다리는 순차 함수, `fun CoroutineScope.` = 즉시 반환하는 동시 함수. 둘을 섞지 마라.
- 기본은 suspend 함수로 선언하고, 동시 실행이 의도된 경우에만 `CoroutineScope` 확장으로 선언하라. `GlobalScope`는 피하라.
