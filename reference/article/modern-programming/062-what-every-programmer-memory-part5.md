## 7장 메모리 성능 도구

프로그램의 성능 특성, 캐시·메모리 사용을 이해하도록 돕는 도구는 다양하다. 최신 프로세서는 성능 모니터링 하드웨어를 제공하고, 정확히 측정하기 어려운 이벤트는 시뮬레이션으로 보완한다. 여기서는 리눅스에서 흔히 쓰이는 도구 몇 가지를 소개한다.

### 7.1 메모리 연산 프로파일링

메모리 연산 프로파일링은 하드웨어의 협조가 필요하다. 소프트웨어만으로도 일부 정보를 모을 수 있지만 성기거나 시뮬레이션에 불과하다. 리눅스에서 성능 모니터링 하드웨어에 접근하는 대표 도구가 **oprofile**이다 — 시스템 전체를 대상으로 한 통계적 프로파일링을 쉬운 인터페이스로 제공한다(pfmon 같은 대안도 개발 중).

oprofile의 인터페이스는 단순하지만 상당히 저수준이다. 사용자는 프로세서가 기록할 수 있는 이벤트 중에서 골라야 하고, 이 이벤트들은 프로세서마다 다르며 해석에도 프로세서에 대한 상세한 지식이 필요하다. 절대값 하나만 봐서는 "얼마가 높은 수치인지" 판단하기 어렵다는 문제도 있다 — 그래서 흔히 두 카운터의 **비율**을 보는 방식을 쓴다(예: 클럭 사이클 대비 명령어 수).

**CPI(사이클/명령어) 측정**: `CPU_CLK_UNHALTED`(클럭 사이클)와 `INST_RETIRED`(완료된 명령어 수)의 비율을 랜덤 "Follow" 테스트의 작업 집합 크기별로 측정하면(그림 7.1), 리스트 원소당 사이클 측정과 비슷한 그림이 나온다 — 작은 작업 집합에서는 비율이 1.0 이하, L1d를 넘어서면 3.0 가까이, L2까지 넘어서면 20 이상으로 뛴다.

**캐시 미스 이벤트**: Core 2에서 주목할 이벤트는 `L1D_REPL`, `DTLB_MISSES`, `L2_LINES_IN`(하드웨어 프리페칭이 아닌 명령어에 의한 미스만 따로 셀 수도 있음)이다. 랜덤 Follow 테스트(그림 7.2)에서는 L1d 미스율이 32KB 부근에서 치솟기 시작하고(64KB까지는 하드웨어 프리페칭 덕에 1%로 유지되다가 그 이후 급증), L2 미스율은 L2 소진 시점(2^21바이트)부터 오르며, L2 디맨드 미스(하드웨어 프리페처가 못 채운 미스)가 0이 아니라는 점이 랜덤 접근에서 프리페칭이 완벽할 수 없음을 보여준다. 반면 순차 접근(그림 7.3)에서는 L2 디맨드 미스가 사실상 0이다 — 거의 모든 L2 미스가 프리페처에 의한 것이고, L1d·L2 미스율이 같다는 것은 L1d 미스가 지연 없이 L2에서 처리된다는, 사실상 이상적인 상황이다. 네 번째 지표인 DTLB 미스율은 랜덤 접근에서 상당한 지연 요인이 되지만(L2 미스보다 먼저 시작됨), 순차 접근에서는 거의 0이다.

6.2.1절의 행렬 곱셈 예제로 돌아가면, `SSE_HIT_PRE`, `SSE_PRE_MISS`, `LOAD_PRE_EXEC` 카운터로 소프트웨어 프리페칭의 효율을 볼 수 있다. 실제 측정에서 유효 NTA 프리페치 비율은 2.84%, 지연 NTA 프리페치 비율은 2.65%로 나왔는데 — 유효(제때 쓰인) 비율이 낮다는 것은 이미 로드되어 있던 캐시 라인에 불필요한 프리페치 명령을 실행해 디코딩·조회 시간만 낭비했다는 뜻이고, 지연 비율은 프리페치가 필요한 시점보다 늦게 도착했다는 뜻이다. 결국 5.5%만 유효했다는 것은 이 코드에 개선 여지(불필요한 프리페치 제거, 하드웨어 특성에 맞춘 프리페치 타이밍 조정)가 있음을 알려준다. `opannotate` 도구로 소스·어셈블리 코드에 이벤트가 어디서 발생했는지 표시할 수 있다(다만 oprofile은 확률적 샘플링이라 N번째 이벤트마다 기록하므로 일부 핫스팟이 누락될 수 있고, 비순차 실행 탓에 명령어 포인터가 정확하지 않을 수도 있다).

**페이지 폴트**: 하드웨어 지원 없이도 측정 가능한 이벤트다. OS가 페이지 폴트를 해결하며 집계하는데, **마이너 폴트**(익명 페이지, COW 페이지, 이미 메모리 어딘가에 있는 페이지)와 **메이저 폴트**(파일 백업이나 스왑에서 실제 디스크 접근이 필요)로 나뉜다. 둘 다 결코 저렴하지 않다(커널 진입, 새 페이지 확보, 초기화/데이터 채움, 페이지 테이블 수정, 필요시 다른 태스크와의 동기화가 필요). `time` 도구(셸 내장이 아닌 실제 `/usr/bin/time`)로 메이저·마이너 폴트 수를 바로 확인할 수 있고, 내부적으로는 `getrusage(2)`(`ru_minflt`, `ru_majflt`)를 쓰며 `/proc/<PID>/stat`의 10~14번째 필드로도 조회할 수 있다. 프로그램 시작 단계가 특히 페이지 폴트에 민감하다(사용하는 페이지마다 폴트가 발생하므로 GUI 앱의 체감 시작 시간에 큰 영향).

### 7.2 CPU 캐시 시뮬레이션

프로그래머는 절대·상대 주소 값을 직접 다루지 않으므로, 실제 프로그램의 캐시 사용 양상을 감으로 파악하기 어렵다. oprofile 같은 CPU 레벨 프로파일러는 실제 하드웨어에 대응하는 데이터를 빠르게 모을 수 있지만, 세밀한 데이터가 필요하거나(스레드를 너무 자주 인터럽트해야 함) 여러 종류의 프로세서에서 동작을 봐야 할 때(그림 3.8의 24개 조합처럼 실제로 존재하지 않는 머신 구성도 포함)는 한계가 있다.

이런 경우를 위한 도구가 **cachegrind**다 — 원래 메모리 처리 문제를 점검하려고 만든 **valgrind** 프레임워크 위에서 프로그램 실행을 시뮬레이션하며 모든 메모리 주소 사용을 가로채, 주어진 크기·라인 크기·연관도의 L1i·L1d·L2 캐시 동작을 시뮬레이션한다.

```
valgrind --tool=cachegrind command arg
```

기본으로는 실행 중인 프로세서의 실제 캐시 구성을 시뮬레이션하며, 종료 시 표준 오류로 전체 명령어·메모리 참조 수, L1i/L1d/L2의 미스 수·미스율 등을 요약해 보여준다(그림 7.5 예시 출력). `--I1`, `--D1`, `--L2` 파라미터로 실제 프로세서와 무관한 임의의 캐시 구성(예: `--L2=8388608,8,64`로 8MB, 8-way, 64바이트 라인)을 시뮬레이션할 수도 있어, 다른 캐시 구성에서 프로그램이 어떻게 동작할지 실험해 볼 수 있다.

프로세스 종료 시 `cachegrind.out.XXXXX` 파일에 함수·소스 파일별 상세 캐시 사용 정보가 저장되고, `cg_annotate` 도구로 이를 볼 수 있다(디버그 정보가 있으면 함수 단위 매칭이 더 정확). 출력(그림 7.6)은 함수별 캐시 히트/미스 수를 보여줘 L2 미스가 큰 곳부터 L1i/L1d 미스 순으로 최적화 우선순위를 정하는 데 쓸 수 있고, 소스 파일명을 주면 줄 단위 주석까지 가능하다.

다만 cachegrind는 어디까지나 **시뮬레이터**임을 유의해야 한다 — 실제 프로세서의 캐시 구현과 다를 수 있다(예: cachegrind는 LRU 축출을 시뮬레이션하지만 실제로 높은 연관도 캐시에서는 LRU가 너무 비싸 다른 정책을 쓸 수 있음). 또한 컨텍스트 스위치·시스템 콜로 인한 L1i/L1d 플러시나 L2 오염을 반영하지 않으므로, 시뮬레이션된 전체 캐시 미스 수는 실제보다 낮게 나온다. 그럼에도 프로그램의 메모리 사용 패턴과 문제점을 배우기에 좋은 도구다.

### 7.3 메모리 사용량 측정

프로그램이 얼마나 많은 메모리를 어디서 할당하는지 아는 것이 메모리 사용 최적화의 첫걸음이다. 재컴파일이나 코드 수정 없이도 쓸 수 있는 도구들이 있다.

**massif**(valgrind 기반)는 컴파일러가 자동 생성하는 디버그 정보만 있으면 시간에 따른 누적 메모리 사용 개요를 보여준다:

```
valgrind --tool=massif command arg
```

할당·해제 함수 호출을 인식해 호출 지점·타임스탬프와 함께 기록하고, 프로그램 전체와 호출 지점별 누적 크기에 반영한다. 종료 시 `massif.XXXXX.txt`(요약)와 `massif.XXXXX.ps`(호출 지점별 색으로 구분된 시간축 그래프, 그림 7.7)를 생성한다. 스택 사용량도 기록할 수 있지만(일부 상황에서는 스택 경계를 알 수 없어 부정확할 수 있어 `--stacks=no`로 끌 수 있음), 자체 메모리 할당기를 쓰거나 시스템 할당 함수를 감싸는 래퍼를 쓰는 프로그램은 `--alloc-fn=xmalloc` 같은 옵션으로 추가 할당 함수를 알려줘야 정확히 잡힌다.

그래프 해석: 같은 호출 지점에서 오랫동안 완만한 기울기로 계속 늘어나는 영역이 보인다면(원문 예시에서 800ms~1,800ms 구간) 상대적으로 작은 할당이 대량으로 일어난다는 뜻으로, GNU C 라이브러리의 **obstack**(큰 블록을 시스템 할당기에서 받아와 순차적으로 나눠주는 할당기, 완전한 malloc 대체는 아니고 해제 능력이 제한적) 같은 벌크 할당 기법을 검토할 후보다.

**memusage**(GNU C 라이브러리의 일부, massif보다 단순하지만 더 오래된 도구)는 힙(옵션으로 `mmap` 포함)과 스택의 총 메모리 사용만 기록하며, 실제 실행 코드에 직접 계측 코드를 심으므로(시뮬레이션이 아님) massif보다 훨씬 빠르다. `-p IMGFILE`로 PNG 그래프를 만들고, 할당 크기별 히스토그램도 표준 오류로 출력한다. `gcc`의 컴파일 단계처럼 드라이버 프로그램이 실행하는 서브프로세스를 관찰해야 할 때는 `-n NAME`으로 관찰 대상 프로그램 이름을 지정한다.

**메모리 레이아웃과 캐시의 관계**: 효율적인 동적 메모리 할당의 핵심은 **선형 할당**과 사용 영역의 **조밀함**이다 — 프리페칭 효율과 캐시 미스 감소로 직결되기 때문이다. 임의 크기 데이터를 처리하려고 연결 리스트로 하나씩 할당하면(단일 연결 리스트라면 오버헤드는 포인터 하나로 작더라도) 순차 할당이 메모리에서도 순차 배치된다는 보장이 없다(할당기 내부에서 뒤에서 앞으로 반환하거나, 청크가 소진돼 다른 영역에서 새로 시작하거나, 크기별로 다른 풀을 쓰거나, 멀티스레드 할당이 서로 뒤섞이는 등의 이유). 나중에 처리할 데이터를 미리 채워 넣어야 한다면 연결 리스트 방식은 피하고, 앞서 언급한 obstack 같은 별도 메모리 관리 계층을 써야 한다.

또한 malloc 구현이 만드는 관리용 헤더·패딩도 무시할 수 없다 — 예컨대 각 블록마다 최소 2워드 헤더(32비트 8바이트, 64비트 16바이트)가 붙고 블록 크기도 특정 배수로 반올림되므로, 작은 블록을 많이 할당하면 헤더·패딩이 실제 데이터 사이사이에 끼어들어 프로세서의 순차 프리페치 효율을 그만큼(예시에서는 최대 50%) 갉아먹을 수 있다. 이는 프로그래머가 할당을 직접 통제할 때만 피할 수 있는 비용이다.

### 7.4 분기 예측 개선

6.2.2절에서 L1i 활용을 위한 두 기법(`__builtin_expect` 기반 정적 예측, PGO/블록 재배치)을 봤다. 여기서는 그 메모리 사용 측면의 개선을 다룬다.

`likely`/`unlikely` 매크로는 쓰기 간단하지만, 프로그래머가 실수로 반대로 표시하기 쉽고 oprofile로도 이런 오류를 잡아내기 어렵다는 문제가 있다. 부록 A.2에 소개된 기법은 이 매크로를 런타임에 실제로 예측이 맞았는지 능동적으로 계측하는 대안 정의로 바꿔, 프로그래머가 직접 검토·조정할 수 있게 해준다.

**PGO**(Profile-Guided Optimization)는 gcc에서 세 단계로 진행된다: (1) 모든 소스 파일을 `-fprofile-generate`로 컴파일해(더 크고 느리지만 분기 실행 여부를 기록하는) 바이너리를 만들고 대표 워크로드로 여러 번 실행해 `.gcda` 파일들을 누적시킨다, (2) 이 데이터를 바탕으로 `-fprofile-use`로 재컴파일한다(소스 파일 위치를 옮기면 안 되고, 공백·주석 외의 변경은 데이터 무효화로 이어짐). 대표 테스트 워크로드 선정이 관건이다 — 실사용 패턴과 다르면 최적화가 오히려 역효과를 낼 수 있어, 사용 시나리오가 다양한 라이브러리에는 PGO보다 정적 `__builtin_expect`가 대체로 더 적합하다. `.gcno`/`.gcda`는 `gcov`(커버리지 분석 도구, 파일 포맷은 PGO와 동일)로도 조회할 수 있다.

### 7.5 페이지 폴트 최적화

리눅스처럼 데맨드 페이징을 지원하는 OS에서 `mmap` 호출은 페이지 테이블만 수정할 뿐 실제 메모리 할당은 첫 접근(읽기·쓰기·실행) 시 페이지 폴트를 통해 이뤄진다. 이 해결 과정은 절대 저렴하지 않고, 프로세스가 쓰는 모든 페이지마다 한 번씩 발생한다.

저자가 valgrind 기반으로 만든 **pagein** 도구는 페이지 폴트가 *왜*, 어떤 순서로 일어나는지 `pagein.<PID>` 파일에 기록한다(주소, 코드/데이터 구분, 첫 폴트 이후 경과 사이클, 가능하면 심볼명). 이 정보로 프로그램의 어떤 코드 시퀀스를 같은 페이지에 인접 배치해야 폴트를 줄이거나 늦출 수 있는지 파악할 수 있다. 정확한 최적 레이아웃을 찾는 것은 시행착오가 필요한 번거로운 과정이지만, 오브젝트 파일 단위의 호출 관계 분석이나 gcc의 `-finstrument-functions`(`__cyg_profile_func_enter`/`exit` 훅)를 이용한 실행 추적으로 근사할 수 있다 — 실제로 함수 재배치만으로 시작 비용을 5% 줄인 사례가 보고된 바 있다(TLB 미스 감소 효과도 있어, 가상화 환경에서는 TLB 미스 비용이 더 크므로 더욱 중요해진다).

리눅스 커널이 제공하는 두 가지 추가 메커니즘:

- **`MAP_POPULATE`** 플래그를 `mmap`에 주면 페이지 테이블 수정뿐 아니라 매핑된 영역 전체를 미리 폴트시킨다. 매핑 즉시 전부 쓰인다면 유리하지만, 매핑한 페이지 상당수가 당분간(또는 영원히) 쓰이지 않는다면 낭비다. 세분성이 너무 거칠다는 한계도 있다.
- **`posix_madvise`의 `POSIX_MADV_WILLNEED`** 힌트는 더 세밀하게(개별 페이지·페이지 범위 단위) 미리 폴트시킬 것을 커널에 제안한다(커널이 무시해도 됨) — 실제로는 쓰이지 않는 데이터가 많은 메모리 매핑 파일에서 `MAP_POPULATE`보다 유리할 수 있다.

**대용량 페이지(huge page)**: 페이지 크기를 키우면(대부분 아키텍처의 기본은 4KB, IA-64·PPC64는 64KB도 흔함) 같은 메모리를 커버하는 데 필요한 페이지 수, 따라서 페이지 폴트 수도 줄어든다. 원문 실험(그림 7.9, NPAD=0 랜덤 Follow, 4KB vs 2MB 페이지)에서 작업 집합 2^20바이트일 때 2MB 페이지 버전이 57% 빠르고, 512MB에서도 38% 빠르다 — 2MB 페이지용 TLB 항목 64개가 2^27바이트를 커버하는 지점까지는 평탄 구간이 유지된다. 이는 대용량 작업 집합에서 비용의 상당 부분이 TLB 미스에서 온다는 것을 보여준다.

대용량 페이지를 쓰려면 물리적으로 연속된 메모리가 필요한데, 시스템이 한동안 돌며 파편화되면 나중에 확보하기 어렵다 — 그래서 리눅스는 부팅 시점에 `/proc/sys/vm/nr_hugepages`로 미리 예약해야 하는 전용 `hugetlbfs` 파일시스템을 쓴다(가상화 환경에서는 VMM이 이를 지원해야 하는 추가 제약이 있음). 프로그램에서 대용량 페이지를 요청하는 방법은 두 가지: System V 공유 메모리의 `SHM_HUGETLB` 플래그, 또는 `hugetlbfs`를 마운트하고 그 안에 파일을 만들어 `mmap`하는 방식(후자가 키(key) 충돌 우려가 없어 더 권장됨). 파일 기반 매핑에는 아직 대용량 페이지를 자동으로 적용하는 방법이 없다(투명한 지원은 `mprotect` 등으로 매핑 일부의 보호 속성이 바뀔 때 발생하는 문제 때문에 구현이 까다로움).

## 8장 다가올 기술

멀티프로세서 처리를 다룬 앞 절들에서 CPU·코어 수가 늘수록 심각한 성능 문제가 생길 수 있음을 봤다. 하지만 바로 이 확장(코어 수 증가)이 앞으로 계속될 방향이다 — 단일 코어 성능이 예전만큼 빠르게 오르지 않으므로 프로그램은 갈수록 더 병렬화되어야 한다.

### 8.1 원자 연산의 문제

공유 자료구조 접근 동기화는 전통적으로 상호 배제(뮤텍스)나 **락-프리 자료구조**로 이뤄진다. 락-프리 자료구조의 문제는 프로세서가 제공하는 원자 연산 프리미티브가 제한적이라는 것 — 대부분의 아키텍처는 워드 단위 원자적 읽기·쓰기 정도만 지원하며(CAS 또는 LL/SC로 구현), x86/x86-64는 이례적으로 더 풍부한 원자 연산 집합, 특히 **DCAS**(double-word CAS)까지 지원한다.

원문은 배열 기반 락-프리 LIFO(스택) 구현을 세 단계로 발전시키며 문제를 짚는다:

1. **원자 연산 없는 버전**: 당연히 스레드 안전하지 않다 — 원소가 사라지거나 이미 제거된 원소가 되살아날 수 있다.
2. **단순 CAS 버전**(`push`/`pop`에서 top 포인터를 CAS로 갱신): 얼핏 동작할 것 같지만 **ABA 문제**에 취약하다 — 스레드 A가 pop을 시작한 사이 다른 스레드가 `pop → push(새 원소) → push(A가 읽었던 원소)` 순서로 조작하면, top 값 자체는 A가 봤던 값으로 되돌아와 A의 CAS는 성공하지만 실제로는 리스트의 두 번째 원소가 바뀌어 있어 새로 push된 원소가 유실된다.
3. **DCAS + 세대 카운터 버전**: top 포인터 옆에 매 연산마다 증가하는 세대 카운터를 함께 두고 이를 DCAS로 원자적으로 갱신하면, 값이 같아 보여도 세대 카운터가 다르므로 ABA 문제를 피할 수 있다.

다만 이 세 번째 해법도 만능은 아니다. `pop`에서 `old.top == NULL` 검사 이후 스레드가 인터럽트되고, 그사이 다른 스레드가 같은 원소를 pop해 메모리를 해제해 버리면, 첫 스레드가 재개될 때 이미 해제된 메모리를 역참조하다가 크래시할 수 있다 — 이를 막으려면 메모리를 절대 해제하지 않거나 참조 여부를 검증해야 하는데, 이는 락-프리 자료구조가 노리는 성능 이점을 상당 부분 상쇄한다(가비지 컬렉션이 있는 언어라면 이 문제를 해결할 수 있지만 그 나름의 비용이 있음). 더 복잡한 자료구조(예: FIFO)일수록 문제는 더 심해지고, 애초에 x86/x86-64의 CAS는 메모리에서 연속된 두 워드에만 적용되므로 이중 연결 리스트의 임의 위치 삽입·삭제 같은 흔한 상황에는 아예 도움이 안 된다. 근본 문제는 여러 메모리 주소가 동시에 관련되고, 그 값들이 동시에 바뀌지 않아야만 연산 전체가 성공하는 상황 — 데이터베이스에서 익숙한 개념이며, 실제로 가장 유망한 해법도 여기서 나온다.

### 8.2 트랜잭셔널 메모리

Herlihy와 Moss는 1993년 논문에서 메모리 연산의 트랜잭션을 소프트웨어만으로는 효율적으로 다룰 수 없으므로 **하드웨어에 구현**하자고 제안했다(당시 DEC가 수십 개 프로세서급 하이엔드 하드웨어에서 이미 비슷한 확장성 문제를 겪고 있었다). 원리는 데이터베이스 트랜잭션과 같다 — 트랜잭션의 결과는 한꺼번에 반영되거나, 실패하면 모든 값이 그대로 유지된다.

일부 프로세서의 LL/SC 명령어 쌍은 사실상 이미 작은 트랜잭션이다 — LL로 값을 읽어 L1d에 적재하고, 그사이 다른 프로세서가 그 메모리 위치를 바꾸면(MESI 프로토콜로 감지) SC가 실패한다. **트랜잭셔널 메모리**는 이 개념을 단순 쌍이 아니라 여러 개의 로드·스토어가 참여하는 트랜잭션으로 확장한 것이다.

**연산 종류**: 일반 읽기(LT), 나중에 쓸 것을 전제로 한 배타적 읽기(LTX), 쓰기(ST), 그리고 트랜잭션 상태를 확인하는 VALIDATE, 트랜잭션을 완료하는 COMMIT이 필요하다. MESI 관점에서 일반 읽기는 'E'·'S' 상태 라인으로 충족되지만, LTX는 'E' 상태 라인을 요구한다.

**예제 코드**(가상의 트랜잭셔널 메모리 프리미티브로 짠 락-프리 LIFO):

```c
struct elem *top;
void push(struct elem *n) {
  while (1) {
    n->c = LTX(top);
    ST(&top, n);
    if (COMMIT()) return;
    ... delay ...
  }
}
struct elem *pop(void) {
  while (1) {
    struct elem *res = LTX(top);
    if (VALIDATE()) {
      if (res != NULL) ST(&top, res->c);
      if (COMMIT()) return res;
    }
    ... delay ...
  }
}
```

이 코드는 스레드 안전하지 않은 원본 코드와 구조가 거의 같다는 점이 큰 이점이다. `pop`에서 `res`가 유효한지 역참조 전에 `VALIDATE()`로 확인하는 부분이, 원자 연산 버전에서 풀지 못했던 "해제된 메모리 역참조" 문제를 해결해 준다 — 트랜잭션이 이미 실패했다면 `VALIDATE`가 그 사실을 알려주고 역참조 자체가 일어나지 않기 때문이다.

**버스 프로토콜과 트랜잭션 캐시**: 트랜잭셔널 메모리는 별도 메모리가 아니라 L1d와 나란히 있는 전용 "트랜잭션 캐시"로 구현되는 것이 유력하다(L1d 자체에 넣는 것은 좋은 생각이 아니라고 원 논문은 지적한다). 이 캐시는 L1d보다 훨씬 작고(트랜잭션이 담을 수 있는 연산 수로 크기가 정해짐, 완전 연관일 가능성이 큼), L1d와 **배타적** 관계를 갖는다. 각 슬롯은 MESI 상태에 더해 EMPTY/NORMAL/XABORT/XCOMMIT라는 트랜잭션 상태를 갖는다 — XABORT는 트랜잭션 실패 시 버릴 데이터, XCOMMIT는 커밋 시 버릴(즉 롤백용 옛 값을 담은) 데이터다. 새 트랜잭션 읽기(T_READ)·배타적 읽기(T_RFO) 버스 요청은 일반 READ/RFO와 비슷하지만, 다른 프로세서가 같은 라인으로 활성 트랜잭션을 진행 중이면 그냥 실패 처리된다(값은 undefined로 취급되며, 이후 반드시 VALIDATE/COMMIT으로 확인해야 함).

이 설계의 핵심 이점은, **VALIDATE·COMMIT 자체가 버스 연산을 유발하지 않는다**는 것이다 — 원자 연산은 동시성을 보장하려고 결국 메인 메모리에 값을 써야 하지만(비용이 크다는 것은 이 문서 전체가 보여준 바), 트랜잭셔널 메모리는 새 트랜잭션을 시작할 때, 그리고 아직 트랜잭션 캐시에 없는 새 캐시 라인을 (성공 중인) 트랜잭션에 추가할 때만 버스 연산을 일으킨다. 실패한 트랜잭션 내부의 연산은 버스 연산을 일으키지 않으므로, 같은 메모리를 두고 여러 스레드가 경쟁해도 캐시 라인이 핑퐁하며 왔다 갔다 하는 일이 없다.

**기타 고려사항**: 6.4.2절에서 본 x86/x86-64의 `lock` 접두사 스킵 트릭은 동시에 도는 스레드가 실제로는 같은 메모리를 두고 경합하지 않는 경우에도 원자 연산을 강제한다는 한계가 있는데, 트랜잭셔널 메모리는 이 문제 자체가 사라진다 — RFO에 준하는 비싼 버스 메시지가 정말 필요한 경우(동시/연속으로 다른 CPU에서 그 메모리를 쓸 때)에만 발생한다. 트랜잭션 캐시가 캐시 라인 단위로 동작하는 배타적 캐시라는 점 때문에, 트랜잭션 데이터와 비트랜잭션 데이터를 같은 캐시 라인에 두면(트랜잭션 도중 그 라인에 대한 일반 접근이 발생해 트랜잭션이 중단될 수 있음) 문제가 된다 — 데이터 정렬이 성능뿐 아니라 정확성 문제로도 이어질 수 있다는 뜻이다.

### 8.3 지연 시간 증가

미래 메모리 기술에서 거의 확실한 한 가지는 **지연 시간이 계속 늘어난다**는 것이다. 2.2.4절에서 봤듯 당시 예정된 DDR3는 DDR2보다 지연이 늘고, FB-DRAM도(특히 데이지 체인 구성일 때) 지연이 늘 수 있다.

두 번째 지연 요인은 **NUMA의 확산**이다 — AMD Opteron은 프로세서가 두 개 이상이면 NUMA 머신이고, 인텔의 CSI도 거의 같은 접근을 쓴다. 프로세서당 대역폭 제약과(예: 10Gb/s 이더넷 카드 여러 장 처리 같은) 서비스 요구 때문에, 소켓당 코어 수가 늘어도 멀티소켓 메인보드 자체는 사라지지 않을 전망이다.

세 번째 요인은 **코프로세서의 부활**이다. 1990년대 초 범용 프로세서에서 수학 코프로세서가 필요 없어진 뒤 사라졌다고 여겨졌지만, 인텔의 Geneseo와 AMD의 Torrenza 같은 플랫폼 확장은 제3자 하드웨어 개발자가 자사 제품을 PCIe 카드가 아니라 CPU에 훨씬 가까운(더 많은 대역폭을 갖는) 위치에 통합할 수 있게 한다. IBM은 다른 방향으로 접근했는데, Cell CPU는 PowerPC 코어 외에 부동소수점 연산에 특화된 8개의 SPU(Synergistic Processing Unit)를 갖는다. 이런 코프로세서·SPU의 공통점은 대체로 진짜 프로세서보다도 메모리 처리 로직이 단순(그래서 더 느림)하다는 것이다 — 캐시 일관성까지 고려한 캐시 처리·프리페칭은 복잡하기 때문이다. Cell CPU의 이론적 최대 성능은 210 GFLOPS로 당시 하이엔드 CPU의 50~60 GFLOPS를 압도했고, GPU는 이미 500 GFLOPS를 넘어서고 있었다 — 고성능 프로그램일수록 코프로세서 의존도가 커질 전망이었다.

이 모든 흐름이 프로그래머에게 주는 결론은 하나다: **프리페칭의 중요성이 갈수록 커진다**. 코프로세서에서는 절대적으로 중요하고, CPU에서도(코어 수가 늘수록) 요청을 몰아서 던지는 대신 FSB를 항상 바쁘게 유지해야 하며, 이를 위해서는 프리페치 명령어를 효율적으로 활용해 CPU에 미래 트래픽에 대한 통찰을 최대한 많이 줘야 한다.

### 8.4 벡터 연산

오늘날 주류 프로세서의 멀티미디어 확장은 벡터 연산을 제한적으로만 구현한다. 벡터 명령어는 한 명령어로 대량의 연산을 수행한다는 특징이 있지만(SIMD), Cray-1이나 IBM 3090의 벡터 유닛에 비하면 훨씬 소박하다(한 명령어당 float 4개 또는 double 2개 수준). 이 제한을 메우려고 주변 루프를 더 자주 반복해야 한다(A.1절 행렬 곱셈 예제에서 캐시 라인 하나당 SM번 반복이 필요했던 것과 같은 맥락).

벡터 레지스터·연산이 넓어지면 루프 반복 횟수가 줄고, 프로세서가 애플리케이션의 메모리 사용 양상을 개별 명령어 동작에서 짜맞추지 않고도 더 잘 파악할 수 있게 된다. 캐시에 영향을 주지 않는 로드/스토어 명령어의 유용성도 커진다 — SSE 레지스터의 16바이트 로드처럼 좁은 단위에서는 캐시 없는 로드가 오히려 손해지만(캐시 미스 시 같은 라인을 다시 메모리에서 읽어야 함), 벡터 레지스터가 캐시 라인 하나 이상을 담을 만큼 넓다면 캐시 없는 로드/스토어가 불리하지 않고 캐시에 못 들어가는 큰 데이터셋 처리도 더 실용적이 된다. 큰 벡터 레지스터라고 반드시 명령어 지연 시간이 늘어나는 것도 아니다 — 벡터 유닛이 코드 흐름을 인식할 수 있다면 데이터 일부만 준비돼도 연산을 시작할 수 있기 때문이다. 다만 실제로는 멀티프로세스·멀티스레드 OS에서 컨텍스트 스위치 비용(레지스터 저장/복원) 때문에 벡터 레지스터 크기에 현실적 제약이 걸린다.

옛 벡터 유닛은 스트라이딩(벡터 원소 간 일정한 간격을 지정해, 예컨대 행렬의 열을 한 번에 벡터 레지스터로 읽는 것)과 인디렉션(로드/스토어 명령어가 실제 주소·오프셋 배열을 가리키는 포인터를 받아 임의 접근 패턴을 구현하는 것)으로 오늘날의 멀티미디어 명령어보다 훨씬 유연한 메모리 접근 패턴을 지원했다. 진짜 벡터 연산이 주류 프로세서에서 부활할지, 아니면 코프로세서 쪽으로 넘어갈지는 이 시점에서 불분명하지만, 만약 벡터 연산에 접근할 수 있게 된다면 코드를 자기완결적이고 교체 가능하게, 그리고 행/열/개별 원소 단위가 아니라 행렬 전체를 다루는 것처럼 충분히 일반적인 인터페이스로 짜는 것이 그만큼 더 중요해진다 — 다루는 빌딩 블록이 클수록 벡터 연산을 활용할 기회도 커지기 때문이다. 다만 큰 레지스터 집합에 따르는 컨텍스트 스위치 비용, 인터럽트 도중 긴 벡터 연산을 중단·재개해야 하는 어려움, RISC 계열 프로세서의 엄격한 정렬 요구 등은 여전히 실현을 가로막는 현실적 장벽으로 남아 있다.

## 부록 개요

원문 부록은 A~G로 구성된다. 상세 재현 대신 각 부록이 다루는 내용과 핵심 요지만 간단히 정리한다.

- **부록 A. 예제·벤치마크 프로그램**: A.1은 6.2.1절 행렬 곱셈의 완전한 벤치마크 코드(SSE2 인트린식, `_MM_HINT_NTA` 프리페치, 64바이트 정렬 활용). A.2는 7.4절에서 언급한, `likely`/`unlikely` 매크로를 런타임에 실제로 계측하는 디버그 버전 구현 — GNU 어셈블러·링커의 커스텀 섹션과 `__start_*`/`__stop_*` 심볼을 이용해 예측 성공/실패 횟수를 소스 위치별로 모아 프로그램 종료 시 출력한다(원자 연산 없이 카운터를 증가시키므로 완벽히 정확하지는 않지만 충분히 실용적). A.3은 6.4.1절 거짓 공유(false sharing) 실험에 쓰인 벤치마크 코드 — 스레드를 `pthread_setaffinity_np`로 특정 CPU에 고정하고, 같은 캐시 라인/다른 캐시 라인에 위치한 변수를 원자적·비원자적으로 증가시켜 오버헤드를 측정한다.

- **부록 B. OProfile 팁**: oprofile은 수집(커널, MSR 접근 권한 필요)과 분석 두 단계로 동작하며, 이벤트마다 오버런 값(몇 번째 이벤트마다 기록할지)·유저/커널 카운트 여부·유닛 마스크를 `opcontrol`로 지정한다. 오버런 값이 작을수록 해상도는 높아지지만 시스템 부하도 커진다. `opreport`로 리포트를, `opannotate`로 소스·명령어 단위 주석을 볼 수 있다. 절대 이벤트 수보다는 **두 이벤트의 비율**이 더 유의미하다는 점을 강조하며, 인텔 최적화 매뉴얼이 제시하는 메모리 관련 비율 지표들(명령어 페치 스톨, ITLB/L1i/L2 미스율, 로드율, 스토어 순서 블록, NTA 프리페치 유효율 등)의 의미와 해석 기준을 표로 정리한다.

- **부록 C. 메모리 타입**: "**레지스터드(registered/buffered)**" DRAM은 모듈에 버퍼 칩을 하나 더 두어 메모리 컨트롤러의 전기적 부담을 줄이고 더 많은 모듈·뱅크를 구동할 수 있게 하지만, 버퍼링 자체가 지연을 늘리고 비용·전력도 더 든다. **ECC(Error Correction Code)** DRAM은 데이터 8비트당 보통 1비트 정도의 오차 정정 비트를 추가로 저장해(해밍 코드 기반, SEC 또는 SEC/DED) 우주선(cosmic ray)이나 알파 붕괴 등으로 인한 셀 오류를 자동 정정한다 — 실제 계산(오차 정정 비트 수 E와 데이터 비트 수 W의 관계식, 행렬을 이용한 해밍 코드 인코딩/디코딩 예시 등)을 통해 W=64, E=8이 실용적인 선택이 되는 이유를 보여준다. 두 기술 모두 비용·지연이라는 트레이드오프 때문에 범용 시스템의 기본값은 아니며, 서버·대용량 메모리 환경에서 주로 쓰인다.

- **부록 D. libNUMA 소개**: 당시 리눅스의 `libnuma`(numactl 패키지)가 NUMA 프로그래밍에 필요한 기능을 충분히 제공하지 못한다는 문제의식에서, 저자가 제안한(당시 미완성·미검증 상태였던) 새 라이브러리의 인터페이스 초안을 소개한다. `/sys` 파일시스템 노출 정보에 크게 의존하며, 메모리 노드 집합을 다루는 `memnode_set_t`와 관련 매크로들을 정의한다. NUMA뿐 아니라 캐시 계층 문제와도 겹치므로 비NUMA 멀티코어 시스템에서도 유용하다고 언급된다.

- **부록 E~G**: 색인(주요 용어의 등장 위치), 참고문헌 목록(AMD·인텔 문서, Herlihy&Moss의 트랜잭셔널 메모리 논문 등 학술 자료), 개정 이력(2007년 6월 내부 초안부터 같은 해 11월 21일 Version 1.0 발행까지의 편집 경과)으로 구성된다.

---

이것으로 Ulrich Drepper의 "What Every Programmer Should Know About Memory" 5부작 정리를 마친다. 전체 흐름을 요약하면: DRAM은 근본적으로 느리고(2장), 이를 CPU 캐시로 가린다(3장). 가상 메모리·NUMA는 여기에 또 다른 계층의 비용과 복잡성을 더하며(4~5장), 결국 프로그래머가 지역성을 살리는 코드·자료구조를 설계하고 프리페칭·정렬·스레드 배치를 신경 써야만 하드웨어의 잠재력을 실제로 끌어낼 수 있다(6장). 측정 도구(7장)로 이를 검증하고, 코어 수 증가와 트랜잭셔널 메모리 같은 다가올 변화(8장)에 대비하는 것이 이 문서 전체가 남기는 실무적 메시지다.
