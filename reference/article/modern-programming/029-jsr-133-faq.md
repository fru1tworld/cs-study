## 메모리 모델이란 무엇인가?

멀티프로세서 시스템에서 프로세서는 일반적으로 한 층 이상의 메모리 캐시를 갖는다. 캐시는 데이터 접근 속도를 높이고(데이터가 프로세서에 더 가까이 있으므로) 공유 메모리 버스의 트래픽을 줄여(많은 메모리 연산을 로컬 캐시에서 처리할 수 있으므로) 성능을 개선한다. 메모리 캐시는 성능을 엄청나게 끌어올릴 수 있지만, 새로운 난제를 잔뜩 안겨 준다. 예를 들어 두 프로세서가 같은 메모리 위치를 동시에 들여다보면 어떻게 되는가? 어떤 조건에서 둘은 같은 값을 보게 되는가?

프로세서 수준에서 메모리 모델은, 다른 프로세서의 메모리 쓰기가 현재 프로세서에 보이고 현재 프로세서의 쓰기가 다른 프로세서에 보인다는 것을 알기 위한 필요충분조건을 정의한다. 어떤 프로세서는 모든 프로세서가 언제나 어떤 메모리 위치든 정확히 같은 값을 보는 강한 메모리 모델을 보인다. 다른 프로세서는 더 약한 메모리 모델을 보이는데, 다른 프로세서의 쓰기를 보거나 이 프로세서의 쓰기를 다른 프로세서에 보이게 만들려면 메모리 배리어라 불리는 특수 명령어로 로컬 프로세서 캐시를 플러시하거나 무효화해야 한다. 이 메모리 배리어는 보통 락과 언락 동작이 수행될 때 함께 수행되며, 고수준 언어의 프로그래머에게는 보이지 않는다.

강한 메모리 모델에서는 메모리 배리어의 필요성이 줄어들기 때문에 프로그램을 작성하기가 더 쉬울 때도 있다. 그러나 가장 강한 축에 드는 메모리 모델에서조차 메모리 배리어가 필요한 경우가 많고, 그 배치가 직관에 어긋나는 일도 꽤 흔하다. 최근의 프로세서 설계 추세는 더 약한 메모리 모델을 선호해 왔다. 캐시 일관성에 대한 완화 덕분에 여러 프로세서와 더 큰 메모리에 걸쳐 더 큰 확장성을 얻을 수 있기 때문이다.

쓰기가 언제 다른 스레드에 보이게 되느냐는 문제는 컴파일러의 코드 재배치 때문에 더 복잡해진다. 예를 들어 컴파일러는 쓰기 연산을 프로그램의 뒤쪽으로 옮기는 것이 더 효율적이라고 판단할 수 있다. 이 코드 이동이 프로그램의 의미를 바꾸지 않는 한, 컴파일러는 자유롭게 그렇게 할 수 있다. 컴파일러가 어떤 연산을 미루면, 다른 스레드는 그 연산이 수행될 때까지 그것을 보지 못한다. 이는 캐싱의 효과와 똑같다.

게다가 메모리 쓰기는 프로그램의 앞쪽으로 옮겨질 수도 있다. 이 경우 다른 스레드는 프로그램에서 실제로 그 쓰기가 "일어나기" 전에 그것을 볼 수 있다. 이 모든 유연성은 의도된 설계다. 컴파일러, 런타임, 하드웨어에 메모리 모델의 한계 안에서 연산을 최적 순서로 실행할 유연성을 줌으로써 더 높은 성능을 달성할 수 있다.

간단한 예를 다음 코드에서 볼 수 있다.

```java
Class Reordering {
  int x = 0, y = 0;
  public void writer() {
    x = 1;
    y = 2;
  }

  public void reader() {
    int r1 = y;
    int r2 = x;
  }
}
```

이 코드가 두 스레드에서 동시에 실행되고, y 읽기가 값 2를 본다고 하자. 이 쓰기가 x에 대한 쓰기 뒤에 왔으므로, 프로그래머는 x 읽기가 반드시 값 1을 본다고 가정할지 모른다. 그러나 쓰기들이 재배치되었을 수 있다. 그렇게 되면 y에 대한 쓰기가 일어난 뒤 두 변수의 읽기가 뒤따르고, 그다음에 x에 대한 쓰기가 일어날 수 있다. 결과적으로 r1은 값 2를 갖지만 r2는 값 0을 갖는다.

Java 메모리 모델은 멀티스레드 코드에서 어떤 동작이 합법인지, 스레드들이 메모리를 통해 어떻게 상호작용할 수 있는지를 기술한다. 프로그램의 변수와, 실제 컴퓨터 시스템에서 그것을 메모리나 레지스터에 저장하고 꺼내는 저수준 세부 사항 사이의 관계를 기술한다. 그것도 다양한 하드웨어와 다양한 컴파일러 최적화를 써서 올바르게 구현할 수 있는 방식으로 기술한다.

Java에는 volatile, final, synchronized를 비롯한 여러 언어 구성물이 있다. 이들은 프로그래머가 프로그램의 동시성 요구사항을 컴파일러에게 기술하도록 돕기 위한 것이다. Java 메모리 모델은 volatile과 synchronized의 동작을 정의하며, 더 중요하게는 올바르게 동기화된 Java 프로그램이 모든 프로세서 아키텍처에서 올바르게 실행되도록 보장한다.

## C++ 같은 다른 언어에도 메모리 모델이 있는가?

C와 C++ 같은 대부분의 다른 프로그래밍 언어는 멀티스레딩을 직접 지원하도록 설계되지 않았다. 컴파일러와 아키텍처에서 일어나는 종류의 재배치로부터 이 언어들이 제공하는 보호는, 사용하는 스레딩 라이브러리(pthreads 등), 사용하는 컴파일러, 코드가 실행되는 플랫폼이 제공하는 보장에 크게 좌우된다.

## JSR 133은 무엇에 관한 것인가?

1997년 이후, Java 언어 명세 17장에 정의된 Java 메모리 모델에서 여러 심각한 결함이 발견되었다. 이 결함들은 혼란스러운 동작(final 필드의 값이 바뀌는 것이 관찰되는 등)을 허용했고, 컴파일러가 흔한 최적화를 수행할 능력을 약화시켰다.

Java 메모리 모델은 야심찬 시도였다. 프로그래밍 언어 명세가 다양한 아키텍처에 걸쳐 동시성에 일관된 의미론을 제공할 수 있는 메모리 모델을 포함하려고 시도한 최초의 사례였다. 불행히도 일관되면서도 직관적인 메모리 모델을 정의하는 일은 예상보다 훨씬 어려웠다. JSR 133은 이전 메모리 모델의 결함을 고치는 Java 언어의 새 메모리 모델을 정의한다. 이를 위해 final과 volatile의 의미론이 바뀌어야 했다.

전체 의미론은 [http://www.cs.umd.edu/users/pugh/java/memoryModel](http://www.cs.umd.edu/users/pugh/java/memoryModel)에서 볼 수 있지만, 형식적 의미론은 마음 약한 사람을 위한 것이 아니다. 동기화처럼 단순해 보이는 개념이 실제로는 얼마나 복잡한지 알게 되면 놀랍고, 또 정신이 번쩍 든다. 다행히 형식적 의미론의 세부 사항을 이해할 필요는 없다. JSR 133의 목표는 volatile, synchronized, final이 어떻게 동작하는지에 대한 직관적인 틀을 제공하는 형식적 의미론 집합을 만드는 것이었다.

JSR 133의 목표는 다음을 포함한다.

- 타입 안전성 같은 기존의 안전성 보장을 보존하고, 다른 것들은 강화한다. 예를 들어 변수 값은 "허공에서(out of thin air)" 만들어질 수 없다. 어떤 스레드가 관찰하는 변수의 각 값은 어떤 스레드가 합리적으로 거기에 놓을 수 있는 값이어야 한다.
- 올바르게 동기화된 프로그램의 의미론은 가능한 한 단순하고 직관적이어야 한다.
- 불완전하게 또는 잘못 동기화된 프로그램의 의미론은 잠재적 보안 위험이 최소화되도록 정의되어야 한다.
- 프로그래머는 멀티스레드 프로그램이 메모리와 어떻게 상호작용하는지 확신을 갖고 추론할 수 있어야 한다.
- 널리 쓰이는 다양한 하드웨어 아키텍처에서 올바르고 고성능인 JVM 구현을 설계할 수 있어야 한다.
- *초기화 안전성(initialization safety)*이라는 새로운 보장이 제공되어야 한다. 객체가 올바르게 생성되면(생성 중에 그 객체에 대한 참조가 탈출하지 않는다는 뜻), 그 객체에 대한 참조를 보는 모든 스레드는 동기화 없이도 생성자에서 설정된 final 필드의 값을 함께 보게 된다.
- 기존 코드에 대한 영향은 최소화되어야 한다.

## 재배치(reordering)란 무엇을 뜻하는가?

프로그램 변수(객체 인스턴스 필드, 클래스 정적 필드, 배열 원소)에 대한 접근이 프로그램에 명시된 것과 다른 순서로 실행되는 것처럼 보일 수 있는 경우가 여럿 있다. 컴파일러는 최적화라는 명목으로 명령어 순서를 자유롭게 바꿀 수 있다. 프로세서는 특정 상황에서 명령어를 순서 없이 실행할 수 있다. 데이터는 레지스터, 프로세서 캐시, 메인 메모리 사이를 프로그램에 명시된 것과 다른 순서로 이동할 수 있다.

예를 들어 스레드가 필드 `a`에 쓴 뒤 필드 `b`에 쓰는데 `b`의 값이 `a`의 값에 의존하지 않는다면, 컴파일러는 이 연산들을 자유롭게 재배치할 수 있고, 캐시는 `a`보다 먼저 `b`를 메인 메모리에 플러시할 수 있다. 컴파일러, JIT, 캐시 등 재배치의 잠재적 원인은 여럿 있다.

컴파일러, 런타임, 하드웨어는 공모하여 마치 순차 실행처럼 보이는(as-if-serial) 의미론의 환상을 만들어 내기로 되어 있다. 즉, 단일 스레드 프로그램에서는 재배치의 효과를 관찰할 수 없어야 한다. 그러나 잘못 동기화된 멀티스레드 프로그램에서는 재배치가 끼어들 수 있다. 한 스레드가 다른 스레드들의 효과를 관찰할 수 있고, 변수 접근이 프로그램에서 실행되거나 명시된 순서와 다른 순서로 다른 스레드에 보이게 된다는 것을 감지할 수 있다.

대부분의 경우 한 스레드는 다른 스레드가 무엇을 하는지 신경 쓰지 않는다. 하지만 신경 써야 할 때, 그럴 때 쓰라고 있는 것이 동기화다.

## 기존 메모리 모델의 무엇이 문제였는가?

기존 메모리 모델에는 여러 심각한 문제가 있었다. 이해하기 어려웠고, 그래서 널리 위반되었다. 예를 들어 기존 모델은 많은 경우, 모든 JVM에서 실제로 일어나던 종류의 재배치를 허용하지 않았다. 기존 모델의 함의를 둘러싼 이런 혼란이 JSR-133 결성을 촉발했다.

예를 들어 널리 퍼진 믿음 하나는, final 필드를 쓰면 다른 스레드가 그 필드의 값을 보도록 보장하기 위한 스레드 간 동기화가 불필요하다는 것이었다. 이것은 합리적인 가정이고 분별 있는 동작이며, 실제로 우리가 원하는 동작이지만, 기존 메모리 모델에서는 그냥 사실이 아니었다. 기존 메모리 모델의 어떤 것도 final 필드를 다른 필드와 다르게 취급하지 않았다. 즉, 생성자가 쓴 final 필드의 값을 모든 스레드가 보도록 보장하는 유일한 방법은 동기화였다. 그 결과 스레드가 필드의 기본값을 본 다음, 나중에 생성자가 설정한 값을 보는 것이 가능했다. 이는 예를 들어 String 같은 불변 객체가 값이 바뀌는 것처럼 보일 수 있다는 뜻이다. 실로 불온한 전망이다.

기존 메모리 모델은 volatile 쓰기가 비volatile 읽기·쓰기와 재배치되는 것을 허용했다. 이는 volatile에 대한 대부분 개발자의 직관과 어긋났고, 그래서 혼란을 일으켰다.

마지막으로, 앞으로 보겠지만, 프로그램이 잘못 동기화되었을 때 무슨 일이 일어날 수 있는지에 대한 프로그래머들의 직관은 틀린 경우가 많다. JSR-133의 목표 중 하나는 이 사실에 주의를 환기하는 것이다.

## "잘못 동기화되었다"는 말은 무슨 뜻인가?

잘못 동기화된 코드는 사람마다 다른 것을 뜻할 수 있다. Java 메모리 모델의 맥락에서 잘못 동기화된 코드라고 말할 때는 다음에 해당하는 코드를 뜻한다.

1. 한 스레드가 어떤 변수에 쓰고,
2. 다른 스레드가 같은 변수를 읽으며,
3. 그 쓰기와 읽기가 동기화로 순서 지어지지 않은 경우

이 규칙이 위반될 때, 그 변수에 *데이터 레이스(data race)*가 있다고 말한다. 데이터 레이스가 있는 프로그램이 잘못 동기화된 프로그램이다.

## 동기화는 무엇을 하는가?

동기화에는 여러 측면이 있다. 가장 잘 이해되는 것은 상호 배제다. 한 번에 하나의 스레드만 모니터를 쥘 수 있으므로, 모니터로 동기화한다는 것은 한 스레드가 모니터로 보호되는 synchronized 블록에 들어가면, 그 스레드가 블록에서 나올 때까지 다른 어떤 스레드도 그 모니터로 보호되는 블록에 들어갈 수 없다는 뜻이다.

그러나 동기화에는 상호 배제 이상이 있다. 동기화는 synchronized 블록 이전 또는 도중에 스레드가 수행한 메모리 쓰기가, 같은 모니터로 동기화하는 다른 스레드들에게 예측 가능한 방식으로 보이도록 보장한다. synchronized 블록에서 나갈 때 우리는 모니터를 **해제**(release)하는데, 이는 캐시를 메인 메모리로 플러시하는 효과가 있어서 이 스레드가 수행한 쓰기가 다른 스레드에 보일 수 있게 된다. synchronized 블록에 들어가기 전에 우리는 모니터를 **획득**(acquire)하는데, 이는 로컬 프로세서 캐시를 무효화하여 변수들이 메인 메모리에서 다시 로드되도록 하는 효과가 있다. 그러면 이전 해제가 보이게 만든 모든 쓰기를 볼 수 있게 된다.

캐시 관점에서 이야기하면 이 문제가 멀티프로세서 머신에만 영향을 주는 것처럼 들릴 수 있다. 그러나 재배치 효과는 단일 프로세서에서도 쉽게 볼 수 있다. 예를 들어 컴파일러는 획득 이전이나 해제 이후로 코드를 옮길 수 없다. 획득과 해제가 캐시에 작용한다고 말할 때, 우리는 가능한 여러 효과를 축약해서 말하는 것이다.

새 메모리 모델의 의미론은 메모리 연산(필드 읽기, 필드 쓰기, 락, 언락)과 다른 스레드 연산(start와 join)에 부분 순서를 만든다. 이 순서에서 어떤 동작은 다른 연산보다 *먼저 일어난다(happen before)*고 말한다. 한 동작이 다른 동작보다 먼저 일어나면, 첫 번째 동작은 두 번째 동작보다 먼저 순서 지어지고 두 번째 동작에 보인다는 것이 보장된다. 이 순서의 규칙은 다음과 같다.

- 스레드 내의 각 동작은 프로그램 순서상 그 뒤에 오는 그 스레드의 모든 동작보다 먼저 일어난다.
- 모니터에 대한 언락은 **같은** 모니터에 대한 이후의 모든 락보다 먼저 일어난다.
- volatile 필드에 대한 쓰기는 **같은** volatile에 대한 이후의 모든 읽기보다 먼저 일어난다.
- 스레드에 대한 `start()` 호출은 시작된 스레드의 모든 동작보다 먼저 일어난다.
- 스레드의 모든 동작은 다른 스레드가 그 스레드에 대한 `join()`에서 성공적으로 반환하는 것보다 먼저 일어난다.

이는 synchronized 블록에서 나가기 전에 스레드에 보였던 모든 메모리 연산이, 같은 모니터로 보호되는 synchronized 블록에 들어간 이후의 어떤 스레드에게든 보인다는 뜻이다. 모든 메모리 연산이 해제보다 먼저 일어나고, 해제는 획득보다 먼저 일어나기 때문이다.

또 다른 함의는, 어떤 사람들이 메모리 배리어를 강제하려고 쓰는 다음 패턴이 동작하지 않는다는 것이다.

```java
synchronized (new Object()) {}
```

이것은 사실상 no-op이며, 컴파일러가 통째로 제거할 수 있다. 다른 어떤 스레드도 같은 모니터로 동기화하지 않으리라는 것을 컴파일러가 알기 때문이다. 한 스레드가 다른 스레드의 결과를 보려면 happens-before 관계를 설정해야 한다.

**중요:** happens-before 관계를 올바르게 설정하려면 두 스레드가 같은 모니터로 동기화하는 것이 중요하다. 스레드 A가 객체 X로 동기화할 때 A에게 보이는 모든 것이, 스레드 B가 객체 Y로 동기화한 뒤 B에게 보이게 되는 것이 아니다. 올바른 의미론을 가지려면 해제와 획득이 "맞아떨어져야"(즉, 같은 모니터에서 수행되어야) 한다. 그렇지 않으면 코드에 데이터 레이스가 있다.

## final 필드의 값이 어떻게 바뀌어 보일 수 있는가?

final 필드의 값이 바뀌는 것처럼 보일 수 있는 최고의 예 중 하나는 `String` 클래스의 특정 구현과 관련이 있다.

`String`은 세 필드를 가진 객체로 구현될 수 있다. 문자 배열, 그 배열 안의 오프셋, 그리고 길이다. 문자 배열만 갖는 대신 이렇게 `String`을 구현하는 근거는, 여러 `String`과 `StringBuffer` 객체가 같은 문자 배열을 공유하여 추가적인 객체 할당과 복사를 피할 수 있기 때문이다. 예를 들어 `String.substring()` 메서드는 원래 `String`과 같은 문자 배열을 공유하면서 길이와 오프셋 필드만 다른 새 문자열을 만드는 것으로 구현될 수 있다. `String`에서 이 필드들은 모두 final 필드다.

```java
String s1 = "/usr/tmp";
String s2 = s1.substring(4);
```

문자열 `s2`는 오프셋 4와 길이 4를 갖게 된다. 그러나 기존 모델에서는 다른 스레드가 오프셋을 기본값 0으로 보았다가, 나중에 올바른 값 4를 보는 것이 가능했다. 마치 문자열 "/usr"가 "/tmp"로 바뀌는 것처럼 보이게 된다.

원래의 Java 메모리 모델은 이 동작을 허용했고, 실제로 여러 JVM이 이 동작을 보였다. 새 Java 메모리 모델은 이를 불법으로 만든다.

## 새 JMM에서 final 필드는 어떻게 동작하는가?

객체의 final 필드 값은 생성자에서 설정된다. 객체가 "올바르게" 생성되었다고 가정하면, 객체가 생성되고 나면 생성자에서 final 필드에 할당된 값은 동기화 없이도 다른 모든 스레드에 보이게 된다. 게다가 그 final 필드가 참조하는 다른 객체나 배열의 보이는 값들도 최소한 final 필드만큼 최신이다.

객체가 올바르게 생성되었다는 것은 무슨 뜻인가? 간단히 말해, 생성 중인 객체에 대한 참조가 생성 도중에 "탈출(escape)"하는 것이 허용되지 않는다는 뜻이다. (예시는 [Safe Construction Techniques](http://www-106.ibm.com/developerworks/java/library/j-jtp0618.html)를 보라.) 다시 말해, 생성 중인 객체에 대한 참조를 다른 스레드가 볼 수 있을 만한 곳에 두지 말라. 정적 필드에 할당하지 말고, 다른 객체에 리스너로 등록하지 말라. 이런 일은 생성자 안이 아니라 생성자가 완료된 뒤에 해야 한다.

```java
class FinalFieldExample {
  final int x;
  int y;
  static FinalFieldExample f;
  public FinalFieldExample() {
    x = 3;
    y = 4;
  }

  static void writer() {
    f = new FinalFieldExample();
  }

  static void reader() {
    if (f != null) {
      int i = f.x;
      int j = f.y;
    }
  }
}
```

위 클래스는 final 필드를 어떻게 써야 하는지 보여 주는 예다. `reader`를 실행하는 스레드는 `f.x`에 대해 값 3을 보는 것이 보장된다. final이기 때문이다. `y`에 대해 값 4를 보는 것은 보장되지 않는다. final이 아니기 때문이다. `FinalFieldExample`의 생성자가 다음과 같다면,

```java
public FinalFieldExample() { // 나쁨!
  x = 3;
  y = 4;
  // 나쁜 생성 방식 - this가 탈출하도록 허용
  global.obj = this;
}
```

`global.obj`에서 `this`에 대한 참조를 읽는 스레드는 `x`에 대해 3을 보는 것이 보장되지 **않는다**.

필드의 올바르게 생성된 값을 볼 수 있다는 것은 좋지만, 필드 자체가 참조라면 그것이 가리키는 객체(또는 배열)의 최신 값도 코드가 보기를 원할 것이다. 필드가 final 필드라면 이것도 보장된다. 그래서 배열에 대한 final 포인터를 가지면서, 다른 스레드가 배열 참조는 올바른 값을 보는데 배열 내용은 잘못된 값을 보지 않을까 걱정할 필요가 없다. 여기서 "올바르다"는 것은 "객체의 생성자가 끝난 시점 기준으로 최신"이라는 뜻이지 "가장 최근에 사용 가능한 값"이라는 뜻이 아니다.

이 모든 것을 말한 다음에 덧붙이자면, 어떤 스레드가 불변 객체(즉, final 필드만 포함하는 객체)를 생성한 뒤 그것이 다른 모든 스레드에 올바르게 보이도록 보장하고 싶다면, **여전히** 보통은 동기화를 써야 한다. 예를 들어 불변 객체에 대한 참조가 두 번째 스레드에 보이도록 보장할 다른 방법은 없다. final 필드로부터 프로그램이 얻는 보장은, 코드에서 동시성이 어떻게 관리되는지에 대한 깊고 신중한 이해로 조심스럽게 다뤄져야 한다.

JNI를 써서 final 필드를 바꾸려는 경우의 동작은 정의되어 있지 않다.

## volatile은 무엇을 하는가?

volatile 필드는 스레드 사이에 상태를 주고받는 데 쓰이는 특별한 필드다. volatile의 각 읽기는 어떤 스레드든 그 volatile에 수행한 마지막 쓰기를 보게 된다. 사실상 프로그래머가 캐싱이나 재배치의 결과로 "낡은(stale)" 값을 보는 것이 결코 용납되지 않는 필드로 지정한 것이다. 컴파일러와 런타임은 volatile을 레지스터에 할당하는 것이 금지된다. 또한 volatile이 쓰인 뒤에는 캐시에서 메인 메모리로 플러시되어 다른 스레드에 즉시 보일 수 있도록 보장해야 한다. 마찬가지로 volatile 필드를 읽기 전에는 캐시를 무효화하여 로컬 프로세서 캐시가 아니라 메인 메모리의 값을 보도록 해야 한다. volatile 변수 접근의 재배치에도 추가 제약이 있다.

기존 메모리 모델에서 volatile 변수 접근은 서로 간에는 재배치될 수 없었지만, 비volatile 변수 접근과는 재배치될 수 있었다. 이는 한 스레드에서 다른 스레드로 조건을 신호하는 수단으로서 volatile 필드의 유용성을 약화시켰다.

새 메모리 모델에서도 volatile 변수가 서로 재배치될 수 없다는 것은 여전히 사실이다. 차이점은 이제 일반 필드 접근을 volatile 접근 주위로 재배치하기가 그리 쉽지 않다는 것이다. volatile 필드에 쓰는 것은 모니터 해제와 같은 메모리 효과를 갖고, volatile 필드에서 읽는 것은 모니터 획득과 같은 메모리 효과를 갖는다. 사실상 새 메모리 모델이 volatile 필드 접근과 다른 필드 접근(volatile이든 아니든)의 재배치에 더 엄격한 제약을 두기 때문에, 스레드 A가 volatile 필드 `f`에 쓸 때 A에게 보였던 모든 것이, 스레드 B가 `f`를 읽을 때 B에게 보이게 된다.

volatile 필드를 어떻게 쓸 수 있는지 보여 주는 간단한 예다.

```java
class VolatileExample {
  int x = 0;
  volatile boolean v = false;
  public void writer() {
    x = 42;
    v = true;
  }

  public void reader() {
    if (v == true) {
      //x 사용 - 42를 보는 것이 보장된다.
    }
  }
}
```

한 스레드가 `writer`를 호출하고 다른 스레드가 `reader`를 호출한다고 하자. `writer`에서 `v`에 대한 쓰기는 `x`에 대한 쓰기를 메모리로 해제하고, `v`의 읽기는 그 값을 메모리에서 획득한다. 따라서 reader가 v에 대해 `true` 값을 보면, 그보다 먼저 일어난 42에 대한 쓰기도 보는 것이 보장된다. 기존 메모리 모델에서는 이것이 참이 아니었을 것이다. `v`가 volatile이 아니라면 컴파일러는 `writer`의 쓰기들을 재배치할 수 있고, `reader`의 `x` 읽기는 0을 볼 수도 있다.

사실상 volatile의 의미론은 거의 동기화 수준까지 대폭 강화되었다. 가시성의 목적에서, volatile 필드의 각 읽기나 쓰기는 동기화의 "절반"처럼 동작한다.

**중요:** happens-before 관계를 올바르게 설정하려면 두 스레드가 같은 volatile 변수에 접근하는 것이 중요하다. 스레드 A가 volatile 필드 `f`에 쓸 때 A에게 보이는 모든 것이, 스레드 B가 volatile 필드 `g`를 읽은 뒤 B에게 보이게 되는 것이 아니다. 올바른 의미론을 가지려면 해제와 획득이 "맞아떨어져야"(즉, 같은 volatile 필드에서 수행되어야) 한다.

## 새 메모리 모델은 "이중 검사 락킹" 문제를 고치는가?

(악명 높은) 이중 검사 락킹(double-checked locking) 이디엄(멀티스레드 싱글턴 패턴이라고도 불린다)은 동기화의 오버헤드를 피하면서 지연 초기화를 지원하도록 고안된 트릭이다. 아주 초기의 JVM에서는 동기화가 느렸고, 개발자들은 그것을 제거하고 싶어 안달이었다. 아마 지나치게 안달이었을 것이다. 이중 검사 락킹 이디엄은 이렇게 생겼다.

```java
// 이중 검사 락킹 - 이렇게 하지 말 것!

private static Something instance = null;

public Something getInstance() {
  if (instance == null) {
    synchronized (this) {
      if (instance == null)
        instance = new Something();
    }
  }
  return instance;
}
```

굉장히 영리해 보인다. 흔히 실행되는 코드 경로에서 동기화를 피하니까. 문제는 딱 하나다. **동작하지 않는다**. 왜 안 되는가? 가장 명백한 이유는, `instance`를 초기화하는 쓰기들과 `instance` 필드에 대한 쓰기가 컴파일러나 캐시에 의해 재배치될 수 있고, 그러면 부분적으로만 생성된 것처럼 보이는 `Something`을 반환하는 효과가 생기기 때문이다. 결과적으로 초기화되지 않은 객체를 읽게 된다. 이것이 잘못된 이유, 그리고 이에 대한 알고리즘적 수정들이 잘못된 이유는 그 밖에도 많다. 기존 Java 메모리 모델로는 고칠 방법이 없다. 더 깊이 있는 정보는 [Double-checked locking: Clever, but broken](http://www.javaworld.com/jw-02-2001/jw-0209-double.html)과 [The "Double Checked Locking is broken" declaration](http://www.cs.umd.edu/~pugh/java/memoryModel/DoubleCheckedLocking.html)에서 찾을 수 있다.

많은 사람이 `volatile` 키워드를 쓰면 이중 검사 락킹 패턴을 쓰려 할 때 생기는 문제가 사라진다고 가정했다. 1.5 이전의 JVM에서 `volatile`은 동작을 보장하지 못했다(환경에 따라 결과가 달랐다). 새 메모리 모델에서는 `instance` 필드를 volatile로 만들면 이중 검사 락킹의 문제가 "고쳐진다". 생성하는 스레드의 `Something` 초기화와, 그 값을 읽는 스레드의 반환 사이에 happens-before 관계가 생기기 때문이다.

~~그러나 이중 검사 락킹의 팬들에게는(정말이지 한 명도 남아 있지 않기를 바란다) 여전히 좋은 소식이 아니다. 이중 검사 락킹의 요점은 동기화의 성능 오버헤드를 피하는 것이었다. Java 1.0 시절 이후 짧은 동기화가 훨씬 저렴해졌을 뿐 아니라, 새 메모리 모델에서는 volatile 사용의 성능 비용이 거의 동기화 비용 수준까지 올라간다. 그러니 이중 검사 락킹을 쓸 좋은 이유는 여전히 없다.~~ *정정 — 대부분의 플랫폼에서 volatile은 저렴하다.*

대신 스레드 안전하면서 이해하기도 훨씬 쉬운 Initialization On Demand Holder 이디엄을 써라.

```java
private static class LazySomethingHolder {
  public static Something something = new Something();
}

public static Something getInstance() {
  return LazySomethingHolder.something;
}
```

이 코드는 정적 필드에 대한 초기화 보장 때문에 올바름이 보장된다. 필드가 정적 초기화 구문에서 설정되면, 그 클래스에 접근하는 어떤 스레드에게든 올바르게 보이는 것이 보장된다.

## 왜 신경 써야 하는가?

왜 신경 써야 할까? 동시성 버그는 디버깅하기가 매우 어렵다. 테스트에서는 잘 나타나지 않고, 프로그램이 높은 부하에서 실행될 때까지 기다렸다가 나타나며, 재현하고 잡아내기 어렵다. 프로그램이 올바르게 동기화되도록 미리 추가 노력을 들이는 편이 훨씬 낫다. 쉬운 일은 아니지만, 잘못 동기화된 애플리케이션을 디버깅하려 애쓰는 것보다는 훨씬 쉽다.
