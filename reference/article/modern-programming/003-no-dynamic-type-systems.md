# 아니요, 동적 타입 시스템이 본질적으로 더 개방적인 것은 아니다

> 원문: [No, dynamic type systems are not inherently more open](https://lexi-lambda.github.io/blog/2020/01/19/no-dynamic-type-systems-are-not-inherently-more-open/)
> 저자: Alexis King | 번역일: 2026-07-10

타이핑 규율(typing discipline)을 둘러싼 인터넷 논쟁에는 동적 타입 시스템이 "열린 세계(open world)" 도메인을 모델링하는 데 본질적으로 더 낫다는 신화가 만연해 있다. 논증은 대개 이렇다. 정적 타이핑의 목표는 모든 것을 최대한 못 박는 것인데, 현실 세계에서 그건 그냥 비현실적이다. 실제 시스템은 느슨하게 결합돼야 하고 데이터 표현에 대한 걱정은 최소화해야 하므로, 동적 타입이 크게 봤을 때 더 견고한 시스템으로 이어진다는 것이다.

이 이야기는 그럴듯하게 들리지만 사실이 아니다. 결함은 전제에 있다. 정적 타입은 "세계를 분류"하거나 시스템 안 모든 값의 구조를 못 박는 것이 _아니다_. 실제로 정적 타입 시스템은 어떤 컴포넌트가 입력의 구조에 대해 정확히 얼마만큼 알아야 하는지, 그리고 역으로 얼마만큼은 몰라도 되는지를 명시할 수 있게 해 준다. 오히려 실무에서 정적 타입 시스템은 구조가 부분적으로만 알려진 데이터를 처리하는 데 뛰어나다. 애플리케이션 로직이 실수로 너무 많은 것을 가정하지 않도록 보장하는 데 쓸 수 있기 때문이다.

## 두 가지 타이핑 오류

이 글은 예전부터 쓰고 싶었는데, 마침내 결심하게 만든 것은 [이전 글](https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/)에 달린 잘못 알고 쓴 댓글들이었다. 특히 두 댓글이 눈에 띄었는데, [첫 번째는 /r/programming에 올라온 것이다](https://www.reddit.com/r/programming/comments/dt0w63/parse_dont_validate/f6ulpsy/).

> 이 글에 강하게 반대한다 […] 이 글은 근본적으로 얽혀 있고 정적인 세계관을 조장한다. 프로그램과 세계 사이의 경계에서 무엇이 "유효한" 입력인지 이론화할 수 있거나 그래야 한다고 가정하며, 그럼으로써 소프트웨어 전체에 강한 결합을 끌어들인다. 어떤 스키마에 맞지 않으면 프로그램이 자동으로 죽어 버리는 것이다.
>
> 여기서는 이걸 장점으로 내세우는데, 인터넷이 이런 식으로 동작한다고 상상해 보라. 어떤 서버가 JSON 출력을 바꾸면 인터넷 전체를 다시 컴파일하고 다시 프로그래밍해야 한다. 이 글이 장점으로 내세우는 게 바로 이 정적인 관점이다. […] "파서 사고방식"은 근본적으로 경직되고 전역적인 반면, 견고한 시스템 설계는 탈중앙화되어야 하고 데이터 해석을 수신자에게 맡겨야 한다.

그 글의 논지, 즉 가능하면 언제나 정밀한 타입을 쓰라는 주장을 감안하면 이 오해가 어디서 오는지는 이해가 된다. 프록시 서버는 페이로드의 구조를 미리 알 수 없는데 어떻게 그런 스타일로 작성할 수 있겠는가? 이 댓글 작성자의 결론은 엄격한 정적 타이핑이 입력 구조를 사전에 알지 못하는 프로그램과 상충한다는 것이다.

[두 번째 댓글은 Hacker News에 달렸고](https://news.ycombinator.com/item?id=21479933), 첫 번째보다 훨씬 짧다.

> 예를 들어 Python의 `pickle.load()`의 타입 시그니처는 뭐가 될까?

이건 다른 종류의 논증이다. 리플렉션 연산의 타입이 런타임 값에 의존할 수 있어서 정적 타입으로 담아내기 어렵다는 사실에 기댄다. 이 논증은 정적 타입이 그런 연산을 아예 금지하기 때문에 표현력을 제한한다고 시사한다.

두 논증 모두 오류지만, 왜 그런지 보이려면 암묵적인 주장 하나를 명시적으로 드러내야 한다. 두 댓글은 주로 정적 타입 시스템이 형태를 알 수 없는 데이터를 처리하지 못한다는 것을 보여 주는 데 집중하지만, 동시에 암묵적인 믿음을 밀어붙인다. 동적 타입 언어는 형태를 알 수 없는 데이터를 처리할 수 _있다_ 는 믿음이다. 곧 보겠지만 이 믿음은 잘못됐다. 타이핑 규율과 무관하게, 프로그램은 형태를 정말로 알 수 없는 데이터를 처리할 능력이 없다. 정적 타입 시스템은 이미 존재하는 가정을 명시적으로 만들 뿐이다.

## 모르는 것은 처리할 수 없다

주장은 단순하다. 정적 타입 시스템에서는 데이터의 형태를 미리 선언해야 하지만, 동적 타입 시스템에서는 타입이, 말 그대로, 동적일 수 있다! 자명하게 들리는 나머지, Rich Hickey는 그 정서적 호소력 위에 강연 커리어를 쌓다시피 했다. 유일한 문제는 그게 사실이 아니라는 것이다.

가상 시나리오는 대개 이렇다. 분산 시스템이 있고, 시스템의 서비스들이 이벤트를 발행하면 그게 필요한 다른 서비스가 소비할 수 있다고 하자. 각 이벤트에는 페이로드가 따라오고, 수신 서비스는 이를 바탕으로 후속 동작을 결정한다. 페이로드 자체는 JSON이나 [EDN](https://github.com/edn-format/edn) 같은 범용 교환 포맷으로 인코딩된, 최소한으로만 구조화된 스키마 없는 데이터다.

간단한 예로, 로그인 서비스는 새 사용자가 가입할 때마다 이런 이벤트를 발행할 수 있다.

```json
{
  "event_type": "signup",
  "timestamp": "2020-01-19T05:37:09Z",
  "data": {
    "user": {
      "id": 42,
      "name": "Alyssa",
      "email": "alyssa@example.com"
    }
  }
}
```

일부 하류 서비스는 이 `signup` 이벤트를 구독하다가 발행될 때마다 후속 동작을 취할 수 있다. 예를 들어 트랜잭션 이메일 서비스는 새 사용자가 가입할 때마다 환영 이메일을 보낼 수 있다. 이 서비스를 JavaScript로 작성한다면 핸들러는 대략 이런 모습일 것이다.

```javascript
const handleEvent = ({ event_type, data }) => {
  switch (event_type) {
    case 'login':
      /* ... */
      break
    case 'signup':
      sendEmail(data.user.email, `Welcome to Blockchain Emporium, ${data.user.name}!`)
      break
  }
}
```

그런데 이 서비스를 Haskell로 작성한다면? [검증 대신 파싱하는](https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/), 현실을 경외하는 착한 Haskell 프로그래머라면 코드는 대략 이렇게 될 것이다.

```haskell
data Event = Login LoginPayload | Signup SignupPayload
data LoginPayload = LoginPayload { userId :: Int }
data SignupPayload = SignupPayload
  { userId :: Int
  , userName :: Text
  , userEmail :: Text }

instance FromJSON Event where
  parseJSON = withObject "Event" \obj -> do
    eventType <- obj .: "event_type"
    case eventType of
      "login" -> Login <$> (obj .: "data")
      "signup" -> Signup <$> (obj .: "signup")
      _ -> fail $ "unknown event_type: " <> eventType

instance FromJSON LoginPayload where { ... }
instance FromJSON SignupPayload where { ... }

handleEvent :: JSON.Value -> IO ()
handleEvent payload = case fromJSON payload of
  Success (Login LoginPayload { userId }) -> {- ... -}
  Success (Signup SignupPayload { userName, userEmail }) ->
    sendEmail userEmail $ "Welcome to Blockchain Emporium, " <> userName <> "!"
  Error message -> fail $ "could not parse event: " <> message
```

분명 보일러플레이트가 더 많지만, 타입 정의에 드는 약간의 추가 비용은 예상 범위 안이고(이렇게 작은 예제에서는 크게 과장돼 보인다), 어차피 지금 논의하는 논증은 보일러플레이트에 관한 게 아니다. 앞의 Reddit 댓글에 따르면 이 버전 코드의 _진짜_ 문제는, 어떤 서비스가 새 이벤트 타입을 추가할 때마다 Haskell 코드를 고쳐야 한다는 것이다! `Event` 데이터 타입에 새 케이스를 추가해야 하고, 새 파싱 로직도 붙여야 한다. 페이로드에 새 필드가 추가되면? 유지보수 지옥이 따로 없다.

그에 비해 JavaScript 코드는 훨씬 관대하다. 새 이벤트 타입이 추가되면 그냥 `switch`를 빠져나가 아무것도 하지 않는다. 페이로드에 필드가 추가돼도 JavaScript 코드는 그냥 무시한다. 동적 타이핑의 승리처럼 보인다.

그런데 아니다, 그렇지 않다. `Event` 타입을 갱신하지 않으면 정적 타입 프로그램이 실패하는 유일한 이유는 우리가 `handleEvent`를 그렇게 작성했기 때문이다. JavaScript 코드에서도 알 수 없는 이벤트 타입을 거부하는 default 케이스를 추가해서 똑같이 할 수 있었다.

```javascript
const handleEvent = ({ event_type, data }) => {
  switch (event_type) {
    /* ... */
    default:
      throw new Error(`unknown event_type: ${event_type}`)
  }
}
```

우리가 그렇게 하지 않은 이유는, 이 경우에는 그게 명백히 바보 같은 짓이기 때문이다. 서비스가 모르는 이벤트를 받으면 그냥 무시해야 한다. 여기서는 관대한 것이 명백히 올바른 동작이고, Haskell 코드에서도 쉽게 그렇게 구현할 수 있다.

```haskell
handleEvent :: JSON.Value -> IO ()
handleEvent payload = case fromJSON payload of
  {- ... -}
  Error _ -> pure ()
```

이것도 여전히 "파싱하라, 검증하지 말라"의 정신에 부합한다. 우리가 _신경 쓰는_ 값은 여전히 최대한 이른 시점에 파싱하고 있으므로 이중 검증의 함정에 빠지지 않는다. 값이 잘 형성돼 있음을 (타입 시스템의 도움으로) 먼저 확인하지 않고서, 그 값이 잘 형성돼 있다는 데 의존하는 코드 경로를 타는 일은 어느 시점에도 없다. 잘못 형성된 값에 반드시 오류로 대응해야 하는 게 아니다! 그것을 무시한다는 사실을 명시하기만 하면 된다.

이는 중요한 논점을 보여 준다. 이 Haskell 코드의 `Event` 타입은 "가능한 모든 이벤트"를 기술하는 게 아니라, 이 애플리케이션이 신경 쓰는 모든 이벤트를 기술한다. 마찬가지로 이벤트 페이로드를 파싱하는 코드는 애플리케이션에 필요한 필드만 신경 쓰고 그 외의 필드는 무시한다. 정적 타입 시스템은 우주 전체에 대한 스키마를 미리 작성하라고 요구하지 않는다. 그저 당신에게 필요한 것을 솔직하게 밝히라고 요구할 뿐이다.

입력에 대한 지식이 제한적인데도 여기에는 기분 좋은 이점이 많다.

- Haskell 프로그램의 가정은 타입 정의만 봐도 쉽게 알 수 있다. 예를 들어 이 애플리케이션이 `timestamp` 필드에 신경 쓰지 않는다는 걸 안다. 어떤 페이로드 타입에도 등장하지 않으니까. 동적 타입 프로그램에서는 그 필드를 들여다보는지 확인하려고 모든 코드 경로를 감사해야 하는데, 그건 실수하기 쉬운 대량의 작업이다!

- 게다가 이 Haskell 코드는 사실 `SignupPayload` 타입 안의 `userId` 필드를 _사용하지_ 않으므로 그 타입은 과하게 보수적이다. 정말 필요 없는지 확인하고 싶다면(예를 들어 그 페이로드에서 사용자 ID 제공을 아예 단계적으로 없애는 중이라면), 그 레코드 필드를 지우기만 하면 된다. 코드가 타입 검사를 통과하면 정말로 그 필드에 의존하지 않는다고 확신할 수 있다.

- 마지막으로, [이전 글에서 언급한](https://lexi-lambda.github.io/blog/2019/11/05/parse-don-t-validate/#the-danger-of-validation) 샷건 파싱 관련 함정을 모두 깔끔하게 피한다. 그 원칙들을 하나도 타협하지 않았기 때문이다.

이로써 주장의 전반부, 즉 정적 타입 언어는 구조가 완전히 알려지지 않은 데이터를 다룰 수 없다는 주장은 이미 무너뜨렸다. 이제 후반부를 보자. 동적 타입 언어는 구조를 전혀 모르는 데이터도 처리할 수 있다는 주장이다. 아직도 맞는 말 같을지 모르지만, 속도를 늦추고 더 곰곰이 생각해 보면 그럴 수 없다는 걸 알게 된다.

위 JavaScript 코드는 Haskell 코드와 정확히 같은 가정을 한다. 이벤트 페이로드가 `event_type` 필드를 가진 JSON 객체라고 가정하고, `signup` 페이로드에 `data.user.name`과 `data.user.email` 필드가 있다고 가정한다. 정말로 알 수 없는 입력으로는 아무 유용한 일도 할 수 없는 게 분명하다! 새 이벤트 페이로드가 추가된다고 해서, 동적 타입이라는 이유만으로 JavaScript 코드가 마법처럼 적응해서 처리하지는 못한다. 동적 타이핑이란 그저 값의 타입이 런타임에 값과 함께 다니며 프로그램이 실행되는 동안 검사된다는 뜻일 뿐이다. 타입은 여전히 거기 있고, 이 프로그램은 여전히 타입이 특정한 모습이라는 데 암묵적으로 의존한다.

## 불투명한 데이터는 불투명하게 유지하기

앞 절에서 정적 타입 시스템은 부분적으로 알려진 데이터를 처리할 수 없다는 생각을 논파했지만, 주의 깊게 읽었다면 원래 주장을 완전히 반박하지는 않았다는 걸 눈치챘을 것이다.

알 수 없는 데이터를 다루긴 했지만 언제나 그냥 버렸는데, 어떤 종류든 프록시를 구현하려 한다면 그걸로는 통하지 않는다. 예를 들어 이벤트를 공용 네트워크로 중계하면서 위조를 막기 위해 각 페이로드에 서명을 붙이는 포워딩 서비스가 있다고 하자. JavaScript로는 이렇게 구현할 수 있다.

```javascript
const handleEvent = (payload) => {
  const signedPayload = { ...payload, signature: signature(payload) }
  retransmitEvent(signedPayload)
}
```

이 경우 페이로드의 구조에는 전혀 신경 쓰지 않지만(`signature` 함수는 유효한 JSON 객체라면 뭐든 처리한다), 모든 정보를 보존해야 한다. 정적 타입 언어라면 페이로드에 정밀한 타입을 부여해야 할 텐데, 어떻게 이걸 할 수 있을까?

이번에도 답은 전제를 거부하는 데 있다. 애플리케이션에 필요한 것보다 더 정밀한 타입을 데이터에 부여할 필요는 없다. 같은 로직을 Haskell로 곧이곧대로 쓸 수 있다.

```haskell
handleEvent :: JSON.Value -> IO ()
handleEvent (Object payload) = do
  let signedPayload = Map.insert "signature" (signature payload) payload
  retransmitEvent signedPayload
handleEvent payload = fail $ "event payload was not an object " <> show payload
```

이 경우 페이로드의 구조에 신경 쓰지 않으므로 `JSON.Value` 타입의 값을 직접 조작한다. 이 타입은 앞의 `Event` 타입에 비해 극도로 부정밀하다. 어떤 형태든 합법적인 JSON 값이라면 뭐든 담을 수 있다. 하지만 이 경우에는 부정밀하기를 _원한다_.

그 부정밀함 덕분에 타입 시스템이 여기서도 우리를 도왔다. 우리가 페이로드를 다른 JSON 값이 아니라 JSON 객체라고 가정하고 있다는 사실을 잡아냈고, 객체가 아닌 케이스를 명시적으로 처리하게 만들었다. 여기서는 오류를 일으키기로 했지만, 물론 이전처럼 원한다면 다른 형태의 복구를 택할 수도 있다. 명시적으로 하기만 하면 된다.

다시 한번 주목하자. Haskell에서 명시하도록 강제당한 그 가정은 JavaScript 코드 _역시_ 하고 있다! JavaScript의 `handleEvent` 함수가 객체가 아니라 문자열로 호출되면 바람직한 동작이 나올 리 없다. 문자열에 객체 스프레드를 적용하면 다음과 같은 깜짝 선물이 나오니까.

```
> { ..."payload", signature: "sig" }
{0: "p", 1: "a", 2: "y", 3: "l", 4: "o", 5: "a", 6: "d", signature: "sig"}
```

이런. 이번에도 파싱 스타일의 프로그래밍이 우리를 구했다. `Object` 케이스에 명시적으로 매칭해서 JSON 값을 객체로 "파싱"하지 않았다면 코드가 컴파일되지 않았을 것이고, 폴스루 케이스를 빼먹었다면 불완전한 패턴에 대한 경고를 받았을 것이다.

* * *

넘어가기 전에 이 현상의 예를 하나만 더 보자. 사용자 ID를 반환하는 API를 소비하고 있고, 그 ID가 마침 UUID라고 하자. "파싱하라, 검증하지 말라"를 곧이곧대로 해석하면 Haskell API 클라이언트에서 사용자 ID를 `UUID` 타입으로 표현하자고 할 수 있겠다.

```haskell
type UserId = UUID
```

하지만 우리의 Reddit 댓글 작성자는 아마 여기에 발끈할 것이다! API 계약이 모든 사용자 ID가 UUID라고 명시하지 않는 한, 이 표현은 우리 권한을 넘어선다. 사용자 ID가 오늘은 UUID일지 몰라도 내일은 아닐 수 있고, 그러면 우리 코드는 아무 이유 없이 깨진다! 이게 정적 타입 시스템의 잘못인가?

역시 답은 아니오다. 이건 부적절한 데이터 모델링의 사례지만 정적 타입 시스템의 잘못이 아니다. 그저 잘못 사용됐을 뿐이다. `UserId`를 표현하는 적절한 방법은 새로운 불투명 타입을 정의하는 것이다.

```haskell
newtype UserId = UserId Text
  deriving (Eq, FromJSON, ToJSON)
```

기존 `UUID` 타입에 새 이름만 붙이는 위의 타입 별칭과 달리, 이 선언은 `Text`를 포함한 다른 모든 타입과 구별되는 완전히 새로운 `UserId` 타입을 만든다. 데이터 타입의 생성자를 비공개로 유지하면(즉 이 타입을 정의하는 모듈에서 내보내지 않으면), `UserId`를 만들어 내는 _유일한_ 방법은 `FromJSON` 파서를 거치는 것뿐이다. 쌍대적으로, `UserId`로 할 수 있는 일은 다른 `UserId`와 동등성을 비교하거나 `ToJSON` 인스턴스로 직렬화하는 것뿐이다. 그 외에는 아무것도 허용되지 않는다. 타입 시스템이 원격 서비스의 사용자 ID 내부 표현에 의존하지 못하게 막아 준다.

이는 완전히 불투명한 데이터를 조작할 때도 정적 타입 시스템이 강력하고 유용한 보장을 제공할 수 있는 또 다른 방식을 보여 준다. `UserId`의 런타임 표현은 사실 그냥 문자열이지만, 타입 시스템은 그것을 실수로 문자열처럼 쓰는 것도, 임의의 문자열에서 새 `UserId`를 허공에서 위조하는 것도 허용하지 않는다.[^1]

타입 시스템은 프로그램에 드나드는 모든 값의 표현을 시시콜콜 기술하도록 강요하는 족쇄가 아니다. 오히려 필요에 가장 잘 맞는 방식으로 쓸 수 있는 도구다.

## 리플렉션은 특별하지 않다

첫 번째 댓글의 주장은 이제 철저히 논파했지만, 두 번째 댓글이 던진 질문은 여전히 우리 논리의 허점처럼 보일 수 있다. Python의 `pickle.load()`의 타입은 무엇 _인가_? 모르는 사람을 위해 설명하면, [귀엽게 이름 붙은 Python의 `pickle` 라이브러리](https://docs.python.org/3/library/pickle.html)는 Python 객체 그래프 전체를 직렬화하고 역직렬화할 수 있게 해 준다. 어떤 객체든 `pickle.dump()`로 직렬화해 파일에 저장했다가 나중에 `pickle.load()`로 역직렬화할 수 있다.

이게 정적 타입 시스템에 어려워 보이는 이유는 `pickle.load()`가 만들어 내는 값의 타입을 예측하기 어렵다는 데 있다. 그건 전적으로 그 파일에 `pickle.dump()`로 무엇이 쓰였느냐에 달려 있다. 컴파일 타임에 어떤 타입의 값이 나올지 도무지 알 수 없으니 본질적으로 동적인 것처럼 보인다. 얼핏 보면 동적 타입 시스템은 해낼 수 있는데 정적 타입 시스템은 그냥 못 하는 일 같다.

하지만 알고 보면 이 상황은 앞의 JSON 예제들과 사실상 동일하며, Python의 피클링이 네이티브 Python 객체를 직접 직렬화한다는 사실도 상황을 바꾸지 않는다. 왜일까? 프로그램이 `pickle.load()`를 호출한 _다음에_ 무슨 일이 일어나는지 생각해 보자. 다음 함수를 작성했다고 하자.

```python
def load_value(f):
  val = pickle.load(f)
  # `val`로 무언가를 한다
```

문제는 이제 `val`이 _어떤_ 타입이든 될 수 있다는 것이다. 정말로 알 수 없는 비구조화 입력으로는 아무 유용한 일도 할 수 없듯이, 값에 대해 최소한 뭔가는 알지 못하면 그 값으로 아무것도 할 수 없다. 결과에 대해 어떤 메서드를 호출하거나 어떤 필드에 접근하는 순간, `pickle.load(f)`가 어떤 종류의 것을 반환했는지에 대한 가정을 이미 한 것이다. 그리고 알고 보면 그 가정들이 _바로_ `val`의 타입이다!

예를 들어 `val`로 하는 유일한 일이 `val.foo()` 메서드를 호출해 그 결과(문자열이라고 기대되는)를 반환하는 것이라고 상상해 보자. Java를 쓰고 있었다면 `val`의 기대 타입은 아주 명료하다. 다음 인터페이스의 인스턴스일 것으로 기대하는 것이다.

```java
interface Foo extends Serializable {
  String foo();
}
```

실제로 `pickle.load()` 같은 함수에는 Java에서 완벽하게 합리적인 타입을 줄 수 있다.

```java
static <T extends Serializable> Optional<T> load(InputStream in, Class<? extends T> cls);
```

트집 잡기 좋아하는 사람은 원하는 타입을 미리 고르는 `Class<T>` 토큰을 넘겨야 하니 `pickle.load()`와 같지 않다고 불평할 것이다. 하지만 `Serializable.class`를 넘기고 객체가 로드된 뒤에 타입으로 분기하는 것을 막는 건 아무것도 없다. 그리고 그게 핵심이다. 객체로 _무엇이든_ 하는 순간, 동적 타입 언어에서조차 그 타입에 대해 뭔가를 알아야 한다! 정적 타입 언어는 그걸 더 명시적으로 하라고 강제할 뿐이다. JSON 페이로드 이야기를 할 때와 똑같이.

* * *

Haskell에서도 이걸 할 수 있을까? 물론이다. 위에서 언급한 Java API와 비슷한 API를 가진 [`serialise` 라이브러리](https://hackage.haskell.org/package/serialise)를 쓰면 된다. 이 라이브러리는 [Haskell JSON 라이브러리 aeson](https://hackage.haskell.org/package/aeson)과도 인터페이스가 아주 비슷한데, 알 수 없는 JSON 데이터를 다루는 문제가 알 수 없는 Haskell 값을 다루는 문제와 크게 다르지 않기 때문이다. 어느 시점에는 값으로 뭔가를 하기 위해 약간의 파싱을 해야 한다.

그렇긴 해도, 타입 검사를 가능한 마지막 순간까지 미루는 방식으로 `pickle.load()`의 동적 타이핑을 정말 원한다면 흉내 낼 수 _있지만_, 현실에서 그렇게 하는 게 실제로 유용한 경우는 거의 없다. 어느 시점에는 값을 사용하기 위해 그 구조에 대한 가정을 해야 하고, 그 가정이 무엇인지는 당신이 안다. _당신이 그 코드를 썼으니까_. 진짜 동적 코드 로딩이 필요한 극히 드문 예외(이를테면 프로그래밍 언어의 REPL 구현)가 있긴 하지만, 일상적인 프로그래밍에서는 나타나지 않으며, 정적 타입 언어의 프로그래머들은 자신의 가정을 미리 밝히는 데 아무 불만이 없다.

이것이 정적 타이핑 진영과 동적 타이핑 진영 사이의 근본적인 단절 중 하나다. 동적 타입 언어에서는 할 수 있는데 정적 타입 언어가 "근본적으로" 막는 일이 있다고 누군가 말하면, 정적 타입 언어에서 일하는 프로그래머는 어리둥절해한다. 그저 값에 충분히 정밀한 타입이 부여되지 않았을 뿐이라고 답할 테니까. 동적 타입 언어 프로그래머의 관점에서 타입 시스템은 합법적인 동작의 공간을 제한하지만, 정적 타입 언어 프로그래머의 관점에서는 합법적인 동작의 집합이 _곧_ 값의 타입이다.

적절한 시각에서 보면 두 관점 모두 부정확하지 않다. 정적 타입 시스템은 실제로 프로그램 구조에 제약을 부과한다. 튜링 완전한 언어에서 좋은 프로그램을 일부 거부하지 않으면서 _모든_ 나쁜 프로그램을 거부하는 것은 증명 가능하게 불가능하기 때문이다([Rice의 정리](https://en.wikipedia.org/wiki/Rice's_theorem)다). 하지만 일반 문제를 풀 수 없다는 사실이 약간 제한된 버전의 문제를 유용하게 푸는 것까지 배제하지는 않는다는 것도 동시에 참이며, 정적 타입 시스템의 이른바 "근본적" 무능력 중 상당수는 전혀 근본적이지 않다.

## 부록: 신화 뒤의 현실

이 글의 핵심 논지는 이제 전달됐다. 정적 타입 시스템은 열려 있거나 부분적으로만 알려진 구조의 데이터를 처리하는 데 동적 타입 시스템보다 근본적으로 열등하지 않다. 글 앞머리에서 인용한 댓글들의 주장은 정적 타입 프로그램 구성이 실제로 어떤 모습인지 정확하게 그리지 못했고, 정적 타이핑 규율의 한계를 오해하는 동시에 동적 타이핑 규율의 능력을 과장했다.

하지만 크게 과장되긴 했어도 이 신화들에는 현실에 뿌리가 어느 정도 있다. 적어도 부분적으로는 구조적 타이핑(structural typing)과 명목적 타이핑(nominal typing)의 차이에 대한 오해에서 자라난 것으로 보인다. 이 차이는 안타깝게도 이 글에서 다루기엔 너무 커서, 그것만으로 글 여러 편을 채울 수 있을 것이다. 약 6개월 전에 이 주제로 글을 쓰려 했지만 별로 설득력 있게 나오지 않아 폐기했다. 언젠가 이 아이디어를 전달할 더 나은 방법을 찾을지도 모르겠다.

지금 이 주제를 온당한 만큼 충분히 다룰 수는 없지만, 관심 있는 독자가 다른 자료를 찾아볼 수 있도록 짧게나마 짚고 넘어가고 싶다. 핵심 아이디어는, 많은 동적 타입 언어가 정적 타입 언어에서라면 대개 맞춤 데이터 타입(보통 클래스나 구조체로 정의되는)으로 표현할 것을 해시맵 같은 단순한 데이터 구조를 재사용해 관용적으로 표현한다는 것이다.

이 두 스타일은 매우 다른 맛의 프로그래밍을 낳는다. JavaScript나 Clojure 프로그램은 레코드를 문자열 또는 심벌 키에서 값으로 가는 해시맵으로 표현하고, 객체 리터럴이나 해시 리터럴로 작성하며, 키와 값을 범용적으로 조작하는 표준 라이브러리의 평범한 함수들로 다룬다. 덕분에 두 레코드를 받아 필드를 합치거나, 기존 레코드에서 필드의 임의(심지어 동적인) 부분 선택을 취하는 게 간단하다.

반면 대부분의 정적 타입 시스템은 레코드의 그런 자유로운 조작을 허용하지 않는다. 레코드가 맵이 아니라 다른 모든 타입과 구별되는 고유한 타입이기 때문이다. 이 타입들은 (완전히 한정된) 이름으로 유일하게 식별된다. 그래서 _명목적 타이핑_ 이라 부른다. 구조체 필드의 부분 선택을 취하고 싶으면 완전히 새로운 구조체를 정의해야 하고, 이러다 보면 어색한 보일러플레이트가 폭발하곤 한다.

이것이 Rich Hickey가 정적 타이핑을 비판하는 여러 강연에서 논해 온 주된 아이디어 중 하나다. 그는 레코드를 유동적으로 병합하고 분리하고 변환하는 이 능력이 동적 타이핑을 분산되고 열린 시스템 도메인에 특히 적합하게 만든다는 생각을 밀어 왔다. 안타깝게도 이 수사에는 중대한 결함이 둘 있다.

1. 이것을 타입 시스템의 근본적 한계라고 부르는 데 너무 가까이 다가간다. 명목적인 정적 타입 시스템에서 그런 시스템을 모델링하는 게 단지 불편한 게 아니라 _불가능_ 하다고 시사하는 것이다. 이건 사실이 아닐 뿐 아니라(이 글이 보였듯이), 그의 주장 중 실제로 가치 있는 논점, 즉 더 구조적인 데이터 모델링 접근이 갖는 실용적이고 실질적인 이점에서 사람들의 시선을 돌려 버린다.

2. 구조적/명목적 구분을 동적/정적 구분과 혼동해서, 키-값 맵으로 표현된 레코드의 유동적인 병합과 분리가 동적 타입 언어에서만 가능하다는 잘못된 인상을 만든다. 사실 정적 타입 언어가 구조적 타이핑을 지원할 수 있을 뿐 아니라, 많은 동적 타입 언어도 명목적 타이핑을 지원한다. 이 두 축은 역사적으로 느슨하게 상관관계가 있었지만 이론적으로는 직교한다.

이 주장들의 반례로, 동적이면서도 꽤 명목적인 Python 클래스와, 정적이면서도 구조적인 TypeScript 인터페이스를 생각해 보라. 실제로 현대 정적 타입 언어들은 구조적으로 타입이 지정되는 레코드에 대한 네이티브 지원을 점점 갖춰 가고 있다. 이런 시스템에서 레코드 타입은 Clojure의 해시와 아주 비슷하게 동작한다. 구별되는 이름 있는 타입이 아니라 익명의 키-값 쌍 모음이며, Clojure의 해시가 지원하는 표현력 있는 조작 연산의 상당수를 정적 타입의 틀 안에서 그대로 지원한다.

구조적 타이핑을 강력하게 지원하는 정적 타입 시스템을 탐험해 보고 싶다면 TypeScript, Flow, PureScript, Elm, OCaml, Reason 중 아무거나 살펴보길 권한다. 모두 구조적으로 타입이 지정되는 레코드를 어떤 형태로든 지원한다. 이 목적으로 권하지 _않는_ 것은 Haskell이다. Haskell의 구조적 타이핑 지원은 형편없다. Haskell은 (이 글의 범위를 벗어나는 여러 이유로) 공격적으로 명목적이다.[^2]

그렇다면 Haskell이 나쁘다는 뜻인가, 아니면 이런 종류의 문제를 실용적으로 풀 수 없다는 뜻인가? 아니, 절대 그렇지 않다. Haskell에서 이런 문제를 충분히 잘 모델링하는 방법은 많다. 다만 그중 일부는 상당한 보일러플레이트에 시달린다. 이 글의 핵심 논지는 위에서 언급한 다른 언어들과 마찬가지로 Haskell에도 그대로 적용된다. 그래도 이 구분을 언급하지 않으면 태만일 것이다. 동적 타입 배경을 가진 프로그래머들이 역사적으로 정적 타입 언어를 훨씬 답답하게 느껴 왔다면, 그렇게 느끼는 _진짜_ 이유를 더 잘 이해하는 데 도움이 될 수 있으니까. (사실상 모든 주류 정적 타입 OOP 언어는 Haskell보다도 더 명목적이다!)

마무리하며. 이 글은 성전을 시작하려는 것도, 동적 타입 프로그래밍에 대한 공격도 아니다. 동적 타입 언어에는 정적 타입 맥락으로 옮기기가 정말로 어려운 패턴이 많고, 그런 패턴에 대한 논의는 생산적일 수 있다고 생각한다. 이 글의 목적은 왜 특정한 한 가지 논의가 생산적이지 _않은지_ 를 분명히 하는 것이다. 그러니 부탁한다. 이런 논증은 이제 그만하자. 타이핑에 대해서라면 이것보다 훨씬 생산적인 대화가 많다.

[^1]: 엄밀히 말하면 `FromJSON` 인스턴스를 악용해 임의의 문자열을 `UserId`로 변환할 수는 있다. 하지만 이건 들리는 것만큼 쉽지 않다. `fromJSON`은 실패할 수 있기 때문이다. 즉 그 실패 케이스를 어떻게든 처리해야 하므로, 이미 입력 파싱을 하고 있는 맥락이 아니라면 이 꼼수로는 멀리 가지 못할 것이다… 그리고 그런 맥락이라면 그냥 올바른 방법을 쓰는 게 더 쉽다. 그러니 그렇다, 타입 시스템은 일부러 제 발등을 찍으러 가는 것까지 막지는 못하지만 올바른 해법으로 인도해 준다(그리고 스스로 삶을 비참하게 만들기로 작정한 프로그래머를 완벽하게 보호할 수 있는 안전장치는 세상에 존재하지 않는다).

[^2]: 이 글을 쓰는 시점에 나는 이것이 Haskell의 가장 중대한 결함이라고 생각한다.
