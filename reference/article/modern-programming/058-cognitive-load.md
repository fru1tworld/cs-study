## 들어가며

세상에는 수많은 유행어와 베스트 프랙티스가 있지만, 대부분 실패했다. 실패한 이유는 그것들이 현실이 아니라 상상에 기반했기 때문이다. 미적 감각과 주관적 판단 위에 세워진 아이디어들이었다. 우리에게는 더 근본적인 것, 틀릴 수가 없는 것이 필요하다.

코드를 읽다 보면 혼란을 느낄 때가 있다. 혼란은 시간과 돈을 잡아먹는다. 혼란의 원인은 높은 *인지 부하(cognitive load)*다. 인지 부하는 무슨 거창한 추상 개념이 아니라 **인간의 근본적인 제약**이다. 상상의 산물이 아니라 실재하며, 우리가 직접 느낄 수 있는 것이다.

코드를 쓰는 시간보다 읽고 이해하는 시간이 훨씬 길기 때문에, 우리는 코드에 과도한 인지 부하를 심고 있지는 않은지 끊임없이 자문해야 한다.

**LLM이 생성한 대량의 코드를 우리 자신의 두뇌로 처리해야 하는 AI 시대에는 특히 중요하다.**

## 인지 부하

> 인지 부하란 개발자가 과제를 완수하기 위해 생각해야 하는 양이다.

코드를 읽을 때 우리는 변수 값, 제어 흐름 로직, 호출 순서 같은 것들을 머릿속에 담는다. 평균적인 사람이 작업 기억(working memory)에 담을 수 있는 청크는 [대략 네 개](https://github.com/zakirullin/cognitive-load/issues/16)다. 인지 부하가 이 한계에 도달하면 이해하기가 훨씬 어려워진다.

*완전히 낯선 프로젝트에 수정 몇 가지를 해달라는 요청을 받았다고 하자. 아주 똑똑한 개발자가 기여했던 프로젝트라고 한다. 멋진 아키텍처, 화려한 라이브러리, 유행하는 기술이 잔뜩 쓰였다. 다시 말해, **원작자는 우리에게 높은 인지 부하를 만들어 놓은 것이다.***

## 인지 부하의 종류

**내재적(intrinsic) 부하** - 과제 자체의 본질적 난이도에서 온다. 줄일 수 없으며, 소프트웨어 개발의 핵심 그 자체다.

**외재적(extraneous) 부하** - 정보가 제시되는 방식에서 온다. 과제와 직접 관련 없는 요인, 예컨대 똑똑한 원작자의 기벽 같은 것이 원인이다. 크게 줄일 수 있다. 우리는 이 유형의 인지 부하에 집중한다.

인지 부하의 수준은 다음과 같이 표기한다:
`🧠`: 작업 기억이 비어 있는 상태, 인지 부하 0
`🧠++`: 작업 기억에 사실 두 개, 인지 부하 증가
`🤯`: 인지 과부하, 사실 4개 초과

> 우리 뇌는 훨씬 복잡하고 아직 미지의 영역이지만, 이 단순한 모델로도 충분하다.

## 복잡한 조건문

```go
if val > someConstant // 🧠+
    && (condition2 || condition3) // 🧠+++, 앞 조건이 참이어야 하고, c2와 c3 중 하나가 참이어야 한다
    && (condition4 && !condition5) { // 🤯, 이 시점에서 이미 뒤죽박죽이다
    ...
}
```

의미 있는 이름을 가진 중간 변수를 도입하자:

```go
isValid = val > someConstant
isAllowed = condition2 || condition3
isSecure = condition4 && !condition5 
// 🧠, 조건들을 기억할 필요가 없다. 서술적인 변수가 있으니까
if isValid && isAllowed && isSecure {
    ...
}
```

## 중첩 if

```go
if isValid { // 🧠+, 좋아, 중첩된 코드는 유효한 입력에만 적용되는군
    if isSecure { // 🧠++, 유효하고 안전한 입력에 대해서만 작업을 하는군
        stuff // 🧠+++
    }
} 
```

조기 반환(early return)과 비교해 보라:

```go
if !isValid
    return
 
if !isSecure
    return

// 🧠, 앞의 return들은 신경 쓸 필요 없다. 여기까지 왔다면 전부 통과한 것이다

stuff // 🧠+
```

해피 패스에만 집중할 수 있으므로, 작업 기억을 온갖 전제 조건에서 해방시킬 수 있다.

## 상속의 악몽

관리자(admin) 사용자를 위해 몇 가지를 바꿔 달라는 요청을 받았다: `🧠`

`AdminController extends UserController extends GuestController extends BaseController`

이런, 기능 일부가 `BaseController`에 있네, 들여다보자: `🧠+`
기본 역할(role) 메커니즘은 `GuestController`에서 도입됐군: `🧠++`
`UserController`에서 일부가 변경됐고: `🧠+++`
드디어 도착, `AdminController`. 이제 코딩하자! `🧠++++`

잠깐, `AdminController`를 상속하는 `SuperuserController`가 있잖아. `AdminController`를 수정하면 상속받은 클래스가 깨질 수 있으니 `SuperuserController`부터 파고들자: `🤯`

상속보다 컴포지션을 우선하라. 자세한 설명은 생략한다 - [자료는 얼마든지 있다](https://www.youtube.com/watch?v=hxGOiiR9ZKg).

## 너무 많은 작은 메서드, 클래스, 모듈

> 이 맥락에서 메서드, 클래스, 모듈은 서로 바꿔 쓸 수 있는 말이다.

"메서드는 15줄보다 짧아야 한다"거나 "클래스는 작아야 한다" 같은 주문(mantra)은 다소 틀린 것으로 드러났다.

**깊은(deep) 모듈** - 단순한 인터페이스, 복잡한 기능
**얕은(shallow) 모듈** - 제공하는 작은 기능에 비해 상대적으로 복잡한 인터페이스

<div align="center">
  <img src="https://raw.githubusercontent.com/zakirullin/cognitive-load/main/img/deepmodulev8.png" alt="Deep module" width="700">
</div>

얕은 모듈이 너무 많으면 프로젝트를 이해하기 어려워진다. **각 모듈의 책임뿐 아니라 그들 사이의 모든 상호작용까지 머릿속에 담아야 하기 때문이다.** 얕은 모듈의 목적을 이해하려면 먼저 관련된 모든 모듈의 기능부터 살펴야 한다. 그런 얕은 컴포넌트 사이를 오가는 일은 정신적으로 지치는 일이며, 인간에게는 [선형적 사고](https://blog.separateconcerns.com/2023-09-11-linear-code.html)가 더 자연스럽다.

> 정보 은닉이 무엇보다 중요한데, 얕은 모듈에는 복잡성이 충분히 숨겨지지 않는다.

나에게는 개인 프로젝트가 두 개 있는데, 둘 다 대략 5천 줄 규모다. 첫 번째는 얕은 클래스 80개로, 두 번째는 [깊은 클래스 7개](https://github.com/zakirullin/files.md/blob/main/server/bot.go)로 되어 있다. 두 프로젝트 모두 1년 반 동안 손을 놓고 있었다.

돌아와 보니, 첫 번째 프로젝트에서 그 80개 클래스 사이의 상호작용을 전부 풀어내는 것은 극도로 어려웠다. 코딩을 시작하기 전에 어마어마한 인지 부하를 다시 쌓아 올려야 했을 것이다. 반면 두 번째 프로젝트는 빠르게 파악할 수 있었다. 단순한 인터페이스를 가진 깊은 클래스가 몇 개뿐이었기 때문이다.

> 최고의 컴포넌트는 강력한 기능을 제공하면서도 단순한 인터페이스를 가진 컴포넌트다.
>
> *존 오스터하우트(John Ousterhout), A Philosophy of Software Design*

Unix I/O의 인터페이스는 매우 단순하다. 기본 호출이 다섯 개뿐이다:

```python
open(path, flags, permissions)
read(fd, buffer, count)
write(fd, buffer, count)
lseek(fd, offset, referencePosition)
close(fd)
```

이 인터페이스의 현대적 구현은 **수십만 줄의 코드**다. 엄청난 복잡성이 밑에 숨어 있다. 그런데도 단순한 인터페이스 덕분에 쓰기는 쉽다.

> 이 깊은 모듈 예시는 존 오스터하우트의 책 [A Philosophy of Software Design](https://web.stanford.edu/~ouster/cgi-bin/book.php)에서 가져왔다. 이 책은 소프트웨어 개발에서 복잡성의 본질을 다룰 뿐 아니라, 파나스(Parnas)의 영향력 있는 논문 [On the Criteria To Be Used in Decomposing Systems into Modules](https://www.win.tue.nl/~wstomv/edu/2ip30/references/criteria_for_modularization.pdf)에 대한 최고의 해석을 담고 있다. 둘 다 필독서다. 관련 읽을거리: [A Philosophy of Software Design vs Clean Code](https://github.com/johnousterhout/aposd-vs-clean-code), [It's probably time to stop recommending Clean Code](https://qntm.org/clean), [Small Functions considered Harmful](https://copyconstruct.medium.com/small-functions-considered-harmful-91035d316c29).

## 한 가지에만 책임진다는 것

우리는 너무나 자주 "모듈은 단 한 가지 일에만 책임져야 한다"는 모호한 원칙을 따르다가 얕은 모듈을 잔뜩 만들고 만다. 그 흐릿한 "한 가지"란 대체 무엇인가? 객체 하나를 인스턴스화하는 것도 한 가지 일이긴 하다. 그렇다면 [MetricsProviderFactoryFactory](https://minds.md/benji/frameworks)도 괜찮다는 얘기가 된다. **이런 클래스들은 이름과 인터페이스가 구현 전체보다 정신적으로 더 부담스러운데, 대체 이게 무슨 추상화인가?** 뭔가 잘못됐다.

우리는 사용자와 이해관계자를 만족시키기 위해 시스템을 변경한다. 우리는 그들에게 책임을 진다.

> 모듈은 단 하나의 사용자 또는 이해관계자에 대해서만 책임져야 한다.

단일 책임 원칙(Single Responsibility Principle)의 본뜻이 바로 이것이다. 쉽게 말해, 한 곳에 버그를 심었는데 서로 다른 비즈니스 담당자 두 명이 항의하러 온다면 원칙을 위반한 것이다. 모듈 안에서 몇 가지 일을 하느냐와는 아무 상관이 없다.

하지만 이렇게 이해해도, 이 규칙은 득보다 실이 클 수 있다. 이 원칙은 사람 수만큼 다양하게 해석될 수 있다. 더 나은 접근은 이 모든 것이 얼마나 많은 인지 부하를 만드는지를 보는 것이다. 한 곳의 변경이 여러 비즈니스 흐름에 걸쳐 연쇄 반응을 일으킬 수 있다는 사실을 기억해야 한다는 것 자체가 정신적으로 부담스럽다. 그게 전부다. 배워야 할 현란한 용어 같은 건 없다.

## 너무 많은 얕은 마이크로서비스

이 얕은-깊은 모듈 원칙은 규모와 무관하며, 마이크로서비스 아키텍처에도 적용할 수 있다. 얕은 마이크로서비스가 너무 많으면 아무 도움이 되지 않는다. 업계는 어느 정도 "매크로서비스", 즉 그리 얕지 않은(=깊은) 서비스 쪽으로 향하고 있다. 가장 악질적이고 고치기 어려운 현상 중 하나가 이른바 분산 모놀리스(distributed monolith)인데, 이는 지나치게 잘게 쪼갠 얕은 분리의 결과인 경우가 많다.

한번은 다섯 명짜리 개발팀이 마이크로서비스를 17개(!)나 도입한 스타트업을 컨설팅했다. 그들은 일정보다 10개월 뒤처져 있었고 공개 출시는 요원해 보였다. 새 요구사항 하나마다 4개 이상의 마이크로서비스에 변경이 필요했다. 그런 분산 시스템에서 문제를 재현하고 디버깅하는 데 엄청난 시간이 들었다. 출시 시간과 인지 부하 모두 용납할 수 없는 수준이었다. `🤯`

이것이 새 시스템의 불확실성에 접근하는 옳은 방법일까? 초기에 올바른 논리적 경계를 찾아내는 것은 엄청나게 어렵다. 핵심은 책임질 수 있는 한 최대한 늦게 결정을 내리는 것이다. 그때가 가장 많은 정보를 손에 쥐고 있는 시점이기 때문이다. 네트워크 계층을 처음부터 도입하면, 설계 결정을 처음부터 되돌리기 어렵게 만드는 셈이다. 그 팀의 유일한 근거는 "FAANG 기업들이 마이크로서비스 아키텍처의 효과를 증명했다"는 것이었다. *이봐요, 꿈 좀 그만 크게 꿔요.*

[타넨바움-토르발스 논쟁](https://en.wikipedia.org/wiki/Tanenbaum%E2%80%93Torvalds_debate)에서는 리눅스의 모놀리식 설계가 결함이 있고 낡았으며 마이크로커널 아키텍처를 써야 한다는 주장이 나왔다. 실제로 마이크로커널 설계는 "이론적, 미학적" 관점에서 우월해 보였다. 그러나 실용의 영역에서는 - 30년이 지난 지금도 마이크로커널 기반 GNU Hurd는 아직 개발 중이고, 모놀리식 리눅스는 어디에나 있다. 이 페이지도 리눅스로 돌아가고, 당신의 스마트 티포트도 리눅스로 돌아간다. 모놀리식 리눅스로.

## 비즈니스 로직과 HTTP 상태 코드

백엔드에서 이렇게 반환한다고 하자:
`401`은 만료된 JWT 토큰
`403`은 접근 권한 부족
`418`은 차단된 사용자

프런트엔드 엔지니어들은 로그인 기능을 구현하려고 백엔드 API를 사용한다. 그러면 다음과 같은 인지 부하를 머릿속에 임시로 만들어야 한다:
`401`은 만료된 JWT 토큰 // `🧠+`, 그래, 일단 임시로 기억해 두자
`403`은 접근 권한 부족 // `🧠++`
`418`은 차단된 사용자 // `🧠+++`

프런트엔드 개발자들은 (바라건대) 자기 쪽에 `숫자 상태 -> 의미` 사전 같은 것을 도입해서, 이후 세대의 기여자들이 이 매핑을 머릿속에서 다시 만들지 않아도 되게 할 것이다.

그다음엔 QA 엔지니어들이 등장한다:
"저기요, `403` 상태를 받았는데, 이거 만료된 토큰인가요, 권한 부족인가요?"
**QA 엔지니어들은 곧바로 테스트에 들어갈 수 없다. 백엔드 엔지니어들이 예전에 만들어 놓은 인지 부하를 먼저 재구성해야 하기 때문이다.**

왜 이런 커스텀 매핑을 작업 기억에 담아 두어야 하나? 비즈니스 세부사항을 HTTP 전송 프로토콜에서 분리하고, 응답 본문에 자기 서술적인(self-descriptive) 코드를 직접 반환하는 편이 낫다:

```json
{
    "code": "jwt_has_expired"
}
```

프런트엔드 쪽 인지 부하: `🧠` (신선한 상태, 머릿속에 담아 둘 사실 없음)
QA 쪽 인지 부하: `🧠`

같은 규칙이 (데이터베이스든 어디든) 온갖 숫자 상태값에 적용된다 - **자기 자신을 설명하는 문자열을 선호하라.** 우리는 메모리를 아끼려고 애쓰던 640K 컴퓨터 시대에 살고 있지 않다.

> 사람들은 자신의 멘털 모델에 근거해 `401`이냐 `403`이냐를 놓고 시간을 들여 논쟁한다. 새 개발자들이 들어오면 그 사고 과정을 다시 재구성해야 한다. 코드에 대한 "왜"를 문서화(ADR)해 두어 신규 인력이 그 결정을 이해하도록 도울 수도 있다. 하지만 결국 그건 별 의미가 없다. 오류를 사용자 관련이냐 서버 관련이냐로 나눌 수는 있어도, 그 이상은 흐릿하기 마련이다.

추신: "인증(authentication)"과 "인가(authorization)"를 구별하는 것도 정신적으로 부담스러운 일이다. ["로그인(login)"과 "권한(permissions)"](https://ntietz.com/blog/lets-say-instead-of-auth/) 같은 더 쉬운 용어를 써서 인지 부하를 줄일 수 있다.

## DRY 원칙 남용

반복하지 말라(Do not repeat yourself) - 소프트웨어 엔지니어가 되면서 처음 배우는 원칙 중 하나다. 이 원칙은 우리 안에 너무 깊이 새겨져서, 몇 줄의 중복 코드조차 견디지 못하게 된다. 대체로 훌륭하고 근본적인 규칙이지만, 과용하면 감당할 수 없는 인지 부하로 이어진다.

요즘은 누구나 논리적으로 분리된 컴포넌트 위에 소프트웨어를 만든다. 그 컴포넌트들이 별도 서비스를 이루는 여러 코드베이스에 분산되어 있는 경우도 많다. 모든 반복을 없애려고 애쓰다 보면, 서로 무관한 컴포넌트들 사이에 강한 결합을 만들어 버릴 수 있다. 그 결과 한 부분의 변경이 겉보기에 무관한 다른 영역에 의도치 않은 결과를 낳을 수 있다. 시스템 전체에 영향을 주지 않고 개별 컴포넌트를 교체하거나 수정하는 능력도 저해된다. `🤯`

사실 같은 문제는 단일 모듈 안에서도 생긴다. 장기적으로는 실재하지 않을지도 모르는, 겉보기에만 비슷한 점들에 근거해 공통 기능을 너무 일찍 추출할 수 있다. 그 결과 수정하거나 확장하기 어려운 불필요한 추상화가 생긴다.

롭 파이크는 이렇게 말했다:

> 약간의 복사가 약간의 의존성보다 낫다.

우리는 바퀴를 다시 발명하지 않으려는 유혹이 너무 강해서, 직접 쉽게 짤 수 있는 작은 함수 하나를 쓰자고 크고 무거운 라이브러리를 통째로 들여올 준비가 되어 있다.

**당신의 모든 의존성은 당신의 코드다.** 들여온 라이브러리의 10단계가 넘는 스택 트레이스를 헤집으며 뭐가 잘못됐는지 알아내는 일은(*왜냐하면 일은 잘못되기 마련이니까*) 고통스럽다.

## 프레임워크와의 강한 결합

프레임워크에는 "마법"이 많다. 프레임워크에 지나치게 의존하면, **앞으로 올 모든 개발자가 그 "마법"부터 배우도록 강제하는 셈이다.** 몇 달이 걸릴 수도 있다. 프레임워크 덕에 MVP를 며칠 만에 출시할 수 있다 해도, 장기적으로는 불필요한 복잡성과 인지 부하를 더하는 경향이 있다.

더 나쁜 것은, 어느 시점에 아키텍처에 도무지 들어맞지 않는 새 요구사항이 나타나면 프레임워크가 심각한 제약이 될 수 있다는 점이다. 그때부터 사람들은 프레임워크를 포크해서 자기들만의 커스텀 버전을 유지하게 된다. 신규 인력이 어떤 가치라도 만들어 내려면 쌓아야 할 인지 부하(즉, 이 커스텀 프레임워크 학습)가 얼마나 될지 상상해 보라. `🤯`

**모든 것을 처음부터 발명하자고 주장하는 게 결코 아니다!**

코드를 어느 정도 프레임워크에 독립적인(framework-agnostic) 방식으로 쓸 수 있다. 비즈니스 로직이 프레임워크 안에 살아서는 안 된다. 오히려 프레임워크의 컴포넌트를 사용해야 한다. 프레임워크를 핵심 로직 바깥에 두라. 프레임워크를 라이브러리처럼 사용하라. 그러면 새 기여자들이 프레임워크 관련 복잡성의 잔해를 먼저 헤치고 나올 필요 없이 첫날부터 가치를 더할 수 있다.

</div>

이 아키텍처는 처음에는 직관적으로 그럴듯했지만, 프로젝트에 적용할 때마다 득보다 실이 컸다. 우리는 명확한 비즈니스 가치가 없는 불필요한 정신 활동과 쓸모없는 접착 코드 작성에 몇 년을 썼다. 오히려 신규 인력이 우리의 접근법(멘털 모델)부터 배우도록 강제해 비즈니스에 해를 끼쳤다. 출시 시간은 나빠졌다. 결국 우리는 전부 포기하고 좋았던 옛 의존성 역전 원칙으로 돌아갔다. **배워야 할 포트/어댑터 용어도 없고, 불필요한 수평 추상화 계층도 없고, 외재적 인지 부하도 없다.**

이런 계층화가 데이터베이스나 다른 의존성을 빠르게 교체할 수 있게 해 준다고 생각한다면 착각이다. 스토리지를 바꾸는 일은 수많은 문제를 일으키는데, 장담하건대 데이터 접근 계층에 대한 추상화가 있느냐 없느냐는 걱정거리 중 가장 사소한 축이다. 잘해야 추상화가 마이그레이션 시간의 10%쯤 아껴 줄까 말까다. 진짜 고통은 데이터 모델 비호환성, 통신 프로토콜, 분산 시스템의 난제들, 그리고 [암묵적 인터페이스](https://www.hyrumslaw.com)에 있다.

> API의 사용자가 충분히 많아지면,
> 계약서에 무엇을 약속했는지는 중요하지 않다:
> 시스템의 관찰 가능한 모든 동작에
> 누군가는 의존하게 될 것이다.

우리는 스토리지 마이그레이션을 했고, 약 10개월이 걸렸다. 구 시스템은 단일 스레드였기 때문에 노출되는 이벤트가 순차적이었다. 우리의 모든 시스템이 그 관찰된 동작에 의존하고 있었다. 이 동작은 API 계약의 일부가 아니었고, 코드에 반영되어 있지도 않았다. 새 분산 스토리지에는 그런 보장이 없었다. 이벤트가 순서 없이(out-of-order) 도착했다. 추상화 덕분에 새 스토리지 어댑터를 코딩하는 데는 몇 시간밖에 안 걸렸다. **그다음 10개월을 순서가 뒤엉킨 이벤트와 그 밖의 난제들을 처리하는 데 썼다.** 추상화가 컴포넌트를 빠르게 교체하게 해 준다는 말이 이제는 우습게 들린다.

**그렇다면 미래에 보상도 없는 계층형 아키텍처에 왜 높은 인지 부하라는 값을 치러야 하나?** 게다가 대부분의 경우, 핵심 컴포넌트를 교체한다는 그 미래는 영영 오지 않는다.

이런 아키텍처들은 근본적인 것이 아니다. 더 근본적인 원칙들에서 파생된 주관적이고 편향된 결과물일 뿐이다. 왜 그런 주관적 해석에 의존하나? 대신 근본 규칙을 따르라: 의존성 역전 원칙, 단일 진실 공급원, 인지 부하, 정보 은닉. 비즈니스 로직은 데이터베이스, UI, 프레임워크 같은 저수준 모듈에 의존해서는 안 된다. 인프라 걱정 없이 핵심 로직에 대한 테스트를 쓸 수 있어야 한다. 그게 전부다. [토론](https://github.com/zakirullin/cognitive-load/discussions/24)

아키텍처를 위한 아키텍처로 추상화 계층을 더하지 말라. 실용적인 이유로 정당화되는 확장 지점이 필요할 때에만 더하라.

**[추상화 계층은 공짜가 아니다](https://blog.jooq.org/why-you-should-not-implement-layered-architecture). 우리의 제한된 작업 기억에 담아 두어야 하는 것들이다.**

## 도메인 주도 설계

도메인 주도 설계(DDD)에는 훌륭한 지점들이 있지만, 자주 오해된다. 사람들은 "우리는 DDD로 코드를 짠다"고 말하는데, 좀 이상한 말이다. DDD는 해결 공간(solution space)보다는 문제 공간(problem space)에 관한 것이기 때문이다.

유비쿼터스 언어, 도메인, 바운디드 컨텍스트, 애그리거트, 이벤트 스토밍은 전부 문제 공간에 관한 것이다. 도메인에 대한 통찰을 얻고 경계를 추출하도록 돕기 위한 도구다. DDD는 개발자, 도메인 전문가, 비즈니스 담당자가 하나의 통일된 언어로 효과적으로 소통할 수 있게 한다. 그런데 우리는 DDD의 이런 문제 공간 측면보다는 특정 폴더 구조, 서비스, 리포지토리 같은 해결 공간의 기법들을 강조하는 경향이 있다.

우리가 DDD를 해석하는 방식은 십중팔구 고유하고 주관적이다. 그리고 그 이해 위에 코드를 쌓는다면, 즉 대량의 외재적 인지 부하를 만들어 낸다면 - 미래의 개발자들은 끝장이다. `🤯`

팀 토폴로지(Team Topologies)는 인지 부하를 팀들에 걸쳐 나누는 데 도움이 되는, 훨씬 낫고 이해하기 쉬운 프레임워크를 제공한다. 엔지니어들은 팀 토폴로지를 배우고 나면 서로 꽤 비슷한 멘털 모델을 형성하는 경향이 있다. 반면 DDD는 10명의 독자에게 10개의 서로 다른 멘털 모델을 만들어 주는 듯하다. 공통 기반이 되는 대신, 불필요한 논쟁의 전쟁터가 된다.

## 익숙한 프로젝트에서의 인지 부하

> 문제는 **익숙함(familiarity)이 단순함(simplicity)과 같지 않다**는 것이다. 둘은 *느낌*이 같다 - 큰 정신적 노력 없이 공간을 누비는 그 편안함 - 하지만 이유는 전혀 다르다. 당신이 쓰는 "영리한"(읽기: "자기만족적인") 비관용적 트릭 하나하나가 다른 모든 사람에게 학습 페널티를 부과한다. 일단 그 학습을 마치고 나면 그 코드로 작업하는 게 덜 어렵게 느껴진다. 그래서 이미 익숙해진 코드를 어떻게 단순화할지 알아보기가 어렵다. 내가 "신입"이 조직에 물들기 전에 코드를 비평하게 하려는 이유가 이것이다!
>
> 이전 작성자(들)는 이 거대한 난장판을 한 번에 만든 게 아니라 한 번에 조금씩 만들었을 가능성이 크다. 그러니 당신은 그 전체를 한꺼번에 이해해 보려고 시도한 최초의 사람인 것이다.
>
> 내 수업에서는 어느 날 우리가 들여다봤던, 거대한 WHERE 절에 조건이 수백 줄 널려 있는 방대한 SQL 저장 프로시저 이야기를 한다. 누군가 어떻게 이 지경이 되도록 놔둘 수 있었느냐고 물었다. 나는 이렇게 답했다: "조건이 2~3개일 때는 하나 더 추가해도 아무 차이가 없다. 그리고 조건이 20~30개쯤 되면, 하나 더 추가해도 아무 차이가 없다!"
>
> 코드베이스에 작용하는 "단순화의 힘" 같은 것은 없다. 당신이 내리는 의도적인 선택 말고는. 단순화에는 노력이 들고, 사람들은 너무 자주 서두른다.
>
> *코멘트를 남겨 준 [Dan North](https://dannorth.net)에게 감사한다.*

프로젝트의 멘털 모델을 장기 기억에 내재화했다면, 높은 인지 부하를 경험하지 않을 것이다.

<div align="center">
  <img src="https://raw.githubusercontent.com/zakirullin/cognitive-load/main/img/mentalmodelsv15.png" alt="Mental models" width="700">
</div>

배워야 할 고유한 멘털 모델이 많을수록, 새 개발자가 가치를 내기까지 걸리는 시간이 길어진다. 인지 부하를 낮게 유지하면, 사람들이 입사 후 몇 시간 안에 코드베이스에 기여할 수 있다. 그렇다고 품질을 희생하거나 진흙 더미가 쌓이도록 허용한다는 뜻이 아니다.

> 그 고유한 멘털 모델이란 무엇인가? 보통 클린 아키텍처/이벤트 주도 아키텍처/DDD가 뒤섞인 어떤 규칙의 집합이다. 원작자를 가장 설레게 한 것들에 대한 그 사람만의 해석이다. 그 사람만의 주관적 멘털 모델이다. **다른 사람들이 내재화해야 하는 외재적 인지 부하다.**

프로젝트에 새 사람을 온보딩할 때, 그들이 겪는 혼란의 양을 측정해 보라(페어 프로그래밍이 도움이 될 수 있다). 40분 넘게 연속으로 혼란스러워한다면 - 코드에 개선할 것이 있다는 뜻이다.

## 사례

> 소프트웨어 시스템은 아마도 (서로 다른 부품 종류의 수라는 면에서) 인류가 만드는 것들 중 가장 정교하고 복잡한 것이다.
>
> *프레드 브룩스, 맨먼스 미신*

- 우리 아키텍처는 표준적인 CRUD 앱 아키텍처, [Postgres 위의 파이썬 모놀리스](https://danluu.com/simple-architectures/)다
- 인스타그램이 [단 3명의 엔지니어](https://read.engineerscodex.com/p/how-instagram-scaled-to-14-million)로 1400만 사용자까지 확장한 방법
- 우리가 "우와, 이 사람들 [머리가 엄청나게 좋은데](https://kenkantzer.com/learnings-from-5-years-of-tech-startup-code-audits/)"라고 감탄했던 회사들은 대부분 망했다
- 시스템 전체를 배선하는 함수 하나. 시스템이 어떻게 동작하는지 알고 싶으면 - [가서 읽으면 된다](https://www.infoq.com/presentations/8-lines-code-refactoring)
- 이해 가능성을 위한 설계: [Raft 합의 알고리즘](https://www.youtube.com/watch?v=vYp4LYbnnW8)

이 아키텍처들은 꽤 지루하고 이해하기 쉽다. 누구든 큰 정신적 노력 없이 파악할 수 있다.

아키텍처 리뷰에 주니어 개발자들을 참여시켜라. 정신적으로 부담스러운 영역을 찾아내는 데 도움이 될 것이다.

**소프트웨어 유지보수는 어렵다.** 뭔가는 부서지기 마련이고, 아낄 수 있는 정신적 노력은 한 톨도 남김없이 필요해진다. 시스템의 컴포넌트가 적을수록 문제도 적다. 디버깅도 정신적으로 덜 부담스러워진다.

> 디버깅은 처음에 코드를 쓰는 것보다 두 배 어렵다. 따라서 코드를 최대한 영리하게 짠다면, 당신은 정의상 그것을 디버깅할 만큼 똑똑하지 않다.
>
> *브라이언 커니핸*

일반적으로 "와, 이 아키텍처 정말 느낌 좋은데!" 하는 마음가짐은 사람을 오도한다. 그것은 "특정 시점의" 주관적 느낌일 뿐, 현실에 대해 아무것도 말해 주지 않는다. 훨씬 나은 접근은 장기적인 결과를 관찰하는 것이다:

- 문제를 재현하고 디버깅하기 쉬운가? 아니면 호출 스택이나 분산 컴포넌트 사이를 뛰어다니며 모든 것을 머릿속에서 짜 맞춰야 하는가?
- 변경을 빠르게 할 수 있는가, 아니면 모르는 것투성이라 사람들이 손대기를 두려워하는가?
- 새 사람들이 기능을 빠르게 추가할 수 있는가? 배워야 할 고유한 멘털 모델이 있는가?

이런 질문들은 추적하기가 훨씬 어렵고, 사람들은 직접 답하기를 꺼리는 경우가 많다. 세상에서 가장 복잡한 소프트웨어 시스템 중 세월의 시험을 견뎌 낸 것들 - Unix, Kubernetes, Chrome, Redis(아래 코멘트 참고) - 을 보라. 화려한 것은 하나도 없고, 대부분 지루하다. 그리고 그것은 좋은 일이다.

똑똑한 개발자들이 최신 디자인 패턴과 마이크로서비스로 인상적인 아키텍처를 만든 프로젝트를 수없이 봤다. 그런데 새 팀원들이 변경을 시도하면, 모든 것이 어떻게 맞물려 있는지 이해하는 데만 몇 주를 썼다. 인지 부하가 너무 높아서 생산성은 곤두박질치고 버그는 늘어났다.

아이러니는? 복잡성을 유발하는 이 패턴들 중 상당수가 "클린 코드"라는 이름으로 도입됐다는 것이다.

정말 중요한 것은 불필요한 인지 부담을 줄이는 것이다. 때로는 얕은 모듈 여러 개 대신 더 적고 더 깊은 모듈을 뜻하고, 때로는 관련 로직을 자잘한 함수로 쪼개지 않고 한데 묶어 두는 것을 뜻한다.

그리고 때로는 영리한 해법 대신 지루하고 직관적인 해법을 고르는 것을 뜻한다. 최고의 코드는 가장 우아하거나 정교한 코드가 아니다. 미래의 개발자들(당신 자신을 포함해서)이 빠르게 이해할 수 있는 코드다.

이 글은 우리가 브라우저 개발에서 마주하는 과제들과 깊이 공명한다. 현대 브라우저가 가장 복잡한 소프트웨어 시스템 중 하나라는 지적은 전적으로 옳다. Chromium에서 그 복잡성을 관리하는 것은 인지 부하에 관해 당신이 짚은 여러 지점과 정확히 맞아떨어지는 상시적인 과제다.

Chromium에서 이를 다루는 한 가지 방법은 신중한 컴포넌트 격리와, 서브시스템(렌더링, 네트워킹, 자바스크립트 실행 등) 사이의 잘 정의된 인터페이스다. Unix I/O의 깊은 모듈 예시와 비슷하게, 우리는 비교적 단순한 인터페이스 뒤의 강력한 기능을 지향한다. 예컨대 렌더링 파이프라인은 엄청난 복잡성(레이아웃, 컴포지팅, GPU 가속)을 처리하지만, 개발자들은 명확한 추상화 계층을 거쳐 상호작용할 수 있다.

불필요한 추상화를 피하라는 지적도 크게 와닿는다. 브라우저 개발에서 우리는 웹 표준과 호환성이라는 내재적 복잡성을 다루면서도 코드베이스를 새 기여자들이 접근하기 쉽게 유지하는 것 사이에서 끊임없이 균형을 잡는다.

때로는 가장 단순한 해법이 최선이다. 복잡한 시스템에서도.

**[antirez](https://x.com/antirez)** *(Redis)*
전적으로 동의한다 :) 그리고 언급된 "A Philosophy of Software Design"에 빠져 있다고 생각하는 것은 "설계 희생(design sacrifice)"이라는 개념이다. 즉, 때로는 무언가를 희생하고 그 대가로 단순함이나 성능, 혹은 둘 다를 얻는다. 나는 이 아이디어를 끊임없이 적용하는데, 이해받지 못할 때가 많다.

좋은 예가, 내가 해시 항목의 만료(expire)를 항상 거부해 왔다는 사실이다. 이것은 설계 희생이다. 특정 속성이 최상위 항목(키 자체)에만 있으면 설계가 더 단순해지고, 값은 그냥 객체이면 되기 때문이다. Redis에 해시 만료가 들어왔을 때, 좋은 기능이긴 했지만 (실제로) 여러 부분에 많은 변경이 필요했고 복잡성이 올라갔다.

또 다른 예는 지금 하고 있는 작업인 Vector Sets, Redis의 새 데이터 타입이다. 나는 Redis가 벡터에 대한 진실 공급원이 되지 않고 그 근사 버전만 받도록 결정했다. 그 덕에 삽입 시 정규화, 커다란 float 벡터를 디스크에 유지하려 애쓰지 않는 양자화 등이 가능해졌다. 많은 벡터 DB는 사용자가 넣은 것을 그대로 기억한다는 점(전체 정밀도 벡터)을 희생하지 못한다.

무작위로 고른 두 예시일 뿐이고, 나는 이 아이디어를 어디에나 적용한다. 문제는 물론 올바른 것을 희생해야 한다는 점이다. 흔히, 아주 큰 복잡성을 차지하는 5%의 기능이 있다. 그런 게 바로 죽이기 좋은 것이다 :D

**[인터넷의 어느 개발자](https://working-for-the-future.medium.com/about)**
당신은 나를 채용하지 않을 것이다... 나는 출시한 엔터프라이즈 프로젝트 실적으로 자신을 판다.

디자인 패턴을 유창하게 말할 줄 아는 사람과 일한 적이 있다. 나는 결코 그렇게 말하지 못했지만, 그를 제대로 이해하는 몇 안 되는 사람 중 하나였다. 관리자들은 그를 사랑했고 그는 어떤 개발 논의든 지배할 수 있었다. 그의 주변에서 일하던 사람들은 그가 파괴의 흔적을 남기고 다닌다고 했다. 내가 그의 프로젝트를 이해한 첫 번째 사람이라는 말을 들었다. 유지보수성은 중요하다. 나는 TCO(*총 소유 비용*)를 가장 중시한다. 어떤 회사들에게는 그것이 전부다.

한동안 안 들어가다가 Github에 로그인했는데, 무슨 이유에서인지 낯선 누군가의 저장소에 있는 글로 이동됐다. "이게 뭐지" 싶었고 내 홈페이지로 돌아가는 데 애를 먹어서, 그냥 읽었다. 당시에는 크게 와닿지 않았는데, 알고 보니 놀라운 글이었다. 모든 개발자가 읽어야 한다. 요지는, 프로그래밍 베스트 프랙티스라고 우리가 들어 온 거의 모든 것이 과도한 "인지 부하"로 이어진다는 것, 즉 우리의 정신이 지적 요구에 두들겨 맞고 있다는 것이었다. 나는 이걸 꽤 오래전부터 알고 있었다. 특히 클라우드, 보안, DevOps의 요구를 겪으면서.

이 글이 좋았던 또 하나의 이유는, 내가 수십 년 동안 해 왔지만 인기가 없어서 잘 인정하지 않던 관행들을 묘사하고 있었기 때문이다... 나는 정말 복잡한 것들을 만들고, 얻을 수 있는 도움은 전부 필요하다.

생각해 보라. 내 짐작이 맞다면, 그 글이 내 앞에 나타난 것은 Github의 매우 똑똑한 사람들이 개발자들이 이 글을 봐야 한다고 생각했기 때문이다. 나도 동의한다.
