# 당신의 함수는 무슨 색인가?

> 원문: [What Color is Your Function?](https://journal.stuffwithstuff.com/2015/02/01/what-color-is-your-function/)
> 저자: Bob Nystrom | 번역일: 2026-07-10

여러분은 어떤지 모르겠지만, 아침에 나를 깨우는 데 좋은 옛날식 프로그래밍 언어 성토만 한 것이 없다. 누군가가 평민들이 쓰는 ["blub"](http://www.paulgraham.com/avg.html) 언어를 꼬치꼬치 찌르는 걸 보면 피가 끓는다. 그 평민들은 StackOverflow를 몰래 들락거리며 하루하루를 그 언어로 버텨내고 있다.

(그동안 당신과 나는 가장 계몽된 언어만 쓴다. 우리 같은 전문 장인의 잘 손질된 손을 위해 설계된, 끌처럼 날카로운 도구 말이다.)

물론 그런 독설의 *저자*로서 나는 위험을 감수한다. 내가 조롱하는 언어가 당신이 좋아하는 언어일 수도 있다! 나도 모르게 쇠스랑과 횃불을 든 군중을 내 블로그에 들인 셈이 될 수 있고, 내 무모한 팸플릿이 그들의 분노를 살 수도 있다!

그 불길의 열기로부터 나를 보호하고, 혹시 모를 여러분의 섬세한 감수성을 건드리지 않기 위해, 대신 내가 방금 지어낸 언어를 성토하겠다. 오로지 불태워지기 위해 존재하는 허수아비다.

이게 무의미해 보인다는 거 안다. 그렇지? 날 믿어라. 끝에 가면 그 허수아비의 짚 머리에 누구의 얼굴이(혹은 얼굴들이!) 그려져 있는지 보게 될 것이다.

## 새로운 언어

블로그 글 하나 때문에 완전히 새로운 (형편없는) 언어를 통째로 배우라는 건 무리한 요구이므로, 이 언어는 당신과 내가 이미 아는 언어와 대체로 비슷하다고 하자. 문법은 JS 비슷하다고 치자. 중괄호와 세미콜론, `if`, `while` 등등. 프로그래밍 동굴의 *공용어* 말이다.

JS를 고른 건 이 글이 JS에 관한 글이어서가 *아니다*. 그저 평균적인 독자의 통계적 표본인 당신이 가장 이해하기 쉬운 언어이기 때문이다. 자:

```javascript
function thisIsAFunction() {
  return "It's awesome";
}
```

우리의 허수아비는 *현대적인* (구린) 언어이므로 일급 함수도 있다. 그래서 이런 걸 만들 수 있다:

```javascript
// collection에서 predicate에 맞는 모든 원소를 담은
// 리스트를 반환한다.
function filter(collection, predicate) {
  var result = [];
  for (var i = 0; i < collection.length; i++) {
    if (predicate(collection[i])) result.push(collection[i]);
  }
  return result;
}
```

이건 그 *고차* 함수라는 것으로, 이름이 암시하듯 아주 격조 있고 엄청나게 유용하다. 아마 컬렉션을 주무를 때 익숙하게 써봤을 텐데, 이 개념을 체득하고 나면 거의 어디서나 쓰게 된다.

테스트 프레임워크에서라든가:

```javascript
describe("An apple", function() {
  it("ain't no orange", function() {
    expect("Apple").not.toBe("Orange");
  });
});
```

데이터를 파싱할 때라든가:

```javascript
tokens.match(Token.LEFT_BRACKET, function(token) {
  // 리스트 리터럴을 파싱...
  tokens.consume(Token.RIGHT_BRACKET);
});
```

그래서 당신은 신나게 함수를 넘기고, 함수를 호출하고, 함수를 반환하면서 온갖 멋진 재사용 가능한 라이브러리와 애플리케이션을 작성한다. 함수 축제다.

## 당신의 함수는 무슨 색인가?

그런데 잠깐. 여기서 이 언어가 이상해진다. 이 언어에는 한 가지 기이한 특징이 있다:

**1. 모든 함수에는 색이 있다.**

익명 콜백이든 이름 있는 일반 함수든, 모든 함수는 빨간색 아니면 파란색이다. `function` 키워드 하나 대신 두 개가 있다:

```
blue_function doSomethingAzure() {
  // 이건 파란 함수다...
}

red_function doSomethingCarnelian() {
  // 이건 빨간 함수다...
}
```

이 언어에 색 없는 함수는 *없다*. 함수를 만들고 싶은가? 색을 골라야 한다. 그게 규칙이다. 그리고 실은 따라야 할 규칙이 몇 개 더 있다:

**2. 함수를 호출하는 방법은 그 함수의 색에 따라 다르다.**

"파란 호출" 문법과 "빨간 호출" 문법이 있다고 상상하자. 이런 식이다:

```
doSomethingAzure()blue;
doSomethingCarnelian()red;
```

함수를 호출할 때는 그 색에 맞는 호출을 써야 한다. 틀리면—빨간 함수를 괄호 뒤에 `blue`를 붙여 호출하거나 그 반대로 하면—나쁜 일이 벌어진다. 침대 밑에 숨어 있던, 팔이 뱀인 광대 같은 어린 시절의 잊고 있던 악몽을 끄집어내 보라. 그게 모니터에서 튀어나와 당신의 유리체를 빨아먹는다.

짜증 나는 규칙이지? 아, 하나 더 있다:

**3. 빨간 함수는 다른 빨간 함수 안에서만 호출할 수 있다.**

빨간 함수 안에서 파란 함수를 호출하는 건 *가능하다*. 이건 괜찮다:

```
red_function doSomethingCarnelian() {
  doSomethingAzure()blue;
}
```

하지만 그 반대는 안 된다. 이렇게 하면:

```
blue_function doSomethingAzure() {
  doSomethingCarnelian()red;
}
```

늙은 거미입 밤광대(Spidermouth the Night Clown)의 방문을 받게 될 것이다.

이 때문에 아까의 `filter()` 같은 고차 함수를 작성하기가 까다로워진다. `filter()` *자체의* 색을 골라야 하고, 그 색이 우리가 넘길 수 있는 함수의 색에 영향을 준다. 뻔한 해법은 `filter()`를 빨간색으로 만드는 것이다. 그러면 빨간 함수든 파란 함수든 받아서 호출할 수 있다. 하지만 그러면 이 언어라는 고행복(hairshirt)의 다음 가려운 지점에 부딪힌다:

**4. 빨간 함수는 호출하기가 더 고통스럽다.**

지금은 "고통스럽다"를 정확히 정의하지 않겠지만, 프로그래머가 빨간 함수를 호출할 때마다 성가신 절차를 거쳐야 한다고 상상해 보라. 아주 장황할 수도 있고, 특정 종류의 문(statement) 안에서는 호출할 수 없을 수도 있다. 소수(prime) 번째 줄에서만 호출할 수 있을지도 모른다.

중요한 건, 함수를 빨간색으로 만들기로 하면 당신의 API를 쓰는 모든 사람이 당신 커피에 침을 뱉고 싶어질 거라는 점이다. 그보다 더 고약한 액체를 넣고 싶어질 수도 있다.

그렇다면 뻔한 해법은 빨간 함수를 *절대* 쓰지 않는 것이다. 모든 걸 파란색으로 만들면, 모든 함수의 색이 같은 정상적인 세계로 돌아온다. 그건 모든 함수에 색이 없는 것과 같고, 이 언어가 완전히 멍청하지는 않은 것과 같다.

하지만 아뿔싸, 가학적인 언어 설계자들이—프로그래밍 언어 설계자가 전부 가학적이라는 건 우리 모두 알지 않는가?—마지막 가시 하나를 우리 옆구리에 꽂아 넣었다:

**5. 코어 라이브러리 함수 중 일부는 빨간색이다.**

플랫폼에 내장된 함수들, 우리가 *반드시* 써야 하고 직접 작성할 수도 없는 함수들 중에 빨간색으로만 제공되는 것들이 있다. 이쯤 되면 합리적인 사람이라면 이 언어가 우리를 증오한다고 생각할 만하다.

## 함수형 프로그래밍 탓이다!

여기서 문제는 우리가 고차 함수를 쓰려 한다는 것이라고 생각할 수도 있다. 그 함수형 허례허식 속에서 나풀거리는 걸 그만두고, 신의 뜻대로 평범한 블루칼라 1차 함수만 쓴다면 이 모든 가슴앓이를 피할 수 있을 거라고.

파란 함수만 호출한다면 내 함수도 파란색으로 만들면 된다. 아니면 빨간색으로 만들면 된다. 함수를 받는 함수를 만들지만 않으면, "함수 색에 대해 다형적"("다색적(polychromatic)"?)이 되려고 애쓸 필요도, 그런 헛소리를 걱정할 필요도 없다.

하지만 아뿔싸, 고차 함수는 한 가지 예일 뿐이다. 이 문제는 프로그램을 재사용되는 별도의 함수들로 쪼개려 할 때마다 어디에나 스며든다.

예를 들어, 뭐랄까, 당신의 소셜 네트워크 구성원들이 서로 얼마나 반해 있는지를 나타내는 그래프 위에서 다익스트라 알고리즘을 구현하는 근사한 코드 조각이 있다고 하자. (그 결과가 대체 뭘 의미할지 정하느라 나는 너무 오래 고민했다. 이행적 비호감도?)

나중에 이 코드 조각을 다른 곳에서도 써야 하게 된다. 당신은 자연스럽게 그걸 별도의 함수로 끌어올린다. 원래 자리에서도 호출하고, 그걸 쓰는 새 코드에서도 호출한다. 그런데 그 함수는 무슨 색이어야 할까? 당연히 가능하면 파란색으로 만들겠지만, 그 코드가 빨간색 전용 코어 라이브러리 함수 중 하나를 쓴다면?

새로 호출하려는 곳이 파란색이라면? 그 함수를 빨간색으로 바꿔야 한다. 그러면 *그걸* 호출하는 함수도 빨간색으로 바꿔야 한다. 으윽. 어떻게 하든 색을 끊임없이 생각해야 한다. 개발이라는 해변 휴가에서 수영복 속의 모래가 될 것이다.

## 색색의 우화

물론 나는 진짜 색 이야기를 하는 게 아니다. 그렇지 않은가? 이건 우화, 문학적 장치다. 《스니치들(The Sneetches)》은 배에 붙은 별 이야기가 아니라 인종 이야기다. 이쯤 되면 색이 실제로 무엇을 나타내는지 짐작이 갈 것이다. 아니라면, 대공개:

**빨간 함수는 비동기 함수다.**

Node.js에서 JavaScript로 프로그래밍하고 있다면, 콜백을 호출해서 값을 "반환"하는 함수를 정의할 때마다 방금 빨간 함수를 만든 것이다. 아까의 규칙 목록을 다시 보면서 내 은유가 어떻게 들어맞는지 확인해 보라:

- 동기 함수는 값을 반환하지만, 비동기 함수는 반환하지 않고 대신 콜백을 호출한다.

- 동기 함수는 결과를 반환값으로 주고, 비동기 함수는 당신이 넘긴 콜백을 호출해서 준다.

- 동기 함수에서 비동기 함수를 호출할 수 없다. 비동기 함수가 나중에 완료되기 전까지는 결과를 알 수 없기 때문이다.

- 비동기 함수는 콜백 때문에 표현식 안에서 합성되지 않고, 에러 처리가 다르며, `try/catch`나 다른 여러 제어 흐름 문 안에서 쓸 수 없다.

- Node의 특기가 바로 코어 라이브러리가 전부 비동기라는 점이다. (그들도 이걸 좀 완화해서 여러 함수에 `___Sync()` 버전을 추가하기 시작하긴 했다.)

사람들이 "콜백 지옥"을 이야기할 때, 그들은 자기 언어에 빨간 함수가 있는 게 얼마나 짜증 나는지 이야기하는 것이다. [비동기 프로그래밍을 위한 라이브러리를 4,089개](https://www.npmjs.com/search?q=async)나 만드는 건, 언어가 떠넘긴 문제를 라이브러리 수준에서 감당해 보려는 몸부림이다.

*2021년 12월 3일 업데이트:* 오늘 기준 비동기 라이브러리는 15,118개다.

## 프로미스, 미래는 더 낫다고 약속하지

Node 커뮤니티 사람들은 콜백이 고통스럽다는 걸 오래전부터 깨닫고 해법을 찾아왔다. 많은 사람을 들뜨게 하는 기법 하나가 [*프로미스(promise)*](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise)다. 래퍼 이름인 "퓨처(future)"로도 알려져 있다.

이건 콜백과 에러 핸들러를 감싼 일종의 강화 래퍼다. 함수에 콜백과 에러백을 넘기는 것을 하나의 *개념*으로 본다면, 프로미스는 기본적으로 그 개념의 *구체화(reification)*다. 비동기 연산을 나타내는 일급 객체인 것이다.

방금 문단에 멋들어진 프로그래밍 언어 용어를 잔뜩 욱여넣었으니 꽤 괜찮은 물건처럼 들리겠지만, 기본적으로는 만병통치약 행세를 하는 가짜 약이다. 프로미스가 비동기 코드를 조금 더 쓰기 쉽게 만들어 주긴 *한다*. 합성도 좀 더 잘 되니 규칙 4번이 *아주* 괴롭지는 않게 된다.

하지만 솔직히, 배를 맞는 것과 급소를 맞는 것의 차이 정도다. 엄밀히는 덜 아프다, 맞다. 하지만 그 가치 제안에 진심으로 열광할 사람은 없을 것이다.

여전히 예외 처리나 다른 제어 흐름 문과 함께 쓸 수 없다. 여전히 동기 코드에서 퓨처를 반환하는 함수를 호출할 수 없다. (뭐, *할 수는* 있다. 하지만 그렇게 하면, 나중에 당신 코드를 유지보수하는 사람이 타임머신을 발명해서 당신이 그 짓을 한 순간으로 돌아가 2번 연필로 당신 얼굴을 찌를 것이다.)

여전히 온 세상을 비동기와 동기의 두 반쪽으로 갈라놓았고, 거기에 따르는 온갖 비참함도 그대로다. 그러니 언어에 프로미스나 퓨처가 있더라도, 그 얼굴은 내 허수아비의 얼굴과 몹시 닮아 있다.

(그렇다, 내가 일하는 언어인 [Dart](http://dartlang.org)도 마찬가지라는 뜻이다. 그래서 팀 일부가 [다른 동시성 모델을 실험](https://github.com/dart-lang/fletch)하고 있다는 게 그렇게 반가운 것이다.)

## await, 해법을 기다리며

C# 프로그래머들은 지금쯤 꽤 우쭐해하고 있을 것이다(Hejlsberg와 그 일당이 달콤한 기능을 하나하나 언어에 쌓아 올리면서 점점 더 자주 걸리는 증상이다). C#에서는 [`await` 키워드](https://msdn.microsoft.com/en-us/library/hh191443.aspx)로 비동기 함수를 호출할 수 있다.

이 덕분에 귀여운 키워드 하나만 살짝 얹으면 동기 호출만큼이나 쉽게 비동기 호출을 할 수 있다. `await` 호출을 표현식 안에 중첩할 수도 있고, 예외 처리 코드에서 쓸 수도 있고, 제어 흐름 안에 넣을 수도 있다. 마음껏 하라. 새 랩 앨범 선인세로 받은 지폐 뿌리듯 `await` 호출을 뿌려대라.

async-await는 확실히 괜찮고, 그래서 우리도 Dart에 추가하고 있다. 비동기 코드를 *작성*하기가 훨씬 쉬워진다. "하지만"이 나올 차례라는 걸 알 것이다. 나온다. *하지만…* 여전히 세계는 둘로 갈라져 있다. 그 비동기 함수들은 작성하기 쉬워졌지만, *여전히 비동기 함수다*.

여전히 두 가지 색이 있다. async-await는 짜증 나는 규칙 4번을 해결한다. 빨간 함수를 호출하는 게 파란 함수보다 크게 나쁘지 않게 만들어 준다. 하지만 나머지 규칙은 전부 그대로다:

- 동기 함수는 값을 반환하고, 비동기 함수는 값을 감싼 `Task<T>`(Dart에서는 `Future<T>`) 래퍼를 반환한다.

- 동기 함수는 그냥 호출하면 되지만, 비동기 함수에는 `await`가 필요하다.

- 비동기 함수를 호출하면 실제로 원하는 건 `T`인데 이 래퍼 객체가 손에 들어온다. *당신의* 함수를 비동기로 만들어 await하지 않는 한 래퍼를 벗겨낼 수 없다. (단, 아래 참조.)

- `await`를 넉넉히 곁들여야 한다는 점만 빼면, 이건 그래도 고쳤다.

- C#의 코어 라이브러리는 사실 async보다 오래됐으니 이 문제는 애초에 없었던 셈이다.

확실히 *더 낫긴 하다*. 맨 콜백이나 퓨처보다는 async-await를 언제든 택하겠다. 하지만 우리 고민이 전부 사라졌다고 생각한다면 스스로를 속이는 것이다. 고차 함수를 쓰려 하거나 코드를 재사용하려 하는 순간, 색은 여전히 거기 있어서 코드베이스 전체에 번지고 있다는 걸 다시 깨닫게 된다.

## 색이 *없는* 언어는 무엇인가?

그러니까 JS, Dart, C#, Python에 이 문제가 있다. CoffeeScript를 비롯해 JS로 컴파일되는 대부분의 언어도 마찬가지다(그래서 Dart가 이 문제를 물려받았다). ClojureScript조차 [core.async](https://github.com/clojure/core.async)로 힘껏 저항했음에도 이 문제가 있다고 나는 *생각한다*.

이 문제가 없는 언어를 알고 싶은가? *Java*다. 그렇다니까? "그래, Java가 이걸 제대로 한 언어야."라고 말할 기회가 얼마나 자주 오겠는가? 그런데 정말 그렇다. 그들의 명예를 위해 덧붙이자면, 그들은 퓨처와 비동기 IO로 옮겨가면서 이 실수를 적극적으로 바로잡으려는 중이다. 바닥을 향한 경주 같달까.

C#도 사실 이 문제를 피할 *수* 있다. 그들은 색을 갖기로 스스로 *선택*한 것이다. async-await와 그 모든 `Task<T>` 물건을 추가하기 전에는 그냥 일반 동기 API 호출을 쓰면 됐다. 이 문제가 없는 언어 세 개 더: Go, Lua, Ruby.

이들의 공통점이 뭔지 짐작이 가는가?

*스레드*다. 더 정확히는, *[서로 전환할 수 있는](https://journal.stuffwithstuff.com/2013/01/13/iteration-inside-and-out/) 여러 개의 독립된 콜스택*이다. 꼭 운영체제 스레드일 필요는 없다. Go의 고루틴, Lua의 코루틴, Ruby의 파이버로도 충분하다.

(C#에 그 작은 단서가 붙은 이유가 이것이다. C#에서는 스레드를 쓰면 async의 고통을 피할 수 있다.)

## 지나간 연산을 찾아서

근본적인 문제는 "연산이 완료됐을 때 하던 일을 어떻게 이어서 하는가?"이다. 커다란 콜스택을 쌓아 올린 상태에서 어떤 IO 연산을 호출한다. 성능을 위해 그 연산은 운영체제의 밑단 비동기 API를 쓴다. 완료를 기다릴 수는 *없다*. 기다려도 완료되지 않기 때문이다. 언어의 이벤트 루프까지 전부 되돌아가서, OS가 일을 마칠 시간을 줘야 한다.

연산이 완료되면 하던 일을 재개해야 한다. 언어가 "자기가 어디에 있었는지 기억하는" 일반적인 방법은 *콜스택*이다. 콜스택은 현재 호출 중인 모든 함수와 각 함수 안에서 명령 포인터가 어디에 있는지를 추적한다.

하지만 비동기 IO를 하려면 C 콜스택 전체를 풀어서 버려야 한다. 일종의 진퇴양난이다. 초고속 IO는 할 수 있는데, 그 결과로는 아무것도 할 수 없는 것이다! 코어에 비동기 IO를 가진 모든 언어는—JS의 경우 브라우저의 이벤트 루프는—어떤 식으로든 이 문제에 대처한다.

끝없이 오른쪽으로 행진하는 콜백을 가진 Node는 그 콜프레임들을 전부 클로저에 욱여넣는다. 이렇게 하면:

```javascript
function makeSundae(callback) {
  scoopIceCream(function (iceCream) {
    warmUpCaramel(function (caramel) {
      callback(pourOnIceCream(iceCream, caramel));
    });
  });
}
```

각 함수 표현식은 자신을 둘러싼 문맥 전체를 *포획(close over)*한다. 그러면 `iceCream`이나 `caramel` 같은 매개변수가 콜스택에서 힙으로 옮겨진다. 바깥 함수가 반환하고 콜스택이 폐기돼도 괜찮다. 그 데이터는 여전히 힙 어딘가를 떠다니고 있으니까.

문제는 이 망할 단계 하나하나를 *수동으로* 구체화해야 한다는 것이다. 이 변환에는 실제로 이름이 있다. [*연속 전달 스타일(continuation-passing style)*](http://en.wikipedia.org/wiki/Continuation-passing_style)이다. 70년대에 언어 해커들이 컴파일러 내부에서 쓸 중간 표현으로 발명했다. 코드를 표현하는 아주 괴상한 방식인데, 우연히도 몇몇 컴파일러 최적화를 하기 쉽게 해준다.

프로그래머가 *실제로 그런 코드를 직접 작성하리라고는* 아무도 단 한 순간도 생각하지 않았다. 그런데 Node가 나타났고, 어느새 우리는 컴파일러 백엔드 흉내를 내고 있다. 우리는 어디서부터 잘못된 걸까?

프로미스와 퓨처도 사실 아무것도 사주지 않는다는 점에 주목하라. 써봤다면 알겠지만, 여전히 거대한 함수 리터럴 더미를 손으로 만들어내고 있다. 그저 그걸 비동기 함수 자체가 아니라 `.then()`에 넘기고 있을 뿐이다.

## 생성된 해법을 await하며

async-await는 확실히 *도움이 된다*. 컴파일러의 두개골을 열고 `await` 호출을 만났을 때 무슨 일을 하는지 들여다보면, 실제로 CPS 변환을 하고 있는 걸 보게 된다. C#에서 `await`를 써야 하는 *이유*가 그것이다. 컴파일러에게 "여기서 함수를 반으로 쪼개라"라고 알려주는 힌트인 것이다. `await` 뒤의 모든 것이 컴파일러가 당신 대신 합성해 주는 새 함수로 끌어올려진다.

그래서 async-await는 .NET 프레임워크에 어떤 *런타임* 지원도 필요 없었다. 컴파일러가 그걸 이미 처리할 수 있는 연쇄 클로저들로 컴파일해 없애 버리기 때문이다. (흥미롭게도 클로저 자체도 런타임 지원이 필요 없다. *클로저는* 익명 클래스로 컴파일된다. C#에서 클로저는 정말로 [가난한 자의 객체](http://c2.com/cgi/wiki?ClosuresAndObjectsAreEquivalent)다.)

내가 언제 제너레이터 이야기를 꺼낼지 궁금했을 것이다. 당신의 언어에 `yield` 키워드가 있는가? 그렇다면 아주 비슷한 일을 할 수 있다.

(사실 나는 제너레이터와 async-await가 동형(isomorphic)이라고 *믿는다*. 내 하드디스크 어두운 구석 어딘가에 async-await만으로 제너레이터 스타일 게임 루프를 구현한 코드가 굴러다니고 있다.)

어디까지 했더라? 아, 그렇지. 결국 콜백이든, 프로미스든, async-await든, 제너레이터든, 최종적으로는 비동기 함수를 힙에 사는 클로저 뭉치로 뭉개 펴 바르게 된다.

당신의 함수는 가장 바깥쪽 클로저를 런타임에 넘긴다. 이벤트 루프나 IO 연산이 끝나면 런타임이 그 함수를 호출하고, 당신은 하던 일을 이어간다. 하지만 그 말은 당신 위의 모든 것도 반환해야 한다는 뜻이다. 여전히 스택 *전체*를 풀어야 한다.

"빨간 함수는 빨간 함수만 호출할 수 있다"는 규칙이 여기서 나온다. `main()`이나 이벤트 핸들러까지 콜스택 전체를 클로저화해야 하기 때문이다.

## 구체화된 콜스택

하지만 스레드(그린 스레드든 OS 수준 스레드든)가 있으면 그럴 필요가 없다. 그냥 스레드 전체를 일시 정지하고, *그 모든 함수에서 반환할 필요 없이* 곧장 OS나 이벤트 루프로 돌아갈 수 있다.

내 생각에 이걸 가장 아름답게 해내는 언어는 Go다. IO 연산을 하는 순간 Go는 그 고루틴을 그냥 세워두고(park), IO에 막혀 있지 않은 다른 고루틴들을 재개한다.

표준 라이브러리의 IO 연산을 보면 동기처럼 보인다. 다시 말해 그냥 일을 하고, 끝나면 결과를 반환한다. 하지만 JavaScript에서 말하는 의미의 동기는 아니다. 이런 연산 하나가 대기 중인 동안에도 다른 Go 코드는 실행될 수 있다. Go는 *동기 코드와 비동기 코드의 구별을 없애 버린* 것이다.

Go에서 동시성은 *당신이* 프로그램을 어떻게 모델링할지 선택하는 문제의 한 단면이지, 표준 라이브러리의 함수 하나하나에 낙인처럼 새겨진 색이 아니다. 덕분에 위에서 말한 다섯 규칙의 고통이 완전히, 남김없이 사라진다.

그러니 다음에 어떤 뜨거운 신생 언어 이야기를 꺼내면서 비동기 API가 있으니 동시성 스토리가 끝내준다고 말하려 한다면, 내가 왜 이를 갈기 시작하는지 이제 알 것이다. 그 말은 곧 빨간 함수와 파란 함수의 세계로 되돌아왔다는 뜻이기 때문이다.
