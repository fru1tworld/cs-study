# Apache Kafka 스키마 관리가 생각보다 더 많은 비용을 초래하는 이유

> **원문:** [Why Managing Your Apache Kafka Schemas Is Costing You More Than You Think](https://www.confluent.io/blog/schema-management-costs/)

> **저자:** Mohtasham Sayeed Mohiuddin, Associate Solutions Architect, Confluent Staff | **게시일:** 2024년 12월 23일 | **읽는 시간:** 10분

---

이벤트 기반 시스템을 구축하는 개발자에게 스키마는 Apache Kafka에서 프로듀서와 컨슈머 간의 데이터 계약을 정의하는 데 필수적이며, 모든 메시지가 올바르게 해석될 수 있도록 보장합니다. 그러나 스키마 관리를 수동으로 또는 DIY(직접 구축) 솔루션을 통해 처리하면, 조직은 배포 규모가 커질수록 복합적으로 증가하는 비용에 직면하게 됩니다.

이 게시물에서는 수동 스키마 관리가 왜 비용 증가, 호환성 위험, 엔지니어링 오버헤드로 이어지는지 살펴보겠습니다. 이러한 문제는 초기 인프라 예산에는 나타나지 않는 방식으로 발생하는 경우가 많습니다.

## 스키마 관리는 어떻게 숨겨진 비용을 초래하는가?

Kafka 여정 초기에는 스키마를 수동으로 관리하는 실제 비용을 과소평가하기 쉽습니다. Kafka에서의 스키마 관리는 프로듀서와 컨슈머 간의 데이터 호환성을 보장합니다. 이는 스트리밍 데이터가 원활하게 흐르도록 유지하는 가드레일입니다. 그러나 조직이 확장됨에 따라, 스키마를 수동으로 처리하거나 임시 레지스트리를 유지하는 "단순한" 접근 방식은 빠르게 숨겨진 비용 센터가 됩니다.

처음에는 스키마를 수동으로 정의하고 진화시키는 것이 무해해 보일 수 있습니다. 여기에 JSON 하나, 저기에 Avro 하나. 그러나 더 많은 팀이 토픽 전반에 걸쳐 데이터를 발행하고 소비하게 되면, 모든 스키마 변경은 운영 위험을 초래합니다. 이제 각 수정 사항은 다운스트림 시스템을 깨뜨리지 않기 위해 조정, 검증, 버전 관리가 필요합니다. 이를 수십 개의 팀, 파이프라인, 환경에 걸쳐 곱하면, 총 엔지니어링 시간(과 위험)은 복합적으로 증가합니다.

Kafka용 중앙 집중식 스키마 레지스트리가 없으면 팀은 종종 다음과 같은 문제에 직면합니다:

- 프로듀서와 컨슈머가 서로 다른 속도로 진화할 때 발생하는 **호환성 장애**
- 통제되지 않는 스키마 진화로 인한 **데이터 불일치**
- 수동 스키마 검증 및 디버깅에서 오는 **운영 오버헤드**
- 데이터 구조가 중앙에서 관리되지 않을 때 증가하는 **컴플라이언스 노출**

대부분의 팀이 컴퓨팅이나 스토리지 비용 최적화에 집중하는 반면, Kafka에서의 스키마 진화가 숨겨진 재작업, 예상치 못한 다운타임, 개발자 생산성 저하를 통해 총 소유 비용(TCO)을 조용히 부풀릴 수 있다는 사실을 깨닫는 팀은 거의 없습니다.

반면, Confluent Schema Registry와 같은 도구를 통해 이 프로세스를 중앙 집중화하고 자동화하면 즉각적인 투자 수익(ROI)을 얻을 수 있습니다: 더 적은 프로덕션 이슈, 더 빠른 개발 주기, 더 강력한 데이터 거버넌스.

## DIY 스키마 관리의 숨겨진 비용은 무엇인가?

자체 Kafka 스키마 관리 프로세스를 구축하는 것은 처음에는 쉬운 승리처럼 느껴질 수 있지만, DIY 접근 방식은 데이터 생태계가 성숙해짐에 따라 확장되는 비효율성의 거미줄을 숨기고 있는 경우가 많습니다. 비용이 항상 즉각적으로 나타나지는 않지만, 엔지니어링, 운영, 컴플라이언스 전반에 걸쳐 축적되며 Kafka TCO를 직접적으로 부풀립니다.

아래는 스키마 관리를 수동으로 또는 일관되지 않게 처리할 때 발생하는 네 가지 주요 숨겨진 비용입니다:

- **수동 검증은 엔지니어링 시간 낭비로 이어집니다:** 개발자가 스키마 변경을 수동으로 검증할 때, 호환성 확인, 프로듀서와 컨슈머 통합 테스트, 다운스트림 애플리케이션 업데이트 등 반복적인 작업에 귀중한 시간을 소비합니다. 이러한 수동 단계는 확장되지 않으며 혁신에서 초점을 빼앗습니다. 시간이 지나면 이는 생산성 손실과 더 높은 인건비로 이어집니다.

- **호환성 문제는 장애와 소방 대응으로 이어집니다:** 내장된 스키마 호환성 검사가 없으면, 필드 이름 변경이나 데이터 타입 변경과 같은 작은 변경도 프로덕션에서 컨슈머를 깨뜨릴 수 있습니다. 이러한 인시던트는 파이프라인 다운타임, 디버깅 마라톤, SLA 지연으로 이어집니다. 소방 대응에 투입되는 매 시간은 비즈니스 가치 손실의 또 다른 시간입니다.

- **스키마 드리프트는 데이터 손상과 재처리로 이어집니다:** "스키마 드리프트"는 중앙 집중식 감독 없이 데이터가 진화할 때 발생합니다. 동일한 스키마의 일관되지 않은 버전은 손상된 데이터, 깨진 분석, 비용이 많이 드는 재처리 작업으로 이어질 수 있습니다. 팀은 일관성을 복원하기 위해 테라바이트의 데이터를 다시 수집하거나 정리해야 할 수도 있으며, 이는 운영 효율성과 클라우드 스토리지 예산에 직접적인 타격을 줍니다.

- **분절된 거버넌스는 컴플라이언스 및 감사 위험을 증가시킵니다:** 스키마가 Git 리포지토리, 스프레드시트 또는 개별 서비스에 존재할 때, 거버넌스를 적용하거나 추적성을 보장하는 것은 거의 불가능해집니다. 누락된 문서화와 버전 격차는 감사를 고통스럽게 만들어, 컴플라이언스 위험과 법적 노출을 증가시킵니다. 특히 계보와 구조가 중요한 금융 서비스와 같은 규제 산업에서 더욱 그렇습니다.

이러한 비효율성은 엔지니어링 속도와 재무 성과를 조용히 갉아먹습니다. Confluent Schema Registry와 같은 중앙 집중식 레지스트리를 통해 거버넌스를 중앙 집중화하면, 조직은 데이터 스트림 전반에 걸쳐 호환성을 보장하고, 드리프트를 최소화하며, 스키마 검증을 자동화하여 이러한 비용 함정이 확장되기 전에 방지할 수 있습니다.

### 스키마 드리프트는 어떻게 Kafka TCO를 부풀리는가?

스키마 드리프트는 Kafka 스키마 관리에서 가장 과소평가된 비용 동인 중 하나입니다. Kafka 환경에서 이것이 발생하면, 동일한 토픽에 대해 서로 다른 프로듀서나 환경이 일치하지 않는 스키마 버전을 사용하고 있다는 것을 의미합니다.

스키마는 종종 분산된 소유권, 수동 프로세스 또는 불량한 버전 관리로 인해 일관되지 않게 진화합니다. 이것은 누락된 필드나 변경된 데이터 타입과 같은 작은 불일치로 시작될 수 있지만, 다운스트림 결과는 전체 데이터 흐름을 통해 파급됩니다. 시간이 지남에 따라, 예상치 못한 데이터 구조 변경의 축적은 데이터에 대한 신뢰를 침식하고, 팀 간의 마찰을 만들며, 비용을 크게 부풀립니다.

스키마 드리프트가 데이터 파이프라인 전반에 걸쳐 숨겨진 비용을 유발하는 네 가지 방법을 분석해 보겠습니다:

#### 비용이 많이 드는 데이터 재처리

호환되지 않는 스키마가 컨슈머에 도달하면, 파이프라인은 종종 조용히 실패하거나 잘못된 형식의 레코드를 생성합니다. 그러면 팀은 일관성을 복원하기 위해 며칠 동안 과거 데이터를 재처리하는 데 시간을 소비합니다. 이는 내장된 스키마 진화 제어로 피할 수 있었을 컴퓨팅, 스토리지, 엔지니어링 시간을 소모합니다.

#### 깨진 파이프라인 디버깅

단일 스키마 불일치는 여러 다운스트림 시스템을 깨뜨릴 수 있습니다. 근본 원인을 감지하는 것은 거의 간단하지 않습니다. 엔지니어는 로그, 스키마 버전, 컨슈머 코드를 추적해야 합니다. 이러한 반응적 디버깅은 장기간의 다운타임, SLA 위반, 개발자 생산성 손실로 이어집니다.

#### 비즈니스 다운타임과 데이터 접근 불가

스키마 드리프트는 단지 기술 부채만 유발하는 것이 아닙니다. 비즈니스 운영을 방해합니다. 분석 대시보드나 머신러닝 모델이 손상된 스트림에 의존할 때, 의사결정자는 실시간 인사이트에 대한 접근을 잃게 됩니다. 데이터 사용 불가의 매 시간은 실질적인 비즈니스 손실로 이어집니다.

#### 인프라 비용 증가

잦은 재처리, 중복 데이터를 위한 추가 스토리지, 반복적인 수집 주기 모두 클라우드 비용에 추가됩니다. 사소해 보이는 스키마 실수가 컴퓨팅 비용을 배로 늘리고 연간 수천 달러씩 TCO를 부풀릴 수 있습니다.

스키마 드리프트는 특히 변경 데이터 캡처(CDC) 파이프라인에 영향을 미칩니다. 여기서 데이터베이스 스키마 변경은 Kafka를 통해 다운스트림 시스템으로 올바르게 전파되어야 합니다. 단일 호환되지 않는 CDC 이벤트가 전체 데이터 동기화 프로세스를 중단시킬 수 있습니다.

## Kafka 스키마 관리에서 피해야 할 일반적인 실수는 무엇인가?

조직들은 동일한 스키마 관리 오류를 반복적으로 범하며, 각 오류는 운영 비용 증가에 기여합니다:

**호환성 모드를 적용하지 않는 것**이 가장 빈번한 실수입니다. 호환성 모드(예: 역방향, 순방향, 전체, 없음)는 어떤 스키마 변경이 허용되는지를 정의합니다. 테스트와 적용 규칙 없이, 개발자는 부주의하게 파괴적인 변경을 배포합니다. 해결책은 간단해 보입니다. 적용을 추가하면 됩니다. 그러나 조직이 필요성을 인식할 때쯤이면, 수백 개의 호환되지 않는 스키마가 이미 프로덕션에 존재합니다.

**각 팀이 독립적으로 스키마를 정의하도록 두면** 조직적 사일로가 생깁니다. 결제 팀은 타임스탬프를 문자열로 사용하고, 재고 팀은 에포크 밀리초를 사용합니다. 둘 다 같은 개념을 다르게 표현하여, 다운스트림 컨슈머가 여러 형식을 처리하도록 강제합니다. 이 분절화는 교차 기능 데이터 제품을 구축하는 것을 기하급수적으로 어렵게 만듭니다.

**레지스트리 대신 임시로 스키마를 저장하면** 중앙 집중식 거버넌스의 가능성이 완전히 사라집니다. 스키마는 애플리케이션 코드, 구성 파일, 위키, 개발자 노트북에 흩어져 있습니다. 버전 관리가 불가능해집니다. 프로덕션에서 실제로 실행 중인 스키마 버전을 발견하려면 여러 시스템에 걸친 탐정 작업이 필요합니다.

**스키마 변경에 대한 모니터링 부재**는 컨슈머가 실패할 때까지 팀이 호환성 문제가 발생했는지 알지 못한다는 것을 의미합니다. 스키마 진화 패턴에 대한 가시성이 없으면, 조직은 위험한 관행을 식별하거나 팀에 적절한 절차를 교육할 수 없습니다.

Kafka 개발자 모범 사례를 따르면 이러한 함정이 개발 문화에 내재되어 비즈니스에 영향을 미치기 전에 피할 수 있습니다.

### 부실한 스키마 관리의 비즈니스 영향

부적절한 스키마 관리의 기술적 비용은 결국 경영진이 주목하는 비즈니스 수준의 재무적 영향으로 이어집니다:

#### 개발자 생산성 손실이 더 높은 인재 비용을 유발

개발자가 수동으로 스키마를 검증하거나, 컨슈머 장애를 디버깅하거나, 데이터 불일치를 조정할 때마다 팀의 효율성이 떨어집니다. 새로운 기능을 구축하는 대신 소방 대응에 나서고 있는 것입니다. 1년에 걸쳐 이러한 비효율성은 팀당 수백 시간의 엔지니어링 시간 낭비로 누적될 수 있으며, 이는 혁신을 가속화할 수 있었을 시간입니다.

#### 장애로 인한 SLA 위약금과 수익 손실

단일 호환되지 않는 스키마가 중요한 프로덕션 파이프라인을 깨뜨릴 수 있습니다. 데이터 기반 비즈니스의 경우, 단 한 시간의 다운타임도 분석을 지연시키고, 고객 경험에 영향을 미치거나, 수익을 창출하는 서비스를 중단시킬 수 있습니다. 이러한 숨겨진 장애는 가동 시간 손실과 계약상 SLA 위약금을 통해 조용히 Kafka TCO를 부풀립니다.

#### 컴플라이언스 및 감사 위험 비용 증가

중앙 집중식 스키마 거버넌스가 없으면, 데이터 계보와 구조적 변경을 추적하는 것이 거의 불가능해집니다. 이러한 가시성 부족은 특히 데이터 추적성과 구조적 무결성이 요구되는 GDPR이나 HIPAA와 같은 규정하에서 주요 컴플라이언스 과제를 제기합니다. 시정 조치나 벌금의 비용은 적절한 거버넌스 비용을 훨씬 초과합니다.

## Confluent Schema Registry는 어떻게 Kafka TCO를 줄이는가?

잘못 관리하면, 스키마 관리는 조용히 데이터 인프라에서 가장 비용이 많이 들고 오류가 발생하기 쉬운 부분 중 하나가 될 수 있습니다. Confluent Schema Registry를 사용하면 검증을 확장 가능하게 자동화하고, 호환성을 적용하며, 엔드투엔드 스키마 거버넌스를 제공할 수 있습니다. 이 모든 것이 총 소유 비용(TCO)을 극적으로 줄여줍니다.

DIY 스키마 관리 접근 방식과 비교했을 때 Confluent Schema Registry를 사용하면 어떻게 측정 가능한 비즈니스 및 운영 효율성을 제공하는지 정확히 분석해 보겠습니다.

- **중앙 집중식 레지스트리 사용으로 DIY 난립을 제거합니다:** 팀, 리포지토리 또는 서비스 전반에 걸쳐 스키마를 저장하는 대신, Confluent Schema Registry는 모든 데이터 정의에 대한 단일 진실 원천 역할을 합니다. 중앙 집중화는 중복을 제거하고, 일관성을 보장하며, 프로듀서와 컨슈머 전반에 걸쳐 즉각적인 스키마 발견을 가능하게 합니다. 이는 여러 분절된 도구나 맞춤형 스키마 저장소를 유지하는 비용을 제거합니다.

- **내장된 호환성 검사로 비용이 많이 드는 파손을 방지합니다:** Confluent는 배포 전에 기존 버전에 대해 스키마 변경을 자동으로 검증합니다. 이러한 사전 호환성 적용은 데이터 손상과 다운스트림 장애를 방지하여 재처리, 디버깅, 프로덕션 복구에 드는 시간을 절약합니다.

- **거버넌스 및 감사 제어로 컴플라이언스 비용을 줄입니다:** Confluent의 스키마 거버넌스를 통해 조직은 스키마 진화를 추적하고, 소유권을 관리하며, 네이밍이나 구조 규약을 대규모로 적용할 수 있습니다. 감사 추적과 버전 이력은 컴플라이언스 감사를 더 빠르게 만들고 위약금 위험을 줄여줍니다. 거버넌스는 더 이상 수동이 아닙니다. 스트리밍 플랫폼에 내장되어 있습니다.

- **완전 관리형 거버넌스로 운영 오버헤드를 줄입니다:** 클라우드에서 Kafka를 실행하는 팀에게 Confluent Cloud는 Schema Registry를 완전 관리형 서비스로 제공합니다. 설정, 유지보수, 스케일링 고민이 필요 없습니다. 이는 DevOps 부담을 줄이고, 다운타임 위험을 제거하며, 예측 가능한 성능을 보장하여 운영 비용과 관리 복잡성을 모두 낮춥니다.

### DIY 스키마 관리 vs Confluent로 스키마 관리

| 측면 | DIY 스키마 관리 | Confluent Schema Registry |
|------|----------------|--------------------------|
| 검증 | 수동 코드 리뷰 | 자동화된 호환성 검사 |
| 저장소 | 리포지토리 전반에 흩어짐 | 중앙 집중식 레지스트리 |
| 거버넌스 | 맞춤형 도구 필요 | 내장된 감사 로그 및 접근 제어 |
| 운영 | 자체 관리 인프라 | 완전 관리형 서비스 |
| 비용 모델 | 엔지니어링 시간에 숨겨져 있음 | 예측 가능한 구독 |
| 장애 모드 | 프로덕션 장애 | 프로덕션 전 검증 오류 |

수동 스키마 관리를 제거함으로써, Confluent는 팀이 비용을 확장하지 않고도 자신감 있게 확장할 수 있도록 지원합니다.

## 스키마 관리 비용을 줄이기 위한 모범 사례

이러한 사례를 구현하면 레지스트리 솔루션에 관계없이 스키마 관련 운영 비용을 줄일 수 있습니다:

- **중앙 집중식 레지스트리 사용:** 항상 단일 레지스트리에서 스키마를 저장하고 관리하십시오. 여러 리포지토리나 서비스에 분산시키지 마십시오. Kafka용 스키마 레지스트리는 권위 있는 진실 원천 역할을 하여 모든 프로듀서와 컨슈머가 정렬된 상태를 유지합니다. 중앙 집중화는 중복을 제거하고 팀 간 스키마 발견을 단순화합니다.

- **기본적으로 호환성 모드 적용:** 명확한 호환성 정책(예: 역방향, 순방향 또는 전체)을 정의하고 자동으로 적용하십시오. 이는 호환되지 않는 스키마 변경이 기존 컨슈머를 깨뜨리는 것을 방지하여 비용이 많이 드는 장애와 재처리 주기를 줄여줍니다.

- **스키마 변경을 지속적으로 모니터링:** 스키마 모니터링을 CI/CD 파이프라인에 통합하십시오. 알림과 감사 로그를 통해 무단 또는 위험한 스키마 변경을 조기에 감지하여 프로덕션 서프라이즈를 피할 수 있습니다. 사전 모니터링은 반응적 디버깅에 드는 수많은 시간을 절약합니다.

- **스키마 거버넌스를 컴플라이언스 정책과 정렬:** 스키마 레지스트리는 개발자만을 위한 것이 아닙니다. 거버넌스 도구입니다. 네이밍 규약을 적용하고, 데이터 계보를 추적하며, 감사 준비가 된 스키마 이력을 제공하는 데 사용하십시오. Confluent의 중앙 집중식 스키마 거버넌스는 추가 오버헤드 없이 조직의 컴플라이언스를 보장합니다.

- **검증 및 버전 관리 자동화:** 수동 스키마 검증은 인적 오류에 취약합니다. Confluent Schema Registry와 같은 도구를 사용하여 검증 단계를 자동화하십시오. 이 도구는 내장된 버전 관리, 검증 API, 호환성 적용을 제공합니다. 자동화는 반복적인 수동 작업을 제거하고 진화를 예측 가능하게 유지합니다.

- **팀에 스키마 진화 사례 교육:** 엔지니어링 팀 전반에 걸쳐 일관된 스키마 진화 원칙을 촉진하십시오. 개발자가 호환성 모드, 폐기 전략, 레지스트리 워크플로우를 이해하도록 교육하면 장기적인 효율성이 보장되고 스키마 관련 이슈가 줄어듭니다.

### 지금 바로 스키마 관리를 단순화하세요

스키마 관리는 데이터 전략에서 숨겨진 비용이 되어서는 안 됩니다. Confluent Cloud의 Schema Registry와 전체 Stream Governance 제품군을 통해, 수동 검증을 제거하고, 데이터 드리프트를 방지하며, 모든 Kafka 데이터 스트림에 걸쳐 호환성을 보장할 수 있습니다. 이 모든 것이 Kafka TCO를 낮추면서 가능합니다.

Confluent Cloud 무료로 시작하여 빌드를 시작하세요. 플랫폼에는 $400의 크레딧, 관리형 Kafka 클러스터, 스트림 처리 기능, 사전 구축된 커넥터가 포함되어 있습니다. 인프라 복잡성 없이 첫 번째 실시간 앱을 만드는 데 필요한 모든 것이 갖춰져 있습니다.

## 스키마 관리 FAQ

### Kafka에서 스키마 관리란 무엇인가?

Kafka에서의 스키마 관리는 토픽을 통해 흐르는 메시지의 구조를 정의하고, 저장하고, 버전 관리하고, 검증하는 것을 포함합니다. 프로듀서와 컨슈머가 데이터 형식에 합의하도록 보장하여 역직렬화 실패와 데이터 손상을 방지합니다. 효과적인 스키마 관리에는 호환성 규칙 적용, 버전 이력 유지, 스키마 정의에 대한 중앙 집중식 접근 제공이 포함됩니다.

### 수동 스키마 관리는 왜 비용이 많이 드는가?

수동 스키마 관리는 호환성 검증, 배포 조정, 파괴적 변경으로 인한 프로덕션 인시던트 소방 대응에 엔지니어링 시간을 소비합니다. 손상된 데이터 재처리, 장애 대응, 맞춤형 거버넌스 도구 구축에 숨겨진 비용을 만들어냅니다. 조직은 일반적으로 이러한 비용을 과소평가하는데, 이는 단일 항목으로 나타나기보다 많은 팀과 인시던트에 분산되어 있기 때문입니다.

### 스키마 드리프트의 위험은 무엇인가?

스키마 드리프트는 서로 다른 서비스가 충돌하는 스키마 버전을 사용함에 따라 시스템 전반에 걸쳐 호환되지 않는 데이터 형식을 만들어냅니다. 위험에는 컨슈머가 메시지를 역직렬화할 수 없을 때 발생하는 프로덕션 장애, 비용이 많이 드는 재처리를 필요로 하는 데이터 손상, 잘못된 비즈니스 인사이트를 생성하는 실패한 분석 쿼리, PII 추적이 불가능해질 때의 컴플라이언스 위반이 포함됩니다. 스키마 드리프트는 팀이 조율되지 않은 변경을 함에 따라 시간이 지날수록 복합적으로 악화됩니다.

### Confluent Schema Registry는 어떻게 TCO를 낮추는가?

Confluent Schema Registry는 수동 엔지니어링 작업이 필요했을 검증을 자동화하고, 사전 배포 호환성 검사를 통해 프로덕션 장애를 방지하고, 자체 관리 인프라의 운영 오버헤드를 제거하며, 컴플라이언스 비용을 줄이는 내장 거버넌스 기능을 제공함으로써 총 소유 비용을 낮춥니다. 반응적 소방 대응에서 사전 예방으로의 전환은 엔지니어링 시간의 직접 비용과 기회 비용을 모두 줄여줍니다.

### Confluent Schema Registry를 사용하면 스키마 관리를 위한 다른 Kafka 도구가 여전히 필요한가?

Confluent Schema Registry가 Kafka에서 스키마 관리의 산업 표준이지만, 전문적인 프로덕션 환경에서 유일하게 사용하는 도구인 경우는 드뭅니다. CI/CD 통합, 가시성, 고급 거버넌스를 위한 도구로 보완할 필요가 있을 것입니다.

예를 들어, Schema Registry를 Maven/Gradle 플러그인, Confluent CLI 또는 코드 생성 도구와 결합하면 Schema Registry의 REST API를 수동으로 사용하는 복잡성을 피할 수 있습니다. 또한 데이터 계약에 대한 사람이 읽을 수 있는 뷰를 얻기 위해 모니터링 및 데이터 포털 도구를 사용하는 것이 좋습니다.

Confluent를 선택하면, Confluent Cloud의 완전 관리형 거버넌스 제품군인 Stream Governance의 일부인 데이터 카탈로그와 데이터 계보 추적기인 Data Portal과 Stream Lineage를 활용할 수 있습니다. 바닐라 Kafka를 실행하는 경우, 스키마를 시각화하고 프로듀서가 등록된 ID와 일치하는 데이터를 전송하는지 확인하는 데 사용할 수 있는 서드파티 Kafka 콘솔 UI가 있습니다.

### Schema Registry와 스키마 스토리지의 차이점은 무엇인가?

스키마 스토리지는 단순히 스키마 정의를 영속합니다. 이는 종종 코드 리포지토리나 구성 관리 시스템에 저장됩니다. Schema Registry는 능동적인 검증, 호환성 검사, 버전 관리, 런타임 스키마 해석을 제공합니다. 스토리지가 수동적인 반면, 레지스트리는 규칙을 능동적으로 적용하고 프로듀서 및 컨슈머와 통합하여 호환성 문제가 프로덕션에 도달하기 전에 방지합니다.
