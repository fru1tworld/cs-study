# 현상(Phenomena)

현상(Phenomenon)이란 데이터베이스가 수행하는 동작 중, 누군가가 어딘가에서 좋지 않다고 판단한 것을 말한다. 특정 현상의 허용 여부는 시스템의 일관성 모델(Consistency Model)에 따라 결정된다. 예를 들어, G1a는 커밋되지 않은 읽기(Read Uncommitted) 수준에서는 허용되지만, 커밋된 읽기(Read Committed) 수준에서는 금지된다.

## Adya 현상

이 현상들은 Adya의 1999년 논문에서 유래하며, 주로 트랜잭션 간의 의존성(dependency)을 통해 일관성 모델을 특성화한다.

## SQL 현상

ANSI SQL 표준과 Berenson 등의 후속 연구에서 제시된 현상들로, SQL 격리 수준(isolation level)을 특성화하는 데 사용된다. 이들은 데이터 흐름(dataflow)이 아닌 이벤트 순서(event order)를 기반으로 정의되어 있어, 분산 시스템에의 적용이 제한적이다.

## 시간적 현상(Temporal Phenomena)

이 현상들은 단일 프로세스 내에서 또는 여러 프로세스에 걸친 이벤트의 순서와 관련된다.

## 기타 현상(Other Phenomena)

## 1. G0 (쓰기 순환 / Write Cycle)

**현상 이름:** G0
**다른 명칭:** 쓰기 순환(Write Cycle)

### 비공식적 정의

트랜잭션들이 서로의 데이터를 덮어쓰는 상황이다. 쓰기 순환(Write Cycle)은 각 트랜잭션이 이전 트랜잭션에 대해 쓰기-쓰기 의존성(write-write dependency)을 가지는 트랜잭션 순환을 말한다.

### 구체적 예시

두 트랜잭션 Ti와 Tj가 객체 x와 y에 대해 작업한다. Ti가 x를 생성하고, Tj가 x를 갱신한 뒤 y를 생성하며, Ti가 y를 갱신한다. 결과적으로 Ti의 y 버전과 Tj의 x 버전이 혼합된 상태가 만들어진다.

### 형식적 정의 (Adya 논문, 3.2.1절)

이력(history) H가 현상 G0을 나타내려면, DSG(H)가 쓰기 의존성 간선(write-dependency edge)으로만 구성된 방향 순환(directed cycle)을 포함해야 한다.

### 일관성 모델

G0은 일반적으로 커밋되지 않은 읽기(Read Uncommitted) 및 그보다 강한 모든 일관성 모델에서 금지되는 것으로 이해된다. 그러나 ANSI SQL 표준에서는 이를 언급하지 않고 있다. ANSI SQL에서는 직렬화 가능(Serializable) 수준만이 G0을 방지한다.

### 관련 개념

G0은 더티 쓰기(Dirty Write, P0)의 보다 일반적이면서도 정밀한 형태를 포착한다. P0은 두 트랜잭션만을 다루며, 일부 직렬화 가능한 이력까지 배제하는 문제가 있다.

**참조:** Adya 논문

## 2. G1a (중단된 읽기 / Aborted Read)

**현상 이름:** G1a
**다른 명칭:** 중단된 읽기(Aborted Read)

### 비공식적 정의

중단된(aborted) 트랜잭션이 수행한 쓰기를 커밋된 트랜잭션이 읽는 현상으로, 중단된 트랜잭션의 정보가 커밋된 트랜잭션으로 누출된다.

### 예시

트랜잭션 Ti가 버전 xi를 쓰고 중단한다. 트랜잭션 Tj가 xi를 읽고 커밋한다. 이는 중단된 트랜잭션으로부터의 정보 누출을 나타낸다.

### 형식적 정의 (Adya 논문, 3.2.2절)

이력 H가 현상 G1a를 나타내려면, 중단된 트랜잭션 Ti와 커밋된 트랜잭션 Tj가 존재하여, Tj가 Ti에 의해 수정된 어떤 객체를 (술어를 통해서일 수도 있음) 읽은 경우에 해당한다.

### 일관성 모델

- **허용됨:** 커밋되지 않은 읽기(Read Uncommitted)
- **금지됨:** 커밋된 읽기(Read Committed) 및 그보다 강한 모든 모델

### 관련 현상 -- 더티 갱신(Dirty Update)

중단된 트랜잭션 Ti가 xi를 쓰고, 커밋된 트랜잭션 Tj가 xj = f(xi)를 쓴다. 최종적으로 커밋된 트랜잭션 Tk가 xj를 읽는다. 이는 변환된 쓰기를 통한 암묵적 정보 흐름을 나타낸다.

### 관련 개념

- ANSI SQL 이상 현상 P1(더티 읽기 / Dirty Read)에 대응
- Adya의 G1 프레임워크의 일부로, G1b(중간 읽기 / Intermediate Read) 및 G1c(순환적 정보 흐름 / Cyclic Information Flow)와 함께 구성됨
- 세 가지 G1 현상을 모두 배제하면 커밋된 읽기(Read Committed) 준수가 보장됨

## 3. G1b (중간 읽기 / Intermediate Read)

**현상 이름:** G1b
**다른 명칭:** 중간 읽기(Intermediate Read)

### 비공식적 정의

G1b는 커밋된 트랜잭션이 다른 트랜잭션이 생성한 중간 상태를 관찰할 때 발생한다. 구체적으로, 트랜잭션 Ti가 객체 x의 여러 버전(중간 버전과 최종 버전)을 쓸 때, 커밋된 트랜잭션 Tj가 최종 버전 대신 중간 버전을 읽는 경우이다.

### 예시

Ti가 x를 1로 설정한 후 2로 설정한다. 트랜잭션 Tj가 x=1을 읽고 커밋하여, Ti의 중간 상태가 누출된다.

### 주요 제약 조건

- G1b는 커밋된 트랜잭션의 읽기만 제약하며, 커밋되지 않은 트랜잭션은 임의로 읽을 수 있다
- 커밋되지 않은 읽기(Read Uncommitted)에서는 허용됨
- 커밋된 읽기(Read Committed)에서는 금지됨

### 형식적 정의 (Adya 논문, 3.2.2절)

이력 H가 현상 G1b를 나타내려면, 커밋된 트랜잭션 Tj가 트랜잭션 Ti가 쓴 객체 x의 버전(술어를 통해서일 수도 있음)을 읽되, 그 버전이 Ti의 x에 대한 최종 수정이 아닌 경우에 해당한다.

### 관련 현상 -- 중간 갱신(Intermediate Update)

중단된 트랜잭션이 쓴 중간 버전이 커밋된 트랜잭션의 쓰기에 영향을 미치고, 그 결과를 또 다른 커밋된 트랜잭션이 읽는 경우이다.

### 관련 개념

G1b는 ANSI SQL 이상 현상 P1(더티 읽기 / Dirty Read)을 구성하는 세 가지 현상 중 하나로, G1a(중단된 읽기 / Aborted Read) 및 G1c(순환적 정보 흐름 / Cyclic Information Flow)와 함께한다. 세 가지를 모두 배제하면 커밋된 읽기(Read Committed) 격리 수준이 보장된다.

## 4. G1c (순환적 정보 흐름 / Cyclic Information Flow)

**현상 이름:** G1c
**다른 명칭:** 순환적 정보 흐름(Cyclic Information Flow)

### 비공식적 정의

일련의 트랜잭션들이 서로의 쓰기를 읽거나 수정하는 방식으로 서로를 관찰하는 상황이다. 구체적으로 쓰기-쓰기(write-write) 또는 쓰기-읽기(write-read) 간선으로만 구성된 의존성 순환이다.

### 예시

객체가 고유한 정수를 추가(append)하는 리스트라고 하자. 트랜잭션 T1이 x에 1을 추가한다. 트랜잭션 T2가 x에 2를 추가한다. 그 후 T1이 x = [1, 2]를 읽는다. 두 트랜잭션 모두 커밋한다. 두 트랜잭션 사이에서 정보가 순환적으로 흘렀다.

### 중요한 제약 조건

G1c는 커밋된 트랜잭션만 제약한다. 중단된 트랜잭션이 수행한 쓰기는 의존성 그래프에 기여하지 않는다.

### 형식적 정의 (Adya 논문, 3.2.2절, 직접 직렬화 그래프 사용)

이력 H가 현상 G1c를 나타내려면, DSG(H)가 의존성 간선(dependency edge)으로만 구성된 방향 순환을 포함해야 한다.

### 일관성 모델

- **허용됨:** 커밋되지 않은 읽기(Read Uncommitted)
- **금지됨:** 커밋된 읽기(Read Committed) 및 그보다 강한 모델

### 관련 개념

- G1c는 ANSI SQL 이상 현상 P1(더티 읽기 / Dirty Read)과 관련됨
- Adya의 G1은 세 가지 현상을 포괄함: G1a(중단된 읽기 / Aborted Read), G1b(중간 읽기 / Intermediate Read), G1c
- G0(쓰기 순환 / Write Cycle)은 항상 G1c이기도 함

## 5. G-single (단일 반의존성 순환 / Single Anti-dependency Cycle)

**현상 이름:** G-single
**다른 명칭:** 단일 반의존성 순환(Single Anti-dependency Cycle)

### 비공식적 정의

트랜잭션 간에 정확히 하나의 읽기-쓰기 의존성(read-write dependency)을 포함하는 특정 유형의 G2 순환이다. 단일 트랜잭션이 다른 트랜잭션을 (쓰기와 읽기의 체인을 통해 간접적으로) 관찰하면서도 해당 트랜잭션의 일부 효과를 놓치는 상황을 설명한다. 이 순환은 임의 개수의 쓰기-쓰기(write-write) 및 쓰기-읽기(write-read) 간선과 정확히 하나의 읽기-쓰기 간선(반의존성 / anti-dependency)으로 구성된다.

### 예시

정수 레지스터에 대한 두 트랜잭션: 트랜잭션 Ti가 x = 1과 y = 1을 삽입한다. 트랜잭션 Tj가 x = 1을 읽고, y의 미생성(unborn) 상태를 읽는다. Tj는 Ti에 대해 쓰기-읽기 의존성을 가지며(Ti의 x 쓰기를 관찰), Ti는 Tj에 대해 읽기-쓰기 의존성을 가진다(Tj가 Ti의 쓰기 전 미생성 y를 관찰). Tj는 Ti의 효과 중 일부만 관찰하고 나머지를 놓쳤다.

### 형식적 정의 (Adya 논문, 4.1.1절)

G-single: 단일 반의존성 순환(Single Anti-dependency Cycles). 이력 H가 현상 G-single을 나타내려면, DSG(H)가 정확히 하나의 반의존성 간선(anti-dependency edge)을 가진 방향 순환을 포함해야 한다.

### 일관성 모델

- **허용됨:** 커밋된 읽기(Read Committed)
- **금지됨:** PL-2+, 스냅샷 격리(Snapshot Isolation) 및 그보다 강한 모델

### 관련 현상

G-nonadjacent, G2, P4, A5A

## 6. G-nonadjacent (비인접 반의존성 순환 / Non-adjacent Anti-dependency Cycle)

**현상 이름:** G-nonadjacent
**다른 명칭:** 비인접 반의존성 순환(Non-adjacent Anti-dependency Cycle)

### 정의

G-nonadjacent는 Jepsen이 명명한 G2 이상 현상의 한 유형으로, 임의 개수의 쓰기-쓰기(write-write) 및 쓰기-읽기(write-read) 간선, 하나 이상의 읽기-쓰기(read-write) 간선으로 구성된 의존성 순환이며, 두 개의 읽기-쓰기 간선이 서로 인접하지 않는다.

### 예시

네 개의 트랜잭션이 관련된 이력:
- T1이 x = 1을 쓴다
- T2가 x = 1을 읽고, y가 존재하지 않음을 관찰한다
- T3이 y = 3을 삽입한다
- T4가 술어 읽기(predicate read)를 수행하여 y = 3만 관찰한다

이는 쓰기-읽기 -> 읽기-쓰기 -> 쓰기-읽기 -> 읽기-쓰기의 순환을 만들며, 읽기-쓰기 간선이 비인접하게 배치된다.

### 일관성 모델

- 스냅샷 격리(Snapshot Isolation)는 G-nonadjacent를 (G1a, G1b, G1c와 함께) 금지한다
- 커밋된 읽기(Read Committed)는 G1a, G1b, G1c를 금지하지만 G-nonadjacent는 허용한다
- 스냅샷 격리는 순환 내에서 인접한 읽기-쓰기 간선만 허용한다

### 형식적 근거 -- 인용된 핵심 정리

- Adya (1999): G-single을 정의함
- Fekete 등 (2005): 직렬화 이력 그래프의 모든 순환은 T1 -rw-> T2 -rw-> T3 형태의 간선 시퀀스를 포함한다
- Cerone & Gotsman (2016): 인접 RW 의존성을 가진 SI 순환을 특성화함

### 관련 현상

- G-single: 정확히 하나의 읽기-쓰기 간선을 가진 특수한 경우
- 긴 포크(Long Fork): 여러 읽기-쓰기 간선을 가진 또 다른 경우
- G2: 임의의 읽기-쓰기 간선 배치를 허용하는 상위 범주

## 7. G2-item (항목 반의존성 순환 / Item Anti-dependency Cycle)

**현상 이름:** G2-item
**다른 명칭:** 항목 반의존성 순환(Item Anti-dependency Cycle)

### 비공식적 정의

단일 객체의 읽기와 관련된 트랜잭션 간의 읽기-쓰기 간선을 포함하는 특정 유형의 G2 순환이다. 이 의존성 순환은 쓰기-쓰기 간선, 쓰기-읽기 간선, 그리고 하나 이상의 항목 읽기-쓰기 간선(item read-write edge)으로 구성된다.

### 예시

두 트랜잭션이 정수 레지스터에 대해 작업한다. 트랜잭션 T1이 x를 읽고 (존재하지 않음을 확인), y = 1을 쓴다. 트랜잭션 T2가 y를 읽고 (존재하지 않음을 확인), x = 1을 쓴다. 각 트랜잭션이 상대방의 효과를 관찰하지 못하여, 반복 가능한 읽기(Repeatable Read)에서 위법인 순환이 생성된다.

### 형식적 정의 (Adya 논문, 3.2.4절)

G2-item: 항목 반의존성 순환(Item Anti-dependency Cycles). 이력 H가 현상 G2-item을 나타내려면, DSG(H)가 하나 이상의 항목 반의존성 간선(item anti-dependency edge)을 가진 방향 순환을 포함해야 한다.

### 일관성 모델

- **허용됨:** 커밋된 읽기(Read Committed)
- **금지됨:** 반복 가능한 읽기(Repeatable Read) 및 그보다 강한 모델

### 역사적 맥락

G2-item은 ANSI SQL의 반복 가능한 읽기(Repeatable Read) 버전을 특성화하기 위해 개발되었다. 해당 버전은 술어 기반 읽기를 제외한 모든 연산에 대해 긴 잠금(long lock)을 사용했다.

### 관련 개념

G2는 술어 읽기-쓰기 의존성을 허용하는 보다 일반적인 형태이다.

## 8. G2 (반의존성 순환 / Anti-dependency Cycle)

**현상 이름:** G2
**다른 명칭:** 반의존성 순환(Anti-dependency Cycle)

### 비공식적 정의

모든 트랜잭션이 서로 의존하는 트랜잭션의 고리로, 최소한 하나의 트랜잭션이 다른 트랜잭션이 읽은 데이터를 덮어쓴다. 구체적으로, 임의 개수의 쓰기-쓰기(write-write) 및 쓰기-읽기(write-read) 간선과 하나 이상의 읽기-쓰기(read-write) 간선으로 구성된 의존성 순환이다.

### 예시

두 트랜잭션이 각각 빈 데이터베이스에 대해 술어 읽기(predicate read)를 수행한 후, 서로 다른 값을 쓴다. 트랜잭션 T1이 x=1을, T2가 y=2를 쓴다. T2는 T1에 대해 읽기-쓰기 의존성을 가진다(x의 미생성 상태를 관찰했으므로). T1은 T2에 대해 읽기-쓰기 의존성을 가진다. 어느 트랜잭션도 상대방의 효과를 관찰하지 못하여, 이 이력은 직렬화 불가능(non-Serializable)하다.

### 형식적 정의 (Adya 논문, 3.2.3절)

G2: 반의존성 순환(Anti-dependency Cycles). 이력 H가 현상 G2를 나타내려면, DSG(H)가 하나 이상의 반의존성 간선(anti-dependency edge)을 가진 방향 순환을 포함해야 한다.

### 일관성 모델

- **허용됨:** 반복 가능한 읽기(Repeatable Read)
- **금지됨:** 직렬화 가능(Serializable) 및 그보다 강한 모델

### 하위 유형

1. **G2-item:** 술어가 관여하지 않음; 반복 가능한 읽기(Repeatable Read)에서 금지됨
2. **G-single:** 정확히 하나의 읽기-쓰기 의존성
3. **G-nonadjacent:** 인접한 읽기-쓰기 의존성 없음; 스냅샷 격리(Snapshot Isolation)에서 금지됨

## 9. P0 (더티 쓰기 / Dirty Write)

**현상 이름:** P0
**다른 명칭:** 더티 쓰기(Dirty Write)

### 비공식적 정의

한 트랜잭션이 다른 트랜잭션이 완료되기 전에 그 트랜잭션의 쓰기를 덮어쓰는 현상이다. 트랜잭션 Ti가 버전 xi를 쓰고, 트랜잭션 Tj가 x의 다음 버전을 쓴다. 이후 두 트랜잭션이 임의의 순서로 커밋하거나 중단한다.

### 주요 특성

- 이벤트의 전체 순서(total order)로 정의됨
- 분산 시스템에서는 이 순서가 의미있게 존재하지 않을 수 있음

### 형식적 정의 (Berenson 등)

더티 쓰기는 트랜잭션 T1이 x를 쓴 후, T1의 중단이나 커밋 전에 T2가 x를 쓸 때 발생한다: P0: w1[x] ... w2[x] ...

### 일관성 모델

P0은 ANSI SQL 정의에서 커밋되지 않은 읽기(Read Uncommitted), 커밋된 읽기(Read Committed), 반복 가능한 읽기(Repeatable Read)에서 허용된다 -- 놀랍게도. ANSI SQL에서는 직렬화 가능(Serializable) 수준만이 이를 방지한다.

### 역사적 맥락

1990년대 후반, Berenson 등과 Adya는 P0의 포함이 우연한 것이었다고 주장했다. 표준의 저자들은 원래 이를 방지하려 했을 것이다. 그러나 표준 문구는 변경되지 않은 채 남아있다.

### 관련 개념

Adya의 G0(쓰기 순환 / Write Cycle)은 여러 트랜잭션의 간섭하는 쓰기로 쓰기 격리를 일반화한다. G0은 더 정밀하다 -- G0을 금지해도 P0 금지처럼 직렬화 가능한 이력까지 배제하지 않는다.

---

## 10. P1 (더티 읽기 / Dirty Read)

**현상 이름:** P1
**다른 명칭:** 더티 읽기(Dirty Read)

### 비공식적 정의

한 트랜잭션이 다른 진행 중인 트랜잭션이 쓴 데이터를, 쓰기 트랜잭션이 완료되기 전에 읽는 현상이다. 이후 두 트랜잭션 모두 커밋하거나 중단할 수 있다.

### 형식적 정의 (Berenson 등, 1995)

**넓은 해석:**
P1: w1[x] ... r2[x] ... ((c1 or a1) and (c2 or a2) in any order)

**엄격한 해석 (A1):**
A1: w1[x] ... r2[x] ... (a1 and c2 in any order)

### ANSI SQL 표준 정의

SQL 트랜잭션 T1이 행을 수정한다. SQL 트랜잭션 T2가 T1이 COMMIT을 수행하기 전에 해당 행을 읽는다. 이후 T1이 ROLLBACK을 수행하면, T2는 커밋되지 않은 행을 읽은 것이 된다.

### 주요 구별점

- **P1 대 A1:** P1은 모든 커밋/중단 조합을 포함하고, A1은 쓰기 트랜잭션이 중단되고 읽기 트랜잭션이 커밋되는 경우만 요구한다
- **Jepsen의 입장:** 시간과 상태에 대한 가정이 보편적으로 성립하지 않을 수 있으므로, P1보다 G1a, G1b, G1c를 선호한다
- **실제 시나리오:** P1 정의는 캐시 읽기나 레플리카 데이터를 고려하지 않는다

### 관련 현상

P1은 다음을 방지한다: 중단된 읽기(aborted read), G1b(중간 읽기 / Intermediate Read), 그리고 특정 G1c(순환적 정보 흐름 / Circular Information Flow) 시나리오

---

## 11. P2 (반복 불가능한 읽기 / Non-Repeatable Read / 퍼지 읽기 / Fuzzy Read)

**현상 이름:** P2
**다른 명칭:** 반복 불가능한 읽기(Non-Repeatable Read), 퍼지 읽기(Fuzzy Read)

### 비공식적 정의

한 트랜잭션이 다른 진행 중인 트랜잭션이 이미 읽은 데이터를 수정하는 현상이다. 트랜잭션 Ti가 버전 xi를 읽고, 다른 트랜잭션 Tj가 Ti가 완료되기 전에 xi를 쓴다. 이후 두 트랜잭션 모두 커밋하거나 중단할 수 있다.

### 정의의 한계에 대한 핵심 통찰

이 현상은 이벤트의 전체 순서로 정의되며, 시간과 상태에 대한 암묵적 가정이 성립하지 않을 수 있다. Ti가 Tj 시작 전에 커밋했다면, Tj가 xi-1을 읽은 후 xi를 읽는 것은 Ti와 동시적이지 않으므로 P2의 정의를 충족하지 않을 것이다.

### 형식적 해석 (Berenson 등, 1995)

**P2 (넓은 해석):** r1[x] ... w2[x] ... ((c1 or a1) and (c2 or a2) in any order)

**A2 (엄격한 해석):** r1[x] ... w2[x] ... c2 ... r1[x] ... c1

두 해석 모두 트랜잭션이 자신이 쓰지 않은 두 개의 서로 다른 버전을 읽는 것을 방지한다. A2는 A5A와 A5B를 허용하지만, P2는 이를 방지한다.

### 원문 출처 -- ANSI SQL 표준 정의

SQL 트랜잭션 T1이 행을 읽는다. SQL 트랜잭션 T2가 해당 행을 수정하거나 삭제하고 COMMIT을 수행한다.

### Jepsen의 입장

Jepsen은 더 정밀한 정의로서 P2보다 G1a, G1b, G1c를 선호한다.

### 관련 현상

- G2-item (보다 일반적인, 데이터 흐름 기반 대안)
- A5A (읽기 왜곡 / Read Skew -- 두 객체에 대한 보다 넓은 변형)

## 12. P3 (팬텀 / Phantom)

**현상 이름:** P3
**다른 명칭:** 팬텀(Phantom), 퍼지 읽기(Fuzzy Read)

### 비공식적 정의

P3은 한 트랜잭션이 술어(predicate)를 통해 읽고 있는 데이터를 다른 동시 트랜잭션이 수정할 때 발생한다. 그 효과로 트랜잭션 내에서 술어 읽기를 반복하면 다른 결과가 관찰된다.

### 형식적 정의 (Berenson 등)

트랜잭션 Ti가 술어 P를 읽고, 그 후 트랜잭션 Tj가 P에 해당하는 어떤 객체 x를 쓴다. 이후 Ti와 Tj가 임의의 순서로 커밋하거나 중단한다.

Jepsen의 해석: P3은 Ti가 술어 읽기 P를 수행하여 VSet(P)에 버전 xi가 포함될 때 발생한다. 그 후 Tj가 버전 xj를 쓴다. xi 또는 xj 중 최소한 하나가 P에 일치한다. 두 트랜잭션 모두 커밋하거나 중단할 수 있다.

### ANSI SQL 표준 정의

SQL 트랜잭션 T1이 특정 검색 조건을 만족하는 행 집합 N을 읽는다. SQL 트랜잭션 T2가 해당 조건에 일치하는 행을 생성하는 문장을 실행한다. T1이 읽기를 반복하면 다른 결과를 얻는다.

### 주요 구별점

- **P3 (넓은 해석):** 한 번의 읽기만 필요하며, 커밋/중단의 모든 조합이 허용됨
- **A3 (엄격한 해석):** 두 번의 읽기가 필요하며, 두 트랜잭션 모두 커밋해야 함

### 이상 현상 예시

Ti가 계좌 x에서 $10을 인출하고 (잔액 -> 0) 계좌 y에 $10을 입금할 때 (잔액 -> $10), "모든 계좌"를 관찰하는 동시 술어 읽기가 부분적인 효과를 보게 되어 잘못된 합계를 산출한다.

### 일관성 모델 맥락

- **금지됨:** 술어 안정성을 강조하는 모델
- **관련:** G2(Adya의 형식화)는 "표준이 의도했다고 생각되는 것"을 보다 일반적인 용어로 포착함

### 추가 사항

- 정의는 시간/상태에 대한 암묵적 가정에 의존함
- 삭제가 술어 변경으로 간주되는지 여부가 ANSI 표준에서 명시되지 않음
- Jepsen은 시간적 한계로 인해 P3보다 G2를 선호함

## 13. P4 (갱신 손실 / Lost Update)

**현상 이름:** P4
**다른 명칭:** 갱신 손실(Lost Update)

### 비공식적 정의

커밋된 트랜잭션의 효과가 사실상 손실되는 현상이다. 일부 또는 전체 효과가 손실될 수 있다.

### 형식적 정의 (Berenson 등)

트랜잭션 Ti가 객체 x의 버전 xi를 읽은 후, 해당 읽기를 기반으로 버전 xk를 쓴다. Ti의 읽기와 쓰기 사이에 트랜잭션 Tj가 xj를 쓴다. Ti가 xj를 관찰하지 못했으므로, Tj의 쓰기는 손실된다. 형식적으로: r1[x]...w2[x]...w1[x]...c1

### 대안적 형식화 (의존성 그래프)

갱신 손실은 의존성 그래프로 표현될 수 있다. Tj가 Ti에 대해 rw-의존하고(Ti가 읽은 버전을 덮어씀), Ti가 Tj에 대해 ww-의존하는(Tj의 쓰기를 덮어씀) 경우이다. 이는 G-single의 특수한 경우이다.

### 일관성 모델

- **허용됨:** 커밋된 읽기(Read Committed)
- **금지됨:** 스냅샷 격리(Snapshot Isolation) 및 반복 가능한 읽기(Repeatable Read) (많은 구현이 실제로는 허용하지만)
- **또한 금지됨:** 읽기에 커서를 사용하는 경우의 커서 안정성(Cursor Stability) (P4C라 불림)

### 예시

블록 파티에서 두 이웃이 각각 디저트를 기여한다. 둘 다 빈 쟁반을 읽고, 각각 쓴다 -- 아이린이 스니커두들 쿠키를 추가하고, 콘레지가 피치 코블러를 추가한다. 콘레지의 쓰기가 아이린의 쓰기를 덮어써서, 아이린의 기여가 사라진다.

### 감지 방법

두 트랜잭션이 같은 객체 버전을 읽고, 서로의 쓰기를 관찰하지 않은 채 둘 다 쓰면, 하나의 쓰기가 반드시 손실된다.

## 14. A5A (읽기 왜곡 / Read Skew)

**현상 이름:** A5A
**다른 명칭:** 읽기 왜곡(Read Skew)

### 비공식적 정의

한 트랜잭션이 다른 트랜잭션의 효과를 관찰하는 동시에 관찰하지 못하는 현상이다. 형식적 정의는 트랜잭션 Ti가 값 xi를 읽고, 트랜잭션 Tj가 xj와 yj를 모두 쓴 후 커밋하며, 이후 Ti가 yj를 읽은 뒤 커밋하거나 중단하는 시나리오를 설명한다.

### 형식적 정의 (Berenson 등)

r1[x]...w2[x]...w2[y]...c2...r1[y]...(c1 or a1)

### 의존성 분석

Tj는 Ti에 대해 읽기-쓰기 의존성을 가진다(Ti가 읽은 x 버전을 덮어썼으므로). 그러나 Ti는 Tj의 y 쓰기를 관찰하여 Tj에 대한 쓰기-읽기 의존성을 생성한다. 읽기 왜곡(Read Skew)은 G-single의 특수한 경우이다.

### 예시

여왕이 커튼 뒤에서 하이힐을 스틸레토에서 펌프스로 바꾼다. 여왕의 어머니가 왼발에서 스틸레토를, 오른발에서 펌프스를 관찰하고, 발목을 삐게 될 것이라고 잘못 결론짓는다.

### 일관성 모델

- **허용됨:** 커밋된 읽기(Read Committed)
- **금지됨:** 스냅샷 격리(Snapshot Isolation) 및 반복 가능한 읽기(Repeatable Read) (ANSI SQL의 반복 가능한 읽기 정의는 실제로 이를 허용하며, MySQL 8.0.34 분석에서 입증됨)

### 관련 현상

P2는 두 객체가 동일한 A5A의 퇴화된 형태이다. P4와 마찬가지로, A5A는 Adya의 연구에 따르면 G-single의 특수한 경우이다.

**참고:** Jepsen은 Adya의 접근 방식을 따라 중단된 트랜잭션이 비일관적인 값을 관찰하는 것을 허용한다. A5A는 Ti가 커밋될 때만 발생한다.

## 15. A5B (쓰기 왜곡 / Write Skew / 짧은 포크 / Short Fork)

**현상 이름:** A5B
**다른 명칭:** 쓰기 왜곡(Write Skew), 짧은 포크(Short Fork)

### 비공식적 정의

A5B는 두 커밋된 트랜잭션이 서로 다른 객체를 쓰고, 각각이 상대방의 쓰기를 관찰하지 못할 때 발생한다. 객체들 사이에 제약 조건(합계 임계값 등)이 존재하면, 해당 제약 조건이 위반될 수 있다.

### 형식적 정의

트랜잭션 Ti가 xi를 읽고, 트랜잭션 Tj가 xi를 읽는다. 그 후 Ti가 yj를 쓰고, Tj가 xj를 쓴다. 두 트랜잭션 모두 커밋한다. 각 트랜잭션은 상대방이 읽은 버전을 덮어썼으므로 상대방에 대한 읽기-쓰기 의존성을 가진다. 쓰기 왜곡(Write Skew)은 G2-item의 특수한 경우이다.

### Berenson 등의 정의

T1이 x와 y를 읽고, 이것이 C()와 일관적이라고 하자. 그 후 T2가 x와 y를 읽고, x를 쓴 뒤 커밋한다. 그러면 T1이 y를 쓴다. x와 y 사이에 제약 조건이 있었다면, 그것이 위반될 수 있다.

이력 표기법: r1[x]...r2[y]...w1[y]...w2[x]...(c1 and c2 occur)

### 예시

두 전기 기사가 전기 패널에서 동시에 작업한다. 각각 현재 회로(200암페어 중 160암페어 사용 중)를 읽고, 서로 다른 슬롯에 30암페어 회로를 설치한다. 완료 시, 패널은 220암페어를 사용하게 되어 안전 제약 조건을 위반한다.

### 일관성 모델

- **허용됨:** 스냅샷 격리(Snapshot Isolation)
- **금지됨:** 반복 가능한 읽기(Repeatable Read), 직렬화 가능(Serializability)

### 관련 현상

- G2-item의 특수한 경우
- 다른 트랜잭션이 분기된 상태와 상호작용하면 긴 포크(Long Fork)가 됨

---

## 16. 긴 포크(Long Fork)

**현상 이름:** 긴 포크(Long Fork)

### 비공식적 정의

긴 포크는 두 트랜잭션이 동시에 서로 겹치지 않는(disjoint) 갱신을 수행하여 시스템 상태가 논리적으로 포크(fork)되고, 다른 트랜잭션들이 포크가 병합되기 전에 분기된 상태와 상호작용할 수 있게 되는 현상이다. 짧은 포크(Short Fork, A5B/쓰기 왜곡)와의 차이점은, 긴 포크에서는 각 포크가 단일 트랜잭션이 아닌 여러 트랜잭션을 포함한다는 것이다.

### 형식적 특성

최소 예시는 네 개의 트랜잭션을 필요로 한다: 서로 겹치지 않는 두 쓰기 트랜잭션(T1, T2)과 두 읽기 트랜잭션 -- T3은 T1을 관찰하되 T2를 관찰하지 못하고, T4는 T2를 관찰하되 T1을 관찰하지 못한다. 이는 다음 의존성 순환을 생성한다: T1 wr T3 rw T2 wr T4 rw T1. 긴 포크는 G-nonadjacent의 하위 유형으로 분류되며, 항상 쓰기-읽기(write-read) 또는 쓰기-쓰기(write-write) 의존성으로 분리된 최소 두 개의 읽기-쓰기(read-write) 의존성을 포함한다.

### 예시

네 명의 협업자(Alexis, Latrice, Abby, Li)가 논문을 작성한다. Latrice는 오프라인으로 작업하고, Alexis는 독립적으로 편집한다. Abby와 Li는 각각 다른 버전을 검토한다. 동기화 시, Alexis의 절 번호 변경이 Latrice의 상호 참조를 부정확하게 만든다 -- 최종 병합 전까지 감지할 수 없다.

### 기술적 예시 (Sovran 등)

초기 상태: A=B=0. T1이 A<-1을 쓰고, T3이 B<-1을 쓴다(동시). T2가 A=1, B=0을 읽고, T4가 A=0, B=1을 읽는다. 최종적으로 T5가 A=B=1을 읽는다.

### 일관성 모델

- **금지됨:** 직렬화 가능(Serializability) 및 스냅샷 격리(Snapshot Isolation)
- **허용됨:** 병렬 스냅샷 격리(Parallel Snapshot Isolation)

### 관련 개념

- 짧은 포크(Short Fork): 각 포크가 단일 트랜잭션만 포함하는 경우
- G-nonadjacent: 긴 포크는 그 특정 하위 유형임

## 17. 프로세스(Process)

**현상 이름:** 프로세스(Process)

### 핵심 정의

두 연산이 동일한 논리적 프로세스 내에서 실행되는 현상이다. 한 연산이 다른 연산보다 먼저 실행되면, 이들은 프로세스 의존성(process dependency)을 가진다. 각 데이터베이스 세션이 단일 프로세스와 동일할 때, 프로세스 현상은 세션 현상(session phenomena)이 된다.

### 비공식적 예시

ADHD를 가진 사람이 상세한 쇼핑 목록을 작성한다. 가게에서 우유를 사려고 메모 앱을 확인하는데, (버그로 인해) 쇼핑 목록의 이전 상태를 관찰한다. 필요한 우유를 사지 못한 채 귀가한다.

### 형식적 정의

의존성 그래프 G에서 순환으로 정의된 현상 P가 주어졌을 때, 현상 P-process는 그래프 (G U P)에서의 대응하는 순환을 나타낸다. 여기서 P는 G 내 트랜잭션 간의 프로세스 의존성 그래프를 나타낸다.

### 관련 개념

- 읽기-쓰기 의존성과 연관됨
- G2-item과의 유사성에 따라 G2-item-process로 명명됨
- 단일 프로세스가 아닌 모든 프로세스에 걸치는 실시간(Real-Time) 현상과 구별됨

### 프로세스 현상을 다루는 일관성 모델

- 순차 일관성(Sequential Consistency)은 단일 객체 시스템에서 프로세스 현상을 방지함
- 강한 세션 직렬화 가능(Strong Session Serializable)은 G1c-process 및 G2-process 순환을 금지함
- 강한 세션 스냅샷 격리(Strong Session Snapshot Isolation)는 G-nonadjacent-process를 배제함

## 18. 실시간(Real-Time)

**현상 이름:** 실시간(Real-Time)

### 비공식적 정의

실시간 현상은 두 연산이 벽시계 시간(wall-clock time)에 의해 순서가 매겨져, 하나가 다른 하나가 시작되기 전에 완료되어야 할 때, 추가적인 의존성 관계와 결합되어 발생하는 현상이다.

### 주요 개념

**실시간 의존성:** 동기화된 시계로 측정하여, 한 이벤트가 다른 이벤트가 시작되기 전에 완료되어야 하는 관계이다.

### 실시간 현상의 예

- **G2-item-realtime:** 쓰기가 완료된 후 읽기가 시작되었지만 쓰기의 효과를 관찰하지 못하는 부실한 읽기(Stale Read)
- **G0-realtime:** 쓰기-쓰기 간선과 실시간 간선으로 구성된 순환

### 설명적 예시

한 화가가 벽화를 그린다. 2주 후 그래피티 아티스트가 태그를 남긴다. 비정상적인 벽돌 특성으로 인해 그래피티가 벽화 아래에 나타난다 -- 실시간에서는 나중에 일어났지만 논리적으로는 이전에 실행된 것으로 보인다. 이 이력은 직렬화 가능(Serializable)이고 강한 세션 직렬화 가능(Strong Session Serializable)이지만, 강한 직렬화 가능(Strong Serializable)은 아니다.

### 형식적 정의

의존성 그래프 G에서 순환으로 정의된 현상 P가 주어졌을 때, 현상 P-realtime은 그래프 (G U R)에서의 대응하는 순환을 나타낸다. 여기서 R은 트랜잭션 간의 실시간 의존성을 나타낸다.

### 일관성 모델

**실시간 현상을 금지하는 모델:**
- 선형화 가능(Linearizable) 시스템
- 강한 직렬화 가능(Strong Serializable) (G1c-realtime과 G2-realtime을 금지)
- 강한 스냅샷 격리(Strong Snapshot Isolation) (G-nonadjacent-realtime을 배제)
- 강한 세션 직렬화 가능(Strong Session Serializable)

**실시간 현상을 허용하는 모델:**
- 직렬화 가능(Serializable) (G1c와 G2를 금지하지만 실시간 변형은 금지하지 않음)
- 스냅샷 격리(Snapshot Isolation)
- 세션 직렬화 가능(Session Serializable)

### 관련 개념

- 부실한 읽기(Stale Read): 나중의 읽기가 이전 쓰기의 효과를 놓치는 실시간 현상
- 프로세스 현상(Process Phenomena): 실시간과 유사하지만 동일 프로세스의 연산에 한정됨

## 19. 부실한 읽기(Stale Read)

**현상 이름:** 부실한 읽기(Stale Read)

### 형식적 정의

읽기가 이전에 완료된 쓰기의 결과를 관찰하지 못하는 현상이다. T1이 객체 x의 버전 x1을 쓰는 연산이고, T2가 실시간으로 T1이 완료된 후에 시작하는 연산이라 하자. 부실한 읽기는 T2가 x의 버전 순서에서 x1 이전의 어떤 버전 x0을 관찰할 때 발생한다.

### 비공식적 예시

누군가가 소셜 미디어에 사진을 올리고 부모님에게 전화하여 알린다. 부모님이 프로필 페이지를 로드하면, 사진이 아직 보이지 않는다 -- 이것이 부실한 읽기를 보여준다.

### 일관성 모델

- **금지됨:** 선형화 가능(Linearizability) 및 엄격한 직렬화 가능(Strict Serializability)
- **허용됨:** 순차 일관성(Sequential Consistency) 및 직렬화 가능(Serializability)

### 역사적 맥락

이 개념은 1970년대 캐시 일관성(cache coherence) 연구를 통한 하드웨어 설계에서 유래했다. 1988년까지 "부실한(stale)"이라는 용어가 무효화된 캐시 라인에 적용되었다. "부실한 읽기(stale read)"라는 용어는 1990년대 중반까지 분산 시스템 문헌에서 확립되었다.

### 관련 개념

- 실시간 현상(Real-Time phenomenon)의 한 유형
- k-부실성(k-staleness) 및 t-가시성(t-visibility) 측정과 관련
- 확률적으로 한정된 부실성(probabilistically bounded staleness)과 연결됨

## 20. 쓰기 손실(Lost Write)

**현상 이름:** 쓰기 손실(Lost Write)
**다른 명칭:** 쓰기 유실(Write Loss)

### 비공식적 정의

시스템이 쓰기의 완료를 확인(acknowledge)했지만, 일정 시간이 지난 후 해당 쓰기의 효과가 더 이상 존재하지 않는 현상이다.

### 주요 특성

- 쓰기는 즉시 손실될 수 있고(예: 디스크 쓰기 전 서버 충돌), 나중에 손실될 수도 있다(예: 안전하지 않은 리더 선출)
- 분산 시스템에서, 노드들은 쓰기가 발생했는지에 대해 의견이 다를 수 있다
- 서버에 존재하지만 읽기에서 반환되지 않는 쓰기는 기능적으로 쓰기 손실이다

### 형식적 정의 (Jepsen의 실무 정의)

쓰기 w가 손실된 것으로 간주되려면, 일정 시간 t 이후에 모든 읽기가 w의 인과적 후손(causal descendant)이 아닌 시스템 상태를 반영해야 한다.

### 관련 현상과의 구별

쓰기 손실은 부실한 읽기(Stale Read)와 구별된다: 부실한 읽기는 결국 해소되지만, 쓰기 손실은 영구적이다. 시스템 안정화(quiescence)를 허용하고 최종 읽기를 수행하여 구별을 판단한다.

### 관련 구별점

성공적인 갱신은 쓰기 손실과 다르다 -- 레지스터에서 x = 1로 설정한 후 x = 2로 설정하는 것은 반드시 쓰기 손실이 아니다.
