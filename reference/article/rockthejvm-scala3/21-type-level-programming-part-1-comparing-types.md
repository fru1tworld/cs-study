# Scala 3의 타입 레벨 프로그래밍: Part 1 - 타입 비교

## 소개

이 글에서는 Scala 3의 타입 레벨 프로그래밍을 다룬다. 값이 아닌 *타입*에 대해 연산을 표현하고, 컴파일러가 컴파일 시점에 타입 관계를 파악하도록 만드는 기법이다. 어떤 계산 문제를 타입 제약이나 타입 관계 강제로 인코딩하면, 컴파일러가 적절한 타입을 탐색하고 추론하고 검증하는 과정에서 우리의 계산을 "풀어낸다".

이 글은 **Scala 3**에 초점을 맞춘 2부작 시리즈의 첫 번째 파트다. 이전에 Scala 2를 위한 타입 레벨 프로그래밍 미니 시리즈를 게시한 적이 있는데, implicit이 여전히 지원되므로 Scala 3에서도 동작한다. 이번 버전에서는 다음을 활용한다:

- Given
- 제네릭
- 타입 멤버 (다음 글에서 다룰 예정)

이 미니 시리즈의 최종 목표는 *컴파일 시점에 타입 레벨 리스트를 정렬*하는 것이다. 다양성을 위해 이번 시리즈에서는 Scala 2 시리즈에서 구현했던 병합 정렬 대신 *타입 레벨 퀵소트*를 수행한다. 이번 글에서는 그보다 훨씬 작은 문제인, 컴파일 시점에 숫자(타입으로 표현된)를 올바르게 비교하는 방법에 집중한다.

## 타입 레벨 숫자 인코딩

자연수 집합은 0부터 무한대까지 하나씩 세어나가는 모든 수로 구성된다. 자연수 집합을 생각하는 한 가지 방법은 끝없이 이어지는 숫자들을 떠올리는 것이다. 또 다른 방법은 — 그리고 훨씬 유용한 방법이기도 한데 — 이 숫자들을 서로 간의 *관계*로 정의하는 것이다. 이를 숫자의 페아노 표현(Peano representation)이라 한다.

```scala
trait Nat
class _0 extends Nat
class Succ[A <: Nat] extends Nat
```

위 코드만으로 자연수 집합 전체를 표현할 수 있다. 숫자 0과 임의의 두 숫자 사이의 후속(succession) 관계, 이 두 가지 규칙만으로 자연수 집합의 어떤 수든 얻을 수 있다. 예를 들어 숫자 127은 `Succ`가 127번 감싸는 `Succ[Succ[Succ[...[_0]]]]`이다. 이후 예제를 위해 타입 별칭 몇 가지를 추가하자:

```scala
type _1 = Succ[_0]
type _2 = Succ[_1] // Succ[Succ[_0]]
type _3 = Succ[_2] // Succ[Succ[Succ[_0]]]
type _4 = Succ[_3] // ... 이하 동일
type _5 = Succ[_4]
```

**참고:** 이 표기법은 Scala 3에 최근 추가된 리터럴 타입과는 아무 관련이 없다. 리터럴 타입은 서로 간에 관계가 없으므로, 관계를 직접 모델링해야 한다.

## 타입 레벨에서 숫자 순서 정하기

Scala 2 버전의 글에서와 마찬가지로, 타입 간의 순서 관계를 표현하는 작은 trait을 만들 것이다. 이름은 `<`로 짓겠다. 가능하니까.

```scala
trait <[A <: Nat, B <: Nat]
```

예를 들어, `<[_1, _2]`의 인스턴스가 *존재한다면* 숫자 1이 2보다 작다고 말할 것이다. 이것이 타입 레벨에서 타입 간 "작다" 관계를 해석하는 방식이다.

이 관계를 모델링하기 위해, "1이 3보다 작은가"라는 질문의 참 값을 컴파일러가 증명하도록 `given` 인스턴스를 만든다. 컴파일러가 적절한 인스턴스에 접근할 수 있으면 — 우리가 명시적으로 정의했든, 컴파일러가 합성으로 구성했든 — 관계가 증명된다. 컴파일러가 적절한 타입의 `given` 인스턴스에 접근하거나 생성할 수 없다면, 관계는 증명 불가능하므로 거짓으로 간주된다.

## 타입 레벨에서 순서 증명하기

먼저 이 관계를 수학적으로 어떻게 증명할 수 있는지 생각해보자. 우리가 가진 것이 "숫자" `_0`과 연속된 두 숫자 사이의 후속 관계뿐일 때, `_3`이 `_5`보다 "작다"는 것을 어떻게 판단할 수 있을까?

`<` 관계의 공리(기본 진리)를 작성하면 이 문제를 풀 수 있다. 다음과 같이 생각해보자:

1. 0은 다른 모든 자연수보다 작다. 0을 제외하면 모든 자연수는 다른 어떤 수의 후속자다. 따라서 자연수 집합의 모든 수에 대해, 그 수의 후속자는 0보다 크다고 말할 수 있다.

2. 숫자 A가 숫자 B보다 작다는 것을 어떻게든 증명할 수 있다면, A의 후속자가 B의 후속자보다 작다는 것도 참이다. 이 공리는 역방향으로도 표현할 수 있다: A가 B보다 작은 것은 A의 선행자가 B의 선행자보다 작을 *때에만, 그리고 그때에 한해* 참이다.

예제를 하나 살펴보자. `_3 < _5`인지 판단하고 싶다고 하자.

- 두 번째 공리에 의해, `_3 < _5`는 `_2 < _4`와 동치다. `_3 = Succ[_2]`이고 `_5 = Succ[_4]`이기 때문이다. 그러므로 `_2 < _4`를 증명해야 한다.
- 마찬가지로 두 번째 공리에 의해, `_2 < _4`는 `_1 < _3`과 동치다.
- 다시 두 번째 공리: `_1 < _3`은 `_0 < _2`와 동치다.
- `_0 < _2`는 첫 번째 공리에 의해 참이다.
- 따라서 역순으로 거슬러 올라가면, `_3 < _5`는 참이다.

이것을 Scala 코드로 어떻게 표현할까? 컴파일러가 `<` 타입에 대한 `given` 인스턴스를 탐색하거나 생성하도록 만든다.

- 첫 번째 공리에 대해, 모든 자연수 `N`에 대해 `<[_0, Succ[N]]` given을 컴파일러가 생성할 수 있게 한다.
- 두 번째 공리에 대해, 스코프에 `<[A, B]` given이 이미 있으면 `<[Succ[A], Succ[B]]` given을 컴파일러가 생성할 수 있게 한다.

코드로 옮기면 다음과 같은 `given`이 된다:

```scala
given basic[B <: Nat]: <[_0, Succ[B]] with {}
given inductive[A <: Nat, B <: Nat](using lt: <[A, B]): <[Succ[A], Succ[B]] with {}
```

**참고:** Scala 3.1(이 글 작성 시점에는 아직 미출시)에서는 간결하게 `<[_0, Succ[B]] with {}` 대신 `<[_0, Succ[B]]()`로 쓸 수 있게 될 예정이다.

편의를 위해 위의 given들을 `<`의 컴패니언 객체에 넣겠다. 코드는 다음과 같다:

```scala
object < {
  given basic[B <: Nat]: <[_0, Succ[B]] with {}
  given inductive[A <: Nat, B <: Nat](using lt: <[A, B]): <[Succ[A], Succ[B]] with {}
  def apply[A <: Nat, B <: Nat](using lt: <[A, B]) = lt
}
```

`apply` 메서드도 추가했는데, 컴파일러가 찾거나 합성할 수 있는 요청된 타입의 `given` 값을 그대로 꺼내주는 역할을 한다.

## 타입 레벨 비교 테스트

위 코드를 갖추면, 컴파일러가 타입 관계를 어떻게 검증하는지(또는 못하는지) 확인할 수 있다. 테스트는 간단하다: 코드가 컴파일되는가?

```scala
val validComparison = <[_3, _5]
```

유효하다. 코드가 컴파일된다. 컴파일러가 내부적으로 수행하는 과정을 살펴보자:

- `<[_3, _5]` 타입의 `given`을 요청하고 있다. 컴파일러는 앞서 정의한 두 given 생성기 중 하나로 만들 수 있는지 확인한다.
- 두 번째 `given`이 동작한다. 컴파일러가 아직 생성되지 않은 given의 시그니처를 보면: `<[Succ[_2], Succ[_4]]`. 하지만 이를 생성하려면 `<[_2, _4]` 타입의 given을 찾거나 만들어야 한다.
- 같은 이유로 다시 두 번째 `given`을 쓸 수 있다. 이를 위해 컴파일러는 `<[_1, _3]` 타입의 given이 필요하다.
- 마찬가지로, 컴파일러가 `<[_1, _3]` 타입의 `given`을 만들려면 `<[_0, _2]` 타입의 given을 찾아야 한다.
- 이번에는 *첫 번째* given을 사용한다. `_2 = Succ[_1]`이므로 컴파일러가 `<[_0, _2]` 타입의 given을 만들 수 있다.
- 역순으로 거슬러 올라가면, 컴파일러는 요청한 given을 만들기 위한 모든 중간 given을 생성할 수 있다.

다시 말해, `<[_3, _5]` 관계, 또는 중위 표기로 `_3 < _5`가 증명 가능하므로 참이다. 컴파일러가 위의 수학적 귀납법에서 우리가 밟은 것과 정확히 같은 순서로 given 인스턴스를 생성했다는 점에 주목하라.

같은 메커니즘으로, 수학적으로 올바른 어떤 "작다" 관계든 타입 간에 증명할 수 있다. `given` 명세는 이전 절에서 정리한 공리의 수학적 기술과 동일하다. 반대로, `_4 < _2`처럼 유효하지 않은 관계는 컴파일러가 적절한 타입의 given을 찾거나 생성할 수 없으므로 증명 불가능하며, 코드가 컴파일되지 않는다.

## 타입 레벨 비교 확장

같은 원리를 적용해 "작거나 같다" 관계도 만들어보자. 이것은 다음 글에서 다룰 퀵소트에 유용하게 쓰일 것이다.

공리는 유사하다:

- 0은 다른 모든 수보다 작거나 같다. Scala로 표현하면, 모든 `N <: Nat`에 대해 `<=[_0, N]` given을 항상 생성할 수 있다는 의미다.
- `<=`의 두 번째 공리는 `<`의 것과 동일하다: 숫자 타입 `A`가 다른 숫자 타입 `B`보다 "작거나 같다"면, `Succ[A] <= Succ[B]`도 성립한다.

정리하면 코드는 다음과 같다:

```scala
trait <=[A <: Nat, B <: Nat]
object <= {
  given lteBasic[B <: Nat]: <=[_0, B] with {}
  given inductive[A <: Nat, B <: Nat](using lte: <=[A, B]): <=[Succ[A], Succ[B]] with {}
  def apply[A <: Nat, B <: Nat](using lte: <=[A, B]) = lte
}
```

올바른 관계는 역시 컴파일된다:

```scala
val lteTest = <=[_3, _3]
val lteTest2 = <=[_3, _5]
```

반면 올바르지 않은 관계는 컴파일되지 않는다:

```scala
val invalidLte = <=[_5, _3] // 적절한 given을 찾을 수 없음
```

## 결론

전체 코드는 다음과 같다:

```scala
trait Nat
class _0 extends Nat
class Succ[N <: Nat] extends Nat

type _1 = Succ[_0]
type _2 = Succ[_1] // Succ[Succ[_0]]
type _3 = Succ[_2]
type _4 = Succ[_3]
type _5 = Succ[_4]

// 처음에는 "LessThan"으로 이름 짓고, 나중에 "<"로 리팩터링
trait <[A <: Nat, B <: Nat]
object < {
  given basic[B <: Nat]: <[_0, Succ[B]] with {}
  given inductive[A <: Nat, B <: Nat](using lt: <[A, B]): <[Succ[A], Succ[B]] with {}
  def apply[A <: Nat, B <: Nat](using lt: <[A, B]) = lt
}
val comparison = <[_1, _3]
/*
  <.apply[_1, _3] -> <[_1, _3] 필요
  inductive[_0, _2] -> <[_0, _2] 필요
  ltBasic[_1] -> <[_0, Succ[_1]] == <[_0, _2] 생성
 */

trait <=[A <: Nat, B <: Nat]
object <= {
  given lteBasic[B <: Nat]: <=[_0, B] with {}
  given inductive[A <: Nat, B <: Nat](using lte: <=[A, B]): <=[Succ[A], Succ[B]] with {}
  def apply[A <: Nat, B <: Nat](using lte: <=[A, B]) = lte
}
val lteTest: _1 <= _1 = <=[_1, _1]
```

이 코드로 자연수를 타입 레벨 페아노 표현으로 나타내고, 비교 "연산자"의 적절한 `given` 값이 존재하는지를 증명하는 방식으로 컴파일 시점에 숫자를 비교하는 방법을 보여주었다(모두 타입 레벨에서).

다음 파트에서는 이 기술을 한 단계 끌어올려 타입에 대한 퀵소트를 구현할 것이다.
