# Scala 3: 들여쓰기 빠르게 살펴보기

## 들어가며

Scala 경험이 많든 적든, 한 번쯤은 Scala 문법의 비일관성이나 대안 문법 때문에 헷갈린 적이 있을 것이다. 이 글에서는 Scala 3가 기존의 다양한 코드 스타일 위에 새로 추가한 들여쓰기 기반 문법을 체계적으로 정리한다.

## if 표현식

Scala 2에서는 if 표현식을 여러 줄에 걸쳐 쓸 수 있다.

```scala
val aCondition = if (2 > 3)
  "bigger"
else "smaller"
```

물론 실무에서 이렇게 쓰는 경우는 드물다. 흔히 쓰이는 스타일은 다음과 같다.

```scala
// 한 줄
val aCondition = if (2 > 3) "bigger" else "smaller"

// Java 스타일
val aCondition2 =
  if (2 > 3) {
    "bigger"
  } else {
    "smaller"
  }

// 간결한 형태
val aCondition3 =
    if (2 > 3) "bigger"
    else "smaller"
```

위 스타일은 전부 Scala 3에서도 그대로 사용할 수 있다. 다만 Scala 3에서는 중괄호와 if 조건의 괄호 없이, 들여쓰기만으로 구조를 나타낼 수 있다.

```scala
val anIndentedCondition =
  if 2 > 3
    "bigger"
  else
    "smaller"
```

`2 > 3` 주변의 괄호를 제거하면, 이 표현식에서 들여쓰기가 의미를 갖게 된다. `else` 분기는 최소한 `if`와 같은 들여쓰기 수준이어야 한다. 실무에서 if/else 분기의 들여쓰기가 뒤섞이면(특히 주니어 코드나 급하게 작성한 코드에서) 매우 혼란스러운데, 컴파일러가 이를 잡아주므로 도움이 된다. Java 스타일에 비해 코드도 좀 더 깔끔해진다.

if 분기의 결과를 조건과 같은 줄에 쓰려면 `then` 키워드를 추가해야 한다.

```scala
// 한 줄
val aCondition = if 2 > 3 then "bigger" else "smaller"

// 간결한 형태
val aCondition =
  if 2 > 3 then "bigger"
  else "smaller"
```

이게 전부다.

## for 컴프리헨션

비슷한 변경이 `for` 컴프리헨션 같은 다른 제어 구조에도 적용되었다. 핵심은 동일하다. 중괄호를 쓰지 않으면 들여쓰기가 의미를 갖는다. 기존에는 이렇게 작성했다.

```scala
for {
  n <- List(1, 2, 3)
  c <- List('a', 'b', 'c')
} yield s"$c$n"
```

Scala 3에서는 다음과 같이 쓸 수 있다.

```scala
for
  n <- List(1, 2, 3)
  c <- List('a', 'b', 'c')
yield s"$c$n"
```

같은 코드처럼 보일 것이다. Scala 2 버전도 적절한 들여쓰기를 포함하고 있었기 때문이다. 중괄호가 없어지면 그 들여쓰기가 필수가 된다는 차이만 있다.

별것 아니다.

## 들여쓰기 영역

구문 분석 컴파일 단계에서 컴파일러는 특정 토큰 뒤에 들여쓰기 영역(indentation region)을 추가한다. 공식 문서에 따르면, `if`, `then`, `else`, `while`, `do`, `try`, `catch`, `finally`, `for`, `yield`, `match`, `return` 키워드와 `=`, `=>`, `<-` 토큰 뒤에서 줄을 바꾸고 윗줄보다 한 단계 깊게 코드를 작성할 수 있다. 이 들여쓰기 수준이 내부에 중첩되는 다른 표현식의 기준선이 된다. 컴파일러는 줄 바꿈 위치에 가상의 토큰(`<indent>` 또는 `<outdent>`)을 삽입해서, 코드를 여러 번 읽지 않고도 들여쓰기 수준을 추적한다.

덕분에 메서드를 중괄호 없이 구현할 수 있다.

```scala
def computeMeaningOfLife(year: Int): Int =
  println("thinking...")

  42 // <-- 들여쓰기가 의미를 가지므로 이 메서드의 스코프에 포함된다
```

이 부분이 특히 헷갈릴 수 있다. 이해하기 쉽게 설명하자면, `=`와 반환값 사이에 컴파일러가 중괄호를 삽입했다고 생각하면 된다. 메서드 구현이 코드 블록이 되고, 코드 블록은 하나의 표현식이며, 그 값은 마지막 구성 표현식의 값으로 결정된다. 여기서 들여쓰기가 의미를 갖는다는 것은, 보이지 않는 코드 블록이 존재한다는 뜻이다.

클래스, 트레이트, 오브젝트, enum을 정의할 때 콜론(`:`)과 줄 바꿈을 사용해도 들여쓰기 영역이 생성된다. 컴파일러는 이 토큰을 "줄 끝의 콜론", 즉 "콜론 뒤에 들여쓰기된 내용을 정의한다"로 해석한다. 예시는 다음과 같다.

```scala
class Animal:
  def eat(): Unit

trait Carnivore:
  def eat(animal: Animal): Unit

object Carnivore:
  def apply(name: String): Carnivore = ???
```

확장 메서드(extension method)와 given 인스턴스에도 같은 규칙이 적용된다(이후 글에서 다룰 예정이다).

```scala
given myOrder as Ordering[Int]: // <-- 들여쓰기 영역 시작
  def compare(x: Int, y: Int) =
    if x < y then 1 // 새 문법에 주목
    else if x > y then -1
    else 0
```

이제 백만 달러짜리 질문이 남았다. 스페이스로 들여쓸까, 탭으로 들여쓸까? Scala 3는 둘 다 지원하며, 컴파일러는 들여쓰기를 비교할 수 있다.

두 줄이 같은 수의 스페이스 또는 같은 수의 탭으로 시작하면, 들여쓰기를 비교할 수 있고 그 뒤의 스페이스/탭 수로 비교한다. 예를 들어 탭 3개 + 스페이스 1개는 탭 3개 + 스페이스 2개보다 "덜 들여쓴 것"이다. 마찬가지로 스페이스 4개 + 탭 1개는 스페이스 4개 + 탭 2개보다 "덜 들여쓴 것"이다. 자연스러운 결과다. 두 줄이 같은 수의 스페이스/탭으로 시작하지 않으면 비교할 수 없다. 예를 들어 탭 3개와 스페이스 6개는 비교 불가능하다. 컴파일러는 들여쓰기 영역의 기준선을 항상 알고 있으므로, 어떤 줄이 기준선과 비교 불가능하면 오류를 발생시킨다.

결론은 이것이다. 스페이스와 탭을 섞지 마라. 자기 진영을 정하고 끝까지 밀고 나가되, 절대 섞지 마라.

## 함수 인자

Scala 2에서 인자가 하나인 메서드에 흔히 쓰이는 스타일이 있다.

```scala
val aFuture = Future {
  // some code
}
```

이것은 `Future(/* 코드 블록 */)` 과 같은데, 처음 접하는 사람에게는 혼란스러울 수 있다. 이 중괄호 문법은 인자가 하나인 모든 메서드에 적용할 수 있다.

핵심은 이것이다. 이 문법은 그대로 유지된다. 여기서는 들여쓰기가 의미를 갖지 않는다. `-Yindent-colons` 컴파일러 옵션을 사용하면, 줄 끝에 `:`를 붙이고 메서드 인자를 들여쓰기로 작성할 수 있다.

```scala
val nextYear = Future:
  2021
```

또는

```scala
List(1,2,3).map:
  x => x + 1
```

하지만 이 기능은 핵심 언어 규칙에 포함되지 않는다.

## end 토큰

들여쓰기 영역이라는 새 개념은 코드가 길어지면 혼란을 줄 수 있다. 특히 클래스 정의 안에 많은 코드를 작성하는 경우가 그렇다. 연쇄적인 if 표현식도 들여쓰기 상태에서 읽기 어려워질 수 있다. 분기별 코드가 여러 줄에 걸치거나 중간에 빈 줄이 들어가면, 어디서부터 어디까지가 어떤 코드에 속하는지 파악하기 어렵다.

이를 해결하기 위해 Scala 3는 `end` 토큰을 도입했다. 어떤 코드가 어떤 들여쓰기 영역에 속하는지 구분할 수 있게 해준다.

```scala
class Animal:
  def eat(): Unit =
    if System.currentTimeMillis() % 2 == 0
      println("even")
    else
      println("odd")
    end if
  end eat
end Animal
```

이 예제는 단순하지만, 같은 파일에 100배 이상의 코드가 들어가거나, 들여쓰기가 깊어지거나, 줄 사이에 빈 줄이 많을 때 `end` 토큰은 확실히 유용하다. `end` 토큰은 컴파일러에 영향을 주지 않지만, 코드를 읽기 쉽게 만들기 위해 추가되었다.

## 마치며

Scala 3의 들여쓰기에 대해 알아야 할 내용은 거의 다 다뤘다. 개인적으로 Scala 3의 새 들여쓰기 규칙을 처음 봤을 때 "대체 뭘 한 거야?!"라고 생각했다. 하지만 체계적으로 접근하고 보니 그렇게 나쁘지 않고, 오히려 도움이 될 수도 있다. 이미 여러 사람이 "이 스타일로 작성하면 Scala가 더 빠르게 쓰이고, 읽기 쉽고, 전반적으로 생산성이 높아진다"고 보고하고 있다. 시간이 지나야 판단할 수 있겠지만, 이 글을 읽고 "생각보다 나쁘지 않네"라고 느낀 사람이 한 명이라도 있다면 이 글은 성공이다.
