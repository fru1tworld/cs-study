## 1. 개요

### ISO 8601이란?

ISO 8601은 국제표준화기구(ISO)가 제정한 날짜와 시간의 표기에 관한 국제 표준이다. 정식 명칭은 "Date and time -- Representations for information interchange"로, 정보 교환을 위한 날짜와 시간의 표현 방법을 정의한다.

### 왜 만들어졌는가?

날짜와 시간의 표기 방식은 국가, 문화권, 시스템마다 크게 다르다. 예를 들어 `01/02/03`이라는 표기는 다음과 같이 해석될 수 있다.

| 해석 | 형식 | 사용 지역 |
|------|------|-----------|
| 2003년 1월 2일 | MM/DD/YY | 미국 |
| 2003년 2월 1일 | DD/MM/YY | 유럽 대부분 |
| 2001년 2월 3일 | YY/MM/DD | 한국, 일본, 중국 |

이러한 혼란은 다음과 같은 문제를 초래한다.

- 국제 커뮤니케이션 오류: 다국적 팀 간 날짜 해석 충돌
- 소프트웨어 호환성 문제: 시스템 간 데이터 교환 시 날짜 파싱 실패
- 법적/재무적 오류: 계약 날짜, 결제일 등의 잘못된 해석
- 정렬 불가능: 문자열 기반 정렬 시 날짜 순서가 보장되지 않음

ISO 8601은 이러한 모호성을 제거하고, 전 세계가 하나의 통일된 형식으로 날짜와 시간을 교환할 수 있도록 설계되었다.

### 핵심 원칙

- 큰 단위에서 작은 단위 순서: 연 -> 월 -> 일 -> 시 -> 분 -> 초
- 고정 자릿수: 각 단위를 고정된 자릿수로 표현 (예: 월은 항상 두 자리)
- 사전식 정렬 가능: 문자열 그대로 정렬하면 시간순 정렬이 됨
- 그레고리력 기반: 그레고리력(Gregorian calendar)을 기준으로 함

## 2. 역사

### 제정 배경

날짜 표기의 국제 표준화 노력은 20세기 중반부터 시작되었다. 각국의 서로 다른 날짜 형식은 국제 무역, 과학 연구, 통신 분야에서 지속적인 문제를 일으켰고, 이를 해결하기 위한 국제적 합의가 필요했다.

### 주요 연혁

| 연도 | 내용 |
|------|------|
| 1971 | ISO 2014 제정 - 날짜의 숫자 표현에 대한 최초의 ISO 표준 |
| 1973 | ISO 2015 제정 - 주 번호(week numbering) 표현 표준 |
| 1976 | ISO 2711 제정 - 시간의 숫자 표현에 대한 표준 |
| 1988 | ISO 8601 초판 제정 - 위 세 표준을 통합하여 날짜와 시간 표현의 단일 표준으로 제정 |
| 2000 | ISO 8601:2000 개정 - 첫 번째 개정판 발행 |
| 2004 | ISO 8601:2004 개정 - 두 번째 개정판, 기존 내용 명확화 및 보완 |
| 2019 | ISO 8601-1:2019 및 ISO 8601-2:2019 발행 - 표준을 두 부분으로 분리 |

### ISO 8601:2019의 구조

2019년 개정에서 표준은 두 부분으로 나뉘었다.

- ISO 8601-1:2019 (Part 1): 기본 규칙 - 날짜와 시간 표현의 기본적인 형식을 정의
- ISO 8601-2:2019 (Part 2): 확장 - 프로파일에 대한 확장 표현을 정의 (불확실한 날짜, 음수 연도, 계절 표현 등)

### 관련 표준

- RFC 3339: IETF에서 정의한 인터넷 프로토콜용 날짜/시간 형식. ISO 8601의 부분 집합(subset)이며, 웹 개발에서 가장 널리 사용된다.
- W3C DTF: W3C에서 정의한 날짜/시간 형식. 역시 ISO 8601의 프로파일이다.

## 3. 날짜 표현 형식

ISO 8601은 날짜를 표현하는 두 가지 형태를 정의한다: 기본 형식(Basic format)과 확장 형식(Extended format).

### 완전한 날짜 표현

#### 확장 형식 (Extended format)

구분자로 하이픈(`-`)을 사용한다.

```
YYYY-MM-DD
```

| 요소 | 설명 | 범위 |
|------|------|------|
| YYYY | 연도 (4자리) | 0000 ~ 9999 |
| MM | 월 (2자리, 앞에 0 채움) | 01 ~ 12 |
| DD | 일 (2자리, 앞에 0 채움) | 01 ~ 28/29/30/31 |

예시:

```
2025-01-31    # 2025년 1월 31일
2000-12-25    # 2000년 12월 25일
1999-06-05    # 1999년 6월 5일
```

#### 기본 형식 (Basic format)

구분자 없이 숫자만 나열한다.

```
YYYYMMDD
```

예시:

```
20250131      # 2025년 1월 31일
20001225      # 2000년 12월 25일
```

### 축약 표현

ISO 8601은 날짜의 일부 요소를 생략한 축약 표현도 허용한다.

```
YYYY-MM       # 연도와 월만 (예: 2025-01)
YYYY          # 연도만 (예: 2025)
--MM-DD       # 월과 일만 (예: --01-31) - 특정 연도 없이
--MM          # 월만 (예: --01)
```

> 주의: 축약 표현은 ISO 8601:2004까지 허용되었으나, ISO 8601-1:2019에서는 일부 축약 형식이 제거되거나 변경되었다. RFC 3339에서는 완전한 날짜 표현(`YYYY-MM-DD`)만 사용한다.

### 확장 연도 표현

4자리를 넘어가는 연도가 필요한 경우, 부호 접두사를 사용한다.

```
+YYYYYY-MM-DD    # 6자리 연도 (양수)
-YYYYYY-MM-DD    # 6자리 연도 (음수, 기원전)
```

예시:

```
+002025-01-31    # 서기 2025년 1월 31일
-000044-03-15    # 기원전 44년 3월 15일 (율리우스 카이사르 암살일 근처)
```

## 4. 시간 표현 형식

ISO 8601의 시간 표현은 24시간제를 사용하며, AM/PM 표기를 사용하지 않는다.

### 완전한 시간 표현

#### 확장 형식

구분자로 콜론(`:`)을 사용한다.

```
hh:mm:ss
```

| 요소 | 설명 | 범위 |
|------|------|------|
| hh | 시 (2자리) | 00 ~ 23 (24는 자정 특수 표현) |
| mm | 분 (2자리) | 00 ~ 59 |
| ss | 초 (2자리) | 00 ~ 59 (윤초 시 60) |

예시:

```
14:30:00      # 오후 2시 30분 0초
00:00:00      # 자정 (하루의 시작)
23:59:59      # 하루의 마지막 초
```

#### 기본 형식

```
hhmmss
```

예시:

```
143000        # 오후 2시 30분 0초
235959        # 오후 11시 59분 59초
```

### 소수 초 (Fractional seconds)

초 이하의 정밀도가 필요한 경우, 소수점 이하 자릿수를 추가한다. 소수점 구분자는 점(`.`) 또는 쉼표(`,`)를 사용할 수 있으며, ISO 8601에서는 쉼표를 선호하지만 실무에서는 점이 더 널리 쓰인다.

```
hh:mm:ss.sss       # 밀리초까지 (3자리)
hh:mm:ss.ssssss    # 마이크로초까지 (6자리)
hh:mm:ss.sssssssss # 나노초까지 (9자리)
```

예시:

```
14:30:00.000       # 밀리초 정밀도
14:30:00.123456    # 마이크로초 정밀도
09:15:30,5         # 쉼표 구분자 사용 (0.5초)
```

### 축약 표현

```
hh:mm             # 시와 분만 (예: 14:30)
hh                # 시만 (예: 14)
```

### 자정의 표현

자정은 두 가지 방법으로 표현할 수 있다.

```
00:00:00          # 하루의 시작으로서의 자정 (권장)
24:00:00          # 하루의 끝으로서의 자정
```

`24:00:00`은 해당 날짜의 끝을 의미하며, 다음 날 `00:00:00`과 동일한 시점이다. ISO 8601:2004에서 정의되어 있으나, 많은 프로그래밍 라이브러리에서 `24:00:00`을 지원하지 않으므로 주의가 필요하다.

## 5. 날짜와 시간 결합

날짜와 시간을 하나의 문자열로 표현할 때는 대문자 `T`를 구분자로 사용한다.

### 형식

```
YYYY-MM-DDThh:mm:ss         # 확장 형식
YYYYMMDDThhmmss             # 기본 형식
```

### 예시

```
2025-01-31T14:30:00         # 2025년 1월 31일 오후 2시 30분
2025-01-31T09:15:30.123     # 밀리초 포함
2025-01-31T00:00:00         # 2025년 1월 31일의 시작
```

### T 구분자의 역할

`T`는 날짜 부분과 시간 부분의 경계를 명확히 한다. 이를 통해 파서(parser)가 문자열에서 날짜와 시간을 명확하게 분리할 수 있다.

```
2025-01-31T14:30:00
|________| |______|
  날짜   T  시간
```

> 참고: RFC 3339에서는 `T` 대신 공백(space)을 허용하는 경우도 있다. 예를 들어 `2025-01-31 14:30:00`과 같은 표현이 가능하지만, 엄격한 ISO 8601 준수를 위해서는 `T`를 사용해야 한다.

## 6. 타임존 표현

시간 표현 뒤에 타임존 지정자(time zone designator)를 붙여 해당 시간이 어느 시간대를 기준으로 하는지 명시한다.

### UTC (협정 세계시)

UTC 시간임을 나타내려면 시간 뒤에 대문자 `Z`를 붙인다. `Z`는 "Zulu time"에서 유래한다.

```
2025-01-31T14:30:00Z        # UTC 기준 오후 2시 30분
2025-01-31T00:00:00.000Z    # UTC 기준 자정
```

### UTC 오프셋

UTC와의 시차를 `+hh:mm` 또는 `-hh:mm` 형식으로 표현한다.

```
2025-01-31T14:30:00+09:00   # UTC+9 (한국 표준시, KST)
2025-01-31T14:30:00-05:00   # UTC-5 (미국 동부 표준시, EST)
2025-01-31T14:30:00+05:30   # UTC+5:30 (인도 표준시, IST)
2025-01-31T14:30:00+00:00   # UTC (Z와 동일)
```

기본 형식에서는 콜론을 생략한다.

```
2025-01-31T14:30:00+0900    # 기본 형식
2025-01-31T14:30:00-0500    # 기본 형식
```

### 주요 타임존 오프셋 예시

| 타임존 | 오프셋 | 지역 |
|--------|--------|------|
| KST | +09:00 | 한국 |
| JST | +09:00 | 일본 |
| CST | +08:00 | 중국 |
| IST | +05:30 | 인도 |
| CET | +01:00 | 중앙 유럽 |
| UTC/GMT | +00:00 / Z | 영국 (겨울) |
| EST | -05:00 | 미국 동부 |
| PST | -08:00 | 미국 서부 |

### 동일 시점의 다른 타임존 표현

다음 세 표현은 모두 동일한 순간을 나타낸다.

```
2025-01-31T14:30:00+09:00   # 한국 시간 오후 2시 30분
2025-01-31T05:30:00Z         # 같은 시점의 UTC 표현
2025-01-31T00:30:00-05:00   # 같은 시점의 미국 동부 시간
```

> 주의: ISO 8601은 타임존 이름(KST, EST 등)을 정의하지 않는다. 오직 UTC 오프셋만을 표현하며, IANA 타임존 데이터베이스(예: `Asia/Seoul`)와는 별개이다. DST(일광절약시간) 전환도 ISO 8601 자체에서는 다루지 않는다.

## 7. 기간(Duration) 표현

기간(duration)은 시간의 길이를 표현하며, 대문자 `P`(Period)로 시작한다.

### 형식

```
PnYnMnDTnHnMnS
```

| 요소 | 설명 |
|------|------|
| P | 기간의 시작을 나타내는 접두사 (필수) |
| nY | n년 |
| nM | n개월 (T 앞에 위치) |
| nD | n일 |
| T | 날짜 요소와 시간 요소의 구분자 |
| nH | n시간 |
| nM | n분 (T 뒤에 위치) |
| nS | n초 |

### 예시

```
P1Y                # 1년
P3M                # 3개월
P30D               # 30일
P1Y6M              # 1년 6개월
P1Y2M3D            # 1년 2개월 3일
PT1H               # 1시간
PT30M              # 30분
PT45S              # 45초
PT1H30M            # 1시간 30분
P1DT12H            # 1일 12시간
P2Y6M15DT10H30M20S # 2년 6개월 15일 10시간 30분 20초
PT0.5S             # 0.5초 (소수 초 허용)
P0D                # 0일 (길이 없는 기간)
```

### 대체 형식

기간을 날짜/시간과 유사한 형식으로도 표현할 수 있다.

```
P0001-02-03T04:05:06   # 1년 2개월 3일 4시간 5분 6초
P00010203T040506       # 동일 (기본 형식)
```

### 주의사항

- `P`는 반드시 맨 앞에 위치해야 한다.
- 시간 요소(H, M, S)가 포함되면 `T` 구분자가 반드시 있어야 한다.
- 월(`M`)과 분(`M`)은 같은 문자를 사용하므로, `T`의 위치로 구분한다. `T` 앞의 `M`은 월, `T` 뒤의 `M`은 분이다.
- 0인 요소는 생략 가능하다. 단, 최소 하나의 요소는 있어야 한다.

## 8. 시간 간격(Interval) 표현

시간 간격(interval)은 두 시점 사이의 범위를 나타내며, 슬래시(`/`)를 구분자로 사용한다. 네 가지 표현 방법이 있다.

### 표현 방법

#### 방법 1: 시작 시점 / 종료 시점

```
<시작>/<종료>
```

```
2025-01-01T00:00:00Z/2025-12-31T23:59:59Z
# 2025년 1월 1일 00:00:00 UTC부터 2025년 12월 31일 23:59:59 UTC까지

2025-01-01/2025-06-30
# 2025년 1월 1일부터 6월 30일까지
```

#### 방법 2: 시작 시점 / 기간

```
<시작>/<기간>
```

```
2025-01-01T00:00:00Z/P1Y
# 2025년 1월 1일 UTC부터 1년간

2025-03-15T09:00:00+09:00/PT2H30M
# 2025년 3월 15일 오전 9시(KST)부터 2시간 30분간
```

#### 방법 3: 기간 / 종료 시점

```
<기간>/<종료>
```

```
P1M/2025-04-01
# 2025년 4월 1일로부터 1개월 전부터 4월 1일까지

PT8H/2025-01-31T18:00:00+09:00
# 2025년 1월 31일 오후 6시(KST)로부터 8시간 전부터 해당 시점까지
```

#### 방법 4: 기간만

```
<기간>
```

```
P1Y2M
# 1년 2개월 (시작/종료 시점 없이 길이만 표현)
```

## 9. 반복(Recurring) 표현

반복 간격은 `R`을 접두사로 사용하며, 반복 횟수와 시간 간격을 결합한다.

### 형식

```
Rn/<interval>
```

| 요소 | 설명 |
|------|------|
| R | 반복을 나타내는 접두사 |
| n | 반복 횟수 (생략하면 무한 반복) |

### 예시

```
R3/2025-01-01T00:00:00Z/P1M
# 2025년 1월 1일 UTC부터 1개월 간격으로 3번 반복
# -> 1월 1일, 2월 1일, 3월 1일

R5/2025-01-01/P1W
# 2025년 1월 1일부터 1주 간격으로 5번 반복
# -> 1/1, 1/8, 1/15, 1/22, 1/29

R/2025-01-01T09:00:00+09:00/PT1H
# 2025년 1월 1일 오전 9시(KST)부터 1시간 간격으로 무한 반복

R12/P1M/2025-12-31
# 2025년 12월 31일까지 1개월 간격으로 12번 반복 (역산)
```

### 활용 예시

| 표현 | 의미 |
|------|------|
| `R365/2025-01-01/P1D` | 매일 1회, 365일간 |
| `R52/2025-01-06/P1W` | 매주 1회, 52주간 |
| `R/PT30M` | 30분마다 무한 반복 |
| `R24/PT1H` | 매시간, 24번 반복 |

## 10. 주차(Week date) 표현

ISO 8601은 연도와 주 번호를 기반으로 날짜를 표현하는 주차 시스템을 정의한다.

### 형식

```
YYYY-Www-D        # 확장 형식
YYYYWwwD          # 기본 형식
```

| 요소 | 설명 | 범위 |
|------|------|------|
| YYYY | ISO 연도 | (그레고리력 연도와 다를 수 있음) |
| W | 주차 접두사 리터럴 | 고정 |
| ww | 주 번호 (2자리) | 01 ~ 52 (또는 53) |
| D | 요일 번호 | 1(월요일) ~ 7(일요일) |

### ISO 주차의 규칙

- 월요일이 주의 첫날이다 (일요일이 아님).
- 제1주는 해당 연도의 첫 번째 목요일이 포함된 주이다.
  - 다시 말해, 1월 4일이 포함된 주가 항상 제1주이다.
- 1년은 52주 또는 53주로 구성된다.

### 요일 번호

| 번호 | 요일 |
|------|------|
| 1 | 월요일 |
| 2 | 화요일 |
| 3 | 수요일 |
| 4 | 목요일 |
| 5 | 금요일 |
| 6 | 토요일 |
| 7 | 일요일 |

### 예시

```
2025-W05-5        # 2025년 제5주 금요일 = 2025-01-31
2025-W01-1        # 2025년 제1주 월요일 = 2024-12-30 (주의!)
2024-W01-1        # 2024년 제1주 월요일 = 2024-01-01
```

### ISO 연도와 그레고리력 연도의 차이

ISO 주차에서의 연도는 그레고리력의 연도와 다를 수 있다. 예를 들어:

- 2025년 1월 1일 (수요일): ISO 표기는 `2025-W01-3` (2025년 제1주 수요일)
- 2024년 12월 30일 (월요일): ISO 표기는 `2025-W01-1` (2025년 제1주 월요일)
  - 그레고리력으로는 2024년이지만, ISO 주차 기준으로는 2025년에 속한다.

이런 연도 경계 차이는 주차 기반 보고서나 회계 시스템에서 특히 주의해야 한다.

## 11. 서수일(Ordinal date) 표현

서수일은 연도와 해당 연도에서의 일수(day of year)로 날짜를 표현한다.

### 형식

```
YYYY-DDD          # 확장 형식
YYYYDDD           # 기본 형식
```

| 요소 | 설명 | 범위 |
|------|------|------|
| YYYY | 연도 (4자리) | 0000 ~ 9999 |
| DDD | 연중 일수 (3자리) | 001 ~ 365 (윤년은 366) |

### 예시

```
2025-001          # 2025년 1월 1일 (연중 제1일)
2025-031          # 2025년 1월 31일 (연중 제31일)
2025-059          # 2025년 2월 28일 (연중 제59일)
2025-060          # 2025년 3월 1일 (평년인 경우)
2025-365          # 2025년 12월 31일 (평년 마지막 날)
2024-366          # 2024년 12월 31일 (윤년 마지막 날)
```

### 월별 누적 일수 참조표 (평년 기준)

| 월 | 시작 서수일 | 종료 서수일 |
|----|------------|------------|
| 1월 | 001 | 031 |
| 2월 | 032 | 059 |
| 3월 | 060 | 090 |
| 4월 | 091 | 120 |
| 5월 | 121 | 151 |
| 6월 | 152 | 181 |
| 7월 | 182 | 212 |
| 8월 | 213 | 243 |
| 9월 | 244 | 273 |
| 10월 | 274 | 304 |
| 11월 | 305 | 334 |
| 12월 | 335 | 365 |

### 활용

서수일은 주로 다음 상황에서 유용하다.

- 항공우주 분야 (임무 경과일 계산)
- 기상 데이터 처리
- 두 날짜 사이의 일수 차이 빠른 계산
- 파일 명명 규칙 (짧은 형식)

## 13. 장점과 단점

### 장점

#### 1. 모호성 제거
- 전 세계 어디서든 동일하게 해석되는 단일 표준
- `01/02/03` 같은 혼란이 발생하지 않음

#### 2. 정렬 가능성 (Sortability)
- 문자열 그대로 사전식(lexicographic) 정렬하면 시간순 정렬이 됨
- 데이터베이스 인덱싱, 파일명 정렬에 유리

```
2025-01-15T08:00:00Z
2025-01-15T14:30:00Z    # 문자열 정렬 = 시간순 정렬
2025-02-01T09:00:00Z
```

#### 3. 기계 파싱 용이
- 고정된 형식으로 프로그래밍적 파싱이 간단
- 정규 표현식으로 유효성 검증 가능

#### 4. 국제 호환성
- 문화적 편향 없는 표현 (월/일 순서 충돌 없음)
- 다국적 팀, API 연동에서 즉시 활용 가능

#### 5. 포괄적 표현력
- 단순 날짜/시간부터 기간, 간격, 반복까지 표현 가능
- 타임존을 명시적으로 포함하여 시간대 오류 방지

#### 6. 널리 채택된 표준
- 대부분의 프로그래밍 언어가 기본 지원
- HTTP, JSON, XML, SQL 등에서 사실상 표준

### 단점

#### 1. 인간 가독성
- `2025년 1월 31일` 같은 자연어 표현보다 직관성이 떨어짐
- 일반 사용자에게는 UI에서 로컬 형식으로 변환 필요

#### 2. 복잡한 고급 기능
- Duration, Interval, Recurring 표현은 학습 곡선이 있음
- 라이브러리 지원이 불균일 (특히 Interval, Recurring)

#### 3. 타임존의 한계
- UTC 오프셋만 표현하며, IANA 타임존 ID를 포함하지 않음
- DST(일광절약시간) 전환 시 오프셋이 변경되므로, 미래 시점의 로컬 시간을 정확히 표현하기 어려움
- 예: `+09:00`이 KST인지 WIT(서인도네시아 시간)인지 구분 불가

#### 4. 모호한 Duration
- `P1M`이 28일인지, 30일인지, 31일인지 문맥에 따라 다름
- 달력 기반 계산이 필요하여 단순 초 변환이 불가능

#### 5. 표준 문서 유료
- ISO 8601 공식 문서는 유료 구매 필요
- 전체 사양을 무료로 확인하기 어려움 (RFC 3339은 무료)

#### 6. 버전 간 차이
- ISO 8601:2004와 ISO 8601-1:2019 사이에 미묘한 차이 존재
- 구현체마다 참조하는 버전이 달라 호환성 문제 발생 가능

## 14. 실제 사용 사례

### API 설계

ISO 8601은 RESTful API에서 날짜/시간 데이터를 교환하는 사실상의 표준이다.

```json
// JSON 응답 예시
{
  "id": "order-12345",
  "created_at": "2025-01-31T14:30:00+09:00",
  "updated_at": "2025-01-31T15:45:30.123Z",
  "delivery_date": "2025-02-05",
  "estimated_duration": "P3D",
  "subscription_period": "2025-01-01/2025-12-31"
}
```

```
// HTTP 헤더
Date: Fri, 31 Jan 2025 05:30:00 GMT     # HTTP는 RFC 7231 형식
Last-Modified: Fri, 31 Jan 2025 05:30:00 GMT

// 쿼리 파라미터
GET /api/events?start=2025-01-01T00:00:00Z&end=2025-01-31T23:59:59Z
GET /api/logs?since=2025-01-31T00:00:00Z&duration=PT24H
```

### 로그 시스템

### 데이터베이스

```sql

SELECT NOW() - INTERVAL 'PT24H';             -- 24시간 전
```

| 데이터베이스 | ISO 8601 지원 |
|------------|--------------|
| PostgreSQL | TIMESTAMP, INTERVAL 타입에서 ISO 8601 형식 완전 지원 |
| MySQL | DATETIME/TIMESTAMP에서 부분 지원, `T` 구분자 인식 |
| MongoDB | ISODate() 타입, ISO 8601 문자열 파싱 지원 |
| SQLite | TEXT 타입으로 저장, datetime() 함수에서 ISO 형식 파싱 |

```
# 백업 파일명
backup-2025-01-31T143000Z.sql.gz

# 로그 파일 로테이션
app-2025-01-31.log

# 사진/미디어 파일
IMG_2025-01-31T143000.jpg

```

ISO 8601 형식의 파일명은 파일 시스템에서 이름순 정렬 시 자동으로 시간순 정렬된다.

### 국제 표준 및 프로토콜

| 표준/프로토콜 | 사용 방식 |
|-------------|----------|
| JSON Schema | `date`, `date-time`, `duration` 포맷으로 ISO 8601 채택 |
| XML Schema | `xs:date`, `xs:dateTime`, `xs:duration` 타입이 ISO 8601 기반 |
| iCalendar (RFC 5545) | 일정/이벤트의 날짜/시간 표현에 ISO 8601 형식 사용 |
| OpenAPI (Swagger) | `format: date-time`이 ISO 8601 (RFC 3339) 형식 |
| GraphQL | DateTime 스칼라 타입에 ISO 8601 형식 권장 |
| TOML | 날짜/시간 리터럴이 ISO 8601 형식 |
| Git | 커밋 타임스탬프에 ISO 8601 오프셋 형식 사용 |
| JWT (RFC 7519) | `exp`, `iat`, `nbf` 클레임에 Unix timestamp 사용 (ISO 8601 아님, 참고) |
