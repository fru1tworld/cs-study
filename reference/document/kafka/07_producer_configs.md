## 개요

이 섹션에서는 Kafka 프로듀서의 구성 설정을 다룹니다. 프로듀서는 Kafka 토픽에 레코드를 게시하는 클라이언트입니다. 각 설정은 중요도(importance)에 따라 높음(high), 중간(medium), 낮음(low)으로 분류됩니다.

### bootstrap.servers

### buffer.memory

프로듀서가 서버로 전송을 기다리는 레코드를 버퍼링하는 데 사용할 수 있는 총 메모리 바이트 수입니다. 레코드가 서버로 전달될 수 있는 것보다 빠르게 전송되면 프로듀서는 `max.block.ms` 동안 블로킹한 후 예외를 발생시킵니다.

이 설정은 프로듀서가 사용하는 총 메모리와 대략적으로 일치해야 하지만, 프로듀서가 사용하는 모든 메모리가 버퍼링에 사용되는 것은 아니므로 하드 바운드는 아닙니다. 일부 추가 메모리는 압축(압축이 활성화된 경우) 및 진행 중인 요청을 유지하는 데 사용됩니다.

| 유효한 값 | [0,...] |

## 중간 중요도 설정

### batch.size

레코드를 함께 그룹화하기 위한 기본 배치 크기(바이트)를 제어합니다. 프로듀서는 동일한 파티션으로 전송되는 레코드를 함께 배치하여 요청 수를 줄입니다. 이렇게 하면 클라이언트와 서버 모두의 성능이 향상됩니다. 이 구성은 기본 배치 크기(바이트)를 제어합니다.

이 크기보다 큰 레코드 배치를 만들려는 시도는 없습니다. 브로커로 전송되는 요청에는 여러 배치가 포함되며 전송할 데이터가 있는 각 파티션당 하나씩입니다.

작은 배치 크기는 배칭을 덜 일반적으로 만들고 처리량을 줄일 수 있습니다(배치 크기가 0이면 배칭이 완전히 비활성화됩니다). 매우 큰 배치 크기는 추가 레코드를 예상하여 항상 지정된 배치 크기의 버퍼를 할당하므로 메모리를 다소 낭비적으로 사용할 수 있습니다.

| 속성 | 값 |
|------|-----|
| 타입 | int |
| 기본값 | 16384 (16KB) |
| 유효한 값 | [0,...] |
| 중요도 | medium |

---

### client.dns.lookup

클라이언트가 DNS 조회를 사용하는 방법을 제어합니다. `use_all_dns_ips`로 설정하면 연결이 설정될 때까지 각 반환된 IP 주소에 순서대로 연결합니다. 연결이 끊어지면 다음 IP가 사용됩니다. 모든 IP가 한 번 사용되면 클라이언트는 호스트 이름에서 IP를 다시 확인합니다(그러나 JVM 및 OS는 모두 DNS 이름 조회를 캐시합니다). `resolve_canonical_bootstrap_servers_only`로 설정하면 각 부트스트랩 주소를 정식 이름 목록으로 확인합니다. 부트스트랩 단계 후 이는 `use_all_dns_ips`와 동일하게 동작합니다.

| 속성 | 값 |
|------|-----|
| 타입 | string |
| 기본값 | use_all_dns_ips |
| 유효한 값 | [use_all_dns_ips, resolve_canonical_bootstrap_servers_only] |
| 중요도 | medium |

### delivery.timeout.ms

`send()` 호출이 반환된 후 성공 또는 실패를 보고하는 시간의 상한입니다. 이는 레코드가 전송되기 전에 지연되는 총 시간, 브로커에서 승인(ack)을 기다리는 시간(예상되는 경우), 재시도 가능한 전송 실패에 허용되는 시간을 제한합니다. 프로듀서는 복구할 수 없는 오류가 발생하거나, 재시도가 소진되거나, 레코드가 이전 전달 만료 기한에 도달한 배치에 추가되면 이 구성보다 일찍 레코드 전송 실패를 보고할 수 있습니다. 이 구성의 값은 `request.timeout.ms`와 `linger.ms`의 합보다 크거나 같아야 합니다.

| 속성 | 값 |
|------|-----|
| 타입 | int |
| 기본값 | 120000 (2분) |
| 유효한 값 | [0,...] |
| 중요도 | medium |

---

### linger.ms

프로듀서는 요청 전송 사이에 도착하는 모든 레코드를 단일 배치 요청으로 그룹화합니다. 일반적으로 이것은 레코드가 전송될 수 있는 것보다 빠르게 도착하는 부하 상태에서만 발생합니다. 그러나 일부 상황에서는 클라이언트가 적당한 부하에서도 요청 수를 줄이고 싶을 수 있습니다. 이 설정은 작은 인위적인 지연을 추가하여 이를 수행합니다. 즉, 레코드를 즉시 보내는 대신 프로듀서는 주어진 지연까지 기다려 레코드를 함께 배치할 수 있도록 합니다. 이는 TCP의 Nagle 알고리즘과 유사하게 생각할 수 있습니다. 이 설정은 배칭 지연의 상한을 제공합니다: 파티션에 대해 `batch.size` 가치의 레코드를 받으면 이 설정에 관계없이 즉시 전송되지만, 이 파티션에 대해 누적된 바이트 수가 이보다 적으면 지정된 시간 동안 더 많은 레코드가 도착하기를 기다립니다. 이 설정의 기본값은 5로, 레코드가 부하 없이 도착할 때 5ms의 약간의 지연을 제공합니다. 그러나 `linger.ms=0`으로 설정하면 레코드는 즉시 전송되며 이 경우 레코드가 도착하는 동안 이전 전송이 진행 중인 경우에만 배칭이 발생합니다.

| 속성 | 값 |
|------|-----|
| 타입 | long |
| 기본값 | 5 |
| 유효한 값 | [0,...] |
| 중요도 | medium |

## 낮은 중요도 설정

### acks

프로듀서가 요청이 완료된 것으로 간주하기 전에 리더가 받아야 하는 승인(acknowledgment) 수입니다. 이는 전송된 레코드의 내구성을 제어합니다. 다음 설정이 허용됩니다:

- `acks=0`: 0으로 설정하면 프로듀서는 서버로부터 어떤 승인도 기다리지 않습니다. 레코드가 즉시 소켓 버퍼에 추가되고 전송된 것으로 간주됩니다. 이 경우 서버가 레코드를 받았다는 보장이 없으며, `retries` 구성은 적용되지 않습니다(클라이언트는 일반적으로 실패를 알지 못하기 때문입니다). 각 레코드에 대해 반환되는 오프셋은 항상 -1로 설정됩니다.

- `acks=1`: 리더는 레코드를 로컬 로그에 쓰지만 모든 팔로워의 완전한 승인을 기다리지 않고 응답합니다. 이 경우 리더가 레코드를 승인한 직후에 실패하지만 팔로워가 복제하기 전에 실패하면 레코드가 손실됩니다.

- `acks=all`: 리더는 모든 동기화된 레플리카가 레코드를 승인할 때까지 기다립니다. 이는 최소 하나의 동기화된 레플리카가 살아 있는 한 레코드가 손실되지 않음을 보장합니다. 이것은 사용 가능한 가장 강력한 보장입니다. 이는 `acks=-1` 설정과 동일합니다.

| 속성 | 값 |
|------|-----|
| 타입 | string |
| 기본값 | all |
| 유효한 값 | [all, -1, 0, 1] |
| 중요도 | low |

### transaction.timeout.ms

### transactional.id

트랜잭션 전달에 사용할 TransactionalId입니다. 이를 통해 클라이언트가 새 트랜잭션을 시작하기 전에 동일한 TransactionalId를 사용하는 트랜잭션이 완료되도록 보장하므로 여러 프로듀서 세션에 걸친 신뢰성 의미론이 가능합니다. TransactionalId가 제공되지 않으면 프로듀서는 멱등 전달로 제한됩니다. 참고: 트랜잭션이 활성화된 경우 `enable.idempotence`가 내포됩니다. 기본적으로 TransactionalId는 구성되지 않으며, 이는 트랜잭션을 사용할 수 없음을 의미합니다. 기본적으로 트랜잭션에는 최소 세 개의 브로커로 구성된 클러스터가 필요하며, 이는 프로덕션에 권장되는 설정입니다. 개발을 위해 `transaction.state.log.replication.factor` 설정을 조정하여 변경할 수 있습니다.

| 속성 | 값 |
|------|-----|
| 타입 | string |
| 기본값 | null |
| 유효한 값 | 비어 있지 않은 문자열 |
| 중요도 | low |
