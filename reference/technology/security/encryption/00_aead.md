# AEAD (Authenticated Encryption with Associated Data)

## 개요

AEAD(Authenticated Encryption with Associated Data)는 데이터의 **기밀성(confidentiality)**, **무결성(integrity)**, **인증(authenticity)** 을 동시에 보장하는 암호화 방식이다. 전통적으로 암호화와 메시지 인증은 별도의 알고리즘으로 수행되었지만, AEAD는 이 두 기능을 하나의 알고리즘으로 통합한다.

## 왜 AEAD가 필요한가

### 암호화만으로는 부족하다

암호화(encryption)는 데이터의 기밀성만 보장한다. 공격자가 암호문을 변조하더라도, 복호화 측에서 이를 감지할 수 없다. 이 문제는 다음과 같은 실제 공격으로 이어진다:

- **비트 플리핑 공격(bit-flipping attack)**: CTR 모드나 스트림 암호에서 암호문의 특정 비트를 뒤집으면, 복호화된 평문의 해당 비트도 뒤집힌다.
- **패딩 오라클 공격(padding oracle attack)**: CBC 모드에서 패딩 검증 오류 메시지를 이용해 평문을 복원할 수 있다.
- **재정렬 공격(reordering attack)**: 암호문 블록의 순서를 바꾸어 의미를 변조할 수 있다.

### MAC만으로도 부족하다

메시지 인증 코드(MAC)는 무결성과 인증을 보장하지만 기밀성은 제공하지 않는다. 암호화와 MAC을 조합할 때도 조합 순서에 따라 보안성이 달라진다:

| 조합 방식 | 설명 | 보안성 |
|-----------|------|--------|
| **Encrypt-and-MAC** | 평문을 암호화하고, 평문에 대해 MAC을 계산 | MAC이 평문 정보를 노출할 수 있어 안전하지 않음 |
| **MAC-then-Encrypt** | 평문에 대해 MAC을 계산하고, 평문+MAC을 암호화 | 패딩 오라클 공격에 취약할 수 있음 (TLS 1.0의 CBC 모드) |
| **Encrypt-then-MAC** | 평문을 암호화하고, 암호문에 대해 MAC을 계산 | 이론적으로 안전하지만, 구현 실수의 여지가 있음 |

AEAD는 이러한 조합의 복잡성과 실수 가능성을 제거하고, 하나의 통합된 인터페이스를 제공한다.

## AEAD의 동작 원리

### 입력

AEAD 알고리즘은 다음 네 가지 입력을 받는다:

1. **키(Key)**: 대칭 암호화에 사용되는 비밀 키
2. **논스(Nonce)**: 같은 키로 여러 메시지를 암호화할 때 각 메시지를 구별하기 위한 고유 값. 초기화 벡터(IV)라고도 부른다.
3. **평문(Plaintext)**: 암호화할 데이터
4. **연관 데이터(Associated Data, AD)**: 암호화하지는 않지만 인증은 필요한 데이터 (예: 헤더, 메타데이터, 프로토콜 버전)

### 출력

- **암호문(Ciphertext)**: 암호화된 평문
- **인증 태그(Authentication Tag)**: 암호문과 연관 데이터 모두의 무결성을 검증하는 값

### 복호화

복호화 시 인증 태그를 먼저 검증한다. 태그가 일치하지 않으면 복호화를 수행하지 않고 즉시 오류를 반환한다. 이는 변조된 데이터가 애플리케이션 계층에 도달하는 것을 방지한다.

```
암호화: (K, N, P, AD) → (C, T)
복호화: (K, N, C, T, AD) → P 또는 오류
```

## 연관 데이터(Associated Data)의 역할

연관 데이터는 AEAD의 핵심 구성 요소다. 네트워크 프로토콜에서는 패킷 헤더처럼 암호화할 수 없지만 변조로부터 보호해야 하는 데이터가 있다:

- **TLS 레코드**: 레코드 타입, 프로토콜 버전, 길이 필드가 연관 데이터로 인증된다.
- **IPsec ESP**: IP 헤더의 일부가 연관 데이터로 처리된다.
- **데이터베이스 암호화**: 행 ID나 컬럼 이름을 연관 데이터로 바인딩하여, 암호문이 다른 위치에 삽입되는 것을 방지한다.

## 논스(Nonce)의 중요성

AEAD의 보안은 **동일한 키에서 논스를 절대 재사용하지 않는 것**에 결정적으로 의존한다. 논스 재사용은 대부분의 AEAD 구성에서 치명적인 보안 결함으로 이어진다:

- **AES-GCM**: 논스 재사용 시 인증 키가 노출되어, 인증 태그 위조와 평문 복원이 가능해진다.
- **ChaCha20-Poly1305**: 논스 재사용 시 키스트림이 동일해져, 두 평문의 XOR이 노출된다.

논스 관리 전략:
- **카운터 기반**: 가장 단순하고 안전하지만, 상태를 유지해야 한다.
- **랜덤 생성**: 상태가 필요 없지만, 논스 공간이 충분히 커야 생일 역설에 의한 충돌을 방지할 수 있다.
- **확장 논스**: XChaCha20-Poly1305처럼 192비트 논스를 사용하여 랜덤 생성의 안전성을 높인다.

## 주요 AEAD 알고리즘

| 알고리즘 | 블록 크기 | 키 크기 | 논스 크기 | 태그 크기 | 특징 |
|----------|----------|---------|----------|----------|------|
| AES-GCM | 128비트 | 128/192/256비트 | 96비트 | 128비트 | 하드웨어 가속 시 최고 성능, 가장 널리 사용 |
| ChaCha20-Poly1305 | 512비트 | 256비트 | 96비트 | 128비트 | 소프트웨어 구현 시 높은 성능, 타이밍 공격에 강함 |
| AES-CCM | 128비트 | 128/192/256비트 | 56~104비트 | 32~128비트 | 제약된 환경(IoT) 용도, 직렬 처리만 가능 |
| AES-SIV | 128비트 | 256/384/512비트 | 가변 | 128비트 | 논스 오용에 안전(nonce-misuse resistant) |
| AES-GCM-SIV | 128비트 | 128/256비트 | 96비트 | 128비트 | 논스 오용에 안전하면서 GCM에 가까운 성능 |
| XChaCha20-Poly1305 | 512비트 | 256비트 | 192비트 | 128비트 | 확장 논스로 랜덤 생성 안전 |

## 표준화

AEAD는 다음 표준과 프로토콜에서 핵심 역할을 한다:

- **NIST SP 800-38D**: AES-GCM 규격
- **RFC 8439**: ChaCha20-Poly1305 규격
- **RFC 5116**: AEAD 알고리즘의 일반적인 인터페이스 정의
- **TLS 1.3 (RFC 8446)**: AEAD만 허용 (CBC 등 비-AEAD 모드 제거)
- **QUIC (RFC 9001)**: AEAD 기반 패킷 보호
- **WireGuard**: ChaCha20-Poly1305 사용
- **Signal Protocol**: AES-256-CBC + HMAC-SHA256 (Encrypt-then-MAC) 또는 AEAD

## AEAD와 양자 내성

현재의 AEAD 알고리즘(AES-GCM, ChaCha20-Poly1305)은 대칭 암호화 기반이므로, 양자 컴퓨터에 대해서도 상대적으로 안전하다. Grover 알고리즘으로 인한 보안 강도 감소(키 크기의 절반)를 감안하면:

- AES-128-GCM: 양자 환경에서 64비트 보안 (불충분)
- AES-256-GCM: 양자 환경에서 128비트 보안 (충분)
- ChaCha20-Poly1305: 양자 환경에서 128비트 보안 (충분)

따라서 양자 내성을 고려할 때는 256비트 키를 사용하는 AEAD 알고리즘을 선택해야 한다.
