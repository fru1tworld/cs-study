# 갈루아/카운터 모드 (Galois/Counter Mode, GCM)

> 원문: https://en.wikipedia.org/wiki/Galois/Counter_Mode

## 개요

**갈루아/카운터 모드(GCM)** 는 대칭 키 암호화 블록 암호의 운용 모드로, 성능을 위해 널리 채택된 인증 암호화(authenticated encryption) 방식이다. GCM은 현대 고속 통신 채널에서 효율적으로 높은 처리량을 달성할 수 있도록 설계되었으며, 특허가 없다.

GCM은 잘 알려진 **카운터 모드(CTR)** 의 암호화 방식과 새로운 **갈루아 모드(Galois mode)** 의 인증 방식을 결합하여 **AEAD(Authenticated Encryption with Associated Data)** 를 제공한다. 인증 전용 변형인 **GMAC(Galois Message Authentication Code)** 도 존재하며, 이는 증분 메시지 인증 코드(incremental MAC)로 사용할 수 있다.

GCM과 GMAC은 모두 128비트 블록 크기의 블록 암호를 받아들인다. **AES(Advanced Encryption Standard)** 가 가장 일반적인 블록 암호이며, AES-GCM이라 불린다.

## 암호화와 인증

GCM의 핵심 개념은 다음과 같다:

### 카운터 모드 암호화

GCM은 카운터 모드(CTR)를 사용하여 암호화를 수행한다:

1. 카운터 값을 블록 암호로 암호화하여 키스트림 블록을 생성한다.
2. 키스트림 블록과 평문 블록을 XOR하여 암호문 블록을 생성한다.
3. 각 블록마다 카운터를 증가시킨다.

카운터 모드의 장점은 각 블록이 독립적으로 처리될 수 있어 **병렬화가 가능**하다는 점이다. CBC 모드와 달리, 이전 블록의 결과가 다음 블록의 입력에 필요하지 않다.

### 갈루아 모드 인증

인증은 유한체 GF(2^128)에서의 산술 연산을 기반으로 한다:

1. 인증 키 H는 블록 암호로 영 블록(all-zero block)을 암호화하여 도출한다: H = E(K, 0^128)
2. 암호문 블록과 연관 데이터(AAD) 블록을 GF(2^128) 상의 다항식으로 취급한다.
3. 이 다항식을 인증 키 H에서 평가하여(호너 방법 사용) 인증 태그를 계산한다.

## GCM 연산의 상세 동작

### 입력

- **키(K)**: 대칭 암호화 키
- **초기화 벡터(IV)**: 고유한 값 (96비트 권장)
- **평문(P)**: 암호화할 데이터
- **연관 인증 데이터(AAD)**: 암호화하지 않지만 인증할 추가 데이터

### 처리 과정

1. **인증 키 생성**: H = E(K, 0^128)
2. **초기 카운터 블록 생성**:
   - IV가 96비트인 경우: J₀ = IV || 0³¹ || 1
   - IV가 96비트가 아닌 경우: J₀ = GHASH(H, {}, IV)
3. **암호화**: 카운터 값을 증가시키면서 각 평문 블록을 암호화
   - C_i = P_i ⊕ E(K, inc₃₂(J₀ + i))
4. **인증 태그 계산**:
   - GHASH 함수를 사용하여 AAD와 암호문에 대한 해시를 계산
   - T = GHASH(H, A, C) ⊕ E(K, J₀)

### GHASH 함수

GHASH는 GCM의 인증 구성 요소로, 다음과 같이 정의된다:

```
GHASH(H, A, C) = X_m+n+1

여기서:
X₀ = 0^128
X_i = (X_{i-1} ⊕ A_i) · H     (i = 1, ..., m, AAD 블록)
X_i = (X_{i-1} ⊕ C_{i-m}) · H  (i = m+1, ..., m+n, 암호문 블록)
X_{m+n+1} = (X_{m+n} ⊕ (len(A) || len(C))) · H
```

여기서 곱셈(·)은 GF(2^128) 상에서 수행되며, 기약 다항식 x^128 + x^7 + x^2 + x + 1을 사용한다.

### 출력

- **암호문(C)**: 암호화된 데이터
- **인증 태그(T)**: 인증 값 (최대 128비트, 일반적으로 128, 120, 112, 104, 96비트 중 선택)

## 수학적 기반

### 유한체 GF(2^128)

GCM의 인증은 이진 갈루아 체 GF(2^128) 상의 산술에 기반한다. 이 체의 원소는 GF(2) 계수를 가진 127차 이하의 다항식으로 표현된다.

**덧셈**: 계수별 XOR 연산 (캐리 없음)

**곱셈**: 다항식 곱셈 후 기약 다항식 p(x) = x^128 + x^7 + x^2 + x + 1로 나눈 나머지

이 기약 다항식은 GF(2) 상의 128차 기약 다항식 중 비영(non-zero) 계수의 개수가 가장 적은 오항식(pentanomial)이다. 이 선택은 효율적인 하드웨어 및 소프트웨어 구현을 가능하게 한다.

### 캐리 없는 곱셈 (Carry-less Multiplication)

최신 프로세서는 GF(2^128) 곱셈을 가속하기 위한 전용 명령어를 제공한다:
- Intel/AMD: **PCLMULQDQ** 명령어 (CLMUL 명령어 세트)
- ARM: **PMULL/PMULL2** 명령어 (ARMv8 Cryptography Extensions)

이러한 하드웨어 지원은 GCM의 성능을 크게 향상시킨다.

## GCM 및 GMAC 규격

### NIST 표준화

- **NIST SP 800-38D** (2007년 11월): GCM의 공식 NIST 권고안으로, AES 블록 암호와 함께 사용하는 GCM을 명시한다.
- GCM에 대한 원래 제안서는 2004년에 David A. McGrew와 John Viega가 제출했다.

### IV 요구사항

NIST는 GCM에서 사용되는 IV에 대해 다음을 규정한다:
- 특정 키에 대해 각 IV는 **고유해야** 한다.
- **96비트(12바이트)** IV가 권장된다. 96비트 이외의 길이도 허용되지만, 추가적인 GHASH 연산이 필요하다.
- 결정론적 방법(카운터)과 랜덤 방법 모두 허용되지만, 랜덤 방법은 96비트 IV에서만 사용해야 한다.

### 인증 태그 길이

- 권장 태그 길이: 128, 120, 112, 104, 96비트
- 허용되지만 권장하지 않는 태그 길이: 64, 32비트
- 짧은 태그는 위조 공격에 대한 보안을 약화시킨다.

## 사용 및 채택

GCM은 다음 프로토콜과 표준에서 광범위하게 사용된다:

### 네트워크 프로토콜

| 프로토콜 | 표준 | 용도 |
|----------|------|------|
| TLS 1.2 | RFC 5288 | HTTPS 등 전송 계층 보안 |
| TLS 1.3 | RFC 8446 | TLS의 최신 버전, AEAD만 지원 |
| IPsec ESP | RFC 4106 | VPN 등 네트워크 계층 보안 |
| IPsec AH | RFC 4543 | GMAC을 이용한 인증 전용 |
| SSH | RFC 5647 | 원격 접속 보안 |
| WPA3-Enterprise | IEEE 802.11ac | 무선 LAN 보안 |

### 스토리지 암호화

| 표준 | 조직 | 용도 |
|------|------|------|
| IEEE 1619.1-2018 | IEEE | 스토리지 데이터 인증 암호화 |
| ANSI/Fibre Channel Security | INCITS | 파이버 채널 보안 |

### 정부 및 군용 표준

- **NSA Suite B Cryptography**: AES-GCM이 기밀(SECRET) 및 일급기밀(TOP SECRET) 등급의 국가 보안 시스템 보호에 승인됨
- **CNSS Policy 15**: AES-GCM을 NSS(National Security Systems)의 기밀 정보 보호에 사용 승인

### AES-256-GCM의 주요 채택 사례

- Google: HTTPS 및 내부 통신에 기본 암호화 방식으로 사용
- Cloudflare: CDN 서비스의 TLS 기본 암호화 방식
- AWS: KMS 및 다양한 서비스의 봉투 암호화(envelope encryption)에 사용

## 성능

### 하드웨어 가속

현대 프로세서에서 GCM은 매우 높은 성능을 낸다:

- **AES-NI** (AES New Instructions): AES 블록 암호 연산을 하드웨어로 가속
- **CLMUL**: GF(2^128) 곱셈(GHASH)을 하드웨어로 가속
- 두 명령어 세트를 조합하면, 소프트웨어 전용 구현 대비 **수십 배**의 성능 향상이 가능하다.

### 병렬화

GCM의 카운터 모드 특성상, 여러 블록의 암호화를 동시에 수행할 수 있다:
- **파이프라인 처리**: AES-NI의 파이프라인 특성을 활용하여 여러 블록을 동시에 처리
- **SIMD 활용**: 벡터 명령어를 사용한 병렬 GHASH 계산

### 처리량 비교 (Intel 최신 프로세서 기준)

하드웨어 가속이 있는 경우 AES-256-GCM은 사이클당 약 1바이트(cycle per byte) 수준의 처리량을 달성하며, 이는 10Gbps 이상의 라인 속도(line rate) 암호화를 가능하게 한다.

## 특허 및 설계

GCM은 명시적으로 **특허가 없도록 설계**되었다. 특허로 보호되는 다른 인증 암호화 모드(예: OCB)와 비교했을 때 GCM의 주요 이점이다.

GCM의 설계자인 David A. McGrew(Cisco Systems)와 John Viega는 2004년에 이 모드를 제안했으며, 특허 무료라는 설계 목표를 명시했다.

## 보안

### 강점

- **증명 가능한 보안(provable security)**: GCM은 사용된 블록 암호가 의사랜덤 순열(pseudorandom permutation)을 제공한다는 가정 하에 보안 증명이 존재한다.
- **증분 MAC**: GMAC은 증분 계산을 지원하여, 메시지의 일부가 변경되었을 때 전체를 재계산하지 않고 태그를 업데이트할 수 있다.

### IV/논스 재사용 문제

GCM의 보안에서 가장 심각한 위협은 **IV 재사용**이다:

- 동일한 키에서 IV를 재사용하면, 인증 키 H가 노출된다.
- H가 노출되면 공격자는 인증 태그를 위조할 수 있다.
- 키스트림이 동일해지므로 평문의 XOR도 노출된다.
- 이 공격은 **실용적(practical)** 이며, Joux가 2006년에 시연했다.

### 짧은 태그의 위험

- 태그 길이 t비트에서, 위조 확률의 상한은 2^(-t)이다.
- 그러나 GCM에서는 짧은 태그가 추가적인 취약성을 만든다: 태그 길이가 128비트 미만이면 GCM의 보안 증명에서 일부 보장이 약화된다.
- **NIST는 64비트 이하의 태그 사용을 권장하지 않는다.**

### 데이터 크기 제한

단일 키-IV 쌍으로 암호화할 수 있는 데이터의 최대 크기:
- **평문**: 최대 2^39 - 256비트 (약 64 GiB)
- **AAD**: 최대 2^64비트

이 한계를 초과하면 보안 증명이 성립하지 않는다.

### 키 노출 공격 (Key Commitment)

GCM은 키 커밋먼트(key commitment) 속성을 제공하지 않는다. 이는 공격자가 동일한 암호문과 태그에 대해 서로 다른 키를 사용하여 서로 다른 평문으로 복호화되도록 만들 수 있음을 의미한다. 이 속성의 부재는 일부 프로토콜에서 문제가 될 수 있다.

### Ferguson의 비판

Niels Ferguson은 2005년에 GCM의 보안 증명에 대해 비판을 제기했다:
- 보안 증명의 바운드가 실용적으로 의미 있는 보안 보장을 제공하기에 너무 약하다.
- 특히 짧은 태그 길이에서 보안 바운드가 현저히 약화된다.
- 2^32개 이상의 호출이 있을 때 보안 마진이 충분하지 않을 수 있다.

## 구현

### 소프트웨어 라이브러리

주요 암호화 라이브러리에서 GCM을 지원한다:

- **OpenSSL**: AES-GCM의 표준 구현, AES-NI 및 CLMUL 가속 지원
- **BoringSSL**: Google의 OpenSSL 포크, TLS에서 AES-GCM 우선 사용
- **libsodium**: `crypto_aead_aes256gcm_*` API 제공
- **Bouncy Castle**: Java/C# 구현
- **wolfSSL**: 임베디드 시스템용 구현
- **GnuTLS**: AES-GCM 지원
- **NSS**: Mozilla의 암호화 라이브러리

### 하드웨어 지원

- **Intel AES-NI + CLMUL**: Westmere 아키텍처(2010) 이후 지원
- **ARM Cryptography Extensions**: ARMv8-A 이후 지원
- **POWER8**: AES 및 GHASH 가속 명령어
- **RISC-V Crypto Extensions**: Zkn(AES) 및 Zkg(GHASH) 확장

### 최적화 기법

고성능 GCM 구현을 위한 주요 최적화 기법:

1. **Karatsuba 곱셈**: GF(2^128) 곱셈의 부분곱 수를 줄이는 기법
2. **4비트/8비트 테이블**: GHASH 연산을 사전 계산된 테이블로 가속하는 기법
3. **집계 감소(aggregated reduction)**: 여러 블록의 감소 연산을 한 번에 수행하는 기법
4. **파이프라인 인터리빙**: AES와 GHASH 연산을 인터리빙하여 파이프라인 지연을 숨기는 기법
