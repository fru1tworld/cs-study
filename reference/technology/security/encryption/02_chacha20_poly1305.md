# ChaCha20-Poly1305

> 원문: https://en.wikipedia.org/wiki/ChaCha20-Poly1305

## 개요

**ChaCha20-Poly1305**는 **ChaCha20** 스트림 암호와 **Poly1305** 메시지 인증 코드를 결합한 AEAD(Authenticated Encryption with Associated Data) 알고리즘이다. 하드웨어 가속이 없는 시스템에서 소프트웨어만으로 구현할 때 일반적으로 AES-GCM보다 빠른 성능을 보이며, 타이밍 공격에 대한 내성이 더 강하다.

## 역사

### 구성 요소의 탄생

ChaCha20-Poly1305를 구성하는 두 요소는 Daniel J. Bernstein이 각각 독립적으로 설계했다:

- **Poly1305** (2005): 일회용 키 기반의 메시지 인증 코드. 원래 AES와 함께 Poly1305-AES로 제안되었다.
- **ChaCha20** (2008): Salsa20 스트림 암호의 변형으로, 라운드당 확산(diffusion)이 향상되었다. ChaCha는 Salsa20 eSTREAM 프로젝트의 수상작을 기반으로 한다.

### IETF 표준화 과정

2013년 3월, IETF TLS 워킹 그룹에 RC4 기반 암호 스위트를 대체하기 위해 Salsa20을 사용하는 제안서가 제출되었다.

이후 논의를 거쳐 Adam Langley가 제안한 변형이 채택되었다:
- 32비트 블록 카운터와 96비트 논스를 사용하는 ChaCha20 변형
- Poly1305의 수정 버전

이 조합은 TLS 및 DTLS용 IETF 인터넷 초안으로 작성되었다.

### OpenSSH의 채택

OpenSSH는 `chacha20-poly1305@openssh.com` 암호를 통해 AEAD 모드를 통합했다. 이 구현은 원래의 64비트 카운터와 논스를 유지한다(IETF 변형의 32비트 카운터/96비트 논스와는 다름).

### 표준화 연혁

| 연도 | RFC/이벤트 | 내용 |
|------|-----------|------|
| 2015 | RFC 7539 | ChaCha20-Poly1305 AEAD 알고리즘 표준화 |
| 2015 | RFC 7634 | IPsec에서의 ChaCha20-Poly1305 사용 |
| 2015 | Cloudflare 채택 | 대체 암호 스위트로 통합 |
| 2016 | RFC 7905 | TLS 1.2 및 DTLS 1.2에서의 구현 기술 |
| 2018 | RFC 8439 | RFC 7539를 대체하는 최신 규격 |

## 상세 설명

### 알고리즘 구조

ChaCha20-Poly1305는 **256비트 키**와 **96비트 논스**를 입력으로 받으며, 암호문에 **128비트(16바이트)** 의 인증 태그가 추가된다.

#### 암호화 과정

1. **Poly1305 키 생성**: ChaCha20의 첫 번째 블록(카운터 = 0)을 사용하여 Poly1305용 일회용 키(256비트)를 생성한다.
2. **키스트림 생성**: ChaCha20의 카운터를 1부터 시작하여 키스트림 블록을 생성한다.
3. **암호화**: 키스트림과 평문을 XOR하여 암호문을 생성한다.
4. **인증**: 연관 데이터(AAD)와 암호문을 Poly1305로 인증하여 128비트 태그를 생성한다.

#### ChaCha20 내부 상태

ChaCha20의 내부 상태는 4x4 행렬의 32비트 워드(총 512비트 = 64바이트)로 구성된다:

```
상태 = [상수, 상수, 상수, 상수,
        키,   키,   키,   키,
        키,   키,   키,   키,
        카운터, 논스, 논스, 논스]
```

- **상수**: "expand 32-byte k"의 ASCII 인코딩 (4워드)
- **키**: 256비트 키 (8워드)
- **카운터**: 32비트 블록 카운터 (1워드)
- **논스**: 96비트 논스 (3워드)

#### 쿼터 라운드 연산

ChaCha20은 **쿼터 라운드(quarter round)** 연산을 핵심 구성 요소로 사용한다. 각 쿼터 라운드는 4개의 32비트 워드(a, b, c, d)에 다음 연산을 적용한다:

```
a += b; d ^= a; d <<<= 16;
c += d; b ^= c; b <<<= 12;
a += b; d ^= a; d <<<= 8;
c += d; b ^= c; b <<<= 7;
```

20라운드(10 더블 라운드)를 수행하며, 각 더블 라운드는:
- **컬럼 라운드**: 4개의 열에 쿼터 라운드 적용
- **대각선 라운드**: 4개의 대각선에 쿼터 라운드 적용

#### Poly1305 인증

Poly1305는 다음과 같이 동작한다:

1. 입력 메시지를 16바이트 블록으로 분할한다.
2. 각 블록을 소수 p = 2^130 - 5에 대한 모듈러 산술로 처리한다.
3. 인증 키 r과 일회용 키 s를 사용하여:
   - tag = ((c₁·r^n + c₂·r^(n-1) + ... + cₙ·r) mod p) + s mod 2^128

여기서 c_i는 메시지 블록을 인코딩한 정수이다.

#### ChaCha20의 블록 크기 이점

ChaCha20의 블록 크기는 **64바이트(512비트)** 로, AES의 **16바이트(128비트)** 보다 4배 크다. 이는 다음과 같은 이점을 제공한다:
- 최신 CPU에서 더 나은 성능 (더 적은 블록 암호 호출)
- 32비트 카운터로도 256 GiB까지 암호화 가능 (AES-CTR의 64 GiB 대비)

#### Poly1305 인증 대상 구성

인증 태그 계산에 사용되는 데이터의 구성:

```
연관 데이터(AAD) || 패딩 || 암호문 || 패딩 || AAD 길이(8바이트) || 암호문 길이(8바이트)
```

패딩은 각 섹션을 16바이트 경계에 맞추기 위해 영(zero) 바이트를 추가한다.

## 변형

### XChaCha20-Poly1305 - 확장 논스 변형

**XChaCha20-Poly1305**는 ChaCha20 대신 **XChaCha20**을 사용하여 논스를 **192비트(24바이트)** 로 확장한다.

#### 동작 원리

XChaCha20은 두 단계로 동작한다:
1. **HChaCha20**: 256비트 키와 192비트 논스의 처음 128비트를 사용하여 256비트 서브키를 생성
2. **ChaCha20**: 서브키와 192비트 논스의 마지막 64비트(+ 32비트 0 패딩)를 사용하여 키스트림 생성

#### 이점

- **랜덤 논스 안전성**: 192비트 논스 공간은 충분히 커서, 논스를 랜덤으로 생성해도 충돌 확률이 무시할 수 있을 정도로 낮다 (생일 역설 기준 약 2^96개의 메시지까지 안전).
- **상태 비유지**: 카운터를 관리할 필요 없이 랜덤 논스를 사용할 수 있다.

#### 표준화 상태

XChaCha20-Poly1305에 대한 인터넷 초안이 작성되었으나, 2020년 7월에 만료되었다. 표준화는 이루어지지 않았지만, libsodium 등 주요 라이브러리에서 지원한다.

### Salsa20-Poly1305 및 XSalsa20-Poly1305

이 변형들은 ChaCha 변형 대신 **Salsa20**과 **XSalsa20**을 사용한다:

- **NaCl**(Networking and Cryptography library)과 **libsodium**에 구현되어 있다.
- 표준화되지 않았다.
- ChaCha 변형이 Salsa20 변형보다 라운드당 확산(diffusion)이 더 우수하여, 실질적으로 ChaCha 변형이 선호된다.

### 축소 라운드 변형

ChaCha20(20라운드) 대신 라운드 수를 줄인 변형:

| 변형 | 라운드 수 | 보안성 | 성능 |
|------|----------|--------|------|
| ChaCha20-Poly1305 | 20 | 전체 보안 마진 | 기준 |
| ChaCha12-Poly1305 | 12 | 충분한 보안 마진 | ~1.7배 빠름 |
| ChaCha8-Poly1305 | 8 | 최소 보안 마진 | ~2.5배 빠름 |

XChaCha20-Poly1305에도 동일한 축소 라운드 변형(XChaCha12, XChaCha8)이 적용될 수 있다. RustCrypto 팀이 이러한 비표준 변형들을 구현하고 있다.

## 성능

### 소프트웨어 성능 특성

ChaCha20-Poly1305는 **AES-NI 하드웨어 가속이 없는 시스템**에서 일반적으로 AES-GCM보다 성능이 우수하다. 이는 다음과 같은 이유 때문이다:

1. **상수 시간 구현 용이**: ChaCha20은 ARX(Addition-Rotation-XOR) 연산만 사용하여, 테이블 조회가 필요 없다. 이는 캐시 타이밍 부채널에 본질적으로 안전하다.
2. **SIMD 친화적**: 쿼터 라운드 연산은 SIMD(단일 명령 다중 데이터) 명령어에 직접 매핑된다.
3. **큰 블록 크기**: 64바이트 블록은 AES의 16바이트 블록보다 호출 오버헤드가 적다.

### ARM 기기에서의 이점

하드웨어 AES 가속이 없는 ARM 기반 CPU를 사용하는 모바일 기기에서 ChaCha20-Poly1305는 특히 중요하다:
- AES-GCM의 소프트웨어 구현보다 상당히 빠르다.
- **전력 소비가 적다**: 더 적은 연산 오버헤드로 인해 배터리 소모가 줄어든다.
- 구형 Android 기기에서 HTTPS 성능을 크게 개선했다 (Google Chrome이 이 이유로 ChaCha20-Poly1305를 우선 선택).

### AES-NI 지원 시스템에서의 비교

AES-NI가 있는 시스템에서는 AES-GCM이 일반적으로 더 빠르다. 그러나 ChaCha20의 AVX2/AVX-512 최적화 구현은 AES-GCM과 경쟁력 있는 성능을 달성할 수 있다.

## 보안

### 보안 증명

ChaCha20-Poly1305 구성은 **표준 모델(standard model)** 과 **이상적 순열 모델(ideal permutation model)** 에서 단일 사용자 및 다중 사용자 환경 모두에 대해 일반적 보안이 증명되어 있다.

### 논스 고유성 요구

GCM과 마찬가지로, ChaCha20-Poly1305의 보안은 **각 암호화 메시지에 고유한 논스를 사용하는 것**에 의존한다. 동일한 키-논스 쌍을 재사용하면:
- 동일한 키스트림이 생성되어 두 평문의 XOR이 노출된다.
- Poly1305 키가 동일해져 인증 태그 위조가 가능해질 수 있다.

### 타이밍 공격 내성

ChaCha20-Poly1305 구현은 AES-GCM 구현에 비해 **타이밍 공격에 덜 취약**하다:
- ChaCha20은 테이블 조회를 사용하지 않으므로, 캐시 타이밍 부채널이 존재하지 않는다.
- Poly1305도 데이터 종속적인 분기나 메모리 접근 패턴이 없다.
- AES의 소프트웨어 구현은 S-box 테이블 조회로 인해 캐시 타이밍 공격에 취약할 수 있다 (하드웨어 AES-NI를 사용하면 이 문제가 해결됨).

### Terrapin 공격

SSH에서 ChaCha20-Poly1305를 사용할 때, **Terrapin 공격**에 취약해진다. 이 공격은 2023년에 공개되었으며:
- SSH 핸드셰이크 중 메시지 시퀀스 번호를 조작하여 초기 메시지를 삭제할 수 있다.
- SSH의 메시지 순서 번호가 논스로 사용되는 특성을 악용한다.
- 이 취약점은 ChaCha20-Poly1305 자체의 결함이 아니라, SSH 프로토콜이 이를 사용하는 방식의 문제이다.
- 대응책으로 "strict key exchange" 확장이 도입되었다.

### 데이터 크기 제한

- 32비트 카운터(IETF 변형): 단일 키-논스 쌍으로 최대 256 GiB 암호화 가능
- 이 한계를 초과해야 하는 경우 키 재생성(rekeying)이 필요하다.
