# Gradle 튜토리얼 5편: 증분 빌드 살펴보기

> **원문:** https://docs.gradle.org/current/userguide/part5_gradle_inc_builds.html

## 개요

이번 편의 목표는 거창한 새 기능을 배우는 게 아니라, Gradle이 "왜 두 번째 빌드부터 빨라지는지"를 눈으로 확인하는 것이다. 핵심은 세 가지다.

- `gradle.properties`로 빌드 동작을 설정하는 방법
- 증분 빌드(incremental build)의 동작 원리
- 태스크 실행 결과를 나타내는 라벨 읽는 법

## 증분 빌드란

증분 빌드는 "직전 빌드 이후 입력이 바뀌지 않은 태스크는 다시 실행할 필요가 없으니 건너뛴다"는 최적화다. 별도로 켜는 옵션이 아니라 Gradle의 기본 동작이며, 대신 전제 조건이 있다.

- 태스크가 자신의 입력(input)과 출력(output)을 명확히 선언하고 있어야 한다.
- Gradle은 매 실행마다 입력/출력 상태를 기록해두고, 다음 실행 때 이전 기록과 비교한다.
- 비교 결과가 같으면 실행을 생략하고, 다르면 실제로 태스크를 수행한다.

즉 "코드를 안 건드렸으면 컴파일도, 테스트도 다시 할 필요 없다"는 상식적인 판단을 Gradle이 태스크 단위로 자동화해주는 것이다.

## 실행 과정 자세히 보기: `--console=verbose`

기본 콘솔 출력은 태스크 실행 여부를 요약해서 보여주지 않는다. 어떤 태스크가 왜 건너뛰어졌는지 확인하려면 verbose 모드가 필요하다. 매번 옵션을 붙이는 대신 프로젝트 루트의 `gradle.properties`에 아래처럼 고정해두면 편하다.

```properties
org.gradle.console=verbose
```

이렇게 설정한 뒤 빌드를 돌리면, 각 태스크 옆에 실행 결과 라벨이 함께 출력된다.

## 실습으로 확인하기

**1) 클린 빌드 (처음 실행)**

```bash
$ ./gradlew :app:clean :app:build
```

캐시나 이전 산출물이 전혀 없는 상태이므로, 모든 태스크(예: 8개)가 라벨 없이 실제로 실행된다.

**2) 같은 명령을 한 번 더 실행**

```bash
$ ./gradlew :app:clean :app:build
```

`clean`이 들어있긴 하지만, 소스 코드나 설정에는 변화가 없으므로 대부분의 태스크가 `UP-TO-DATE` 라벨을 달고 실행을 건너뛴다. 결과적으로 빌드 소요 시간이 수 초에서 수십~수백 밀리초 수준으로 확 줄어든다. 이 차이가 바로 증분 빌드의 효과다.

## 태스크 실행 결과 라벨

| 라벨 | 의미 |
| --- | --- |
| (라벨 없음) | 입력이 바뀌어 태스크가 실제로 새로 실행됨 |
| `UP-TO-DATE` | 직전 실행과 입력·출력이 동일해서 실행을 건너뜀 (증분 빌드) |
| `SKIPPED` | 조건이나 옵션에 의해 의도적으로 실행이 제외됨 |
| `FROM-CACHE` | 로컬 재실행이 아니라 빌드 캐시에서 결과물을 그대로 가져옴 |
| `NO-SOURCE` | 처리할 입력 자체가 없어서 실행되지 않음 |

**핵심 포인트**
- `UP-TO-DATE`는 "같은 워크스페이스에서 직전에 이미 해봤다"는 뜻이고, `FROM-CACHE`는 "다른 브랜치·다른 실행 주체가 만든 결과물을 재사용했다"는 뜻으로 서로 다른 메커니즘의 산물이다.
- 라벨이 없는 태스크만 실제로 시간이 걸리는 작업이며, 나머지 라벨들은 모두 "이번엔 일을 안 했다"는 신호다.
- verbose 콘솔은 디버깅용 정보 제공 수단일 뿐, 빌드 결과 자체를 바꾸지는 않는다.
