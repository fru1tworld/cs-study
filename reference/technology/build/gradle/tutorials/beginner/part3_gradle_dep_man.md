# Gradle 튜토리얼 Part 3: 의존성 이해하기

> **원문:** https://docs.gradle.org/current/userguide/part3_gradle_dep_man.html

## 개요

Part 1(프로젝트 초기화)과 Part 2(태스크 실행)를 마쳤다면, 이제 프로젝트가 외부 라이브러리를 어떻게 가져오고 관리하는지 볼 차례다. 이 파트에서는 버전 카탈로그, 의존성 좌표, 전이 의존성, 의존성 트리 조회, Build Scan을 이용한 시각화, 버전 업데이트까지 한 흐름으로 다룬다.

## 버전 카탈로그부터 살펴보기

Gradle 프로젝트를 새로 만들면 `gradle/libs.versions.toml` 파일이 함께 생성된다. 여기가 의존성 버전을 관리하는 출발점이다.

- 라이브러리 버전을 빌드 스크립트에 하드코딩하지 않고 이 파일 한 곳에 모아 둔다.
- 여러 서브프로젝트가 같은 버전 정의를 공유하므로 버전이 제각각 흩어지는 문제를 막는다.
- IntelliJ 같은 IDE는 카탈로그 항목을 인식해 `libs.xxx` 형태의 타입-세이프 자동완성을 제공한다.

```toml
[versions]
guava = "33.3.1-jre"

[libraries]
guava = { module = "com.google.guava:guava", version.ref = "guava" }
```

빌드 스크립트에서는 문자열 좌표 대신 `libs` 접근자로 참조한다.

```kotlin
dependencies {
    implementation(libs.guava)
}
```

### 핵심 포인트: 의존성의 네 갈래

`dependencies {}` 블록에 선언되는 것들은 성격이 다르다.

- **외부 의존성** — 리포지토리(대표적으로 Maven Central)에서 내려받는 라이브러리.
- **프로젝트 의존성** — 같은 빌드 안의 다른 서브프로젝트를 참조.
- **로컬 파일 의존성** — 저장소를 거치지 않고 직접 지정하는 JAR 파일.
- **컨피규레이션별 의존성** — `implementation`처럼 컴파일·런타임에 필요한 것과 `testImplementation`처럼 테스트 코드에만 필요한 것을 구분.

## 좌표(GAV)로 라이브러리를 식별하기

리포지토리에 있는 라이브러리 하나를 정확히 가리키려면 세 가지 정보가 필요하다. 이를 줄여서 **GAV**라 부른다.

| 구분 | 의미 | 예시 |
|---|---|---|
| Group | 배포 주체(조직) | `com.google.guava` |
| Artifact | 라이브러리 이름 | `guava` |
| Version | 배포 버전 | `33.3.1-jre` |

리포지토리는 이 좌표를 실제 아티팩트(jar 등)로 바꿔 주는 저장소 역할을 한다. Java 생태계에서는 Maven Central이 사실상 표준 리포지토리로 쓰인다.

## 전이 의존성: 딸려 오는 라이브러리들

내가 선언한 라이브러리가 또 다른 라이브러리를 필요로 한다면, Gradle은 그것까지 자동으로 끌어온다. 이렇게 간접적으로 포함되는 의존성을 **전이 의존성(transitive dependency)**이라 한다.

- 예를 들어 Guava를 선언하면, Guava가 내부적으로 요구하는 `failureaccess` 같은 라이브러리는 따로 선언하지 않아도 함께 내려받힌다.
- 개발자가 신경 쓸 것은 직접 필요한 라이브러리뿐이고, 나머지 의존 관계 해석은 Gradle이 처리한다.
- 다만 프로젝트 규모가 커지면 이 전이 의존성들이 서로 다른 버전을 요구하면서 충돌이 생길 수 있어, 실제로 무엇이 들어와 있는지 눈으로 확인하는 절차가 필요해진다.

## 의존성 트리로 실제 구성 확인하기

`dependencies` 태스크를 실행하면 컴파일/런타임 클래스패스별로 어떤 라이브러리가 어떤 경로로 들어왔는지 트리 형태로 볼 수 있다.

```bash
./gradlew :app:dependencies
```

- 직접 선언한 의존성과 그로부터 파생된 전이 의존성이 계층 구조(트리)로 함께 출력된다.
- 컨피규레이션(예: `compileClasspath`, `runtimeClasspath`)별로 결과가 나뉘어, "이 라이브러리가 왜 여기 들어와 있는지"를 추적할 수 있다.

## Build Scan으로 한눈에 시각화하기

터미널 출력만으로 구조를 파악하기 어렵다면 Build Scan을 활용할 수 있다.

```bash
./gradlew build --scan
```

- 최초 실행 시 Gradle 이용약관 동의와 이메일 인증 절차를 거쳐야 상세 리포트에 접근할 수 있다.
- 결과는 웹 페이지 형태로 제공되며, 의존성 그래프를 컨피규레이션별로 펼쳐 보거나 특정 라이브러리가 어느 경로를 통해 들어왔는지 시각적으로 추적할 수 있다.
- 팀원과 링크를 공유해 같은 화면을 보며 의존성 문제를 논의하기에도 유용하다.

## 버전을 바꾸고 다시 확인하기

의존성 버전을 올리거나 내릴 때는 빌드 스크립트가 아니라 버전 카탈로그(`libs.versions.toml`)의 버전 값을 수정한다.

- 카탈로그 한 곳만 고치면 그 버전을 참조하는 모든 서브프로젝트/모듈에 동일하게 반영된다.
- IntelliJ 같은 IDE에는 변경 후 프로젝트 인덱스를 다시 동기화하는 버튼이 있어, 자동완성 정보가 최신 버전 기준으로 갱신된다.
- 버전을 바꾼 뒤 다시 `dependencies` 태스크나 Build Scan을 돌려 보면, 트리 구조가 새 버전을 기준으로 어떻게 달라졌는지 비교할 수 있다.

마지막으로 `./gradlew run`을 실행해 애플리케이션이 정상 동작하면, 의존성 선언·해석·컴파일까지 문제없이 끝났다는 뜻이다.

## 정리

- 의존성 버전은 빌드 스크립트가 아니라 버전 카탈로그(`libs.versions.toml`) 한 곳에서 관리한다.
- 라이브러리는 Group·Artifact·Version(GAV) 세 가지로 식별되며, 리포지토리(예: Maven Central)가 이를 실제 아티팩트로 해석해 준다.
- 직접 선언하지 않아도 함께 들어오는 라이브러리가 전이 의존성이며, 버전 충돌의 주요 원인이 되기도 한다.
- `./gradlew :app:dependencies`는 텍스트 트리로, `./gradlew build --scan`은 웹 기반 시각화로 실제 의존성 구성을 확인하는 두 가지 방법이다.
- 버전을 바꿀 때는 카탈로그만 수정하면 되고, 변경 후에는 트리나 Build Scan으로 결과를 다시 검증하는 습관이 필요하다.
