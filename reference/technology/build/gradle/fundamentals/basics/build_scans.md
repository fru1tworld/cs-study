# Build Scan 기본 개념

> **원문:** https://docs.gradle.org/current/userguide/build_scans.html

## 개요
Build Scan은 Gradle 빌드를 실행하는 동안 수집되는 메타데이터를 시각화한 결과물입니다. 빌드가 끝나면 Gradle이 실행 정보를 Build Scan Service로 전송하고, 이 서비스가 원시 메타데이터를 사람이 분석하기 쉬운 웹 리포트 형태로 가공해 줍니다.

## 왜 필요한가
- **트러블슈팅**: 빌드 실패나 성능 저하의 원인을 파악할 때, 수집된 데이터가 근거 자료가 됩니다.
- **협업/질문**: 커뮤니티나 동료에게 도움을 요청할 때 에러 메시지와 환경 정보를 일일이 복사해서 붙여넣을 필요 없이, Build Scan 링크 하나만 공유하면 됩니다.
- **성능 분석**: 태스크별 소요 시간, 캐시 적중 여부 등을 확인해 빌드 성능을 개선하는 데 활용할 수 있습니다.

## 핵심 포인트
- Build Scan은 "빌드가 실행되면서 어떤 일이 있었는지"를 기록한 스냅샷이며, 실행할 때마다 새로 생성됩니다.
- 기본적으로 리포트는 공개 서비스인 `scans.gradle.com`에 업로드되며, 링크를 아는 사람은 누구나 접근할 수 있습니다.
- 조직 자체 Develocity 서버로 업로드 대상을 바꿀 수도 있습니다.

## 활성화 방법

### 커맨드라인에서 즉시 생성
Gradle 명령 뒤에 `--scan` 플래그만 붙이면 됩니다.

```
./gradlew build --scan
```

- 처음 사용할 때는 Build Scan 이용 약관에 동의하라는 프롬프트가 표시될 수 있습니다.
- 빌드가 끝나면 콘솔에 리포트로 이동할 수 있는 URL이 출력됩니다.

### 자체 서버(Develocity)로 전송
공개 서비스가 아니라 조직에서 운영하는 Develocity 서버로 리포트를 보내고 싶다면, `--develocity-url` 옵션으로 대상 서버를 지정합니다.

```
./gradlew build --develocity-url=https://develocity.example.com
```

- 지속적으로 특정 서버를 사용하려면 매번 옵션을 주는 대신 Develocity Gradle 플러그인을 통해 프로젝트 설정에 고정해 두는 방식이 권장됩니다.

## 어떤 정보가 담기는가
- Build Scan에는 실행 환경(OS, JVM 버전), 태스크 실행 결과, 소요 시간, 의존성 해석 과정, 콘솔 출력 등 빌드 관련 메타데이터가 포함됩니다.
- 구체적으로 어떤 항목이 수집되는지, 그리고 데이터가 어떻게 보호되는지에 대한 세부 내용은 Gradle Develocity 플러그인 문서에서 별도로 다룹니다.
- 민감한 정보가 포함될 수 있으므로, 공개 서비스에 업로드하기 전에는 어떤 데이터가 전송되는지 확인하는 것이 좋습니다.

## 정리
- 빌드에 문제가 생기거나 성능을 점검하고 싶을 때는 `--scan` 플래그를 붙여 실행하는 것만으로 진단용 리포트를 얻을 수 있습니다.
- 리포트는 기본적으로 공개 서비스(`scans.gradle.com`)에 저장되지만, `--develocity-url`로 자체 서버를 지정할 수도 있습니다.
- Build Scan 링크는 텍스트로 상황을 설명하는 것보다 훨씬 정확하고 빠르게 문제를 공유하는 방법이므로, 이슈 리포트나 팀 내 질문에 적극 활용하면 좋습니다.
