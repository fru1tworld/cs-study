# Gradle 캐싱 기초 (증분 빌드와 빌드 캐시)

> **원문:** https://docs.gradle.org/current/userguide/gradle_optimizations.html

## 개요

Gradle은 매번 모든 태스크를 처음부터 다시 실행하지 않는다. 이전 실행 결과를 기억해두고, 바뀐 부분만 다시 계산하는 방식으로 빌드 속도를 크게 끌어올린다. 이를 가능하게 하는 두 가지 핵심 메커니즘이 **증분 빌드(incremental build)**와 **빌드 캐시(build cache)**다.

- 증분 빌드: 로컬에서 "이전 실행과 입력이 같으면 건너뛴다"는 원리
- 빌드 캐시: "누군가(나 자신 또는 팀원, CI)가 이미 만들어둔 결과물을 재사용한다"는 원리

두 기능 모두 태스크의 **입력(input)과 출력(output)**을 비교해서 동작 여부를 판단하기 때문에, 태스크가 입력/출력을 제대로 선언하고 있어야 효과를 볼 수 있다.

## 태스크 실행 결과 레이블

`--console=verbose` 옵션으로 빌드를 실행하면 각 태스크가 어떤 이유로 실행되었거나 건너뛰어졌는지 라벨로 확인할 수 있다.

| 라벨 | 의미 |
| --- | --- |
| (라벨 없음, 실행됨) | 입력이 바뀌어 태스크가 실제로 수행됨 |
| `UP-TO-DATE` | 이전 실행과 입력·출력이 동일해서 실행을 건너뜀 |
| `FROM-CACHE` | 로컬 또는 원격 빌드 캐시에서 출력을 그대로 복원함 |
| `NO-SOURCE` | 처리할 소스 파일 자체가 없어서 건너뜀 (예: 컴파일할 `.java` 파일 없음) |
| `SKIPPED` | `onlyIf` 조건이나 커맨드라인 옵션 등으로 인해 실행되지 않음 |
| `FAILED` | 태스크 실행 중 오류가 발생함 |

**핵심 포인트**
- `UP-TO-DATE`와 `FROM-CACHE`는 둘 다 "태스크를 다시 실행하지 않았다"는 뜻이지만, 전자는 증분 빌드, 후자는 빌드 캐시에 의한 결과라는 점이 다르다.
- 한 번 `FROM-CACHE`로 복원된 태스크는, 다음 빌드부터는 입력이 그대로라면 `UP-TO-DATE`로 표시된다. 즉 캐시 복원 이후에는 증분 빌드 판단이 다시 그 자리를 대체한다.

## 증분 빌드 (Incremental Build)

증분 빌드는 **직전 빌드와 비교해 입력이 바뀌지 않은 태스크의 실행을 생략**하는 기능이다. Gradle에서는 별도 설정 없이 기본적으로 항상 켜져 있다.

동작 원리는 단순하다.

- 태스크마다 선언된 입력(소스 파일, 설정 값 등)과 출력(생성 파일 등)의 상태(주로 해시 값)를 기록해둔다.
- 다음 빌드 때 현재 상태와 기록된 상태를 비교한다.
- 값이 같으면 실행을 건너뛰고 `UP-TO-DATE`로 표시하고, 다르면 태스크를 실제로 실행한다.

실행 과정을 자세히 보려면 verbose 콘솔 모드를 사용한다.

```bash
$ ./gradlew compileJava --console=verbose
> Task :app:compileJava UP-TO-DATE
BUILD SUCCESSFUL in 374ms
```

매번 옵션을 입력하기 번거롭다면 `gradle.properties`에 다음처럼 고정해둘 수 있다.

```properties
org.gradle.console=verbose
```

**핵심 포인트**
- 증분 빌드는 "로컬 이력"만 참고하는 최적화다. 다른 브랜치나 다른 머신에서 만든 결과는 알지 못한다.
- 효과를 보려면 태스크 작성 시 입력/출력을 정확히 선언하는 것이 전제 조건이다.

## 빌드 캐시 (Build Cache)

빌드 캐시는 한 단계 더 나아가, **이전에 실행했던 (같은 입력을 가진) 태스크의 결과물을 저장해두고, 필요할 때 재사용**하는 기능이다. 증분 빌드가 "같은 워크스페이스, 같은 이력"을 가정하는 반면, 빌드 캐시는 브랜치가 다르거나 실행 주체(팀원, CI 서버)가 달라도 입력이 같으면 결과를 그대로 가져다 쓸 수 있다는 점이 다르다.

전형적인 활용 시나리오는 다음과 같다.

- 같은 코드를 여러 브랜치에서 반복 빌드하는 경우
- 여러 팀원이 동일한 모듈을 각자 빌드하는 경우
- CI 파이프라인에서 동일한 커밋을 여러 번 빌드하는 경우

이런 상황에서는 이미 누군가 만들어 놓은 컴파일/테스트 결과를 그대로 재사용해 중복 작업을 없앤다.

빌드 캐시는 기본적으로 꺼져 있으며, `--build-cache` 옵션으로 활성화한다.

```bash
$ ./gradlew compileJava --build-cache
> Task :app:compileJava FROM-CACHE
BUILD SUCCESSFUL in 364ms
```

**핵심 포인트**
- 빌드 캐시가 적용된 태스크는 `FROM-CACHE`로 표시되며, 이는 "실행하지 않고 캐시에서 출력을 복원했다"는 의미다.
- 팀 전체나 CI 환경처럼 여러 실행 주체가 결과를 공유할수록 빌드 캐시의 효과가 커진다.

## 두 기능 비교

| 구분 | 증분 빌드 | 빌드 캐시 |
| --- | --- | --- |
| 기본 활성화 여부 | 항상 켜짐 | `--build-cache`로 켜야 함 |
| 비교 대상 | 같은 워크스페이스의 직전 실행 결과 | 로컬/원격 캐시에 저장된 임의의 과거 결과 |
| 결과 표시 | `UP-TO-DATE` | `FROM-CACHE` |
| 공유 범위 | 나 혼자, 이 프로젝트 디렉터리 | 팀원·CI 등 여러 실행 주체 간 공유 가능 |

결국 두 메커니즘은 서로 배타적이지 않다. 로컬에서는 증분 빌드로 빠르게 건너뛰고, 로컬 이력에 없는 경우에는 빌드 캐시로 다시 한번 재사용 기회를 찾는 식으로 함께 작동한다.
