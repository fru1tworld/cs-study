# Gradle 초급 튜토리얼 2부 - 태스크 실행하기

> **원문:** https://docs.gradle.org/current/userguide/part2_gradle_tasks.html

## 개요

이번 장에서는 Gradle 빌드의 기본 단위인 **태스크(Task)**를 다룬다. 프로젝트에 어떤 태스크가 있는지 확인하는 법, 태스크가 어디서 생겨나는지, 태스크 사이의 의존 관계가 어떻게 실행 순서를 결정하는지, 그리고 터미널과 IDE에서 태스크를 실행하는 방법을 순서대로 살펴본다. 1부에서 만든 Java 애플리케이션 프로젝트가 실습 대상이다.

## 사용 가능한 태스크 조회하기

`tasks` 태스크를 실행하면 현재 프로젝트에서 쓸 수 있는 태스크 목록을 볼 수 있다.

```bash
./gradlew tasks
```

- 출력은 **Application**, **Build**, **Documentation**, **Other** 등 카테고리별로 정리되어 나온다.
- 멀티 프로젝트 구성에서는 특정 서브프로젝트만 지정해 조회할 수 있다. 예: `./gradlew :app:tasks`
- 기본 출력에는 사용자에게 노출할 만한 태스크만 표시된다. 내부용/라이프사이클 태스크까지 모두 보려면 `--all` 옵션을 붙인다.

```bash
./gradlew tasks --all
```

## 태스크는 어디서 오는가

태스크는 세 가지 경로로 프로젝트에 추가된다.

- **Gradle 기본 제공** — 별도 설정 없이도 존재하는 태스크
- **적용된 플러그인** — 예를 들어 `application` 플러그인을 적용하면 `run`, `build`, `compileJava`, `test`, `jar` 등의 태스크가 자동으로 생긴다
- **사용자 정의** — 빌드 스크립트에서 직접 등록한 태스크

사용자 정의 태스크는 `tasks.register<타입>("이름") { ... }` 형태로 등록한다. 예를 들어 특정 디렉터리의 `.war` 파일을 다른 곳으로 복사하는 태스크는 다음처럼 작성할 수 있다.

```kotlin
tasks.register<Copy>("copyTask") {
    from("source")
    into("target")
    include("*.war")
}
```

Gradle이 기본으로 제공하는 대표적인 태스크 타입은 아래와 같다.

- `Copy` — 파일을 한 위치에서 다른 위치로 복사
- `Delete` — 파일/디렉터리 삭제
- `Exec` — OS 명령 실행
- `Zip` — 파일을 압축 아카이브로 묶음

## 태스크 간 의존성

대부분의 태스크는 독립적으로 실행되지 않고, 다른 태스크가 먼저 끝나야 의미가 있다. Gradle은 이 순서를 두 가지 방식으로 파악한다.

- **명시적 의존성** — `dependsOn()`으로 "이 태스크는 저 태스크 다음에 실행되어야 한다"고 직접 선언
- **암묵적 의존성** — 한 태스크의 출력이 다른 태스크의 입력으로 쓰이는 관계를 Gradle이 자동으로 감지해 순서를 정함

명시적 의존성 예시는 다음과 같다.

```kotlin
tasks.register("hello") {
    doLast { println("Hello!") }
}
tasks.register("greet") {
    doLast { println("How are you?") }
    dependsOn("hello")
}
```

`greet`를 실행하면 `dependsOn("hello")` 때문에 `hello`가 먼저 실행되어 `Hello! How are you?` 순서로 출력된다. 개발자가 실행 순서를 일일이 지정하지 않아도, Gradle이 의존성 그래프를 계산해 올바른 순서로 처리해 준다.

## IDE에서 태스크 확인·실행하기 (선택)

IntelliJ IDEA 같은 IDE를 쓴다면 화면 우측의 Gradle 패널에서 프로젝트의 태스크를 카테고리별로 시각적으로 확인할 수 있다. 목록에서 원하는 태스크를 더블클릭하면 바로 실행되고, 콘솔 창에서 진행 상황과 성공 여부를 볼 수 있다. 예를 들어 `build` 태스크를 더블클릭하면 컴파일부터 테스트, 패키징까지 이어지는 전체 과정이 순서대로 실행된다.

## 터미널에서 태스크 실행하기

Gradle Wrapper로 원하는 태스크를 직접 실행할 수 있다.

```bash
./gradlew build
./gradlew jar
./gradlew run
```

- `build`는 컴파일, 리소스 처리, 테스트, 검증 등 여러 하위 태스크를 순서대로 실행하는 라이프사이클 태스크다.
- `jar`는 실행 가능한 JAR 파일을 `app/build/libs/` 아래에 만든다.
- `run`은 애플리케이션을 실제로 구동하며, 이 예제에서는 `Hello World!` 출력을 확인할 수 있다.

이 과정에서 `compileJava → jar → build` 같은 순서로 태스크가 자동 실행되는 것을 로그에서 직접 확인할 수 있는데, 이는 앞서 설명한 태스크 의존성이 실제로 동작하는 모습이다.

## 핵심 포인트

- `./gradlew tasks`로 프로젝트에서 실행 가능한 태스크를 카테고리별로 확인할 수 있고, `--all` 옵션으로 내부 태스크까지 볼 수 있다.
- 태스크는 Gradle 기본 제공, 플러그인 적용, 사용자 정의라는 세 가지 경로로 생겨난다.
- 태스크 의존성은 `dependsOn()`을 통한 명시적 방식과, 입출력 관계를 통한 암묵적 방식으로 결정되며, Gradle이 이를 바탕으로 실행 순서를 자동 계산한다.
- 태스크는 IDE의 Gradle 패널에서 더블클릭으로도, 터미널에서 `./gradlew <태스크 이름>`으로도 실행할 수 있다.
- `build`, `jar`, `run` 같은 태스크를 실행하면 그에 필요한 선행 태스크(`compileJava` 등)가 먼저 자동으로 실행된다.
