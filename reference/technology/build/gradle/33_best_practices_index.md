# Gradle 모범 사례 체크리스트 (Best Practices Index)

> **원문:** https://docs.gradle.org/current/userguide/best_practices_index.html

## 개요

Gradle 공식 문서가 "일반, 빌드 구조화, 의존성, 태스크, 성능, 보안, 테스트" 7개 범주로 정리해 둔 모범 사례 목록입니다. 개별 규칙 하나하나가 어렵지는 않지만, 실제 프로젝트에서 놓치기 쉬운 항목들이라 체크리스트 형태로 정리해 둡니다.

## 일반 (General)

- **Kotlin DSL을 사용하라**: Groovy DSL보다 타입 안정성과 IDE 자동완성이 뛰어나므로 신규 빌드는 Kotlin DSL(`build.gradle.kts`)로 작성한다.
- **Gradle의 최신 마이너 버전을 유지하라**: 메이저 업그레이드가 아니어도 마이너 버전 업데이트만으로 버그 수정과 성능 개선을 받을 수 있다.
- **플러그인은 `plugins {}` 블록으로 적용하라**: `apply plugin:` 같은 레거시 방식 대신 `plugins {}` 블록을 사용해야 버전 관리와 클래스패스 해석이 안전해진다.

  ```kotlin
  plugins {
      id("java")
      id("org.springframework.boot") version "3.3.0"
  }
  ```

- **Gradle 내부 API를 사용하지 마라**: `internal` 패키지나 문서화되지 않은 API는 버전 간 예고 없이 바뀔 수 있다.
- **빌드 플래그는 `gradle.properties`에 설정하라**: JVM 힙, 병렬 실행 등 환경 설정은 스크립트가 아닌 `gradle.properties`에 두어 환경별 일관성을 유지한다.
- **루트 프로젝트에 이름을 명시하라**: `settings.gradle(.kts)`에서 `rootProject.name`을 지정해 멀티 프로젝트 빌드에서 프로젝트를 명확히 구분한다.
- **서브프로젝트에는 `gradle.properties`를 두지 마라**: 설정이 여러 곳에 흩어지면 우선순위 혼란이 생기므로 루트에만 둔다.
- **`afterEvaluate`를 남용하지 마라**: 평가 순서에 의존하는 코드는 디버깅이 어렵고 성능도 떨어지므로 최소화한다.

## 빌드 구조화 (Structuring Builds)

- **빌드를 모듈화하라**: 하나의 거대한 프로젝트 대신 기능 단위로 서브프로젝트를 나누면 병렬 빌드와 캐시 재사용성이 좋아진다.
- **루트 프로젝트에 소스 파일을 두지 마라**: 루트는 조율(coordination) 역할만 하고, 실제 소스 코드는 서브프로젝트에 둔다.
- **공용 빌드 로직은 `build-logic` 컴포지트 빌드로 분리하라**: `buildSrc`나 스크립트 복붙 대신 별도의 컴포지트 빌드로 관리하면 재사용과 테스트가 쉬워진다.
- **의도치 않은 빈 프로젝트 생성을 피하라**: `include`로 등록했지만 실제 코드가 없는 프로젝트는 빌드 구조를 불필요하게 복잡하게 만든다.
- **컨벤션 플러그인을 사용하라**: 여러 서브프로젝트에 반복되는 설정은 컨벤션 플러그인으로 뽑아내 한 곳에서 관리한다.

## 의존성 관리 (Dependencies)

- **단일 GAV 문자열을 사용하라**: `group`, `name`, `version`을 따로 쓰지 않고 `"group:artifact:version"` 한 줄로 표기해 가독성을 높인다.
- **버전 카탈로그로 의존성 버전을 중앙화하라**: `libs.versions.toml`에 버전을 모아두면 여러 모듈 간 버전 불일치를 막을 수 있다.

  ```toml
  [versions]
  junit = "5.10.2"

  [libraries]
  junit-jupiter = { module = "org.junit.jupiter:junit-jupiter", version.ref = "junit" }
  ```

- **버전 카탈로그 항목 이름은 명확하게 지어라**: 알파벳 순 정렬이 아니라 용도가 드러나는 이름을 붙여 검색성을 높인다.
- **의존성 저장소는 Settings 파일에 설정하라**: `dependencyResolutionManagement`를 `settings.gradle(.kts)`에 두어야 초기화 시점부터 저장소가 확정된다.
- **`kotlin-stdlib`를 명시적으로 의존성에 추가하지 마라**: Kotlin 플러그인이 자동으로 포함시켜 주므로 중복 선언이다.
- **중복 의존성 선언을 피하라**: 같은 라이브러리를 여러 곳에서 다른 버전으로 선언하면 카탈로그나 플랫폼으로 통합한다.
- **여러 저장소 사용 시 콘텐츠 필터링을 적용하라**: 어떤 저장소가 어떤 그룹의 의존성을 제공하는지 명시하면 불필요한 네트워크 조회를 줄인다.
- **`exclude`는 좁은 범위에만 적용하라**: 전역 제외 대신 특정 의존성에만 국한된 제외 규칙을 써서 의도치 않은 부작용을 막는다.

## 태스크 (Task)

- **`dependsOn` 사용을 최소화하라**: 태스크 간 순서만 강제하는 `dependsOn` 대신 입력/출력을 연결하면 캐시와 병렬화가 제대로 동작한다.
- **`cacheIf`/`doNotCacheIf` 대신 `@CacheableTask`, `@DisableCachingByDefault` 애너테이션을 써라**: 조건부 메서드보다 애너테이션이 캐싱 의도를 더 명확히 드러낸다.
- **커스텀 태스크에는 `group`과 `description`을 지정하라**: `gradle tasks` 출력에서 잘 분류되고 검색되도록 한다.
- **태스크 액션 밖에서 `Provider.get()`을 호출하지 마라**: 설정 단계에서 값을 즉시 꺼내면 지연 평가(lazy evaluation)의 이점이 사라진다.
- **태스크 실행 전에 `Configuration`을 resolve하지 마라**: 조기 resolve는 구성 캐시 효율과 빌드 유연성을 해친다.
- **`FileCollection`에 즉시 평가 API를 쓰지 마라**: 지연 API를 사용해야 증분 빌드와 캐싱이 정상 동작한다.
- **파일 입력은 `PathSensitivity.NONE`, 디렉터리는 `PathSensitivity.RELATIVE`를 우선하라**: 경로 민감도를 적절히 낮춰야 캐시 적중률이 올라간다.
- **출력 파일·디렉터리 경로는 고유하게 지정하라**: 여러 변형(variant) 빌드에서 경로가 겹치면 캐싱 충돌과 잘못된 결과를 유발한다.

## 성능 (Performance)

- **UTF-8 인코딩을 명시하라**: 플랫폼마다 기본 인코딩이 달라 컴파일 결과가 갈릴 수 있으므로 명시적으로 고정한다.
- **빌드 캐시를 사용하라**: 입력이 바뀌지 않은 태스크는 재실행 없이 캐시 결과를 재사용한다.
- **구성 캐시(Configuration Cache)를 사용하라**: 빌드 모델 계산 자체를 캐싱해 설정 단계 시간을 절약한다.
- **구성 단계에서 비용이 큰 연산을 피하라**: 무거운 계산은 태스크 액션으로 미뤄야 구성 캐시의 효과를 온전히 얻는다.
- **Gradle 배포판은 `-bin`을 우선하라**: 소스가 포함된 `-all` 배포판 대신 바이너리 전용 배포판을 써서 다운로드/시작 시간을 줄인다.

## 보안 (Security)

- **Gradle 배포판의 SHA-256 체크섬을 검증하라**: 다운로드한 배포판이 변조되지 않았는지 확인한다.
- **Wrapper 업그레이드마다 무결성을 검증하라**: `gradle-wrapper.properties`의 체크섬을 매 업그레이드 시 확인해 공급망 공격을 예방한다.

## 테스트 (Testing)

- **커스텀 태스크·플러그인은 TestKit으로 테스트하라**: `gradle-testkit`을 이용해 격리된 환경에서 반복 가능한 통합 테스트를 작성한다.

## 핵심 포인트: 왜 이런 규칙들이 존재하는가

- 대부분의 규칙은 결국 "지연 평가(laziness)를 지키고, 설정 단계와 실행 단계를 명확히 분리하라"는 하나의 원칙으로 수렴한다.
- 의존성·태스크 관련 규칙은 캐시 적중률과 병렬 빌드 효율을 높이는 방향으로 설계되어 있다.
- 보안 규칙(체크섬 검증)은 CI 환경에서 Gradle 배포판 자체가 공급망 공격의 표적이 될 수 있다는 점을 전제로 한다.
