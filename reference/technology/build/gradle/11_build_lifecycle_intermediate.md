# Gradle 빌드 생명주기 (Gradle Build Lifecycle)

> **원문:** https://docs.gradle.org/current/userguide/build_lifecycle_intermediate.html

## 개요

빌드 생명주기란 Gradle이 빌드 스크립트와 소스 코드를 실제 결과물로 바꾸기 위해 거치는 일련의 단계를 말합니다. 빌드 환경을 초기화하고, 프로젝트를 설정하고, 마지막으로 태스크를 실행하는 순서로 진행되며, 이 세 단계는 항상 고정된 순서로만 실행됩니다.

## 세 가지 빌드 단계

Gradle 빌드는 언제나 아래 세 단계를 순서대로 거칩니다.

### 1단계: 초기화(Initialization)

- `settings.gradle(.kts)` 설정 파일을 찾아 `Settings` 인스턴스를 생성합니다.
- 설정 파일을 평가해서 이 빌드에 어떤 프로젝트와 컴포짓 빌드(included build)가 포함되는지 결정합니다.
- 발견된 프로젝트마다 `Project` 인스턴스를 만듭니다.

### 2단계: 구성(Configuration)

- 빌드에 참여하는 모든 프로젝트의 빌드 스크립트를 평가합니다.
- 각 태스크의 입력·출력 등 설정 정보를 처리합니다.
- 요청된 태스크를 기준으로 태스크 그래프를 만듭니다.

### 3단계: 실행(Execution)

- 구성 단계에서 만들어진 태스크 그래프를 바탕으로 선택된 태스크만 스케줄링해서 실행합니다.
- 필요한 경우 여러 태스크를 병렬로 실행할 수 있습니다.

## 핵심 포인트: 어떤 코드가 언제 실행되는가

빌드 스크립트에 적힌 코드가 정확히 어느 단계에서 실행되는지는 흔히 헷갈리는 부분입니다. 다음 예시로 구분해 볼 수 있습니다.

```kotlin
// 구성 단계에서 실행
println("configuration phase")

tasks.register("configured") {
    // 이 블록 본문도 구성 단계에서 실행
    println("still configuration phase")
}

tasks.register("test") {
    doLast {
        // doFirst/doLast 블록 내부만 실행 단계에서 실행
        println("execution phase")
    }
}
```

- 태스크 등록 블록(`tasks.register { ... }`) 본문 자체는 **구성 단계**에 실행됩니다.
- `doFirst`/`doLast`로 감싼 코드만 **실행 단계**로 미뤄집니다.
- Gradle은 실행하도록 요청받은 태스크와 그 의존 태스크만 구성합니다. 즉 `./gradlew test`를 실행하면 `test`와 관련 없는 다른 태스크(`configured` 등)는 구성조차 되지 않습니다. 이 특성 때문에 대규모 멀티 프로젝트 빌드에서도 필요 없는 부분을 건너뛰어 구성 시간을 아낄 수 있습니다.

## 태스크 그래프(Task Graph)

- 빌드 스크립트와 플러그인은 태스크 사이의 의존 관계를 선언합니다. 예를 들어 `assembleDocs`가 `buildHtml`에, `createDocs`가 `assembleDocs`에 의존한다고 선언하면 Gradle은 `buildHtml → assembleDocs → createDocs` 순서의 그래프를 구성합니다.
- 의존 관계는 `dependsOn`처럼 명시적으로 선언할 수도 있고, 태스크의 입력·출력을 연결하는 방식으로 암묵적으로 선언할 수도 있습니다.
- Gradle은 태스크를 실행하기 **전에** 반드시 전체 그래프를 먼저 구성합니다. 그래서 구성 단계가 끝나야 실행 단계로 넘어갈 수 있습니다.
- 빌드에 참여하는 모든 프로젝트의 태스크를 통틀어 하나의 방향성 비순환 그래프(DAG, Directed Acyclic Graph)를 이룹니다. 순환 의존이 있으면 그래프를 만들 수 없어 빌드가 실패합니다.

## 핵심 포인트: 생명주기 후킹

- 빌드 로직이 복잡해지면 특정 단계가 시작되거나 끝나는 시점에 커스텀 동작을 끼워 넣고 싶을 때가 있습니다.
- Gradle은 이런 요구를 위해 초기화·구성·실행 각 단계의 주요 이벤트를 감지할 수 있는 리스너·콜백 API를 제공합니다.
- 구체적인 후킹 지점과 사용법은 별도의 "Lifecycle API" 문서에서 다루므로, 이 문서는 세 단계의 흐름과 태스크 그래프의 개념을 잡는 데 집중합니다.

## 정리

- Gradle 빌드는 초기화 → 구성 → 실행이라는 고정된 순서를 따르며, 각 단계의 역할이 명확히 분리되어 있습니다.
- 빌드 스크립트 코드가 구성 단계에서 도는지 실행 단계에서 도는지 구분하는 것이 태스크 작성의 기본입니다.
- 태스크 그래프는 실행 전에 완성되는 DAG이며, 요청된 태스크와 그 의존 관계만 구성 대상에 포함됩니다.
