# 6. 브로커

브로커는 다른 브로커와 협력해서 더 큰 시스템을 구성하는데 사용한다.

## 6.1 브로커의 역할

```
┌────────────────────────────────────────────────────────────────┐
│                        Kafka Broker                             │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                    Request Handler                        │  │
│  │  - Producer Requests (메시지 저장)                        │  │
│  │  - Consumer Requests (메시지 페치)                        │  │
│  │  - Metadata Requests (클러스터 정보)                      │  │
│  └──────────────────────────────────────────────────────────┘  │
│                           │                                    │
│                           ▼                                    │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                    Log Manager                            │  │
│  │  - 세그먼트 파일 관리                                     │  │
│  │  - 로그 보존 정책 적용                                    │  │
│  │  - 로그 컴팩션 실행                                       │  │
│  └──────────────────────────────────────────────────────────┘  │
│                           │                                    │
│                           ▼                                    │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                  Replica Manager                          │  │
│  │  - 리더/팔로워 역할 관리                                  │  │
│  │  - ISR 관리                                               │  │
│  │  - 복제 동기화                                            │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 6.2 주요 브로커 설정

| 설정 | 설명 | 기본값 |
|------|------|--------|
| `broker.id` | 브로커 고유 ID | - |
| `log.dirs` | 로그 저장 디렉토리 | /tmp/kafka-logs |
| `num.partitions` | 기본 파티션 수 | 1 |
| `default.replication.factor` | 기본 복제 팩터 | 1 |
| `min.insync.replicas` | 최소 ISR 수 | 1 |
| `unclean.leader.election.enable` | 비동기 레플리카 리더 선출 허용 | false |

## 6.3 파티션 리더 레플리카와 그 역할

### ISR (In-Sync Replicas)

ISR는 카프카를 이해하기 위한 핵심 부분이다.

```
┌────────────────────────────────────────────────────────────────┐
│                    ISR (In-Sync Replicas)                       │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  Partition 0:                                                  │
│  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐      │
│  │   Broker 0  │     │   Broker 1  │     │   Broker 2  │      │
│  │   (Leader)  │────▶│  (Follower) │     │  (Follower) │      │
│  │             │     │   (ISR)     │     │   (Out-Sync)│      │
│  │  Offset:100 │     │  Offset:100 │     │  Offset: 95 │      │
│  └─────────────┘     └─────────────┘     └─────────────┘      │
│                                                                │
│  ISR = {Broker 0, Broker 1}                                    │
│  Broker 2는 replica.lag.time.max.ms 초과로 ISR에서 제외됨      │
│                                                                │
│  High Watermark (HW): 100 (모든 ISR이 동기화된 오프셋)         │
│  Log End Offset (LEO): 100                                     │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

새 토픽의 경우 특정 수의 레플리카가 생성되어 초기 ISR 목록에 추가된다.

이 번호는 매개변수 또는 기본적으로 브로커 구성에서 가져올 수 있다.

### ISR 관리 파라미터

| 파라미터 | 설명 | 기본값 |
|----------|------|--------|
| `replica.lag.time.max.ms` | ISR에서 제외되는 지연 시간 | 30000 |
| `min.insync.replicas` | 쓰기 성공에 필요한 최소 ISR 수 | 1 |

카프카에서 주목해야 할 세부 사항 중 하나는 레플리카가 기본적으로 자체 복구되지 않는다는 것이다.

파티션 사본 중 하나가 존재하는 브로커를 잃어버리면 카프카는 새 사본을 생성하지 않는다.

우리 시스템의 상태를 모니터링할 때 살펴봐야 할 중요한 항목은 얼마나 많은 ISR이 실제로 우리가 원하는 수와 일치하는지다.

### 6.3.1 리더 선출

ISR이 없고 장애로 인해 리더 레플리카가 손실되면 어떻게 될까 ?

```
┌────────────────────────────────────────────────────────────────┐
│                      리더 선출 시나리오                          │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  Clean Leader Election (기본값):                               │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ ISR 목록에 있는 레플리카 중에서만 새 리더 선출            │  │
│  │ → 데이터 유실 없음                                        │  │
│  │ → ISR이 비어있으면 파티션 사용 불가                       │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                │
│  Unclean Leader Election (unclean.leader.election.enable=true):│
│  ┌──────────────────────────────────────────────────────────┐  │
│  │ ISR이 아닌 레플리카도 리더가 될 수 있음                   │  │
│  │ → 가용성 우선                                             │  │
│  │ → 데이터 유실 가능!                                       │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

unclean.leader.election.enable 값이 true이면 컨트롤러는 레플리카가 최신이 아니더라도 시스템이 계속 실행되도록 파티션의 리더를 선택한다.

이 방식의 문제점은 리더가 실패했을 때 살아 있는 레플리카 중 어느 것도 모든 데이터를 갖고 있지 않기 때문에 데이터가 손실될 수 있다는 것이다.

### 6.3.2 Preferred Replica

각 파티션에는 "선호 레플리카"가 있다. 토픽 생성 시 첫 번째 레플리카가 선호 레플리카로 지정된다.

```bash
# 리더 재균형 실행
kafka-leader-election.sh --bootstrap-server localhost:9092 \
  --election-type PREFERRED --all-topic-partitions
```

## 6.4 데이터 복제 프로토콜

```
┌────────────────────────────────────────────────────────────────┐
│                     복제 프로토콜 동작                           │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  1. Producer → Leader (Write)                                  │
│     ┌──────────┐                                               │
│     │ Producer │──── Write ────▶ Leader                        │
│     └──────────┘                                               │
│                                                                │
│  2. Follower ← Leader (Fetch)                                  │
│     Leader ◀──── Fetch Request ──── Follower                   │
│     Leader ──── Fetch Response ────▶ Follower                  │
│                                                                │
│  3. High Watermark 갱신                                        │
│     - 모든 ISR이 복제 완료한 오프셋                            │
│     - Consumer는 HW까지만 읽을 수 있음                         │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 6.5 모니터링

모니터링 도구는 Grafana와 Prometheus가 좋다.

### 주요 모니터링 메트릭

| 메트릭 | 설명 | 임계값 |
|--------|------|--------|
| `UnderReplicatedPartitions` | ISR이 부족한 파티션 수 | > 0 경고 |
| `OfflinePartitionsCount` | 오프라인 파티션 수 | > 0 위험 |
| `ActiveControllerCount` | 활성 컨트롤러 수 | = 1 |
| `RequestHandlerAvgIdlePercent` | 요청 핸들러 유휴율 | < 30% 경고 |
| `UncleanLeaderElectionsPerSec` | 비정상 리더 선출 빈도 | > 0 위험 |

Kafka는 런타임에 수평확장할 수 있다는 장점이 존재한다.

## 6.6 브로커 운영

### 브로커 추가 시 고려사항

1. 새 브로커는 빈 상태로 시작
2. 기존 파티션이 자동 재분배되지 않음
3. `kafka-reassign-partitions.sh`로 수동 재분배 필요

### 롤링 재시작 절차

```bash
# 1. 브로커를 ISR에서 제거 (Controlled Shutdown)
# 2. 브로커 재시작
# 3. ISR 복귀 확인 후 다음 브로커 진행
```
