# 8. 카프카 스토리지

## 8.1 로그 저장 구조

카프카는 메시지를 디스크에 저장하는 분산 커밋 로그 시스템이다.

```
┌────────────────────────────────────────────────────────────────┐
│                    카프카 스토리지 계층                          │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  /kafka-logs/                                                  │
│  │                                                             │
│  ├── orders-0/          ← Topic: orders, Partition: 0         │
│  │   ├── 00000000000000000000.log                              │
│  │   ├── 00000000000000000000.index                            │
│  │   ├── 00000000000000000000.timeindex                        │
│  │   ├── 00000000000000523400.log                              │
│  │   └── leader-epoch-checkpoint                               │
│  │                                                             │
│  ├── orders-1/          ← Topic: orders, Partition: 1         │
│  │   └── ...                                                   │
│  │                                                             │
│  ├── __consumer_offsets-0/  ← 내부 토픽                       │
│  │   └── ...                                                   │
│  │                                                             │
│  └── recovery-point-offset-checkpoint                          │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

## 8.2 세그먼트 파일

### 세그먼트 구성 요소

| 파일 유형 | 확장자 | 설명 |
|-----------|--------|------|
| **로그 파일** | `.log` | 실제 메시지 데이터 |
| **오프셋 인덱스** | `.index` | 오프셋 → 파일 위치 매핑 |
| **타임스탬프 인덱스** | `.timeindex` | 타임스탬프 → 오프셋 매핑 |
| **트랜잭션 인덱스** | `.txnindex` | 트랜잭션 인덱스 (EOS 사용 시) |

### 세그먼트 롤링 조건

새 세그먼트가 생성되는 조건:
- `log.segment.bytes` 도달 (기본: 1GB)
- `log.segment.ms` 경과 (기본: 7일)
- 인덱스/타임인덱스 파일이 가득 참

## 8.3 로그 정리 정책

### 8.3.1 Delete 정책 (기본)

```
┌────────────────────────────────────────────────────────────────┐
│                    Delete 정책 동작                             │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  retention.ms 또는 retention.bytes 기준으로 오래된 세그먼트 삭제 │
│                                                                │
│  [Seg 1]    [Seg 2]    [Seg 3]    [Seg 4]    [Seg 5]          │
│   삭제 ←     삭제 ←     보존       보존       활성 (쓰기 중)   │
│  (7일 초과)  (7일 초과)                                        │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 8.3.2 Compact 정책

```
┌────────────────────────────────────────────────────────────────┐
│                    Log Compaction 동작                          │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  Before Compaction:                                            │
│  [K1:V1] [K2:V1] [K1:V2] [K3:V1] [K2:V2] [K1:V3]              │
│                                                                │
│  After Compaction:                                             │
│  [K3:V1] [K2:V2] [K1:V3]  ← 각 키의 최신 값만 유지            │
│                                                                │
│  사용 사례:                                                     │
│  - CDC (Change Data Capture)                                   │
│  - 상태 저장 (State Store)                                     │
│  - 캐시 복원                                                   │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### Compaction 설정

| 설정 | 설명 | 기본값 |
|------|------|--------|
| `log.cleanup.policy` | compact 또는 delete,compact | delete |
| `log.cleaner.min.cleanable.ratio` | 컴팩션 트리거 비율 | 0.5 |
| `log.cleaner.delete.retention.ms` | Tombstone 보존 시간 | 86400000 (1일) |
| `min.cleanable.dirty.ratio` | 더티 비율 임계값 | 0.5 |

### Tombstone (삭제 마커)

키를 삭제하려면 값을 null로 설정한 메시지를 전송한다:

```java
producer.send(new ProducerRecord<>("topic", key, null));  // Tombstone
```

Tombstone은 `delete.retention.ms` 기간 동안 유지된 후 제거된다.

## 8.4 Debezium (CDC)

데베지움은 데이터베이스를 이벤트 스트림으로 전환하는데 도움이 되는 분산 플랫폼이라고 설명되어 있다.

즉 데이터베이스 업데이트를 이벤트로 처리할 수 있다.

```
┌────────────────────────────────────────────────────────────────┐
│                    Debezium CDC 아키텍처                        │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  ┌──────────┐     ┌─────────────┐     ┌──────────┐            │
│  │  MySQL   │────▶│  Debezium   │────▶│  Kafka   │            │
│  │ (Binlog) │     │  Connector  │     │  Topics  │            │
│  └──────────┘     └─────────────┘     └──────────┘            │
│                                              │                 │
│  ┌──────────┐     ┌─────────────┐            ▼                 │
│  │PostgreSQL│────▶│  Debezium   │     ┌──────────┐            │
│  │  (WAL)   │     │  Connector  │────▶│Consumers │            │
│  └──────────┘     └─────────────┘     └──────────┘            │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

변경 데이터 캡처(CDC)라는 기술을 활용한다.

데베지움은 애플리케이션 카프카에서 일반 클라이언트가 소비하는 이벤트를 기록하기 위해 카프카 커넥트와 커넥터를 사용한다.

### 지원하는 데이터베이스

| 데이터베이스 | 캡처 방식 |
|-------------|----------|
| MySQL | Binlog |
| PostgreSQL | WAL (Write-Ahead Log) |
| MongoDB | Oplog |
| SQL Server | CDC Tables |
| Oracle | LogMiner |

## 8.5 티어드 스토리지 (Tiered Storage)

Kafka 3.6+에서 도입된 기능으로, 오래된 데이터를 저렴한 원격 스토리지로 이동할 수 있다.

```
┌────────────────────────────────────────────────────────────────┐
│                    Tiered Storage 아키텍처                      │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  ┌─────────────────┐     ┌─────────────────────────────────┐  │
│  │   Local Tier    │     │         Remote Tier             │  │
│  │  (브로커 디스크) │────▶│  (S3, GCS, Azure Blob 등)       │  │
│  │                 │     │                                 │  │
│  │  최근 데이터     │     │  오래된 데이터 (저렴한 스토리지) │  │
│  │  빠른 접근       │     │  무한 확장 가능                 │  │
│  └─────────────────┘     └─────────────────────────────────┘  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```
