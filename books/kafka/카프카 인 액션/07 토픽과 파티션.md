# 7. 토픽과 파티션

## 7.1 토픽

토픽은 물리적 구조가 아니라 논리적 개념이다.

```
┌────────────────────────────────────────────────────────────────┐
│                         Topic 구조                              │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  Topic: "orders"                                               │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                                                          │  │
│  │  Partition 0 ──▶ [msg0][msg1][msg2][msg3][msg4]...      │  │
│  │                  Broker 1 (Leader)                       │  │
│  │                  Broker 2, 3 (Followers)                 │  │
│  │                                                          │  │
│  │  Partition 1 ──▶ [msg0][msg1][msg2][msg3]...            │  │
│  │                  Broker 2 (Leader)                       │  │
│  │                  Broker 1, 3 (Followers)                 │  │
│  │                                                          │  │
│  │  Partition 2 ──▶ [msg0][msg1][msg2][msg3][msg4][msg5]...│  │
│  │                  Broker 3 (Leader)                       │  │
│  │                  Broker 1, 2 (Followers)                 │  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

토픽은 일반적으로 하나의 브로커에만 존재하는 것은 아니다.

대부분의 애플리케이션은 해당 데이터를 단일 토픽에 있는 것으로 보며, 구독하는 데 다른 세부 정보가 필요한 것은 아니다.

그러나 토픽 이름 뒤에는 실제로 데이터를 보유하는 하나 이상의 파티션이 있다.

### 7.1.1 토픽 생성 옵션

```bash
# 토픽 생성
kafka-topics.sh --bootstrap-server localhost:9092 \
  --create \
  --topic my-topic \
  --partitions 6 \
  --replication-factor 3 \
  --config retention.ms=604800000  # 7일

# 토픽 설정 변경
kafka-configs.sh --bootstrap-server localhost:9092 \
  --alter \
  --entity-type topics \
  --entity-name my-topic \
  --add-config retention.ms=86400000  # 1일로 변경
```

### 주요 토픽 설정

| 설정 | 설명 | 기본값 |
|------|------|--------|
| `partitions` | 파티션 수 | 1 |
| `replication.factor` | 복제 팩터 | 1 |
| `retention.ms` | 메시지 보존 시간 | 604800000 (7일) |
| `retention.bytes` | 파티션당 최대 크기 | -1 (무제한) |
| `cleanup.policy` | 정리 정책 (delete/compact) | delete |
| `min.insync.replicas` | 최소 ISR 수 | 1 |

또 중요한 결정은 토픽을 생성하기 위해 삭제해야 하는지 여부이다.

- 설정을 통해 삭제할 수 있게 설정할 수 있음 (`delete.topic.enable=true`)

### 7.1.2 복제 팩터

실질적으로 운영되기 위해서는 총 레플리카 수가 브로커 수보다 적거나 같도록 계획해야 한다.

| 환경 | 권장 복제 팩터 |
|------|---------------|
| 개발/테스트 | 1 |
| 프로덕션 | 3 |
| 높은 내구성 요구 | 3 + `min.insync.replicas=2` |

실제로 총 브로커 수보다 많은 레플리카 수로 토픽을 생성하려고 시도하면 InvalidReplicationFactorException 오류가 발생한다.

## 7.2 파티션

컨슈머 입장에서 파티션은 변경할 수 없는 메시지 로그이다.

이 데이터가 실제로 영원히 증가하지 않지만 데이터를 제자리에서 수정하는 것이 아니라 추가하는 것이 유지 관리하기에 좋은 멘탈 모델이다.

또한 컨슈머 클라이언트는 메시지를 직접 삭제할 수 없다.

### 7.2.1 파티션 물리적 구조

파티션은 실제 디스크에서 파티션이 하나의 단일 파일이 아니라 여러 세그먼트로 분할됨을 의미한다.

```
┌────────────────────────────────────────────────────────────────┐
│                    파티션 세그먼트 구조                          │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  /kafka-logs/orders-0/                                         │
│  │                                                             │
│  ├── 00000000000000000000.log      ← 세그먼트 파일 (메시지)    │
│  ├── 00000000000000000000.index    ← 오프셋 인덱스             │
│  ├── 00000000000000000000.timeindex← 타임스탬프 인덱스         │
│  │                                                             │
│  ├── 00000000000000105000.log      ← 두 번째 세그먼트         │
│  ├── 00000000000000105000.index                                │
│  ├── 00000000000000105000.timeindex                            │
│  │                                                             │
│  └── 00000000000000210000.log      ← 활성 세그먼트 (쓰기 중)  │
│      ...                                                       │
│                                                                │
│  파일명 = Base Offset (해당 세그먼트의 첫 번째 메시지 오프셋)  │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 7.2.2 파티션 수 결정

```
┌────────────────────────────────────────────────────────────────┐
│                    파티션 수 결정 공식                           │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  파티션 수 = max(처리량 기반, 컨슈머 기반)                      │
│                                                                │
│  처리량 기반:                                                   │
│    필요 파티션 수 = 목표 처리량 / 단일 파티션 처리량            │
│    예: 100MB/s 목표, 파티션당 10MB/s → 10개 파티션             │
│                                                                │
│  컨슈머 기반:                                                   │
│    파티션 수 ≥ 최대 병렬 컨슈머 수                              │
│                                                                │
│  주의사항:                                                      │
│  - 파티션 수는 증가만 가능 (감소 불가)                         │
│  - 파티션 증가 시 키 기반 파티셔닝 무효화                      │
│  - 파티션당 브로커 리소스 소비 (메모리, 파일 핸들)             │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```
