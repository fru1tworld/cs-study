# 07 객체 분해

## 프로시저 추상화와 데이터 추상화

소프트웨어를 설계하는 두 가지 방법:
- **프로시저 추상화**: 기능 분해(함수 중심)
- **데이터 추상화**: 타입 추상화(데이터 중심)

## 프로시저 추상화와 기능 분해

### 하향식 설계 (Top-Down)
- 시스템을 세분화하는 방법
- 최상위 기능을 하위 기능으로 분해
- 각 단계를 정제해 나감

### 문제점
- 시스템은 하나의 메인 함수로 구성되지 않음
- 기능 추가와 변경에 취약
- 비즈니스 로직과 사용자 인터페이스 강결합
- 데이터 변경의 파급 효과

## 모듈

### 모듈의 장점
- **분해의 목적**: 변경을 관리하기 위함
- **정보 은닉**: 내부 정보를 감춤
- **모듈 독립성**: 낮은 결합도, 높은 응집도

### 모듈화 원칙
- 비밀을 결정하고 숨긴다
- 외부에 제공할 인터페이스를 결정
- 나머지 세부사항은 모듈 내부에 감춤

## 데이터 추상화와 추상 데이터 타입

### 추상 데이터 타입 (ADT)
- 타입 정의를 선언부와 구현부로 분리
- 외부에서는 오퍼레이션만 사용
- 내부 표현 방식을 감춤

### 특징
- 타입을 기준으로 프로그램 구조화
- 오퍼레이션 추가가 어렵움
- 타입 추가는 용이

## 클래스

### 객체지향 vs 추상 데이터 타입
- **추상 데이터 타입**: 타입을 외부에 노출
- **객체지향**: 다형성을 통한 확장

### 클래스의 장점
- 상속을 통한 코드 재사용
- 다형성을 통한 유연성
- 새로운 타입 추가 용이

## 책임 주도 설계

### 객체 분해 방법
1. 협력을 설계한다
2. 책임을 식별한다
3. 적절한 객체를 찾는다
4. 메시지를 수신할 객체를 선택

### 책임 할당 원칙
- **높은 응집도**: 관련된 책임만 할당
- **낮은 결합도**: 최소한의 의존성
- **정보 전문가**: 정보를 알고 있는 객체에 책임 할당

## 변경 관리

### 변경을 대비하는 설계
- 캡슐화를 통한 변경 영향 최소화
- 추상화를 통한 변경 흡수
- 다형성을 통한 확장

### 설계 개선
- 응집도가 높은가?
- 결합도가 낮은가?
- 변경이 용이한가?

## 교훈

1. **프로시저 추상화보다 데이터 추상화를 선호하라**
2. **객체를 데이터로 보지 말고 협력하는 단위로 봐라**
3. **책임을 중심으로 설계하라**
4. **변경을 관리할 수 있는 구조로 분해하라**
5. **모듈의 비밀을 결정하고 감춰라**
