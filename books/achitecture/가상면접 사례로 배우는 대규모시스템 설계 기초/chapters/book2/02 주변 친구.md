# 02 주변 친구 (Nearby Friends)

## 개요

사용자의 친구 목록에서 지리적으로 가까이 있는 친구를 실시간으로 표시하는 기능 설계. 페이스북의 "Nearby Friends" 기능이 대표적인 예시. 근접성 서비스와 달리 사용자 위치가 동적으로 계속 변함.

---

## 1단계: 문제 이해 및 설계 범위 확정

### 기능 요구사항

- 모바일 앱에서 **5마일(약 8km) 반경** 내 친구 위치 확인
- 친구까지의 거리와 마지막 갱신 시각 표시
- 친구 목록은 ** 몇 초 단위** 로 갱신
- 10분 이상 비활성 상태인 친구는 목록에서 제외

### 비기능 요구사항

- ** 낮은 지연 시간**: 위치 변경을 신속하게 반영
- ** 높은 가용성**: 일부 데이터 손실 허용
- ** 약한 일관성**: 최종 일관성(Eventual Consistency) 허용
- ** 신뢰성**: 가끔 위치 갱신 누락은 허용

### 개략적 규모 추정

| 항목 | 수치 |
|------|------|
| 일간 활성 사용자(DAU) | 1억 명 |
| 동시 접속 사용자 | 1천만 명 (DAU의 10%) |
| 위치 전송 주기 | 30초 |
| 위치 갱신 QPS | ~334,000 (10M / 30) |
| 평균 친구 수 | 400명 |
| 동시 활성 친구 수 | 40명 (10%) |

## 2단계: 개략적 설계안

### 프로토콜 선택

| 프로토콜 | 특징 | 적합성 |
|----------|------|--------|
| HTTP | 요청-응답 기반, 단방향 | X |
| HTTP Long Polling | 서버 푸시 가능, 비효율적 | △ |
| **WebSocket** | 양방향, 실시간, 저지연 | **O** |

**WebSocket 선택 이유:**
- 양방향 실시간 통신 지원
- 연결 유지로 HTTP보다 낮은 지연 시간
- 위치 전송(클라이언트→서버) + 친구 위치 수신(서버→클라이언트) 모두 필요

### API 설계

#### WebSocket 초기화

클라이언트 연결 시:
1. 사용자의 모든 친구 목록 조회
2. 위치 캐시에서 모든 친구 위치 일괄 조회
3. 검색 반경 내 친구 위치만 클라이언트에 전송

#### 주기적 위치 업데이트 (클라이언트 → 서버)

```json
{
  "type": "location_update",
  "user_id": "user123",
  "latitude": 37.7749,
  "longitude": -122.4194,
  "timestamp": 1699123456
}
```

#### 친구 위치 푸시 (서버 → 클라이언트)

```json
{
  "type": "friend_location",
  "friend_id": "friend456",
  "latitude": 37.7750,
  "longitude": -122.4195,
  "distance": 150,
  "last_updated": 1699123450
}
```

### 데이터 모델

#### 위치 캐시 (Redis)

```
Key: location:{user_id}
Value: {latitude, longitude, timestamp}
TTL: 10분 (비활성 시 자동 삭제)
```

#### 위치 이력 (Cassandra)

| 필드 | 타입 | 설명 |
|------|------|------|
| user_id | UUID | PK |
| timestamp | TIMESTAMP | 클러스터링 키 |
| latitude | DOUBLE | 위도 |
| longitude | DOUBLE | 경도 |

### 개략적 설계

```
┌─────────────────────────────────────────────────────────────────────────┐
│                            Load Balancer                                │
└───────────────────────────────┬─────────────────────────────────────────┘
                                │
        ┌───────────────────────┴───────────────────────┐
        ▼                                               ▼
┌───────────────────┐                         ┌───────────────────┐
│   RESTful API     │                         │  WebSocket 서버    │
│     서버          │                         │   (Stateful)      │
│ (친구관리, 설정)   │                         │                   │
└───────────────────┘                         └─────────┬─────────┘
                                                        │
                                    ┌───────────────────┼───────────────────┐
                                    ▼                   ▼                   ▼
                          ┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
                          │ Redis Pub/Sub   │ │ Redis 위치 캐시  │ │    사용자 DB    │
                          │   (채널 라우팅)  │ │  (TTL 기반)     │ │   (친구 관계)   │
                          └─────────────────┘ └─────────────────┘ └─────────────────┘
                                                        │
                                                        ▼
                                              ┌─────────────────┐
                                              │   Cassandra     │
                                              │  (위치 이력)     │
                                              └─────────────────┘
```

## 3단계: 상세 설계

### 웹소켓 서버

#### 특징

- ** 유상태(Stateful) 서버**: 클라이언트와 지속적 연결 유지
- 각 서버가 자신과 연결된 사용자들의 연결 정보 관리
- 거리 계산 및 변경사항 실시간 푸시

#### 연결 초기화 흐름

```
1. 클라이언트 WebSocket 연결
         │
         ▼
2. 사용자 DB에서 친구 목록 조회
         │
         ▼
3. Redis 위치 캐시에서 친구 위치 일괄 조회
         │
         ▼
4. 각 친구와의 거리 계산
         │
         ▼
5. 반경 내 친구만 필터링하여 클라이언트에 전송
         │
         ▼
6. Redis Pub/Sub에서 모든 친구 채널 구독
```

#### 위치 업데이트 처리 흐름

```
1. 클라이언트에서 위치 업데이트 수신
         │
         ▼
2. Redis 위치 캐시 갱신 (TTL 리프레시)
         │
         ▼
3. Redis Pub/Sub 채널에 위치 발행
         │
         ▼
4. 해당 채널 구독 중인 친구들의 WebSocket 핸들러 호출
         │
         ▼
5. 각 핸들러에서 거리 재계산
         │
         ▼
6. 반경 내면 친구 클라이언트에 새 위치 푸시
```

---

### Redis 위치 캐시

#### 설계

```
Key: location:user_123
Value: {"lat": 37.7749, "lng": -122.4194, "ts": 1699123456}
TTL: 600초 (10분)
```

#### 특징

- 위치 업데이트마다 TTL 갱신
- TTL 만료 = 사용자 비활성 = 자동 삭제
- 단일 Redis 서버로 모든 활성 사용자 위치 저장 가능
  - 1천만 사용자 × 100바이트 ≈ 1GB

---

### Redis Pub/Sub

#### 채널 구조

```
채널명: location:{user_id}

예시:
- location:user_123  ← user_123의 위치 업데이트 채널
- location:user_456  ← user_456의 위치 업데이트 채널
```

#### 동작 원리

```
┌─────────────────────────────────────────────────────────────────┐
│                     Redis Pub/Sub 서버                          │
│                                                                 │
│  location:user_A    location:user_B    location:user_C         │
│       │                  │                  │                   │
│       │                  │                  │                   │
└───────┼──────────────────┼──────────────────┼───────────────────┘
        │                  │                  │
        ▼                  ▼                  ▼
   ┌─────────┐       ┌─────────┐       ┌─────────┐
   │Friend B │       │Friend A │       │Friend A │
   │Friend C │       │Friend C │       │Friend B │
   │ (구독)  │       │ (구독)  │       │ (구독)  │
   └─────────┘       └─────────┘       └─────────┘
```

**User A가 위치 업데이트 시:**
1. WebSocket 서버가 `location:user_A` 채널에 발행
2. 해당 채널을 구독 중인 Friend B, C의 핸들러가 메시지 수신
3. 각 핸들러에서 거리 계산 후 반경 내면 클라이언트에 푸시

#### Pub/Sub 장점

- ** 채널 생성 비용 저렴**: 수백만 개 채널 생성 가능
- ** 낮은 지연**: 메모리 기반 메시지 전달
- ** 단순한 구조**: 발행자-구독자 모델

#### Pub/Sub 한계

- ** 메시지 영속성 없음**: 구독자가 없으면 메시지 손실
- **At-most-once 전달**: 메시지 유실 가능

---

### 분산 Redis Pub/Sub

#### 병목 현상

- 1천만 동시 사용자 × 평균 40명 활성 친구
- 초당 약 **1,400만 개** 위치 업데이트 메시지 구독자에게 전달
- 단일 Redis 서버로 처리 불가 (CPU 병목)

#### 샤딩 전략

** 일관된 해싱(Consistent Hashing) 사용:**

```
┌──────────────────────────────────────────────────────────────┐
│                      Service Discovery                        │
│                      (ZooKeeper/etcd)                        │
│                                                              │
│              Redis Pub/Sub 서버 해시 링 관리                  │
└──────────────────────────────────────────────────────────────┘
                              │
           ┌──────────────────┼──────────────────┐
           ▼                  ▼                  ▼
    ┌────────────┐     ┌────────────┐     ┌────────────┐
    │  Redis #1  │     │  Redis #2  │     │  Redis #3  │
    │ (채널 A~H) │     │ (채널 I~P) │     │ (채널 Q~Z) │
    └────────────┘     └────────────┘     └────────────┘
```

- 약 **100개 이상** 의 Redis Pub/Sub 서버 필요
- 채널 ID로 해시하여 담당 서버 결정
- 서버 추가/제거 시 영향 최소화

---

### WebSocket 서버 확장

#### 유상태 서버의 어려움

- 서버 제거 시 연결된 클라이언트 영향
- 로드밸런싱 복잡

#### 해결 방안

1. **Graceful Shutdown**: 서버 제거 전 연결 드레이닝
2. ** 일관된 해싱**: 사용자 ID 기반 서버 할당
3. ** 재연결 로직**: 클라이언트에서 자동 재연결

---

### 위치 이력 저장

#### Cassandra 선택 이유

- ** 막대한 쓰기 부하** 처리 (초당 33만 건)
- ** 시계열 데이터** 에 최적화
- ** 수평 확장** 용이
- ** 최종 일관성**: 실시간성보다 내구성 중요

#### 데이터 모델

```sql
CREATE TABLE location_history (
    user_id UUID,
    timestamp TIMESTAMP,
    latitude DOUBLE,
    longitude DOUBLE,
    PRIMARY KEY (user_id, timestamp)
) WITH CLUSTERING ORDER BY (timestamp DESC);
```

## 4단계: 마무리

### 핵심 포인트

1. **WebSocket**: 실시간 양방향 통신의 핵심
2. **Redis Pub/Sub**: 경량 메시지 라우팅, 채널 기반 구독
3. **Redis 위치 캐시**: TTL 기반 활성 사용자 관리
4. ** 일관된 해싱**: Pub/Sub 서버 샤딩

### 추가 논의 주제

- ** 지오펜싱**: 특정 지역 진입/이탈 알림
- ** 랜덤 주변 사람**: 친구 외 근처 사용자 표시 (지오해시 기반 채널)
- ** 프라이버시**: 위치 공유 설정, 정확도 조절

### 확장 아이디어

```
지오해시 기반 공개 채널:
┌─────────────────────────────────────────┐
│  geohash:9q8yy  │  geohash:9q8yz  │ ... │
│   (지역 채널)    │   (지역 채널)    │     │
└─────────────────────────────────────────┘
        │                  │
        ▼                  ▼
  해당 지역의          해당 지역의
  사용자들 구독        사용자들 구독
```

랜덤 주변 사람 기능은 친구 채널 대신 ** 지오해시 기반 지역 채널** 을 구독하여 구현 가능.
