# 08 분산 이메일 서비스 (Distributed Email Service)

## 개요

Gmail과 같은 대규모 이메일 서비스 설계. 수십억 사용자를 지원하며, 전통적인 이메일 프로토콜(SMTP, IMAP, POP)의 한계를 극복한 현대적 아키텍처.

---

## 1단계: 문제 이해 및 설계 범위 확정

### 기능 요구사항

- 이메일 송신 및 수신
- 받은편지함 조회 (폴더 관리)
- 이메일 검색
- 첨부파일 지원
- 읽음/안읽음 상태 관리
- 스팸 및 바이러스 필터링

### 비기능 요구사항

- ** 고가용성**: 이메일은 중요 통신 수단
- ** 확장성**: 수십억 사용자 지원
- ** 낮은 지연**: 이메일 전송/수신 빠르게
- ** 데이터 내구성**: 이메일 손실 방지

### 개략적 규모 추정

| 항목 | 수치 |
|------|------|
| 사용자 수 | 10억 명 |
| 일간 이메일 수 | 100억 통 |
| 평균 이메일 크기 | 50KB |
| 일간 저장 용량 | ~500TB |

## 2단계: 개략적 설계안

### 이메일 프로토콜

#### SMTP (Simple Mail Transfer Protocol)

- 이메일 ** 송신** 표준 프로토콜
- 메일 서버 간 통신에 사용
- 포트: 25 (기본), 587 (제출)

#### POP (Post Office Protocol)

- 이메일 ** 수신** 프로토콜
- 다운로드 후 서버에서 삭제 (기본)
- 단일 기기에서만 접근
- 포트: 110

#### IMAP (Internet Message Access Protocol)

- 이메일 ** 수신** 프로토콜
- 서버에 이메일 유지
- 여러 기기에서 동기화
- 포트: 143

### 전통적 이메일 흐름

```
전통적 송신 경로:

발신자 클라이언트
      │
      ▼ (SMTP)
발신자 메일 서버
      │
      ▼ (SMTP)
수신자 메일 서버
      │
      ▼ (IMAP/POP)
수신자 클라이언트
```

** 전통적 시스템의 한계:**
- 확장성 부족 (수십억 사용자 처리 어려움)
- 대용량 첨부파일 처리 어려움
- 현대적 기능 지원 제한 (실시간 검색 등)

### 현대적 이메일 아키텍처

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              클라이언트                                      │
│     (웹앱: HTTP/REST)           (모바일: SMTP/IMAP)                          │
└─────────────────────────────────┬───────────────────────────────────────────┘
                                  │
                   ┌──────────────┴──────────────┐
                   ▼                             ▼
        ┌─────────────────┐           ┌─────────────────┐
        │   웹 서버        │           │  WebSocket 서버  │
        │  (REST API)     │           │  (실시간 알림)   │
        └────────┬────────┘           └─────────────────┘
                 │
                 ▼
        ┌─────────────────┐
        │   메시지 큐      │
        │    (Kafka)      │
        └────────┬────────┘
                 │
    ┌────────────┼────────────┐
    ▼            ▼            ▼
┌─────────┐  ┌─────────┐  ┌─────────┐
│  SMTP   │  │ 메일 처리│  │  스팸   │
│ Workers │  │ Workers │  │ 필터링  │
└────┬────┘  └────┬────┘  └─────────┘
     │            │
     ▼            ▼
┌─────────────────────────────────────────────────────────────┐
│                        저장소 계층                           │
│  ┌───────────────┐  ┌───────────────┐  ┌───────────────┐   │
│  │ 메타데이터 DB  │  │ 첨부파일 저장소│  │ 검색 인덱스    │   │
│  │  (Cassandra)  │  │    (S3)       │  │(Elasticsearch)│   │
│  └───────────────┘  └───────────────┘  └───────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

---

### 이메일 전송 흐름

```
1. 사용자가 이메일 작성 & 전송
           │
           ▼
2. 웹 서버가 요청 수신
           │
           ▼
3. 메시지 검증 (스팸, 바이러스)
           │
           ▼
4. Kafka 큐에 전송 요청 저장
           │
           ▼
5. SMTP Worker가 큐에서 가져옴
           │
           ├── 내부 사용자 → 직접 저장
           │
           └── 외부 도메인 → DNS 조회 후 SMTP 전송
                    │
                    ▼
           수신자 메일 서버
```

### 이메일 수신 흐름

```
1. 외부 SMTP 서버에서 이메일 수신
           │
           ▼
2. SMTP 수신 서버
           │
           ▼
3. Kafka 큐에 저장
           │
           ▼
4. 메일 처리 Worker
           │
           ├── 스팸/바이러스 검사
           │
           ├── 첨부파일 분리 → S3 저장
           │
           ├── 메타데이터 → Cassandra 저장
           │
           └── 검색 인덱스 → Elasticsearch
           │
           ▼
5. WebSocket으로 실시간 알림
```

## 3단계: 상세 설계

### 메타데이터 데이터베이스

#### Cassandra 선택 이유

- ** 높은 쓰기 처리량**: 이메일 수신은 쓰기 집약적
- ** 수평 확장**: 노드 추가로 용량 확장
- ** 높은 가용성**: 복제로 장애 대응

#### 데이터 모델

```sql
-- 사용자별 이메일 목록 (user_id로 샤딩)
CREATE TABLE emails_by_user (
    user_id UUID,
    folder VARCHAR,
    email_id TIMEUUID,
    subject TEXT,
    from_address TEXT,
    is_read BOOLEAN,
    received_at TIMESTAMP,
    PRIMARY KEY ((user_id, folder), received_at, email_id)
) WITH CLUSTERING ORDER BY (received_at DESC);

-- 이메일 본문 (별도 테이블)
CREATE TABLE email_body (
    email_id TIMEUUID PRIMARY KEY,
    body TEXT,
    attachments LIST<TEXT>  -- S3 URL 목록
);
```

#### 읽음 상태 관리

```
비정규화로 빠른 조회:

emails_by_user 테이블에 is_read 컬럼 포함
→ 폴더 조회 시 별도 조인 불필요
→ 상태 변경 시 해당 행만 업데이트
```

### 첨부파일 저장소

```
첨부파일 처리 흐름:

1. 이메일 수신
      │
      ▼
2. 첨부파일 추출
      │
      ▼
3. S3에 저장
      │
      ▼
4. S3 URL을 메타데이터에 저장

      ┌─────────────────────────────────────────┐
      │              Amazon S3                  │
      │                                         │
      │  /attachments/{user_id}/{email_id}/     │
      │    ├── file1.pdf                        │
      │    └── file2.jpg                        │
      └─────────────────────────────────────────┘
```

**S3 선택 이유:**
- 대용량 파일 저장에 최적화
- 높은 내구성 (99.999999999%)
- CDN 연동으로 빠른 다운로드

### 이메일 전달 가능성

#### SPF, DKIM, DMARC

```
이메일 인증 프로토콜:

SPF (Sender Policy Framework):
- 발신 서버 IP 검증
- DNS TXT 레코드에 허용 IP 목록

DKIM (DomainKeys Identified Mail):
- 이메일에 디지털 서명 추가
- 수신자가 서명 검증

DMARC (Domain-based Message Authentication):
- SPF + DKIM 정책 정의
- 인증 실패 시 처리 방법 지정
```

#### 전송 평판 관리

```
평판 점수 영향 요소:
- 스팸 신고 비율
- 반송률 (bounce rate)
- 수신자 참여도

낮은 평판 → 스팸 폴더 또는 차단
```

### 검색

#### Elasticsearch 활용

```
검색 인덱스 구조:

{
  "email_id": "...",
  "user_id": "...",
  "subject": "회의 일정 안내",
  "body": "다음 주 월요일 회의...",
  "from": "sender@example.com",
  "to": ["recipient@example.com"],
  "labels": ["work", "important"],
  "date": "2024-01-15T10:30:00Z"
}
```

** 특징:**
- 이메일은 ** 쓰기가 읽기보다 많음**
- 비동기로 인덱스 업데이트
- 역색인(Inverted Index)으로 빠른 전문 검색

### 확장성 및 가용성

#### 샤딩 전략

```
user_id 기준 샤딩:

┌─────────────────────────────────────────────────────────────┐
│                      Cassandra Cluster                       │
│                                                             │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │   Node 1    │  │   Node 2    │  │   Node 3    │        │
│  │ (user A~H)  │  │ (user I~P)  │  │ (user Q~Z)  │        │
│  └─────────────┘  └─────────────┘  └─────────────┘        │
│                                                             │
│  Virtual Nodes로 자동 리밸런싱                               │
└─────────────────────────────────────────────────────────────┘
```

#### 복제

```
복제 팩터 = 3

각 이메일 데이터가 3개 노드에 저장
→ 2개 노드 장애에도 서비스 유지
```

## 4단계: 마무리

### 핵심 포인트

1. ** 현대적 아키텍처**: 전통 프로토콜 + 현대적 저장소/처리
2. **Cassandra**: 높은 쓰기 처리량, 사용자 기준 샤딩
3. **S3**: 대용량 첨부파일 저장
4. **Elasticsearch**: 빠른 전문 검색
5. ** 이메일 인증**: SPF, DKIM, DMARC로 전달 가능성 확보

### 추가 논의 주제

- ** 스마트 답장**: ML 기반 자동 답장 제안
- ** 분류 및 필터링**: 자동 라벨링, 중요 메일 분류
- ** 암호화**: End-to-End 암호화
- ** 컴플라이언스**: GDPR, 데이터 보존 정책
