# 06 광고 클릭 이벤트 집계 (Ad Click Event Aggregation)

## 개요

디지털 광고 플랫폼에서 광고 클릭 이벤트를 실시간으로 집계하는 시스템 설계. 광고 과금, RTB(Real-Time Bidding), 리포팅에 활용. 정확성이 매우 중요 (수백만 달러 차이 가능).

---

## 1단계: 문제 이해 및 설계 범위 확정

### 기능 요구사항

- 지난 M분 동안 특정 광고(ad_id)의 클릭 수 집계
- 매분 가장 많이 클릭된 Top 100 광고 반환
- IP, 사용자 ID, 국가 등 다양한 속성으로 필터링
- 데이터 정확성 우선 (속도보다 정확성이 중요)

### 비기능 요구사항

- **정확성**: 광고 과금에 사용되므로 높은 정확성 필수
- **지연 허용**: 수 분의 end-to-end 지연 허용
- **장애 내성**: 일부 장애에도 데이터 손실 없음
- **확장성**: 트래픽 증가 대응

### 개략적 규모 추정

| 항목 | 수치 |
|------|------|
| 일간 클릭 수 | 10억 건 |
| 광고 수 | 200만 개 |
| 피크 QPS | ~50,000 |
| 연간 성장률 | 30% |

## 2단계: 개략적 설계안

### 질의 API 설계

#### 1. 광고 클릭 수 조회

```
GET /v1/ads/{ad_id}/aggregated_count?
    from=1699000000&
    to=1699003600&
    filter=country:US
```

**응답:**
```json
{
  "ad_id": "ad_123",
  "count": 125000,
  "from": 1699000000,
  "to": 1699003600
}
```

#### 2. Top N 광고 조회

```
GET /v1/ads/popular?
    minute=202311150930&
    top_n=100
```

## 3단계: 상세 설계

### 스트리밍 vs 배치

| 방식 | 특징 | 장점 | 단점 |
|------|------|------|------|
| **배치** | 주기적 처리 (시간/일) | 구현 간단, 정확 | 높은 지연 |
| **스트리밍** | 실시간 처리 | 낮은 지연 | 구현 복잡 |
| **Lambda** | 배치 + 스트리밍 | 정확성 + 실시간 | 두 코드베이스 유지 |
| **Kappa** | 스트리밍으로 통합 | 단일 코드베이스 | 재처리 시 비용 |

**선택: Kappa 아키텍처**
- 실시간 처리와 과거 데이터 재처리를 동일한 스트리밍 엔진으로 처리
- 단일 코드베이스로 유지보수 용이

### 데이터 집계 서비스

#### MapReduce 파이프라인

```
┌──────────────────────────────────────────────────────────────────┐
│                    MapReduce DAG 모델                             │
│                                                                  │
│  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐        │
│  │   Map Node  │     │   Map Node  │     │   Map Node  │        │
│  │  (ad_id%3=0)│     │  (ad_id%3=1)│     │  (ad_id%3=2)│        │
│  └──────┬──────┘     └──────┬──────┘     └──────┬──────┘        │
│         │                   │                   │                │
│         ▼                   ▼                   ▼                │
│  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐        │
│  │  Aggregate  │     │  Aggregate  │     │  Aggregate  │        │
│  │    Node     │     │    Node     │     │    Node     │        │
│  │(분당 카운트)  │     │(분당 카운트)  │     │(분당 카운트)  │        │
│  └──────┬──────┘     └──────┬──────┘     └──────┬──────┘        │
│         │                   │                   │                │
│         └───────────────────┼───────────────────┘                │
│                             ▼                                    │
│                      ┌─────────────┐                             │
│                      │ Reduce Node │                             │
│                      │ (최종 집계)  │                             │
│                      └─────────────┘                             │
└──────────────────────────────────────────────────────────────────┘
```

#### 각 노드의 역할

| 노드 | 역할 |
|------|------|
| **Map** | 데이터 파티셔닝 (ad_id 기준), 필터링/변환 |
| **Aggregate** | 분 단위 클릭 카운트, Top-N 힙 유지 |
| **Reduce** | 여러 Aggregate 결과 병합, 최종 결과 생성 |

### 시간 및 집계 윈도우

#### 이벤트 시간 vs 처리 시간

```
이벤트 발생 시점과 처리 시점의 차이:

이벤트 시간: 10:00:01  →  처리 시간: 10:00:05 (정상)
이벤트 시간: 10:00:02  →  처리 시간: 10:01:30 (지연!)
```

**이벤트 시간 기준 집계 사용**
- 정확한 시간대별 집계
- 지연된 이벤트 처리 필요

#### 워터마크 (Watermark)

```
집계 윈도우: 10:00 ~ 10:01

워터마크 없이:
10:01에 윈도우 종료 → 10:00:59에 발생했지만 10:01:05에 도착한 이벤트 누락

워터마크 (15초):
10:01:15까지 윈도우 열어둠 → 지연 이벤트 포함

┌──────────────────────────────────────────────┐
│         10:00         10:01      10:01:15    │
│           │             │            │       │
│  ◀────────┼─────────────┼────────────┼──▶    │
│      윈도우 시작      윈도우 끝    워터마크    │
│                    (논리적)     (실제 종료)   │
└──────────────────────────────────────────────┘
```

**트레이드오프:**
- 워터마크 길면: 정확도 ↑, 지연 ↑
- 워터마크 짧으면: 정확도 ↓, 지연 ↓

### 배달 보장

#### 왜 Exactly-once가 필요한가?

```
At-least-once 문제:
광고 클릭 100만 건 → 중복으로 105만 건 집계
→ 광고주에게 5% 더 과금 (수백만 달러 차이)
```

#### Exactly-once 구현

```
┌─────────────────────────────────────────────────────────────┐
│                  Exactly-once 처리                          │
│                                                             │
│  1. Kafka에서 이벤트 읽기                                    │
│                │                                            │
│                ▼                                            │
│  2. 집계 처리                                                │
│                │                                            │
│                ▼                                            │
│  3. 원자적 트랜잭션:                                         │
│     - 집계 결과 저장 + Kafka 오프셋 커밋                      │
│                                                             │
│  실패 시: 마지막 커밋된 오프셋부터 재시작                      │
│  중복 방지: 같은 오프셋의 이벤트는 이미 처리됨                 │
└─────────────────────────────────────────────────────────────┘
```

### 정확히 한 번 처리

```
분산 트랜잭션으로 정확성 보장:

┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   Kafka에서     │     │   집계 결과      │     │   오프셋 커밋    │
│   이벤트 읽기    │ ──▶ │   DB에 저장     │ ──▶ │   (원자적)      │
└─────────────────┘     └─────────────────┘     └─────────────────┘
                                    │
                          ┌────────┴────────┐
                          │ 모두 성공 or 롤백 │
                          └─────────────────┘
```

### 시스템 규모 확장

#### 메시지 큐 확장

```
파티션 키 = ad_id

┌─────────────────────────────────────────────────────────────┐
│                      Kafka Topic                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│  │Partition0│  │Partition1│  │Partition2│  │Partition3│    │
│  │ad_id%4=0 │  │ad_id%4=1 │  │ad_id%4=2 │  │ad_id%4=3 │    │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘    │
└───────┼─────────────┼─────────────┼─────────────┼───────────┘
        ▼             ▼             ▼             ▼
    Consumer 0    Consumer 1    Consumer 2    Consumer 3
```

#### 핫스팟 처리

특정 광고(예: 슈퍼볼 광고)가 트래픽 집중:

```
핫스팟 광고 처리:

일반 광고: ad_123 → Partition (ad_123 % N)
핫 광고:   ad_HOT → 여러 파티션에 분산 + 추가 Reduce 단계

┌──────────┐     ┌──────────┐     ┌──────────┐
│ ad_HOT_0 │     │ ad_HOT_1 │     │ ad_HOT_2 │
│Partition0│     │Partition1│     │Partition2│
└────┬─────┘     └────┬─────┘     └────┬─────┘
     │                │                │
     └────────────────┼────────────────┘
                      ▼
               ┌────────────┐
               │   Reduce   │
               │  (ad_HOT)  │
               └────────────┘
```

#### 스냅샷 기반 복구

```
주기적 스냅샷 저장:
- Kafka 오프셋
- 현재 집계 상태 (카운트, Top-N 힙)

┌─────────────────────────────────────────────────────────────┐
│                      정상 동작                               │
│                                                             │
│  Offset: 1000  ──▶  1001  ──▶  1002  ──▶  1003            │
│  State:  A0    ──▶  A1    ──▶  A2    ──▶  A3              │
│                             │                               │
│                        스냅샷 저장                           │
│                     (Offset:1002, State:A2)                │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                      장애 복구                               │
│                                                             │
│  스냅샷 로드: Offset:1002, State:A2                         │
│        │                                                    │
│        ▼                                                    │
│  Kafka에서 1002부터 재처리                                   │
│        │                                                    │
│        ▼                                                    │
│  상태 복구 완료                                              │
```

## 4단계: 마무리

### 핵심 포인트

1. **Kappa 아키텍처**: 스트리밍으로 실시간 + 재처리 통합
2. **MapReduce DAG**: 병렬 분산 처리로 대규모 데이터 집계
3. **워터마크**: 지연 이벤트 처리와 정확성 균형
4. **Exactly-once**: 광고 과금 정확성을 위한 필수 요소
5. **일간 조정**: 배치로 실시간 집계 결과 검증

### 추가 논의 주제

- **사기 탐지**: 클릭 사기 필터링
- **실시간 대시보드**: 광고주용 실시간 리포팅
- **멀티 리전**: 글로벌 광고 플랫폼의 지역 분산
