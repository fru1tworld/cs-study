# 12 디지털 지갑 (Digital Wallet)

## 개요

디지털 지갑 서비스의 잔액 관리 및 이체 시스템 설계. 정확한 잔액 관리, 분산 트랜잭션 처리, 완전한 감사 추적이 핵심.

---

## 1단계: 문제 이해 및 설계 범위 확정

### 기능 요구사항

- **잔액 조회**: 사용자 지갑 잔액 실시간 조회
- **입금/출금**: 외부 계좌와 지갑 간 자금 이동
- **이체**: 사용자 간 P2P 송금
- **거래 내역 조회**: 전체 거래 기록 확인

### 비기능 요구사항

- **정확성**: 단 1원의 오차도 허용 불가
- **가용성**: 99.99% 이상
- **감사 추적**: 모든 거래의 완전한 기록
- **동시성 처리**: 동일 계좌 동시 접근 처리

### 개략적 규모 추정

| 항목 | 수치 |
|------|------|
| 사용자 수 | 1억 명 |
| 일간 트랜잭션 | 1,000만 건 |
| TPS | ~100 (피크 ~1,000) |

### 데이터 모델

#### 지갑 테이블

```sql
CREATE TABLE wallets (
    wallet_id VARCHAR(64) PRIMARY KEY,
    user_id VARCHAR(64),
    balance DECIMAL(18, 4),
    currency VARCHAR(3),
    version INT DEFAULT 0,  -- 낙관적 락용
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);
```

#### 거래 내역 테이블

```sql
CREATE TABLE transactions (
    transaction_id VARCHAR(64) PRIMARY KEY,
    wallet_id VARCHAR(64),
    type ENUM('DEPOSIT', 'WITHDRAW', 'TRANSFER_IN', 'TRANSFER_OUT'),
    amount DECIMAL(18, 4),
    balance_after DECIMAL(18, 4),
    counterparty_wallet_id VARCHAR(64),
    idempotency_key VARCHAR(64) UNIQUE,
    created_at TIMESTAMP
);
```

## 3단계: 상세 설계

### 분산 트랜잭션

#### 문제 상황

```
A 지갑 → B 지갑 이체:

┌─────────────────────────────────────────────────────────────┐
│                    분산 트랜잭션 문제                         │
│                                                             │
│  1. A 지갑에서 10,000원 차감                                 │
│  2. ← 이 시점에서 장애 발생!                                 │
│  3. B 지갑에 10,000원 입금 (실행 안됨)                        │
│                                                             │
│  결과: A에서는 돈이 빠졌지만 B에는 안 들어옴                  │
│        → 시스템 불일치 (돈 증발)                             │
└─────────────────────────────────────────────────────────────┘
```

### 2단계 커밋 (2PC)

#### 프로토콜

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      2PC (Two-Phase Commit)                              │
│                                                                         │
│                    ┌───────────────────┐                                │
│                    │  트랜잭션 코디네이터 │                               │
│                    └─────────┬─────────┘                                │
│                              │                                          │
│  ┌───────────────────────────┴───────────────────────────┐             │
│  │                                                        │             │
│  │  Phase 1: Prepare (준비)                               │             │
│  │  ─────────────────────────                             │             │
│  │                                                        │             │
│  │  코디네이터 ──"준비하세요"──▶ 참여자 A (A 지갑 서비스)   │             │
│  │  코디네이터 ──"준비하세요"──▶ 참여자 B (B 지갑 서비스)   │             │
│  │                                                        │             │
│  │  참여자 A ──"준비 완료"──▶ 코디네이터                   │             │
│  │  참여자 B ──"준비 완료"──▶ 코디네이터                   │             │
│  │                                                        │             │
│  │  (이 시점에 참여자들은 락을 보유한 채 대기)               │             │
│  │                                                        │             │
│  └───────────────────────────┬───────────────────────────┘             │
│                              │                                          │
│  ┌───────────────────────────┴───────────────────────────┐             │
│  │                                                        │             │
│  │  Phase 2: Commit (커밋)                                │             │
│  │  ───────────────────────                               │             │
│  │                                                        │             │
│  │  모두 "준비 완료" → 코디네이터 ──"커밋"──▶ 모든 참여자   │             │
│  │  하나라도 실패   → 코디네이터 ──"롤백"──▶ 모든 참여자   │             │
│  │                                                        │             │
│  └───────────────────────────────────────────────────────┘             │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 장단점

| 장점 | 단점 |
|------|------|
| 강한 일관성 보장 | 성능 저하 (락 대기) |
| 구현 상대적 단순 | 코디네이터 단일 장애점 |
| | 참여자 장애 시 블로킹 |

### TC/C (Try-Confirm/Cancel)

#### 프로토콜

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     TC/C (TCC) 패턴                                      │
│                                                                         │
│  Phase 1: Try (시도)                                                    │
│  ─────────────────                                                      │
│  - 리소스를 예약 (실제 커밋 X)                                           │
│  - 비즈니스 검증 수행                                                   │
│                                                                         │
│  예: A 지갑                        B 지갑                               │
│      ├── 잔액: 50,000원            ├── 잔액: 30,000원                   │
│      └── 예약: 10,000원 (동결)      └── (대기)                          │
│                                                                         │
│  Phase 2-a: Confirm (확정) - 모든 Try 성공 시                           │
│  ────────────────────────────────────────────                           │
│  - 예약된 리소스 확정                                                   │
│                                                                         │
│      A 지갑                        B 지갑                               │
│      ├── 잔액: 40,000원 (확정)     ├── 잔액: 40,000원 (확정)            │
│      └── 예약: 0                   └── 예약: 0                          │
│                                                                         │
│  Phase 2-b: Cancel (취소) - 하나라도 Try 실패 시                         │
│  ──────────────────────────────────────────────                         │
│  - 예약된 리소스 해제                                                   │
│                                                                         │
│      A 지갑                        B 지갑                               │
│      ├── 잔액: 50,000원 (복원)     ├── 잔액: 30,000원 (변경 없음)       │
│      └── 예약: 0                   └── 예약: 0                          │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 장단점

| 장점 | 단점 |
|------|------|
| 락 보유 시간 짧음 | 비즈니스 로직 복잡 |
| 성능 우수 | 3개 인터페이스 구현 필요 |
| 유연한 보상 로직 | Try/Confirm/Cancel 모두 멱등해야 함 |

### 사가 (Saga)

#### 개념

```
Saga: 로컬 트랜잭션의 연속 + 보상 트랜잭션

T1 → T2 → T3 → T4  (정상 흐름)

T3 실패 시:
T1 → T2 → T3(X) → C2 → C1  (보상 트랜잭션으로 롤백)

Ci: Ti의 보상 트랜잭션
```

### 이벤트 소싱 (Event Sourcing)

#### 개념

```
전통적 방식 vs 이벤트 소싱:

┌─────────────────────────────────────────────────────────────────────────┐
│  전통적 방식: 현재 상태만 저장                                           │
│                                                                         │
│  wallets 테이블:                                                        │
│  ┌────────────┬───────────┐                                            │
│  │ wallet_id  │  balance  │                                            │
│  ├────────────┼───────────┤                                            │
│  │ wallet_123 │  40,000   │  ← 최종 잔액만 알 수 있음                   │
│  └────────────┴───────────┘                                            │
│                                                                         │
│  문제: 어떻게 40,000원이 되었는지 알 수 없음                              │
└─────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────┐
│  이벤트 소싱: 모든 이벤트를 순서대로 저장                                 │
│                                                                         │
│  events 테이블 (Append-Only):                                           │
│  ┌────────┬────────────┬────────────┬───────────┐                      │
│  │ seq_no │ wallet_id  │ event_type │  amount   │                      │
│  ├────────┼────────────┼────────────┼───────────┤                      │
│  │   1    │ wallet_123 │ CREATED    │     0     │                      │
│  │   2    │ wallet_123 │ DEPOSIT    │ +100,000  │                      │
│  │   3    │ wallet_123 │ WITHDRAW   │  -30,000  │                      │
│  │   4    │ wallet_123 │ TRANSFER   │  -30,000  │                      │
│  └────────┴────────────┴────────────┴───────────┘                      │
│                                                                         │
│  현재 잔액 = 0 + 100,000 - 30,000 - 30,000 = 40,000원                   │
│                                                                         │
│  장점:                                                                   │
│  - 모든 상태 변화 추적 가능                                              │
│  - 어느 시점의 잔액도 재계산 가능                                        │
│  - 완전한 감사 추적 (Audit Trail)                                       │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 이벤트 스토어 설계

```sql
CREATE TABLE wallet_events (
    event_id BIGSERIAL PRIMARY KEY,
    wallet_id VARCHAR(64),
    event_type VARCHAR(32),
    amount DECIMAL(18, 4),
    payload JSONB,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 인덱스: 특정 지갑의 이벤트 빠르게 조회
CREATE INDEX idx_wallet_events ON wallet_events(wallet_id, event_id);
```

#### 상태 재구성 (Replay)

```python
def get_wallet_balance(wallet_id):
    events = db.query("""
        SELECT event_type, amount
        FROM wallet_events
        WHERE wallet_id = %s
        ORDER BY event_id
    """, [wallet_id])

    balance = 0
    for event in events:
        if event.event_type == 'DEPOSIT':
            balance += event.amount
        elif event.event_type in ('WITHDRAW', 'TRANSFER_OUT'):
            balance -= event.amount
        elif event.event_type == 'TRANSFER_IN':
            balance += event.amount

    return balance
```

### CQRS (Command Query Responsibility Segregation)

#### 개념

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    CQRS: 명령과 조회 분리                                │
│                                                                         │
│                         ┌───────────────┐                               │
│                         │   클라이언트   │                               │
│                         └───────┬───────┘                               │
│                     ┌───────────┴───────────┐                          │
│                     │                       │                           │
│              명령 (쓰기)                조회 (읽기)                      │
│                     │                       │                           │
│                     ▼                       ▼                           │
│           ┌─────────────────┐     ┌─────────────────┐                  │
│           │   Command Side  │     │   Query Side    │                  │
│           │  (이체, 입출금)  │     │ (잔액, 내역조회)│                  │
│           └────────┬────────┘     └────────┬────────┘                  │
│                    │                       │                           │
│                    ▼                       ▼                           │
│           ┌─────────────────┐     ┌─────────────────┐                  │
│           │   이벤트 스토어  │────▶│   Read Model    │                  │
│           │  (이벤트 저장)   │이벤트│  (잔액 캐시)    │                  │
│           └─────────────────┘  동기화└─────────────────┘                 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 구현 상세


#### Eventual Consistency

```
CQRS의 특성: 최종 일관성

┌─────────────────────────────────────────────────────────────┐
│                   일관성 지연                                │
│                                                             │
│  시간 ──────────────────────────────────────────────▶       │
│                                                             │
│  T1: 이체 명령 실행 (A→B 10,000원)                          │
│                                                             │
│  T2: 이벤트 스토어에 저장 완료                               │
│      Command Side: A=-10,000, B=+10,000 반영                │
│                                                             │
│  T3: 이벤트 발행 → 메시지 큐                                 │
│      (이 시점에 조회하면 이전 잔액 반환 가능)                │
│                                                             │
│  T4: Query Side에서 이벤트 수신                              │
│      Read Model 업데이트                                    │
│                                                             │
│  T5: 일관성 달성                                             │
│      Query Side도 최신 잔액 반환                            │
│                                                             │
│  지연 시간: 보통 수십~수백 ms                                │
└─────────────────────────────────────────────────────────────┘

처리 방법:
- UI에서 "처리 중" 상태 표시
- 요청 후 즉시 반환 대신 polling/webhook
- 중요한 조회는 Command Side에서 직접
```

### 멱등성 (Idempotency)

#### 중복 요청 방지

```sql
-- 멱등성 키 테이블
CREATE TABLE idempotency_keys (
    idempotency_key VARCHAR(64) PRIMARY KEY,
    response JSONB,
    created_at TIMESTAMP,
    expires_at TIMESTAMP
);

-- TTL 인덱스 (만료된 키 자동 삭제)
CREATE INDEX idx_expires ON idempotency_keys(expires_at);
```

#### 처리 로직

```python
def transfer(request):
    key = request.idempotency_key

    # 1. 기존 처리 결과 확인
    existing = db.query(
        "SELECT response FROM idempotency_keys WHERE idempotency_key = %s",
        [key]
    )

    if existing:
        return existing.response  # 이전 결과 반환

    # 2. 새 요청 처리
    try:
        result = execute_transfer(request)

        # 3. 멱등성 키와 결과 저장
        db.execute("""
            INSERT INTO idempotency_keys (idempotency_key, response, expires_at)
            VALUES (%s, %s, NOW() + INTERVAL '24 hours')
        """, [key, json.dumps(result)])

        return result
    except IntegrityError:
        # 동시 요청으로 키 이미 존재 → 기존 결과 조회
        return get_existing_result(key)
```

## 4단계: 마무리

### 핵심 포인트

1. **분산 트랜잭션**: 2PC, TCC, Saga 중 상황에 맞게 선택
2. **이벤트 소싱**: 완전한 감사 추적, 상태 재구성 가능
3. **CQRS**: 읽기/쓰기 분리로 성능 최적화
4. **멱등성**: 모든 API에서 중복 요청 안전 처리
5. **Eventual Consistency**: 최종 일관성 허용으로 확장성 확보

### 추가 논의 주제

- **동시성 제어**: 낙관적 락, 비관적 락, 계좌별 직렬화
- **잔액 부족 처리**: 실시간 검증 vs 사후 처리
- **다중 통화**: 환율 처리, 통화별 잔액 관리
- **한도 관리**: 일일/월간 이체 한도
- **사기 탐지**: 비정상 패턴 실시간 감지
