# 05 지표 모니터링 및 경보 시스템 (Metrics Monitoring and Alerting System)

## 개요

대규모 인프라의 지표를 수집, 저장, 시각화하고 이상 징후 발생 시 경보를 보내는 시스템 설계. Prometheus, Datadog, Grafana 등이 대표적인 예시.

---

## 1단계: 문제 이해 및 설계 범위 확정

### 기능 요구사항

- ** 지표 수집**: CPU, 메모리, 디스크 등 인프라 지표 수집
- ** 지표 저장**: 시계열 데이터 저장 및 보존
- ** 시각화**: 대시보드를 통한 지표 시각화
- ** 경보**: 임계값 초과 시 알림 전송
- ** 유연한 쿼리**: 다양한 집계 및 필터링

### 비기능 요구사항

- ** 확장성**: 수백만 개 지표 처리
- ** 낮은 지연**: 대시보드 및 경보의 빠른 응답
- ** 고가용성**: 중요한 경보 누락 방지
- ** 유연성**: 새로운 지표 및 경보 규칙 쉽게 추가

### 개략적 규모 추정

| 항목 | 수치 |
|------|------|
| 서버 풀 | 1,000개 |
| 서버 수 | 100,000대 |
| 지표 수 | ~1,000만 개 |
| 보존 기간 | 1년 |
| 데이터 수집 주기 | 10초 ~ 1분 |

** 보존 정책:**
- 원시 데이터: 7일
- 1분 해상도: 30일
- 1시간 해상도: 1년

## 2단계: 개략적 설계안

### 데이터 모델

시계열 데이터(Time-Series Data) 구조:

```
지표 이름 + 라벨(태그) + 타임스탬프 + 값

예시:
cpu.load{host=server01, env=prod} 1699123456 0.85
memory.used{host=server01, env=prod} 1699123456 8589934592
```

#### 지표 구성 요소

| 구성 요소 | 설명 | 예시 |
|-----------|------|------|
| ** 지표 이름** | 측정 대상 | cpu.load, memory.used |
| ** 라벨/태그** | 메타데이터 (키-값) | host, env, region |
| ** 타임스탬프** | Unix 시간 | 1699123456 |
| ** 값** | 측정값 | 0.85, 8589934592 |

#### 카운터 vs 게이지

| 유형 | 특성 | 예시 |
|------|------|------|
| ** 카운터** | 단조 증가, 리셋 가능 | 요청 수, 에러 수 |
| ** 게이지** | 증가/감소 가능 | CPU 사용률, 메모리 |

### 데이터 수집

#### Pull 모델

```
지표 수집기가 주기적으로 대상 서버에서 지표를 가져옴

┌────────────────┐         GET /metrics         ┌────────────────┐
│ Metrics        │ ──────────────────────────▶  │   Server 1     │
│ Collector      │                              │   :9090/metrics│
│                │ ◀──────────────────────────  │                │
└────────────────┘         지표 데이터           └────────────────┘
```

** 장점:**
- 수집기가 상태 파악 용이 (응답 없음 = 서버 다운)
- 디버깅 편리 (직접 엔드포인트 호출 가능)
- 일관된 수집 간격 보장

** 단점:**
- 방화벽 뒤 서버 접근 어려움
- 짧은 수명의 작업(Job) 지표 수집 어려움

** 예시**: Prometheus

#### Push 모델

```
에이전트가 지표를 수집기에 전송

┌────────────────┐                              ┌────────────────┐
│   Server 1     │ ──────────────────────────▶  │ Metrics        │
│   Agent        │         POST /metrics        │ Collector      │
└────────────────┘                              │                │
┌────────────────┐                              │                │
│   Server 2     │ ──────────────────────────▶  │                │
│   Agent        │         POST /metrics        │                │
└────────────────┘                              └────────────────┘
```

** 장점:**
- 짧은 수명 작업 지원
- 방화벽 친화적

** 단점:**
- 서버 상태 파악 어려움 (전송 없음 = 다운? 지표 없음?)
- 수집기 과부하 가능

** 예시**: StatsD, Datadog Agent

#### 서비스 디스커버리

```
┌──────────────────────────────────────────────────────────┐
│                   Service Discovery                       │
│                   (etcd, Consul, K8s)                    │
│                                                          │
│   서버 목록: [server1:9090, server2:9090, ...]            │
└──────────────────────────────┬───────────────────────────┘
                               │
                               ▼
                    ┌─────────────────────┐
                    │   Metrics Collector │
                    │                     │
                    │   대상 서버 동적 파악  │
                    └─────────────────────┘
```

### 데이터 저장

#### 왜 시계열 데이터베이스(TSDB)인가?

| DB 유형 | 장점 | 단점 |
|---------|------|------|
| 관계형 DB | 범용적 | 시계열 쿼리 비효율, 튜닝 필요 |
| **TSDB** | ** 시계열 최적화, 압축, 집계** | 범용 쿼리 제한 |

**TSDB 특징:**
- ** 높은 쓰기 처리량**: 초당 수백만 데이터 포인트
- ** 효율적인 압축**: 타임스탬프 델타 인코딩
- ** 시계열 전용 쿼리 언어**: PromQL, InfluxQL
- ** 자동 만료**: 오래된 데이터 자동 삭제

** 대표적인 TSDB:**
- InfluxDB
- Prometheus
- TimescaleDB
- OpenTSDB

#### 저장소 최적화

##### 압축

```
타임스탬프 델타 인코딩:

원본:    1699123456, 1699123466, 1699123476
델타:    1699123456, +10, +10

저장 공간: 32비트 → 4비트
```

##### 다운샘플링 (Downsampling)

```
시간이 지나면 해상도를 낮춤:

원시 데이터 (10초 간격):
t0: 0.5, t10: 0.6, t20: 0.55, t30: 0.58, t40: 0.52, t50: 0.61

1분 해상도로 다운샘플링:
t0-t50: avg=0.56, min=0.5, max=0.61
```

** 보존 정책 예시:**
| 기간 | 해상도 | 용도 |
|------|--------|------|
| 0~7일 | 원본 (10초) | 상세 분석 |
| 7~30일 | 1분 | 일반 분석 |
| 30일~1년 | 1시간 | 장기 트렌드 |

### 경보 시스템

#### 경보 규칙 예시

```yaml
alert: HighCPUUsage
expr: avg(cpu_usage{env="prod"}) > 0.8
for: 5m
labels:
  severity: critical
annotations:
  summary: "CPU usage is above 80%"
  description: "CPU usage on {{ $labels.host }} has been above 80% for 5 minutes"
```

#### 경보 시스템 아키텍처

```
┌─────────────────────────────────────────────────────────────┐
│                     경보 규칙 설정                            │
│                     (YAML/UI)                               │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
                 ┌─────────────────────┐
                 │   경보 관리자         │
                 │  (Alert Manager)    │
                 │                     │
                 │  1. 규칙 평가         │
                 │  2. 상태 관리         │
                 │  3. 중복 제거         │
                 │  4. 그룹화            │
                 └──────────┬──────────┘
                            │
            ┌───────────────┼───────────────┐
            ▼               ▼               ▼
      ┌──────────┐    ┌──────────┐    ┌──────────┐
      │  Email   │    │  Slack   │    │ PagerDuty│
      └──────────┘    └──────────┘    └──────────┘
```

#### 경보 상태 관리

```
┌─────────────┐     조건 충족     ┌─────────────┐
│  Inactive   │ ──────────────▶  │   Pending   │
│  (비활성)    │                  │   (대기)     │
└─────────────┘                  └──────┬──────┘
      ▲                                 │
      │                            for 기간 경과
      │                                 │
      │     조건 미충족                  ▼
      │◀──────────────────────  ┌─────────────┐
      │                         │   Firing    │
      │◀──────────────────────  │   (발생)     │
      │     조건 미충족          └──────┬──────┘
      │                                 │
      │                            알림 전송
      │                                 │
      │     해결됨                       ▼
      └────────────────────────  ┌─────────────┐
                                 │  Resolved   │
                                 │   (해결됨)   │
                                 └─────────────┘
```

** 왜 자체 개발하지 않는가?**
- Grafana는 이미 성숙한 시각화 도구
- 다양한 TSDB와 통합 지원
- 풍부한 플러그인 생태계

## 3단계: 상세 설계

### Kafka를 통한 디커플링

```
수집기 장애 시에도 데이터 유실 방지:

┌──────────────┐      ┌───────────────┐      ┌──────────────┐
│   Collector  │ ──▶  │     Kafka     │ ──▶  │     TSDB     │
└──────────────┘      │   (버퍼링)     │      │   (저장)     │
                      └───────────────┘      └──────────────┘
                             │
                             │ TSDB 다운 시
                             │ 메시지 보존
                             ▼
                      ┌───────────────┐
                      │  복구 후 처리   │
                      └───────────────┘
```

**Kafka 활용 이점:**
- 수집과 저장 분리
- 버퍼링으로 트래픽 급증 대응
- 여러 소비자 지원 (저장, 경보, 분석)

## 4단계: 마무리

### 핵심 포인트

1. **TSDB 선택**: 범용 DB가 아닌 시계열 전용 DB 사용
2. **Push vs Pull**: 환경에 따른 적절한 수집 모델 선택
3. **Kafka 버퍼**: 수집과 저장 분리로 안정성 확보
4. ** 다운샘플링**: 장기 저장 시 해상도 낮춤
5. **Grafana 활용**: 시각화는 검증된 도구 사용

### 추가 논의 주제

- ** 하이 카디널리티 처리**: 라벨 조합이 많을 때의 성능 문제
- ** 분산 TSDB**: 단일 노드 한계 극복
- ** 로그와의 상관관계**: 지표와 로그 연계 분석
- **AIOps**: ML 기반 이상 탐지
