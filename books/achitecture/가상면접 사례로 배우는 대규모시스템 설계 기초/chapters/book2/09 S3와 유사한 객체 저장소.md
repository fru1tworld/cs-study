# 09 S3와 유사한 객체 저장소 (S3-like Object Storage)

## 개요

Amazon S3와 같은 대규모 객체 저장소 설계. 무한에 가까운 확장성, 높은 내구성, 다양한 데이터 유형 저장 지원.

---

## 1단계: 문제 이해 및 설계 범위 확정

### 객체 저장소 특성

| 저장소 유형 | 특성 | 예시 |
|-------------|------|------|
| **블록 스토리지** | 저수준, 고성능, OS 의존 | EBS, 로컬 디스크 |
| **파일 스토리지** | 계층적 디렉토리, POSIX | NFS, EFS |
| **객체 스토리지** | 플랫 구조, HTTP API, 메타데이터 | S3, GCS |

### 기능 요구사항

- **버킷 생성**: 객체를 담는 논리적 컨테이너
- **객체 업로드/다운로드**: PUT/GET 연산
- **객체 버전 관리**: 삭제/덮어쓰기 복구
- **버킷 내 객체 목록 조회**

### 비기능 요구사항

- **데이터 내구성**: 99.9999% (6 nines)
- **서비스 가용성**: 99.99% (4 nines)
- **확장성**: 수백 페타바이트 저장
- **저비용**: 콜드 데이터 저렴하게 저장

### 개략적 규모 추정

| 항목 | 수치 |
|------|------|
| 저장 객체 수 | 수조 개 |
| 총 저장 용량 | 수백 페타바이트 |
| 일간 요청 수 | 수십억 건 |

## 2단계: 개략적 설계안

### 객체 저장소 개념

#### 버킷 (Bucket)

- 객체를 담는 **최상위 컨테이너**
- 전역적으로 고유한 이름
- 접근 제어 및 설정 단위

#### 객체 (Object)

- **객체 데이터 (Payload)**: 실제 저장할 데이터
- **객체 메타데이터**: 이름, 크기, 생성일, 사용자 정의 메타데이터

### API 설계

#### 버킷 API

```
PUT /buckets/{bucket_name}     # 버킷 생성
DELETE /buckets/{bucket_name}  # 버킷 삭제
GET /buckets/{bucket_name}     # 객체 목록 조회
```

#### 객체 API

```
PUT /buckets/{bucket_name}/objects/{object_key}    # 객체 업로드
GET /buckets/{bucket_name}/objects/{object_key}    # 객체 다운로드
DELETE /buckets/{bucket_name}/objects/{object_key} # 객체 삭제
```

### 개략적 설계

```
┌─────────────────────────────────────────────────────────────────────────┐
│                              클라이언트                                  │
└─────────────────────────────────┬───────────────────────────────────────┘
                                  │
                                  ▼
                       ┌─────────────────────┐
                       │   Load Balancer     │
                       └──────────┬──────────┘
                                  │
                                  ▼
                       ┌─────────────────────┐
                       │    API 서비스        │
                       │  (무상태, 수평확장)  │
                       └──────────┬──────────┘
                                  │
         ┌────────────────────────┼────────────────────────┐
         ▼                        ▼                        ▼
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  메타데이터 DB   │     │   데이터 저장소    │     │   IAM 서비스    │
│  (버킷/객체 정보)│     │   (실제 데이터)   │     │   (인증/인가)   │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

### 객체 업로드

```
1. 클라이언트가 PUT 요청
           │
           ▼
2. API 서비스가 메타데이터 DB에 객체 정보 저장
           │
           ▼
3. 데이터 저장소에 실제 데이터 저장
           │
           ▼
4. 복제 (내구성을 위해 여러 노드에 저장)
           │
           ▼
5. 성공 응답 반환
```

### 객체 다운로드

```
1. 클라이언트가 GET 요청
           │
           ▼
2. 메타데이터 DB에서 객체 위치 조회
           │
           ▼
3. 데이터 저장소에서 데이터 읽기
           │
           ▼
4. 클라이언트에 스트리밍 응답
```

## 3단계: 상세 설계

### 데이터 저장소

#### 분산 파일 시스템 구조

```
데이터 노드 클러스터:

┌─────────────────────────────────────────────────────────────┐
│                     데이터 저장소 클러스터                    │
│                                                             │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │ Data Node 1 │  │ Data Node 2 │  │ Data Node 3 │        │
│  │  (Disk 1~N) │  │  (Disk 1~N) │  │  (Disk 1~N) │        │
│  └─────────────┘  └─────────────┘  └─────────────┘        │
│                                                             │
│  복제 팩터 = 3 (각 객체가 3개 노드에 저장)                   │
└─────────────────────────────────────────────────────────────┘
```

#### Erasure Coding

복제 대신 더 효율적인 방식:

```
Erasure Coding (예: 6+3):
- 데이터를 6개 청크로 분할
- 3개 패리티 청크 생성
- 총 9개 청크 중 6개만 있으면 복구 가능

저장 오버헤드:
- 3x 복제: 300%
- 6+3 Erasure Coding: 150%
```

### 객체 버전 관리

#### 버전 활성화

```sql
-- 버킷 테이블
ALTER TABLE buckets ADD COLUMN is_versioning_enabled BOOLEAN;

-- 객체 테이블
ALTER TABLE objects ADD COLUMN version_id INT;
```

#### 버전 관리 동작

```
버전 관리 활성화 시:

PUT object_A (첫 번째) → object_A, version_id=1
PUT object_A (두 번째) → object_A, version_id=2

GET object_A → version_id=2 (최신 버전)
GET object_A?versionId=1 → version_id=1

DELETE object_A → Delete Marker 추가 (version_id=3)
GET object_A → 404 Not Found

실제 삭제되지 않음, 이전 버전 접근 가능
```

#### 삭제 마커 (Delete Marker)

```
버전 기록:

version_id=1: 원본 데이터 (100KB)
version_id=2: 수정된 데이터 (120KB)
version_id=3: Delete Marker (0KB)  ← 삭제 표시

GET /bucket/object → 404 (Delete Marker가 최신)
GET /bucket/object?versionId=2 → 수정된 데이터 반환
```

### 대용량 파일 업로드

#### Multipart Upload

```
5GB 파일 업로드:

1. Initiate Multipart Upload
   → upload_id 발급

2. Upload Parts (병렬)
   ┌─────────────────────────────────────────────────┐
   │  Part 1 (100MB) ───▶  데이터 노드 1             │
   │  Part 2 (100MB) ───▶  데이터 노드 2             │
   │  Part 3 (100MB) ───▶  데이터 노드 3             │
   │  ...                                           │
   │  Part 50 (100MB) ──▶  데이터 노드 N             │
   └─────────────────────────────────────────────────┘

3. Complete Multipart Upload
   → 모든 파트를 하나의 객체로 조합
```

**장점:**
- **병렬 업로드**: 빠른 전송
- **재시도**: 실패한 파트만 재전송
- **대용량 지원**: 최대 5TB 객체

### 가비지 컬렉션

#### 삭제된 데이터 정리

```
가비지 컬렉션 대상:
1. 삭제된 객체의 실제 데이터
2. Multipart Upload 미완료 파트
3. 이전 버전 (보존 기간 초과)

┌─────────────────────────────────────────────────────────────┐
│                    GC 프로세스                               │
│                                                             │
│  1. 메타데이터 DB 스캔 (삭제 표시된 객체)                     │
│                │                                            │
│                ▼                                            │
│  2. 데이터 저장소에서 실제 데이터 삭제                        │
│                │                                            │
│                ▼                                            │
│  3. 메타데이터 DB에서 레코드 삭제                            │
│                                                             │
│  (비피크 시간에 배치로 실행)                                 │
└─────────────────────────────────────────────────────────────┘
```

### 메타데이터 데이터 모델

#### 버킷 테이블

```sql
CREATE TABLE buckets (
    bucket_id UUID PRIMARY KEY,
    bucket_name VARCHAR(255) UNIQUE,
    owner_id UUID,
    is_versioning_enabled BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP
);
```

#### 객체 테이블

```sql
CREATE TABLE objects (
    bucket_id UUID,
    object_key VARCHAR(1024),
    version_id INT,
    object_id UUID,
    size BIGINT,
    etag VARCHAR(64),
    is_deleted BOOLEAN DEFAULT FALSE,
    storage_location TEXT,  -- 데이터 노드 위치
    created_at TIMESTAMP,
    PRIMARY KEY ((bucket_id), object_key, version_id)
);
```

### 데이터 일관성

#### 읽기 후 쓰기 일관성 (Read-After-Write)

```
PUT object_A → 성공 응답
바로 GET object_A → 방금 업로드한 데이터 반환

구현:
- 동기 복제 (최소 쿼럼)
- 메타데이터 커밋 후 응답
```

### 복제

#### 동일 리전 복제

```
데이터 센터 내 복제:

┌─────────────────────────────────────────────────────────────┐
│                     데이터 센터 A                            │
│                                                             │
│  원본 ──복제──▶ 복제본1 ──복제──▶ 복제본2                   │
│                                                             │
│  쿼럼 쓰기: 2/3 노드 성공 시 응답                            │
└─────────────────────────────────────────────────────────────┘
```

#### 교차 리전 복제 (CRR)

```
재해 복구를 위한 리전 간 복제:

┌──────────────┐       비동기 복제       ┌──────────────┐
│  리전 A      │ ──────────────────────▶ │  리전 B      │
│  (원본)      │                         │  (복제본)    │
└──────────────┘                         └──────────────┘
```

## 4단계: 마무리

### 핵심 포인트

1. **분리된 저장**: 메타데이터와 데이터 분리
2. **Multipart Upload**: 대용량 파일 효율적 업로드
3. **버전 관리**: Delete Marker로 삭제 복구
4. **Erasure Coding**: 복제보다 효율적인 내구성
5. **가비지 컬렉션**: 비동기 정리로 성능 유지

### 추가 논의 주제

- **스토리지 클래스**: Standard, Infrequent Access, Glacier
- **수명 주기 정책**: 자동 클래스 전환, 삭제
- **암호화**: 서버 측/클라이언트 측 암호화
- **이벤트 알림**: 객체 생성/삭제 시 Lambda 트리거
