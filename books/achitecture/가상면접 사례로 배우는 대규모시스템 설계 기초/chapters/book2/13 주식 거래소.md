# 13 주식 거래소 (Stock Exchange)

## 개요

증권 거래소의 주문 매칭 시스템 설계. 매수/매도 주문을 받아 실시간으로 매칭하고 체결하는 시스템. 극도로 낮은 지연 시간과 높은 처리량이 핵심.

---

## 1단계: 문제 이해 및 설계 범위 확정

### 기능 요구사항

- **주문 접수**: 매수/매도 주문 접수
- **주문 매칭**: 매수가 ≥ 매도가일 때 체결
- **주문 취소**: 미체결 주문 취소
- **시장 데이터 제공**: 실시간 호가, 체결가 전파
- **주문 유형**: 지정가(Limit), 시장가(Market)

### 비기능 요구사항

- **초저지연**: 밀리초(ms) 또는 마이크로초(μs) 수준
- **높은 처리량**: 초당 수백만 주문 처리
- **정확성**: 주문 순서 보장, 공정한 매칭
- **고가용성**: 99.99% 이상

### 개략적 규모 추정

| 항목 | 수치 |
|------|------|
| 상장 종목 수 | 5,000개 |
| 일간 주문 수 | 수억 건 |
| 피크 TPS | 100,000+ |
| 지연 목표 | < 1ms |

## 3단계: 상세 설계

### 매칭 엔진

#### 핵심 역할

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         매칭 엔진 역할                                   │
│                                                                         │
│  1. 시퀀스 검증: 주문이 올바른 순서로 도착했는지 확인                    │
│                                                                         │
│  2. 주문 매칭: 매수가 ≥ 매도가일 때 체결                                 │
│                                                                         │
│  3. 주문서 관리: 미체결 주문 보관                                        │
│                                                                         │
│  특징:                                                                   │
│  - 결정론적 (Deterministic): 동일 입력 → 동일 출력                      │
│  - 순차 처리 (Sequential): 한 번에 하나씩 처리                          │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 매칭 알고리즘

```
매칭 원칙:

1. 가격 우선 (Price Priority)
   - 매수: 높은 가격 우선
   - 매도: 낮은 가격 우선

2. 시간 우선 (Time Priority)
   - 같은 가격이면 먼저 들어온 주문 우선 (FIFO)

예시:
┌─────────────────────────────────────────────────────────────┐
│  매수 주문서        │        매도 주문서                    │
│  ─────────────────────────────────────────────────────────  │
│  150.50 - 100주 (A) │        150.55 - 200주 (D)            │
│  150.45 - 200주 (B) │        150.60 - 150주 (E)            │
│  150.40 - 300주 (C) │        150.65 - 100주 (F)            │
└─────────────────────────────────────────────────────────────┘

새 매도 주문: 150.45에 150주 매도 → A와 100주 체결, B와 50주 체결
```

#### 매칭 로직

```python
def match_order(incoming_order, order_book):
    if incoming_order.side == 'BUY':
        opposite_book = order_book.asks  # 매도 주문서
        compare = lambda price1, price2: price1 >= price2
    else:
        opposite_book = order_book.bids  # 매수 주문서
        compare = lambda price1, price2: price1 <= price2

    trades = []
    remaining_qty = incoming_order.quantity

    while remaining_qty > 0 and len(opposite_book) > 0:
        best_order = opposite_book.peek()  # 최우선 주문

        # 가격 조건 확인
        if incoming_order.type == 'LIMIT':
            if not compare(incoming_order.price, best_order.price):
                break  # 가격 불일치, 매칭 중단

        # 체결 수량 결정
        trade_qty = min(remaining_qty, best_order.quantity)

        # 체결 기록
        trades.append(Trade(
            buy_order=incoming_order if incoming_order.side == 'BUY' else best_order,
            sell_order=best_order if incoming_order.side == 'BUY' else incoming_order,
            quantity=trade_qty,
            price=best_order.price
        ))

        remaining_qty -= trade_qty
        best_order.quantity -= trade_qty

        if best_order.quantity == 0:
            opposite_book.pop()  # 완전 체결된 주문 제거

    # 미체결 수량이 남으면 주문서에 추가
    if remaining_qty > 0 and incoming_order.type == 'LIMIT':
        incoming_order.quantity = remaining_qty
        order_book.add(incoming_order)

    return trades
```

### 주문서 (Order Book)

#### 자료구조

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      주문서 자료구조                                     │
│                                                                         │
│  종목별로 두 개의 주문서 유지:                                           │
│  - Bids (매수): 높은 가격 우선 정렬                                      │
│  - Asks (매도): 낮은 가격 우선 정렬                                      │
│                                                                         │
│  ┌─────────────────────────────────────────────────────────────┐       │
│  │                    Order Book (AAPL)                         │       │
│  │                                                              │       │
│  │  Bids (매수)                        Asks (매도)             │       │
│  │  ────────────                        ────────────            │       │
│  │  Price Level: 150.50                Price Level: 150.55     │       │
│  │    ├── Order A (100주, 10:00:01)      ├── Order D (200주)   │       │
│  │    └── Order B (50주, 10:00:05)       └── Order E (100주)   │       │
│  │                                                              │       │
│  │  Price Level: 150.45                Price Level: 150.60     │       │
│  │    └── Order C (200주)                └── Order F (150주)   │       │
│  │                                                              │       │
│  └─────────────────────────────────────────────────────────────┘       │
│                                                                         │
│  자료구조:                                                               │
│  - 가격 레벨: 정렬된 맵 (Red-Black Tree / Skip List)                    │
│  - 각 가격의 주문: 이중 연결 리스트 (FIFO)                               │
│  - 주문 ID 인덱스: 해시맵 (O(1) 취소용)                                 │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 시간복잡도

| 연산 | 시간복잡도 |
|------|-----------|
| 주문 추가 | O(log N) |
| 주문 취소 | O(1) - 인덱스 사용 |
| 최우선 주문 조회 | O(1) |
| 매칭 (M개 체결) | O(M) |

#### 구현 예시

```python
from sortedcontainers import SortedDict
from collections import deque

class OrderBook:
    def __init__(self, symbol):
        self.symbol = symbol
        # 매수: 높은 가격 우선 (-price로 정렬)
        self.bids = SortedDict()
        # 매도: 낮은 가격 우선
        self.asks = SortedDict()
        # 주문 ID → 주문 객체 매핑 (O(1) 취소용)
        self.order_index = {}

    def add_order(self, order):
        book = self.bids if order.side == 'BUY' else self.asks
        price_key = -order.price if order.side == 'BUY' else order.price

        if price_key not in book:
            book[price_key] = deque()

        book[price_key].append(order)
        self.order_index[order.order_id] = order

    def cancel_order(self, order_id):
        if order_id not in self.order_index:
            return False

        order = self.order_index[order_id]
        order.quantity = 0  # Lazy deletion
        del self.order_index[order_id]
        return True

    def get_best_bid(self):
        if not self.bids:
            return None
        return self.bids.peekitem(0)[1][0]  # 첫 번째 가격의 첫 번째 주문

    def get_best_ask(self):
        if not self.asks:
            return None
        return self.asks.peekitem(0)[1][0]
```

### 시퀀서 (Sequencer)

#### 역할

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          시퀀서 역할                                     │
│                                                                         │
│  1. 전역 순서 번호 부여                                                  │
│     - 모든 주문, 취소, 체결에 고유 번호 할당                             │
│     - 순서 보장, 공정성 확보                                             │
│                                                                         │
│  2. 메시지 타입별 큐 관리                                                │
│     ┌───────────────┐                                                   │
│     │    시퀀서      │                                                   │
│     └───────┬───────┘                                                   │
│             │                                                           │
│     ┌───────┼───────┬───────────┐                                      │
│     ▼       ▼       ▼           ▼                                      │
│  [주문 큐] [취소 큐] [체결 큐] [시장데이터 큐]                           │
│                                                                         │
│  3. 결정론적 재생 (Deterministic Replay) 지원                           │
│     - 장애 복구 시 순서번호 기반으로 상태 재구성                         │
│     - 동일 순서 → 동일 결과                                              │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 구현

```python
import threading
from dataclasses import dataclass

@dataclass
class SequencedMessage:
    sequence_number: int
    timestamp: int
    message_type: str
    payload: dict

class Sequencer:
    def __init__(self):
        self._sequence = 0
        self._lock = threading.Lock()

    def assign_sequence(self, message_type, payload):
        with self._lock:
            self._sequence += 1
            return SequencedMessage(
                sequence_number=self._sequence,
                timestamp=time.time_ns(),
                message_type=message_type,
                payload=payload
            )

    def get_current_sequence(self):
        return self._sequence
```

#### 링 버퍼 (Ring Buffer)

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         Ring Buffer                                      │
│                                                                         │
│  Lock-Free 자료구조로 고성능 메시지 전달                                 │
│                                                                         │
│      Write Index                                                        │
│           ↓                                                             │
│    ┌───┬───┬───┬───┬───┬───┬───┬───┐                                   │
│    │ 0 │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ 7 │                                   │
```

## 4단계: 마무리

### 핵심 포인트

1. **매칭 엔진**: 가격-시간 우선 원칙, 결정론적 처리
2. **주문서**: 효율적 자료구조 (정렬맵 + 연결리스트 + 해시인덱스)
3. **시퀀서**: 전역 순서 보장, 공정성 확보
4. **저지연 설계**: 단일 서버, 공유 메모리, CPU 고정
5. **고가용성**: Active-Passive + 이벤트 소싱 복구

### 추가 논의 주제

- **서킷 브레이커**: 급격한 가격 변동 시 거래 중단
- **공매도 관리**: 대차 잔고 확인, 업틱 룰
- **고빈도 거래 (HFT)**: 코로케이션, 마이크로초 경쟁
- **다크풀**: 익명 거래소 운영
- **크로스 마켓 주문**: 여러 거래소 동시 주문
