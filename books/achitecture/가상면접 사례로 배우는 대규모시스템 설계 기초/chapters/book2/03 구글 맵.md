# 03 구글 맵 (Google Maps)

## 개요

위치 기반 서비스, 경로 안내, 지도 렌더링 등을 제공하는 구글 맵과 같은 서비스 설계. 2005년 개발 시작, 2021년 기준 10억 DAU, 전 세계 99% 커버리지 달성.

---

## 1단계: 문제 이해 및 설계 범위 확정

### 기능 요구사항

1. ** 사용자 위치 갱신**: 현재 위치 추적 및 기록
2. ** 지도 렌더링**: 경로, 지도 표시
3. ** 경로 안내(내비게이션)**: 출발지에서 목적지까지 최적 경로 계산
4. ** 예상 도착 시간(ETA)**: 실시간 교통 정보 기반 도착 시간 예측

### 비기능 요구사항

- ** 정확성**: 사용자에게 잘못된 경로 안내 X
- ** 낮은 지연**: 부드러운 지도 렌더링, 신속한 경로 계산
- ** 데이터 효율성**: 모바일 데이터/배터리 최소화
- ** 고가용성**: 안정적인 서비스

### 개략적 규모 추정

- 일간 활성 사용자(DAU): 10억 명
- 지도 타일 저장소: 수백 페타바이트
- 도로 데이터: 수십억 개 노드/엣지

### 위치 서비스 (Location Service)

사용자의 위치 갱신 정보를 기록하는 역할.

#### 데이터 흐름

```
모바일 클라이언트 → 위치 서비스 → Kafka → 다양한 소비자
                                    │
                         ┌──────────┼──────────┐
                         ▼          ▼          ▼
                    교통 정보   지도 개선   위치 이력
                      분석      서비스      저장
```

#### 위치 데이터 활용

- ** 새로운 도로/폐쇄 도로 감지**: 많은 사용자가 새 경로로 이동하면 새 도로 추정
- ** 지도 정확도 향상**: 실제 주행 데이터로 도로 데이터 검증
- ** 실시간 교통 정보**: 속도 데이터 수집으로 혼잡도 계산

#### 데이터 모델

```
위치 갱신 메시지:
{
  "user_id": "user_123",
  "latitude": 37.7749,
  "longitude": -122.4194,
  "timestamp": 1699123456,
  "accuracy": 10,
  "speed": 30
}
```

#### 배치 전송

- 클라이언트에서 위치 데이터를 ** 배치** 로 묶어 전송
- 네트워크/배터리 효율성 향상
- Cassandra에 위치 이력 저장 (쓰기 최적화)

---

### 지도 렌더링 (Map Rendering)

세계 지도를 2D 이미지로 투영하여 작은 ** 타일(Tile)** 단위로 분할.

#### 지도 타일 (Map Tiles)

```
줌 레벨별 타일 구조:

줌 0: 전 세계 = 1개 타일
     ┌─────────────┐
     │   세계 전체  │
     └─────────────┘

줌 1: 전 세계 = 4개 타일 (2×2)
     ┌──────┬──────┐
     │  NW  │  NE  │
     ├──────┼──────┤
     │  SW  │  SE  │
     └──────┴──────┘

줌 2: 전 세계 = 16개 타일 (4×4)
     ┌───┬───┬───┬───┐
     │   │   │   │   │
     ├───┼───┼───┼───┤
     │   │   │   │   │
     ├───┼───┼───┼───┤
     │   │   │   │   │
     ├───┼───┼───┼───┤
     │   │   │   │   │
     └───┴───┴───┴───┘

줌 N: 타일 수 = 4^N
```

#### 타일 수 계산

| 줌 레벨 | 타일 수 | 용도 |
|---------|---------|------|
| 0 | 1 | 전 세계 개관 |
| 5 | 1,024 | 대륙 수준 |
| 10 | ~100만 | 도시 수준 |
| 15 | ~10억 | 동네 수준 |
| 21 | ~4.3조 | 건물 수준 |

** 줌 레벨 21 저장소 추정:**
- 4.3조 타일 × 100KB(평균) ≈ **430 페타바이트**

#### 타일 식별자

```
타일 URL 형식: /{줌레벨}/{x}/{y}.png

예시:
- /0/0/0.png (전 세계)
- /10/301/384.png (특정 도시)
- /15/16345/10983.png (특정 블록)
```

#### CDN 활용

```
클라이언트 → CDN (엣지 서버) → S3 (원본 저장소)
                 │
                 └── 캐시 히트 시 즉시 반환
```

- ** 정적 콘텐츠**: 지도 타일은 거의 변경되지 않음
- **CDN 캐싱**: 사용자 근처 엣지 서버에서 타일 제공
- ** 지연 감소**: 원본 저장소 접근 최소화

#### 모바일 최적화

- ** 벡터 타일**: 래스터 이미지 대신 벡터 데이터 전송
- ** 데이터 절감**: 클라이언트에서 렌더링
- ** 오프라인 지도**: 자주 사용하는 지역 미리 다운로드

### 경로 안내 서비스 (Navigation Service)

출발지에서 목적지까지 최적 경로 계산.

#### 구성 요소

```
경로 안내 서비스
       │
       ├── 지오코딩 서비스 (Geocoding)
       │        └── 주소 → 위/경도 변환
       │
       └── 경로 계획 서비스 (Route Planner)
                ├── 최단 경로 계산
                ├── ETA 추정
                └── 경로 순위화
```

#### API 설계

** 경로 요청:**
```
GET /v1/routes?origin=37.7749,-122.4194&destination=37.3382,-121.8863
```

** 응답:**
```json
{
  "routes": [
    {
      "route_id": "route_1",
      "distance_km": 77.5,
      "duration_min": 65,
      "eta": "2024-01-15T15:30:00Z",
      "polyline": "encoded_polyline_string",
      "steps": [...]
    }
  ]
}
```

## 3단계: 상세 설계

### 경로 탐색 알고리즘

#### 그래프 모델링

도로 네트워크를 ** 가중치 방향 그래프** 로 모델링:

```
노드(Node) = 교차로, 도로 끝점
엣지(Edge) = 도로 구간
가중치(Weight) = 이동 시간 또는 거리

      [A] ──(5분)──→ [B]
       │              │
     (3분)          (4분)
       │              │
       ▼              ▼
      [C] ──(2분)──→ [D]
```

#### 다익스트라 알고리즘 (Dijkstra's Algorithm)

- ** 단일 출발점** 최단 경로 알고리즘
- 모든 노드까지의 최단 거리 계산
- 시간 복잡도: O((V + E) log V)

** 한계:**
- 전 세계 그래프에 적용 시 너무 느림
- 수십억 개 노드 탐색 필요

#### A* 알고리즘

- ** 휴리스틱** 기반 탐색으로 효율성 향상
- 목적지 방향으로 우선 탐색
- 휴리스틱: 직선 거리(유클리드 거리)

```
f(n) = g(n) + h(n)

g(n) = 출발점에서 n까지 실제 비용
h(n) = n에서 목적지까지 추정 비용 (휴리스틱)
f(n) = 총 예상 비용
```

** 장점:**
- 다익스트라보다 빠름
- 휴리스틱이 적절하면 최적 경로 보장

### 경로 안내 타일 (Routing Tiles)

전 세계 도로를 한 번에 로드하면 비효율적 → ** 계층적 타일** 구조 사용.

#### 계층 구조

```
┌─────────────────────────────────────────────────────────────┐
│                    고속도로 계층 (Level 3)                   │
│                 도시 간 연결, 장거리 경로                     │
├─────────────────────────────────────────────────────────────┤
│                    간선도로 계층 (Level 2)                   │
│                 구역 간 연결, 중거리 경로                     │
├─────────────────────────────────────────────────────────────┤
│                    지역도로 계층 (Level 1)                   │
│                 동네 내 이동, 단거리 경로                     │
└─────────────────────────────────────────────────────────────┘
```

#### 계층별 라우팅

```
장거리 경로 계산:
1. 출발지 → 가까운 간선도로 (Level 1)
2. 간선도로 → 고속도로 진입 (Level 2)
3. 고속도로 구간 계산 (Level 3)
4. 고속도로 → 도착지 근처 간선도로 (Level 2)
5. 간선도로 → 목적지 (Level 1)

      [출발지]
          │ Level 1 (지역도로)
          ▼
      [간선도로]
          │ Level 2 (간선도로)
          ▼
      [고속도로 IC]
          │ Level 3 (고속도로)
          ▼
      [고속도로 IC]
          │ Level 2 (간선도로)
          ▼
      [도착지 근처]
          │ Level 1 (지역도로)
          ▼
      [목적지]
```

#### 타일 저장

- S3와 같은 ** 객체 저장소** 에 경로 타일 저장
- 지역별, 계층별로 분리
- 필요한 타일만 로드하여 메모리 효율성 확보

### 실시간 교통 정보

#### 데이터 수집

```
사용자 위치 업데이트
        │
        ▼
      Kafka (스트림 처리)
        │
        ├──→ 교통 상황 분석 서비스
        │           │
        │           ▼
        │     엣지 가중치 갱신
        │           │
        │           ▼
        │     라우팅 타일 갱신
        │
        └──→ 이상 탐지 (사고, 도로 폐쇄)
```

#### 교통 상황 계산

```
특정 도로 구간의 속도:
- 해당 구간을 지나는 사용자들의 평균 속도 계산
- 이동 중인 사용자 수로 혼잡도 추정

엣지 가중치 갱신:
- 기본 가중치: 도로 길이 / 제한 속도
- 실시간 가중치: 도로 길이 / 현재 평균 속도
```

#### ETA 계산

```
예상 도착 시간 = 현재 시간 + Σ(각 구간 예상 소요 시간)

각 구간 예상 소요 시간:
- 현재 교통 상황 (실시간 데이터)
- 과거 패턴 (요일/시간대별 평균)
- 기상 조건
- 예정된 이벤트
```

## 4단계: 마무리

### 핵심 포인트

1. ** 지도 타일**: CDN + 객체 저장소로 정적 콘텐츠 효율적 제공
2. ** 계층적 라우팅 타일**: 장거리 경로를 효율적으로 계산
3. ** 실시간 교통**: 사용자 위치 데이터로 교통 상황 파악
4. **A* 알고리즘**: 휴리스틱 기반 효율적 경로 탐색

### 추가 논의 주제

- ** 대중교통 경로**: 버스, 지하철 시간표 통합
- ** 다중 경유지**: 여러 목적지 경로 최적화 (TSP 변형)
- ** 오프라인 지도**: 네트워크 없이 사용 가능한 기능
- ** 실시간 재경로**: 경로 이탈 또는 교통 상황 변경 시 자동 재계산
