# 11 결제 시스템 (Payment System)

## 개요

전자상거래 플랫폼의 결제 시스템 설계. 고객으로부터 대금을 수령하고 판매자에게 정산하는 과정에서 정확성, 신뢰성, 보안이 핵심.

---

## 1단계: 문제 이해 및 설계 범위 확정

### 기능 요구사항

- **대금 수신 (Pay-in)**: 고객으로부터 판매자 대신 결제 수령
- **대금 정산 (Pay-out)**: 판매자에게 판매 대금 송금
- **다양한 결제 수단**: 신용카드, 체크카드, 페이팔 등
- **결제 상태 조회**: 실시간 결제 상태 확인

### 비기능 요구사항

- **신뢰성**: 결제 실패 적절히 처리
- **내결함성**: 장애 상황에서도 데이터 일관성 유지
- **정확성**: 단 1원도 오차 없이 처리
- **보안**: 결제 정보 보호, PCI DSS 준수

### 개략적 규모 추정

| 항목 | 수치 |
|------|------|
| 일간 트랜잭션 | 100만 건 |
| TPS | ~10 (피크 ~50) |
| 정확성 > 처리량 | 금융 시스템 특성 |

### API 설계

#### 결제 실행

```
POST /v1/payments
{
  "payment_order_id": "uuid-12345",  // 멱등성 키
  "buyer_id": "user_123",
  "seller_id": "merchant_456",
  "amount": "10000",                 // 문자열로 저장 (정밀도 보존)
  "currency": "KRW",
  "payment_method": "CREDIT_CARD"
}

Response:
{
  "payment_id": "pay_789",
  "status": "EXECUTING",
  "redirect_url": "https://psp.com/checkout/token_abc"
}
```

#### 결제 상태 조회

```
GET /v1/payments/{payment_id}

Response:
{
  "payment_id": "pay_789",
  "status": "SUCCESS",
  "amount": "10000",
  "currency": "KRW",
  "created_at": "2024-01-15T10:30:00Z"
}
```

**금액을 문자열로 저장하는 이유:**
- float/double 연산 시 정밀도 손실 발생 가능
- 시스템 간 직렬화/역직렬화 과정에서 오차 방지

### 데이터 모델

#### 결제 이벤트 테이블

```sql
CREATE TABLE payment_events (
    checkout_id UUID PRIMARY KEY,
    buyer_id VARCHAR(64),
    is_payment_done BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP
);
```

#### 결제 주문 테이블

```sql
CREATE TABLE payment_orders (
    payment_order_id UUID PRIMARY KEY,
    checkout_id UUID REFERENCES payment_events,
    seller_id VARCHAR(64),
    amount DECIMAL(18, 4),
    currency VARCHAR(3),
    status ENUM('NOT_STARTED', 'EXECUTING', 'SUCCESS', 'FAILED'),
    wallet_updated BOOLEAN DEFAULT FALSE,
    ledger_updated BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP
);
```

**결제 이벤트와 결제 주문 관계:**
```
하나의 결제 이벤트 (checkout_id)
    │
    ├── 결제 주문 1 (판매자 A)
    ├── 결제 주문 2 (판매자 B)
    └── 결제 주문 3 (판매자 C)

예: 장바구니에 여러 판매자 상품이 담긴 경우
```

## 3단계: 상세 설계

### PSP 연동

#### PSP (Payment Service Provider)

대부분의 기업은 PCI DSS 규정 준수 부담으로 카드 정보를 직접 저장하지 않음.

```
PSP 역할:
- 카드사/은행 연동 대행
- 민감 정보(카드번호) 처리
- 다양한 결제 수단 제공
- 사기 탐지

예: Stripe, PayPal, Adyen, Toss Payments
```

#### PSP 연동 흐름 (호스트 결제 페이지)

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      PSP 호스트 결제 페이지 흐름                          │
│                                                                         │
│  1. 클라이언트가 결제 시작 요청                                           │
│  2. 결제 서비스가 PSP에 결제 등록                                         │
│  3. PSP가 결제 토큰(UUID) 반환                                           │
│  4. 토큰을 DB에 저장                                                     │
│  5. 클라이언트에게 PSP 결제 페이지 URL 전달                                │
│                                                                         │
│            ┌───────────────────────────────────────────┐                │
│            │            PSP 결제 페이지                 │                │
│            │                                           │                │
│            │  ┌─────────────────────────────────┐     │                │
│            │  │  카드 번호: ________________    │     │                │
│            │  │  유효기간: ____/____            │     │                │
│            │  │  CVV: ____                      │     │                │
│            │  └─────────────────────────────────┘     │                │
│            │                                           │                │
│            │  * 민감 정보는 PSP에서만 처리              │                │
│            │  * 자사 서버에 카드 정보 전송 안됨         │                │
│            └───────────────────────────────────────────┘                │
│                                                                         │
│  6. 사용자가 PSP 페이지에서 결제 정보 입력                                 │
│  7. PSP가 결제 처리 후 결과 반환                                          │
│  8. 브라우저가 결과 URL로 리다이렉트                                      │
│  9. PSP가 웹훅으로 결제 서비스에 비동기 알림                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**웹훅(Webhook) 중요성:**
- 브라우저 리다이렉트보다 신뢰성 높음
- 네트워크 장애 시에도 재시도
- 결제 상태 최종 확인 수단

### 조정 (Reconciliation)

#### 조정의 필요성

```
문제 상황:
- PSP에서는 결제 성공, 내부 시스템에서는 실패로 기록
- 네트워크 장애로 웹훅 수신 실패
- 비동기 처리 중 데이터 불일치

해결: 정기적 조정 (Reconciliation)
```

### 정확히 한 번 처리 (Exactly-Once)

#### 공식

```
Exactly-Once = At-Least-Once + At-Most-Once

At-Least-Once: 재시도로 최소 한 번 처리 보장
At-Most-Once: 멱등성으로 최대 한 번 처리 보장
```

#### At-Least-Once (재시도 전략)

```
재시도 전략 종류:

1. 즉시 재시도 (Immediate)
   - 일시적 오류에 적합
   - 서버 과부하 가능

2. 고정 간격 (Fixed Interval)
   - 5초마다 재시도
   - 예측 가능

3. 증분 간격 (Incremental)
   - 5초, 10초, 15초...
   - 선형 증가

4. 지수 백오프 (Exponential Backoff) ★ 권장
   - 1초, 2초, 4초, 8초...
   - 네트워크 문제에 효과적
   - Jitter 추가로 동시 재시도 분산

5. 취소 (Cancel)
   - 영구적 오류 시 재시도 중단
```

#### At-Most-Once (멱등성 구현)

```sql
-- 멱등성 키를 unique constraint로 사용
CREATE TABLE payment_orders (
    payment_order_id UUID PRIMARY KEY,
    idempotency_key VARCHAR(64) UNIQUE,
    ...
);

-- 삽입 시도
INSERT INTO payment_orders (payment_order_id, idempotency_key, ...)
VALUES (uuid_generate_v4(), 'client_key_123', ...);

-- 결과:
-- 성공: 첫 요청, 정상 처리
-- 실패 (중복 키): 이전 결과 조회하여 반환
```

### 원장과 복식부기 (Double-Entry Ledger)

#### 복식부기 원칙

```
모든 거래는 두 개의 원장 계좌에 기록:
- 한 계좌: 차변 (Debit)
- 다른 계좌: 대변 (Credit)

합계는 항상 0

예: 고객이 10,000원 결제
┌─────────────────────────────────────────────────────────────┐
│  계좌              │   차변(Debit)   │   대변(Credit)       │
│  ─────────────────────────────────────────────────────────  │
│  고객 계좌          │   -10,000원     │                      │
│  판매자 계좌        │                 │   +10,000원          │
│  ─────────────────────────────────────────────────────────  │
│  합계              │   -10,000원     │   +10,000원 = 0      │
└─────────────────────────────────────────────────────────────┘
```

#### 원장 테이블 설계

```sql
CREATE TABLE ledger_entries (
    entry_id UUID PRIMARY KEY,
    payment_id UUID,
    account_id VARCHAR(64),
    entry_type ENUM('DEBIT', 'CREDIT'),
    amount DECIMAL(18, 4),
    currency VARCHAR(3),
    created_at TIMESTAMP
);

-- 잔액 조회
SELECT
    account_id,
    SUM(CASE WHEN entry_type = 'CREDIT' THEN amount ELSE 0 END) -
    SUM(CASE WHEN entry_type = 'DEBIT' THEN amount ELSE 0 END) AS balance
FROM ledger_entries
GROUP BY account_id;
```

### 결제 보안

#### 보안 요소

## 4단계: 마무리

### 핵심 포인트

1. **멱등성**: 중복 결제 방지의 핵심
2. **PSP 연동**: 민감 정보는 PSP에서 처리
3. **복식부기**: 모든 거래의 합은 0
4. **조정**: 비동기 시스템의 최종 안전장치
5. **Exactly-Once**: 재시도 + 멱등성 조합

### 추가 논의 주제

- **다중 통화 지원**: 환율 처리, 통화 변환
- **구독 결제**: 정기 결제 시스템
- **분할 결제**: 여러 결제 수단 조합
- **환불 처리**: 부분/전체 환불 흐름
- **분산 트랜잭션**: Saga 패턴 적용
