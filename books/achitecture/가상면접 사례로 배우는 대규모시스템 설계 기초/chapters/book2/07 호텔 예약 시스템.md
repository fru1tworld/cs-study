# 07 호텔 예약 시스템 (Hotel Reservation System)

## 개요

호텔 객실 검색, 예약, 결제를 처리하는 예약 시스템 설계. Booking.com, Airbnb, Expedia 등이 대표적인 예시. 동시성 문제 해결이 핵심 과제.

---

## 1단계: 문제 이해 및 설계 범위 확정

### 기능 요구사항

- 호텔/객실 정보 조회
- 특정 날짜의 객실 가용성 확인
- 객실 예약 (결제 포함)
- 예약 취소
- **10% 오버부킹** 지원 (일부 취소 예상)

### 비기능 요구사항

- ** 높은 동시성**: 성수기에 많은 사용자가 동시 예약
- ** 적절한 지연**: 예약 확인까지 수 초 이내
- ** 데이터 정확성**: 이중 예약 방지

### 개략적 규모 추정

| 항목 | 수치 |
|------|------|
| 호텔 수 | 5,000개 |
| 객실 수 | 100만 개 |
| 일간 예약 수 | 수십만 건 |
| 예약 QPS | ~100 (피크 ~1,000) |

** 특징**: QPS는 높지 않지만 ** 동시성 처리** 가 핵심

#### 예약 테이블

```sql
CREATE TABLE reservations (
    reservation_id VARCHAR(64) PRIMARY KEY,

```

```sql
CREATE TABLE room_inventory (

    PRIMARY KEY (hotel_id, room_type_id, date)

```

## 3단계: 상세 설계

### 개선된 데이터 모델

재고 확인과 예약을 원자적으로 처리하기 위한 설계:

```sql
-- 가용 객실 = total_rooms * (1 + overbooking_pct) - total_reserved

예시:
total_rooms = 100
overbooking_pct = 0.1 (10%)
total_reserved = 105

가용 객실 = 100 * 1.1 - 105 = 5
```

### 동시성 문제

#### 문제 상황

```
시나리오: 1개 객실 남음, 2명이 동시 예약 시도

User A                          User B
   │                               │
   │  SELECT total_reserved = 99   │
   │  (가용: 1개)                   │
   │                               │  SELECT total_reserved = 99
   │                               │  (가용: 1개)
   │                               │
   │  UPDATE total_reserved = 100  │
   │                               │  UPDATE total_reserved = 100
   │                               │
   ▼                               ▼
결과: 둘 다 예약 성공 → 이중 예약!
```

### 해결 방안 비교

#### 1. 비관적 락 (Pessimistic Locking)

```sql
-- Transaction 1
BEGIN;
SELECT * FROM room_inventory
WHERE hotel_id = 'H1' AND room_type_id = 'R1' AND date = '2024-01-15'
FOR UPDATE;  -- 행 잠금

-- 예약 처리...
UPDATE room_inventory SET total_reserved = total_reserved + 1 WHERE ...;
COMMIT;
```

```
User A                          User B
   │                               │
   │  SELECT ... FOR UPDATE        │
   │  (락 획득)                    │
   │                               │  SELECT ... FOR UPDATE
   │                               │  (대기...)
   │  UPDATE & COMMIT              │
   │  (락 해제)                    │
   │                               │  (락 획득)
   │                               │  SELECT → 가용 0개
   │                               │  예약 실패
```

** 장점**: 구현 간단, 정확성 보장
** 단점**: 성능 저하, 데드락 가능, 확장성 제한

#### 2. 낙관적 락 (Optimistic Locking)

```sql
-- 버전 컬럼 추가
ALTER TABLE room_inventory ADD COLUMN version INT DEFAULT 0;

-- 읽기
SELECT total_reserved, version FROM room_inventory WHERE ...;
-- 결과: total_reserved = 99, version = 5

-- 업데이트 (버전 체크 포함)
UPDATE room_inventory
SET total_reserved = 100, version = version + 1
WHERE hotel_id = 'H1' AND room_type_id = 'R1' AND date = '2024-01-15'
  AND version = 5;  -- 버전 확인

-- affected_rows = 0 이면 재시도
```

```
User A                          User B
   │                               │
   │  SELECT ver=5, reserved=99    │
   │                               │  SELECT ver=5, reserved=99
   │                               │
   │  UPDATE WHERE ver=5           │
   │  (성공, ver=6)                │
   │                               │  UPDATE WHERE ver=5
   │                               │  (실패! ver != 5)
   │                               │  → 재시도
```

** 장점**: 락 없이 높은 동시성
** 단점**: 충돌 시 재시도 필요, 높은 동시성에서 재시도 폭증

#### 3. 데이터베이스 제약 조건

```sql
-- 체크 제약 조건
ALTER TABLE room_inventory
ADD CONSTRAINT check_inventory
CHECK (total_reserved <= total_rooms * 1.1);  -- 10% 오버부킹

-- 또는 직접 조건부 업데이트
UPDATE room_inventory
SET total_reserved = total_reserved + 1
WHERE hotel_id = 'H1'
  AND room_type_id = 'R1'
  AND date = '2024-01-15'
  AND total_reserved < total_rooms * 1.1;

-- affected_rows = 0 이면 재고 없음
```

** 장점**: 애플리케이션 로직 간소화, DB 수준 보장
** 단점**: DB 의존성, 복잡한 비즈니스 로직 처리 어려움

### 권장 접근법

| 상황 | 권장 방식 |
|------|----------|
| 낮은 경쟁 | 낙관적 락 |
| 높은 경쟁 (인기 객실) | 비관적 락 또는 DB 제약 |
| 일반적 시스템 | **DB 제약 조건 + 낙관적 락** |

### 확장성

#### 데이터베이스 샤딩

```
샤딩 키: hotel_id

┌─────────────────────────────────────────────────────────────┐
│                      Shard Router                            │
└──────────────────────────┬──────────────────────────────────┘
                           │
        ┌──────────────────┼──────────────────┐
        ▼                  ▼                  ▼
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   Shard 1   │    │   Shard 2   │    │   Shard 3   │
│  (hotel A~H)│    │  (hotel I~P)│    │  (hotel Q~Z)│
└─────────────┘    └─────────────┘    └─────────────┘
```

**hotel_id 기준 샤딩 이점:**
- 같은 호텔의 예약은 같은 샤드에서 처리
- 크로스 샤드 트랜잭션 최소화

#### 캐싱

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   클라이언트     │ ──▶ │   Redis Cache   │ ──▶ │   Database      │
└─────────────────┘     │  (재고 정보)     │     └─────────────────┘
                        └─────────────────┘
                              │
                              ▼
                    캐시 무효화 전략:
                    - Write-through
                    - TTL (짧게 설정)
```

** 주의**: 예약은 캐시 사용 X, 항상 DB에서 처리

### 데이터 일관성

#### 분산 트랜잭션 문제

예약 + 결제를 원자적으로 처리해야 함:

```
예약 서비스                     결제 서비스
     │                              │
     │  1. 예약 생성 (pending)       │
     │                              │
     │  2. ──────────────────────▶  │  결제 요청
     │                              │
     │                              │  3. 결제 처리
     │                              │
     │  4. ◀──────────────────────  │  결제 결과
     │                              │
     │  5. 예약 확정 or 취소         │
```

** 해결 방안:**
1. ** 사가 패턴 (Saga)**: 보상 트랜잭션으로 롤백
2. **2PC**: 분산 트랜잭션 (성능 저하)
3. ** 이벤트 기반**: 최종 일관성

## 4단계: 마무리

### 핵심 포인트

1. ** 동시성 제어**: 낙관적 락 + DB 제약 조건 조합
2. ** 오버부킹**: 비즈니스 요구사항 반영
3. ** 샤딩**: hotel_id 기준으로 크로스 샤드 트랜잭션 최소화
4. ** 분산 트랜잭션**: 사가 패턴으로 최종 일관성 확보
