# 13 스레드 안전성과 락 최적화

## 13.1 들어가며

동시성 프로그래밍에서 스레드 안전성은 핵심 개념이다. 이 장에서는 스레드 안전성의 정의와 구현 방법, 그리고 JVM이 제공하는 다양한 락 최적화 기법을 살펴본다.

## 13.2 스레드 안전성

스레드 안전성이라는 용어가 있다.
스레드 안전성을 브라이언 게츠가 다음과 같이 정의하였다.

여러 스레드가 한 객체에 동시에 접근할 때 어떤 런타임 환경이든 다음 두 조건을 모두 충족하면 객체를 호출하는 행위가 올바른 결과를 얻을 수 있다면 그 객체는 스레드 안전하다

- 특별한 스레드 스케줄링이나 대체 실행 수단을 고려할 필요 없다.
- 추가적인 동기화 수단이나 호출자 측에서 조율이 필요없다.

한편 많은 시나리오에서는 많이 누그러진 정의를 사용한다.
객체를 호출하는 행위를 단일 호출로 제한한다.

### 13.2.1 자바 언어의 스레드 안전성

자바에서 스레드 안전성을 제대로 이해하려면 안전하냐 아니냐라는 이분법적 사고에서 벗어나야 한다.

스레드 안전성은 안전함의 정도에 따라 여러 단계로 분류된다.

**1. 불변 (Immutable)**
- 변하지 않는다는 뜻이다. final 키워드는 불변이므로 안전하다.
- String의 경우 불변이다. 그 외에도 열거 타입, Long, Double과 같은 타입은 불변이다.
- 단 Number의 AtomicInteger와 AtomicLong은 불변이 아니다.
- 불변 객체는 본질적으로 스레드 안전하다.

**2. 절대적 스레드 안전 (Absolute Thread Safety)**
- 절대적 스레드 안전은 스레드 안전성 정의를 완벽하게 충족한다.
- "어떤 런타임 환경에서든 호출자가 추가적인 동기화 조치를 할 필요가 없다"는 조건은 사실 비용이 많이 들거나 비현실적일 수 있다.
- 그래서 스레드 안전하다고 표시된 클래스 대부분이 절대적 스레드 안전을 의미하지 않는다.
- synchronized가 붙어 있어도 호출자가 추가로 동기화할 필요가 없다는 뜻은 아니다.

**3. 조건부 스레드 안전 (Conditionally Thread-Safe)**
- 흔히 스레드 안전하다고 말할 때의 안전 수준을 말한다.
- 안전한 객체는 단일한 작업을 별도 보호 조치 없이 스레드로부터 안전하게 수행한다.
- 하지만 특정 순서로 연달아 호출하는 경우는 추가로 동기화해야 할 수 있다.
- 예: Vector, Hashtable

**4. 스레드 호환 (Thread-Compatible)**
- 호출자가 적절히 조치하면 멀티스레드에서 안전할 수 있다.
- 예: ArrayList, HashMap

**5. 스레드 적대적 (Thread-Hostile)**
- Thread 클래스의 suspend와 resume 메서드는 한 스레드 객체를 공유하면서 동시에 suspend와 resume를 호출했을 때 교착에 빠질 수 있다.
- 이런 이유로 두 메서드는 폐기 대상으로 지정됐다.

### 13.2.2 스레드 안전성 구현 방법

**1. 상호 배제 동기화 (Mutual Exclusion Synchronization)**

가장 일반적인 동시성 보장 수단이다. 임계 영역(critical section)을 보호하여 한 번에 하나의 스레드만 접근하도록 한다.

```java
// synchronized 키워드 사용
public synchronized void method() {
    // 임계 영역
}

// synchronized 블록 사용
public void method() {
    synchronized(this) {
        // 임계 영역
    }
}
```

**synchronized의 특징:**
- 자바에서는 단일 연산 특성을 보장하기 위해 synchronized 구문을 사용한다.
- 락은 스레드가 synchronized 블록에 들어가기 전에 자동 확보되며, 해당 블록을 벗어날 때 자동으로 해제된다.
- 자바에서 암묵적인 락은 뮤텍스(Mutex)로 동작한다.
- synchronized 블록은 객체 단위로 락을 다룬다.

**ReentrantLock 사용:**

```java
ReentrantLock lock = new ReentrantLock();

public void method() {
    lock.lock();
    try {
        // 임계 영역
    } finally {
        lock.unlock();
    }
}
```

**2. 비차단 동기화 (Non-blocking Synchronization)**

낙관적 락킹 방식으로 CAS(Compare-And-Swap) 연산을 사용한다.

```java
// AtomicInteger 사용 예시
private AtomicInteger count = new AtomicInteger(0);

public void increment() {
    count.incrementAndGet();  // CAS 기반 원자적 연산
}
```

**CAS 연산:**
- 비관적 락처럼 모든 스레드의 접근을 차단하기보다 경쟁이 발생할 것 같은 스레드만 비교하고 차단한다.
- 메모리 위치의 현재 값을 예상 값과 비교하여, 일치하면 새 값으로 교체한다.
- 일치하지 않으면 실패하고 재시도한다.

**3. 동기화 없는 방식**

- **스레드 로컬 저장소 (ThreadLocal)**: 각 스레드가 자신만의 변수 복사본을 가짐
- **불변 객체**: 객체가 변경 불가능하면 동기화가 필요 없음

## 13.3 락 최적화

JVM은 synchronized의 성능을 향상시키기 위해 다양한 최적화 기법을 적용한다.

### 13.3.1 스핀 락과 적응형 스핀

**스핀 락 (Spin Lock):**

스핀락은 임계 구역(critical section)에 진입이 불가능할 때 진입이 가능할 때까지 루프를 돌면서 재시도하는 방식으로 구현된 락이다. 스핀락이라는 이름은 락을 획득할 때까지 해당 스레드가 빙빙 돌고 있다(spinning)는 것을 의미한다.

```java
// 스핀락 개념 (실제 구현과 다름)
while (!lock.tryLock()) {
    // 락을 얻을 때까지 반복
}
try {
    // 임계 영역
} finally {
    lock.unlock();
}
```

**스핀락의 장점:**
- 운영체제의 스케줄링 지원을 받지 않기 때문에, 해당 스레드에 대한 문맥 교환이 일어나지 않는다.
- 임계 구역에 짧은 시간 안에 진입할 수 있는 경우에 문맥 교환을 제거할 수 있어 효율적이다.

**스핀락의 단점:**
- 스핀락이 오랜 시간을 소요하면 다른 스레드를 실행하지 못하고 대기하게 되어 CPU 자원 낭비가 심해진다.

**적응형 스핀 (Adaptive Spinning):**
- JDK 6부터 도입
- 같은 락에 대한 이전 스핀 성공/실패 이력을 기반으로 스핀 시간을 동적으로 조절
- 스핀이 자주 성공하면 스핀 시간을 늘리고, 실패하면 스핀을 포기하고 바로 블로킹

### 13.3.2 락 제거 (Lock Elimination)

JIT 컴파일러가 탈출 분석(Escape Analysis)을 통해 락이 불필요하다고 판단되면 락을 제거한다.

```java
// 컴파일러가 락 제거 가능
public String concat(String s1, String s2, String s3) {
    StringBuffer sb = new StringBuffer();  // sb는 메서드 내에서만 사용
    sb.append(s1);
    sb.append(s2);
    sb.append(s3);
    return sb.toString();
}
```

StringBuffer는 동기화되어 있지만, sb 객체가 메서드를 벗어나지 않으므로 락이 제거될 수 있다.

### 13.3.3 락 범위 확장 (Lock Coarsening)

연속된 동기화 블록이 같은 락을 사용하면 하나의 큰 블록으로 합친다.

```java
// 락 확장 전
synchronized(lock) { a(); }
synchronized(lock) { b(); }
synchronized(lock) { c(); }

// 락 확장 후
synchronized(lock) {
    a();
    b();
    c();
}
```

**주의사항:**
- 락으로 보호한 부분은 동시에 한 스레드에 의해서만 실행된다.
- 큰 단위로 락을 만들 경우 안전성을 확보할 수 있지만 성능이 저하된다.
- 동기화 블록의 범위를 최소화해야 한다.

### 13.3.4 경량 락 (Lightweight Lock)

경합이 없는 상황에서 OS 수준의 무거운 락 대신 CAS 연산을 사용하여 동기화한다.

**동작 방식:**

1. 객체가 잠기지 않은 상태에서 스레드가 synchronized 블록에 진입하면 경량 락을 시도
2. 스레드는 자신의 스택에 락 레코드(Lock Record)를 생성
3. CAS 연산으로 객체 헤더의 Mark Word를 락 레코드를 가리키는 포인터로 변경
4. 성공하면 경량 락 획득, 실패하면 락 확장(중량 락으로 업그레이드)

**MarkWord 구조:**
- 객체 자체의 런타임 데이터(MarkWord)는 hashCode, GC 세대 나이, 락 유형 플래그, 편향 락 스레드 ID, 스레드 LockRecord에 대한 CAS 락 포인터 등의 정보를 저장한다.
- MarkWord의 가장 낮은 3비트는 락 상태를 나타낸다.

**락 상태:**
- 01: 잠금 해제 또는 편향 락
- 00: 경량 락
- 10: 중량 락
- 11: GC 마킹

### 13.3.5 편향 락 (Biased Locking)

편향 락은 HotSpot 가상 머신에서 경쟁이 없는 락킹의 오버헤드를 줄이기 위한 최적화 기술이다. 단일 스레드가 반복적으로 락을 획득하는 경우를 최적화한다.

**동작 방식:**

1. 객체가 처음 잠기면 해당 스레드의 ID를 MarkWord에 기록
2. 같은 스레드가 다시 락을 획득하면 CAS 없이 바로 진입
3. 다른 스레드가 락을 경쟁하면 편향을 철회하고 경량 락으로 업그레이드

```
스레드가 편향락 상태에 있을 때, MarkWord는 현재 스레드의 ID를 기록한다.
다음 스레드가 편향락을 경쟁할 때, 시스템은 먼저 MarkWord에 저장된 스레드 ID가
이 스레드의 ID와 일치하는지 확인한다. 일치하지 않으면 시스템은 즉시 편향락을
철회하고 경량락으로 업그레이드한다.
```

**편향 락의 현재 상태 (JDK 15 이후):**
- JEP 374에 따라 편향 락은 더 이상 기본적으로 활성화되지 않는다.
- `-XX:+UseBiasedLocking` 옵션으로 활성화 가능
- 과거에 보였던 성능 향상은 오늘날에는 훨씬 덜 명확하다.
- 편향 락의 혜택을 받은 많은 애플리케이션은 모든 접근에서 동기화하는 초기 Java 컬렉션 API(예: Hashtable, Vector)를 사용하는 오래된 레거시 애플리케이션이다.
- 최신 애플리케이션은 일반적으로 비동기화 컬렉션(예: HashMap, ArrayList)이나 더 성능이 좋은 동시성 데이터 구조를 사용한다.

### 13.3.6 중량 락 (Heavyweight Lock)

중량 락은 일반적으로 synchronized의 객체 락이라고 하며, 운영체제의 뮤텍스를 사용한다.

**특징:**
- 락 식별 비트는 10
- 포인터는 모니터 객체(Monitor)의 시작 주소를 가리킴
- 스레드 블로킹/언블로킹에 운영체제 개입 필요
- 컨텍스트 스위칭 비용 발생

### 13.3.7 락 업그레이드 순서

synchronized 락의 업그레이드 순서:

```
편향 락 (Biased Lock) → 경량 락 (Lightweight Lock) → 중량 락 (Heavyweight Lock)
```

**업그레이드 과정:**
1. 처음 락 획득 시 → 편향 락
2. 다른 스레드가 경쟁 → 경량 락으로 업그레이드
3. CAS 스핀이 일정 횟수 실패 → 중량 락으로 업그레이드

**참고:** 락은 업그레이드만 가능하고 다운그레이드는 불가능하다.

## 13.4 java.util.concurrent

저수준의 스레드 문제를 직접 처리하는 것보다 java.util.concurrent 패키지에서 추상화된 동시 프로그래밍 지원 기능을 골라 쓰는 것이 좋다.

**주요 클래스:**

**1. 원자적 클래스:**
- AtomicInteger, AtomicLong, AtomicReference 등
- CAS 기반의 락프리 연산

**2. 락:**
- ReentrantLock: 재진입 가능한 락
- ReadWriteLock: 읽기-쓰기 락
- StampedLock: 낙관적 읽기 지원

**3. 동시성 컬렉션:**
- ConcurrentHashMap: 세그먼트 단위 락킹 (Java 8부터는 노드 단위)
- CopyOnWriteArrayList: 쓰기 시 복사
- BlockingQueue: 생산자-소비자 패턴

**4. 동기화 도구:**
- CountDownLatch: 카운트다운 래치
- CyclicBarrier: 사이클릭 배리어
- Semaphore: 세마포어

**ConcurrentHashMap 예시:**

```java
// ConcurrentHashMap의 put 메소드는 해시 충돌이 발생했을 때
// 해시 충돌 발생 노드에 대해서만 lock을 획득한다.
// 다른 노드들에 접근하는 스레드들은 blocking이 발생하지 않으므로
// 성능 이슈가 개선된다.
ConcurrentHashMap<String, String> map = new ConcurrentHashMap<>();
map.put("key", "value");
```

## 13.5 마치며

스레드 안전성은 동시성 프로그래밍의 핵심이다. 불변 객체를 우선 고려하고, 필요한 경우에만 동기화를 사용하며, JVM이 제공하는 락 최적화를 이해하면 효율적인 동시성 코드를 작성할 수 있다.

**핵심 정리:**
1. 스레드 안전성은 이분법이 아닌 스펙트럼으로 이해하라
2. 불변 객체를 우선 고려하라
3. synchronized보다 java.util.concurrent를 활용하라
4. 락의 범위를 최소화하라
5. JVM의 락 최적화를 이해하고 활용하라
