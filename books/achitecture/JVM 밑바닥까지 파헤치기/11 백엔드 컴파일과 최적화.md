# 11 백엔드 컴파일과 최적화

## 11.1 들어가며

자바 프로그램은 처음에 인터프리터에 의해 바이트코드 형태로 실행된다. 하지만 자주 실행되는 코드(핫스팟)가 발견되면 JIT 컴파일러가 이를 네이티브 기계어로 컴파일하여 성능을 향상시킨다. 이 장에서는 JIT 컴파일러의 동작 원리와 다양한 최적화 기법을 살펴본다.

## 11.2 JIT 컴파일러

핫스팟 가상 머신은 인터프리터로 실행해 해석하고, 그 다음 자주 실행되는 메서드나 코드 블록이 발견되면 해당 코드를 네이티브 코드로 컴파일하고 다양한 최적화를 통해 효율을 높인다.

### 11.2.1 인터프리터와 컴파일러

핫스팟 가상 머신은 인터프리터와 컴파일러를 함께 사용하는 혼합 모드를 채택했다.

**인터프리터의 장점:**
- 프로그램 시작 시간이 빠르다
- 메모리 사용량이 적다
- 디버깅과 프로파일링에 유리하다

**컴파일러의 장점:**
- 실행 속도가 빠르다
- 다양한 최적화를 적용할 수 있다

**핫스팟의 컴파일러:**

1. **C1 컴파일러 (Client Compiler)**
   - 빠른 컴파일 속도
   - 간단한 최적화만 수행
   - 프로그램 시작 시간 최소화에 집중

2. **C2 컴파일러 (Server Compiler)**
   - 고급 최적화 기법 적용
   - 컴파일 시간은 길지만 더 빠른 코드 생성
   - 전체적인 성능 최적화에 집중

3. **Graal 컴파일러**
   - Java로 작성된 새로운 세대의 컴파일러
   - JDK 10에서 실험적 기능으로 추가 (JEP 317)
   - 플러그인 가능하고 사용자 확장 가능

### 11.2.2 계층적 컴파일 (Tiered Compilation)

JDK 7부터 도입된 계층적 컴파일은 C1과 C2 컴파일러의 장점을 결합한다:

**컴파일 계층:**
- **레벨 0**: 인터프리터 실행 (프로파일링 정보 수집)
- **레벨 1**: C1 컴파일 (프로파일링 없음)
- **레벨 2**: C1 컴파일 (제한된 프로파일링)
- **레벨 3**: C1 컴파일 (전체 프로파일링)
- **레벨 4**: C2 컴파일 (최대 최적화)

코드는 먼저 C1 컴파일러로 빠르게 컴파일되고, 많이 사용되면 역최적화 후 C2 컴파일러로 재컴파일된다.

### 11.2.3 컴파일 대상과 촉발 조건

JIT 컴파일의 대상이 되는 핫 코드는 두 가지 유형이 있다:

1. 여러번 호출되는 메서드
2. 여러번 실행되는 순환문의 본문

1번이 가장 직관적인데 2는 한 번만 실행되어도 핫 코드일 수 있다.
그런데 2 유형 모두 컴파일 대상이 본문이 아니라 메서드 전체이다.

**핫스팟 감지 방식:**

핫스팟이라는 이름은 코드 컴파일에 대한 접근법에서 유래했다. 일반적인 프로그램은 전체 코드 중 일부만 자주 실행되며, 애플리케이션의 성능은 이 일부가 얼마나 빠르게 실행되는가에 의해 좌우된다. 이 중요한 영역을 애플리케이션의 핫스팟이라고 한다.

여러번의 의미는 구체적으로 무엇일까?
메서드 카운터 호출을 증가시키고 임계값을 넘으면 컴파일러에 컴파일을 요청한다.
그런데 절대 횟수가 아닌 단위 시간당 호출 횟수를 계산한다.
만약 임계값을 넘지 못하면 값을 반으로 줄인다.

이 방식을 카운터 감쇠 메서드 호출이라 한다.

기본 임계값은 참고로 10000(클라이언트 모드) 또는 10000(서버 모드)이다.

**온-스택 치환 (On-Stack Replacement, OSR):**

컴파일을 일으킬 정도로 호출 빈도가 높지는 않지만 메서드 내부에 핫 루프가 포함된 경우가 있다. 대표적인 예시로 main() 메서드가 있다. 핫스팟은 이런 코드를 온-스택 치환(OSR)을 이용해서 최적화한다. 인터프리터가 루프 백 브랜치 횟수를 세어보고 특정 한계치를 초과하면 루프를 컴파일한 후 치환해서 실행한다.

한편 클라이언트 컴파일러는 간단한 컴파일만해서 빠르지만 서버 컴파일러는 최적화를 적용해서 더 느리다.
그래서 우선 클라이언트 컴파일러를 실행하고 백그라운드로 서버 컴파일러를 수행한다.

## 11.3 컴파일 최적화 기법

JIT 컴파일러는 다양한 최적화 기법을 적용하여 코드 성능을 향상시킨다.

### 11.3.1 인라이닝 (Inlining)

인라이닝은 JIT 컴파일러가 가장 먼저 적용하는 최적화로, 관문 최적화라고도 한다.

**인라이닝이란?**
메서드 호출을 해당 메서드의 본문 코드로 대체하는 것이다.

**인라이닝의 장점:**
- 메서드 호출 오버헤드 제거
- 다른 최적화의 범위를 확장
- 탈출 분석, 죽은 코드 제거, 루프 펼치기, 락 생략 등의 최적화가 가능해짐

**인라이닝 결정 요소:**
- 메서드 호출 빈도
- 메서드 크기 (기본적으로 35바이트 이하)
- 가상 메서드 여부

```
// 인라이닝 전
int result = add(a, b);

// 인라이닝 후
int result = a + b;
```

**관련 JVM 옵션:**
- `-XX:-Inline`: 인라이닝 비활성화 (기본적으로 활성화)
- `-XX:MaxInlineSize=35`: 인라이닝할 최대 메서드 크기

### 11.3.2 탈출 분석 (Escape Analysis)

탈출 분석은 객체의 스코프를 분석해서 특정 스코프를 벗어날 수 있는지 여부를 판단하는 기법이다.

**탈출 상태:**
1. **NoEscape**: 객체가 메서드나 스레드를 벗어나지 않음
2. **ArgEscape**: 객체가 메서드의 인자로 전달되지만 스레드를 벗어나지 않음
3. **GlobalEscape**: 객체가 메서드나 스레드를 벗어남

**탈출 분석을 통한 최적화:**

1. **스택 할당 (Stack Allocation)**
   - 힙이 아닌 스택에 객체 할당
   - GC 부하 감소

2. **스칼라 치환 (Scalar Replacement)**
   - 객체 필드를 지역 변수로 분해
   - 객체 자체를 제거

3. **동기화 제거 (Lock Elision)**
   - 비탈출 객체의 동기화 락 제거

**관련 JVM 옵션:**
- `-XX:+DoEscapeAnalysis`: 탈출 분석 활성화 (기본값)
- `-XX:+EliminateLocks`: 락 제거 활성화
- `-XX:+EliminateAllocations`: 스칼라 치환 활성화

### 11.3.3 루프 최적화

**루프 언롤링 (Loop Unrolling):**

백브랜치(back branch)가 일어나면 CPU는 명령어 파이프라인을 덤프하기 때문에 성능상 바람직하지 않다. 루프 바디가 짧을수록 백 브랜치 비용은 상대적으로 높기 때문에 이를 기준으로 루프 펼치기 여부를 결정한다.

```java
// 언롤링 전
for (int i = 0; i < 4; i++) {
    sum += array[i];
}

// 언롤링 후
sum += array[0];
sum += array[1];
sum += array[2];
sum += array[3];
```

**루프 불변 코드 이동 (Loop Invariant Code Motion):**
루프 내에서 변하지 않는 코드를 루프 밖으로 이동한다.

**루프 피일링 (Loop Peeling):**
루프의 첫 번째 또는 마지막 반복을 분리하여 최적화한다.

### 11.3.4 락 최적화

**락 제거 (Lock Elision):**
탈출 분석을 통해 락이 걸린 객체가 로컬 범위를 벗어나지 않는다는 것이 증명되면 해당 락을 제거한다.

**락 확장 (Lock Coarsening):**
같은 락을 공유한 연속된 락이 걸린 영역을 하나로 병합한다.

```java
// 락 확장 전
synchronized(obj) { a(); }
synchronized(obj) { b(); }

// 락 확장 후
synchronized(obj) {
    a();
    b();
}
```

**중첩 락 (Nested Lock):**
락을 해제하지 않고 같은 락을 반복 획득한 블록을 찾아낸다.

### 11.3.5 기타 최적화 기법

**죽은 코드 제거 (Dead Code Elimination):**
실행되지 않는 코드나 결과가 사용되지 않는 코드를 제거한다.

**상수 전파 (Constant Propagation):**
컴파일 시점에 알 수 있는 상수를 미리 계산한다.

**공통 부분식 제거 (Common Subexpression Elimination):**
동일한 계산을 여러 번 수행하는 경우 한 번만 계산하고 결과를 재사용한다.

**널 체크 제거 (Null Check Elimination):**
불필요한 널 체크를 제거한다.

**가상 호출 최적화 (Virtual Call Optimization):**
- **인라인 캐시**: 이전 호출 대상을 캐시
- **바이모픽 인라인 캐시**: 두 가지 호출 대상까지 캐시
- **탈가상화**: 단일 구현체만 있는 경우 직접 호출로 변환

## 11.4 AOT 컴파일러

AOT 컴파일러는 프로그램을 실행하기 전에 바이트코드를 기계어로 변환한다.

### 11.4.1 AOT 컴파일러의 장점과 단점

AOT는 프로그램 실행 전에 네이티브 코드로 컴파일하는 단계 혹은 원래 JIT 컴파일러가 런타임에 수행해야하는 작업을 미리 수행해 캐시에 저장해두고 다음번 실행 시 사용하는 형태다.

**장점:**
- 애플리케이션 시작 시간 단축
- 예측 가능한 성능
- 메모리 사용량 감소
- 워밍업 시간 불필요

**단점:**
- 런타임 프로파일링 정보를 활용할 수 없음
- JIT의 적응형 최적화 불가
- 플랫폼 종속적
- 리플렉션, 동적 프록시 등 동적 기능 제한

### 11.4.2 GraalVM Native Image

GraalVM의 Native Image는 AOT 컴파일의 대표적인 구현체이다.

**특징:**
- 자바 애플리케이션을 독립 실행형 네이티브 바이너리로 컴파일
- 밀리초 단위의 시작 시간
- 낮은 메모리 사용량
- 클라우드 네이티브 환경에 적합

### 11.4.3 JIT vs AOT

| 특성 | JIT | AOT |
|------|-----|-----|
| 시작 시간 | 느림 | 빠름 |
| 최고 성능 | 높음 | 중간 |
| 메모리 사용 | 높음 | 낮음 |
| 적응형 최적화 | 가능 | 불가능 |
| 배포 크기 | 작음 | 큼 |

## 11.5 컴파일러 튜닝

### 11.5.1 주요 JVM 옵션

**컴파일러 선택:**
- `-client`: C1 컴파일러 사용
- `-server`: C2 컴파일러 사용
- `-XX:+TieredCompilation`: 계층적 컴파일 사용 (기본값)

**컴파일 임계값:**
- `-XX:CompileThreshold=10000`: 컴파일 촉발 임계값

**코드 캐시:**
- `-XX:ReservedCodeCacheSize`: 코드 캐시 크기

**컴파일 로그:**
- `-XX:+PrintCompilation`: 컴파일 로그 출력
- `-XX:+UnlockDiagnosticVMOptions -XX:+PrintInlining`: 인라이닝 로그

## 11.6 마치며

백엔드 컴파일러는 자바의 성능을 좌우하는 핵심 요소다. JIT 컴파일러의 다양한 최적화 기법을 이해하면 더 효율적인 코드를 작성하고, 성능 문제를 진단할 수 있다. AOT 컴파일러는 특정 사용 사례에서 JIT의 한계를 보완하며, GraalVM과 같은 새로운 기술은 자바의 가능성을 더욱 확장하고 있다.
