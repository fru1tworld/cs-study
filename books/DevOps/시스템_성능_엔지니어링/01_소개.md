# 1장 소개

## 1.1 시스템 성능

시스템 성능 분석은 소프트웨어와 하드웨어의 모든 핵심 구성 요소를 포괄하여 컴퓨터 시스템 전체의 성능을 연구하는 과정이다.

데이터가 지나는 경로의 모든 요소, 저장 장치에서부터 애플리케이션 소프트웨어에 이르기까지, 이 모든 요소가 성능에 영향을 미칠 수 있고 성능 분석의 대상이 된다.

대상 애플리케이션의 경로가 포함된 다이어그램이 없다면 직접 그려보는 것이 좋다.

시스템 성능 분석의 목적은 컴퓨팅 비용을 절감하는 것이다. 비효율성을 제거하고, 시스템 스루풋을 향상시키며 시스템 전반을 튜닝함으로써 비용을 절감할 수 있다.

소프트웨어에서 말하는 풀스택과 시스템 성능에서 말하는 풀스택이 다르다. 하드웨어까지 포함하며 심지어 컴파일러도 포함하는데 이조차 성능에 영향을 미칠 수 있기 때문이다.

수용량 계획(Capacity Planning)은 앞서 언급한 여러 활동 중 일부를 포함하는 개념으로, 설계 단계에서는 개발 중인 소프트웨어의 자원 사용량을 분석하여, 설계가 목표 성능을 충족할 수 있는지 평가한다.

## 1.2 역할

시스템 성능 분석에는 다양한 역할을 가진 사람들이 관여한다. 각 역할은 고유한 관점과 책임을 가지고 성능 문제에 접근한다.

### 1.2.1 시스템 관리자 (System Administrator)

시스템 관리자는 운영체제, 특히 서버와 같은 다중 사용자 시스템에 대한 깊고 포괄적인 지식을 갖추고 있다. 소프트웨어가 해당 시스템에서 실행되고 계속 실행되도록 하는 방법을 알고 있으며, 물리적으로 하드웨어를 구축, 구성, 배선 및 설치하는 방법도 알고 있다. 가상 환경에서도 유사한 작업을 수행할 수 있다.

주요 업무:

- 시스템 신뢰성과 성능 유지
- 문제 및 오류 수정
- 작업 자동화
- 인시던트 대응
- 온콜 책임 관리

### 1.2.2 사이트 신뢰성 엔지니어 (Site Reliability Engineer, SRE)

SRE는 소프트웨어 엔지니어링과 시스템 관리를 결합하여 대규모 클라우드 기반 애플리케이션과 인프라의 확장성, 성능, 신뢰성을 보장하는 고급 DevOps 역할이다. Google에서 처음 만든 용어로, 운영을 소프트웨어 문제처럼 다루어 복잡한 운영 문제에 대한 자동화된 솔루션을 개발하는 것을 목표로 한다.

주요 책임:

- 시스템의 성능, 가용성, 신뢰성, 효율성, 변경 관리, 모니터링, 비상 대응
- SLI(Service Level Indicator) 모니터링 및 SLO(Service Level Objective) 설정
- 인시던트 대응 및 MTTR(Mean Time to Respond) 관리
- 수용량 계획(Capacity Planning)
- 자동화 구현

### 1.2.3 성능 엔지니어 (Performance Engineer)

성능 엔지니어는 성능 노력을 주도하며, 전략을 설계하고, 테스트를 수행하고, 소프트웨어 수명 주기 전반에 걸쳐 병목 현상을 해결한다.

시스템 개발 수명 주기의 모든 단계에서 적용되는 역할, 기술, 활동, 실천 사항, 도구 및 결과물의 집합을 담당하며, 솔루션이 정의된 성능 요구 사항을 충족하도록 설계, 구현 및 운영 지원되도록 보장한다.

### 1.2.4 애플리케이션 개발자

애플리케이션 개발자는 성능 최적화된 코드를 작성하고 설계 결정에 협력한다. 자신의 코드에 대한 깊은 지식을 가지고 있으며, 애플리케이션 레벨에서의 성능 문제를 가장 잘 이해하고 해결할 수 있다.

### 1.2.5 네트워크 엔지니어

네트워크 프로토콜과 네트워킹 장비에 대한 전문 지식을 보유하고 있다. 네트워크 관련 성능 문제를 진단하고 해결하는 데 핵심적인 역할을 한다.

### 1.2.6 데이터베이스 관리자 (DBA)

데이터베이스 시스템의 성능 튜닝, 쿼리 최적화, 인덱스 관리 등을 담당한다. 데이터베이스가 애플리케이션 성능에 미치는 영향을 이해하고 최적화한다.

## 1.3 활동

성능 엔지니어링은 소프트웨어 개발 수명 주기의 다양한 단계에서 수행되는 여러 활동을 포함한다.

### 1.3.1 성능 요구 사항 정의

시스템이 달성해야 하는 성능 목표를 명확히 정의한다. 응답 시간, 처리량, 자원 사용률 등의 구체적인 지표를 설정한다.

### 1.3.2 성능 모델링 및 설계

전체 개발 전에 엔지니어는 트래픽 급증이나 느린 데이터베이스 응답과 같은 다양한 조건에서 시스템이 어떻게 동작할지 시뮬레이션한다. 가정을 테스트하여 잠재적인 병목 현상을 예측하고 다양한 워크로드에서 구성을 테스트한다.

### 1.3.3 성능 테스트

부하 테스트, 스트레스 테스트, 내구성 테스트를 수행하여 시스템의 성능 특성을 파악한다:

- **부하 테스트 (Load Testing)**: 예상 부하에서 시스템 동작 확인
- **스트레스 테스트 (Stress Testing)**: 극한 부하에서 시스템의 한계점 확인
- **내구성 테스트 (Endurance Testing)**: 장시간 부하에서 시스템 안정성 확인

### 1.3.4 성능 분석

수집된 데이터를 분석하여 병목 현상을 식별하고 근본 원인을 파악한다. 지연시간, 처리량, 자원 사용률에 대한 통계적 측정치를 기반으로 분석한다.

### 1.3.5 성능 최적화

코드 최적화, 아키텍처 리뷰, 자원 재구성 등을 통해 식별된 성능 문제를 해결한다.

### 1.3.6 지속적 모니터링

운영 환경(프로덕션 배포 후)에서 성능 엔지니어링은 주로 세 가지 영역에 집중한다:

- **서비스 수준 관리**: SLA, SLO, SLI 모니터링 및 관리
- **수용량 관리**: 자원 사용량 추적 및 미래 수요 예측
- **문제 관리**: 성능 관련 인시던트 대응 및 해결

### 1.3.7 수용량 계획 (Capacity Planning)

수용량 계획은 예상되는 워크로드 수요와 패턴을 기반으로 의사 결정을 내리는 미래 지향적인 프로세스이다. 지속적인 부하와 피크 부하 시나리오 모두에서 워크로드 성능을 최적화하는 것이 목표이다.

수용량 계획의 핵심 구성 요소:

- **수요 예측**: 통계 분석, 추세 분석, 예측 모델링 기법을 사용하여 미래 수요를 예측
- **추세 분석**: 과거 데이터를 검토하여 일관된 패턴을 식별하고 미래로 추정
- **피크 부하 처리**: 성능 저하나 장애 없이 피크 사용 기간 동안 시스템이 처리할 수 있는 최대 부하 계획
- **시나리오 계획**: 최선의 경우와 최악의 경우 등 여러 시나리오를 계획하여 성장이 예상을 초과하더라도 대비

## 1.4 관점

성능 분석은 두 가지 주요 관점에서 접근할 수 있다:

### 1.4.1 자원 분석 (Resource Analysis)

자원 분석은 시스템 자원에서 시작하여 성능 문제를 조사하고 수용량을 계획하는 접근 방식이다.

네 가지 주요 시스템 자원에 초점을 맞춘다:

- **CPU**: 프로세서 사용률, 컨텍스트 스위칭, 런큐 길이
- **메모리**: 메모리 사용량, 페이징, 스와핑
- **디스크 I/O**: 디스크 사용률, I/O 대기 시간, 큐 길이
- **네트워크**: 대역폭 사용률, 패킷 손실, 지연시간

자원 분석의 핵심 지표:

- **사용률 (Utilization)**: U = B / T (U = 사용률, B = 총 사용 시간, T = 관찰 기간)
- **포화도 (Saturation)**: 자원이 처리할 수 없는 대기 작업의 정도
- **오류 (Errors)**: 자원과 관련된 오류 발생 빈도

### 1.4.2 워크로드 분석 (Workload Analysis)

워크로드 분석은 시스템에 가해지는 부하의 특성을 분석하는 접근 방식이다.

워크로드 특성화는 미래 컴퓨터 시스템 설계에서 중요한 단계이며 하드웨어 플랫폼에서 운영체제에 이르기까지 모든 시스템 수준의 설계에 영향을 미친다.

워크로드 특성화에서 답해야 할 질문들:

- **누가** 부하를 발생시키는가?
- **왜** 부하가 호출되는가?
- 부하의 **특성**은 무엇인가?

워크로드 모델링:

- **개방형 워크로드**: 주어진 평균 비율과 통계적 변동(예: 푸아송 프로세스)을 가진 이벤트 스트림
- **폐쇄형 워크로드**: 요청 간 무작위 "생각 시간"을 가진 고정된 수의 사용자 또는 소스

## 1.5 성능 분석의 어려움

### 1.5.1 주관성

일반적으로 버그나 오류는 흑백으로 판별할 수 있지만 성능은 주관적인 면이 꽤 있다.

### 1.5.2 복잡성

시스템 자체의 복잡성과 분석을 어디서부터 시작해야 할지 명확하지 않다는 점 때문에 도전적인 분야이다.

단일 컴포넌트 자체에서의 문제일 수 있고 "캐스케이딩 실패"로 인한 발생일 수 있다.

## 1.7 관측가능성

### 1.7.1 카운터, 통계, 지표

애플리케이션과 커널은 작업 수행 횟수, 바이트 수, 지연시간 측정치, 자원 사용량, 오류 비율 등 자신들의 상태와 활동에 관한 데이터를 제공한다.

이들은 대체로 소프트웨어 내에 하드코딩된 카운터라는 정수 변수로 구현된다.

### 1.7.2 프로파일링

기본적인 프로파일링은 샘플링을 수행하는 도구를 의미하는데 일부 측정값을 샘플로 잡아서 대상의 전반적인 상태를 대략적으로 파악하는 방식이다.

**플레임 그래프**를 살펴보면 CPU 프로파일링 데이터를 효과적으로 분석할 수 있는 시각화 도구이다.

이를 통해 시간에 따른 CPU 사용량 변화를 시각적으로 표현함으로써 단순한 CPU 문제뿐 아니라 사용 패턴을 통해 다양한 유형의 문제를 식별할 수 있다.

### 1.7.3 트레이싱

트레이싱은 시스템 내에서 발생하는 이벤트들을 기록하고 추적하는 과정이다.

이렇게 비교된 이벤트 데이터는 저장되어 추후 분석에 활용되거나 실시간으로 사용자가 원하는 요약 정보나 다른 처리를 위해 쓰일 수 있다.

이러한 트레이싱 도구들 중에는 strace 같이 시스템 콜을 트레이싱하거나, 리눅스 tcpdump 같이 네트워크 패킷을 트레이싱하기 위한 특수 목적 도구들도 있다.

또한 모든 소프트웨어 및 하드웨어 이벤트의 실행을 분석할 수 있는 범용 트레이싱 도구도 있다.

이러한 범용 트레이서들은 정적 및 동적 계측, 그리고 프로그래밍을 위해 BPF와 같은 다양한 이벤트 소스를 활용한다.

#### 정적 계측

정적 계측은 tracepoint라 불리며 사용자 공간 소프트웨어의 정적 계측 기술은 USDT라고 불린다.

#### 동적 계측

실행 중에 계측점을 생성하는 방식으로 메모리 내의 명령어를 수정하여 계측 루틴을 삽입한다.

동적 계측은 1990년대에 처음 개발되었으며 이와 함께 동적 트레이싱 도구라 불리는 도구들도 함께 만들어졌다.

#### BPF

원래 버클리 패킷 필터의 약자였지만 현재는 리눅스의 최신 동적 트레이싱 도구를 구동하는 핵심 기술로 발전했다.

### 1.7.4 eBPF (extended Berkeley Packet Filter)

BPF는 원래 Berkeley Packet Filter의 약자로 패킷 필터링을 위해 만들어졌지만, eBPF(extended BPF)로 확장되면서 패킷 필터링 이상의 기능을 수행할 수 있게 되었다.

eBPF는 리눅스 커널 내에서 작은 특수 목적 프로그램을 실행할 수 있게 해주는 기술이다.

이 프로그램들은 시스템 콜, 네트워크 이벤트, 함수 호출 등 커널 내에서 발생하는 다양한 이벤트에 의해 트리거된다.

#### eBPF의 관측 가능성 이점

- **최소한의 오버헤드**: eBPF 프로그램은 커널 내에서 직접 실행되어 성능을 최대화한다. 컨텍스트 스위치와 데이터 복사가 필요한 전통적인 사용자 공간 모니터링 도구와 달리, eBPF 프로그램은 커널 데이터 구조에 직접 접근할 수 있다.
- **동적 로딩**: eBPF 프로그램은 커널 재컴파일이나 시스템 재부팅 없이 동적으로 로드/언로드될 수 있다.
- **안전한 샌드박스**: eBPF 프로그램은 제한된 샌드박스에서 실행되어 시스템 안정성에 영향을 주지 않는다.

#### 주요 프론트엔드 도구

BPF 트레이싱의 주요 프론트엔드는 BCC와 bpftrace이다:

- **BCC (BPF Compiler Collection)**: 복잡한 도구와 데몬을 위한 툴킷으로, C로 커널 계측 프로그램을 작성하고 Python과 Lua 프론트엔드를 제공한다.
- **bpftrace**: 원라이너와 짧은 스크립트를 위한 고수준 트레이싱 언어로, awk, C, DTrace, SystemTap에서 영감을 받았다.

#### eBPF 기반 도구들

- **Cilium**: 쿠버네티스 환경에서 eBPF 기반 네트워킹, 보안, 관측 가능성을 제공하는 오픈소스 프로젝트
- **Hubble**: Cilium과 eBPF 위에 구축된 완전 분산형 네트워킹 및 보안 관측 가능성 플랫폼
- **Tracee**: eBPF 기술을 사용하여 운영체제 이벤트를 탐지하고 필터링하는 보안 도구

### 1.7.5 모니터링의 새로운 미래: 관측 가능성 (Observability)

#### 모니터링 vs 관측 가능성

**모니터링(Monitoring)**: 시스템 상태를 정의하는 사전에 정해진 메트릭에 대한 데이터를 수집하고 보고서를 생성하는 프로세스. 본질적으로 반응적(reactive)이며, 정적이고 잘 이해된 네트워크에 적합하다.

**관측 가능성(Observability)**: 시스템이 생성하는 외부 출력(로그, 메트릭, 트레이스)을 분석하여 복잡한 시스템의 내부 상태를 이해할 수 있는 능력. 본질적으로 능동적(proactive)이며, 마이크로서비스나 클라우드 네이티브 환경에 필수적이다.

> "모니터링은 무언가 잘못되었을 때 알려주고, 관측 가능성은 무엇이 일어나고 있는지, 왜 일어나고 있는지, 어떻게 고칠 수 있는지를 알려준다."

**핵심 차이점**:

- 모니터링은 "알려진 문제(known unknowns)"를 찾는 데 집중
- 관측 가능성은 "알려지지 않은 문제(unknown unknowns)"도 발견 가능
- 모니터링은 관측 가능성의 부분집합

#### 관측 가능성의 3가지 축 (Three Pillars of Observability)

관측 가능성은 세 가지 텔레메트리 데이터 기둥에 의존한다:

##### 1. 메트릭 (Metrics)

- 시스템의 성능과 동작을 측정한 **수치 데이터**
- 특정 시점에서 시스템의 현재 상태를 나타내는 수치 값
- CPU 사용량, 메모리 사용량, 요청 수, 오류율 등
- **"무엇이 잘못되었는가?"** 에 대한 답을 제공

##### 2. 로그 (Logs)

- 시스템에서 발생하는 개별 이벤트에 대한 **상세한 기록**
- 타임스탬프가 찍힌 불변의 이벤트 레코드
- 구조화된 로그(JSON)와 비구조화된 로그(텍스트)
- **"왜 그런 일이 발생했는가?"** 에 대한 상세한 컨텍스트 제공

##### 3. 트레이스 (Traces)

- 요청이 분산 시스템을 통과하는 **전체 여정**을 추적
- 마이크로서비스 간의 요청 흐름을 시각화
- 각 서비스에서의 지연시간과 오류 지점 파악
- **"어디서 발생했는가?"** 에 대한 답을 제공

> 세 가지 모두 중요하지만, 관측 가능성은 단순히 이들을 수집하는 것 이상이다. 공유된 컨텍스트를 통해 이들을 연결하고 "무슨 일이 일어나고 있는가?"라는 질문에 명확하고 실행 가능한 답변을 얻는 것이 핵심이다.

#### OpenTelemetry: 관측 가능성의 표준

OpenTelemetry(OTel)는 텔레메트리 데이터 수집을 위한 **사실상의 표준(de facto standard)**으로 부상했다.

##### OpenTelemetry란?

- CNCF(Cloud Native Computing Foundation)에서 Kubernetes 다음으로 두 번째로 큰 프로젝트
- 메트릭, 로그, 트레이스 수집을 표준화하는 오픈소스 프레임워크
- 벤더 종속성(vendor lock-in) 없이 다양한 백엔드 시스템과 호환

##### 2024년 성과

- **로그 지원 GA(General Availability)**: 세 가지 축 모두 안정화 달성
- **85%의 조직**이 OpenTelemetry를 널리 채택
- 평균 **52%의 MTTR(평균 복구 시간) 감소** 달성
- 일부 조직은 시스템 이상 탐지 능력이 최대 67% 향상

#### 네 번째 축: 연속 프로파일링 (Continuous Profiling)

2024년, OpenTelemetry는 세 가지 축을 넘어 **프로파일링(Profiling)**을 네 번째 신호로 도입했다.

##### 연속 프로파일링이란?

런타임에 애플리케이션 코드의 동작과 성능을 동적으로 검사하는 방법으로:

- 코드 수준에서 자원 사용량에 대한 인사이트 제공
- 시간에 따른 프로파일링 데이터 저장, 쿼리, 분석 가능
- 정확히 코드의 어느 부분에서 문제가 발생하는지 파악

##### Elastic의 eBPF 기반 프로파일링 에이전트 기증

- Elastic이 OpenTelemetry에 연속 프로파일링 에이전트를 기증
- eBPF 기반으로 다양한 프로그래밍 언어와 런타임 관찰
- 코드 계측, 재컴파일, 서비스 재시작 불필요
- 프로덕션 환경에서 낮은 CPU(~1%)와 메모리 오버헤드

##### 신호 간 상관관계 (Signal Correlation)

프로파일은 다른 신호들과 양방향 링크를 지원한다:

- **메트릭 → 프로파일**: CPU나 메모리 사용량 급증에서 해당 자원을 소비하는 특정 코드로 이동
- **트레이스 → 프로파일**: 서비스 간 지연시간의 위치뿐 아니라, 지연이 코드의 어떤 부분에서 발생하는지 확인

#### 2025년 관측 가능성 트렌드

##### 1. 통합 관측 가능성 플랫폼

로그, 트레이스, 메트릭, 이벤트, 프로파일을 하나의 중앙화된 뷰로 통합

##### 2. AI 기반 관측 가능성 (AIOps)

- AI/ML을 활용한 이상 탐지 자동화
- 자동 근본 원인 분석
- 예측적 알림과 자동 복구
- GenAI 애플리케이션의 모니터링과 보안

##### 3. SecOps 관측 가능성 강화

보안 운영과 관측 가능성의 통합으로 위협 탐지 강화

##### 4. 풀스택 가시성 (Full-Stack Visibility)

프론트엔드부터 백엔드, 인프라까지 전체 스택에 대한 가시성 확보

## 1.8 실험

성능 분석에서 실험은 가설을 검증하고 시스템 동작을 이해하는 데 필수적인 활동이다.

### 1.8.1 벤치마킹 (Benchmarking)

벤치마킹은 시스템의 성능을 정량적으로 측정하고 비교하는 프로세스이다.

**기준 지표 설정**: 시스템의 정상 성능 지표를 식별하고 문서화한다. 여기에는 평균 응답 시간, 오류율, 시스템 처리량이 포함된다. 이러한 지표는 도입된 실패의 영향을 측정할 수 있는 벤치마크 역할을 한다.

벤치마킹 유형:

- **마이크로 벤치마킹**: 개별 컴포넌트나 함수의 성능 측정
- **매크로 벤치마킹**: 전체 시스템이나 애플리케이션의 성능 측정
- **비교 벤치마킹**: 서로 다른 시스템이나 구성 간의 성능 비교

### 1.8.2 카오스 엔지니어링 (Chaos Engineering)

> "카오스 엔지니어링은 프로덕션 환경에서 격동적인 조건을 견딜 수 있는 시스템의 능력에 대한 신뢰를 구축하기 위해 시스템에서 실험하는 분야이다."

카오스 엔지니어링은 혼란을 만드는 것이 아니라, 통제된 실험을 사용하여 문제가 발생하기 전에 시스템의 잠재적인 실패 지점을 식별하는 것이다.

#### 핵심 원칙

**가설 형성**: 카오스 엔지니어링은 시스템 동작에 대한 가설을 수립하고 테스트하는 과학적 원칙에 따라 운영된다. 예를 들어, "서버 인스턴스가 실패하면 로드 밸런서가 상당한 지연시간 증가 없이 트래픽을 재분배할 것이다"라는 가설을 세울 수 있다.

**통제된 실험**: 카오스 실험은 시스템 운영에 장애나 이상을 체계적으로 도입하여 그러한 조건에 대한 시스템의 반응을 관찰하고 이해하기 위해 신중하게 설계되고, 통제되고, 모니터링되는 프로세스이다.

#### 성능 엔지니어링과의 관계

카오스 엔지니어링은 단위 테스트, 통합 테스트, 성능 벤치마킹의 대안이 아니다. 이들과 보완적으로, 심지어 병렬로 작동한다. 예를 들어, 카오스 엔지니어링 테스트와 성능 테스트를 동시에 실행하면 부하 상태에서만 발생하는 문제를 찾는 데 도움이 될 수 있다.

#### 주요 도구

- **Chaos Monkey**: Netflix가 개발한 도구로, 프로덕션 환경에서 인스턴스를 무작위로 종료한다
- **Gremlin**: CPU 급증, 네트워크 파티셔닝, 블랙홀 공격 등 다양한 실패 주입 기능을 제공하는 상용 플랫폼
- **Azure Chaos Studio**: 하드투파인드 문제의 발견을 가속화하기 위한 완전 관리형 카오스 엔지니어링 실험 플랫폼

#### 이점

카오스 엔지니어링을 수행하는 대부분의 팀은 가용성 향상, 평균 복구 시간(MTTR) 감소, 평균 탐지 시간(MTTD) 감소, 프로덕트에 출시되는 버그 감소, 장애 감소를 경험한다. 카오스 엔지니어링 실험을 자주 수행하는 팀은 99.9% 이상의 가용성을 달성할 가능성이 높다.

## 1.9 클라우드 컴퓨팅

클라우드 컴퓨팅은 시스템 성능 분석에 새로운 차원을 추가한다. 클라우드 환경에서의 성능 분석은 기존 온프레미스 환경과 다른 고려 사항을 필요로 한다.

### 1.9.1 클라우드 서비스 모델

#### IaaS (Infrastructure as a Service)

IaaS는 가상화된 컴퓨팅 자원을 제공하며, 성능 관점에서 가장 많은 제어권을 제공한다.

성능 특성:

- **낮은 지연시간과 향상된 성능**: IaaS 제공자는 일반적으로 여러 위치에서 데이터 센터를 운영하여, 사용자와 가까운 곳에서 앱과 서비스를 실행하여 지연시간을 최소화
- **베어 메탈 서버**: 높은 성능, 엄격한 규정 준수, 또는 최소 지연시간이 필요한 워크로드에 유용

#### PaaS (Platform as a Service)

PaaS에서 데이터베이스 쿼리 응답 시간은 특히 지연시간에 민감하다. 연구에 따르면 지연시간 증가가 PaaS 기반 데이터베이스 성능에 상당한 영향을 미친다.

#### SaaS (Software as a Service)

다양한 지연시간 조건에서 SaaS 애플리케이션을 분석한 연구에 따르면, CRM 및 ERP 시스템과 같은 애플리케이션의 응답 시간이 지연시간이 높을수록 상당히 증가한다.

> SaaS가 시간과 유지보수에서 절약하는 것이 제어, 보안, 성능 측면에서 비용이 될 수 있으므로, 신뢰할 수 있는 제공자를 선택하는 것이 중요하다.

### 1.9.2 클라우드 성능 분석의 도전 과제

**네트워크 지연시간의 영향**: 클라우드 컴퓨팅의 부상은 현대 IT 인프라를 혁신시켰지만, 클라우드 시스템의 효율성은 네트워크 성능, 특히 네트워크 지연시간에 크게 의존한다. 사용자와 클라우드 서버 간의 데이터 전송 지연인 지연시간은 애플리케이션의 응답성과 전반적인 사용자 경험에 영향을 미친다.

**가상화와 APM**: 가상화와 세 가지 클라우드 플랫폼(IaaS, PaaS, SaaS)은 애플리케이션 성능 모니터링(APM) 분야에 큰 영향을 미친다. 클라우드 컴퓨팅 성능 관리는 큰 기술적 도전일 뿐만 아니라 실제 게임 체인저이기도 하다.

### 1.9.3 서버리스 vs 서버 기반 성능 비교

Compute Engine과 Vertex AI 서비스를 JMeter를 통해 테스트한 연구에 따르면:

- Vertex AI 성능이 Compute Engine보다 우수 (CPU 사용률 0.10%, 메모리 사용률 0.94%, 지연시간 17.34ms)
- 단, 비용 효율성은 Compute Engine이 유리

세 가지 서버리스 클라우드 컴퓨팅 플랫폼(Google Cloud Run, AWS App Runner, Azure Container Apps)을 분석한 연구에서는 Google의 Cloud Run이 경쟁자보다 더 나은 성능과 사용성을 보여주었다.

### 1.9.4 멀티 클라우드 환경

IaaS, PaaS, SaaS는 클라우드 컴퓨팅에서 상호 배타적이지 않다. 많은 중견 및 대기업이 둘 이상 또는 세 가지 모두를 사용한다. 이러한 환경에서의 성능 분석은 각 서비스 모델의 특성을 이해하고 통합된 관점에서 모니터링해야 한다.

## 1.10 방법론

성능 분석에는 여러 가지 체계적인 방법론이 사용된다. 자세한 내용은 2장에서 다루지만, 여기서는 간단히 소개한다.

### 1.10.1 60초 리눅스 성능 분석

Netflix에서 개발한 이 방법론은 성능 문제의 빠른 진단을 위한 10가지 명령어를 제공한다:

```bash
# 1. uptime - 부하 평균 확인
uptime

# 2. dmesg | tail - 최근 시스템 메시지 확인
dmesg | tail

# 3. vmstat 1 - 가상 메모리 통계
vmstat 1

# 4. mpstat -P ALL 1 - CPU별 시간 분석
mpstat -P ALL 1

# 5. pidstat 1 - 프로세스별 CPU 사용량
pidstat 1

# 6. iostat -xz 1 - 블록 디바이스 I/O 통계
iostat -xz 1

# 7. free -m - 메모리 사용량
free -m

# 8. sar -n DEV 1 - 네트워크 인터페이스 처리량
sar -n DEV 1

# 9. sar -n TCP,ETCP 1 - TCP 통계
sar -n TCP,ETCP 1

# 10. top - 전체 시스템 요약
top
```

이 명령어들이 노출하는 지표는 USE 방법론(Utilization, Saturation, Errors)을 완성하는 데 도움이 된다.

#### 각 명령어 설명

1. **uptime**: 시스템 부하 평균을 빠르게 확인. 1분, 5분, 15분 평균 제공
2. **dmesg | tail**: 마지막 10개의 시스템 메시지 확인. oom-killer, TCP dropping 등 성능 문제를 유발하는 오류 확인
3. **vmstat 1**: 가상 메모리 통계. 1초 간격으로 서버의 주요 통계 요약 출력
4. **mpstat -P ALL 1**: CPU별 시간 분석. 불균형 확인 가능하며, 단일 핫 CPU는 단일 스레드 애플리케이션의 증거
5. **pidstat 1**: top과 유사하지만 화면을 지우지 않고 롤링 요약 출력
6. **iostat -xz 1**: 블록 디바이스(디스크)의 워크로드와 결과 성능 이해에 유용
7. **free -m**: 사용 가능한 메모리 확인. 특히 available 컬럼은 회수 가능한 메모리량 표시
8. **sar -n DEV 1**: 네트워크 인터페이스 처리량 확인 (rxkB/s, txkB/s)
9. **sar -n TCP,ETCP 1**: 주요 TCP 지표 요약 (active/s, passive/s 등)
10. **top**: 전체 시스템 상태. 단, 시간에 따른 패턴 파악이 어려움

### 1.10.2 USE 방법론

모든 자원에 대해 세 가지를 확인하는 방법론:

- **U (Utilization)**: 자원이 얼마나 사용되고 있는가?
- **S (Saturation)**: 자원에 대기 중인 작업이 있는가?
- **E (Errors)**: 자원에서 오류가 발생하고 있는가?

### 1.10.3 RED 방법론

모든 서비스에 대해 확인하는 방법론:

- **R (Rate)**: 초당 요청 수
- **E (Errors)**: 실패한 요청 수
- **D (Duration)**: 요청 처리 시간

### 1.10.4 기타 방법론

- **과학적 방법론**: 질문 → 가설 → 예측 → 테스트 → 분석
- **진단 사이클**: 가설 → 계측 → 데이터 → 가설
- **도구 방법론**: 사용 가능한 성능 도구 목록 → 각 도구에서 지표 수집 → 수집된 지표 해석
- **드릴다운 분석**: 모니터링 → 식별 → 분석의 3단계 방법론

## 1.11 사례 연구

### 1.11.1 느린 디스크

**문제 상황**: 온라인 게임 회사가 게임 로딩 시간이 느리다는 불만을 받기 시작했다. 문제는 SQL Server 데이터베이스로 추적되었다.

**분석 과정**: 성능 지표를 확인한 결과, 디스크 I/O 서브시스템이 병목 현상을 일으키고 있었으며, 높은 디스크 큐 길이와 디스크 지연시간이 관찰되었다. 조사 결과, 모든 데이터베이스가 단일의 느린 디스크에 호스팅되어 있었다.

**해결 방법**:

- I/O 부하를 분산하기 위해 TempDB와 로그 파일을 별도의 빠른 SSD 드라이브로 이동
- 데이터 파일의 생성 및 증가 속도를 높이기 위해 Instant File Initialization(IFI) 활성화

**결과**: 이전에 느렸던 시스템의 응답 시간이 50% 이상 개선되었다.

### 1.11.2 소프트웨어 변경

**문제 상황**: 회사의 IT 팀이 데이터베이스 성능 지표를 분석한 결과, 시스템이 과도하게 인덱싱되어 저장 공간 낭비와 쿼리 실행 중 불필요한 속도 저하가 발생하고 있음을 발견했다.

**분석 과정**: 거의 70%의 느린 쿼리가 부적절한 인덱스 활용에서 비롯된다. SQL Server의 DMV나 PostgreSQL의 EXPLAIN 명령을 사용하여 누락된 인덱스를 식별하는 체계적인 접근 방식이 필요했다.

**해결 방법**:

- 불필요한 인덱스 제거
- 누락된 인덱스 추가
- 쿼리 최적화

**핵심 교훈**:

- 최적화는 일회성 이벤트가 아니라 지속적인 프로세스로 보아야 함
- 정기적인 감사와 성능 지표가 워크플로우에 통합되어야 함
- 데이터베이스 최적화를 목적지가 아닌 지속적인 여정으로 접근

### 1.11.3 추가 자료

지속적인 모니터링은 데이터베이스 성능 유지에 필수적이다:

- **Nagios, Prometheus, New Relic** 등의 도구가 실시간 모니터링과 알림 제공
- 이러한 도구를 사용하는 회사들은 성능 문제를 사전에 해결 가능
- 지속적인 모니터링은 관리자가 사용자에게 영향을 미치기 전에 성능 문제를 발견하는 데 도움
