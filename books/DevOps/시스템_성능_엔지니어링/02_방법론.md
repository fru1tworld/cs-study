# 2장 방법론

## 2.1 용어

- **IOPS (Input/Output Operations Per Second)**: 초당 입출력 연산 횟수. 저장 장치의 성능을 측정하는 핵심 지표
- ** 스루풋 (Throughput)**: 단위 시간당 처리량. 보통 초당 데이터 전송량(bytes/sec) 또는 초당 연산 수로 표현
- ** 응답 시간 (Response Time)**: 연산이 완료될 때까지 걸리는 전체 시간. 서비스 시간 + 대기 시간
- ** 지연시간 (Latency)**: 요청에 대한 처리를 받기까지 기다려야 하는 시간. 휴지(latent) 상태인 시간
- ** 사용률 (Utilization)**: 자원이 작업을 처리하는 데 사용된 시간의 비율 (0~100%)
- ** 포화도 (Saturation)**: 자원이 처리하지 못해 대기 중인 작업의 정도
- ** 병목 (Bottleneck)**: 시스템 성능을 제한하는 가장 느린 구성 요소
- ** 워크로드 (Workload)**: 시스템에 적용되는 입력 또는 부하
- ** 캐시 (Cache)**: 빠른 저장 영역으로 느린 저장 장치의 데이터를 임시 저장

## 2.2 모델

### 2.2.1 테스트 중인 시스템 (System Under Test)

성능 분석의 대상이 되는 시스템을 정의한다. 명확한 경계를 설정하여:

- 입력(inputs): 시스템에 들어오는 워크로드
- 출력(outputs): 시스템이 생성하는 결과
- 환경: 시스템이 동작하는 컨텍스트

### 2.2.2 큐 시스템

시스템을 큐 모델로 표현:

```
도착(Arrival) → [큐] → [서비스 센터] → 완료(Departure)
```

- 도착률(λ): 단위 시간당 도착하는 요청 수
- 서비스 시간: 요청을 처리하는 데 걸리는 시간
- 큐 길이: 대기 중인 요청의 수

## 2.3 개념

### 2.3.1 지연시간

지연시간은 성능 분석에서 가장 중요한 지표 중 하나다.

** 지연시간 vs 응답시간:**

- 지연시간: 요청이 처리되기를 기다리는 순수 대기 시간
- 응답시간: 서비스 시간 + 대기 시간 + 네트워크 지연 등 전체 시간

** 지연시간이 발생하는 곳:**

- CPU 스케줄러 큐
- 디스크 I/O 큐
- 네트워크 큐
- 락 경합 (Lock contention)

### 2.3.2 시간 스케일

프로그래머가 알아야 할 지연시간 수치 (Jeff Dean의 유명한 자료):

| 연산                          | 시간   |
| ----------------------------- | ------ |
| L1 캐시 참조                  | 0.5 ns |
| 분기 예측 실패                | 3 ns   |
| L2 캐시 참조                  | 7 ns   |
| 뮤텍스 락/언락                | 25 ns  |
| 메인 메모리 참조              | 100 ns |
| 1KB Snappy 압축               | 2 µs   |
| 1Gbps 네트워크로 1KB 전송     | 10 µs  |
| SSD 랜덤 읽기                 | 16 µs  |
| 메모리에서 1MB 순차 읽기      | 250 µs |
| 같은 데이터센터 내 왕복       | 500 µs |
| SSD에서 1MB 순차 읽기         | 1 ms   |
| HDD 탐색                      | 10 ms  |
| HDD에서 1MB 순차 읽기         | 30 ms  |
| 캘리포니아↔네덜란드 패킷 왕복 | 150 ms |

** 시사점:**

- 메모리는 디스크보다 수만 배 빠르다
- SSD는 HDD보다 수십~수백 배 빠르다
- 네트워크 지연시간은 매우 가변적이다

### 2.3.3 트레이드오프

성능 튜닝의 대표적인 트레이드오프:

** 메모리 vs CPU:**

- 메모리: 결과를 캐시하여 CPU 사용 감소
- CPU: 데이터를 압축하여 메모리 사용 감소

** 파일 시스템 레코드 크기:**

- 작은 레코드: 임의 접근 I/O 성능 향상, 캐시 효율성 증가
- 큰 레코드: 순차 접근 I/O 성능 향상

** 네트워크 버퍼 크기:**

- 작은 버퍼: 연결당 메모리 오버헤드 감소, 더 많은 연결 처리 가능
- 큰 버퍼: 대역폭 활용도 향상, 지연시간 감소

### 2.3.4 튜닝을 위한 노력

성능 튜닝 가능한 스택의 각 계층:

| 계층         | 튜닝 포인트                           |
| ------------ | ------------------------------------- |
| 애플리케이션 | 로직, 요청 큐 크기, DB 쿼리           |
| 데이터베이스 | 테이블 구조, 인덱스, 버퍼             |
| 시스템 콜    | 메모리 매핑, I/O 플래그 (동기/비동기) |
| 파일 시스템  | 레코드 크기, 캐시 크기, 저널링        |
| 저장 장치    | RAID 수준, 디스크 개수/유형           |

### 2.3.5 적합성의 수준

ROI(투자 수익률)는 단순 비용 절감보다는 ** 고객 만족도 향상** 에 초점을 맞춰야 한다.

- 성능 개선의 가치는 비즈니스 목표와 연결되어야 함
- 사용자 경험에 직접적인 영향을 주는 지표에 집중

### 2.3.6 분석을 언제 중단할 것인가

분석 중단 기준:

- 성능 목표 달성
- 추가 분석의 비용 > 예상 이득
- 시간 제약
- 더 이상 의미 있는 개선점을 찾을 수 없을 때

**80/20 법칙**: 80%의 문제는 20%의 원인에서 발생

### 2.3.7 성능 개선의 한시성

성능 개선은 영구적이지 않다:

- 워크로드 변화
- 사용자 증가
- 소프트웨어 업데이트
- 하드웨어 노후화

→ 지속적인 모니터링과 재평가가 필요

### 2.3.8 부하 vs. 아키텍처

성능 문제의 원인 구분:

- ** 부하 문제**: 더 많은 자원 추가로 해결 (스케일 아웃/업)
- ** 아키텍처 문제**: 설계 변경이 필요 (리팩토링, 재설계)

부하 문제로 보이는 것이 실제로는 아키텍처 문제인 경우가 많다.

### 2.3.9 규모 확장성

확장성(Scalability): 부하 증가에 따라 성능을 유지하는 능력

** 수직 확장 (Scale Up):**

- 단일 시스템의 자원 증가 (CPU, 메모리 추가)
- 한계가 있음

** 수평 확장 (Scale Out):**

- 시스템 수를 늘림
- 분산 시스템의 복잡성 증가

### 2.3.10 지표

좋은 성능 지표의 특성:

- 측정 가능 (Measurable)
- 의미 있음 (Meaningful)
- 실행 가능 (Actionable)

** 선행 지표 vs 후행 지표:**

- 선행 지표: 문제 예측 가능 (큐 길이, 사용률)
- 후행 지표: 문제 발생 후 확인 (오류율, 응답시간)

### 2.3.11 사용률

** 사용률 (Utilization)**: 자원이 사용 중인 시간의 비율

```
사용률 = (사용 시간 / 전체 시간) × 100%
```

** 해석:**

- 100% 사용률 → 일반적으로 병목
- 70%+ 사용률 → I/O 자원의 경우 문제 가능성
- 시간 간격 주의: 5분 평균 60%는 3분간 100% + 2분간 0%일 수 있음

** 용량 기반 사용률:**

```
사용률 = (사용된 용량 / 전체 용량) × 100%
```

### 2.3.12 포화도

** 포화도 (Saturation)**: 자원이 수용할 수 없는 초과 작업의 정도

측정 방법:

- 큐 길이 (run queue length)
- 대기 시간

** 해석:**

- 포화도 > 0 → 지연시간 발생
- 높은 포화도 → 심각한 성능 저하

### 2.3.13 프로파일링

** 프로파일링**: 프로그램 실행 시간이 어디에 소비되는지 분석

** 샘플링 (Sampling):**

- 주기적으로 콜 스택 캡처
- 낮은 오버헤드
- 통계적 정확성

** 트레이싱 (Tracing):**

- 모든 이벤트 기록
- 높은 정확성
- 높은 오버헤드

**Flame Graph (불꽃 그래프):**

- Brendan Gregg가 발명
- x축: 샘플 비율 (시간 순서 아님)
- y축: 스택 깊이
- 넓은 박스 = 더 많은 CPU 시간 소비
- 병목 지점을 시각적으로 빠르게 파악

### 2.3.14 캐싱

캐시 성능 지표:

- ** 캐시 적중률 (Hit Ratio)**: 캐시에서 찾은 요청의 비율
- ** 캐시 미스율 (Miss Ratio)**: 1 - 적중률

** 캐시 교체 정책:**

| 정책                             | 설명                           | 사용 사례          |
| -------------------------------- | ------------------------------ | ------------------ |
| LRU (Least Recently Used)        | 가장 오래전에 사용된 항목 제거 | 일반적인 워크로드  |
| LFU (Least Frequently Used)      | 가장 적게 사용된 항목 제거     | 빈도 기반 워크로드 |
| MRU (Most Recently Used)         | 가장 최근에 사용된 항목 제거   | 순환 스캔 패턴     |
| ARC (Adaptive Replacement Cache) | LRU + LFU 하이브리드           | 적응형 워크로드    |

** 캐시 워밍 (Cache Warming):**

- 콜드 캐시: 비어있거나 오래된 데이터
- 웜 캐시: 유효한 데이터로 채워진 상태

### 2.3.15 모른다는 것을 아는 것들

**Known Unknowns (알려진 미지수):**

- 측정해야 하지만 아직 측정하지 않은 것들
- 도구가 없어서 측정 못하는 것들

**Unknown Unknowns (미지의 미지수):**

- 존재 자체를 모르는 성능 문제
- 가장 위험한 영역

→ USE 방법론 등을 통해 Known Unknowns를 체계적으로 식별

## 2.4 관점

### 2.4.1 자원 분석 (Resource Analysis)

** 시스템 관리자 관점**: 물리적 자원 중심

분석 대상:

- CPU
- 메모리
- 디스크 I/O
- 네트워크 I/O

질문:

- 어떤 자원이 병목인가?
- 사용률과 포화도는?
- 오류가 발생하는가?

### 2.4.2 워크로드 분석 (Workload Analysis)

** 애플리케이션 개발자 관점**: 워크로드 중심

분석 대상:

- 요청 유형
- 요청 빈도
- 요청 분포

질문:

- 누가 부하를 발생시키는가?
- 왜 부하가 발생하는가?
- 부하의 특성은?

## 2.5 방법론

### 2.5.1 가로등 반방법론 (Streetlight Anti-Method)

** 잘못된 접근**: 익숙한 도구만 사용

```
"열쇠를 왜 가로등 아래서 찾나요?"
"여기가 밝으니까요."
```

문제점:

- 도구가 보여주는 것만 분석
- 실제 문제를 놓칠 수 있음

### 2.5.2 임의 변경 반방법론 (Random Change Anti-Method)

** 잘못된 접근**: 무작위로 설정 변경

문제점:

- 효과를 측정하지 않음
- 원인-결과 관계 파악 불가
- 새로운 문제 발생 가능

### 2.5.3 다른 사람 비난 반방법론 (Blame Someone Else Anti-Method)

** 잘못된 접근**: 다른 팀/컴포넌트 탓

```
"네트워크 문제인 것 같아요"
"DB팀에 문의해보세요"
```

문제점:

- 실제 원인 파악 지연
- 팀 간 갈등 유발

### 2.5.4 전용 체크리스트 방법론 (Ad Hoc Checklist Method)

조직별 맞춤 체크리스트 작성:

```markdown
□ CPU 사용률 확인
□ 메모리 사용량 확인
□ 디스크 I/O 확인
□ 네트워크 대역폭 확인
□ 애플리케이션 로그 확인
```

장점: 경험 기반, 빠른 진단
단점: 새로운 문제 유형에 취약

### 2.5.5 문제 내역서 (Problem Statement)

성능 문제를 명확하게 정의:

1. ** 무엇이** 문제인가? (증상)
2. ** 언제** 시작되었나? (시점)
3. ** 무엇이** 변경되었나? (변경사항)
4. ** 누가** 영향을 받는가? (범위)
5. ** 어디서** 발생하는가? (위치)

### 2.5.6 과학적 방법론 (Scientific Method)

```
1. 질문 (Question)
   ↓
2. 가설 (Hypothesis)
   ↓
3. 예측 (Prediction)
   ↓
4. 실험 (Test)
   ↓
5. 분석 (Analysis)
   ↓
6. (반복 또는 결론)
```

예시:

1. 질문: "왜 응답시간이 느린가?"
2. 가설: "디스크 I/O가 병목이다"
3. 예측: "디스크 사용률이 높을 것이다"
4. 실험: iostat으로 측정
5. 분석: 결과 해석 및 다음 단계

### 2.5.7 진단 사이클 (Diagnosis Cycle)

```
가설 → 도구 선택 → 데이터 수집 → 데이터 분석 → 가설 수정/확인
                              ↓
                         (반복)
```

### 2.5.8 도구 방법론 (Tools Method)

사용 가능한 도구를 순차적으로 실행:

```bash
# 시스템 개요
uptime
vmstat 1
iostat -xz 1
mpstat -P ALL 1

# 네트워크
sar -n DEV 1
sar -n TCP,ETCP 1

# 프로세스
top
pidstat 1
```

단점: 가로등 반방법론과 유사한 위험

### 2.5.9 USE 방법론 (USE Method)

**Brendan Gregg가 개발한 방법론**

```
모든 자원에 대해:
  → 사용률 (Utilization) 확인
  → 포화도 (Saturation) 확인
  → 오류 (Errors) 확인
```

** 자원 목록:**

- CPU (소켓, 코어, 스레드)
- 메모리 용량
- 네트워크 인터페이스
- 저장 장치 (I/O, 용량)
- 컨트롤러 (저장 장치, 네트워크)
- 상호연결 (버스)

**USE 체크리스트 예시:**

| 자원     | 유형   | 지표          | 도구 (Linux)        |
| -------- | ------ | ------------- | ------------------- |
| CPU      | 사용률 | CPU 사용률    | vmstat, mpstat, top |
| CPU      | 포화도 | 실행 큐 길이  | vmstat (r 열)       |
| CPU      | 오류   | -             | perf                |
| 메모리   | 사용률 | 메모리 사용량 | free, vmstat        |
| 메모리   | 포화도 | 스왑 사용량   | vmstat (si/so)      |
| 메모리   | 오류   | OOM 이벤트    | dmesg               |
| 디스크   | 사용률 | 장치 busy%    | iostat              |
| 디스크   | 포화도 | 대기 큐 길이  | iostat (avgqu-sz)   |
| 디스크   | 오류   | 장치 오류     | dmesg               |
| 네트워크 | 사용률 | 대역폭 사용률 | sar, nicstat        |
| 네트워크 | 포화도 | 버퍼 오버런   | netstat -s          |
| 네트워크 | 오류   | 패킷 오류     | netstat -i          |

** 장점:**

- 체계적이고 완전함
- 약 80%의 서버 문제를 5%의 노력으로 발견
- Known Unknowns를 명확히 식별

### 2.5.10 RED 방법론 (RED Method)

**Tom Wilkie가 개발 (Google의 Four Golden Signals 기반)**

마이크로서비스 모니터링에 최적화:

```
모든 서비스에 대해:
  → Rate: 초당 요청 수
  → Errors: 실패한 요청 수
  → Duration: 요청 처리 시간 분포
```

**USE vs RED:**
| 방법론 | 관점 | 대상 |
|--------|------|------|
| USE | 인프라 | 자원 (CPU, 메모리, 디스크) |
| RED | 사용자 | 서비스 (요청 기반) |

** 함께 사용:** USE로 인프라, RED로 서비스 모니터링

### 2.5.11 워크로드 특성화 (Workload Characterization)

워크로드를 이해하기 위한 질문:

- **Who**: 누가 요청을 생성하는가?
- **What**: 무엇을 요청하는가?
- **When**: 언제 요청이 발생하는가?
- **How**: 어떤 패턴으로 발생하는가?

특성화 항목:

- 요청 유형 분포
- 요청 크기 분포
- 요청 빈도 (시간대별)
- 피크 부하 패턴

### 2.5.12 드릴다운 분석 (Drill-Down Analysis)

**Top-Down 접근**: 높은 수준에서 낮은 수준으로

```
시스템 전체
    ↓
서브시스템 (CPU, 메모리, I/O)
    ↓
개별 프로세스
    ↓
함수/코드 레벨
```

** 예시:**

1. 시스템 전체 CPU 사용률 90%
2. 특정 프로세스가 80% 사용
3. 해당 프로세스의 특정 함수가 원인
4. 비효율적인 알고리즘 발견

### 2.5.13 지연시간 분석 (Latency Analysis)

응답시간을 구성요소별로 분해:

```
총 응답시간 = 네트워크 지연 + 큐 대기 + 처리 시간 + 응답 전송
```

각 구성요소를 측정하여 병목 식별

** 분포 분석:**

- 평균만으로는 부족
- 백분위수 (p50, p90, p99) 분석
- 히스토그램으로 분포 확인

### 2.5.14 R 방법론 (Method R)

데이터베이스 성능 분석용 방법론:

1. 사용자 경험 기반 목표 설정
2. 응답 시간 측정
3. 응답 시간 구성요소 분석
4. 가장 큰 구성요소 개선
5. 반복

### 2.5.15 이벤트 트레이싱 (Event Tracing)

시스템 이벤트를 추적:

- 시스템 콜
- 함수 호출
- I/O 연산
- 네트워크 패킷

도구:

- Linux: strace, perf, eBPF, ftrace
- 분산 시스템: Jaeger, Zipkin

### 2.5.16 기준 통계 (Baseline Statistics)

** 기준선 (Baseline)**: 정상 상태의 성능 지표

용도:

- 이상 탐지
- 추세 분석
- 용량 계획

수집 항목:

- 시간대별 평균/최대값
- 요일별 패턴
- 계절적 변동

### 2.5.17 정적 성능 튜닝 (Static Performance Tuning)

시스템 구성을 검토하여 문제 식별:

체크 항목:

- 커널 파라미터
- 파일 시스템 마운트 옵션
- 네트워크 설정
- 애플리케이션 설정

부하 없이도 문제점 발견 가능

### 2.5.18 캐시 튜닝 (Cache Tuning)

캐시 성능 최적화:

1. ** 캐시 크기**: 적중률과 메모리 사용 균형
2. ** 교체 정책**: 워크로드에 맞는 정책 선택
3. **TTL (Time To Live)**: 데이터 신선도와 성능 균형
4. ** 워밍 전략**: 콜드 스타트 방지

### 2.5.19 마이크로 벤치마킹 (Micro-Benchmarking)

작은 단위의 성능 측정:

- 특정 함수
- 특정 연산
- 특정 시스템 콜

** 주의사항:**

- 실제 워크로드와 다를 수 있음
- 캐시 웜업 고려
- 반복 측정으로 통계적 유의성 확보

### 2.5.20 성능 최적화를 위한 지침

1. ** 측정 먼저**: 추측하지 말고 측정하라
2. ** 병목 집중**: 가장 큰 문제부터 해결
3. ** 한 번에 하나**: 변경을 격리하여 효과 측정
4. ** 자동화**: 반복 가능한 테스트 구축
5. ** 문서화**: 변경사항과 결과 기록

## 2.6 모델링

### 2.6.1 엔터프라이즈 환경 vs. 클라우드 환경

** 엔터프라이즈:**

- 고정된 하드웨어
- 수직 확장 중심
- 용량 계획 중요

** 클라우드:**

- 탄력적 자원
- 수평 확장 용이
- 비용 최적화 중요

### 2.6.2 시각적 성능 식별

그래프를 통한 패턴 인식:

- 선형 증가: 정상적인 확장
- 지수적 증가: 문제 징후
- 급격한 변화: 이벤트 발생

### 2.6.3 암달의 확장성 법칙 (Amdahl's Law)

병렬화의 이론적 한계:

```
속도 향상 = 1 / ((1 - P) + P/N)

P = 병렬화 가능한 비율
N = 프로세서 수
```

** 시사점:**

- 순차 실행 부분이 병목
- 병렬화 비율이 높을수록 효과적
- 무한한 프로세서로도 순차 부분에 의해 제한

** 예시:**

- P = 0.9 (90% 병렬화 가능)
- N = 10 프로세서
- 최대 속도 향상 = 1 / (0.1 + 0.9/10) = 5.26배

### 2.6.4 일반 확장성 법칙 (Universal Scalability Law, USL)

**Neil Gunther가 개발**, Amdahl's Law의 확장:

```
C(N) = N / (1 + σ(N-1) + κN(N-1))

N = 프로세서/노드 수
σ = 경합 (contention) 계수
κ = 일관성 (coherency) 계수
```

**USL이 고려하는 것:**

- **σ (Sigma)**: 순차 실행으로 인한 제한 (Amdahl's Law)
- **κ (Kappa)**: 프로세스 간 통신 오버헤드

**USL의 장점:**

- 역행 속도 저하 (retrograde speedup) 모델링 가능
- 실제 벤치마크 데이터로 최적 노드 수 예측
- 확장성 한계 정량화

### 2.6.5 큐 이론 (Queueing Theory)

**Little's Law (리틀의 법칙):**

```
L = λ × W

L = 시스템 내 평균 항목 수
λ = 도착률 (arrival rate)
W = 평균 대기 시간
```

** 응용:**

- 용량 계획
- 병목 분석
- SLA 설정

**M/M/1 큐:**

- Markovian 도착 (포아송)
- Markovian 서비스 (지수)
- 1개 서버

```
평균 대기 시간 = 1 / (μ - λ)

μ = 서비스율
λ = 도착률 (λ < μ 필수)
```

** 사용률과 응답시간:**

```
사용률 ρ = λ / μ

평균 응답시간 = 1 / (μ × (1 - ρ))
```

사용률이 높아질수록 응답시간이 급격히 증가 (비선형)

## 2.7 수용량 계획 (Capacity Planning)

### 2.7.1 리소스 한계

각 자원의 한계 식별:

- CPU: 코어 수, 클럭 속도
- 메모리: 용량, 대역폭
- 디스크: IOPS, 스루풋, 용량
- 네트워크: 대역폭, 연결 수

### 2.7.2 요인 분석 (Factor Analysis)

성능에 영향을 주는 요인 분석:

- 사용자 수
- 동시 연결 수
- 데이터 크기
- 요청 복잡도

### 2.7.3 스케일링 솔루션

** 수직 확장 (Scale Up):**

- 더 강력한 하드웨어
- 한계가 명확

** 수평 확장 (Scale Out):**

- 인스턴스 추가
- 로드 밸런싱 필요
- 상태 관리 복잡

** 자동 확장 (Auto Scaling):**

- 부하에 따른 자동 조정
- 클라우드 환경에서 효과적

## 2.8 통계

### 2.8.1 성능 향상 정량화

성능 개선 측정:

```
향상률 = (이전 - 이후) / 이전 × 100%
속도 향상 = 이전 시간 / 이후 시간
```

### 2.8.2 평균 (Average)

** 산술 평균의 한계:**

- 극단값에 민감
- 분포를 반영하지 못함

** 기하 평균:**

- 비율/배수에 적합
- 벤치마크 점수 비교에 사용

### 2.8.3 표준 편차, 백분위, 중앙값

** 표준 편차 (Standard Deviation):**

- 데이터의 분산 정도
- 일관성 평가에 유용

** 백분위수 (Percentile):**

- p50 (중앙값): 절반이 이 값 이하
- p90: 90%가 이 값 이하
- p99: 99%가 이 값 이하 (테일 지연시간)

** 중앙값 (Median):**

- 극단값에 강건함
- 평균보다 대표성 높은 경우 많음

### 2.8.4 변동 계수 (Coefficient of Variation)

```
CV = 표준 편차 / 평균
```

- 상대적 변동성 측정
- 다른 스케일의 데이터 비교 가능

### 2.8.5 다봉 분포 (Multimodal Distribution)

단일 평균으로 설명 불가능한 분포:

- 캐시 히트 vs 캐시 미스
- SSD vs HDD 접근
- 빠른 경로 vs 느린 경로

→ 히스토그램으로 확인 필수

### 2.8.6 극단값 (Outliers)

극단값 처리:

- 제거: 측정 오류인 경우
- 분석: 실제 문제인 경우

** 테일 지연시간 (Tail Latency):**

- p99, p99.9의 중요성
- 대규모 시스템에서 영향 증폭
