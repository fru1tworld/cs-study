# IAM 정책 평가 로직

## 정책 평가 기본 원칙

AWS는 요청을 받으면 해당 요청에 적용되는 모든 정책을 평가하여 허용 또는 거부 여부를 결정합니다.

### 핵심 원칙

1. ** 기본 거부 (Default Deny)**: 명시적으로 허용되지 않은 모든 요청은 거부
2. ** 명시적 허용 (Explicit Allow)**: Allow 문이 기본 거부를 재정의
3. ** 명시적 거부 (Explicit Deny)**: Deny 문이 모든 허용을 재정의

```
우선순위: 명시적 거부 > 명시적 허용 > 암묵적 거부
```

## 정책 유형별 평가

### 자격 증명 기반 정책 평가

```
사용자의 유효 권한 =
    사용자 직접 연결 정책
    + 사용자가 속한 그룹의 정책

역할의 유효 권한 =
    역할에 연결된 정책
```

### 권한 경계와의 조합

```
유효 권한 = 자격증명 기반 정책 ∩ 권한 경계

예시:
자격증명 정책: S3:*, EC2:*, IAM:*
권한 경계:     S3:*, EC2:*

유효 권한:     S3:*, EC2:*  (IAM은 권한 경계에 없으므로 제외)
```

### SCP와의 조합 (Organizations)

```
유효 권한 = 자격증명 기반 정책 ∩ SCP ∩ 권한 경계

예시:
자격증명 정책: S3:*, EC2:*, RDS:*
SCP:           S3:*, EC2:*
권한 경계:     S3:*, RDS:*

유효 권한:     S3:*  (모든 정책에서 허용하는 것만)
```

## 교차 계정 접근 평가

다른 계정의 리소스에 접근할 때는 ** 양쪽 계정 모두** 에서 허용해야 합니다.

```
┌─────────────────────┐          ┌─────────────────────┐
│      계정 A         │          │      계정 B         │
│   (요청 발신 계정)   │   ───→   │   (리소스 소유 계정) │
├─────────────────────┤          ├─────────────────────┤
│                     │          │                     │
│ 자격증명 기반 정책   │   AND    │ 리소스 기반 정책     │
│ 에서 허용 필요      │          │ 에서 허용 필요       │
│                     │          │                     │
└─────────────────────┘          └─────────────────────┘
```

### 교차 계정 접근 예시

** 계정 A의 사용자 정책:**
```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": "s3:GetObject",
            "Resource": "arn:aws:s3:::account-b-bucket/*"
        }
    ]
}
```

** 계정 B의 S3 버킷 정책:**
```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Principal": {
                "AWS": "arn:aws:iam::111111111111:user/johndoe"
            },
            "Action": "s3:GetObject",
            "Resource": "arn:aws:s3:::account-b-bucket/*"
        }
    ]
}
```

## 같은 계정 내 리소스 기반 정책

같은 계정 내에서는 자격 증명 기반 정책 ** 또는** 리소스 기반 정책 중 하나만 허용해도 접근 가능합니다.

```
같은 계정 내 접근:
유효 권한 = 자격증명 기반 정책 ∪ 리소스 기반 정책

예외:
- 권한 경계가 있으면 교집합 적용
- 세션 정책이 있으면 교집합 적용
```

## 다중 문장 평가

### 같은 정책 내 여러 Statement

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": "s3:*",
            "Resource": "*"
        },
        {
            "Effect": "Deny",
            "Action": "s3:DeleteBucket",
            "Resource": "*"
        }
    ]
}
```

평가 결과:
- `s3:GetObject` → 첫 번째 문장에서 ** 허용**
- `s3:DeleteBucket` → 두 번째 문장에서 ** 거부** (명시적 거부 우선)

### 여러 정책의 조합

```
정책 A: s3:GetObject 허용
정책 B: s3:PutObject 허용
정책 C: s3:Delete* 거부

최종 결과:
- s3:GetObject → 허용
- s3:PutObject → 허용
- s3:DeleteObject → 거부
```

## NotAction, NotResource, NotPrincipal

### NotAction 평가

```json
{
    "Effect": "Allow",
    "NotAction": "iam:*",
    "Resource": "*"
}
```

의미: IAM을 ** 제외한** 모든 서비스의 모든 작업 허용

### NotResource 평가

```json
{
    "Effect": "Deny",
    "Action": "*",
    "NotResource": "arn:aws:s3:::protected-bucket/*"
}
```

의미: protected-bucket을 ** 제외한** 모든 리소스에 대한 모든 작업 거부

### NotPrincipal 사용 주의

```json
// ⚠️ 주의 필요
{
    "Effect": "Deny",
    "NotPrincipal": {"AWS": "arn:aws:iam::123456789012:user/admin"},
    "Action": "*",
    "Resource": "*"
}
```

** 경고:** `NotPrincipal`과 `Deny`를 함께 사용하면 admin을 제외한 모든 주체가 거부됩니다. 이는 의도치 않게 다른 사용자의 접근을 차단할 수 있습니다.

권장 대안:
```json
{
    "Effect": "Deny",
    "Principal": "*",
    "Action": "*",
    "Resource": "*",
    "Condition": {
        "ArnNotEquals": {
            "aws:PrincipalArn": "arn:aws:iam::123456789012:user/admin"
        }
    }
}
```

## 조건 평가

### AND 로직

같은 Condition 블록 내의 여러 조건은 AND로 평가됩니다.

```json
"Condition": {
    "StringEquals": {"aws:RequestTag/Env": "Prod"},
    "IpAddress": {"aws:SourceIp": "10.0.0.0/8"}
}
```

결과: 두 조건 ** 모두** 만족해야 함

### OR 로직 (조건 키의 여러 값)

```json
"Condition": {
    "StringEquals": {
        "aws:RequestTag/Env": ["Dev", "Test", "Prod"]
    }
}
```

결과: Env 태그가 Dev, Test, Prod 중 ** 하나** 라도 일치하면 조건 충족

## 정책 시뮬레이터

정책 평가 결과를 미리 테스트할 수 있습니다.

### 콘솔 시뮬레이터

```
IAM 콘솔 → 정책 시뮬레이터
1. 테스트할 사용자/역할 선택
2. 서비스와 작업 선택
3. 리소스 지정
4. 시뮬레이션 실행
```

### CLI 시뮬레이터

```bash
# 정책 시뮬레이션
aws iam simulate-principal-policy \
    --policy-source-arn arn:aws:iam::123456789012:user/johndoe \
    --action-names s3:GetObject \
    --resource-arns arn:aws:s3:::my-bucket/file.txt
```

## 문제 해결

### 접근 거부 원인 파악

1. ** 명시적 거부 확인**
   - SCP에서 거부하는지 확인
   - 권한 경계에서 거부하는지 확인
   - 자격 증명 기반 정책에서 거부하는지 확인

2. ** 허용 누락 확인**
   - 자격 증명 기반 정책에 허용이 있는지 확인
   - 리소스 기반 정책이 필요한지 확인 (교차 계정)

3. ** 조건 확인**
   - 조건이 현재 요청 컨텍스트와 일치하는지 확인
   - MFA, IP 주소, 시간 등

### CloudTrail 활용

```
CloudTrail 이벤트에서 확인:
- errorCode: AccessDenied
- errorMessage: 거부 원인 상세 정보
- userIdentity: 요청 주체 정보
```
