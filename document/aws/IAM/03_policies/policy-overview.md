# IAM 정책 개요

## IAM 정책이란?

IAM 정책은 자격 증명이나 리소스와 연결될 때 해당 권한을 정의하는 JSON 문서입니다. 정책은 사용자, 그룹, 역할이 수행할 수 있는 작업, 대상 리소스, 적용 조건을 제어합니다.

## 정책의 기본 동작

### 기본 규칙
1. ** 모든 요청은 기본적으로 거부됩니다** (암묵적 거부)
2. ** 명시적 허용이 암묵적 거부를 재정의** 합니다
3. ** 명시적 거부가 모든 허용을 재정의** 합니다

### 정책 평가 흐름

```
요청 수신
    │
    ▼
┌───────────────────┐
│  명시적 거부 확인  │ ──거부 있음──→ 요청 거부
└───────────────────┘
    │ 거부 없음
    ▼
┌───────────────────┐
│  명시적 허용 확인  │ ──허용 있음──→ 요청 허용
└───────────────────┘
    │ 허용 없음
    ▼
  요청 거부
(암묵적 거부)
```

## 7가지 정책 유형

### 1. 자격 증명 기반 정책 (Identity-based Policies)

IAM 사용자, 그룹, 역할에 연결되어 해당 자격 증명이 수행할 수 있는 작업을 정의합니다.

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Action": "s3:GetObject",
            "Resource": "arn:aws:s3:::my-bucket/*"
        }
    ]
}
```

** 특징:**
- 관리형 정책 (AWS 관리형, 고객 관리형)
- 인라인 정책
- Principal 요소 불필요 (연결된 자격 증명이 주체)

### 2. 리소스 기반 정책 (Resource-based Policies)

S3 버킷, SQS 대기열 등 리소스에 직접 연결됩니다.

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Principal": {
                "AWS": "arn:aws:iam::123456789012:user/johndoe"
            },
            "Action": "s3:GetObject",
            "Resource": "arn:aws:s3:::my-bucket/*"
        }
    ]
}
```

** 특징:**
- 인라인만 지원 (관리형 없음)
- Principal 요소 필수
- 교차 계정 접근에 유용

### 3. 권한 경계 (Permissions Boundaries)

자격 증명 기반 정책이 부여할 수 있는 최대 권한을 설정합니다.

```
권한 경계의 역할:
┌─────────────────────────────────────┐
│         권한 경계                    │
│    (최대 허용 가능한 권한)           │
│  ┌─────────────────────────────┐   │
│  │    자격 증명 기반 정책       │   │
│  │      (부여된 권한)          │   │
│  └─────────────────────────────┘   │
└─────────────────────────────────────┘
유효 권한 = 교집합
```

** 특징:**
- 권한을 부여하지 않음
- 최대 권한만 설정
- 사용자와 역할에만 적용 (그룹 불가)

### 4. AWS Organizations 서비스 제어 정책 (SCP)

조직 내 계정의 최대 권한을 제한합니다.

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Deny",
            "Action": [
                "ec2:*Reserved*",
                "ec2:*Purchase*"
            ],
            "Resource": "*"
        }
    ]
}
```

** 특징:**
- 권한을 부여하지 않음
- 조직, OU, 계정 수준에서 적용
- 관리 계정에는 적용 안 됨

### 5. AWS Organizations 리소스 제어 정책 (RCP)

조직 내 리소스의 최대 권한을 제한합니다.

** 특징:**
- 리소스 기반 관점에서 제어
- 조직 전체 리소스에 적용

### 6. 액세스 제어 목록 (ACL)

JSON 형식이 아닌 유일한 정책 유형입니다.

```
S3 ACL 예시:
- private: 소유자만 접근
- public-read: 모든 사람이 읽기 가능
- authenticated-read: 인증된 AWS 사용자만 읽기
```

** 특징:**
- 교차 계정 권한만 제어
- S3, VPC에서 주로 사용
- 레거시 방식 (정책 권장)

### 7. 세션 정책 (Session Policies)

역할을 맡거나 연합 사용자로 접근할 때 세션에 적용되는 정책입니다.

```bash
aws sts assume-role \
    --role-arn arn:aws:iam::123456789012:role/MyRole \
    --role-session-name my-session \
    --policy '{"Version":"2012-10-17","Statement":[...]}'
```

** 특징:**
- 역할 정책을 더 제한
- 권한을 확장할 수 없음
- 세션 기간 동안만 유효

## 정책 유형별 사용 사례

| 정책 유형 | 사용 사례 |
|-----------|----------|
| 자격 증명 기반 | 사용자/그룹/역할에 권한 부여 |
| 리소스 기반 | 교차 계정 접근, 리소스별 접근 제어 |
| 권한 경계 | 권한 위임 시 최대 권한 제한 |
| SCP | 조직 전체 보안 가드레일 |
| RCP | 리소스 수준 조직 제어 |
| ACL | 레거시 S3 접근 제어 |
| 세션 정책 | 임시 세션 권한 제한 |

## 정책 평가 순서

여러 정책 유형이 적용될 때의 평가 순서입니다.

```
┌───────────────────────────────────────────────────────────┐
│                    요청 평가 시작                          │
├───────────────────────────────────────────────────────────┤
│                                                           │
│  1. 명시적 거부 확인 (모든 정책)                           │
│     └─→ 하나라도 Deny면 즉시 거부                         │
│                                                           │
│  2. SCP 확인 (Organizations 사용 시)                      │
│     └─→ 허용하지 않으면 암묵적 거부                       │
│                                                           │
│  3. 리소스 기반 정책 확인                                  │
│     └─→ 교차 계정: 양쪽 모두 허용 필요                    │
│     └─→ 같은 계정: 하나만 허용해도 가능                   │
│                                                           │
│  4. 자격 증명 기반 정책 확인                               │
│     └─→ 허용 필요                                         │
│                                                           │
│  5. 권한 경계 확인 (설정된 경우)                           │
│     └─→ 허용 필요                                         │
│                                                           │
│  6. 세션 정책 확인 (역할 세션인 경우)                      │
│     └─→ 허용 필요                                         │
│                                                           │
│  모든 조건 충족 → 요청 허용                               │
│                                                           │
└───────────────────────────────────────────────────────────┘
```
